Your environment decoded
What your draw.io diagram actually means for OpenShift

Before planning anything, a senior engineer reads the existing fabric and asks: "what does OpenShift need from each layer?" Here is exactly that reading, layer by layer.

Network layers OpenShift cares about

L1 — Internet Edge ISP A/B → Edge Routers → Palo Alto NGFW

OpenShift's ingress traffic from external customers will arrive here. The route path is: Cloudflare DNS → DigiCert-signed TLS → ISP → Edge router → Palo Alto (untrust→DMZ) → F5 BIG-IP (TLS termination/WAF) → PA inside interface → spine → leaf → OpenShift ingress pod.

Your Palo Alto policy will need rules to allow the F5 pool IPs to reach OpenShift node IPs on port 443/80 (or whatever NodePort/LoadBalancer you use).

L2 — Spine-Leaf fabric Nexus 9504 spine × 2 → Nexus 93180YC-FX leaf × 3

This is a pure L3 spine, L2 leaf design (no NSX-T, no VCF — confirmed in your notes). Each leaf is ToR (Top-of-Rack). OpenShift VMs will live on ESXi hosts connected to these leafs. All OCP traffic within the cluster stays on the leaf/spine fabric — 25G downlinks, 100G uplinks are excellent for OpenShift's SDN overlay (OVN-Kubernetes).

Key action: You will need a dedicated VLAN for OpenShift nodes (we call it VLAN 50, 10.50.0.0/24) trunked to all three leafs, with an SVI on the spine for L3 routing.

L3 — Compute 5 × HP DL380 Gen11 running ESXi + vSAN

OpenShift VMs will run as VMware virtual machines on these hosts. You have 5 hosts. That matters enormously for node planning (covered in section 07). vSAN provides the shared datastore — OCP VMs will be stored there. Hosts 2 and 3 have FC HBAs (used for Oracle RAC), hosts 4 and 5 do not. This is fine — OpenShift VMs use vSAN, not raw FC.

You will need a vSphere / VMware IPI (Installer Provisioned Infrastructure) install or UPI with VMs pre-created in vCenter.

L4 — Security & PKI AD CS Internal CA + DigiCert + Cloudflare

This is where most candidates get confused. OpenShift needs certificates for: (a) its API server, (b) its internal etcd, (c) the default ingress wildcard. You have two CA hierarchies and each has a job. This is covered in full detail in section 10.

Installation methods
IPI vs UPI vs Agent-based — what to choose in your environment and why

The three main methods

Recommended for you

IPI — Installer Provisioned Infrastructure

The installer creates VMs in vCenter for you. You give it a service account and a network, and it handles the rest: VM creation, RHCOS injection, bootstrap sequence.

Best for: VMware environments with vCenter API access.

AutomationHigh — nearly full auto
ControlLess granular
PrereqsvCenter creds + DNS + LB

UPI — User Provisioned Infrastructure

You create the VMs manually (or via Terraform/Ansible). The installer generates ignition configs; you feed them to the VMs at boot.

Best for: Heavily regulated environments where every VM must be approved and documented before creation, or where vCenter permissions are restricted.

AutomationLower — you own VM lifecycle
ControlMaximum
PrereqsDNS + LB + DHCP or static IPs

Agent-based

Newer method. You create a bootable agent ISO and boot nodes from it. No bootstrap VM needed. Good for disconnected/air-gapped environments.

Best for: Air-gapped or restricted environments where bootstrap VMs are disallowed.

AutomationMedium
ControlHigh
PrereqsDNS + LB; no bootstrap VM
For NovaCrestBank — which method?
Choose IPI If your vCenter service account can create VMs and the security team approves automated provisioning. Fastest path, least manual error. This is the most common choice in enterprise banking environments with standard vSphere setups.
Choose UPI If your change management process requires VMs to be created via approved ITSM tickets before installation. Many banks with strict ITIL processes use UPI so that each VM has a change record before it exists.
What actually happens in real enterprise banking
Most Tier-1 and Tier-2 banks use UPI with Terraform on VMware. This gives them a full IaC audit trail (every VM has a Terraform plan/apply in a Git commit) which satisfies change management, PCI-DSS audit requirements, and gives the infrastructure team maximum control. The Terraform code is stored in a protected GitLab repo, reviewed by two senior engineers, and applied only from a dedicated automation account. Consider this approach if your bank has similar governance requirements.
Node planning deep-dive
How to answer "how many VMs?" with confidence — before anyone asks
The core question answered upfront
When a senior official asks "how many VMs?", you say: "Minimum 7 for a functional HA cluster, 10–13 for production with infrastructure separation, and we will grow from there." Here is exactly why, and how to arrive at those numbers for NovaCrestBank.

Node roles — what each one does

Control Plane nodes — always 3 (or 5 in very large clusters)

Control plane nodes (formerly called "master nodes") run the Kubernetes API server, etcd, scheduler, and controller manager. They are the brain of the cluster.

Why always 3? etcd uses the Raft consensus algorithm and requires a quorum — more than half the members must be healthy to elect a leader. With 3 nodes, you can lose 1 and still have quorum (2 of 3). With 2 nodes, losing 1 = total cluster loss. With 1 node = no HA at all.

Why not 2 or 4? Only odd numbers work correctly with Raft. 3, 5, or 7 — never even numbers. For most enterprise clusters, 3 is the answer. 5 only if you have 2 data centers and need geographic split-brain protection.

Minimum3 (non-negotiable for HA)
Recommended3 for NovaCrestBank (single DC)
Spec per node8 vCPU, 32 GB RAM, 120 GB OS disk
Scheduling workloads?No — control plane nodes are tainted; app pods do not land here
Worker nodes — where your applications run

Worker nodes run application workloads (pods). The number is entirely driven by the services you plan to run. This is the most flexible part of the answer.

Minimum for meaningful HA: 2 workers (pods can be scheduled across 2 nodes for basic redundancy). But 2 is bare minimum — real production starts at 3+ workers.

Minimum (functional)2 workers
Minimum (production HA)3 workers
Spec per node16 vCPU, 64 GB RAM, 120 GB OS disk (baseline)
Grow later?Yes — add workers anytime with MachineSet or manual node join
Bootstrap node — temporary, destroyed after install

The bootstrap node is a temporary VM that exists only during installation. It hosts a temporary control plane and serves ignition configs to the master nodes. Once the masters are up and etcd is healthy, the bootstrap node is destroyed.

Important: The bootstrap is not part of your permanent cluster. In IPI, the installer creates and destroys it automatically. In UPI, you create and then delete it manually.

Lifespan~30–45 minutes during install only
Spec4 vCPU, 16 GB RAM, 100 GB disk
Permanent?No — delete after install completes

After bootstrap is removed: 5 permanent VMs. This is the theoretical minimum. Do not use this for anything production.

Infrastructure nodes
The "hidden" third tier most textbooks skip — critical for production
This is what separates enterprise OCP from textbook OCP
Most beginners think OpenShift has only two node types: masters and workers. Real production clusters have a third type — infrastructure nodes. Missing this leads to resource contention, licensing cost overruns, and operational nightmares.

What are infrastructure nodes?

Infrastructure nodes are worker nodes with a special label and taint — node-role.kubernetes.io/infra — that makes OpenShift's own internal platform components run there instead of on your application worker nodes.

What runs on infrastructure nodes?

ComponentWhy it must go on infra nodes
OpenShift Router (HAProxy Ingress)Receives all external app traffic — needs dedicated resources, not competing with app pods
Image Registry (internal)Stores all container images built in-cluster — high storage I/O, needs dedicated disk
Monitoring stack (Prometheus, Alertmanager, Thanos)Stores metrics — can consume 50–200 GB RAM in production; must not starve app pods
Logging (OpenShift Logging / Loki / EFK)High I/O, high memory — definitely not on workers
OpenShift Cluster Sampling / TelemetryInternal platform component

Why does this matter for licensing?

Cost impact — very important for management conversations
Red Hat charges per core for OpenShift subscriptions. Infrastructure nodes running only platform components (not your application workloads) are not counted in your subscription core count in many licensing agreements. This means separating infra nodes can reduce your licensing cost significantly. Verify this with your Red Hat account team — it is a real enterprise conversation.

How many infrastructure nodes?

Minimum

2 infra nodes (not recommended — no HA for the ingress router)

Recommended for NovaCrestBank

3 infra nodes — one per availability zone (in your case, one per ESXi host group / rack). This gives full HA for the ingress router (3 replicas across 3 nodes) and monitoring.

Spec per node16 vCPU, 64 GB RAM, 500 GB storage disk

How to make a node an infrastructure node

# Label the node
oc label node worker03.ocp.novacrestbank.local node-role.kubernetes.io/infra=''
oc label node worker03.ocp.novacrestbank.local node-role.kubernetes.io/worker-

# Taint it so only infra-aware workloads schedule here
oc adm taint node worker03.ocp.novacrestbank.local \
  node-role.kubernetes.io/infra=reserved:NoSchedule

# Then move the router to infra nodes
oc patch ingresscontroller/default -n openshift-ingress-operator \
  --type=merge \
  -p '{"spec":{"nodePlacement":{"nodeSelector":{"matchLabels":{"node-role.kubernetes.io/infra":""}},"tolerations":[{"effect":"NoSchedule","key":"node-role.kubernetes.io/infra"}]}}}'
Sizing decisions
How to actually decide how many nodes — a mental framework for real environments

The three sizing approaches — which one applies?

Approach A Start minimum, grow later (most common)

You start with the minimum viable cluster (3 masters + 3 infra + 3 workers) and add worker nodes as services are onboarded. This is extremely common in enterprise. OpenShift's MachineSet API makes adding nodes almost trivial.

When to use this: The applications/services that will run on the cluster are not yet fully defined, or are being migrated gradually. You don't know the full workload picture yet.

Risk: Under-sizing at launch can cause disruption as you scramble to add nodes when workloads arrive. Mitigate by adding at least one extra worker node beyond your first wave of workloads.

Approach B Capacity-based (service-driven sizing)

Before install, you collect resource requests (CPU/memory) from every team/application that will migrate to OCP. Sum them up, add 40% headroom buffer, divide by per-node capacity — that gives you the worker node count.

Formula(Total vCPU needed × 1.4) ÷ vCPU per worker = worker count
Do same forRAM and storage separately
Take the maxWhichever resource (CPU/RAM/disk) gives highest node count — that's your answer

Example for NovaCrestBank: 10 banking microservices, each requesting 4 vCPU / 16 GB RAM. Total: 40 vCPU / 160 GB RAM. With 40% buffer: 56 vCPU / 224 GB RAM. With 16-vCPU/64-GB workers: ceil(56/16) = 4, ceil(224/64) = 4. Answer: 4 workers.

Approach C VMware host constraint-driven

In your environment, you have 5 ESXi hosts. This physically limits how many VMs you can place, especially if vSAN needs host resources. A realistic anti-affinity rule (OCP masters never on the same ESXi host) means your 3 masters each need to go on a different host. That leaves 2 hosts for infra + worker VMs.

This is the real constraint in your environment. You may plan 9 permanent OCP VMs but also share those ESXi hosts with Oracle RAC, banking apps, and management VMs. Calculate the remaining vCPU and RAM on each host, then fit OCP VMs into that space.

PKI & certificate strategy
Three CAs, two trust domains — exactly which CA signs what, and why
Your PKI landscape
You have: (1) AD CS — offline Root CA → online Issuing CA (private, internally trusted), (2) DigiCert — publicly trusted, internet-facing, (3) Cloudflare — not a CA, it's your public DNS + CDN. Here is the exact job each CA does for OpenShift.

Certificate responsibility map

CertificateSigned byWhy
OCP API server (api.ocp.novacrestbank.local) AD CS Issuing CA Internal endpoint — only accessed by internal systems and admins via VPN. Public CA would be wasteful and risky.

How to inject the AD CS CA into OpenShift — the additionalTrustBundle

OpenShift does not trust your internal AD CS by default. You must inject the CA chain into the cluster's trust store. This is done in two places:

# Step 1: Export your AD CS Issuing CA + Root CA chain to PEM format
# (Do this from a Windows machine with the AD CS role or from MMC)

# Step 2: Add it to install-config.yaml (before install)
additionalTrustBundle: |
  -----BEGIN CERTIFICATE-----
  [paste your Root CA PEM here]
  -----END CERTIFICATE-----
  -----BEGIN CERTIFICATE-----
  [paste your Issuing CA PEM here]
  -----END CERTIFICATE-----

# Step 3: Post-install — add to the cluster-wide proxy/CA config
oc create configmap custom-ca \
  --from-file=ca-bundle.crt=/path/to/novacrestbank-ca-chain.pem \
  -n openshift-config

oc patch proxy/cluster \
  --type=merge \
  -p '{"spec":{"trustedCA":{"name":"custom-ca"}}}'

Certificate for the API server — replacing the self-signed default

Always do this in production
OpenShift ships with a self-signed certificate for the API server. Replace it with your AD CS Issuing CA-signed cert immediately after install. Leaving the self-signed cert causes constant TLS warnings in tools like oc and CI/CD pipelines. 
# Create a secret with your API server cert and key (signed by AD CS Issuing CA)
oc create secret tls api-server-cert \
  --cert=api.ocp.novacrestbank.local.crt \
  --key=api.ocp.novacrestbank.local.key \
  -n openshift-config

# Patch the API server to use it
oc patch apiserver cluster \
  --type=merge \
  -p '{"spec":{"servingCerts":{"namedCertificates":[{"names":["api.ocp.novacrestbank.local"],"servingCertificate":{"name":"api-server-cert"}}]}}}'

Networking & VLANs
What VLANs to create, how to trunk them, and how OVN-Kubernetes overlays everything

New VLAN for OpenShift — VLAN 50

Add VLAN 50 (10.50.0.0/24) to the Nexus fabric — trunked on all three leaf switches. Create an SVI on the spine for inter-VLAN routing. This VLAN carries all OCP node traffic (both management and pod-to-pod at the IP level).

OVN-Kubernetes overlay — the "invisible" layer
OCP uses OVN-Kubernetes as its SDN (Software Defined Networking). It creates a second IP space (the pod network, e.g. 10.128.0.0/14) that lives on top of your VLAN 50. Pod-to-pod traffic is encapsulated in Geneve tunnels between nodes. From the Nexus fabric's perspective, it just sees VLAN 50 IP-to-IP traffic — it has no visibility into the pod overlay. You do NOT need any changes to the Nexus fabric for pod networking. That is the magic of OVN-Kubernetes.

vCenter objects, service accounts, and vSphere settings to configure before running the installer

vCenter objects to create

  • 1
    Create a vCenter folder for OCP VMs

    In vCenter → VMs & Templates → Create folder: OCP-Production. All OCP VMs will go here. Helps with organization, change tracking, and applying vSphere permissions.

  • 2
    Create a vCenter service account for the installer

    Create a dedicated AD service account: svc-ocp-installer@novacrestbank.local. Assign it the following vCenter roles on the relevant objects: VM create/modify/delete at the cluster level, datastore access on the vSAN datastore, network access on the VLAN 50 portgroup. Never use a personal account for the installer.

  • 3
    Create a vSphere Distributed Port Group for VLAN 50

    In vCenter → Networking → Your Distributed Switch → Add Port Group. Name: VLAN50-OCP-Nodes, VLAN ID: 50. This is the network all OCP VMs will attach to.

  • 4
    Upload RHCOS OVA (for UPI only)

    If doing UPI: download the RHCOS OVA from mirror.openshift.com and upload it to the vSAN datastore as a VM template. For IPI: skip this — the installer handles it automatically.

  • 5
    Create VM anti-affinity rules

    vCenter → Cluster → Configure → VM/Host Rules → Add Rule. Create one rule: "OCP-Masters-Anti-Affinity" — type "Separate VMs", members: master01, master02, master03. This ensures they never land on the same ESXi host.

  • 6
    Download and trust vCenter root CA

How to think & decide
The mental model senior engineers use — from zero information to confident answers

The senior engineer's decision tree

When faced with any OpenShift planning question, senior engineers move through layers:

  • 1
    Start from the constraint, not the ideal

    Ask: "What physical/organizational constraints exist?" In NovaCrestBank's case: 5 ESXi hosts, existing vSAN, existing Palo Alto policy, AD CS, change management process. These constraints shape every answer before you open a Red Hat doc.

  • 2
    Separate "must-have day-1" from "can grow into"

    Control plane: always 3, always on day 1. Infra nodes: strongly recommended day 1. Worker count: start with 3, grow based on workload. This framing lets you give a concrete answer to management while leaving room for growth.

  • 3
    Think in failure domains

    "What happens if ESXi host 1 goes down?" → It must not take down more than one master. "What if the F5 VIP for API is unreachable?" → That is a cluster outage. Work backwards from failure scenarios to your placement and redundancy decisions.

  • 4
    DNS and PKI before networking, networking before compute

    This is the correct prep order. DNS errors cause the hardest-to-debug installation failures. PKI errors surface at strange times post-install. Get these solid first, then firewall rules, then VM creation.

  • 5
    When in doubt, answer with the minimum + growth path

    Saying "we start with 9 nodes and scale to 15 as services are onboarded" is a more confident, credible answer than giving a precise number you can't justify. The minimum is fixed (3 masters, 3 infra); the workers are a function of workload.

Answering management's question — the script

Management: "How many VMs do we need for OpenShift?"

Your answer:

"For a production-grade, highly available OpenShift cluster in our environment, we start with 9 permanent VMs: 3 control plane nodes (the brain of the cluster — always odd-numbered for consensus), 3 infrastructure nodes for platform services like load balancing and monitoring, and 3 worker nodes for our initial application workloads. There is also 1 temporary bootstrap VM that exists only during the 45-minute install window and is then automatically destroyed. Worker nodes scale horizontally as we onboard more services — each additional node gives us roughly 16 vCPUs and 64 GB RAM of application capacity. The control plane stays at 3 unless we expand to a second data center."

Q&A — Real-world concerns
The actual questions that come up in real enterprise deployments — answered directly

Q: Can we install OpenShift without internet access?

Yes — this is called a disconnected or air-gapped install. You mirror all OCP images to an internal registry (Red Hat Quay or Harbor), then point the installer at your internal registry instead of quay.io. For NovaCrestBank, your Palo Alto probably allows outbound HTTPS to specific destinations — verify with your security team whether a connected or semi-disconnected install is appropriate. Banking environments often do connected-but-filtered installs: the cluster reaches Red Hat registries through a Palo Alto URL-filtered policy during install, then is locked down post-install.

Q: What if the install fails partway through?

Run openshift-install gather bootstrap first — this collects logs from the bootstrap VM. Common failure causes in enterprise environments: DNS missing (PTR records forgotten), firewall blocking port 6443 or 22623, NTP skew, vCenter cert not trusted by installer, pull secret expired. Never re-run the installer on top of a failed install — always destroy and start clean: openshift-install destroy cluster --dir=~/ocp-install.

Q: Our change management process requires a Change Request before any VM is created. How does that work with IPI?

This is the most common governance concern in banking. The answer: raise a single Change Request that describes "Creation of 10 VMs (1 bootstrap + 3 masters + 3 infra + 3 workers) in vCenter folder OCP-Production on VLAN 50" with a full VM specification sheet. The IPI installer then executes this approved change. Alternatively, use UPI with Terraform — generate the Terraform plan, attach it as evidence to the Change Request, have it approved, then apply. Many banks prefer this because each VM has a code commit as its audit trail.

Q: We have HSMs for key management. Does OpenShift integrate with Thales Luna?

Yes — OpenShift's etcd encryption and key management can be integrated with external key management systems. Red Hat Advanced Cluster Security integrates with KMIP-compliant HSMs including Thales Luna Network HSM. For PCI-DSS environments (which NovaCrestBank certainly is), the HSM integration for etcd encryption at rest is strongly recommended. This is a post-install configuration requiring the OpenShift KMS provider and Thales KMIP configuration.

Q: The F5 is already used for other applications. Can OCP share it?

Yes — the F5 BIG-IP is perfectly capable of hosting multiple virtual servers. You simply create two new virtual server objects (VIPs): one for the API (10.50.0.200:6443) and one for ingress (10.50.0.201:443/80). Each has its own pool pointing to the relevant OCP nodes. This does not interfere with existing VIPs. The F5 admin will need pool member IPs and health check settings — provide them the node IPs and the health check paths (/readyz on port 6443 for API, HTTP on 1936 for the router status page).

Q: How does OCP know about our NetApp NAS for persistent storage?

After install, deploy the NetApp Trident CSI driver (also called Astra Trident). It creates a StorageClass in OpenShift that maps to NFS exports on your NetApp AFF A400. When a developer creates a PersistentVolumeClaim, Trident automatically creates an NFS volume on the NetApp and mounts it into the pod. This is how the internal image registry, logging, and any stateful application pods get their storage.

Q: After install, where is the admin password?

The installer creates a temporary admin account called kubeadmin. Its password is stored in ~/ocp-install/auth/kubeadmin-password. Store this immediately in CyberArk. This account has cluster-admin privileges and should be disabled after you configure Active Directory authentication. The kubeconfig is at ~/ocp-install/auth/kubeconfig — this gives direct API access without a password, treat it like a root private key.
