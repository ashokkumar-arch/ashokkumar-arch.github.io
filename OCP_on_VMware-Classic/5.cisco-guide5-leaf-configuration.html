The leaf switch does what the spine cannot — and vice versa
The spine is pure Layer 3. It has no VLANs, no MAC learning, no server ports. The leaf is the boundary between Layer 2 (VLANs, server ports) and Layer 3 (routing, BGP). Every VLAN in NovaCrestBank terminates at the leaf switch — the leaf holds the SVI (the IP gateway for that VLAN) and routes traffic between VLANs and up to the spines. The spine then routes between leafs.

The leaf switch's three roles in NovaCrestBank

  • 1
    Layer 2 access switch — carries VLANs to servers

    The ESXi downlink ports (Eth1/1 through Eth1/12 on leaf01) are trunk ports. They carry all 7 production VLANs simultaneously: management (10), vMotion (20), vSAN (25), NAS (30), OCP (50), VM production (60), and Windows infrastructure (70). The ESXi host's virtual distributed switch then separates this single trunk into distinct port groups — one per VLAN. From the leaf's perspective: one physical port, many VLANs.

  • 2
    Layer 3 gateway — one SVI per VLAN

    For each VLAN the leaf carries, it also has an SVI (Switched Virtual Interface) — a virtual Layer 3 interface that acts as the default gateway for all devices in that VLAN. When an ESXi host's vmkernel adapter on VLAN 10 needs to send a packet outside the VLAN 10 subnet, it sends it to the gateway IP (10.10.0.1). That IP lives on the leaf's SVI for VLAN 10. The leaf routes the packet — either to another VLAN SVI (inter-VLAN routing) or up to the spine (inter-leaf routing).

  • 3
    BGP speaker — advertises SVI subnets to spines

    Each leaf's SVI subnets (10.10.0.0/24, 10.20.0.0/24, etc.) are advertised via BGP to both spines. The spine then knows that "10.10.0.0/24 is reachable via leaf01" and can route traffic there from any other leaf. This is how an ESXi host on leaf03 reaches the management VLAN gateway — BGP tells the spine where it lives.

vPC — virtual Port Channel explained
The technology that lets two physical leaf switches appear as one switch to an ESXi host — why it exists and exactly how it works
The problem vPC solves — single points of failure in server connectivity
Without vPC: each ESXi host's NIC pair (vmnic0 + vmnic1) would connect to a single leaf switch. If that leaf switch fails, both NICs lose connectivity — the ESXi host is isolated from the network. With vPC: vmnic0 connects to leaf01 and vmnic1 connects to leaf02. The host bonds them with LACP into a single logical link. If leaf01 fails, vmnic1 keeps the host connected via leaf02. This is true active-active redundancy at the leaf layer.

How vPC works — the three components

vPC keepalive link

A dedicated heartbeat link between leaf01 and leaf02 — used only to detect if the peer switch is alive. Runs over the management network (mgmt0 on OOB). If the peer link goes down but the keepalive is still up, the switch with the lower vPC role priority becomes secondary and suspends its vPC interfaces — preventing a split-brain scenario where both switches think they are the only active one.

vPC peer link

A high-bandwidth trunk link between leaf01 and leaf02 — typically two 100G ports bonded into a port channel. Carries all vPC VLANs. Used to synchronise MAC tables, IGMP snooping state, and to forward traffic when needed. Critical: if the peer link goes down and the keepalive is also down, the secondary leaf suspends its vPC member ports (the ESXi connections) — this is the correct behaviour.

leaf01 configuration order
The exact sequence of configuration steps — why the order matters and what depends on what
Always configure vPC infrastructure before VLANs and SVIs
vPC must be set up first because the peer link is what carries VLAN traffic between the two leaf switches. If you create VLANs before vPC is running, the VLANs will work but without the redundancy that vPC provides. More importantly: the vPC keepalive and peer link must be established before the ESXi downlinks are configured — otherwise a brief traffic black-hole can occur as vPC negotiates which ports are active.

vPC infrastructure — keepalive and domain
The first step in making two leaf switches work as one — configuring the heartbeat and establishing the vPC domain
leaf01 — vPC keepalive and domain configuration 
[admin@jumphost ~]$ ssh admin@10.40.0.20
Password: NovaCrest$Leaf2026!
leaf01# configure terminal
leaf01(config)#

# ── Step 1: Configure the vPC keepalive link ───────────────────────
# The keepalive runs over the management VRF (OOB network).
# leaf01 mgmt0 = 10.40.0.20, leaf02 mgmt0 = 10.40.0.21
# The keepalive is a UDP heartbeat sent every 1 second.
# If 5 keepalives are missed → peer is declared dead.

leaf01(config)# vpc domain 10
↑ Domain ID = 10. Both leaf01 and leaf02 must use the SAME domain ID.
  Any number 1-1000 works — we use 10 to match NovaCrestBank's VLAN 10 management.
  The domain ID identifies this vPC pair to the rest of the network.

leaf01(config-vpc-domain)# peer-keepalive destination 10.40.0.21 source 10.40.0.20 vrf management
↑ Keepalive: this switch (10.40.0.20) sends heartbeats to peer (10.40.0.21).
  "vrf management" = use the OOB management VRF (mgmt0 interface).
  This means the keepalive travels over the OOB switch — completely independent
  of the production network. Even if ALL production links fail, the keepalive
  continues via OOB, preventing false split-brain detection.

leaf01(config-vpc-domain)# role priority 100
↑ Role priority: lower number = preferred primary.
  leaf01 = 100 → will become vPC PRIMARY.
  leaf02 = 200 (configured next) → will become vPC SECONDARY.
  The primary makes decisions about vPC consistency checks.
  If primary fails: secondary takes over automatically (no manual intervention).

leaf01(config-vpc-domain)# peer-switch
↑ peer-switch mode: makes both leaf switches appear as a single STP root bridge.
  Without this: each switch has its own STP bridge MAC — ESXi hosts see two different
  switches. With peer-switch: both use the same virtual MAC for STP — ESXi sees one switch.
  This simplifies STP and prevents brief STP reconvergence when the primary fails.

leaf01(config-vpc-domain)# peer-gateway
↑ peer-gateway: allows either leaf switch to route traffic for the other leaf's MAC.
  Without this: if leaf02 has the gateway MAC for an SVI but a packet arrives at leaf01,
  leaf01 drops it (wrong destination MAC). With peer-gateway: leaf01 responds on behalf
  of leaf02's SVI MAC. Critical for proper active-active forwarding.

leaf01(config-vpc-domain)# ip arp synchronize
↑ Synchronise ARP table between leaf01 and leaf02.
  When a device ARPs for the gateway on leaf01, leaf02 also learns the entry.
  Prevents ARP floods when traffic shifts from one leaf to the other.

leaf01(config-vpc-domain)# auto-recovery
↑ If both links (peer link AND keepalive) go down simultaneously, and then one comes back,
  auto-recovery allows the switch to restore its vPC ports without manual intervention.
  Essential for preventing prolonged outages during failure recovery.

leaf01(config-vpc-domain)# auto-recovery reload-delay 300
↑ After a reboot, wait 300 seconds before forming vPC.
  Gives time for BGP to converge and the peer to be ready.
  Prevents a just-rebooted switch from taking over prematurely.

leaf01(config-vpc-domain)# exit

# Verify the vPC domain is configured and keepalive is working
leaf01(config)# do show vpc peer-keepalive

vPC peer link — Po1 between leaf01 and leaf02
The high-bandwidth trunk that synchronises the two leaf switches — two 100G ports bonded with LACP
The peer link must use dedicated ports — not shared with other traffic
The vPC peer link carries synchronisation traffic (MAC tables, IGMP state, ARP sync) in addition to any transit traffic that needs to cross from one leaf to the other. Best practice: use dedicated 100G ports for the peer link — do not share with server connections. NovaCrestBank uses Ethernet1/47 and Ethernet1/48 on each leaf (the two ports just below the 100G uplinks) for the peer link.
leaf01 — vPC peer link configuration (Po1) 
leaf01(config)#

# ── Step 1: Create Port-channel 1 — the peer link ──────────────────
leaf01(config)# interface port-channel1
leaf01(config-if)# description vPC peer link to leaf02 — Po1
leaf01(config-if)# switchport
↑ This port channel IS a Layer 2 (switchport) interface — not routed.
  The peer link carries VLANs (Layer 2 traffic) between the two leafs.
leaf01(config-if)# switchport mode trunk
leaf01(config-if)# switchport trunk allowed vlan 10,20,25,30,50,60,70
↑ Only allow the production VLANs on the peer link trunk.
  VLAN 40 (OOB) is NOT trunked here — OOB is only on the Catalyst 9200 OOB switch.
  No need to pass OOB traffic through the leaf peer link.
leaf01(config-if)# spanning-tree port type network
↑ "network" port type = this port connects to another switch (not an end device).
  Enables Bridge Assurance — prevents a unidirectional link from silently dropping traffic.
  Required on all peer-link ports in a vPC domain.
leaf01(config-if)# vpc peer-link
↑ Designate this port channel as the vPC peer link.
  NX-OS now knows this is the synchronisation path between the two leaf switches.
  Only one port channel per vPC domain can be the peer-link.
leaf01(config-if)# no shutdown
leaf01(config-if)# exit

leaf01(config-if)# channel-group 1 mode active

SVIs — the Layer 3 gateway for every VLAN
One SVI per VLAN — the IP address that every device in each VLAN uses as its default gateway
What an SVI is and what it does
SVI stands for Switched Virtual Interface. It is a virtual Layer 3 interface associated with a specific VLAN. When you create SVI VLAN 10 on leaf01 and assign it the IP 10.10.0.1/24, any device in VLAN 10 that needs to route traffic (to a different VLAN or to a different leaf's subnet) sends the packet to 10.10.0.1. leaf01 receives it on the SVI, looks up the destination in its routing table, and forwards it — either to another SVI (inter-VLAN routing) or to the spine via BGP (inter-leaf routing). The SVI is effectively a router interface that also participates in Layer 2 switching within its VLAN.

DHCP relay — forwarding requests to the Windows DC
Why DHCP relay is needed in a routed network — and the exact configuration on each SVI
Why DHCP relay is necessary here
DHCP works by broadcast — a client broadcasts a DHCPDISCOVER packet to 255.255.255.255. Routers do not forward broadcasts — they drop them. Without DHCP relay, a VM on VLAN 60 sends a DHCP broadcast, the leaf SVI blocks it, and the VM never gets an IP address. DHCP relay converts the broadcast into a unicast packet directed to the Windows DC's DHCP server IP (10.70.0.10). The leaf acts as a proxy — it receives the broadcast and forwards it as unicast to the DC. The DC responds with a unicast DHCPOFFER that the relay forwards back to the client.

Jumbo frames — MTU 9000 for vSAN and NFS
Why vSAN and NFS need jumbo frames — and exactly which interfaces to configure and in what order
MTU must be consistent end-to-end — partial jumbo is worse than no jumbo
If the leaf switch supports 9000-byte frames but the ESXi VMkernel adapter is still at 1500, packets are fragmented at the ESXi side — causing performance degradation. If the leaf is at 1500 but ESXi VMkernel is at 9000, packets that exceed 1500 bytes are silently dropped at the switch — causing mysterious vSAN or NFS failures with no obvious error. MTU must match on EVERY hop: ESXi VMkernel adapter → leaf switch port → leaf SVI → spine interface → remote leaf SVI → remote ESXi VMkernel adapter. Configure all of them.
