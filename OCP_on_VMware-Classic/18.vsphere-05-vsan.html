vSAN — hyperconverged storage built into the cluster you already have
Traditional storage architectures require a separate Storage Area Network (SAN) — physical Fibre Channel switches, HBAs in each server, and a dedicated storage array (like the Dell EMC PowerStore). This adds complexity, cost, and another failure domain. vSAN uses the NVMe SSDs already inside the HP ProLiant DL380 Gen11 servers and pools them across all five hosts using the 25G vSAN VMkernel network (vmk2 on VLAN 25, configured in vSphere-04). The result is a distributed, redundant, shared storage pool that all five hosts access — without any additional hardware.
vSAN vs the Dell EMC PowerStore — why both exist in NovaCrestBank
NovaCrestBank has a Dell EMC PowerStore 1000T. Why use vSAN at all instead of putting everything on the PowerStore?
Because vSAN and the PowerStore serve different workload profiles — and having both gives NovaCrestBank the optimal storage tier for each workload type.

vSAN: Hyperconverged, all-NVMe SSD storage. Extremely low latency (sub-0.5ms for reads, sub-1ms for writes). Excellent for random I/O-heavy workloads like databases, OLTP systems, and containerised banking applications on OpenShift. The NVMe SSDs in the DL380 Gen11 are faster than any iSCSI-connected SAN for latency-sensitive workloads. vSAN also provides HA-integrated storage — vSphere HA and DRS work natively with vSAN without needing to configure storage HA separately.

Dell EMC PowerStore (via FC/SAN): Enterprise SAN with dual-port FC HBAs. Better for large sequential I/O workloads (backup staging, large file operations), workloads that need FC-specific features, or VMs that require Fibre Channel connectivity for regulatory or application reasons.

NetApp AFF A400 (via NFS): Best-in-class NAS. Ideal for shared file services, large capacity workloads, and protocols that benefit from NAS features (snapshots, clones, CIFS/NFS sharing).

vSAN is the primary datastore for most VMs. PowerStore and NetApp serve specific workloads. This is the correct enterprise storage strategy for a financial institution.

How vSAN stores data — objects, components, and RAID-like protection
The internal architecture that makes vSAN both fast and redundant — understanding this prevents surprises when interpreting health alerts and capacity numbers
Objects and components — vSAN's storage unit model
What is a vSAN "object" and how is it different from a traditional LUN or VMDK?
A vSAN object is a logical storage entity (like a VMDK file, a VM home namespace, or a swap file) that is stored as multiple components distributed across multiple hosts' disk groups.

When a VM's VMDK is stored on vSAN:
— The VMDK is an object
— vSAN splits the object into components
— Each component is stored on a different host's NVMe disk
— Additional components provide redundancy (mirrored copies or RAID parity)
— There is also a witness component — a small metadata file stored on a third host that acts as a tiebreaker if two copies disagree

Example with FTT=1 (tolerate 1 failure) and RAID-1 (mirroring):
— VMDK component 1: stored on esxi01's NVMe
— VMDK component 2 (mirror): stored on esxi02's NVMe
— Witness: stored on esxi03's NVMe (tiny — just metadata)
— If esxi01 fails: esxi02 still has the full VMDK component, VM continues running
— If esxi01's NVMe fails: same result — mirror on esxi02 is unaffected
FTT — Failures to Tolerate — the most important vSAN concept
What does "Failures to Tolerate" (FTT) mean in vSAN storage policies?
FTT is the number of simultaneous host (or disk) failures a VM's storage can survive without losing data or becoming inaccessible.

FTT=1: Can tolerate 1 failure. Requires minimum 3 hosts (2 data copies + 1 witness). NovaCrestBank has 5 hosts — FTT=1 works well.
FTT=2: Can tolerate 2 simultaneous failures. Requires minimum 5 hosts (3 data copies + 2 witnesses). NovaCrestBank has exactly 5 hosts — FTT=2 is the maximum possible with RAID-1 on a 5-host cluster.
FTT=3: Requires 7 hosts minimum. Not possible for NovaCrestBank with 5 hosts.

For banking-critical VMs (DCs, CA servers, database tier): FTT=2 with RAID-1 — maximum redundancy.
For standard VMs (development, test, non-critical services): FTT=1 with RAID-5 — efficient use of capacity.

The minimum host count formula: FTT×2 + 1 for RAID-1. For FTT=2: 2×2+1 = 5 hosts minimum. NovaCrestBank is right at the minimum for FTT=2.

RAID-1 vs RAID-5 in vSAN — which to use when

Disk groups — the cache tier and capacity tier
How vSAN organises the physical disks inside each ESXi host — the two-tier model that delivers SSD-class performance
What a disk group is and why there are two tiers
vSAN separates disks into "cache tier" and "capacity tier." What is the difference and why does it matter?
A disk group is a set of physical disks on one ESXi host that vSAN manages together. Each disk group has one cache tier device and one or more capacity tier devices.

Cache tier (one per disk group): The fastest available device in the host. For the HP DL380 Gen11, this is the highest-endurance NVMe SSD. vSAN uses the cache tier as a read cache and write buffer. Write I/O from VMs is committed to the cache tier first (extremely fast — NVMe latency under 0.1ms) and then destaged to the capacity tier. Read I/O is served from the cache tier if the data is cached (a cache hit avoids hitting the slower capacity devices).

Capacity tier (one or more per disk group): The bulk storage devices. These hold the actual vSAN objects. They can be NVMe SSDs or SATA/SAS SSDs. For NovaCrestBank's HP DL380 Gen11, the capacity tier is also NVMe SSD — making this an All-Flash vSAN configuration.

All-Flash vs Hybrid: In a Hybrid vSAN, the cache tier is an SSD and the capacity tier is spinning HDD. The cache tier dramatically improves performance by absorbing the worst of the HDD latency. In an All-Flash vSAN (NovaCrestBank's design), all disks are SSDs — the cache tier still provides write buffering and read caching, but the performance difference between tiers is smaller. All-Flash vSAN supports deduplication and compression, which Hybrid does not.

Why the M.2 RAID-1 boot volume is NOT claimed by vSAN
The M.2 RAID-1 volume contains the ESXi OS and local datastores. Can vSAN claim it?
No — and attempting to claim the boot volume would destroy the ESXi installation. vSAN cannot and must not claim disks that are already in use by the operating system.

The M.2 RAID-1 volume (480 GB) is formatted as VMFS and contains: the ESXi boot partition, the VMFS datastore (datastore1), and any VMs currently stored there (including vCenter, before it is migrated to vSAN in Section 15). vSAN only claims disks that are empty and unformatted. The wizard will show only unclaimed disks — the M.2 volume will not appear as available for vSAN claiming because it is already in use.
One disk group per host — not multiple
vSAN supports multiple disk groups per host. Should NovaCrestBank configure one or multiple?
One disk group per host — NovaCrestBank has one cache-tier NVMe per host, which determines the maximum number of disk groups (one cache device per disk group).

Multiple disk groups would require multiple cache-tier devices per host. The DL380 Gen11 has one high-endurance NVMe designated for the cache role. Using multiple disk groups would require designating additional capacity NVMes as cache devices — reducing overall capacity inefficiently. One disk group with the maximum capacity tier devices is the correct configuration for this hardware.

Storage policies — the mechanism that defines how each VM's data is protected
vSAN storage policies are what make vSAN's protection model explicit and per-VM configurable — this is a fundamental concept before the wizard is opened
What a vSAN storage policy is and why it replaces RAID volume selection
In traditional storage, you select a RAID level for an entire LUN. How does vSAN storage policy differ?
vSAN storage policies are applied per VM (or per VMDK) — not per LUN or per datastore. Every VM or disk object on vSAN gets an explicit policy that defines exactly how its data is protected, striped, and replicated.

Traditional approach: you create a RAID-5 LUN and every VM on that LUN gets RAID-5 protection whether it needs it or not. If some VMs need higher protection, you need a separate LUN with a different RAID level — multiplying management complexity.

vSAN approach: you create one datastore. Each VM is assigned a storage policy. A banking-critical VM gets FTT=2 RAID-1 (survives 2 host failures). A development VM gets FTT=1 RAID-5 (efficient, sufficient for non-critical workloads). The physical storage is the same pool — the policy controls how that pool protects each object.

This is the most powerful concept in vSAN: one shared storage pool with per-object protection policies. The DBA sets a policy on the database VM that says "this must survive 2 simultaneous failures" — and vSAN enforces that guarantee automatically, placing components on appropriate hosts.

The two policies NovaCrestBank will create

NovaCrestBank-Banking-Critical

  • FTT = 2 (survives 2 simultaneous failures)
  • RAID method: RAID-1 (mirroring)
  • Stripe width = 1
  • Applied to: DC VMs, CA VMs, database tier, vCenter
  • Capacity overhead: 3× (3 full copies)

NovaCrestBank-Standard

  • FTT = 1 (survives 1 failure)
  • RAID method: RAID-5 (erasure coding)
  • Stripe width = 4
  • Applied to: general VMs, dev/test workloads
  • Capacity overhead: 1.33× (33% overhead)

vSAN recommended slack space — why 30% must remain free
If the raw capacity is 76.8 TB, can the cluster use all 76.8 TB?
No. VMware recommends keeping at least 30% of raw vSAN capacity free at all times — and operational reality requires even more headroom.

Why 30% minimum free:
1. Resync operations: when a host fails and rejoins, vSAN resyncs all its objects to restore compliance. Resync writes temporary copies of data — if there is no free space, resync fails and objects remain non-compliant.
2. Component replacement: when a disk fails, vSAN moves its data to other disks. Free space is needed as a buffer.
3. Deduplication staging: dedup operates on a disk group level and needs staging space.
4. Snapshots and rapid VM provisioning: VM snapshots and clones temporarily double the storage requirement.

For NovaCrestBank with 76.8 TB raw, VMware recommends keeping at least 23 TB free (30% of raw). Effective usable budget for VMs: approximately 53 TB raw, or ~18–40 TB depending on policy mix. This is more than sufficient for the initial deployment of ~50 VMs.

Enable deduplication and compression — yes
Deduplication and compression are optional. Why enable them on the initial vSAN deployment?
Because NovaCrestBank's workloads (Windows VMs with similar OS images, containerised applications with shared base layers) have significant deduplication potential, and All-Flash vSAN can run dedup efficiently without the performance penalty that affects hybrid vSAN.

In an All-Flash vSAN, dedup and compression run on the capacity tier NVMe SSDs without significantly impacting write latency (the cache tier absorbs write bursts before they reach the dedup engine). Expected dedup/compression ratios for typical banking VM workloads: 1.5x to 2.5x effective capacity gain. With 76.8 TB raw, dedup/compression can effectively make it feel like 115 TB–192 TB before FTT overhead. This meaningfully extends the useful life of the hardware investment.

Note: enabling dedup/compression cannot be done on a running vSAN cluster without reformatting all disk groups — it must be enabled at initial configuration time. Disabling it later is also disruptive. Enable it now.
Encryption — deferred to KMS integration guide
vSAN at-rest encryption is a banking regulatory requirement (PCI-DSS, GDPR). Why defer it?
Because vSAN encryption requires a configured Key Management Server (KMS) before it can be enabled — and NovaCrestBank's Thales Luna Network HSM 7 has not yet been integrated with vCenter.

vSAN encryption uses the KMS to encrypt and store the vSAN encryption key. Without the KMS: the encryption key has nowhere to be stored — vCenter will reject the encryption activation. The Thales Luna HSM integration with vCenter (KMIP protocol) is a separate configuration step that requires the HSM to be provisioned, partitioned, and the vCenter KMS client certificate exchange completed.

When the KMS integration guide is completed: navigate to Cluster → Configure → vSAN → Services → Encryption → Enable. The existing vSAN datastore is re-encrypted in the background without any VM downtime. Enabling encryption after initial vSAN setup is fully supported and non-disruptive once the KMS is in place.

Tick Deduplication and Compression. Leave Encryption and Stretched Cluster unchecked. Click Next.

What the Claim Disks screen shows

The wizard displays all eligible (unclaimed) disks grouped by host. For each host you will see 5 NVMe devices. vSAN's auto-claim algorithm identifies the cache tier candidate based on the disk's endurance class — the 1.6 TB Samsung PM9A3 has a higher write endurance rating (DWPD) than the 3.84 TB drives, making it the correct cache tier candidate. The wizard colour-codes each assignment: orange/yellow for cache tier, blue for capacity tier.

Verifying the cache tier selection is correct
The wizard auto-selects the cache tier. How do you verify it chose the right disk?
Verify by cross-referencing the disk size and model against the hardware spec sheet for the HP DL380 Gen11 storage configuration.

The cache tier NVMe should be: the smallest NVMe (1.6 TB vs 3.84 TB) AND the highest endurance model (typically labelled "P4610," "PM9A3 Write Intensive," or similar in the model number). If the wizard selects a 3.84 TB drive as the cache tier, click on that assignment and change it to "Capacity Tier," then click the 1.6 TB drive and change it to "Cache Tier." The cache tier disk is designated for random write I/O — using a capacity drive as cache would degrade its endurance faster than intended.

You can verify disk endurance ratings on any host via SSH: esxcli storage core device smart get -d naa.xxx — look for the "Percentage Used Endurance Indicator" field. The cache tier device should have the highest endurance rating among the NVMe drives.

Review the disk assignments carefully. Confirm the 1.6 TB disk is assigned as Cache Tier on every host. Confirm all four 3.84 TB disks are assigned as Capacity Tier. If everything looks correct, click Next.

Fault domains — rack-aware protection
Telling vSAN about the physical topology of the hosts so it can place data components on hosts in different failure zones
What fault domains are and why NovaCrestBank needs them
vSAN works fine without fault domains — why configure them?
Without fault domains, vSAN places components on different hosts — but it does not know that some hosts might share the same failure zone (same rack, same PDU, same ToR switch).

NovaCrestBank's five HP DL380 Gen11 servers are in one server rack. All five connect to the same pair of Cisco Nexus 93180YC-FX leaf switches. If those leaf switches fail simultaneously (vPC peer link failure plus uplink failure — an unlikely but possible scenario), all five hosts lose connectivity at the same time. In that scenario, it does not matter which hosts hold the vSAN components — all are affected equally.

More relevant to NovaCrestBank's immediate setup: all five hosts are on the same pair of PDUs and share the rack's cooling. A single PDU failure taking out PDUs A and B simultaneously would affect all servers in the rack.

With fault domains: you group hosts by their physical failure zone (e.g., two hosts on PDU-A in fault domain 1, two hosts on PDU-B in fault domain 2, one remaining host in fault domain 3). vSAN places components in different fault domains. An FTT=1 policy with fault domains ensures that one complete failure zone can fail without losing data.

For NovaCrestBank with 5 hosts in a single rack: configure 3 fault domains (two hosts in each of the first two, one host in the third). This enables FTT=1 rack-aware protection — a worst-case PDU or switch failure affecting some hosts does not compromise FTT=1 protected data. FTT=2 with fault domains would require additional hosts.

VMware Series · vSphere-05 · Section 11
Default vSAN storage policy — what it is and why it is not sufficient
Understanding what vSAN assigns to VMs by default — and why NovaCrestBank needs custom policies
The default vSAN storage policy — FTT=1 RAID-1
What policy does vSAN assign to VMs that have no explicit policy?
The default vSAN storage policy uses FTT=1 with RAID-1 (mirroring). Every VM placed on vSAN without an explicit policy gets 2 data copies and a witness — tolerating 1 host or disk failure.

For most VMs, FTT=1 RAID-1 is adequate. However, NovaCrestBank has two categories that need different treatment:

Banking-critical VMs (DC1, DC2, CA, database tier): FTT=1 means one failure causes the VM to run non-compliant (with only 1 copy). For a Domain Controller, running non-compliant even briefly is unacceptable — if a second failure occurs during the resync window, data is lost. FTT=2 with RAID-1 provides 3 copies and tolerates 2 simultaneous failures.

General VMs: FTT=1 is fine, but the default uses RAID-1 (200% overhead). FTT=1 with RAID-5 provides the same single-failure protection at only 133% overhead — significantly more capacity-efficient. Creating a Standard policy with RAID-5 frees up roughly 40% more usable capacity for general workloads compared to the default.

The default policy should not be modified — VMs without explicit policies falling back to a safe default is intentional. Instead, create two custom policies (next sections) and assign them to the appropriate VMs during provisioning.

Why stripe width = 4 for the Standard RAID-5 policy
Stripe width (disk stripes per object) defaults to 1. Why set it to 4 for the RAID-5 Standard policy?
Because vSAN RAID-5 (erasure coding) distributes data across 4 components (3 data + 1 parity) on 4 different hosts. Setting stripe width = 4 aligns with this distribution model and ensures the erasure coding algorithm has data spread across the correct number of hosts.

For RAID-5 with FTT=1, vSAN needs data on 4 different hosts (3 data stripes + 1 parity stripe). The stripe width setting of 4 tells vSAN to spread each object across 4 disk groups on 4 different hosts. NovaCrestBank has 5 hosts and 3 fault domains — this is perfectly compatible. Setting stripe width = 1 with RAID-5 would cause vSAN to use fewer components and potentially not spread data across enough hosts for the erasure coding to function optimally.

This is one of the few "under the hood" vSAN settings where matching the stripe width to the RAID level's distribution requirement matters for performance and correctness.

Create this policy the same way as the Banking-Critical policy (Menu → Policies and Profiles → VM Storage Policies → + Create). After saving, NovaCrestBank has two custom policies visible in the VM Storage Policies list alongside the built-in defaults.

    After enabling, the Performance charts page shows graphs for: Backend IOPS, Backend Throughput, Backend Latency, Congestion, and Resync Activity. On a freshly configured cluster with few or no VMs, all graphs should show near-zero values with no congestion. These charts become the baseline for future performance comparison — if latency increases after adding more VMs, these charts show exactly when the increase began and which disk group is affected.

Why enable the Performance Service immediately
Performance monitoring is optional. Why enable it now rather than when a performance problem is suspected?
Because performance baseline data is only useful if collected from the beginning. When a performance problem is eventually reported, you need historical data to determine: was this always slow, or did it change when a specific event occurred?

Without historical data: "vSAN is slow" — no context, no baseline, no way to know if this is normal or a regression. With historical data: "vSAN latency increased from 0.4ms baseline to 3.2ms at 14:37 on Tuesday — coinciding exactly with when the batch processing job started running." The Performance Service stores statistics in the vSAN datastore (a small overhead — typically less than 10 GB) and graphs are available for the past 90 days. The cost is minimal; the diagnostic value is high.

What comes next — vSphere-06 HA and DRS

Next guideWhat it doesvSAN dependency
vSphere-06 — HA and DRS Enable vSphere HA so VMs restart automatically on surviving hosts after a failure. Enable DRS for load balancing. Configure anti-affinity rules to keep DC1 and DC2 on separate hosts. Set DRS to fully automated mode for the cluster. vSAN provides shared storage required for HA. Without vSAN, HA cannot restart VMs on other hosts (no shared disk access). Both HA and DRS require VMs to be on shared storage — vsanDatastore satisfies this requirement.
Thales Luna HSM — KMS integration Configure the Thales Luna Network HSM 7 as the Key Management Server for vCenter. Establish KMIP trust between vCenter and the HSM. Enable vSAN at-rest encryption. vSAN encryption uses the KMS to protect the vSAN encryption key. Once KMS is configured: Cluster → Configure → vSAN → Services → Encryption → Enable. The cluster re-encrypts in the background with no VM downtime.
Dell EMC PowerStore — FC SAN Zone the Fibre Channel fabric. Create storage volumes on the PowerStore. Present FC LUNs to ESXi hosts. Create VMFS datastores on FC LUNs for workloads that require SAN storage. PowerStore datastores complement vSAN for specific workloads. Zoning and LUN presentation are separate from vSAN — they use the FC HBAs in the DL380 Gen11, not the 25G Ethernet NICs used by vSAN.
