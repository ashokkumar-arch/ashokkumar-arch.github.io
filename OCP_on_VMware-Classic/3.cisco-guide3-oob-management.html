Cisco Fabric · Guide 3 · Section 01
Why OOB exists — the 3 AM scenario
Understanding what happens without OOB — so you never underestimate why it is built first
The scenario that makes every engineer wish they had OOB
It is 3 AM. An alert fires. The banking application is down. You SSH to the leaf switch to investigate — connection refused. You SSH to the spine — connection refused. The production network is broken. Without OOB, you have two options: (1) wake up someone who has physical data center access and send them in with a console cable at 3 AM, or (2) drive there yourself. With OOB, you open a laptop, SSH to 10.40.0.20, and fix it from your home office. The OOB network runs on separate physical cables, a separate switch, and a separate VLAN — it is completely independent from the production fabric you just broke.

What "out-of-band" actually means

"In-band" management means you manage a device using the same network that carries production traffic. If production traffic breaks, your management access breaks too. "Out-of-band" management means you manage devices using a completely separate, dedicated network that has nothing to do with production traffic. The two networks share no physical cables, no VLANs, and no routing paths.

In-band management (risky)

You SSH to leaf01's VLAN 10 IP (10.10.0.20). That SSH packet travels through the production network — the same leaf switch, the same spine, the same uplinks that carry vMotion, vSAN, and application traffic. If a misconfigured ACL drops all traffic, your SSH session is also dropped. You are locked out.

Out-of-band management (safe)

You SSH to leaf01's mgmt0 IP (10.40.0.20). That SSH packet travels through the OOB network — a physically separate Catalyst 9200 switch, separate Cat6 cables, and the mgmt0 interface which is completely isolated from production interfaces. A broken production config cannot affect your management access.

The three things OOB protects you from

  • 1
    Your own configuration mistakes

    The most common reason you need OOB. You misconfigure an ACL, a routing policy, or a VLAN trunk. The production network breaks. Without OOB you just locked yourself out. With OOB you SSH in via mgmt0, check show running-config, and fix the mistake. The lesson: always have a way in that is independent of what you are configuring.

  • 2
    Hardware failures

    A line card fails on a spine switch. Several uplinks go down. Traffic reroutes but your management path via the affected uplinks is also gone. Via OOB (mgmt0 on a separate physical cable to the OOB switch) you still have full CLI access to the spine, can diagnose which line card failed, check error counters, and call Cisco TAC with full diagnostic information — while still connected.

  • 3
    Security incidents

    An attacker compromises the production network. They may be able to intercept or disrupt in-band management SSH sessions. OOB management is on a physically separate network that the attacker cannot reach from the production network — there is no route between VLAN 40 and production VLANs except through the Palo Alto NGFW, which enforces strict access control.

How the jump host reaches OOB devices — the path

The jump host (CyberArk PAM server, 10.40.0.50 on VLAN 40 via a dedicated access port, OR reached via VPN) connects to the OOB switch's SVI at 10.40.0.1. The OOB switch then routes the management traffic directly to each device on VLAN 40. The Palo Alto NGFW enforces that only the jump host subnet can reach 10.40.0.0/24 — no other traffic can enter the OOB network from production VLANs.

What HP iLO is and how it works
iLO is the server equivalent of the OOB network — a completely independent management processor built into every HP ProLiant server
iLO — Integrated Lights-Out — HP's out-of-band server management
Every HP ProLiant DL380 Gen11 in NovaCrestBank has a dedicated hardware module called iLO (Integrated Lights-Out) built directly into the motherboard. iLO has its own CPU, its own memory, its own network port, and its own firmware. It runs completely independently of the server's main operating system. Even if ESXi crashes, if the server is powered off, or if all production network cables are disconnected — iLO continues to run as long as the server has power (even standby power).

What iLO lets you do

You see the BIOS, the boot process, ESXi installer, and any crash screen. Installing ESXi for the first time. Diagnosing a server that will not boot. Reading a PSOD (Purple Screen of Death) on a crashed ESXi host.

iLO vs the server's production NICs — they are completely separate

Look at the back of a ProLiant DL380 Gen11. You will see multiple network ports. The iLO port is clearly labelled "iLO" or "MGMT" — it is physically separate from the production NICs (vmnic0 through vmnic5). The iLO port connects to the OOB switch. The production NICs connect to the leaf switch. They share no physical connection, no software stack, and no driver — they are completely independent hardware.

The result — what this independence gives you
You can be in the middle of re-installing ESXi on a server (which means ESXi is completely gone and the server is booting from an ISO) while simultaneously using the production OCP cluster that ran on that server. The other ESXi hosts pick up the VMs via vMotion. You use iLO to watch the ESXi installer on the screen of the server being reinstalled. Zero disruption to the cluster. Zero physical data center visits.

iLO initial configuration — using the physical server console
Before iLO has a network IP, you must configure it through the server's BIOS/UEFI setup — what you see on screen when you press F9 at boot
You only do this once — and you need physical access to the server
The very first iLO configuration must be done either via the server's BIOS/UEFI (pressing F9 during POST) or via a physical console (monitor + keyboard directly connected to the server's VGA/USB ports). After the iLO IP address is set, all further iLO configuration is done remotely via its web interface — no physical access needed again.

When a server behaves unexpectedly, check this log before anything else. It often reveals the hardware root cause (failing fan, high temperature, PSU switching to redundancy) before it escalates to an outage.

Management VRF — deep explanation
Understanding why NX-OS puts mgmt0 in its own VRF — the concept that confuses most CCNA candidates
VRF — the concept that changes how you think about routing on Nexus
A VRF (Virtual Routing and Forwarding) instance is a completely separate routing table inside a single switch. The switch can have multiple VRFs — each has its own routing table, its own routes, and its own interfaces. Traffic in one VRF cannot cross into another VRF without explicit configuration. Think of each VRF as a completely separate router running inside the same physical hardware.

Why mgmt0 needs its own VRF

Nexus switches have a default VRF — this is the routing table for production traffic (BGP routes, SVI gateways, etc.). When you configure BGP for the spine-leaf fabric in Guide 4, all those routes will live in the default VRF.

The mgmt0 interface is in the management VRF — a completely separate routing table. This separation means:

  • No route leaking — management traffic cannot accidentally be routed via a production path (which might be congested, broken, or firewalled differently)
  • Security isolation — the management VRF only has one route: the default gateway via the OOB switch. There is no way for a production network attack to reach the management VRF through routing
  • Stability — even if the entire production BGP routing table is corrupted or cleared, the management VRF (and your SSH access) is completely unaffected

What this means for your commands

spine01 — VRF awareness in NX-OS commands 
# When you ping or traceroute FROM the switch, you must specify the VRF
# Otherwise NX-OS uses the default VRF (production routing table)

# WRONG — will fail (OOB switch is not in the production routing table)
spine01# ping 10.40.0.1
No route to host   ← fails because 10.40.0.0/24 is not in the DEFAULT VRF

The Palo Alto management interface — physically separate from traffic interfaces
The Palo Alto PA-5250 has a dedicated management port labelled MGT on the rear panel. This port is completely separate from the traffic interfaces (ethernet1/1 through ethernet1/8 and beyond) that carry firewall zones. The MGT port connects only to the OOB switch — it is never used for production traffic. This means even if every traffic interface is down, you can still reach PAN-OS (Palo Alto's operating system) management via the MGT port.

What needs to be connected and why

Physical connectionCat6 cable from PA-5250 MGT port → oob-sw01 Gi1/0/20
OOB switch port configAccess port, VLAN 40 (already configured in Guide 2)
Management IP10.40.0.50/24 (assigned to PA-5250 MGT interface)
Default gateway10.40.0.1 (OOB switch)
Used forPAN-OS web UI (HTTPS on port 443), SSH CLI, Panorama management, API calls
Cannot be used forProduction traffic (untrust/DMZ/trust zone traffic uses the traffic interfaces only)

Why this matters for OCP and VMware

When you manage the Palo Alto to add firewall rules for OCP (as done in the OpenShift guides), every rule change is made through the MGT interface at 10.40.0.50. If the production network has a problem, you can still reach the Palo Alto management interface via OOB and add or remove rules without being affected by the production outage. This is the exact scenario that happened in OpenShift Guide 7 (troubleshooting) — adding Palo Alto rules to fix blocked OVN traffic while the production network had issues.
