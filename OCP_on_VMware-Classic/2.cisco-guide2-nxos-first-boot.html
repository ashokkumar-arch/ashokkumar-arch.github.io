Cisco Fabric · Guide 2 · Section 01
What NX-OS is — and how it differs from IOS
Before your first command, understand the operating system you are working with
NX-OS vs IOS — important differences for a CCNA candidate
Your CCNA knowledge is based on Cisco IOS — the OS on routers and Catalyst switches. Nexus switches run NX-OS, which is a different OS sharing some IOS conventions but with important differences. Knowing these differences prevents confusion when commands behave differently than expected.

Key differences between IOS and NX-OS

ConceptCisco IOS (CCNA knowledge)NX-OS (what Nexus uses)
Features / services Most features are always available — just configure them Most features are disabled by default to save resources. You must explicitly enable each feature with feature bgp, feature lacp, etc. before you can configure it.
Interface numbering GigabitEthernet0/1 or Gi0/1 Ethernet1/1 or Eth1/1. The format is module/port. On a fixed switch like the 93180YC-FX, module is always 1. On a modular chassis like the 9504, module is the line card slot number.
Startup config copy running-config startup-config or wr Same command works. NX-OS also accepts copy run start. Critical difference: NX-OS does NOT auto-save. If you configure the switch and it reboots before you save, all your work is lost. Save after every significant change.
VDC — Virtual Device Context Not present in IOS Nexus 9504 modular chassis supports VDCs — virtual partitions of the switch that act as separate devices. NovaCrestBank uses the default VDC (VDC 1). You do not need to configure additional VDCs.
Layer 2 vs Layer 3 ports Interfaces default to Layer 3 (routed) on routers, Layer 2 (switchport) on switches Nexus interfaces default to Layer 3 (routed, no switchport). To make a port a Layer 2 switchport, you must type switchport explicitly. This is the opposite of IOS switches.
Interface ranges interface range Gi0/1-24 interface Ethernet1/1-24 (same concept, slightly different syntax)
Privileged mode enable → # prompt NX-OS uses role-based access. The admin account logs straight into a full-privilege shell. The prompt is switch# or switch(config)#. No separate enable password needed for the admin account.
Spanning Tree STP enabled by default Nexus supports STP but for spine-to-leaf uplinks (which are routed/Layer 3) STP is irrelevant. STP runs only on Layer 2 (switchport) interfaces — leaf switch downlinks to servers.

NX-OS command mode hierarchy — where you are at each prompt

NX-OS prompt hierarchy explained 
spine01#                     ← Privileged EXEC mode. Read-only commands. show, ping, copy.
                                 You start here after logging in.

spine01# configure terminal
spine01(config)#             ← Global Configuration mode. Configure anything system-wide.
                                 Enter by typing: configure terminal (or conf t)
                                 Exit by typing: end (goes back to #) or exit (goes up one level)

spine01(config)# interface Ethernet1/1
spine01(config-if)#          ← Interface Configuration mode. Configure a specific interface.
                                 Exit by typing: exit (back to config) or end (back to #)

spine01(config)# router bgp 65001
spine01(config-router)#      ← Router/Protocol Configuration mode. Configure BGP, OSPF, etc.

spine01(config)# vlan 10
spine01(config-vlan)#        ← VLAN Configuration mode. Name VLANs, set state.

# Most useful navigation shortcuts:
end             ← From ANYWHERE back to privileged EXEC (#)
exit            ← One level up (config-if → config → #)
Ctrl+Z          ← Same as 'end' — emergency exit to # prompt
do show ...     ← Run a show command while inside (config)# without exiting
?               ← Context-sensitive help — shows all commands available at current prompt
Tab             ← Auto-complete (same as IOS)

Configuration order & strategy
Which device to configure first, second, and third — and the logic behind the order
The golden rule of initial configuration order
Configure the OOB management switch first. Once you have OOB working, every other device is accessible via SSH over the OOB network — even if you make a mistake on the production interfaces. Without OOB, a configuration error that breaks connectivity means driving to the data center with a console cable.

Why NTP must be verified before anything else
BGP, TLS certificates, logging timestamps, and audit records all depend on accurate time. A switch that is 5 minutes off in time will have logging that does not correlate with other devices. A certificate that is checked by a switch with wrong time may appear expired when it is not, or valid when it is. NTP must be synchronised and verified on every device before Guide 3 begins.
