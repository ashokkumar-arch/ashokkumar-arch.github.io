The honest truth about a fresh OCP install
Right now, your cluster has: one shared emergency password (kubeadmin) with no audit trail, a self-signed TLS certificate on the API server that every tool complains about, zero AD users able to log in, the ingress router competing with app pods on worker nodes, and unencrypted etcd data at rest. In a banking environment, none of this is acceptable. This guide closes every gap — step by step, terminal output by terminal output.

The order matters — do not skip ahead

Each task depends on the previous one. You cannot disable kubeadmin before AD users can log in. You cannot move the ingress router to infra nodes before the infra nodes exist. Follow this exact order.

Task 1 · Immediate
Active Directory identity provider
Connecting OpenShift to NovaCrestBank's Active Directory so real users can log in
What this actually does
OpenShift does not integrate directly with AD via Kerberos or NTLM. It uses LDAP — which Active Directory speaks natively. You configure OpenShift to authenticate users by querying your AD LDAP interface. When a user logs in at the web console, OpenShift sends their credentials to AD LDAP and AD confirms whether they are valid. The user's groups from AD are also imported, allowing you to set cluster permissions based on AD group membership.

Step 1 — Create a bind account in AD

OpenShift needs a read-only service account to query AD. Create this in Active Directory Users and Computers (or via PowerShell).

Domain controller or admin workstation — PowerShell 
# Create the LDAP bind service account in AD
PS C:\> New-ADUser `
  -Name "svc-ocp-ldap" `
  -SamAccountName "svc-ocp-ldap" `
  -UserPrincipalName "svc-ocp-ldap@novacrestbank.local" `
  -Path "OU=ServiceAccounts,DC=novacrestbank,DC=local" `
  -AccountPassword (ConvertTo-SecureString "Ldap$Bind2026!" -AsPlainText -Force) `
  -PasswordNeverExpires $true `
  -CannotChangePassword $true `
  -Enabled $true

   # Account created successfully

# Verify it exists
PS C:\> Get-ADUser svc-ocp-ldap | Select Name, SamAccountName, Enabled
Name         SamAccountName  Enabled
----         --------------  -------
svc-ocp-ldap svc-ocp-ldap    True

# Find your AD LDAP base DN — you will need this
PS C:\> (Get-ADDomain).DistinguishedName
DC=novacrestbank,DC=local    ← this is your base DN

# Find your AD LDAP server IP/hostname
PS C:\> (Get-ADDomainController).HostName
dc01.novacrestbank.local    ← your LDAP server

Step 2 — Export the AD CS cert for LDAPS trust

OpenShift will connect to AD via LDAPS (LDAP over TLS — port 636) for security. You need the AD CS Issuing CA certificate so OpenShift trusts the LDAP server's certificate.

Step 3 — Configure the LDAP identity provider

jumphost.novacrestbank.local — bash 
# Create the OAuth configuration YAML
[svc-ocp@jumphost ~]$ cat > /tmp/ldap-oauth.yaml << 'EOF'
apiVersion: config.openshift.io/v1
kind: OAuth
metadata:
  name: cluster
spec:
  identityProviders:
  - name: NovaCrestBank-AD              # display name on login page
    mappingMethod: claim
    type: LDAP
    ldap:
      attributes:
        id:
          - sAMAccountName              # AD login name (unique)
        email:
          - mail                        # AD email attribute
        name:
          - displayName                 # full name for display
        preferredUsername:
          - sAMAccountName
      bindDN: "CN=svc-ocp-ldap,OU=ServiceAccounts,DC=novacrestbank,DC=local"
      bindPassword:
        name: ldap-bind-secret          # references the secret you created
      ca:
        name: ldap-ca                   # references the CA configmap
      insecure: false                   # false = use LDAPS (TLS) — mandatory
      url: "ldaps://dc01.novacrestbank.local:636/OU=Users,DC=novacrestbank,DC=local?sAMAccountName?sub?(objectClass=person)"
EOF

Task 2 · Immediate
Replace the API server certificate
Swapping the installer's self-signed API cert for one signed by your AD CS Issuing CA
Why this cannot wait
Every oc command, every CI/CD pipeline, every Kubernetes client connecting to api.ocp.novacrestbank.local:6443 currently gets an x509 warning or error because the certificate is self-signed and unknown. In a banking environment with security monitoring, these TLS errors generate alerts. Replace this on day one.

Step 1 — Generate the API server certificate from AD CS

You need to request a certificate from your AD CS Issuing CA for the hostname api.ocp.novacrestbank.local. This is done on a Windows machine with access to the Certificate Authority.

Task 5 · Day 1
Infrastructure nodes — setup and migration
Creating 3 dedicated infra nodes and moving the ingress router and monitoring there
Recap: what infra nodes are
Infrastructure nodes run OpenShift's own platform components — the ingress router, the internal image registry, and the monitoring stack. They are worker nodes with a special label and taint that says "only platform components, not application pods, go here." This separates platform traffic from application traffic and protects your licensing cost.

Step 1 — Label and taint the infra nodes

In your environment, infra nodes will be worker01, worker02, and worker03 — you will rename them conceptually (the hostname stays the same, only the role label changes). In production you would typically add 3 new VMs as dedicated infra nodes, but for a starting cluster, relabelling existing workers is acceptable.

jumphost.novacrestbank.local — label and taint infra nodes 
# Add the infra role label to the designated infra nodes
[svc-ocp@jumphost ~]$ for node in infra01 infra02 infra03; do
  oc label node ${node}.ocp.novacrestbank.local \
    node-role.kubernetes.io/infra='' \
    --overwrite
  echo "Labelled: $node"

Task 6 · Day 1
etcd encryption at rest
Encrypting all secrets, ConfigMaps, and OAuth tokens stored in etcd — mandatory for PCI-DSS
PCI-DSS requirement for NovaCrestBank
PCI-DSS Requirement 3.5 mandates protection of stored cardholder data and cryptographic keys. OpenShift stores Kubernetes Secrets (which can include API tokens, database passwords, TLS private keys) in etcd. Without encryption, anyone with access to the etcd data files can read all secrets in plain text. Enable this before any application workloads are deployed.

[svc-ocp@jumphost ~]$ oc patch apiserver cluster \
  --type=merge \
  -p '{"spec":{"encryption":{"type":"aescbc"}}}'

Task 8 · Week 1
Move monitoring stack to infra nodes
Prometheus, Alertmanager, and Thanos isolated to infra nodes with persistent storage
Why monitoring needs dedicated nodes
In production, Prometheus can consume 50–200 GB of RAM and significant CPU. If it runs on your worker nodes, it will compete with your banking application pods for resources — and in resource pressure situations, Prometheus can cause pods to be evicted. Move it to infra nodes where it has dedicated capacity.

Task 9 · Week 1
Configure the internal image registry
Giving the registry persistent NFS storage from the NetApp AFF A400
Why the registry is Removed by default
On vSphere IPI installs, OCP sets the internal registry to Removed state because it does not know what storage to use yet. You need to configure a PVC backed by NetApp NFS (or vSAN) and then set the registry to Managed state. Until you do this, builds that push images to the internal registry will fail.
jumphost.novacrestbank.local — configure image registry with persistent storage 
# Check current registry state
[svc-ocp@jumphost ~]$ oc get configs.imageregistry.operator.openshift.io cluster \
  -o jsonpath='{.spec.managementState}'
Removed    ← registry is not running. Fix this.

# Create a PVC for the registry (using thin-csi = vSAN)
# Note: for production, you would use ReadWriteMany from NetApp NFS
# For a starting cluster, ReadWriteOnce from vSAN is acceptable
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: registry-storage
  namespace: openshift-image-registry
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: thin-csi
  resources:
    requests:
      storage: 200Gi
EOF
persistentvolumeclaim/registry-storage created

# Configure the registry to use the PVC and run on infra nodes
[svc-ocp@jumphost ~]$ oc patch configs.imageregistry.operator.openshift.io cluster \
  --type=merge \
  -p '{
    "spec": {
      "managementState": "Managed",
      "replicas": 1,
      "storage": {
        "pvc": {
          "claim": "registry-storage"
        }
      },
      "nodeSelector": {
        "node-role.kubernetes.io/infra": ""
      },
      "tolerations": [
        {
          "effect": "NoSchedule",
          "key": "node-role.kubernetes.io/infra",
          "value": "reserved"
        }
      ]
    }
  }'
config.imageregistry.operator.openshift.io/cluster patched
