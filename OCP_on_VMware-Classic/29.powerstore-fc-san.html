Why FC SAN alongside vSAN — two storage architectures for different workloads
NovaCrestBank already has vSAN with 76.8 TB. Why add a separate Fibre Channel SAN?
Because vSAN and FC SAN have different performance profiles and vSAN is hyperconverged storage that runs on the same network infrastructure as VM traffic. FC SAN provides dedicated, isolated storage bandwidth with deterministic latency for workloads where consistent sub-millisecond response is non-negotiable.

vSAN is optimal for: Most VM workloads, random I/O-heavy databases, OpenShift persistent volumes, any workload that benefits from policy-based per-object protection (FTT=2 RAID-1). vSAN's strength is its integration with vSphere — storage policies, HA, and DRS all work natively.

FC SAN (PowerStore) is optimal for: Large sequential I/O workloads (backup targets, log archiving), VMs that have regulatory requirements for physical storage isolation, workloads that require FC-specific features (synchronous replication to a DR site, PowerStore Metro Volume), and VMs that were historically on FC and whose performance baseline was established on FC.

The practical NovaCrestBank split: Day-to-day banking applications, all OCP workloads, and most VMs run on vSAN. The PowerStore hosts backup staging volumes, DR-replicated critical database LUNs, and any application where the banking architecture team has specified FC connectivity as a requirement.

PowerStore FC SAN · Section 02
FC fundamentals — WWNs, fabrics, zoning, and why each matters
Fibre Channel is a dedicated storage network with its own addressing, topology, and security model — understanding these prevents the most common FC configuration mistakes
What a WWN is and why it is the foundation of FC addressing
Ethernet uses MAC addresses for addressing. What does Fibre Channel use and how is it different?
Fibre Channel uses World Wide Names (WWNs) — 64-bit globally unique identifiers assigned by the hardware manufacturer to every FC port on every device. Unlike MAC addresses, WWNs do not change when hardware is replaced (they are hardware-burned), and they come in two types: World Wide Port Names (WWPNs) identifying individual ports, and World Wide Node Names (WWNNs) identifying devices.

For SAN configuration, WWPNs are what matter — each HBA port in each ESXi server has its own unique WWPN. The PowerStore's storage processors also have WWPNs on each FC port. Zoning (the access control layer) uses WWPNs to define which initiators (host HBA ports) can see which targets (storage array ports).

WWPNs are written as 8 bytes in colon-separated hex: 21:00:00:24:ff:8a:1b:2c. The first byte's first nibble indicates the port type (2x = N_Port — host or storage). Every WWPN in the world is unique — there are no IP address collision risks.
What FC zoning does — the access control layer
All FC devices in a fabric can technically see each other. What does zoning add?
Zoning restricts which FC devices can communicate with each other — it is the FC equivalent of network segmentation. Without zoning, every HBA on every server can access every storage port on every array in the fabric — a misconfiguration on one server could corrupt another server's storage.

NovaCrestBank uses single-initiator single-target zoning — each zone contains exactly one host HBA port and one storage port. This is the most restrictive and safest configuration:

Zone NCB-esxi01-portA-PSA-portA contains: [esxi01 HBA Port A WWPN] + [PowerStore SP-A Port A WWPN]
Zone NCB-esxi01-portB-PSB-portA contains: [esxi01 HBA Port B WWPN] + [PowerStore SP-B Port A WWPN]

With this design: esxi01 can only access PowerStore SP-A via Fabric A, and PowerStore SP-B via Fabric B. It cannot accidentally reach esxi02's storage paths or any other device not in its zones. If esxi02's FC driver malfunctions and starts sending data to unintended addresses, zoning prevents it from reaching any of esxi01's LUNs.

PowerStore FC SAN · Section 05
FC zoning — what it does and the single-initiator single-target best practice
Understanding the zoning strategy before running any commands — the design decision that determines whether the SAN is secure and manageable
Single-initiator single-target vs multi-initiator zoning
A zone could contain one initiator and one target, or many initiators and many targets. What is the difference operationally?
Single-initiator single-target zoning (one host port + one storage port per zone) is the most restrictive and safest configuration. Multi-initiator zoning is simpler to manage but creates blast radius risk — a host that behaves badly in a multi-initiator zone can affect all other hosts in that zone.

With single-initiator single-target: each zone has exactly 2 members — one host HBA WWPN and one storage port WWPN. If esxi01 sends a SCSI reset to the storage (which happens during some error recovery paths), only esxi01's own path is affected. Other hosts' paths to the same storage port are in separate zones and are not disrupted.

With multi-initiator zoning (e.g., all 5 host ports + 2 storage ports in one zone): a SCSI bus reset from esxi01 affects all 5 hosts' paths simultaneously. One misbehaving host can disrupt all storage access for the entire cluster — exactly what happened in several high-profile SAN incidents in banking environments before single-initiator zoning became standard practice.

For NovaCrestBank with 5 hosts and 4 storage ports: 10 host ports × 2 fabrics = 20 zones (10 per fabric). This is more zones to manage but each is simple (2 members) and operationally the safest design.

Why 512e sector size — not 4Kn (native 4K)
Modern drives use 4K native (4Kn) sectors internally. Why use 512e (512 emulated)?
Because VMware vSphere and VMFS require 512-byte sector compatibility. 512e presents 4K physical sectors as 512-byte logical sectors — providing full compatibility with VMware's block storage stack while using the storage array's modern 4K-aligned physical layout.

4Kn native volumes are not supported by VMware VMFS 6 as of vSphere 8.x — attempting to format a 4Kn LUN as VMFS will fail with an unsupported sector size error. 512e is the correct choice for all VMware-attached FC LUNs.
PowerStore FC SAN · Section 10
Create a host group and map volumes to all ESXi hosts
Grouping all five ESXi hosts into a host group and presenting all three volumes to the group simultaneously
Why use a host group — not individual host mappings
A LUN could be mapped to each ESXi host individually. Why use a host group?
Because a host group allows a single volume mapping operation to apply to all members simultaneously — and more importantly, ensures consistent LUN IDs (Logical Unit Numbers) across all hosts.

When a LUN is mapped to multiple hosts individually, each mapping gets its own LUN ID by default. esxi01 might see the volume as LUN 1 while esxi02 sees it as LUN 2. VMFS lock management and vSphere cluster features require that all hosts see a shared LUN using the same canonical path structure — different LUN IDs can cause VMFS resignaturing prompts or access failures.

With a host group: all hosts see the same volume with the same LUN ID at the same time. The mapping is consistent. Adding a new ESXi host to the host group immediately gives it access to all volumes with consistent LUN IDs.

PowerStore FC SAN · Section 12
VMFS on FC — what it provides and how it differs from vSAN
VMFS is a cluster-aware file system specifically designed for VMware — understanding what it adds over a raw block device
VMFS vs vSAN — why use VMFS at all when vSAN exists
vSAN provides distributed VM storage. VMFS provides shared VM storage. For VMware workloads, what does VMFS add over just using the raw FC LUN?
VMFS (VMware File System) is a cluster-aware file system that allows multiple ESXi hosts to concurrently mount and use the same LUN without corrupting each other's data. A raw FC LUN without VMFS would allow only one host to safely access it at a time.

VMFS uses on-disk distributed locking to coordinate concurrent access — when esxi01 opens a VMDK file for a running VM, VMFS locks that file. If esxi02 tries to open the same file (e.g., for a vMotion migration), VMFS coordinates the lock transfer. Without VMFS, both hosts could write to the same blocks simultaneously, causing immediate and total data corruption.

VMFS supports:
— Concurrent mounting by all cluster hosts (enabling vMotion and HA)
— VM snapshots (incremental delta VMDK files)
— Thin provisioning at the file level (not just at the LUN level)
— Per-file permissions that vSphere uses to track which VMs own which files

VMFS on FC is the classic "shared SAN datastore" model that existed before vSAN. vSAN replaced it for most new deployments, but FC VMFS remains relevant for specific workloads and for organisations transitioning from FC to HCI gradually.

Multipathing — ALUA and Path Selection Policy
Configuring how ESXi hosts use the two FC fabric paths to the PowerStore — balancing load and handling path failures
ALUA and Path Selection Policy — why the default may need changing
ESXi has multiple Path Selection Policies (PSP). Which does NovaCrestBank use for PowerStore FC LUNs?
NovaCrestBank uses Round Robin (RR) Path Selection Policy for PowerStore FC LUNs. The PowerStore 1000T supports ALUA Active-Optimised paths — both SP-A and SP-B paths are active-optimised for all LUNs, making Round Robin across both fabrics the optimal configuration.

PSP options:
Fixed (default in some cases): Always uses the same preferred path. If the preferred path fails, it switches to an alternative — but never load-balances. All I/O goes through one SP even when both are healthy.

Most Recently Used (MRU): Uses the most recently active path. Similar to Fixed in practice.

Round Robin (RR): Distributes I/O across all active-optimised paths in rotation. For a PowerStore with both SP-A and SP-B presenting ALUA Active-Optimised paths, Round Robin provides genuine load balancing across both storage processors and both fabrics. This doubles the effective bandwidth to each ESXi host and provides faster failover (the next I/O immediately uses the surviving path).

The PowerStore 1000T with its dual-controller Active-Active ALUA configuration is specifically designed for Round Robin path balancing. Dell EMC's best practice documentation recommends RR for ESXi workloads on PowerStore.

# Set Round Robin PSP for all PowerStore (Dell EMC) LUNs on all hosts

    # Get all FC LUNs from Dell EMC (PowerStore identifies as DGC or EMC)
