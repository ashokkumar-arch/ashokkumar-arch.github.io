The purpose of this guide — stated plainly
This guide runs structured tests against every component of NovaCrestBank's network. The tests produce documented, timestamped pass/fail results. When every test passes, the network engineer signs a handoff document and hands the keys to the VMware team. The VMware team should not spend a single minute debugging network issues — those must be resolved before they touch their first ESXi installer ISO. This guide is how you prove that boundary was met.

The three things this guide establishes

  • 1
    Network correctness. Every path that should work, works. Every path that should not work, does not. The routing table contains exactly the prefixes that the design specifies. vPC is healthy. STP has no loops. BGP sessions are all Established. No interface has unexpected errors or drops.
  • 2
    Baseline state. The current state of the network is documented — BGP route counts, interface speeds, NTP sync status. This baseline becomes the reference point for the future: if something changes, the baseline shows what it used to look like before it changed.
  • 3
    Handoff accountability. A signed document records exactly what was verified, by whom, and when. If the VMware team encounters a network issue after handoff, the signed document makes clear: was this issue present at handoff (network engineering fault) or did it appear after VMware was installed (VMware engineering scope)?

Pre-test checklist — before the first command is run
The five things that must be true before verification begins — running tests on a not-ready system produces misleading results

Checklist — confirm all five before proceeding

Pre-test readiness checklist5 items
PRE-01
All Nexus switches and ASR routers have been running continuously for at least 30 minutes since last reboot or configuration change. BGP needs time to converge — testing immediately after a reboot produces false negatives.
Confirm: show version | include uptime on each device shows ≥ 30 minutes uptime.
READY
PRE-02
NTP is synchronised on all devices. Log timestamps must be accurate for the test results to be meaningful as baseline evidence.
Confirm: show ntp status shows "Clock is synchronized" on all 5 Nexus switches and 2 ASRs.
READY
PRE-03
The jump host (10.40.0.x) has SSH connectivity to all devices via OOB. All verification commands in this guide are run from the jump host.
Confirm: jump host can ping 10.40.0.1 through 10.40.0.52 (all OOB devices).
READY
PRE-04
All configurations have been saved (copy running-config startup-config on all Nexus and IOS-XE devices). Running tests on unsaved configs means a reboot would lose everything verified.
Confirm: show running-config matches show startup-config on critical devices.
READY
PRE-05
No active change window or planned maintenance is in progress. Testing during a change window produces unreliable results and could be confused with change-induced behaviour.
Confirm: ITSM system shows no open network changes for today.
READY

Critical notes for VMware configuration

  • vSAN VMkernel (vmk2) must use MTU 9000. The switch is already configured for jumbo frames on VLAN 25. If VMware is configured at 1500, vSAN will degrade silently. Set it to 9000.
  • NFS VMkernel (vmk3) must use MTU 9000. Same reason. The NetApp and leaf switches support 9000. VMware must match.
  • LACP must be enabled on the vDS uplink teaming policy. The switch is configured with LACP active mode on all ESXi-facing port channels. The vDS must also use LACP (not load-based teaming or failover).
  • vCenter IP: 10.10.0.100. Deploy the VCSA to this static IP on VLAN 10. DNS record vcenter.novacrestbank.local already exists in AD DNS pointing to this IP.
  • Default gateway for esxi01–04 is 10.10.0.1 (leaf01 primary). For esxi05 the gateway is 10.10.0.3 (leaf03 — it has no leaf01/02 connection).
  • OCP node IPs use VLAN 50 (10.50.0.0/24). The OpenShift install-config.yaml should specify this subnet. API VIP: 10.50.0.200. Ingress VIP: 10.50.0.201.
  • DHCP is available on VLANs 10, 20, 50, 60, 70 (pool .200–.254). Static IPs are required for all infrastructure components (ESXi hosts, vCenter, DCs). Use DHCP only for general VMs in VLAN 60.

If it was present and not detected, it is a network engineering defect and will be resolved without delay.
