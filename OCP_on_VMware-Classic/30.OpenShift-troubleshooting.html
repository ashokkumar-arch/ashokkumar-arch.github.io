Guide 7 · Foundation · Section 01
Troubleshooting mindset & methodology
How senior engineers think when something breaks — the mental framework that prevents panic and leads to fast resolution
The senior engineer's first rule
When something breaks, the instinct is to immediately start changing things — restart pods, force-delete objects, patch configurations. Resist this. The first 3 minutes must be spent observing and understanding, not acting. A change made without understanding the cause can make the situation worse and destroy the evidence you need to find the real root cause.

The five-step troubleshooting framework

  • 1
    Establish the blast radius — what is actually broken?

    Is it one pod? One namespace? One node? The entire API server? Start from the widest view and narrow down. oc get nodes → oc get co → oc get pods -A | grep -v Running. Know the scope before doing anything else.

  • 2
    Find the error message — the exact text matters

    Every OCP failure leaves a trace. It might be in: pod Events, operator logs, node journald, etcd logs, or the kube-apiserver audit log. The exact error text is the key — copy it verbatim. "x509: certificate has expired" tells you exactly what is wrong. "connection refused" narrows it to networking or a crashed process.

  • 3
    Form a hypothesis — what single thing could cause this?

    Based on the error message, form one hypothesis. Not five. One. "The API server cert is expired." "The etcd member lost its data." "The Palo Alto firewall rule changed." Test that hypothesis with one verification step. If it is correct, fix it. If not, form the next hypothesis.

  • 4
    Fix the cause — not the symptom

    Restarting a pod that keeps crashlooping is treating a symptom. The cause is in the pod logs or the resource it cannot reach. Fix the cause — the pod will restart healthy on its own. Always ask "why is this failing?" before "how do I make it stop failing?"

  • 5
    Verify the fix — and write the incident report

    After fixing, run the same checks you ran in step 1. Confirm the blast radius has closed. Then write a brief incident report: what broke, what caused it, what fixed it, what prevents recurrence. This is your learning record and your PCI-DSS audit evidence.

Node goes NotReady
The most common P1 incident — a master or worker node stops responding to the API server
P1 — Severity 1
master03.ocp.novacrestbank.local is NotReady
Alert: NodeNotReady fired at 02:14 UTC Impact: etcd has 2 of 3 members — still quorate but no redundancy Time limit: Fix before second master fails or etcd loses quorum
Symptom

oc get nodes shows master03 as NotReady. Pods that were running on master03 are being rescheduled to other nodes.

Possible causes

kubelet crashed or stopped. Node ran out of memory and OOM-killed kubelet. ESXi host had an issue. Network connectivity to node lost. Disk full — kubelet cannot write.

Resolution path

SSH to node via iLO/OOB → check kubelet → check resources → check network → restart kubelet or reboot node.

Guide 7 · Scenario B · Section 06
etcd loses quorum — 2 masters down
The most severe control plane incident — the API server becomes read-only when etcd loses quorum
P1 — Severity 1 — Cluster Outage
etcd quorum lost — only 1 of 3 members alive
Impact: API server in read-only mode — no new pods, no config changes Time limit: Existing workloads still run but cannot be managed Recovery path: Restore failed masters OR restore from etcd backup
Understanding etcd quorum — critical knowledge
etcd uses the Raft algorithm. With 3 members, quorum requires 2 members (majority). If only 1 member is alive, there is no quorum. The cluster becomes read-only — existing pods keep running, but you cannot create, modify, or delete anything. The API server returns "etcdserver: request timed out" on writes. You cannot fix this without recovering the failed masters.
jumphost — etcd quorum loss diagnosis and recovery decision 
# Confirm quorum is lost
[svc-ocp@jumphost ~]$ oc get etcd cluster \
  -o jsonpath='{.status.conditions[?(@.type=="EtcdMembersAvailable")].message}'

[svc-ocp@jumphost ~]$ oc get nodes | grep master
master01.ocp.novacrestbank.local  Ready    master

master03.ocp.novacrestbank.local  NotReady master

[svc-ocp@jumphost ~]$ oc create configmap test-write -n default --from-literal=key=value
error: failed to create configmap: etcdserver: request timed out
              ↑ confirmed: API is in read-only mode due to lost quorum

# Decision tree: Can the failed masters be recovered?

# ── Option A: Masters are recoverable (VMs exist, just crashed) ────
# If master02 and master03 VMs are still in vCenter (just crashed):
# 1. Power on the VMs via vCenter or iLO
# 2. Wait for RHCOS to boot and kubelet to restart
# 3. Watch etcd re-form quorum automatically

OVN-Kubernetes — how to think about network problems
The mental model for diagnosing pod networking issues — understanding the four layers where things can go wrong
The OVN-Kubernetes networking stack — four layers
When a pod cannot reach another pod, the problem is in one of four layers. Knowing which layer to check first cuts diagnosis time from hours to minutes.

The four diagnostic layers

LayerWhat it handlesFirst checkIf broken
L1 — Physical ESXi host network, Nexus leaf/spine, VLAN 50 connectivity between nodes ping NODE_IP from another node Node-to-node ping fails → check vSphere port group, Nexus VLAN config, Palo Alto
L2 — OVN tunnel Geneve tunnels between OVN nodes (UDP 6081). Pod packets are encapsulated and sent between nodes via these tunnels. oc exec ovnkube-node -- ovn-nbctl show Geneve blocked → UDP 6081 firewall rule missing in Palo Alto
L3 — NetworkPolicy Kubernetes NetworkPolicy rules — what pods are allowed to talk to what Check if default-deny is applied; check if allow rule exists for the path Missing allow rule → add NetworkPolicy (Guide 5 Section 13)
L4 — DNS / service CoreDNS resolution of service names to ClusterIPs, kube-proxy/OVN load balancing to pod IPs nslookup SERVICE.NAMESPACE.svc.cluster.local from pod DNS fails → CoreDNS pod issue; service not found → missing Service object

The connectivity test ladder — run these in order

The universal OVN connectivity test sequence 
# From inside the source pod (use oc exec or oc debug)
# Run each test — the first one that fails tells you which layer is broken

SOURCE_POD $ ping 10.128.2.15        ← Test L2: ping destination pod IP directly (L2 OVN tunnel)
SOURCE_POD $ curl 10.128.2.15:8080    ← Test L2+L3: reach pod IP on its port (NetworkPolicy)
SOURCE_POD $ nslookup svc-name.namespace.svc.cluster.local  ← Test L4: DNS resolution
SOURCE_POD $ curl svc-name.namespace.svc.cluster.local:8080  ← Test L4: full service path

# Interpretation:
# ping pod IP FAILS     → L2 OVN tunnel broken (check Geneve, check ovnkube pods)
# curl pod IP FAILS     → L3 NetworkPolicy blocking (check NetworkPolicy)
# nslookup FAILS        → L4 DNS issue (check CoreDNS pods)
# curl service FAILS    → L4 Service/Endpoint issue (check Service object, check Endpoints)
