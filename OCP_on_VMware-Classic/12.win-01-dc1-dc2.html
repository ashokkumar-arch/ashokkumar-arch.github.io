Why DC1 and DC2 must exist before almost everything else
Domain Controllers provide four services that every other system in NovaCrestBank depends on: Authentication (Kerberos and NTLM — vCenter AD login, OpenShift LDAP IDP, Windows machine logins), DNS (name resolution — vcenter.novacrestbank.local, esxi01.novacrestbank.local, every service FQDN), NTP (time synchronisation — ESXi hosts, Cisco switches, and Palo Alto all point to the DC for time), and DHCP (IP address assignment for VMs — configured in Win-02). Until DCs exist, none of these services are operational.

DC1 vs DC2 — which one does what

DC1 (dc01.novacrestbank.local)

10.70.0.10 · Primary Domain Controller. Holds all five FSMO roles initially. Primary DNS server. PDC Emulator (primary NTP source). The first DC in the forest — creates the domain. All other DCs replicate FROM DC1 initially.

DC2 (dc02.novacrestbank.local)

10.70.0.11 · Additional Domain Controller. Replicates all AD data from DC1. Secondary DNS server. Provides redundancy — if DC1 fails, DC2 handles all authentication and DNS automatically. In the same AD site as DC1 (one physical location).

Windows Server · Win-01 · Section 02
AD DS concepts — forest, domain, domain controller
The vocabulary and hierarchy of Active Directory — understanding these terms before the wizard asks them
Forest, domain, and domain controller — what each one is
The promotion wizard uses all three terms. What is the difference between a forest, a domain, and a domain controller?
Forest — the outermost boundary of an Active Directory deployment. A forest contains one or more domains and defines the trust boundary. NovaCrestBank has one forest. All DCs in the forest share a common schema (the definition of every object type that can exist in AD). The forest name is typically the name of the root domain.

Domain — a partition within the forest. NovaCrestBank has one domain: novacrestbank.local. The domain stores user accounts, computer accounts, groups, and policies. It defines the Kerberos authentication boundary — all machines in the domain trust each other's authentication.

Domain Controller (DC) — a Windows Server with the AD DS role installed and promoted to serve the domain. The DC stores a writable copy of the domain database (ntds.dit) and runs the authentication services. You can have multiple DCs for the same domain — they replicate automatically. NovaCrestBank has two DCs (DC1 and DC2) both serving the same novacrestbank.local domain.
FSMO roles — what they are and why DC1 holds them all initially
The wizard mentions FSMO roles. What are they and do you need to configure them manually?
FSMO (Flexible Single Master Operation) roles are five special roles within Active Directory that only one DC can hold at a time. They handle operations that require a single authoritative source:

Schema Master — controls changes to the AD schema (what types of objects and attributes exist). NovaCrestBank will not change the schema.
Domain Naming Master — controls adding and removing domains from the forest.
RID Master — issues pools of Relative IDs used to create new AD objects (users, computers, groups).
PDC Emulator — handles password changes, account lockouts, and is the authoritative NTP time source for the domain.
Infrastructure Master — maintains references between objects in different domains.

You do not need to configure FSMO roles manually. When you promote the first DC in a new forest, Windows automatically assigns all five FSMO roles to DC1. DC2 promotion does not take any FSMO roles — DC1 holds all five until you explicitly transfer them. For a two-DC single-domain environment, this is fine — FSMO roles only become a concern if DC1 is taken offline for an extended period.
Why ".local" domain — not ".com" or ".novacrestbank.com"
NovaCrestBank has a public domain (novacrestbank.com). Should the internal AD domain use the same name?
NovaCrestBank uses novacrestbank.local — a non-routable internal namespace — for clear separation from the public DNS name.

Using novacrestbank.com as the internal AD domain creates a DNS split-brain problem: all internal machines resolve novacrestbank.com via the internal AD DNS — but external clients and internet systems also use novacrestbank.com. Without careful split-horizon DNS configuration, internal machines may fail to reach public services (like cloud APIs) because internal DNS gives different answers from public DNS for the same name.

Using novacrestbank.local is clean: this namespace only exists internally, it is never registered in public DNS, and there is no ambiguity between internal and external resolution. The one limitation: .local can conflict with Bonjour/mDNS on some client networks — not a concern in a data center environment.

Microsoft's current guidance (since Windows Server 2012) actually recommends using a subdomain of the public DNS name (e.g., corp.novacrestbank.com) to avoid the .local limitation on modern clients. NovaCrestBank chooses novacrestbank.local for operational simplicity — it is a bank data center with no end-user devices, not a corporate desktop environment where .local limitations matter.

Why DC2 runs on esxi02 — not esxi01
Why does it matter which ESXi host each DC runs on?
Host anti-affinity — the two DCs must be on different physical hosts so a single host failure does not take both DCs offline simultaneously.

If DC1 and DC2 both run on esxi01 and esxi01 has a hardware failure, both Domain Controllers fail. Authentication fails for every machine in the domain — vCenter cannot authenticate AD users, ESXi hosts cannot validate Kerberos tickets, OpenShift cannot verify LDAP credentials. The entire infrastructure loses its identity fabric simultaneously.

By running DC1 on esxi01 and DC2 on esxi02, a failure of either host takes down only one DC. The remaining DC handles all authentication alone — users and services are unaffected. This is the fundamental availability design principle for AD: always separate DCs onto different physical hardware.

In Phase 2 (after vSAN and HA are configured), a DRS anti-affinity rule will be created in vCenter to enforce this separation automatically — if someone tries to vMotion DC2 onto esxi01, DRS will refuse. For now, it is enforced by initial placement.

DSRM password — what it is and why it matters
The Directory Services Restore Mode (DSRM) password is a special local administrator password used to boot a Domain Controller into recovery mode — outside of Active Directory. If AD becomes corrupted and the DC cannot boot normally, DSRM allows you to access the server and restore the AD database. This password is separate from the domain Administrator password. It is set once during DC promotion and then essentially never used — until the one moment you desperately need it. It goes in CyberArk immediately. Do not lose it.

Windows Server · Win-01 · Section 06
VM network — connecting DC1 to VLAN 70
Setting the correct vDS port group on the VM NIC — VLAN 70 is where all Windows infrastructure VMs live
Why PG-VLAN70-Windows — not PG-VLAN10-Management
Domain Controllers serve the entire infrastructure. Why are they not on the management VLAN (VLAN 10) with ESXi and vCenter?
Because Domain Controllers are a workload — not infrastructure management devices.

VLAN 10 (ESXi Management) is for VMware infrastructure management traffic: the ESXi vmk0 VMkernel adapters and the vCenter appliance. These devices are the VMware management plane. They are administered by the VMware/infrastructure team.

VLAN 70 (Windows Infrastructure) is for the Windows-based identity and certificate infrastructure: Domain Controllers, DHCP servers, and CA servers. These are administered by the Windows/identity team. The separation means: a problem on VLAN 70 (such as a DNS storm caused by a misconfigured DC) does not affect VLAN 10 connectivity. The ESXi hosts and vCenter remain manageable even if the Windows infrastructure has an issue.

The communication between VLAN 10 and VLAN 70 that is needed (vCenter querying AD LDAP on port 389, ESXi hosts resolving DNS from DC) is allowed by the Cisco leaf inter-VLAN routing and Palo Alto firewall rules. Separation does not mean no communication — it means controlled, inspectable communication.

Setting the port group in the hardware wizard

In the Customize Hardware wizard screen, find the Network adapter 1 row. There is a dropdown next to the adapter type (VMXNET3) that currently shows "VM Network" or a blank. Click this dropdown. A list of available port groups appears — you will see all 7 port groups from NCB-vDS-01 plus any standard vSwitch port groups. Select PG-VLAN70-Windows. The dropdown shows the selected port group. Ensure the checkbox "Connect at power on" is ticked.

ℹThe DC1 VM's NIC is now configured to connect to VLAN 70 (10.70.0.0/24) when powered on. The static IP address (10.70.0.10) will be configured inside Windows after installation — the VM itself does not know about IP addresses at this layer.

Why Standard (Desktop Experience) — not Server Core
Server Core (no GUI) is more secure and uses less RAM. Why use Desktop Experience for domain controllers?
Because this guide is for learning — and GUI tools are the fastest way to build and understand Active Directory configuration for the first time.

Server Core requires all AD DS configuration to be done via PowerShell and command-line tools. This is the production-hardened approach and reduces the attack surface (no GUI processes, no IE, no Windows Explorer). However, it requires familiarity with AD PowerShell cmdlets and has a steeper learning curve.

Desktop Experience provides: Server Manager (the central configuration hub), Active Directory Users and Computers (ADUC), DNS Manager, Active Directory Sites and Services, Group Policy Management — all with graphical interfaces. Every action in this guide can be seen, clicked, and verified visually.

For NovaCrestBank in a production environment: Server Core is the correct long-term choice. After this guide series is complete and the candidate is comfortable with the concepts, the DCs should be rebuilt as Server Core and managed via Remote Server Administration Tools (RSAT) from a management workstation. For now, Desktop Experience is the right choice for learning.

Why DNS is installed together with AD DS — not separately
DNS Server is a separate role from AD DS. Why install them simultaneously rather than adding DNS later?
Because AD DS requires DNS to function, and AD DS installs best when DNS is already present on the same server.

When you promote a server to a Domain Controller, the AD DS promotion wizard looks for a DNS server that can host the AD DS DNS zones. If no DNS server is available, the wizard offers to install DNS and create the zones as part of promotion. This is fine — but installing DNS before AD DS gives you cleaner control over DNS setup.

Why the defaults are correct for NovaCrestBank
Some AD administrators put the database and logs on separate disks for performance. Should NovaCrestBank do this?
No — the default C:\Windows\NTDS path is correct for NovaCrestBank's VM-based DC.

Separating the AD database (ntds.dit) from its log files onto different physical disks was important in the physical-server era. Transaction logs need to be written sequentially — if the database and log files competed for the same disk I/O, performance degraded. On separate disks, the sequential log writes did not compete with random database reads/writes.

On a virtual machine running on vSAN (once vSAN is configured), all disk I/O is distributed across the cluster's NVMe SSDs and is effectively handled by a very fast, distributed storage backend. The performance difference between single-disk and separated-disk configurations for a DC handling authentication for 50 VMs is immeasurable. The default paths are correct and keep the configuration simple.

Leave all three paths as defaults. Click Next.

Why there are two zones — what _msdcs.novacrestbank.local is
Two DNS zones were created. What is _msdcs.novacrestbank.local and why does it exist?
_msdcs (Microsoft Domain Controller Services) is a special DNS zone that Windows clients use to locate domain controllers and global catalog servers.

When a Windows computer needs to authenticate, it does not know which server is a DC by IP — it asks DNS. Specifically, it queries for SRV records in the _msdcs zone: _ldap._tcp._msdcs.novacrestbank.local returns the list of DCs. _kerberos._tcp._msdcs.novacrestbank.local returns the DCs that offer Kerberos. These SRV records are created automatically when AD DS is promoted and are updated automatically when DCs are added or removed.

The _msdcs zone being AD-integrated means it replicates between DC1 and DC2 automatically — no manual zone transfer configuration needed. If DC1 is unavailable, clients query DC2's DNS and still find the SRV records they need.

Both zones should show "Active Directory-Integrated" in the Type column. This is the most important confirmation — it means DNS zones are stored in the AD database and will replicate automatically to DC2 when it is promoted. Standard (file-based) zones would show just "Primary" — if you see that, something went wrong during promotion and DNS was not set up as AD-integrated.

NTP — configure DC1 as stratum 2 time source
Making DC1 the authoritative NTP server for NovaCrestBank's entire infrastructure — the Windows Time service configuration
The NTP hierarchy — why DC1 is stratum 2
NTP stratum numbers indicate distance from an accurate time source. Stratum 1 = directly connected to a hardware clock (GPS, atomic clock). Stratum 2 = synchronised to a stratum 1 source. DC1 syncs to public NTP servers (stratum 1) over the internet — making DC1 stratum 2. All ESXi hosts, Cisco switches, and other infrastructure sync to DC1 — making them stratum 3. This hierarchy ensures all devices in NovaCrestBank have the same time, which is essential for Kerberos authentication (which fails if clocks are more than 5 minutes apart), for certificate validation (certificates have validity periods), and for log correlation.

Windows Server · Win-01 · Section 35
Active Directory Sites and Services — match the physical topology
Configuring AD Sites so replication traffic follows the physical network — and understanding why Sites matter
What AD Sites are and why they must match the physical topology
Active Directory automatically created a site called "Default-First-Site-Name". What is a Site and why configure it?
An AD Site represents a physical or logical network location with reliable high-speed connectivity between its members. Sites control two things: DC-to-DC replication and client-to-DC authentication locality.

Replication control: DCs within the same site replicate changes immediately (within 15 seconds). DCs in different sites replicate on a schedule (typically every 3 hours or on a configured interval). NovaCrestBank has all DCs in one physical location — all in the same site — so replication is immediate.

Client locality: When a Windows machine authenticates, it queries DNS for a DC in its "closest" site. If sites match the physical network, clients find DCs they can reach efficiently. If sites are misconfigured, a machine in London might authenticate against a DC in Sydney — working but slow.

For NovaCrestBank: all infrastructure is in one data center — one AD site is correct. The configuration task is simply renaming "Default-First-Site-Name" to something meaningful, adding the subnets, and confirming both DCs are in the site.

DC2 must point to DC1 for DNS BEFORE joining the domain
When DC2 is a standalone server and you set its DNS to 10.70.0.10 (DC1), DC2 can resolve novacrestbank.local DNS queries — which is required for domain join. If DC2 points to itself (127.0.0.1) before promotion, there is no DNS server running on DC2 yet and name resolution fails. The order is: set DNS to DC1 → join domain → promote to DC → then change DNS to prefer itself.
