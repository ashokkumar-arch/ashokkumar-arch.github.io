Phase 0 · Pre-flight
Pre-flight checks
Run every single one of these before touching the installer. Each one is a gate. If any fails — stop and fix it first.
Non-negotiable rule
Every experienced OpenShift engineer has a war story about skipping a pre-flight check and spending 6 hours debugging a failure that could have been caught in 30 seconds. Run all of these. Every time. No exceptions.

Check 1 — DNS forward + reverse lookups

Run these from the jump host. The jump host must be on a network that uses your internal DNS server (Windows DNS / novacrestbank.local). Every single one of these must resolve correctly before proceeding.

# Wildcard apps record
[svc-ocp@jumphost ~]$ dig +short test.apps.ocp.novacrestbank.local
10.50.0.201              ← good: F5 Ingress VIP

# ── Reverse lookups (PTR records) ────────────────────
[svc-ocp@jumphost ~]$ dig +short -x 10.50.0.11
master01.ocp.novacrestbank.local.   ← good

[svc-ocp@jumphost ~]$ dig +short -x 10.50.0.12
master02.ocp.novacrestbank.local.

[svc-ocp@jumphost ~]$ dig +short -x 10.50.0.13
master03.ocp.novacrestbank.local.
✗ If dig returns empty (no output) or NXDOMAIN — STOP. Add the missing record to Windows DNS, wait 60 seconds for replication, then re-check before proceeding.
✓ All forward AND reverse lookups must return the correct value. Both directions are mandatory — OpenShift verifies both during bootstrap.

Creating install-config.yaml
The most critical file in the entire process — every field explained line by line
Single most important warning in this entire guide
The installer reads install-config.yaml and then deletes it. This is intentional — it prevents accidental re-use. Always back it up before running the installer. If you forget and the install fails, you will need to recreate this file from scratch.

Phase 3 · Manifests
Generating ignition manifests
Understanding what gets generated and where you can (and should) customise before committing
What are ignition files?
Ignition is the first-boot configuration system for RHCOS (Red Hat CoreOS). When a node boots for the first time, it fetches its ignition JSON file and uses it to configure itself — disks, network, users, systemd services, TLS certificates. Think of it as "cloud-init but specifically for CoreOS." The installer generates three ignition files: one for bootstrap, one for masters, one for workers.

Symptom: Installer hangs at "Waiting for Kubernetes API"

What you see — installer stuck

Diagnosis — SSH into bootstrap and check

Diagnosing DNS on the bootstrap node 
[svc-ocp@jumphost ~]$ ssh -i ~/.ssh/ocp_novacrestbank_ed25519 \
  core@10.50.0.10   ← use IP if hostname DNS doesn't work

# Check what DNS server the bootstrap is using
[core@bootstrap ~]$ cat /etc/resolv.conf
nameserver 10.10.0.10
search ocp.novacrestbank.local novacrestbank.local

# Test DNS resolution FROM the bootstrap node
[core@bootstrap ~]$ dig +short api.ocp.novacrestbank.local
                          ← empty output = DNS record missing or wrong server

[core@bootstrap ~]$ dig +short api-int.ocp.novacrestbank.local
                          ← also empty = api-int record missing

# Check bootkube logs for specific error
[core@bootstrap ~]$ journalctl -u bootkube.service | grep -i "error\|fail\|dns"
Apr 13 10:12:44 bootstrap bootkube[4821]: error: failed to connect to api.ocp.novacrestbank.local:6443
Apr 13 10:12:44 bootstrap bootkube[4821]: dial tcp: lookup api.ocp.novacrestbank.local: no such host
                                                    ↑ "no such host" = DNS resolution failing
✗ The DNS record api.ocp.novacrestbank.local is not resolving from the bootstrap node. Most likely cause: the VM's DHCP-provided DNS server is not your internal DNS server (it's using the default gateway's DNS which doesn't know about novacrestbank.local).

Fix: two possible causes and their fixes

Cause A — DNS record not created

Go to Windows DNS Manager → Add the missing A record for api.ocp.novacrestbank.local pointing to 10.50.0.200. Wait 60 seconds, then verify from jump host with dig.

Then destroy and re-run the install:
openshift-install destroy cluster --dir=~/ocp-kit/install/

Cause B — VM using wrong DNS server

The VLAN 50 DHCP scope is handing out the wrong DNS server IP (e.g. the ISP's DNS instead of your internal one). Fix the DHCP scope's DNS option (option 6) to point to your Windows DNS server 10.10.0.10. Destroy and restart install.
