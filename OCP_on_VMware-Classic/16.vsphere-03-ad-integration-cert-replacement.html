The three transformations

Transformation 1 — AD Integration

Before: Only administrator@vsphere.local can log into vCenter. This is a shared credential that cannot be audited per-person.

After: Every NovaCrestBank infrastructure engineer logs in with their own NOVACRESTBANK\username credentials. AD groups map to vCenter roles. Every action is attributed to a specific person in the audit log.

Transformation 2 — vCenter Certificate

Before: vCenter uses a self-signed certificate. Every browser shows "Your connection is not private." Administrators click through the warning — training them to ignore security alerts.

After: vCenter has a CA-signed certificate from NovaCrestBank-Issuing-CA. The padlock is green. No warnings. Trusted by all domain machines automatically via GPO.

Transformation 3 — ESXi Certificates

Before: Each ESXi host uses a self-signed certificate. vCenter accepts them uncritically. An attacker could intercept the vCenter-ESXi management connection undetected.

After: Each ESXi host has a CA-signed certificate. vCenter validates these certificates before connecting. The vCenter-ESXi management channel is cryptographically verified end-to-end.

Unlocked by transformations 1+2+3

ESXi lockdown mode becomes safe to enable (because AD authentication works and cert-based vCenter-ESXi trust means no manual host access needed). vCenter can be managed by named AD accounts. SSH to ESXi is restricted to authenticated channels. The environment is ready for production hardening.

VMware Series · vSphere-03 · Section 02
How vCenter AD integration works
The authentication flow from AD credentials to vCenter access — understanding the mechanism before configuring it
What "Identity Source" means in vCenter SSO
vCenter has its own SSO (Single Sign-On) with a vsphere.local domain. How does Active Directory fit into this?
vCenter SSO supports multiple Identity Sources — directories that SSO can query to authenticate users. Adding Active Directory as an Identity Source allows vCenter SSO to validate NOVACRESTBANK\username credentials against DC01 and DC02.

The authentication flow works like this:
1. Engineer goes to https://vcenter.novacrestbank.local/ui
2. On the login page, selects "NOVACRESTBANK" from the domain dropdown (or types NOVACRESTBANK\username)
3. vCenter SSO receives the credentials and queries DC01 (via LDAPS on port 636) with the bind credentials you configure
4. DC01 validates the username/password and returns the user's group memberships
5. vCenter SSO checks if any of the user's AD groups have been mapped to a vCenter role
6. If yes — the user is logged in with the permissions of their role
7. Every action they take is recorded in the vCenter Events log as NOVACRESTBANK\username — auditable

The vsphere.local SSO domain continues to exist alongside AD — it is not replaced. administrator@vsphere.local remains the break-glass account for when AD is unavailable.
LDAP vs LDAPS — why LDAPS is mandatory
vCenter supports both LDAP (port 389) and LDAPS (port 636). Why must NovaCrestBank use LDAPS?
Because LDAP transmits credentials in clear text — any device on the same network segment can capture the bind password that vCenter sends to the DC. LDAPS encrypts the entire connection using TLS, protecting the credentials.

When vCenter queries Active Directory to validate a user's credentials, it sends a bind request with a service account username and password to the DC. Over plain LDAP (port 389), this bind request is not encrypted — the service account password is visible to anyone with network access who can capture traffic between vCenter (10.10.0.100) and DC01 (10.70.0.10). Even on a well-segmented network, this is unacceptable for a banking environment.

Over LDAPS (port 636), the entire connection from vCenter to DC01 is encrypted with TLS using the Domain Controller's certificate (which DC01 received via auto-enrolment in Win-04). The bind password is never transmitted in clear text. LDAPS requires the Root CA certificate to be trusted by vCenter — which means installing the Root CA certificate into vCenter's trust store, done in Section 07.

For UK financial institutions, encrypted directory communication is expected by FCA, PRA, and GDPR compliance frameworks. LDAP is not an acceptable option.

Certificate replacement — what changes and why it matters
Understanding the scope of certificate replacement in a vSphere environment — not just one certificate, but an entire chain of trust
The certificates being replaced in this guide
vCenter currently has two distinct self-signed certificates: the Machine SSL certificate (used for browser HTTPS connections to the vSphere Client) and the STS (Security Token Service) signing certificate (used for SSO token validation). This guide replaces the Machine SSL certificate — the one browsers connect to. The STS certificate is managed separately and can be replaced later when hardware security modules (Thales HSM) are configured.

What self-signed vs CA-signed means operationally

AspectSelf-signed (current)CA-signed (after this guide)
Browser warning "Your connection is not private" — every access, every user, every browser Green padlock — no warning, trusted automatically via GPO Root CA distribution
ESXi host trust vCenter accepts any certificate from an ESXi host — cannot detect impostor vCenter validates each ESXi certificate against the CA chain — impostor detected immediately
vMotion security vMotion traffic uses ESXi certificate for encryption — self-signed, cannot verify authenticity vMotion encrypted and authenticated — CA-signed ESXi cert verified before migration begins
Audit and compliance Certificate warning bypassed — security warning habituation (dangerous behaviour) Clean HTTPS — engineers see the lock, trust it, and report anomalies when it changes
Why the certificate replacement must happen after AD integration
AD integration and certificate replacement are independent operations. Why do them in this specific order?
Because certificate replacement requires adding the Root CA certificate to vCenter's trust store — and vCenter's certificate trust store (VECS, the vSphere Certificate Store) must be in a stable state before certificates are replaced. Doing AD integration first confirms vCenter is stable and tests LDAPS connectivity (which itself validates the DC's certificate), before the more disruptive certificate replacement operation.

Additionally: if certificate replacement fails and vCenter becomes unreachable, you need another way to access it for recovery. With AD integration already in place and lockdown mode not yet enabled, ESXi hosts are still directly accessible. Doing things in this order maintains the maximum number of recovery paths throughout the process.

Every field — exactly what to enter and why

Identity source type — "Active Directory as LDAP Server"
The first option is "Active Directory (Integrated Windows Authentication)." Sounds simpler — why not use that?
Because Integrated Windows Authentication only works when vCenter is running on a Windows machine that is domain-joined. The VCSA (vCenter Server Appliance) is a Linux appliance — it is not domain-joined and cannot use Kerberos machine account authentication.

"Active Directory as LDAP Server" uses an explicit service account (a bind account) to authenticate to AD over LDAP/LDAPS. This is the correct approach for all VCSA deployments. It requires a dedicated service account in AD with read permissions to the directory.
Identity source type: Active Directory as LDAP Server

Base DN for users: DC=novacrestbank,DC=local
Base DN for groups: DC=novacrestbank,DC=local
Domain name: novacrestbank.local
Domain alias: NOVACRESTBANK
Username (bind acct): NOVACRESTBANK\svc-vcenter-ldap
Password: [from CyberArk — created in next step]
Primary server: ldaps://dc01.novacrestbank.local:636
Secondary server: ldaps://dc02.novacrestbank.local:636
The bind account — svc-vcenter-ldap
What is a "bind account" and why does it need to be a dedicated service account rather than an administrator account?
A bind account is the credential vCenter uses to authenticate to Active Directory so it can search the directory (look up users and groups). It only needs read access — no write access, no elevated privileges whatsoever.

Why dedicated instead of Administrator: if NOVACRESTBANK\Administrator were used as the bind account and that password needed to change, every system using it would need to be updated simultaneously. A dedicated service account (svc-vcenter-ldap) has a password that only needs to be known to vCenter's configuration — it never changes because no human uses it interactively, and if it does need to change, only vCenter's identity source configuration needs updating.

The service account only needs: Read access to the entire directory (default for all domain users). No additional permissions are required. The account should be in a dedicated "Service Accounts" OU with a strong password policy.

Create the bind account in AD before filling in the form

VMware Series · vSphere-03 · Section 07
Configure LDAPS — add the Root CA certificate to vCenter's trust store
The step that many guides skip — without the Root CA certificate in vCenter's trust store, LDAPS fails with a certificate validation error
Without this step, LDAPS will fail even if port 636 is open
vCenter validates the TLS certificate presented by DC01 when making an LDAPS connection. DC01 presents its Domain Controller certificate — signed by NovaCrestBank-Issuing-CA, which is signed by NovaCrestBank-Root-CA. If vCenter's trust store does not contain the Root CA certificate, it cannot validate the chain and rejects the connection with "SSL certificate verification failed." This is the most common reason vCenter LDAPS configuration fails.

Method 1 — Use the certificate upload in the Add Identity Source dialog

When the "Active Directory as LDAP Server" type is selected and LDAPS URLs are entered, the Add Identity Source dialog shows a "Browse" button next to "SSL Certificates." Click Browse and upload the Root CA certificate (.cer or .crt file). vCenter adds the certificate to its LDAPS trust store, allowing it to validate the DC's certificate chain.

  • 1
    In the Add Identity Source dialog — find the SSL Certificates field

    After entering the LDAPS URLs (ldaps://dc01.novacrestbank.local:636), the dialog shows a section labelled "SSL Certificates" with a Browse button. This upload adds the CA certificate that signed DC01's LDAPS certificate to vCenter's trust for this identity source.

  • 2
    Click Browse — upload the Root CA certificate file

    Click Browse. A file upload dialog opens. Navigate to the Root CA certificate file. You previously placed it at http://pki.novacrestbank.local/pki/rootca.novacrestbank.local_NovaCrestBank-Root-CA.crt. Download it to your laptop from that URL first (open it in a browser, then save). Upload it via the Browse dialog. The certificate is validated and shown: "NovaCrestBank-Root-CA — Valid until April 2046."

Method 2 — Add the Root CA to vCenter's trusted store via VAMI (more permanent)

The dialog's SSL certificate upload applies to that specific identity source. For a more permanent solution (trusted by all vCenter services, not just this LDAPS connection), also add the Root CA certificate to the VCSA's trusted root store via VAMI:

VCSA SSH — add Root CA to VECS (vSphere Certificate Store)
# SSH to the VCSA from the jump host
[jumphost]$ ssh root@10.10.0.100
# Password: NovaCrest$vCenter2026! (VCSA root password)

# First, copy the Root CA certificate to the VCSA
# (from the pki web server — curl it directly)
root@vcsa []# curl -o /tmp/rootca.crt http://pki.novacrestbank.local/pki/rootca.novacrestbank.local_NovaCrestBank-Root-CA.crt
  % Total    % Received  100  1350  100  1350  -- done

# Add the Root CA certificate to VECS TRUSTED_ROOTS store
root@vcsa []# /usr/lib/vmware-vmafd/bin/dir-cli trustedcert publish --cert /tmp/rootca.crt --login administrator@vsphere.local
Password for administrator@vsphere.local: NovaCrest$vCenter2026!
Certificate published to VMware Directory Service.

# Also add to the VCSA system trust store
root@vcsa []# cp /tmp/rootca.crt /etc/ssl/certs/NovaCrestBank-Root-CA.crt
root@vcsa []# update-ca-certificates
1 added, 0 removed; done.

# Verify the Root CA is trusted
root@vcsa []# openssl s_client -connect dc01.novacrestbank.local:636 -CAfile /tmp/rootca.crt 2>&1 | grep -E "Verify|issuer"
issuer=CN=NovaCrestBank-Issuing-CA, DC=novacrestbank, DC=local
Verify return code: 0 (ok)

↑ Verify return code: 0 (ok) ← DC01's LDAPS cert validates against Root CA ✓
  vCenter can now establish a trusted LDAPS connection to DC01
✓Root CA certificate is trusted by the VCSA. openssl verifies DC01's LDAPS certificate successfully. The LDAPS connection will work without SSL errors when the identity source is added.

Why set AD as default — not leave vsphere.local as default
vsphere.local is the original SSO domain. Why make AD the default instead?
Because 99% of all vCenter logins will use AD credentials. Having engineers select the domain from a dropdown every time is unnecessary friction — and increases the chance of accidentally logging in with the wrong domain.

The vsphere.local domain and its administrator@vsphere.local account still exist and still work — the default selection only changes what is pre-selected in the dropdown. When an engineer needs to use administrator@vsphere.local (break-glass scenario), they manually select vsphere.local from the dropdown. This is intentional — it adds a deliberate step to using the break-glass account, which should feel slightly different from normal login.
VMware Series · vSphere-03 · Section 10
Create AD security groups for vCenter roles
The principle: AD groups map to vCenter roles — never individual AD users. Groups make permissions manageable and auditable.
Groups not users — why this is the only correct approach
Why map AD groups to vCenter roles rather than individual user accounts?
Because individual user-to-role mappings do not scale and create operational problems when staff changes occur.

If NOVACRESTBANK\jsmith is directly assigned the vCenter Administrator role, and jsmith leaves the organisation, someone must remember to remove jsmith from vCenter before the account is disabled in AD. With group-based permissions: NOVACRESTBANK\vCenter-Admins is assigned the Administrator role. When jsmith leaves, they are removed from the vCenter-Admins AD group in AD — which immediately revokes vCenter access without any vCenter-specific action. When a new engineer joins, they are added to the appropriate AD group — immediately granting vCenter access without any vCenter-specific action.

Groups also make auditing easier: a vCenter admin can look at the AD group membership to see who has what access, without needing to query vCenter's global permissions.
PowerShell — DC01 — create vCenter AD security groups
# Create a dedicated OU for vCenter groups (clean separation)
PS C:\> New-ADOrganizationalUnit -Name "vCenter Groups" -Path "DC=novacrestbank,DC=local"

# Create four AD security groups — one per vCenter permission level
# Group 1: vCenter Administrators — full control of the vSphere environment
PS C:\> New-ADGroup -Name "vCenter-Admins" -GroupScope Global -GroupCategory Security `
    -Path "OU=vCenter Groups,DC=novacrestbank,DC=local" `
    -Description "Full vCenter Administrator role — infrastructure team only"

# Group 2: vCenter Read-Only — view all but change nothing (auditors, helpdesk)
PS C:\> New-ADGroup -Name "vCenter-ReadOnly" -GroupScope Global -GroupCategory Security `
    -Path "OU=vCenter Groups,DC=novacrestbank,DC=local" `
    -Description "vCenter read-only access — monitoring and auditing"

# Group 3: vCenter Operators — can power VMs on/off, view consoles, but not modify config
PS C:\> New-ADGroup -Name "vCenter-Operators" -GroupScope Global -GroupCategory Security `
    -Path "OU=vCenter Groups,DC=novacrestbank,DC=local" `
    -Description "vCenter operator access — VM power management and console access"

# Group 4: vCenter No Access — explicitly denied (useful for break-glass prevention)
PS C:\> New-ADGroup -Name "vCenter-NoAccess" -GroupScope Global -GroupCategory Security `
    -Path "OU=vCenter Groups,DC=novacrestbank,DC=local" `
    -Description "Explicitly no vCenter access — security team managed"

# Add the first admin to the vCenter-Admins group
# (Replace 'administrator' with the actual NovaCrestBank infrastructure admin account)
PS C:\> Add-ADGroupMember -Identity "vCenter-Admins" -Members "Administrator"
Group membership updated ✓

# Verify groups exist
PS C:\> Get-ADGroup -Filter "Name -like 'vCenter*'" | Select Name,DistinguishedName
Name              DistinguishedName
----              -----------------
vCenter-Admins    CN=vCenter-Admins,OU=vCenter Groups,DC=novacrestbank,DC=local
vCenter-ReadOnly  CN=vCenter-ReadOnly,OU=vCenter Groups,DC=novacrestbank,DC=local

Certificate replacement strategy — what approach to use
VMware provides multiple methods for replacing vCenter certificates — understanding which to use and why
Choosing the certificate replacement method
VMware offers three approaches: VMCA (VMware Certificate Authority), Custom Certificates via VAMI, and Certificate Manager CLI. Which does NovaCrestBank use?
NovaCrestBank uses Custom Certificates via VAMI (the vCenter Appliance Management Interface) for the Machine SSL certificate. This is the method that works with an external CA (the NovaCrestBank Issuing CA) and gives the most explicit control over what is being replaced.

VMCA (subordinate): You can make VMCA a subordinate of the NovaCrestBank Issuing CA — VMCA then signs all vCenter and ESXi certificates automatically. This is elegant but requires adding VMCA as a sub-CA under the Issuing CA, which adds another CA tier to the PKI. Not chosen for NovaCrestBank — unnecessary complexity for a small environment.

Custom Certificates via VAMI: You generate a CSR from VAMI, sign it at the Issuing CA Web Enrollment, and install the signed certificate back in VAMI. Explicit, auditable, and uses the same Issuing CA as all other infrastructure certificates. This is what NovaCrestBank uses.

Certificate Manager CLI: A command-line tool on the VCSA that offers similar functionality. Useful for automation but the VAMI GUI approach is more transparent for a learning environment.

VMware Series · vSphere-03 · Section 20
ESXi certificate replacement — the approach
Why ESXi host certificates matter and the two approaches available
Why ESXi certificates must also be replaced
vCenter's certificate is now CA-signed. Why do the ESXi hosts also need CA-signed certificates?
Because vCenter connects to ESXi hosts over HTTPS — and without CA-signed certificates on ESXi, vCenter cannot cryptographically verify it is connecting to the real host rather than a man-in-the-middle.

Currently, vCenter accepts ESXi self-signed certificates without validation (thumbprint-based trust — vCenter trusts whatever certificate a host presents once it is added to the cluster). A compromised network device between vCenter and an ESXi host could present a fake ESXi certificate and intercept management traffic — vCenter would not detect this.

With CA-signed ESXi certificates and Custom certificate mode enabled in vCenter: vCenter validates each ESXi host's certificate against the NovaCrestBank CA chain before connecting. An impostor cannot present a valid certificate because only the Issuing CA can sign certificates for esxi01.novacrestbank.local — and the Issuing CA requires authentication before signing anything. The vCenter-ESXi management channel is end-to-end verified.

The two approaches to ESXi certificate replacement

Approach 1 — VMCA Subordinate (automatic)

Make VMCA a subordinate CA under the NovaCrestBank Issuing CA. VMCA then automatically signs all ESXi certificates. Simple operation but adds a CA tier to the PKI.

Not used — adds PKI complexity

Approach 2 — Custom Certificates (explicit)

Generate a CSR for each ESXi host, sign it at the Issuing CA Web Enrollment, and install via vCenter. Explicit control, uses the existing PKI, no new CA tier. This is what NovaCrestBank uses.

✓ Selected approach

Enable ESXi lockdown mode
Now that AD authentication works and certificates are CA-signed, lockdown mode can be safely enabled — this is the security hardening step that was deferred until the PKI was ready
What lockdown mode does and why it is now safe to enable
Lockdown mode was left disabled during the build. What does it do and what made it unsafe to enable earlier?
Lockdown mode restricts direct ESXi host access — all management must flow through vCenter. Direct root SSH, direct Host Client browser access, and direct API calls to the ESXi host are blocked for all users except those explicitly whitelisted in the DCUI access list.

Why it was unsafe to enable earlier: During the build, direct ESXi access (SSH, Host Client, DCUI) was used frequently — to configure the vSwitch before vCenter existed, to troubleshoot network issues, to manage VMs before the cluster was stable. Enabling lockdown during the build would have blocked these critical access paths.

Why it is now safe: vCenter is running with CA-signed certificates and AD authentication. All infrastructure management happens through vCenter. The DCs, DHCP, PKI, and the vDS are all configured. The iLO emergency access path is always available regardless of lockdown mode — iLO is not affected. Engineers authenticated via NOVACRESTBANK AD credentials can manage everything through vCenter. There is no operational reason to allow direct ESXi access for routine administration.

What lockdown does not block: vCenter's management connection to ESXi (vpxa agent — this is how vCenter manages hosts). iLO remote console access (physical access equivalent). Explicitly whitelisted accounts in the Exception Users list (e.g., root can be kept in the exception list for emergencies).
