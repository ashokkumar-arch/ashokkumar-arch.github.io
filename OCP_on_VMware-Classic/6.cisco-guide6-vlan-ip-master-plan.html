Cisco Fabric · Guide 6 · Section 01
How to think about VLAN design — the decision framework
Every VLAN in NovaCrestBank exists because of a specific reason. This guide teaches you to think like the engineer who designed it, not just remember what was configured.
The mental model shift this guide creates
After Guides 1–5, you know what was configured — the commands, the outputs, the verification. This guide answers why. Knowing why is what allows you to make decisions in a new environment you have never seen before. When someone asks "should we put vSAN on the same VLAN as vMotion?" you will know immediately why the answer is no — because you understand the principle, not because you memorised a rule.

The four questions every VLAN decision must answer

Before any VLAN is created, a senior network engineer asks four questions. This guide walks through all four questions for every VLAN in NovaCrestBank.

1. Why does this traffic need its own VLAN?

What would happen if this traffic shared a VLAN with something else? Would it cause a security problem? A performance problem? An operational problem? If none of those — is a separate VLAN really needed?

2. What traffic type is this?

Is it management traffic (admin access to devices)? Storage traffic (high bandwidth, latency-sensitive)? VM traffic (unpredictable, user-generated)? Knowing the traffic type determines QoS policy, MTU settings, redundancy requirements, and security controls.

3. Who and what connects to this VLAN?

Which devices have IPs in this VLAN? Which switch ports carry this VLAN? Which firewall zones does this VLAN touch? Answering this question defines the trunk policy — which ports carry this VLAN and which do not.

4. What are the performance and security requirements?

Does this traffic need jumbo frames? Low latency? Rate limiting? Firewall inspection? Encryption? These requirements determine MTU settings, QoS markings, and whether the traffic goes through the Palo Alto before reaching its destination.

The five reasons to separate traffic into different VLANs

ReasonExplanationNovaCrestBank example
Security isolation Traffic that should not be reachable from other traffic — even within the data center. A VLAN boundary forces all cross-VLAN traffic through the router (and potentially the firewall), creating a choke point for security inspection. Windows infrastructure (VLAN 70) is isolated from VM production (VLAN 60). A compromised VM cannot directly reach a domain controller — traffic must be routed and is subject to firewall rules.
Performance isolation Traffic that, if mixed with other traffic, would cause one to degrade the other. High-bandwidth storage traffic competes with interactive user traffic for buffer space and bandwidth. vSAN (VLAN 25) is isolated so its replication traffic cannot cause congestion on the management VLAN (VLAN 10). An ESXi host under heavy vSAN load does not affect admin SSH sessions.
QoS policy differences Different traffic types need different quality-of-service treatment. Storage traffic needs low latency. Management traffic needs guaranteed delivery but tolerates some delay. VM traffic is best-effort. vMotion (VLAN 20) is marked with DSCP AF21. vSAN (VLAN 25) is marked with DSCP AF31 (higher priority than vMotion — storage latency matters more than migration speed).
Operational clarity When something breaks, you need to know immediately which type of traffic is affected. A separate VLAN means a separate subnet, which means tools like Wireshark, flow monitoring, and SNMP traps immediately tell you which traffic category has a problem. An alert fires for 10.25.0.0/24. You immediately know: vSAN is affected. You do not need to look at packet captures to figure out what type of traffic is failing.
Compliance and audit PCI-DSS requires cardholder data to be separated from other traffic. PCI-DSS also requires that management access to in-scope systems be auditable and separate from data traffic. The OOB management VLAN (VLAN 40) is the only path for admin access to switches and servers during a security audit. Its complete separation makes it trivial to demonstrate compliance.

Why separate VLANs at all — what happens without them
The clearest way to understand why VLAN separation is essential is to imagine NovaCrestBank without it — one flat network.
The flat network scenario — what happens without VLANs
Imagine all of NovaCrestBank's traffic — ESXi management, vMotion, vSAN, NFS, VMs, Windows DCs, OpenShift nodes — on a single /16 network: 10.0.0.0/16. Every device can reach every other device directly. Let us walk through what that means in practice.

Problem 1 — Security: A compromised VM can reach everything

A banking application VM gets compromised. The attacker now has a foothold with an IP in 10.0.0.0/16. From this IP, the attacker can directly reach:

  • The Windows Domain Controllers (port 389, 636) — full AD query access
  • The ESXi host management interface (port 443) — potential hypervisor access
  • The vCenter server — can enumerate and potentially power-cycle any VM
  • The NetApp NFS server — can mount NFS shares directly
  • Other VMs without any firewall inspection

With VLAN separation: the compromised VM is in VLAN 60 (VM Production). To reach the Domain Controller in VLAN 70, traffic must be routed through the SVI, which routes it to the spine, which sends it to the VLAN 70 SVI. Every hop is a point where firewall rules can be applied. The Palo Alto inspects inter-zone traffic. The attack surface is dramatically reduced.

Problem 2 — Performance: vSAN kills management access

vSAN replication during a disk rebuild can saturate a 25G link for several minutes. In a flat network, this congestion affects every device on that link — including the ESXi management VMkernel adapter used for SSH and vCenter connectivity. During a disk rebuild you cannot reliably manage the ESXi host because management packets are competing with gigabytes of storage traffic in the same queue.

With VLAN separation and QoS: vSAN traffic is in VLAN 25, marked with a specific DSCP value that places it in a medium-priority queue. Management traffic in VLAN 10 is marked with a higher-priority DSCP value. The switch's QoS policy guarantees management traffic always gets through, even during full vSAN saturation. You always have console access to the hosts you are trying to fix.

Problem 3 — Troubleshooting: You cannot isolate the problem

An alert fires: "Something is wrong with storage." On a flat network, your Wireshark capture shows: vSAN traffic, NFS traffic, management traffic, VM traffic, vMotion traffic — all mixed together. Finding the specific anomaly takes an hour. With VLAN separation, the same alert fires with source subnet 10.25.0.0/24 — you immediately know it is vSAN, not NFS. Your entire troubleshooting scope narrows to one VLAN and one set of devices in under 10 seconds.

IP address philosophy — the choices made and why
Understanding the principles behind NovaCrestBank's entire IP addressing scheme — choices that affect every device, forever
IP address design is a permanent decision — get it right the first time
Unlike a VLAN name or a port description, renumbering an IP address scheme in a production data center is extremely disruptive. Every device, every firewall rule, every monitoring alert, every certificate SAN, every DNS record references these IP addresses. The IP design made on day one will be lived with for 5–10 years. These are the principles NovaCrestBank used.

Principle 1 — Use RFC 1918 private address space exclusively

All internal addresses use 10.x.x.x — part of the RFC 1918 private address space (10.0.0.0/8). NovaCrestBank chose the 10.x.x.x range over 172.16.x.x or 192.168.x.x for one reason: the 10.x.x.x space is large enough to accommodate the entire data center with meaningful, readable subnets without running out of room. 192.168.0.0/16 gives only 65,534 addresses — too small for a production enterprise. 172.16.0.0/12 is often confused with public addresses by less-experienced admins. 10.0.0.0/8 is unambiguous and universal.

Principle 2 — Encode the VLAN ID in the third octet

Design decision
Why is VLAN 10 given subnet 10.10.0.0/24 — and not 10.1.0.0/24 or 192.168.10.0/24?
Because the third octet matches the VLAN ID. VLAN 10 = 10.10.0.0/24. VLAN 25 = 10.25.0.0/24. VLAN 70 = 10.70.0.0/24. This is not accidental — it is a deliberate convention that makes every IP address self-documenting. When you see the IP 10.50.0.11 in a log, you immediately know: this is a device in VLAN 50 (OCP Nodes). You do not need to look up a spreadsheet. This convention pays dividends thousands of times over the lifetime of the data center.

Principle 3 — Reserve the 10.0.x.x block for infrastructure links

Design decision
Why do the spine-leaf BGP underlay links use 10.0.x.x addresses — separate from everything else?
Because infrastructure links (the /30 point-to-point links between switches) must be distinguishable from device addresses at a glance. A firewall rule that says "deny 10.0.0.0/16" blocks all infrastructure link traffic — useful in some security policies. A BGP prefix list that says "reject any prefix within 10.0.0.0/16 from leaf switches" specifically blocks a leaf from accidentally advertising infrastructure links as usable routes. The separation makes routing policy simple and unambiguous.

Principle 4 — Consistent gateway addressing (.1, .2, .3)

Design decision
Why do leaf01's SVIs always end in .1, leaf02's in .2, and leaf03's in .3?
Because consistent addressing makes documentation, troubleshooting, and training dramatically simpler. An engineer who knows this convention can derive any gateway IP in their head. "What is the vSAN gateway on leaf02?" — 10.25.0.2. No spreadsheet needed. This convention also makes firewall rule authoring easier: rules that reference "the management gateway" are always x.x.x.1, never a random IP that requires lookup.

Principle 5 — Leave room for growth within each /24

A /24 gives 254 usable host addresses. NovaCrestBank currently has 5 ESXi hosts — that is only 5 IPs in VLAN 25 (vSAN). Why waste a /24 on 5 devices? Because the data center is designed to grow to 20+ hosts without renumbering. Renumbering a vSAN network means taking every ESXi host offline to change its VMkernel IP — a multi-hour maintenance window. A /24 costs nothing (private address space is effectively unlimited) and buys permanent flexibility.

VLAN 10 — ESXi Management
The most important VLAN in the data center — if this breaks, all management access to the infrastructure is at risk
10
VLAN 10
ESXi Management
10.10.0.0/24 · Gateway .1/.2/.3 per leaf · 254 usable addresses
Why does this need its own VLAN?

ESXi management traffic is the most sensitive traffic in the data center — it includes SSH sessions to ESXi hosts, vCenter API calls, vCenter UI access, and host-to-host management communication. If this traffic shared a VLAN with VM traffic, a compromised VM could attempt to reach the ESXi management interface (port 443) directly. Separating it means a compromised VM must route through the SVI and through firewall inspection to reach any management address. The separation also means management traffic quality is protected — a noisy VM workload cannot cause management packet loss.

What traffic lives here?

VMkernel adapter vmk0 on every ESXi host. vCenter Server Appliance (VCSA) at 10.10.0.100. ESXi SSH management (port 22). ESXi HTTPS management (port 443). vCenter HTTPS API (port 443, 902). iSCSI software initiator (if used — NovaCrestBank uses NFS, not iSCSI, but the SVI serves both). NFS client traffic from ESXi hosts (although NovaCrestBank puts NFS on its own dedicated VLAN 30 — see below).

10.10.0.3leaf03 SVI — gateway for Rack 3esxi05 and devices in Rack 3

10.10.0.12esxi02 vmk0Host 2, Rack 1

10.10.0.14esxi04 vmk0Host 4, Rack 2

Performance and MTU considerations

Standard MTU (1500 bytes). Management traffic — SSH sessions, vCenter API calls, HTTPS web UI — consists of small packets. Jumbo frames provide no benefit here and would add complexity. VLAN 10 uses 1500-byte MTU throughout.

QoS marking: DSCP CS6. Management traffic is marked with DSCP CS6 (network control class) — this gives it the highest non-reserved priority in the switch's QoS queues. Management access must always be available, even during periods of heavy storage or VM traffic.

Security considerations

This VLAN is in PCI-DSS scope because vCenter and ESXi hosts are part of the infrastructure that processes cardholder data. All SSH sessions to ESXi hosts are logged via the audit trail. vCenter SSO is integrated with Active Directory — no local vCenter accounts for daily operations. The Palo Alto has explicit rules allowing only the jump host (in VLAN 40) to reach VLAN 10 management interfaces from external networks.

VLAN 20 — vMotion
The live migration highway — why vMotion must be completely isolated and what happens if it is not
20
VLAN 20
vMotion
10.20.0.0/24 · Gateway .1/.2/.3 per leaf · 254 usable addresses
Why does vMotion need its own VLAN?

vMotion is a live VM migration protocol. When an ESXi host migrates a running VM to another host, it copies the VM's memory over the network in real time. A 64 GB RAM VM migrating between hosts generates 64 GB of network traffic — at 25G speeds, this takes about 20 seconds. During those 20 seconds, the vMotion VMkernel link is saturated.

If vMotion shared the management VLAN: those 20 seconds of saturation would cause management packet drops — vCenter might temporarily lose communication with the host, triggering false alarms or, worse, triggering HA failover on a host that is perfectly healthy but just busy migrating VMs. The separation prevents this false alarm scenario entirely.

There is also a security argument: the memory contents of a running VM transit the vMotion network in cleartext (unless vMotion encryption is enabled). If vMotion shared a VLAN with general VM traffic, a compromised VM with a sniffer could capture memory pages of other VMs as they migrate. Isolation ensures only the VMkernel adapters of ESXi hosts are on this VLAN — no VM workloads, no management interfaces.

What traffic lives here?

Exclusively VMware vMotion migration traffic. VMkernel adapter vmk1 on every ESXi host. No other traffic belongs on this VLAN. Not NFS, not management, not VM workloads. The VMkernel adapter on this VLAN is explicitly tagged for vMotion traffic only — ESXi enforces this at the hypervisor level.

IP assignments
10.20.0.11–15esxi01–05 vMotion VMkernel (vmk1)One per host, sequential
10.20.0.1–3Leaf SVI gatewaysNot typically pinged — SVIs exist for routing
Performance — why standard MTU is acceptable here

vMotion works well at 1500-byte MTU because VMware's vMotion protocol handles segmentation internally. However, enabling jumbo frames (9000 bytes) on vMotion reduces the CPU overhead of segmentation on both source and destination hosts. NovaCrestBank uses standard 1500 MTU on VLAN 20 in the initial deployment — this can be changed to 9000 later if vMotion performance becomes a bottleneck. Changing MTU on vMotion requires placing each ESXi host in maintenance mode briefly.

VLAN 25 — vSAN
The most performance-sensitive VLAN in the data center — vSAN has requirements that affect every design decision
25
VLAN 25
vSAN
10.25.0.0/24 · MTU 9000 · Dedicated NICs · No sharing
Why VLAN 25 — not VLAN 20 or VLAN 30?

vSAN is the most demanding traffic in NovaCrestBank's fabric. It requires:

  • Jumbo frames (MTU 9000) — VMware requires MTU 9000 for vSAN. This is not optional — vSAN will degrade significantly on 1500-byte MTU because storage block sizes are much larger than 1500 bytes, causing fragmentation on every write.
  • Dedicated bandwidth — During a disk failure or rebuild event, vSAN generates sustained gigabytes of replication traffic for hours. This must not affect any other workload.
  • Low latency — vSAN reads and writes directly affect VM disk I/O latency. Any congestion in the vSAN path becomes disk latency for every VM.
  • No broadcast storms — vSAN uses multicast for certain operations. Keeping vSAN on its own VLAN limits multicast propagation to only ESXi hosts.

The VLAN ID 25 was chosen to fall between vMotion (20) and NAS/NFS (30) — a logical grouping of storage-related VLANs in the 20s and 30s range that makes the VLAN plan readable.

Why dedicated NICs for vSAN?

NovaCrestBank assigns vmnic4 and vmnic5 (a dedicated NIC pair) exclusively to vSAN. These are physically separate NICs from the management NICs (vmnic0/1) and the VM/OCP NICs (vmnic2/3). This is a VMware best practice — not a preference.

If vSAN shared NICs with management: a vSAN rebuild event (generating 50 Gbps of replication traffic on a 25G NIC) could completely starve management traffic — making the ESXi host unmanageable at the exact moment when you are trying to diagnose a storage failure. Dedicated NICs mean vSAN and management are physically independent paths.

Why MTU 9000 must be verified end-to-end

MTU 9000 is the most common misconfiguration source in vSAN deployments. The path for vSAN traffic is: ESXi VMkernel adapter (MTU 9000) → physical NIC → leaf switch port (MTU 9000) → leaf SVI VLAN 25 (MTU 9000) → spine interface (MTU 9000) → remote leaf SVI (MTU 9000) → remote leaf port → remote ESXi NIC → remote VMkernel. Every single hop must be 9000. One hop at 1500 causes silent packet drops on oversized frames — vSAN degrades but does not produce an obvious error. The symptom is slow storage performance, not an error message. Always verify with vmkping -d -s 8972.

Cisco Fabric · Guide 6 · Section 07
VLAN 30 — NAS / NetApp NFS
The network file system VLAN — why NFS gets its own dedicated path even though it is also storage traffic
30
VLAN 30
NAS — NetApp NFS
10.30.0.0/24 · MTU 9000 · NetApp AFF A400 · Shared by ESXi and OCP pods
Why is NFS separate from vSAN (VLAN 25)?

Both vSAN and NFS are storage traffic — but they are fundamentally different in nature and serve different purposes in NovaCrestBank.

vSAN is distributed storage built from the local SSDs inside the ESXi hosts. vSAN traffic flows between ESXi hosts only — host to host. It is always internal to the cluster.

NFS is centralised storage from the NetApp AFF A400 NAS appliance. NFS traffic flows from ESXi hosts and OpenShift pods to a single physical device (the NetApp). Mixing these means: if the NetApp generates a traffic burst (large backup job), it competes with vSAN replication — potentially causing vSAN heartbeat timeouts that trigger unnecessary disk rebuild events.

Separating them also means separate QoS policies can be applied if needed, and troubleshooting is simpler — a vSAN latency issue is immediately distinguishable from an NFS latency issue by VLAN.

What connects to VLAN 30?

The NetApp AFF A400 NFS LIFs (Logical Interfaces — virtual IPs on the storage controller), connected via Ethernet1/13-16 on leaf03. ESXi host VMkernel adapters for NFS (configured in the VMware guide). OpenShift pods via the Trident CSI driver (which creates NFS-backed PVCs — as covered in the OpenShift storage guide). The jump host does NOT connect to VLAN 30 — NFS traffic is data-plane only, never admin traffic.

Why does NFS need MTU 9000?

NFS read operations transfer large amounts of data per operation — often 64 KB or 128 KB I/O sizes. At standard 1500-byte MTU, a 64 KB NFS read requires approximately 43 separate IP packets. At 9000-byte MTU, the same read requires only 8 packets. Fewer packets means less interrupt overhead on the ESXi host CPU and the NetApp controller, and less header overhead consuming bandwidth. NFS performance is measurably better with jumbo frames — VMware recommends MTU 9000 for all NFS datastores.

VLAN 40 — OOB Management
The most unusual VLAN in the data center — it exists on completely different physical hardware from everything else
40
VLAN 40
OOB Management
10.40.0.0/24 · Catalyst 9200 ONLY · Never on leaf/spine production fabric
Why is OOB management on a physically separate switch?

This is the most important architectural decision in NovaCrestBank's entire network design. The OOB management network must survive any failure in the production network. If VLAN 40 were simply a VLAN on the leaf-spine fabric, then a misconfiguration that broke the fabric would also break management access — the exact moment when you need management access most.

By putting VLAN 40 on a completely separate Catalyst 9200 switch with completely separate Cat6 cables to every device's dedicated management port (mgmt0 on Nexus, iLO port on servers, MGT port on Palo Alto), the OOB network is physically independent. A catastrophic leaf failure does not affect OOB. A BGP misconfiguration that drops all production routes does not affect OOB. A spanning tree loop does not affect OOB. Nothing in the production fabric can touch OOB.

Security — why OOB is a separate security zone

The OOB network is also a strict security zone. The only way to reach 10.40.0.0/24 from the outside world is via the jump host (CyberArk PAM), which is the only system with a firewall rule allowing it. A developer, a monitoring tool, or even a compromised system in the production network cannot reach 10.40.0.0/24 — there is literally no route from the production VLANs to the OOB subnet without going through the Palo Alto and being explicitly permitted.

This is PCI-DSS Requirement 2.2 in practice: "system components are protected from known vulnerabilities by installing applicable security patches." The OOB network means you can always apply patches and configurations to infrastructure devices, regardless of what is happening on the production network.

The critical rule: VLAN 40 never appears on leaf/spine switches

The Nexus leaf and spine switches carry VLANs 10, 20, 25, 30, 50, 60, and 70. They do NOT carry VLAN 40. If someone mistakenly adds VLAN 40 to a leaf switch trunk port, OOB traffic could theoretically traverse the production fabric — immediately defeating the purpose of OOB. This is why the VLAN database on leaf01 through leaf03 never contains VLAN 40. It is simply not created. If you run show vlan brief on any leaf switch and see VLAN 40, something is wrong.

VLAN 50 — OpenShift Nodes
The Kubernetes infrastructure VLAN — why OCP gets its own segment and how it connects back to everything you built in the OpenShift guides
50
VLAN 50
OpenShift Nodes
10.50.0.0/24 · Masters · Infra nodes · Workers · API VIP · Ingress VIP
Why does OpenShift need its own VLAN — not just VLAN 60 (VM Production)?

This decision deserves careful thought because OpenShift nodes are VMs — they could theoretically live in VLAN 60 (VM Production). There are three strong reasons they do not.

Reason 1 — Network policies and security. OpenShift runs its own internal network (OVN-Kubernetes — 10.128.0.0/14 by default for pod traffic). The Palo Alto NGFW needs to apply specific security policies to OpenShift node IPs as a group. Having all OCP nodes in a dedicated /24 means the firewall policy can say "permit from OCP nodes to OCP API VIP" using a single subnet match — not a list of individual IPs scattered across VLAN 60.

Reason 2 — Traffic patterns. OCP nodes generate significant east-west traffic: pod-to-pod OVN tunnels (Geneve, UDP 6081), API server communication, etcd replication. This traffic pattern is distinct from general VM traffic. Isolating it allows QoS policies to be applied specifically to the OCP subnet.

Reason 3 — Future growth. OpenShift clusters scale independently. New worker nodes are always added to VLAN 50. Having a dedicated /24 means the address space for OCP is clear and never competes with other VMs.

How this VLAN connects to the OpenShift guides

Every IP address in the OpenShift guide series (Guides 1–7) lives in 10.50.0.0/24. When the OpenShift installer runs, it assigns IPs from this subnet to each node VM. The vCenter port group PG-VLAN50-OCP-Nodes (configured in the VMware guide) connects directly to the leaf's vPC port channel that carries VLAN 50. The entire OpenShift network stack — from node IP to pod overlay to service ClusterIP — sits on top of this one VLAN.

VLAN 60 — VM Production
The general-purpose VM network — why it is the broadest VLAN and what that implies
60
VLAN 60
VM Production
10.60.0.0/24 · General banking application VMs · Broad access · Firewall-inspected
What goes in VLAN 60 — and the important boundary

VLAN 60 is for production VM workloads that are not Windows infrastructure (VLAN 70) and not OpenShift nodes (VLAN 50). This includes: Oracle database VMs, legacy banking application servers that are not containerised, middle-tier application VMs, batch processing VMs.

The important boundary: VLAN 60 is not a "catch-all" for anything that doesn't fit elsewhere. Every VM placed in VLAN 60 should be a deliberate decision. The temptation in enterprise environments is to put everything in the "general VM" VLAN. This creates a flat, uncontrolled network that gradually accumulates every workload with no security segmentation. NovaCrestBank's governance rule: any new VM workload must be assessed — does it belong in an existing VLAN, or does it need a new dedicated VLAN?

Security posture for VLAN 60

VMs in VLAN 60 are treated as untrusted by default. This does not mean they are hostile — it means the security policy assumes any VM could be compromised, and designs controls accordingly. Firewall rules from VLAN 60 to VLAN 70 (Windows Infrastructure) are explicit and minimal — a VLAN 60 VM should never need to reach a Domain Controller directly unless specifically justified. All outbound traffic from VLAN 60 to the internet goes through the Palo Alto for inspection.

Cisco Fabric · Guide 6 · Section 11
VLAN 70 — Windows Infrastructure
The identity and certificate authority network — why Windows services are strictly isolated from everything else
70
VLAN 70
Windows Infrastructure
10.70.0.0/24 · AD DS · AD CS · DHCP · DNS · NTP — the identity fabric
Why is Windows infrastructure separate from VM Production?

The Windows infrastructure VMs — Domain Controllers, Certificate Authority servers — are the most sensitive systems in NovaCrestBank's internal network. Here is why their compromise is catastrophic:

  • A compromised Domain Controller gives an attacker full control of every machine joined to the domain — ESXi hosts, vCenter, all Windows VMs.
  • A compromised Certificate Authority allows an attacker to issue certificates for any hostname — enabling man-in-the-middle attacks on any TLS connection including banking applications and OpenShift API.
  • A compromised DHCP server can poison IP assignments — redirecting traffic to attacker-controlled systems.

Placing these systems in their own VLAN means: a compromised VLAN 60 application VM cannot directly reach the Domain Controllers on port 389/636 — it must be routed through the leaf SVI and inspected by the Palo Alto firewall. The attack surface is minimised to only systems explicitly permitted by firewall rules.

Key IP assignments in VLAN 70
10.70.0.10dc01.novacrestbank.local — primary DC, DNS, DHCP, NTPStatic — never DHCP
10.70.0.11dc02.novacrestbank.local — secondary DC, DNS failoverStatic — never DHCP
10.70.0.20rootca.novacrestbank.local — offline Root CAPowered OFF after issuing issuing CA cert
10.70.0.21issuingca.novacrestbank.local — online Issuing CAAlways on — issues day-to-day certificates
10.70.0.30–99Reserved for management serversMonitoring, Ansible, jump hosts
Why the Root CA VM is in VLAN 70 but powered off

The offline Root CA design is a PKI best practice: the Root CA signs the Issuing CA certificate, and is then powered off and never connected to the network again. Even if the Issuing CA is compromised, an attacker cannot use the Root CA because it is offline. It has an IP in VLAN 70 for the brief periods when it must be powered on for maintenance — but its IP never appears in routing tables or firewall logs during normal operations.

How VLAN 70 connects to the OpenShift and VMware guides

VLAN 70 is the foundation that every other component in NovaCrestBank depends on. The OpenShift AD LDAP IDP (Guide 3) authenticates against dc01 on VLAN 70. The AD CS Issuing CA (VLAN 70) signed all the TLS certificates referenced throughout the OpenShift guide series. The NTP hierarchy starts at dc01 — all ESXi hosts, all Nexus switches, all Palo Alto interfaces synchronise their clocks from dc01. The DHCP server on dc01 provides IP leases to VMs across VLANs 10, 20, 50, 60, and 70 via the DHCP relay configured on the leaf SVIs (Guide 5, Section 10).

Cisco Fabric · Guide 6 · Section 12
Subnet sizing — why every VLAN is a /24
The decision behind giving every VLAN the same subnet size — and when a different size would be appropriate
The most common question about this design
"Why does vSAN (VLAN 25) get a /24 with 254 addresses when you only have 5 ESXi hosts? That is 249 wasted addresses. Why not a /29 (6 usable addresses)?"

The /24 argument — why uniformity beats efficiency here

Decision: all production VLANs are /24
Why not use appropriately-sized subnets (/28, /29, /30) for each VLAN based on its current device count?
Three reasons — all of them about operational cost over time:

1. Growth. NovaCrestBank is planning to expand from 5 ESXi hosts to 20+ over the next 3 years. A /29 for vSAN (6 usable IPs) would need to be renumbered at host 7 — a maintenance window affecting all storage. A /24 accommodates 254 hosts without ever renumbering. The cost of a /24 (a few unused IP addresses in private space) is vastly smaller than the operational cost of renumbering a VLAN.

2. Firewall rule simplicity. A firewall rule that says "permit 10.25.0.0/24" is permanent and correct regardless of how many hosts are added. A firewall rule for a /29 would need to be updated (or replaced with a /28 or /27) as the network grows. Every firewall rule change is a change management event — a risk, a ticket, an approval. Avoiding that is worth the "wasted" addresses.

3. Consistent routing. Every route in the BGP routing table is a /24. The spine switches always see clean, identically-sized subnets for every VLAN. Summarisation is trivial: all VLAN subnets can be summarised as 10.10.0.0/13 (covering 10.10-70.x.x). No variable-length masks to reason about.

When a different subnet size IS appropriate

ScenarioAppropriate maskReasoning
BGP point-to-point links between switches /30 Only 2 hosts needed per link. /30 wastes only 2 addresses (network + broadcast). The 10.0.x.x block uses /30 throughout for underlay links.
Loopback interfaces /32 A loopback is a single address — a /32 is the only correct size. There is no "network" and no "broadcast" for a loopback.
DMZ or internet-facing services /27 or /28 Internet-facing networks often have a hard limit on the number of public IPs assigned by the ISP. /27 (30 usable) or /28 (14 usable) is common for DMZ segments.
Very large VM pools (100+ VMs) /23 or /22 A /24 accommodates 254 VMs. If a single VLAN needs more than 254 IPs, use /23 (510 usable) or /22 (1022 usable) rather than creating two adjacent /24s with duplicate gateways.

Why static IPs for infrastructure and DHCP only for workloads
Every ESXi VMkernel adapter, every switch mgmt0, every Domain Controller, every vCenter — these get static IPs. The reason: these devices are referenced in DNS records, firewall rules, monitoring tools, and security certificates. If a DHCP lease expires and is re-issued with a different IP, every reference breaks simultaneously. The operational cost of a single IP change on a Domain Controller is enormous. Static addressing for infrastructure is not optional — it is mandatory.

Trunk philosophy — least privilege for every port
The single most important trunk configuration principle — and why violating it is a security and performance risk
The most common trunk mistake — and why NovaCrestBank avoids it
The tempting shortcut: configure every trunk port as switchport trunk allowed vlan all. One command. Done. Every VLAN on every port. The switch CLI even suggests this as a default. This is wrong in a production data center. It means: vSAN replication traffic flows on management switch ports. OOB management traffic (if accidentally trunked) can reach production switches. A VM in VLAN 60 could potentially receive broadcasts from VLAN 70 (Windows Infrastructure). NovaCrestBank never uses "allow all" on any trunk port.

The least-privilege trunk principle

Design principle — trunk configuration
What is the rule for deciding which VLANs appear on a trunk port?
A port carries only the VLANs that the device connected to that port legitimately needs.

For an ESXi management NIC (vmnic0/1): this NIC handles ESXi management and Windows VM traffic. It carries VLAN 10 (management) and VLAN 70 (Windows VMs). It does NOT carry VLAN 25 (vSAN), VLAN 50 (OCP), or VLAN 60 (VM Production) — these travel on different NIC pairs.

For an ESXi vSAN NIC (vmnic4/5): this NIC exclusively handles storage replication. It carries VLAN 25 (vSAN) and VLAN 30 (NFS). Nothing else.

The test for any VLAN on any port: "Does the device connected to this port have a legitimate reason to send or receive traffic on this VLAN?" If the answer is no — even if technically harmless — the VLAN is not on the port.

Why per-NIC VLAN assignment was chosen over a single all-VLAN trunk

NovaCrestBank's approach: per-NIC VLAN assignment

  • vmnic0/1 carries only VLANs 10, 70
  • vmnic2/3 carries only VLANs 20, 50, 60
  • vmnic4/5 carries only VLANs 25, 30
  • vSAN congestion cannot affect management
  • A bug in ESXi vSAN code cannot send rogue packets into VLAN 10
  • Monitoring is per-NIC-pair — easy to correlate interface utilisation to workload type

Alternative rejected: single trunk with all VLANs

  • All VLANs on all NICs (single bond of all 6 NICs)
  • Simpler cabling — fewer port channels
  • But: vSAN saturates the shared NICs → management drops
  • But: no hardware separation between storage and management traffic
  • But: ESXi scheduler must make real-time traffic prioritisation decisions
  • But: much harder to troubleshoot performance issues

Alternatives considered and rejected
The design decisions that were NOT made — and understanding why the rejected option is wrong helps you defend the chosen option

Alternative 1 — Combine vSAN and NFS into one storage VLAN

Rejected alternative
Why not put vSAN and NFS on the same VLAN (e.g., "Storage" VLAN 25, 10.25.0.0/24)?
Rejected because of conflicting MTU and QoS requirements. vSAN requires MTU 9000. NFS benefits from MTU 9000 but can function at 1500. More importantly: vSAN and NFS have different latency tolerance. vSAN responds poorly to latency spikes — a vSAN heartbeat timeout can trigger disk rebuild events. NFS is more tolerant. Separate VLANs allow separate QoS policies. Additionally, separating them means a NetApp firmware upgrade (which might require a brief NFS service interruption) is visibly separate from vSAN traffic — clear operational boundaries. Finally, troubleshooting is cleaner: a Wireshark capture on 10.25.0.0/24 shows only vSAN traffic.

Alternative 2 — Combine management and OOB into one VLAN

Rejected alternative
Why not put iLO, mgmt0, and ESXi management in a single "management" VLAN on the production leaf switches?
Rejected because it defeats the entire purpose of OOB. If iLO were on VLAN 10 (ESXi management), and the leaf switches had a configuration error that broke VLAN 10 forwarding, iLO access would be lost simultaneously. You would have no way to power-cycle the server or connect to its console to diagnose what happened to the leaf. OOB only provides its value if it is physically independent. Combining it with production management creates a management network with a single point of failure — exactly what OOB is designed to prevent.

Alternative 3 — Put OpenShift nodes in VLAN 60 (VM Production)

Rejected alternative
OpenShift nodes are VMs. Why not put them in VLAN 60 (VM Production) with other VMs?
Rejected for three reasons: security policy granularity, BGP/routing clarity, and DNS separation. OCP generates unique traffic patterns (Geneve overlay UDP 6081, etcd replication on port 2379/2380, API server on port 6443) that deserve their own firewall policy category. Firewall rules for "OCP traffic" are distinct from "general VM traffic" — mixing them makes the rules harder to reason about and audit. From a routing perspective, advertising 10.50.0.0/24 (OCP nodes) as a distinct prefix allows fine-grained routing policies on the Palo Alto — e.g., "traffic to 10.50.0.200 (API VIP) from external sources goes through deep packet inspection." With OCP in VLAN 60, this requires per-IP rules instead of per-subnet rules.

Alternative 4 — Use VLAN 100, 200, 300 instead of 10, 20, 25

Rejected alternative
Some designs use VLAN IDs like 100 (management), 200 (vMotion), 300 (vSAN). Why use 10, 20, 25?
Rejected because the third-octet encoding convention only works with smaller VLAN IDs. VLAN 10 → subnet 10.10.0.0/24. VLAN 100 → subnet 10.100.0.0/24. The 100 pattern still works, but the IDs are larger and less readable. VLAN 20 (vMotion) and VLAN 25 (vSAN) are deliberately close together — the teens and twenties group represents "VMware infrastructure" VLANs. VLAN 40 (OOB) and VLAN 50 (OCP) follow logically. The grouping is intentional: 10–39 = VMware infrastructure, 40 = management, 50–70 = workloads. This mental grouping makes the design intuitive for anyone who learns the pattern.
