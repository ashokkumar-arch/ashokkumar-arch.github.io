DHCP delivers four things — not just an IP address
When a VM sends a DHCP discovery broadcast, the DHCP server responds with four pieces of information that together give the VM everything it needs to communicate: (1) an IP address, (2) a subnet mask, (3) a default gateway, and (4) DNS server addresses. Without all four, the VM can have an IP but still cannot resolve names, reach other subnets, or access the internet. DHCP options are what carry items 2, 3, and 4 alongside the IP address.

Where DHCP runs in NovaCrestBank

DHCP runs as a Windows Server role on DC01 (10.70.0.10). The same server that is the primary Domain Controller is also the primary DHCP server — this is standard practice for small-to-medium Windows environments. The DHCP role is added to DC01 (no new VM needed). DHCP failover is then configured to DC02 so if DC01 is unavailable, DC02 takes over lease issuance automatically.

Why DHCP runs on the DC — not on a dedicated server
Some designs run DHCP on a dedicated server separate from the Domain Controllers. Why combine them on DC01?
Because for NovaCrestBank's scale (5 ESXi hosts, ~50 VMs initially), the DHCP load is trivially small and does not justify a dedicated VM.

DHCP is one of the lowest-resource network services. At NovaCrestBank's scale, DC01 handles perhaps 50–100 DHCP leases. Each lease renewal is a single UDP packet exchange. The total DHCP load on DC01's CPU and RAM is unmeasurable in practice.

The historical reason for separating DHCP from DCs was that a poorly configured DHCP server could hand out wrong gateway or DNS addresses — potentially isolating machines from the domain and causing authentication failures. This risk is mitigated by: (1) careful scope configuration (done in this guide), (2) DHCP authorisation in AD (only authorised servers can issue leases — configured in Section 07), and (3) DHCP failover to DC02 (so a DC01 failure does not cause DHCP outage).

DHCP concepts — scope, pool, exclusion, reservation, option, lease
The vocabulary the DHCP Manager uses — understanding every term before the wizard asks for it
Scope
What is a DHCP scope?
A scope is a defined range of IP addresses that the DHCP server can assign for a specific subnet. In NovaCrestBank, each VLAN that uses DHCP gets its own scope. The VLAN 60 scope covers 10.60.0.200–10.60.0.254. The VLAN 10 scope covers 10.10.0.200–10.10.0.254. A scope must match the subnet of the VLAN it serves — you cannot assign VLAN 10 addresses to VLAN 60 clients. Each scope also carries scope-specific options (like the default gateway) that differ per VLAN.
Pool
What is the DHCP pool?
The pool is the range of IP addresses within a scope that are available for assignment. If a scope covers 10.60.0.1–10.60.0.254, and you exclude 10.60.0.1–10.60.0.199 (the static address range), the pool is 10.60.0.200–10.60.0.254 — 55 addresses available for dynamic assignment. The pool shrinks as leases are issued and grows back as leases expire.
Exclusion
What is an exclusion and why is it essential?
An exclusion is a range of IPs within the scope that DHCP will never assign — protecting addresses that are statically configured on specific devices.

Example: VLAN 10 (10.10.0.0/24) has ESXi hosts statically configured at 10.10.0.11–10.10.0.15 and vCenter at 10.10.0.100. If no exclusions were set, the DHCP scope for VLAN 10 (10.10.0.1–10.10.0.254) could assign 10.10.0.11 to a new VM — creating a duplicate IP with esxi01. The duplicate IP causes network errors on both the ESXi host and the new VM. Exclusions prevent this by telling the DHCP server "never assign these addresses — they belong to static devices."
Reservation
What is a DHCP reservation and how does it differ from a static IP?
A reservation ties a specific IP address to a specific MAC address within the DHCP system — the device always receives the same IP via DHCP, without being manually configured with a static IP in its network adapter settings.

The benefit: the device is configured to use DHCP (simpler, works everywhere). The DHCP server recognises its MAC address and always issues the same IP. If the device is reimaged or replaced, the new device gets the same IP as long as you update the MAC address in the reservation. Without reservations, the same goal requires manually configured static IPs on the device — which is harder to track and update centrally.
Lease duration
What is a lease and how long should it be?
A lease is the time period for which a DHCP assignment is valid. When a lease expires, the client must renew it (or gets a new IP).

Short leases (1–4 hours): good for environments with rapid VM turnover (development, testing). The DHCP server quickly reclaims IPs from VMs that have been deleted. However, frequent renewals increase DHCP server load.

Long leases (8 days — the Windows default): good for stable environments where VMs run for weeks without being recreated. Fewer DHCP renewals. Risk: if a VM is deleted, its IP is "stuck" for 8 days before DHCP can reassign it.

NovaCrestBank uses: VLAN 60 (VM production) = 8 days (stable long-running VMs). VLAN 50 (OCP nodes) = 8 days (OCP nodes are not frequently recreated). VLAN 10/20/70 = 8 days (these are managed infrastructure with stable VMs).

DHCP relay — how cross-VLAN delivery works
The most important concept in DHCP for a multi-VLAN network — and why it was already configured in the Cisco guides
The fundamental DHCP problem in a multi-VLAN network
DHCP discovery packets are broadcast packets — they are sent to 255.255.255.255 (all devices on the local network segment). Routers do not forward broadcasts between VLANs. This creates a problem: a VM on VLAN 60 (10.60.0.0/24) broadcasts a DHCP discovery, but the DHCP server is on VLAN 70 (10.70.0.0/24). The router (leaf switch) would normally drop the broadcast at the VLAN boundary — the DHCP server never receives it, and the VM never gets an IP.

How DHCP relay solves this

The solution is a DHCP relay agent configured on the router (the Cisco leaf switch SVIs). The relay agent intercepts DHCP broadcasts from clients on its directly-connected VLANs and converts them into unicast packets addressed directly to the DHCP server (10.70.0.10). The DHCP server receives the unicast packet, sees which subnet the client is on (included in the relay packet header), selects the matching scope, and sends a unicast reply back through the relay agent. The relay agent delivers the reply to the client.

This is already configured — in Cisco Guide 5
The Cisco guides already configured ip helper-address. Does anything need to be done on the network side?
No — the Cisco relay agent (ip helper-address 10.70.0.10) is already configured on every leaf SVI in Guide 5. Every VLAN SVI on every leaf switch has the helper address pointing to DC01. When this guide creates and activates the DHCP scopes, the relay agent will immediately start forwarding DHCP requests from clients in those VLANs to DC01. The network configuration and the Windows DHCP configuration must both exist — neither works without the other. The Cisco side was done first (in Guide 5) and is waiting for the Windows side (this guide).

The relay path — step by step for a VLAN 60 VM

1. VM on VLAN 60 boots, sends DHCP Discover broadcast → 255.255.255.255:67
2. leaf01 VLAN 60 SVI (10.60.0.1) receives the broadcast (ip helper-address active)
3. leaf01 converts broadcast → unicast to 10.70.0.10:67
4. Adds GIADDR field = 10.60.0.1 (the relay agent's IP — tells DHCP which subnet)
5. DC01 DHCP service receives the unicast packet
6. DC01 checks GIADDR (10.60.0.1) → matches VLAN 60 scope (10.60.0.0/24)
7. DC01 selects an IP from the VLAN 60 pool (e.g., 10.60.0.210)
8. DC01 sends DHCP Offer back to leaf01 (10.60.0.1) via unicast
9. leaf01 forwards the offer to the VM on VLAN 60
10. VM sends DHCP Request → leaf01 relays → DC01 sends ACK → VM is configured

Why the exclusion covers .1 through .199 — not just .1 through .15
In VLAN 10, only .1–.3 are SVIs, .11–.15 are ESXi hosts, and .100 is vCenter. Why exclude .1–.199 (196 addresses) when only a handful are actually used?
Because the exclusion is not just protecting currently-used addresses — it is reserving the entire static address range for future growth.

The IP assignment convention from Cisco Guide 6 established that .1–.199 are the static address range (gateways at .1/.2/.3, infrastructure at .10–.99, VIPs at .100–.199) and .200–.254 are the dynamic DHCP range. This is a permanent architectural decision, not a reflection of current usage.

If only the currently-used addresses were excluded (say .1–.3, .11–.15, .100), the DHCP pool would include addresses like 10.10.0.50 — currently unused but reserved by convention for a future infrastructure device. If DHCP ever assigned 10.10.0.50 to a random VM and then a new infrastructure device was later configured at 10.10.0.50, the duplicate IP conflict would be immediately painful and confusing. The exclusion protects the entire static range permanently, so the convention can always be relied upon.

DHCP authorisation in Active Directory — why it is mandatory
Understanding what AD DHCP authorisation does and verifying it completed correctly
Why DHCP must be authorised in Active Directory
DHCP authorisation sounds like a formality. What actually happens if a DHCP server is not authorised?
An unauthorised DHCP server in an AD domain does not issue leases at all — it detects the AD domain and refuses to start serving because it knows it could be a rogue server.

This is a security mechanism Microsoft introduced to prevent "rogue DHCP" attacks. The scenario it prevents: someone connects a home router (which has a built-in DHCP server) to the corporate network. The router starts issuing IPs with its own gateway (pointing to itself) and its own DNS (pointing outside the corporate network). Corporate machines receive these leases and lose access to internal resources while potentially sending traffic through an attacker-controlled device.

Windows DHCP Server prevents this by checking Active Directory at startup: "Is this server listed as an authorised DHCP server for this domain?" If yes — start serving. If no (or if the server cannot contact AD at all) — remain silent. An unauthorised DHCP server (like a home router) cannot be registered in AD, so it stays silent.

The post-install wizard (Section 06) completed the authorisation. To verify: in DHCP Manager, the server name should show a green upward arrow (▲) icon rather than a red downward arrow (▼). The red arrow means "not authorised."

Why start from .1 not .200 when the pool is .200–.254
The DHCP pool is .200–.254. Why define the scope as .1–.254 and then add an exclusion for .1–.199 rather than just defining the scope as .200–.254?
Because the scope represents the entire subnet — not just the DHCP portion. The exclusions within the scope protect the static range.

If you define the scope as 10.10.0.200–10.10.0.254, DHCP Manager treats this as a scope covering only 55 addresses. The subnet mask becomes meaningless — DHCP does not know that 10.10.0.0/24 is the full subnet. More importantly, the scope description and options (gateway, DNS domain) must reference the correct subnet for the DHCP relay to work. The relay agent (on the leaf switch) sends DHCP requests with GIADDR=10.10.0.1 (the SVI IP). The DHCP server matches this against scopes whose IP range contains 10.10.0.1. If the scope only covers .200–.254, 10.10.0.1 falls outside the scope range and DHCP may not match it correctly on some Windows Server versions.

Defining the scope as the full subnet (.1–.254) and using exclusions for the static range is the architecturally correct approach.

Why the gateway is 10.10.0.1 — not 10.10.0.2 or 10.10.0.3
There are three VLAN 10 SVIs (.1, .2, .3 on leaf01, leaf02, leaf03). Which one is the default gateway for DHCP clients?
.1 (leaf01) is the primary gateway — set in DHCP Option 003.

The vPC design ensures that if a DHCP client sends its first packet to 10.10.0.1 and leaf01 is unavailable, leaf02 intercepts and processes the packet via the peer-gateway feature (configured in Cisco Guide 5). So setting .1 as the gateway provides a single, memorable gateway address while the vPC infrastructure handles redundancy transparently.

Setting multiple gateways (10.10.0.1 and 10.10.0.2) in Option 003 is technically possible but creates asymmetric routing — some clients use .1 as their gateway, some use .2. When traffic between two clients on the same VLAN takes different gateway paths, return traffic can arrive on a different path from what was expected, which can cause issues with stateful firewall sessions on the Palo Alto. Single gateway, vPC handles redundancy — this is the correct design.

OCP nodes get DHCP but their IPs must be predictable — use reservations
OCP nodes need stable IPs for the cluster to function correctly. Should they use DHCP or static IPs?
OCP nodes should use DHCP with reservations — not fully static IPs on the VMs, and not fully dynamic DHCP.

During OCP installation, the install-config.yaml specifies the expected IPs for each node (master01=10.50.0.11, master02=10.50.0.12, etc.). If nodes receive random IPs from DHCP, the installation fails because the installer cannot find the nodes at their expected addresses.

The solution is DHCP reservations (configured in Section 20): each OCP node VM has its MAC address mapped to a specific IP in the DHCP server. The node uses DHCP (simpler, consistent), always gets the same IP (because of the reservation), and the install-config.yaml can specify the expected IPs. Infrastructure is managed centrally in DHCP rather than scattered across individual VM network adapter configurations.

Windows Server · Win-02 · Section 14
Why VLAN 25, 30, and 40 have no DHCP scope
Not every VLAN needs DHCP — and the reason for each exclusion is a design decision worth understanding
VLAN 25 — vSAN — no DHCP scope
Why does the vSAN network have no DHCP scope?
Because every device on VLAN 25 has a statically-assigned IP that is part of the vSAN configuration — no dynamic assignment is ever appropriate here.

The vSAN VMkernel adapters (vmk2 on each ESXi host) are configured in vCenter with static IPs: 10.25.0.11 through 10.25.0.15. vSAN uses these IPs for all replication and heartbeat traffic. If a vSAN VMkernel adapter received a DHCP address instead of its configured static IP, vSAN would fail to form its cluster — it would try to use 10.25.0.11 (its configured identity) but the actual IP would be something from a DHCP pool, causing all vSAN communication to fail silently. Additionally, a DHCP address could change if the lease expired and was not renewed (e.g., after a long maintenance window). vSAN absolutely requires stable, permanent IPs. Static configuration is the only correct approach.
VLAN 30 — NFS — no DHCP scope
Why does the NFS network have no DHCP scope?
Same reason as vSAN: NFS VMkernel adapters and NetApp NFS LIFs have statically assigned IPs that must never change.

The ESXi NFS VMkernel adapters (vmk3) are configured in vCenter with static IPs (10.30.0.11–10.30.0.15). The NetApp AFF A400 NFS LIFs are at 10.30.0.100 and 10.30.0.101 — configured on the NetApp controller. If the NetApp LIF IP changed due to a DHCP lease, every ESXi host's NFS datastore mount would break simultaneously — all VMs using that datastore would see disk errors. The consequences in a banking environment would be severe. NFS IPs are permanent and static. No DHCP scope needed.
VLAN 40 — OOB Management — no DHCP scope
Why does the OOB management network have no DHCP scope?
Because the OOB network carries critical management access — every device on it must have a known, permanent IP address.

The OOB network (10.40.0.0/24 on the Catalyst 9200 switch) connects iLO ports, mgmt0 interfaces, the Palo Alto management port, and edge router management ports. Every one of these is statically configured: iLO at .31–.35, spine mgmt at .10–.11, leaf mgmt at .20–.22, Palo Alto at .50, edge routers at .51–.52. These are the addresses engineers use during a 3 AM outage to access devices whose production network may be broken. If DHCP served VLAN 40 and a DHCP server became unavailable during an outage (which is when you need OOB the most), devices trying to renew their leases would lose their IPs — making the OOB network unavailable at the exact moment it is needed. Static IPs on OOB — always.
Windows Server · Win-02 · Section 15
DHCP options — what they are and the three levels
Options are the configuration data delivered alongside the IP address — understanding the three levels prevents confusion about which options apply where
The three levels of DHCP options — server, scope, and client
DHCP Manager shows "Server Options" and "Scope Options" for each scope. What is the difference and which takes priority?
Server Options — applied to every client from every scope on this DHCP server. Use for options that are the same across all VLANs: DNS servers (10.70.0.10, 10.70.0.11 — the same for all VLANs), DNS domain name (novacrestbank.local — the same everywhere), NTP servers (10.70.0.10, 10.70.0.11 — the same everywhere).

Scope Options — applied only to clients in a specific scope. Use for options that differ per VLAN: the default gateway (Option 003) differs per VLAN — VLAN 10 clients get 10.10.0.1, VLAN 60 clients get 10.60.0.1. If you set Option 003 at the server level with one gateway, all clients get the same gateway regardless of VLAN — which breaks routing.

Priority (if the same option is set at multiple levels): Client reservation options override scope options, which override server options. In NovaCrestBank's design, the gateway (Option 003) is set per scope and the DNS/NTP options are set at server level — no conflicts.

The practical rule: Set options at the highest level that applies correctly. Everything the same across all scopes → server level. Per-VLAN differences → scope level.

Windows Server · Win-02 · Section 21
DHCP failover — why a single DHCP server is not enough
Understanding what happens when the only DHCP server goes offline — and why the answer is unacceptable for a bank
What happens to existing leases when DHCP fails
If DC01 (the DHCP server) goes offline: existing DHCP clients with valid, non-expired leases continue to function normally — they already have their IP, gateway, and DNS configuration and do not need the DHCP server for anything until their lease expires. However, new VMs that power on while DHCP is down cannot get an IP address — they get an APIPA address (169.254.x.x) and cannot communicate on the managed network. Existing leases at 50% of their duration start attempting renewal — if DHCP is still down, they will eventually lose their lease at expiry. With 8-day leases, an outage of several hours has no impact on existing clients. A multi-day outage starts causing problems.
Why configure DHCP failover to DC02
With 8-day leases, a short DHCP outage is tolerable. Why add complexity with failover?
Because in a banking environment, "tolerable for a short time" is not an acceptable design posture for any critical service — and new VM deployments during a DHCP outage are a real operational problem.

Scenario: DC01 needs an emergency firmware update. It must be shut down. During the 30-minute maintenance window, the OpenShift team deploys three new OCP worker nodes for a capacity expansion. The workers boot and request DHCP — no server responds. The workers get APIPA addresses. The OCP deployment fails. The team spends time troubleshooting. Eventually someone realises DHCP is down. This scenario is preventable with DHCP failover.

Windows Server DHCP failover is configured once, requires no manual intervention during failover, and adds no ongoing operational overhead. The cost of configuring it is 10 minutes. The benefit is a DHCP service that survives DC01 downtime without any impact. This is the correct choice for a production banking environment.

Hot Standby for NovaCrestBank — not Load Balance
Load balance distributes the DHCP load. Why choose Hot Standby (which puts all load on DC01)?
Because NovaCrestBank's DHCP load is trivially small and the operational simplicity of Hot Standby outweighs any load distribution benefit.

Hot Standby advantages for NovaCrestBank:
— All leases are issued by DC01. The DHCP lease database is simple: one authoritative server, DC02 is a pure standby. Troubleshooting a lease problem involves looking at only one server's database.
— The address pool on DC01 is almost fully available (5% reserved for DC02). For small pools (55 addresses), a 50/50 load-balance split would give each server only 27 addresses — if one server's 27 are all leased, new clients cannot get an IP even though the other server has free leases.
— Failover is automatic: if DC01 is unreachable for more than the "Maximum Client Lead Time" (MCLT — 1 hour by default), DC02 takes over the full pool and starts issuing leases. No manual intervention.

Load Balance advantages (not applicable here):
— Distributes DHCP load evenly — useful if DHCP traffic is significant (thousands of requests per second). Not relevant for 55-address pools.
— Both servers are active, so failover is instantaneous rather than requiring an MCLT wait. This matters only in environments where clients are very time-sensitive to DHCP delays.
