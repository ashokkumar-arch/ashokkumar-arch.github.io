Windows Server · Win-05 · Section 02
How Group Policy works — processing order, precedence, and the LSDOU rule
The foundational concept before any GPO is created — understanding processing order prevents misconfiguration
LSDOU — the four levels of Group Policy application
A domain computer receives GPOs from multiple levels simultaneously. Which one wins when they conflict?
Group Policy applies in LSDOU order: Local → Site → Domain → Organizational Unit. Higher levels (OU) override lower levels (Domain) when they conflict. Within a level, GPOs are processed in link order with higher-numbered links applied last (and therefore winning).

Local: Settings in the local Group Policy on each machine. Applied first — lowest precedence.
Site: GPOs linked to the AD site. Applied second. (NovaCrestBank has one site — NovaCrestBank-London-DC1 — so site GPOs affect all machines.)
Domain: GPOs linked to novacrestbank.local. Applied third. The Default Domain Policy lives here.
Organizational Unit: GPOs linked to the specific OU containing the computer or user. Applied last — highest precedence. A GPO on OU=Domain Controllers overrides the same setting in a Domain-level GPO.

The practical implication for NovaCrestBank: the security baseline GPO at the domain level sets the default for everything. Server-specific settings in the Servers OU GPO override domain-level settings for server machines. DC-specific settings in the Domain Controllers OU GPO override domain-level settings for DCs. This layering allows a single consistent baseline with targeted overrides — no duplication needed.
Computer Configuration vs User Configuration — what each applies to
Every GPO has both "Computer Configuration" and "User Configuration" sections. What is the difference?
Computer Configuration settings are applied when the machine starts up and are tied to the computer account — they apply regardless of which user logs in. User Configuration settings are applied when a user logs in and follow the user — they apply regardless of which machine the user logs into.

Security baseline settings (firewall rules, audit policy, SMB signing, UAC) belong in Computer Configuration — they must apply to the machine itself regardless of who is logged in. An attacker who gains physical access to an unlocked server does not need to log in to benefit from missing firewall rules.

User environment settings (mapped drives, desktop backgrounds, software restrictions per user) belong in User Configuration. For NovaCrestBank's server infrastructure, most settings are in Computer Configuration — servers should be consistently hardened regardless of the logged-in administrator's identity.

Why 14 characters minimum — not 8 (Windows default) or 12
The Windows default minimum password length is 7 characters. Why set 14?
Because 14 characters is the NCSC (UK National Cyber Security Centre) minimum recommendation for privileged accounts, and FCA guidance for banking systems expects password controls aligned with NCSC standards. A 14-character minimum with complexity requirements makes brute force attacks computationally infeasible within any realistic attack timeframe.

The maths: a 14-character password using uppercase, lowercase, digits, and symbols from a ~95-character space has 95^14 ≈ 4.9×10^27 combinations. At a theoretical online attack rate of 100 guesses per second (with account lockout, the realistic rate is far lower), cracking a single password would take longer than the age of the universe. Even offline dictionary-based attacks against a captured NTLM hash are computationally expensive against 14+ character complex passwords.

The 90-day expiry is a balance between security (limiting how long a stolen credential is valid) and usability (too-frequent changes cause users to adopt predictable patterns like incrementing numbers). NCSC's most recent guidance actually recommends against mandatory periodic changes in favour of longer passwords — for NovaCrestBank's initial deployment, 90 days is a reasonable starting point aligned with most banking regulatory expectations.
Windows Server · Win-05 · Section 05
Fine-Grained Password Policy — different rules for service accounts
Service accounts cannot expire — but they need stronger passwords. Fine-Grained Password Policies apply different password rules to specific users or groups without changing the domain-wide policy
What a Fine-Grained Password Policy is and why it exists
The Default Domain Policy sets one password policy for the entire domain. Service accounts need PasswordNeverExpires. Why not just tick that on the account directly?
Ticking PasswordNeverExpires directly on an account bypasses all password policy controls for that account — it can then have any password of any length, complexity, or age. Fine-Grained Password Policies (FGPPs) let you define a separate, more restrictive password policy that applies to specific users or groups, with PasswordNeverExpires as one of the policy settings.

For NovaCrestBank service accounts (svc-vcenter-ldap, and future OCP service accounts): they must have non-expiring passwords (because changing them requires updating every system that uses them — if the password expires and is not updated everywhere simultaneously, all dependent services break at midnight). However, they must also have very long, complex passwords — precisely because they never expire, they are higher-value targets for credential theft. A 32-character randomly-generated password that never expires is more secure than a 14-character human-chosen password that expires every 90 days.
PowerShell — DC01 — create Fine-Grained Password Policy for service accounts
# Create the Fine-Grained Password Policy for service accounts
# FGPP is created with New-ADFineGrainedPasswordPolicy
# It must be applied to groups (best practice) or individual users
PS C:\> New-ADFineGrainedPasswordPolicy `
    -Name "NovaCrestBank-ServiceAccount-Policy" `
    -Precedence 10 `
    -MinPasswordLength 32 `
    -PasswordHistoryCount 24 `
    -MaxPasswordAge "0.00:00:00" `
    -MinPasswordAge "0.00:00:00" `

    -LockoutThreshold 5 `

    -LockoutDuration "0.01:00:00" `

Account lockout policy
Configuring the settings that protect against brute-force password attacks — in the Default Domain Policy
Default Domain Policy → Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Account Lockout Policy
Account lockout threshold5 invalid logon attempts
Account lockout duration30 minutes (auto-unlocks after 30 min)
Reset account lockout counter after30 minutes
5 attempts and 30-minute lockout — the banking tradeoff
Why 5 attempts — not 3 (stricter) or 10 (more lenient)?
Because 5 attempts balances security against the operational cost of administrator-reset lockouts from genuine user mistakes, while still making online brute-force attacks infeasible.

At 5 attempts with a 30-minute reset window: an attacker can try 5 passwords every 30 minutes = 10 passwords per hour = 240 per day. To exhaust a 14-character complex password space at this rate would take astronomical time — the lockout policy is effective.

3 attempts: users who fat-finger their password twice then misremember the correct one are locked out frequently, generating a high volume of helpdesk calls and pressure to disable the lockout policy entirely — worse for security than 5 attempts. 10 attempts: gives attackers twice as many guesses before detection, with no meaningful improvement in usability. 5 is the standard for UK financial services.

30-minute auto-unlock: the account unlocks automatically after 30 minutes without administrator intervention. This prevents helpdesk flooding for genuine user lockouts while still blocking automated attacks (30-second attack tools hit the lockout immediately and must wait).

Open the Default Domain Policy → navigate to Account Lockout Policy. Double-click each setting and configure the three values above. Apply and close.

Why "Deny access to this computer from the network" includes "Local account"
The Deny access list includes a built-in group called "Local account." What does this group do and why is it important?
The "Local account" built-in group (introduced in Windows 8.1/2012 R2) contains all local user accounts on a machine. Adding it to "Deny access from network" prevents pass-the-hash attacks — where an attacker who steals a local account's NTLM hash uses it to authenticate to other machines on the network.

Before this setting existed: if the local Administrator account on esxi-vm01 had the same password as on esxi-vm02, an attacker who compromised esxi-vm01 and stole the local Administrator hash could use that hash (without knowing the plaintext password) to authenticate to esxi-vm02, and then every other server with that same local password. This is the core of "pass-the-hash" lateral movement attacks that were responsible for many major breaches.

With "Local account" in the deny list: local accounts cannot authenticate over the network at all — they can only log in at the physical (or RDP) console. Network authentication requires domain accounts. This forces attackers to obtain domain credentials rather than reusing local account hashes — a much higher bar. Domain accounts are logged in AD, subject to lockout policies, and can be immediately disabled centrally.

Audit policy — what to log in a banking environment
The audit settings that create the forensic trail required by FCA, PRA, and PCI-DSS — every sensitive action on every server must be logged
Advanced Audit Policy Configuration vs Legacy Audit Policy
Windows has both "Audit Policy" (simple) and "Advanced Audit Policy Configuration" (detailed). Which one does NovaCrestBank use and why?
Advanced Audit Policy Configuration — it provides subcategory-level control over exactly what is audited, preventing log flooding while ensuring comprehensive coverage of security-relevant events.

Legacy Audit Policy has 9 broad categories (e.g., "Audit account logon events"). Enabling this category generates events for every Kerberos ticket — hundreds per minute per server, filling the Security log with noise. Advanced Audit Policy Configuration has 58 subcategories. NovaCrestBank can enable "Audit Kerberos Authentication Service — Failure" (failed Kerberos auth attempts — suspicious) without enabling "Audit Kerberos Authentication Service — Success" (every successful ticket — too noisy).

This precision is essential for banking SIEM integration. A SIEM ingesting 10,000 events per second from 50 servers cannot distinguish signal from noise without careful audit policy tuning. Advanced Audit Policy gives the team control over signal-to-noise ratio.

Why SMB signing is mandatory in a banking environment
SMB signing adds overhead to file transfers. Why require it for all SMB traffic?
Because without SMB signing, an attacker with network access can perform an SMB relay attack — intercepting an SMB authentication from a client, replaying it to a server, and authenticating as that client — without ever needing to know the password.

An SMB relay attack requires only network access between two machines and a user who can be tricked into connecting to the attacker's machine (via various techniques including LLMNR/NBT-NS poisoning). With SMB signing enforced on both client and server: every SMB packet is signed with the session key. A relayed connection cannot forge valid signatures — the relay attack fails immediately. The overhead of SMB signing on modern hardware (NVMe storage, 25G networking) is negligible — typically less than 2% throughput reduction.

NTLMv2 enforcement

Security Options — LAN Manager Authentication
Network security: LAN Manager authentication level Send NTLMv2 response only. Refuse LM & NTLM
NTLMv2 only — refusing LM and NTLMv1
Windows supports LM, NTLMv1, and NTLMv2. Why refuse the older protocols?
Because LM hashes are trivially crackable (rainbow tables crack them in seconds) and NTLMv1 uses DES encryption that modern hardware breaks in minutes. Both are completely insecure against any capable attacker. Refusing them forces all authentication to NTLMv2 (or Kerberos, which is always preferred).

LM hashes split a password into two 7-character chunks and compute DES independently on each — a 14-character password is no more resistant to LM cracking than a 7-character one. If any machine on the network still accepts LM (Windows 95/98/NT-era compatibility), an attacker can capture LM hashes and crack them offline in seconds with freely available tools.

Setting "Send NTLMv2 response only. Refuse LM & NTLM" ensures: (1) this machine will only send NTLMv2 challenges to authenticate to other servers, (2) this machine will only accept NTLMv2 from connecting clients — it refuses connections using LM or NTLMv1. No Windows machine produced in the last 25 years needs LM or NTLMv1. NovaCrestBank's environment has no legacy clients requiring these protocols.

What OpenShift needs from Active Directory
Understanding the AD integration requirements before creating the accounts — so the accounts are created correctly the first time
How OpenShift integrates with Active Directory
OpenShift is a Linux-based Kubernetes platform. How does it authenticate users against a Windows Active Directory domain?
OpenShift uses LDAP as its identity provider — it connects to Active Directory via LDAPS (port 636) using a service account (the bind account) to search the directory and authenticate users when they log into the OpenShift web console or CLI.

The integration works at three levels:

1. LDAP Identity Provider (IdP): OpenShift's OAuth server queries AD via LDAPS to authenticate users. When a developer logs into the OCP web console with their NOVACRESTBANK username and password, OpenShift sends those credentials to DC01/DC02 via LDAPS and receives confirmation of authentication. The user is then mapped to OCP groups based on their AD group membership.

2. Group sync: OpenShift periodically syncs AD group memberships to OCP groups. An AD group (e.g., "OCP-ClusterAdmins") maps to an OCP group of the same name. When a user is added to the AD group, they automatically gain the corresponding OCP permissions — no OCP-specific permission management needed.

3. Service accounts for applications: Some OCP workloads (CI/CD pipelines, configuration management) need their own AD accounts for accessing domain resources. These are separate service accounts distinct from the LDAP bind account.

The accounts and groups OpenShift needs

AppLocker — application whitelisting for servers
AppLocker restricts which executables can run on servers — a defence-in-depth control that limits what malware can execute even if it reaches a server
What AppLocker does and why it matters for banking servers
What does AppLocker prevent that antivirus does not?
AppLocker prevents execution of binaries that are not on the whitelist — regardless of whether they are known malware. Antivirus detects known malware signatures. AppLocker blocks unknown/zero-day executables because they are not on the approved list.

Scenario: an attacker exploits a web application on a server and uploads a custom reverse shell executable. The executable is new, has no antivirus signature — antivirus does nothing. AppLocker sees an attempt to execute an unsigned binary from a non-whitelisted path and blocks it. The attacker's payload never runs.

AppLocker works by enforcing rules based on: publisher (digitally signed by a trusted certificate), file hash (exact file match), or path (specific folder location). For NovaCrestBank's servers, a publisher-based whitelist (only executables signed by Microsoft, HP, VMware Tools, or other approved vendors can run) combined with a path-based rule (executables in C:\Windows and C:\Program Files are allowed) covers the vast majority of legitimate software while blocking anything dropped by an attacker.

AppLocker rules for NovaCrestBank servers
Configuring AppLocker in the NovaCrestBank-Servers-Baseline GPO — starting with Audit mode before switching to Enforce
NovaCrestBank-Servers-Baseline → Computer Configuration → Policies → Windows Settings → Security Settings → Application Control Policies → AppLocker
  • 1
    Configure AppLocker enforcement — start with Audit Only

    Right-click AppLocker → Properties. The Configure Rule Enforcement dialog opens. Set all three rule types (Executable rules, Windows Installer rules, Script rules) to Audit only initially. Click OK. Audit only logs what would have been blocked without actually blocking anything — essential for testing before enforcement.

  • 2
    Create default rules — right-click Executable Rules → Create Default Rules

    Right-click Executable Rules → Create Default Rules. Three default rules are created automatically: (1) Allow Everyone to run executables in %PROGRAMFILES%, (2) Allow Everyone to run executables in %WINDIR%, (3) Allow Administrators to run all executables. These defaults allow all legitimately installed software to run while giving administrators an escape hatch.

  • 3
    Add a publisher rule for Microsoft-signed executables

    Right-click Executable Rules → Create New Rule → Next → Allow → Publisher. The wizard asks for a reference file. Browse to any Microsoft executable (e.g., C:\Windows\System32\notepad.exe). Set the publisher slider to "Publisher: O=Microsoft Corporation" — allowing any executable signed by Microsoft regardless of version. This covers all Windows system binaries and Microsoft management tools.

  • 4
    After testing in Audit mode for 2 weeks — switch to Enforce

    Review the AppLocker event log (Event Viewer → Applications and Services Logs → Microsoft → Windows → AppLocker → EXE and DLL). Any application that would have been blocked appears in the Audit log as Event ID 8003. Add publisher or path rules for any legitimate application that appears in the log. After 2 weeks with no unexpected blocks, change the enforcement mode to Enforce rules.

Enable the Application Identity service — required for AppLocker enforcement
AppLocker requires the Application Identity service (AppIDSvc) to be running. It is disabled by default. Configure it via GPO: Computer Configuration → Preferences → Control Panel Settings → Services → Application Identity → Set to Automatic start. Without this service running, AppLocker rules have no effect even when set to Enforce mode.
