Guide 5 · Foundation · Section 01
Security landscape & real threats
What you are actually defending against in a banking environment — and which OCP security control stops each threat
Why banking security is different from generic OCP security
NovaCrestBank processes customer financial data, card transactions, and KYC documents. This puts the cluster in PCI-DSS scope. A breach is not just an IT incident — it is a regulatory event with mandatory disclosure, potential fines of millions, and personal liability for executives. Every security control in this guide has a direct link to either a PCI-DSS requirement or a documented threat pattern from real banking breaches.

The threat landscape — what attackers actually do

ThreatHow it manifests in OCPControl that stops itSeverity
etcd data theft Attacker with node access reads etcd files directly — all Secrets in plain text including DB passwords, TLS private keys, payment API tokens HSM-backed etcd encryption at rest Critical
Container escape Exploit in a containerised app breaks out to the node OS, gaining access to other pods and the kubelet ACS runtime detection + SCCs (no privileged containers) Critical
Lateral movement Compromised pod in one namespace connects directly to pods in another namespace — e.g. a payment processing pod reaching a database it should never talk to NetworkPolicy default-deny + namespace isolation Critical

Privilege escalation Pod running as root or with excessive capabilities gains control of the node SCCs (restricted), ACS policy enforcement High

NovaCrestBank security components — the full picture

1
Thales Luna HSM
FIPS 140-2 Level 3 hardware. Stores and never releases the etcd encryption master key. All cryptographic operations (key wrapping, key derivation) happen inside tamper-evident hardware. Already physically present in your draw.io diagram — 2 HSMs in HA pair.
2
Red Hat ACS
Runtime security platform. Scans every image for CVEs before deployment. Monitors every running container for anomalous behaviour (shell spawned, network connection opened, file written to /etc). Enforces policies — block, alert, or scale-to-zero on violation.
3
NetworkPolicy
Kubernetes-native firewall at the pod level. Default-deny everything. Explicit allow only the exact communication paths each namespace needs. A compromised pod in the kyc-service namespace cannot reach payment-processing — the OVN-Kubernetes kernel enforces it.
4
API Audit Logging
Every call to the Kubernetes API server is recorded with: who made it (user identity from AD), what they did (verb + resource), what the outcome was (allowed/denied), and when. Forwarded to your SIEM. PCI-DSS Req 10 compliance evidence.
5
Security Context Constraints
OpenShift's pod admission control. Enforces: no root containers, no privilege escalation, no host network access, read-only root filesystem where possible. Applied by default — developers must explicitly request elevated permissions and get them approved.

Guide 5 · Thales Luna HSM · Section 03
What the HSM does — and why software encryption is not enough
The difference between aescbc (software) encryption and HSM-backed encryption — a critical distinction for PCI-DSS
The problem with software-only encryption
In Guide 3, you enabled etcd encryption with type: aescbc. This encrypts etcd data, but the encryption key is itself stored in a Kubernetes Secret on the master nodes — in memory and on disk. An attacker who compromises a master node can read the encryption key from the master's filesystem and decrypt all etcd data. The key is only as protected as the host it lives on. This is not acceptable for PCI-DSS where cardholder data encryption keys must be protected to FIPS 140-2 Level 3.

How the Thales Luna HSM changes everything

KEK
Key Encryption Key (in HSM)
The master key — a 256-bit AES key — is generated inside the HSM and never leaves the hardware boundary. Even Thales engineers with physical access cannot extract it. The HSM stores it in tamper-evident hardware that self-destructs on physical intrusion. This is your FIPS 140-2 Level 3 protection.
DEK
Data Encryption Keys (wrapped by HSM)
When OCP needs to encrypt an etcd Secret, the KMS plugin asks the HSM to wrap (encrypt) the DEK using the KEK. The wrapped DEK is stored alongside the data. To decrypt: OCP sends the wrapped DEK to the HSM, HSM unwraps it using the KEK (inside the hardware boundary), returns the unwrapped DEK for a single operation. The KEK never crosses the HSM boundary.
KMS
KMS Plugin (on master nodes)
A small gRPC server running on each master node that bridges the Kubernetes KMS API (what OCP uses) to the PKCS#11 interface of the Thales Luna HSM. OCP calls the KMS plugin; the KMS plugin calls the HSM via the Luna client software. The plugin never sees the KEK — only the HSM does.
Your NovaCrestBank HSM setup from the draw.io diagram
You have 2 × Thales Luna Network HSM 7 in an HA pair, already physically installed. FIPS 140-2 Level 3 certified. Connected to the trust zone via the OOB management switch. Each master node will run the Luna Client software and connect to both HSMs simultaneously (HA client). If one HSM is unreachable, the other serves all requests without interruption.

Guide 5 · Thales Luna HSM · Section 05
KMS plugin — HSM-backed etcd encryption
Deploying the KMS plugin DaemonSet on masters and migrating etcd from software encryption to HSM-backed encryption
What we are doing and why the order matters
OpenShift's KMS v2 provider connects to a gRPC KMS plugin running locally on each master. The plugin holds no keys itself — it forwards every encrypt/decrypt request to the Luna HSM via PKCS#11. We must first deploy the plugin as a static pod on the masters, then patch the apiserver to switch from aescbc (software) to KMSv2 (HSM). The transition is seamless — existing data is re-encrypted in the background.

Step 1 — Deploy the KMS plugin as a static pod

master01.ocp.novacrestbank.local — create KMS plugin static pod manifest 
# The KMS plugin is deployed as a static pod — a pod defined by a file in
# /etc/kubernetes/manifests/ that the kubelet starts automatically
# Do this on ALL THREE master nodes

[core@master01 ~]$ sudo cat > /etc/kubernetes/manifests/kms-plugin.yaml << 'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: kms-plugin
  namespace: kube-system
spec:
  hostNetwork: true                        # uses host network — no CNI needed
  containers:
  - name: kms-plugin
    image: registry.novacrestbank.local/infra/luna-kms-plugin:v1.2.0  # internal registry
    command:
    - /luna-kms-plugin
    - --listen-addr=unix:///var/run/kms-plugin/kms.sock
    - --hsm-slot=0                         # slot 0 = ocp-ha-group

    - --hsm-pin-env=HSM_PIN                # read partition PIN from environment
    - --key-label=ocp-etcd-kek             # label for the KEK inside the HSM partition
    env:
    - name: HSM_PIN
      valueFrom:
        secretKeyRef:
          name: hsm-partition-pin
          key: pin
    volumeMounts:
    - name: kms-socket-dir
      mountPath: /var/run/kms-plugin
    - name: luna-config

      path: /etc/safenet

      path: /usr/safenet

Guide 5 · Red Hat ACS · Section 07
What ACS does — runtime security for containers
The three things ACS does that nothing else in OCP does — and why each one matters in a banking environment
The gap that ACS fills
OpenShift has RBAC, SCCs, NetworkPolicy, and audit logging. But none of them can answer: "Is this running container doing something it was not supposed to do?" A pod can pass every admission check and then, 10 minutes after starting, spawn a reverse shell, connect to an external IP, or read files it should not. ACS watches what running containers actually do — in real time — and alerts or blocks when behaviour deviates from what is expected.

The three capabilities of ACS

Image scanning

Every image pulled into the cluster is scanned for CVEs before the pod starts. ACS checks against 15+ vulnerability databases. You define policies: "block any image with a Critical CVE that has a fix available" — ACS enforces it at admission time.

Example: A developer tries to deploy an image with CVE-2024-1234 (Critical, fix available). ACS blocks the deployment automatically and sends an alert.

Runtime detection

ACS deploys a Collector DaemonSet that uses eBPF to watch system calls on every node. It detects: unexpected processes (a shell spawning inside a Node.js container), unexpected file access (/etc/passwd read by an app container), unexpected network connections (outbound to a new external IP).

Example: Container spawns /bin/bash — ACS immediately alerts "Bash process spawned in container" with process args, user, and timestamp.

Compliance & policy

ACS has built-in compliance checks for PCI-DSS, HIPAA, NIST SP 800-190, CIS Kubernetes Benchmark. It continuously evaluates your cluster against these standards and gives you a percentage compliance score with drill-down on every failure.

Example: ACS shows "PCI-DSS compliance: 87%" with a list of 6 failing controls and remediation steps for each one.

ACS architecture in your environment

ComponentWhere it runsWhat it does
Central 3 pods on infra nodes in rhacs-operator namespace The brain of ACS. Stores vulnerability data, policies, alerts, audit history. Provides the ACS web UI. Receives data from Sensors.
Sensor 1 pod per OCP cluster (on infra node) Connects to the OCP API server. Watches for new deployments, policy changes, configuration drift. Enforces admission policies.
Collector DaemonSet — 1 pod per node (all nodes) Uses eBPF to intercept system calls. Captures: process execution, network connections, file access. Sends events to Sensor for policy evaluation.
Scanner 2 pods on infra nodes Scans container images for CVEs. Uses Red Hat's internal vulnerability database plus NVD and other feeds. Integrated with the internal image registry.

Guide 5 · Network Policies · Section 12
Default deny — zero-trust networking
How NetworkPolicy works in OCP with OVN-Kubernetes, and why default-deny is the only acceptable starting point for banking
What happens without NetworkPolicy
By default, every pod in every namespace in OpenShift can reach every other pod in every other namespace — directly, no firewall. A compromised pod in your test-app namespace can make direct TCP connections to your payment-processing database. The OVN-Kubernetes SDN provides routing but not isolation. NetworkPolicy is what adds the isolation — and without it you have zero internal segmentation regardless of what Palo Alto does at the perimeter.

How OVN-Kubernetes enforces NetworkPolicy

OVN-Kubernetes implements NetworkPolicy using OpenFlow rules in the kernel on each node. When a pod tries to make a connection, the kernel checks the OVN rules before the packet leaves the node. If no NetworkPolicy allows it, the packet is dropped — silently, at the kernel level. This is not a software firewall that can be bypassed — it is enforced at the hypervisor level before the packet is even sent on the wire.

Step 1 — Apply a global default-deny to all namespaces

jumphost.novacrestbank.local — apply default-deny to all namespaces

# This blocks ALL ingress and egress traffic to/from pods in the namespace
# by default — you then add explicit allow rules for each required path

[svc-ocp@jumphost ~]$ for ns in banking-backend kyc-service payment-processing; do
  cat << EOF | oc apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: $ns
spec:
  podSelector: {}         # {} = applies to ALL pods in this namespace
  policyTypes:
    - Ingress
    - Egress              # deny both directions
EOF
  echo "Default-deny applied to: $ns"
