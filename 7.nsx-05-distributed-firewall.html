PCI-DSS v4.0 Requirement 1.2.1 and 1.3 — This entire guide is a PCI-DSS control
PCI-DSS Req 1.2.1 requires all network security controls to be documented with business justification. Req 1.3 requires restricting inbound and outbound traffic to what is necessary. The DFW policy in this guide implements both requirements simultaneously — the default-deny rule blocks all traffic not explicitly permitted, and every allow rule in this guide has a documented business justification.
NSX-05 · Section 02
DFW policy model — policies, sections, rules, and evaluation order
Understanding how the NSX DFW evaluates rules before creating any — the evaluation order is what makes the default-deny work correctly
How DFW evaluates rules — top-to-bottom, first match wins
A packet from an OCP node (10.50.0.11) arrives at DC01 (10.70.0.10) on TCP port 636. How does the DFW decide whether to allow or block it?
The DFW evaluates rules from top to bottom, in order of policy section and rule sequence within each section. The first rule that matches the packet's source, destination, and service is applied — and evaluation stops. Subsequent rules are not checked.

For the example packet (OCP node → DC01 on TCP 636):
1. DFW checks the Management Policy rules first — no match (packet is not from a management workstation).
2. DFW checks the Inter-Zone Policy rules — it finds Rule: "OCP Nodes → Infra Services on TCP 636 — ALLOW." This matches. The packet is allowed. Evaluation stops. DC01 receives the LDAPS query.
3. The Default Deny rule is never reached for this packet — the ALLOW rule matched first.

Now a different packet: OCP node → DC01 on TCP 3389 (RDP). Rules 1 and 2 are evaluated — no ALLOW rule matches RDP. The Default Deny rule (last in the table) matches. The packet is silently dropped. DC01 never receives it.

This is why rule order matters critically, and why the Default Deny rule must always be last.

NSX-05 · Section 04
What Security Groups are and why rules use them instead of raw IPs
Security Groups are dynamic membership sets — they automatically include VMs that match a defined criteria. Rules referencing groups remain valid even when VM IPs change.
Security Group vs IP address in a DFW rule — why groups are always preferred
A DFW rule could say "Allow traffic from 10.50.0.11, 10.50.0.12, 10.50.0.13 to 10.70.0.10, 10.70.0.11." Why use Security Groups instead of specific IPs?
Three reasons: dynamic membership, operational clarity, and scale.

Dynamic membership: A Security Group defined as "all VMs on seg-ocp-nodes" automatically includes every VM connected to that segment — today and in the future. When a new OCP worker node is added (10.50.0.34), it immediately inherits all DFW rules referencing NCB-SG-OCP-Nodes without any rule modification. With IP-based rules, every new node would require manual rule updates.

Operational clarity: "NCB-SG-OCP-Nodes → NCB-SG-Infra-Services on TCP 636" reads as a business intent: "OCP cluster nodes are allowed to authenticate against infrastructure services." "10.50.0.11, .12, .13 → 10.70.0.10, .11 on TCP 636" is a list of numbers with no inherent meaning — it does not document the business reason.

Scale: NSX Security Groups are evaluated in the ESXi kernel — they scale to hundreds of members with no performance impact. Adding or removing a member from a group is an O(1) operation; the DFW rules themselves do not change.

PCI-DSS v4.0 Req 1.2.1 — Groups with descriptive names make the business justification for each rule self-documenting, satisfying the requirement to document the purpose of each network security control.

NovaCrestBank's four Security Groups

NCB-SG-Management

Management workstations and jump hosts used by infrastructure administrators. Membership: IP addresses of management workstations (e.g., 10.10.0.100–.110 — the administrator laptop subnet on VLAN 10).

NCB-SG-Infra-Services

Infrastructure service VMs — DC01, DC02, IssuingCA. Membership: Segment: seg-infra-services — all VMs on this segment are automatically members.

NCB-SG-OCP-Nodes

All OpenShift nodes — masters, infra, workers. Membership: Segment: seg-ocp-nodes — any VM connected to this segment is an OCP node and automatically a member.

NCB-SG-Any-External

Represents external traffic arriving via the NSX LB VIPs. In practice this is the "Any" source — the NSX DFW uses "Any" for external sources, and this group acts as a documentation placeholder to make the ingress rule intent clear.

Custom services — port definitions not in the NSX built-in catalogue
NSX has built-in services for common ports (DNS, HTTPS, SSH). NovaCrestBank needs two custom services for KMIP and OCP etcd.
NSX built-in services usedDNS (TCP+UDP 53), HTTPS (TCP 443), HTTP (TCP 80), SSH (TCP 22), LDAP (TCP 389), LDAPS (TCP 636) — no custom definition needed
Custom service needed: KMIPTCP 5696 — Key Management Interoperability Protocol. Not in NSX built-in catalogue. Used for vCenter KMS communication with the Thales Luna HSM appliances.
Custom service needed: OCP-etcdTCP 2379–2380 — etcd client port (2379) and peer port (2380). NSX has no built-in service for this range.
Custom service: OCP-KubeletTCP 10250 — Kubernetes kubelet API. Not in NSX built-in catalogue.

NSX-05 · Section 08
Create the four DFW Policy sections
Creating the policy containers before adding rules — the section order determines rule evaluation order
Why create all sections before adding any rules
Rules could be added to sections as the sections are created. Why create all four sections first?
Because the section order in the DFW determines the evaluation order of rules. If the Default Deny section is accidentally created first (before the Allow sections), publishing the policy would block all traffic immediately — before any allow rules are in place. Creating all sections first, in the correct order, and then adding rules to each section is the safe and correct approach.

Field-by-field guide for the most important rules

Rule 4 — Allow-OCP-LDAPS: how to select LDAPS as the service

In the Service cell, click the edit icon. A service selector panel opens. Type "LDAP" in the search box. NSX shows built-in services: LDAP (TCP 389) and LDAPS (TCP 636). Select LDAPS. Click OK. The service cell shows "LDAPS."

Why log LDAPS but not DNS? LDAPS is an authentication protocol — every successful and failed OCP authentication event generates an LDAPS query. Logging LDAPS allows security monitoring to detect authentication anomalies (brute force, credential stuffing). DNS queries are orders of magnitude more frequent and logging them generates noise that obscures security events. PCI Req 10.2.1 — log authentication events.

Rule 6 — Allow-OCP-KMIP: selecting the custom service

In the Service cell, click the edit icon. In the service selector, type "KMIP". The built-in catalogue has no KMIP entry. Scroll to find the custom services section — NCB-SVC-KMIP appears. Select it. PCI Req 3.7.2 — encryption key operations to/from the HSM must be explicitly permitted and logged.

Rule 8 — Allow-External-to-OCP-Ingress: source is Any, not a group

This rule allows external application traffic (banking users, APIs) to reach OCP nodes on ports 443 and 80 via the NSX LB VIP. The source is "Any" because external clients come from the internet via Palo Alto → Cisco → NSX T0 → T1 → OCP nodes. Their source IPs are not predictable or manageable with a Security Group. The destination is NCB-SG-OCP-Nodes, and the service is limited to HTTPS (443) and HTTP (80) only — all other ports are denied by the Default Deny rule. PCI Req 1.3.1

NSX-05 · Section 11
OCP Internal Policy rules — node-to-node cluster traffic
Six rules in NCB-Policy-OCP-Internal — the intra-cluster communication that OCP requires to function
Why OCP internal traffic needs explicit DFW rules
OCP nodes all communicate with each other on the same segment (seg-ocp-nodes). Without any DFW rules, they could all communicate freely. Why add specific rules instead of just allowing all OCP-to-OCP traffic?
Because the default-deny rule (Section 12) will block all OCP internal traffic that is not explicitly permitted. Without the OCP internal rules, the cluster would fail to form — etcd replication would be blocked, kubelet communication would fail, and pod networking would not establish.

The approach here is precise: allow only the specific ports OCP requires for cluster operation. Any OCP node trying to communicate with another OCP node on a port not in this list is blocked — preventing lateral movement if one OCP node is compromised.

A simpler alternative — "Allow NCB-SG-OCP-Nodes → NCB-SG-OCP-Nodes on Any" — would also work operationally but would permit an attacker who compromised one OCP node to probe all other OCP nodes on any port. The per-port approach limits the blast radius of a compromised node. PCI Req 1.3.2

If OCP nodes form but pods fail to communicate across nodes, the most common DFW cause is a blocked OVN-Kubernetes port. NSX-05 · Section 12
Default Deny Policy — the last rule that catches everything not explicitly permitted
One rule in NCB-Policy-Default-Deny — the most important rule in the entire DFW table
Why the Default Deny must be in its own policy section — and why it must be last
The Default Deny rule could be the last rule in NCB-Policy-InterZone. Why put it in its own separate policy section?
Two reasons: visibility and operational safety.

Visibility: A dedicated "NCB-Policy-Default-Deny" section makes the default-deny intent unmistakably visible. Anyone reviewing the DFW policy sees immediately that there is a named section whose sole purpose is default-deny. If the deny rule were buried as the last rule in a section alongside allow rules, its significance could be missed.

Operational safety: When adding new allow rules in the future, administrators add them to NCB-Policy-InterZone or NCB-Policy-OCP-Internal — never to NCB-Policy-Default-Deny. The dedicated section creates a clear boundary: "rules above this section are allow rules; this section is the deny boundary." This prevents accidental rule placement errors.

Why it must be last: DFW evaluates top-to-bottom. If Default Deny were above the allow rules, it would match all traffic before any allow rule was reached — blocking everything. The default-deny rule must always be the very last rule evaluated.

NSX-05 · Section 13
Publish the DFW configuration
Rules exist in NSX Manager's database — they are not enforced on ESXi hosts until Published. This is the step that pushes rules to all 9 ESXi hosts simultaneously.
What "Publish" does and why it is a separate step from creating rules
Rules have been created in NSX Manager. Why are they not already enforcing? What does clicking Publish do?
NSX Manager maintains a "candidate" version of the DFW policy — changes you make in the GUI are staged but not yet active on ESXi hosts. Publishing pushes the candidate policy to all Transport Nodes (ESXi hosts) simultaneously. This separation exists to allow policy review before enforcement — a batch of rule changes can be staged and reviewed, then published atomically. All rules go live simultaneously, not one at a time.

Why this matters: if rules were applied one at a time as they were created, there would be a window where the Default Deny rule was active before the allow rules — locking out everything temporarily. With staged publishing, all 15 rules are pushed to all 9 ESXi hosts in a single operation. The ESXi hosts update their kernel DFW rule tables and begin enforcing within seconds of the publish operation completing.
