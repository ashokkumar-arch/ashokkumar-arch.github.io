The edge router's unique role — it speaks two languages simultaneously
Every other device in NovaCrestBank speaks a single language: the internal 10.x.x.x address space. The edge router speaks two: the internal network (talking to the Palo Alto and the fabric) AND the public internet (talking to ISP routers using public IP addresses and public AS numbers). It is the translation boundary between NovaCrestBank's private addressing and the public internet. Nothing passes between these two worlds without going through the ASR and through the Palo Alto NGFW.

The complete traffic path — internet to banking application

── Internet ────────────────────────────────────────────────────────────── ISP-A router ISP-B router │ BGP eBGP │ BGP eBGP ▼ ▼ edge01 (ASR 1002-X) edge02 (ASR 1002-X) │ WAN: 203.0.113.1/30 │ WAN: 198.51.100.1/30 │ HA link: 10.0.200.1/30 │ HA link: 10.0.200.2/30 │ Inside: 10.0.100.2/30 │ Inside: 10.0.101.2/30 ▼ ▼ Palo Alto PA-5250 (HA pair) │ Inspects and filters ALL internet-bound traffic │ BGP: receives default route from ASRs │ BGP: advertises internal summary to ASRs ▼ spine01 / spine02 │ Routes to leaf SVIs ▼ leaf01 / leaf02 / leaf03 │ ESXi hosts → VMs → banking applications ▼ Banking application / OpenShift pods

Why two ISPs and two routers — the banking availability argument

Single ISP risk

ISP fibre cuts happen. ISP equipment fails. ISP BGP route leaks occur. In 2023, a major ISP in the UK lost all connectivity for 6 hours due to a BGP misconfiguration. Banks using that ISP as their only internet connection lost online banking access for 6 hours. Regulatory fines for service disruption under PSD2 and FSB guidelines can reach millions. A single ISP is not acceptable for a bank.

Dual ISP + dual router

ISP-A and ISP-B are on completely separate physical circuits, different carrier networks, and ideally different physical entry points to the building. If ISP-A loses all connectivity, edge01 detects the BGP session failure within seconds, withdraws its routes, and edge02 takes over all traffic via ISP-B. RTO (Recovery Time Objective) is under 30 seconds. Customers see at most one failed page load.

Design decision — why separate physical routers, not router redundancy protocols
Why use two physical ASR 1002-X routers rather than a single router with redundant line cards?
Because a software bug, firmware update, or hardware failure in a single chassis takes down both ISP connections simultaneously. With two physical devices: a power supply failure in edge01's chassis does not affect edge02. A firmware update on edge01 does not require edge02 to be taken offline. A BGP misconfiguration applied only to edge01 does not affect edge02's ISP-B peering. Physical separation gives you independent failure domains — the fundamental requirement for true redundancy.

ASR 1002-X — what this router does and how it differs from the Nexus
Understanding the ASR before touching the CLI — what makes it different from everything else in NovaCrestBank's network
ASR 1002-X vs Nexus 9504 — different tools for different jobs
The Nexus switches are optimised for data center east-west traffic — high port count, low latency, VXLAN, vPC. The ASR 1002-X is optimised for WAN edge — internet-scale BGP, NAT, QoS on variable-rate WAN interfaces, encryption (VPN), and carrier-grade features. These two device families share the same CLI conventions (both run IOS-XE or NX-OS derived from the same heritage) but are built for fundamentally different purposes.

ASR 1002-X key characteristics

VRFs Supports multiple VRF instances (same as Nexus). Management interface uses Mgmt-intf VRF. Management access via OOB uses Mgmt-intf VRF — same pattern as NX-OS management VRF. Production routing is in the global (default) VRF.

The three interfaces on edge01 that matter

GE1 — WAN interface (ISP-A side)

Faces the internet. Has a public IP address assigned by ISP-A. Runs eBGP with ISP-A's router. This interface's IP appears in internet routing tables. Pings from the internet hit this IP first.

Security: access-list applied outbound to prevent internal RFC 1918 addresses from being advertised to the internet. IP TTL security and BGP TTL security (GTSM) applied to prevent spoofed BGP packets.

GE2 — Inside interface (Palo Alto side)

Faces the Palo Alto PA-5250's inside port channel. Uses a private /30 address (10.0.100.x). Runs eBGP with the Palo Alto. This is where the edge router hands off internet traffic to the firewall for inspection.

GE3 — HA link (edge01 ↔ edge02)

Direct cable between edge01 and edge02. Uses a private /30 (10.0.200.x). Carries iBGP session between the two routers. Allows the two ASRs to exchange routing information — edge01 knows what ISP-B routes look like via edge02's iBGP advertisement, and vice versa. This is how failover works: if edge01 loses its ISP-A BGP session, it can still forward traffic via edge02's ISP-B path through the HA link, then fall back gracefully.

edge01 — base configuration post-wizard hardening
The IOS-XE post-wizard hardening steps specific to an edge router — additional security controls that are not needed on internal switches
edge01 — SSH via OOB (10.40.0.51) — edge-specific hardening 
[admin@jumphost ~]$ ssh admin@10.40.0.51
Password: NovaCrest$Edge2026!
edge01# configure terminal
edge01(config)#

# ── Loopback — router identity anchor ─────────────────────────────
edge01(config)# interface Loopback0
edge01(config-if)# description edge01 router-ID and BGP source
edge01(config-if)# ip address 10.0.0.51 255.255.255.255
edge01(config-if)# exit

# ── Control plane security — critical on an internet-facing device ──
edge01(config)# ip access-list extended MGMT-ACCESS
edge01(config-ext-nacl)# permit tcp host 10.40.0.1 any eq 22
↑ Only the OOB switch gateway can initiate SSH to this router.
  No direct internet SSH access. Ever.
edge01(config-ext-nacl)# deny ip any any log
edge01(config-ext-nacl)# exit

edge01(config)# line vty 0 4
edge01(config-line)# access-class MGMT-ACCESS in
↑ Apply the access list to VTY lines — only OOB subnet can SSH
edge01(config-line)# transport input ssh
edge01(config-line)# exec-timeout 15 0
edge01(config-line)# logging synchronous
edge01(config-line)# exit

# ── Disable unused services — attack surface reduction ─────────────
edge01(config)# no ip http server
↑ Disable HTTP management (port 80) — use SSH only
edge01(config)# no ip http secure-server
↑ Disable HTTPS management (port 443) — we do not use the router's web UI
edge01(config)# no cdp run
↑ Disable CDP — never run CDP on an internet-facing device.
  CDP advertises device model, IOS version, and IP addresses in cleartext.
  An ISP or on-path attacker could use this for reconnaissance.
edge01(config)# no ip source-route
↑ Disable IP source routing — an old attack vector that allows packets to
  specify their own route through the network. Always disabled on edge devices.
edge01(config)# no ip proxy-arp
edge01(config)# no service finger
edge01(config)# no service tcp-small-servers
edge01(config)# no service udp-small-servers
↑ Legacy diagnostic services — never needed, occasionally exploited

# ── Anti-spoofing on WAN interface — Unicast Reverse Path Forwarding ─
edge01(config)# ip cef
↑ Enable Cisco Express Forwarding — required for uRPF

# ── NTP synchronisation ────────────────────────────────────────────
edge01(config)# ntp server vrf Mgmt-intf 10.70.0.10 prefer
↑ Synchronise to Windows DC (dc01) now that it exists.
  Use Mgmt-intf VRF — NTP travels over OOB management network.
edge01(config)# ntp server vrf Mgmt-intf 10.70.0.11
edge01(config)# clock timezone UTC 0 0

# ── Syslog to dc01 ────────────────────────────────────────────────

WAN interface to ISP-A demarc — GigabitEthernet1
The internet-facing interface — the most security-sensitive interface in the entire data center
The WAN interface faces the untrusted internet — security is non-negotiable
GigabitEthernet1 on edge01 connects to the ISP-A demarc (demarcation) device — the point where the ISP's network ends and NovaCrestBank's begins. Every packet from the entire internet can potentially reach this interface. Three essential controls are applied: (1) uRPF (Unicast Reverse Path Forwarding) to drop spoofed source IPs, (2) an inbound access list to block RFC 1918 addresses coming from the internet (they should never appear there), and (3) BGP TTL security to prevent TTL-spoofed BGP attacks from non-directly-connected hosts.
edge01 — WAN interface GigabitEthernet1 (ISP-A facing) 
edge01(config)#

# ── Anti-spoofing access list — applied inbound on WAN ─────────────
edge01(config)# ip access-list extended WAN-INBOUND-FILTER
edge01(config-ext-nacl)# deny ip 10.0.0.0 0.255.255.255 any log
edge01(config-ext-nacl)# deny ip 172.16.0.0 0.15.255.255 any log
edge01(config-ext-nacl)# deny ip 192.168.0.0 0.0.255.255 any log
↑ Drop RFC 1918 addresses arriving from the internet.
  These should never appear as source IPs on the WAN — they are private addresses.
  If they arrive, it is either a misconfigured ISP or an active spoofing attack.
  The "log" keyword records every dropped packet in syslog — evidence for the SIEM.
edge01(config-ext-nacl)# deny ip 127.0.0.0 0.255.255.255 any log
↑ Drop loopback addresses arriving from outside
edge01(config-ext-nacl)# deny ip 0.0.0.0 0.255.255.255 any log
↑ Drop 0.0.0.0/8 source addresses
edge01(config-ext-nacl)# permit ip any any
↑ Permit everything else — the Palo Alto does the real inspection.
  This ACL is not a firewall — it is a coarse sanity filter for obviously bogus traffic.
edge01(config-ext-nacl)# exit

# ── WAN interface configuration ─────────────────────────────────────
edge01(config)# interface GigabitEthernet1
edge01(config-if)# description ISP-A WAN — 203.0.113.1 (ISP-A router)
edge01(config-if)# ip address 203.0.113.2 255.255.255.252
↑ /30 subnet from ISP-A. edge01 = .2 (host), ISP-A router = .1 (gateway).
  ISP-A provides these addresses in the service agreement.
  This is a public IP — it appears in internet routing tables.
edge01(config-if)# ip access-group WAN-INBOUND-FILTER in
↑ Apply anti-spoofing filter inbound — before any other processing
edge01(config-if)# ip verify unicast source reachable-via rx
↑ uRPF (Unicast Reverse Path Forwarding) — loose mode.
  For each incoming packet: "can I route a reply back to this source IP via this same interface?"
  If not, the source IP is spoofed — drop the packet.
  This is the most effective anti-spoofing mechanism at the network layer.
edge01(config-if)# no ip proxy-arp
edge01(config-if)# no ip redirects
edge01(config-if)# no cdp enable
↑ Never run CDP on internet-facing interfaces
edge01(config-if)# no shutdown
edge01(config-if)# exit

# ── Verify WAN interface is up ──────────────────────────────────────
edge01(config)# do show interface GigabitEthernet1 | include "line protocol\|Internet address"
GigabitEthernet1 is up, line protocol is up
  Internet address is 203.0.113.2/30   ← WAN interface up with ISP IP ✓

# ── Test reachability to ISP-A router (one hop away) ──────────────
edge01(config)# do ping 203.0.113.1 repeat 5
!!!!!
Success rate is 100 percent (5/5), round-trip min/avg/max = 2/3/4 ms
   ↑ ISP-A's router is reachable ✓ — the physical WAN circuit is working.
     BGP session will be established in Section 08.
ℹThe ISP-A router IP (203.0.113.1) is confirmed reachable — this means the physical fibre from the ISP demarc to edge01's GE1 port is working correctly. The next step is establishing the BGP session, which will exchange routing information and allow actual internet traffic to flow.

Cisco Fabric · Guide 7 · Section 07
HA link between edge01 and edge02
The direct router-to-router link that enables iBGP synchronisation and makes graceful failover possible
Why the HA link is essential — the asymmetric routing problem
Without the HA link, if ISP-A fails: edge01 loses its default route and starts sending traffic to edge02 via the internal fabric (through the Palo Alto and spines). But edge02 would receive return traffic from ISP-B and send it back through ISP-B — the packet goes in via ISP-A's path but returns via ISP-B's path. Stateful firewall sessions on the Palo Alto would break (the PA tracks sessions by interface — if traffic enters on one interface and returns on another, the session is asymmetric and gets dropped). The HA link solves this by allowing edge01 and edge02 to exchange routing information via iBGP — both routers know the full picture and can make routing decisions that maintain symmetric paths.
edge01 — HA link GigabitEthernet3 (direct cable to edge02) 
edge01(config)# interface GigabitEthernet3
edge01(config-if)# description edge02:GE3 — direct HA link (iBGP)
edge01(config-if)# ip address 10.0.200.1 255.255.255.252
↑ /30: edge01 = .1, edge02 = .2
  This is a direct physical cable between the two ASR routers.
  No switching in between — any failure in this link is immediately detected.
edge01(config-if)# no ip proxy-arp
edge01(config-if)# no cdp enable
↑ No CDP even on internal links — security discipline
edge01(config-if)# carrier-delay msec 0
↑ Detect link failure immediately (no carrier delay dampening).
  We want to know immediately if the HA cable is pulled.
edge01(config-if)# no shutdown
edge01(config-if)# exit

# Test — edge02 must also have its GE3 configured at this point
edge01(config)# do ping 10.0.200.2 repeat 5
!!!!!
Success rate is 100 percent (5/5), round-trip min/avg/max = 0/0/1 ms
   ↑ Sub-millisecond — edge02 is directly connected ✓

eBGP with ISP-A — the internet peering session
The BGP session that connects NovaCrestBank to the internet — every statement explained with the why behind it
edge01 — eBGP configuration with ISP-A 
edge01(config)# router bgp 65200
↑ NovaCrestBank's BGP AS for edge routers = 65200.
  This is separate from the spine AS (65000) and leaf ASes (65001-65003).
  Using a different AS at the edge gives a cleaner eBGP topology:
  the spines only see edge AS 65200, never NovaCrestBank's internal leaf ASes.

edge01(config-router)# bgp router-id 10.0.0.51
↑ Use loopback0 as router-ID — stable, unique, easy to identify in "show bgp summary"

edge01(config-router)# bgp log-neighbor-changes
↑ Log every BGP state change to syslog — essential for ISP failover auditing.
  Every time ISP-A goes down or comes back, this creates a timestamped log entry.

edge01(config-router)# no auto-summary
↑ Disable automatic route summarisation — always off in modern BGP deployments.
  Auto-summary would aggregate our internal routes in unexpected ways.

# ── eBGP with ISP-A ──────────────────────────────────────────────
edge01(config-router)# neighbor 203.0.113.1 remote-as 12345
↑ ISP-A's router IP = 203.0.113.1, ISP-A's AS = 12345.
  Different AS from edge01 (65200) → this is eBGP.

edge01(config-router)# neighbor 203.0.113.1 description ISP-A primary peering

edge01(config-router)# neighbor 203.0.113.1 password 0 NovaCrest$BGP-ISP-A2026!
↑ MD5 authentication on ISP BGP session.
  Requires coordination with ISP-A — they must configure the same password.
  Prevents TCP RST injection attacks that could tear down the BGP session.
  This is standard practice — all ISPs support MD5 BGP authentication.

edge01(config-router)# neighbor 203.0.113.1 timers 10 30
↑ keepalive=10s, holdtime=30s for ISP BGP session.
  Default (60/180) means a failed session is not detected for 3 minutes.
  10/30 detects failure in 30 seconds. This matches the SLA NovaCrestBank
  expects from its ISP contracts. Faster timers increase the keepalive traffic
  but this is negligible on a dedicated WAN circuit.

edge01(config-router)# neighbor 203.0.113.1 ttl-security hops 1
↑ GTSM (Generalised TTL Security Mechanism) — BGP TTL security.
  Only accept BGP packets with TTL ≥ 254 (meaning the packet was sent from
  exactly 1 hop away — directly connected). A BGP packet from the internet
  claiming to be from ISP-A would have TTL reduced by each router hop — it would
  arrive with TTL much less than 254 and be dropped. Prevents remote BGP hijacking.

edge01(config-router)# address-family ipv4
edge01(config-router-af)# neighbor 203.0.113.1 activate

edge01(config-router-af)# neighbor 203.0.113.1 prefix-list ISP-A-IN in
↑ Filter what we accept from ISP-A — configured in Section 11.
  We only want: the default route (0.0.0.0/0). Nothing else from the internet.
  This prevents ISP-A from accidentally advertising a customer's route to us.

edge01(config-router-af)# neighbor 203.0.113.1 prefix-list NOVACREST-OUT out
↑ Filter what we advertise to ISP-A — configured in Section 11.
  We only advertise NovaCrestBank's public IP block (if we have one) or nothing.
  We NEVER advertise internal RFC 1918 routes to the internet.

edge01(config-router-af)# neighbor 203.0.113.1 route-map SET-LOCAL-PREF-ISP-A in
↑ Set local preference on routes received from ISP-A.
  local-preference 200 = prefer ISP-A over ISP-B (which gets local-pref 100).
  Higher local-preference = preferred outbound path.
  This makes ISP-A the primary path and ISP-B the backup.

edge01(config-router-af)# neighbor 203.0.113.1 soft-reconfiguration inbound
↑ Store received routes before policy filtering — enables show commands
  for troubleshooting without needing to tear down and re-establish the session.

edge01(config-router-af)# maximum-paths 1
↑ Only one best path per prefix on the edge — we do not ECMP between ISPs.
  Traffic exits via the preferred ISP (ISP-A) when both are available.

edge01(config-router-af)# exit
edge01(config-router)# exit

iBGP with edge02 and BGP to Palo Alto
The internal BGP sessions — synchronising route information between the two ASRs and exchanging routes with the firewall
edge01 — iBGP with edge02 and eBGP with Palo Alto 
edge01(config)# router bgp 65200

# ── iBGP with edge02 ─────────────────────────────────────────────
edge01(config-router)# neighbor 10.0.200.2 remote-as 65200
↑ Same AS (65200) → iBGP.
  Uses the HA link IP (10.0.200.2) — direct physical connection.

edge01(config-router)# neighbor 10.0.200.2 description edge02 iBGP HA peer
edge01(config-router)# neighbor 10.0.200.2 password 0 NovaCrest$BGP2026!
edge01(config-router)# neighbor 10.0.200.2 timers 3 9
↑ Fast timers for HA link — detect edge02 failure quickly (9 seconds)
edge01(config-router)# neighbor 10.0.200.2 next-hop-self
↑ Critical for iBGP: when edge01 forwards a route learned from ISP-A to edge02,
  it sets itself (10.0.200.1) as the next-hop.
  Without this: edge02 would see the next-hop as 203.0.113.1 (ISP-A's router IP)
  which is not reachable from edge02 — edge02 cannot use the route.
edge01(config-router)# address-family ipv4
edge01(config-router-af)# neighbor 10.0.200.2 activate
edge01(config-router-af)# neighbor 10.0.200.2 route-reflector-client
↑ edge01 acts as a route reflector for edge02 in this simplified design.
  In a standard iBGP mesh, every router must peer with every other router.
  With only two iBGP routers, this means one full-mesh session — route reflector
  is technically not needed, but designating edge01 as RR is future-proof if
  additional edge routers are added later.
edge01(config-router-af)# exit

# ── eBGP with Palo Alto ─────────────────────────────────────────
edge01(config-router)# neighbor 10.0.100.1 remote-as 65100
↑ Palo Alto is AS 65100 — configured on the PA's inside interface.
  Different AS from edge01 (65200) → eBGP.
  The BGP session travels over the GE2 /30 link (10.0.100.x).

edge01(config-router)# neighbor 10.0.100.1 description Palo-Alto-PA5250-inside
edge01(config-router)# neighbor 10.0.100.1 password 0 NovaCrest$BGP2026!
edge01(config-router)# neighbor 10.0.100.1 timers 3 9
edge01(config-router)# address-family ipv4
edge01(config-router-af)# neighbor 10.0.100.1 activate
edge01(config-router-af)# neighbor 10.0.100.1 default-originate
↑ Advertise a default route (0.0.0.0/0) to the Palo Alto.
  The PA uses this as its default route to send internet-bound traffic
  back to the ASR for forwarding to the ISP.
  The Palo Alto only advertises the default route into the fabric (to the spines)
  AFTER it has received one from an ASR. If both ISPs fail, the PA stops
  advertising the default — the fabric no longer forwards internet traffic (correct).
edge01(config-router-af)# neighbor 10.0.100.1 route-map PA-OUT out
↑ Only send the default route and NovaCrestBank's internal summary to PA.
  The PA does not need to know about ISP-A's prefixes.
edge01(config-router-af)# neighbor 10.0.100.1 prefix-list PA-IN in
↑ Only accept the internal summary (10.0.0.0/8) from the PA.
  The PA should advertise NovaCrestBank's internal routes back to the ASR
  so the ASR knows how to route return traffic from the internet.
edge01(config-router-af)# exit

Default route — primary and backup paths
How the default route flows through the architecture — and why it is the most important route in NovaCrestBank's routing table
The default route journey — from ISP to banking application
The default route (0.0.0.0/0) is the route that handles all traffic with no more-specific match. For NovaCrestBank, this means: "if you do not know where to send a packet, send it toward the internet via the ASR." The default route propagates inward through the network in a chain: ISP-A → edge01 (via BGP) → Palo Alto (via eBGP) → spine switches (via eBGP) → leaf SVIs (via BGP). Every device in the fabric gets a default route pointing toward the internet via the Palo Alto and ASR.
edge01 — default route handling and local preference tuning

↑ Routes from ISP-A get local-preference 200.
  Routes from ISP-B (via iBGP from edge02) get local-preference 100 (default).

  Result: all outbound internet traffic prefers ISP-A as long as ISP-A is up.

Cisco Fabric · Guide 7 · Section 11
Prefix lists — what we advertise to the ISP and what we accept
The most important security controls on the edge router — preventing accidental route leaks that could cause internet routing catastrophes
Route leaks — the most dangerous BGP misconfiguration
In 2019, a small ISP in Pennsylvania accidentally leaked 70,000 routes from Amazon, Cloudflare, and others to its upstream ISP. Traffic for large portions of the internet was redirected through a tiny ISP that could not handle the load — causing widespread outages. This happened because the ISP had no outbound prefix filtering. NovaCrestBank's prefix lists on edge01 prevent this from ever happening: we explicitly define every prefix we will advertise to ISPs, and explicitly define what we will accept from ISPs.
edge01 — prefix lists for ISP BGP sessions 
edge01(config)#

# ── What we ACCEPT from ISP-A ──────────────────────────────────────
# NovaCrestBank chooses to receive only the default route from ISPs.
# We do NOT receive the full internet BGP table (800,000+ routes).
# Why: a full table would require significant memory and would not change
# our routing decisions — we always send internet-bound traffic to the ISP
# regardless of what the optimal path is. A default route is sufficient.

edge01(config)# ip prefix-list ISP-A-IN description Accept only default route from ISP-A
edge01(config)# ip prefix-list ISP-A-IN seq 10 permit 0.0.0.0/0
↑ Accept the default route (0.0.0.0/0) only
edge01(config)# ip prefix-list ISP-A-IN seq 1000 deny 0.0.0.0/0 le 32
↑ Deny everything else — no internet prefixes accepted into our routing table
  This prevents ISP-A from accidentally leaking customer routes to us

# ── What we ADVERTISE to ISP-A ────────────────────────────────────
# NovaCrestBank advertises its public IP block (if it has PI space — Provider Independent)
# OR nothing at all (if using ISP-provided addresses / full NAT).
# For this guide: NovaCrestBank has PI space 192.0.2.0/24 (documentation example).
# Replace with actual PA space at deployment.

edge01(config)# ip prefix-list NOVACREST-OUT description Advertise NovaCrestBank public IP space
edge01(config)# ip prefix-list NOVACREST-OUT seq 10 permit 192.0.2.0/24
↑ Our public IP block — customers reach banking applications at IPs in this block.
  The Palo Alto applies NAT: internal 10.50.0.200 (OCP API VIP) is published
  as a public IP within 192.0.2.0/24.
edge01(config)# ip prefix-list NOVACREST-OUT seq 1000 deny 0.0.0.0/0 le 32
↑ Deny everything else — CRITICAL security control.
  This prevents us from accidentally advertising:
  - Internal RFC 1918 routes to the internet (10.x.x.x, 172.16.x.x, 192.168.x.x)
  - Routes learned from other sources (customers, peering)
  - Prefixes we do not own
  Without this, a routing table error could make us transit traffic for
  other organisations — the "route leak" scenario described above.

# ── What we send TO and accept FROM the Palo Alto ──────────────────
edge01(config)# ip prefix-list PA-OUT description Routes sent to Palo Alto
edge01(config)# ip prefix-list PA-OUT seq 10 permit 0.0.0.0/0
↑ Send the default route to PA so PA knows to send internet traffic to us
edge01(config)# ip prefix-list PA-OUT seq 1000 deny 0.0.0.0/0 le 32

edge01(config)# ip prefix-list PA-IN description Routes accepted from Palo Alto
edge01(config)# ip prefix-list PA-IN seq 10 permit 10.0.0.0/8
↑ Accept the internal summary — PA tells edge01 "10.0.0.0/8 is behind me"
  so edge01 can route return internet traffic to the correct internal destination
edge01(config)# ip prefix-list PA-IN seq 1000 deny 0.0.0.0/0 le 32

# ── Route map for PA outbound ──────────────────────────────────────
edge01(config)# route-map PA-OUT permit 10
edge01(config-route-map)# match ip address prefix-list PA-OUT
edge01(config-route-map)# exit

QoS — banking traffic classification and prioritisation
How NovaCrestBank ensures real-time banking transactions always get through — even when the WAN link is congested
Why QoS matters specifically at the WAN edge
Inside the data center, the 25G and 100G links between switches are rarely saturated. QoS is helpful but not critical inside the fabric. At the WAN edge, the internet connection is typically 1G or less — a fraction of the internal bandwidth. When a bulk batch job (end-of-day transaction file transfer, backup to off-site) saturates the WAN link, interactive banking transactions (customer login, card authorisation) would queue behind the bulk transfer and experience seconds of latency. This is unacceptable. QoS at the WAN edge prioritises time-sensitive banking traffic over bulk transfers regardless of congestion.

NovaCrestBank QoS traffic classes

Real-time (VoIP/video)
EF (46)
10%
Expedited Forwarding — voice and video calls for banking operations. Strict priority queue — always served first.
Banking transactions
AF31 (26)
40%
Online banking, card authorisations, payment APIs. Guaranteed 40% bandwidth with burst capability. Includes HTTPS to OCP ingress VIP.
Management / signalling
CS6 (48)
5%
SSH, SNMP, syslog, monitoring traffic. Small but always available — ensures management access even during WAN congestion.
Business applications
AF21 (18)
25%
General business traffic — email, internal web applications, non-critical APIs. Gets remaining capacity after priority classes.
Best effort / bulk
BE (0)
20%
Batch transfers, backups, updates. Gets the remainder. During congestion, this class suffers first — by design.
edge01 — QoS policy configuration (Modular QoS CLI — MQC) 
edge01(config)#

# ── Step 1: Classify traffic into categories (class-maps) ──────────
edge01(config)# class-map match-any REALTIME
edge01(config-cmap)# match dscp ef
↑ Match packets marked EF (Expedited Forwarding, DSCP 46).
  VoIP and video conferencing tools set DSCP EF themselves.
edge01(config-cmap)# exit

edge01(config)# class-map match-any BANKING-TRANSACTIONS
edge01(config-cmap)# match dscp af31
↑ Match AF31-marked packets.
  The Palo Alto marks outbound banking traffic (HTTPS to known payment endpoints
  and to 10.50.0.201 — the OCP ingress VIP) with AF31 via a QoS policy.
  Edge router honours the mark — it does not reclassify.
edge01(config-cmap)# match dscp af32
edge01(config-cmap)# exit

edge01(config)# class-map match-any MANAGEMENT
edge01(config-cmap)# match dscp cs6
edge01(config-cmap)# exit

edge01(config)# class-map match-any BUSINESS
edge01(config-cmap)# match dscp af21
edge01(config-cmap)# match dscp af22
edge01(config-cmap)# exit

# ── Step 2: Define per-class actions (policy-map) ──────────────────
edge01(config)# policy-map WAN-QOS-OUT
↑ This policy applies to OUTBOUND traffic on the WAN interface (toward ISP).
  "Out" = traffic leaving NovaCrestBank going to the internet.

edge01(config-pmap)# class REALTIME
edge01(config-pmap-c)# priority percent 10
↑ Strict priority queue — 10% of link bandwidth always reserved.
  Traffic in this class is NEVER delayed behind lower-priority traffic.
  If the link is 1Gbps: 100Mbps always available for voice/video.
edge01(config-pmap-c)# exit

edge01(config-pmap)# class BANKING-TRANSACTIONS
edge01(config-pmap-c)# bandwidth percent 40
↑ Guaranteed 40% for banking transactions (400 Mbps on a 1G link).
  "bandwidth percent" = weighted fair queuing, not strict priority.
  Can burst above 40% if other classes are idle.
edge01(config-pmap-c)# queue-limit 100 packets
↑ Limit the queue depth — prevents head-of-line blocking.
  A transaction that waits in queue longer than ~100ms (at 1G) is worse than
  a packet drop (TCP will retransmit but the old queued packet is stale).
edge01(config-pmap-c)# exit

edge01(config-pmap)# class MANAGEMENT
edge01(config-pmap-c)# bandwidth percent 5
edge01(config-pmap-c)# exit

edge01(config-pmap)# class BUSINESS
edge01(config-pmap-c)# bandwidth percent 25
edge01(config-pmap-c)# exit

edge01(config-pmap)# class class-default
edge01(config-pmap-c)# bandwidth percent 20
↑ class-default catches everything not matched above (best-effort / bulk).
  Gets 20% — last to be served during congestion.
edge01(config-pmap-c)# fair-queue
↑ Within best-effort class: use WFQ so multiple bulk flows share fairly.
  Without this, a single bulk flow could starve other best-effort flows.
edge01(config-pmap-c)# exit
edge01(config-pmap)# exit

# ── Step 3: Apply policy to the WAN interface ───────────────────────
edge01(config)# interface GigabitEthernet1
edge01(config-if)# service-policy output WAN-QOS-OUT
↑ Apply QoS policy outbound on the WAN interface.
  "output" = applies to traffic leaving GE1 toward ISP-A.
  Traffic coming IN from ISP-A is not shaped (we cannot control what ISP sends).
  QoS on WAN is always applied outbound — that is the direction we control.
edge01(config-if)# exit
ℹThe bandwidth percentages (10+40+5+25+20=100%) ensure the policy is valid. In IOS-XE, a priority class is always served first (strict priority). The remaining bandwidth percentages apply to the non-priority classes and are weighted relative to each other. If only banking traffic and best-effort are active, banking gets (40/90 = 44%) and best-effort gets (20/90 = 22%) of the non-priority bandwidth, with the unused management and business shares redistributed proportionally.

The critical difference — local preference makes ISP-A primary
Both edge01 and edge02 receive a default route from their respective ISPs. Both advertise this default route via iBGP to each other and via eBGP to the Palo Alto. The local preference values determine which default route the network prefers as its primary outbound path. edge01 sets local-preference 200 on ISP-A's default route. edge02 sets local-preference 100 on ISP-B's default route. Higher local-preference wins. All outbound internet traffic uses ISP-A (via edge01) as long as ISP-A is healthy. If ISP-A fails, edge01's default route disappears, and ISP-B's route (local-pref 100) becomes the only option — traffic automatically shifts to edge02 and ISP-B.
