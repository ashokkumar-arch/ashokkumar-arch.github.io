What HA gives NovaCrestBank — in one sentence
When an ESXi host fails at 3 AM, vSphere HA automatically restarts all the VMs that were running on that host on the surviving hosts in the cluster — without any human intervention, in approximately 2–4 minutes per VM restart — and then sends an alert so the on-call engineer knows which host failed.

What HA does

HA protects against host failure

If a physical ESXi host loses power, loses its management network, or its underlying hardware crashes, HA detects the absence of heartbeats from that host and restarts all VMs that were running on it on surviving hosts. The VMs experience a reboot — but come back up automatically.

HA protects against VM operating system crash

If a VM's guest OS crashes (BSOD, kernel panic) and cannot be detected as running, HA's VM Monitoring feature can detect the crash (via VMware Tools heartbeat absence) and restart the VM automatically — even if the underlying host is healthy.

What HA does NOT do — three common misconceptions

MisconceptionReality
"HA provides zero-downtime for applications" HA restarts VMs after failure — there is a restart period of 1–5 minutes where the VM is unavailable. Applications must tolerate a brief outage or use application-level HA (clustering, load balancing) for zero-downtime. HA is infrastructure-level protection, not application-level HA.
"HA works without shared storage" HA requires that VM disk files be on shared storage accessible to all hosts — that is vSAN (configured in vSphere-05). A VM whose VMDK lives only on a local host's disk cannot be restarted on another host. This is why vCenter was migrated to vSAN before HA is enabled.
"HA migrates VMs off a failing host gracefully" HA restarts VMs — it does not migrate them. Live migration (vMotion) is what gracefully moves a running VM. HA kicks in only after a host has failed and the VMs are already down. The pre-failure graceful option is DRS with maintenance mode (covered in this guide).
VMware Series · vSphere-06 · Section 02
How HA detects failure and restarts VMs — the internal mechanism
Understanding what happens inside the cluster from the moment a host fails to the moment the last VM restarts
The HA master/slave election and heartbeat system
How does vSphere HA know a host has failed versus just being temporarily unreachable?
Each cluster has one HA master host elected from the cluster members. The master monitors all other hosts via two independent heartbeat mechanisms — network heartbeats and datastore heartbeats. Both must indicate failure before HA declares a host failed and initiates VM restarts.

Network heartbeats: Every host sends a UDP heartbeat packet to the master every second via the management VMkernel (vmk0). If the master does not receive a heartbeat for 10 seconds (configurable via das.failuredetectiontime), it suspects the host has failed.

Datastore heartbeats: As a second opinion, the master checks a heartbeat file on the vSAN datastore that the suspected host writes to every 5 seconds. If the host is merely isolated (its management network is down but ESXi is still running), it continues updating the datastore heartbeat. The master sees the datastore heartbeat still updating and declares the host as "Isolated" rather than "Failed" — triggering the isolation response instead of a full VM restart.

The distinction matters: An isolated host that is still running might try to keep running its VMs. If the master also tries to restart those VMs on other hosts, you get duplicate VMs writing to the same VMDKs — catastrophic data corruption. The two-heartbeat system prevents this split-brain scenario.

The failure-to-restart timeline for NovaCrestBank

T+0s esxi02 loses power (hardware failure)
T+10s HA master (esxi01) stops receiving network heartbeats from esxi02
T+25s HA master confirms datastore heartbeat also stopped — esxi02 declared FAILED (not isolated)
T+30s HA master begins VM restart sequence on esxi02's orphaned VMs
T+30s Admission control check: is there enough reserved capacity on remaining hosts? Yes → proceed
T+35s DC02 (highest restart priority) restarts on esxi03
T+90s DC02 is back up — AD services restored on surviving DC01
T+120s IssuingCA (second priority) restarts on esxi04
T+300s All VMs from esxi02 restarted on surviving hosts
T+300s Alert sent to monitoring system — on-call engineer receives page
The HA master election — which host becomes master
When HA is enabled, one host becomes the master. How is the master elected and what happens if the master itself fails?
The HA master is elected via a distributed algorithm based on the number of datastores each host has access to — the host with access to the most datastores becomes the master. If hosts have the same datastore count, the host with the lowest MoID (Managed Object ID) wins.

For NovaCrestBank with all 5 hosts connected to vsanDatastore, the election is based on MoID order — typically whichever host was added to the cluster first (esxi01). This is automatic and transparent.

If the master fails: a new master is elected from the remaining slave hosts using the same algorithm — the host with the most datastores accessible (still all of vsanDatastore for the 4 surviving hosts). The re-election takes approximately 10–15 seconds. VMs on the failed master are restarted by the new master after election. There is no single point of failure in the HA master architecture.
VMware Series · vSphere-06 · Section 03
What DRS does — automated load balancing and intelligent VM placement
DRS is the feature that prevents any single host from becoming a hotspot while others sit idle — it continuously monitors and rebalances the cluster
What DRS monitors and what it acts on
DRS "load balances" the cluster — what metrics does it actually measure and what action does it take?
DRS measures per-host CPU and memory utilisation every 5 minutes and calculates the "imbalance" of the cluster. When the imbalance exceeds a configurable threshold, DRS generates migration recommendations — or executes them automatically — using vMotion to move VMs from overloaded hosts to underutilised ones.

The DRS balance score ranges from 1 (perfectly balanced) to 5 (severely imbalanced). DRS calculates the standard deviation of resource utilisation across hosts normalised to host capacity. A host running at 90% CPU while its peers are at 20% contributes to a high imbalance score.

Initial placement: DRS also controls where new VMs are powered on. When a VM is powered on, DRS selects the host with the most available capacity (considering CPU and memory) for the initial placement. Without DRS, VMs power on wherever vCenter chooses arbitrarily.

The critical distinction from HA: DRS acts on running healthy VMs to improve efficiency. HA acts on dead VMs to restore availability. They solve completely different problems and work together: DRS keeps the cluster balanced during normal operation, HA keeps VMs running after failures.

DRS automation levels — three choices

LevelWhat it doesNovaCrestBank choice
Manual DRS generates recommendations but takes no action. An administrator must review and approve each migration in vCenter. DRS never moves a VM without explicit approval. Not chosen — too much administrative overhead
Partially Automated DRS automatically places VMs on power-on (initial placement) but generates recommendations for ongoing rebalancing migrations that require manual approval. Not chosen — ongoing rebalancing still requires approval
Fully Automated DRS automatically places VMs at power-on AND automatically executes migrations to rebalance the cluster when the imbalance score exceeds the configured threshold. ✓ Selected — appropriate for a stable, monitored banking cluster
Why Fully Automated is the right choice for NovaCrestBank
Fully Automated means DRS can migrate any VM at any time. Does that risk disrupting banking applications?
No — vMotion migrations are transparent to applications and users, and DRS adds several safeguards before migrating production VMs.

DRS only migrates VMs when the cluster imbalance exceeds the threshold (configurable in Section 11). It does not migrate VMs continuously for small improvements — there is a cost-benefit calculation that weighs migration overhead against the benefit of better balance. A threshold of 3 (moderate) means DRS only acts on meaningful imbalance.

DRS respects several constraints before migrating: anti-affinity rules (DC1 and DC2 must stay on different hosts — DRS will never violate this), VM-host affinity rules (if configured), vMotion compatibility (DRS never migrates a VM to an incompatible host), and the available vMotion bandwidth. For critical VMs with Override settings, the DRS automation can be set to Manual on a per-VM basis even when the cluster is Fully Automated — giving fine-grained control without sacrificing automation for general VMs.

In a banking context, Fully Automated DRS is preferable because it prevents the gradual resource imbalance that accumulates when administrators forget to rebalance manually. An imbalanced cluster is more likely to cause VM performance problems than a DRS vMotion migration is.
VMware Series · vSphere-06 · Section 04
HA and DRS together — how they complement each other
Two features designed to work as a pair — understanding how they interact avoids misconfiguration

The interaction between HA and DRS

Scenario 1 — Host failure with DRS enabled

esxi02 fails at 2 AM. HA restarts its VMs on esxi03, esxi04, and esxi05. The cluster is now imbalanced — three hosts share all the workload while esxi01 is relatively idle. DRS detects the imbalance within the next 5-minute monitoring cycle. DRS migrates some VMs from overloaded hosts to esxi01, restoring balance. HA handled the failure; DRS handled the aftermath.

Scenario 2 — DRS during a rolling upgrade

An administrator puts esxi01 into maintenance mode for a firmware upgrade. DRS automatically vMotions all running VMs off esxi01 to the other four hosts (because maintenance mode triggers a DRS migration sweep). The administrator patches esxi01, exits maintenance mode, and DRS rebalances VMs back onto it. Completely automated — no manual vMotion needed.

Scenario 3 — Admission control and DRS interaction

HA admission control reserves capacity for one host's worth of VMs (configured in Section 07). DRS is aware of this reservation — it will not use the reserved capacity for balancing migrations. DRS balances within the available (non-reserved) capacity. The two features share a consistent view of cluster resources.

Host Isolation Response — why "Leave Powered On" is correct for NovaCrestBank
The Host Isolation Response has three options: Leave Powered On, Power Off, or Shut Down. Why choose "Leave Powered On"?
Because an isolated host is not necessarily a failed host — it may still be running all its VMs normally with only its management network down. Powering off VMs on an isolated host causes unnecessary downtime.

Consider: esxi02's management NIC cable is accidentally unplugged during a cable management exercise. esxi02 cannot communicate with the HA master (no management network heartbeat). However, esxi02 is still powered on, still running all its VMs, and still connected to vSAN (via vmk2 on the storage NIC pair — a different NIC). The vSAN datastore heartbeat continues.

With "Leave Powered On": VMs on esxi02 continue running normally. The administrator notices the management network is down, re-plugs the cable, esxi02 rejoins the cluster, no downtime occurred.

With "Power Off" or "Shut Down": all VMs on esxi02 are shut down immediately, and HA tries to restart them on other hosts. A cable management accident causes a full cluster event — completely unnecessary.

The vSAN datastore heartbeat is exactly the mechanism that makes "Leave Powered On" safe — if the isolated host were also cut off from storage, the datastore heartbeat would also fail, and HA would correctly declare it a failed host and restart its VMs elsewhere.

Admission control — the capacity reservation that guarantees HA can restart VMs
The most misunderstood HA setting — the reason vSphere sometimes refuses to power on a VM even when the cluster shows available resources
What admission control is and why it exists
What is admission control and why does it sometimes prevent powering on a VM?
Admission control is HA's mechanism for guaranteeing that, even after a host failure, enough capacity remains to restart all the failed host's VMs. It works by reserving a portion of cluster capacity exclusively for HA restarts — refusing to let normal VM operations consume that reserved capacity.

The problem without admission control: a cluster with 5 hosts each at 80% CPU utilisation has only 20% headroom per host. If esxi01 fails, its VMs need to restart on the 4 remaining hosts. But those 4 hosts are already at 80% — there is not enough spare capacity to restart esxi01's VMs without causing resource contention or VM restart failures. HA's promise of "restart your VMs after failure" cannot be kept if the cluster is too full.

Admission control prevents this by saying: "I will never allow the cluster to be more than X% utilised. I reserve Y% for HA. If a new VM power-on would push utilisation above the threshold, I reject the power-on with 'Insufficient resources to satisfy configured failover level for HA.'"

This is not a bug — it is the correct behaviour. When you see that error, it means the cluster is genuinely too full to guarantee HA protection for all running VMs. The fix is either to reduce other VMs' reservations, add a host to the cluster, or accept reduced HA protection level by adjusting admission control.

Why 20% reservation — not 25% or more
With 5 hosts, losing one host means losing 20% of capacity. The reservation should match this — why exactly 20%?
Because 20% equals exactly one host's capacity in a 5-host cluster — which is the correct reservation for tolerating one host failure.

The formula: 1 host ÷ 5 hosts = 20%. Reserving 20% CPU and 20% memory ensures that if any single host fails, its full CPU and memory contribution is covered by the reservation. VMs from the failed host can be restarted without the reservation being exhausted.

Using 25% would reserve more than one host's capacity — correctly protecting against a worst-case scenario where the largest host fails, but unnecessarily restrictive for a homogeneous cluster where all hosts are identical HP DL380 Gen11s.

Using a lower percentage (e.g., 10%) would under-protect — if the failed host had more VMs than 10% of cluster capacity can accommodate, HA cannot restart all of them. For a homogeneous cluster, 1/n where n = number of hosts is the correct formula.

Advanced HA settings — VM restart priorities and dependency order
Configuring which VMs restart first after a failure — essential for banking infrastructure where the restart sequence matters
Why restart priority order matters for NovaCrestBank
If esxi02 fails with DC02, IssuingCA, and several application VMs running on it, why does the order in which they restart matter?
Because infrastructure services that other VMs depend on must restart before the VMs that depend on them.

Consider the dependency chain: application VMs authenticate against Active Directory (DCs). If application VMs restart before the DCs are available, their boot-time AD authentication attempts fail, services fail to start, and those VMs may require manual intervention. If DC02 restarts first, it rejoins the domain with DC01, AD replication resumes, and all subsequently-restarted application VMs find AD available and boot cleanly.

Similarly: the Issuing CA provides certificates to vCenter and other services. If vCenter restarts before the Issuing CA, certificate validation during startup may fail. Restarting the Issuing CA before vCenter (though vCenter has a higher restart priority due to being the management plane) is the ideal sequence.

vSphere HA restart priorities: High (restarts first, within the first restart wave), Medium (default, second wave), Low (third wave). The restart dependency feature allows you to specify "do not restart VM X until VM Y is powered on and responsive."

NovaCrestBank restart priority assignments

These priorities are set per-VM in VM Overrides (Section 17). For now, understand the intended priority order:

VMPriorityReason
dc01.novacrestbank.local High Active Directory is required by all other infrastructure. DC01 is on esxi01 — if esxi01 fails, DC01 is the highest priority restart.
dc02.novacrestbank.local High Second DC — must restart quickly to restore AD replication and redundancy. Anti-affinity rule ensures DC01 and DC02 are never on the same host (so both cannot fail simultaneously in a single-host failure).
vcenter.novacrestbank.local High Without vCenter, the administrator cannot manage the cluster, trigger manual interventions, or access monitoring. Must be up quickly after a failure.

DRS migration threshold — what level 3 means and when to adjust it
Understanding the five threshold levels and the tradeoffs at each — so you know when the setting is working correctly and when to change it

The five DRS threshold levels

LevelPriority recommendations appliedBehaviourUse when
1 — Conservative Priority 1 only (most critical) Only migrates for the most extreme imbalance — very few migrations occur High-sensitivity workloads where any migration disruption is undesirable
2 Priority 1 and 2 Migrates for significant imbalance Moderately sensitive workloads
3 — Moderate Priority 1, 2, and 3 Balances regularly on meaningful imbalance — most common production setting NovaCrestBank default — correct balance of stability and efficiency
4 Priority 1–4 More frequent migrations, tighter balance Resource-intensive clusters where tight balance matters
5 — Aggressive All recommendations Constant migrations to maintain perfect balance Development/test clusters where migrations are not disruptive
Threshold 3 is correct for banking — why not level 5 for maximum efficiency
Level 5 keeps the cluster most balanced. Why use level 3 for NovaCrestBank?
Because each DRS migration is a vMotion, and while vMotion is transparent at the network level, it does consume CPU and memory bandwidth on both source and destination hosts during the migration. Aggressive DRS generates many migrations continuously, each consuming resources to perform the balancing act itself.

At threshold 5 on a busy cluster, you can end up in a situation where DRS is continuously migrating VMs to achieve marginal balance improvements — the migrations themselves consuming more resources than the imbalance they are correcting would have wasted. This is sometimes called "DRS thrashing."

For a banking production cluster running critical financial workloads, threshold 3 is the correct balance: DRS acts on real imbalance (reducing the risk of resource contention on overloaded hosts) but does not micro-optimise with constant migrations. The cluster is stable, well-balanced, and migrations occur when they meaningfully improve utilisation — not for fractional percentage gains.
VMware Series · vSphere-06 · Section 12
Predictive DRS and vSAN integration
How DRS uses vSAN performance history to anticipate load before it happens
What Predictive DRS is and when it helps
Standard DRS reacts to current utilisation. What does "Predictive DRS" add?
Predictive DRS uses historical performance data (from vSAN Performance Service and vRealize Operations if connected) to anticipate load spikes before they occur — and pre-migrates VMs to create headroom on the host that is about to become overloaded.

Example: every day at 09:00, the batch processing VM on esxi03 starts a report generation job that saturates esxi03's CPU for 30 minutes. Standard DRS would react after esxi03 is already overloaded — waiting for the next 5-minute monitoring cycle and then migrating VMs as esxi03 struggles. Predictive DRS, having seen this pattern in historical data, migrates VMs off esxi03 at 08:55 — before the CPU spike starts. esxi03 handles the batch job with headroom, other VMs were moved to balanced hosts in advance.

For NovaCrestBank, Predictive DRS is useful for predictable daily patterns (end-of-day reporting, overnight batch jobs, morning login storms). Enable it — the Performance Service installed in vSphere-05 provides the historical data it needs.
VMware Series · vSphere-06 · Section 13
Affinity and anti-affinity rules — why placement rules are critical for NovaCrestBank
DRS rules override DRS's automated placement decisions — they are the mechanism for ensuring critical VMs are always separated
VM-VM affinity vs anti-affinity — when to use each
vSphere supports both affinity rules (keep VMs together) and anti-affinity rules (keep VMs apart). What is each used for?
Anti-affinity rules keep specified VMs on different hosts — ensuring a single host failure cannot take down multiple related services simultaneously. Affinity rules keep specified VMs on the same host — used for latency-sensitive applications that benefit from co-location.

Anti-affinity use cases (NovaCrestBank's primary need):
— DC01 and DC02 must be on different hosts. A single host failure must never take down both Domain Controllers — otherwise the entire domain is unavailable until HA restarts at least one DC. Anti-affinity ensures that even DRS, in its automated balancing, can never move DC01 and DC02 to the same host.
— vCenter and DC01 should ideally be separated — if the host running both fails, the management plane (vCenter) and one DC go down simultaneously. This makes HA restart sequencing harder (you are managing the restart from a failed management plane).

Affinity use cases:
— A database server and its read replica that do high-volume inter-VM replication benefit from being co-located (sub-millisecond inter-VM latency on the same host vs slightly higher across the vSAN network).
— Not commonly needed in NovaCrestBank's initial configuration.

Name: NCB-AntiAffinity-CA-Infrastructure
Type: Separate Virtual Machines
VMs: issuingca.novacrestbank.local, vcenter.novacrestbank.local, dc01.novacrestbank.local
# Issuing CA, vCenter, and DC01 on different hosts # A single host failure cannot eliminate certificate services AND domain management # Note: with 5 hosts and 3 rules constraining 4 VMs, the rules are satisfiable
Verifying the rules are mutually satisfiable
With three anti-affinity rules covering five infrastructure VMs, can DRS actually satisfy all of them simultaneously on a 5-host cluster?
Yes — with 5 hosts and the three rules as defined, every rule can be satisfied simultaneously. The worst case: DC01, DC02, vCenter, and IssuingCA each need their own host. That is 4 hosts out of 5 used by these rules — leaving 1 host unconstrained for any of them. The four critical infrastructure VMs can all be on separate hosts (Rule 1: DC01≠DC02, Rule 2: DC01≠DC02≠vCenter, Rule 3: DC01≠vCenter≠IssuingCA).

A valid placement satisfying all rules:
— esxi01: vCenter
— esxi02: DC01
— esxi03: DC02
— esxi04: IssuingCA
— esxi05: (free for any VM)

If a fifth rule were added that also separated IssuingCA from DC02, it would require all four infrastructure VMs to be on four different hosts — still satisfiable. Adding more constraints on a 5-host cluster risks making rules mutually unsatisfiable — DRS would report a rule conflict. Always verify satisfiability before saving a new rule.

Why staggered restart delays — not simultaneous restart
All High-priority VMs restart in the same wave. Why give vCenter a 30-second delay and IssuingCA 120 seconds?
Because dependency services must be available before dependent services try to start — and the restart delays provide a rough time-based sequencing within priority tiers.

DC01 and DC02 restart immediately (0 second delay). They take approximately 60–90 seconds to fully boot and have AD services running. vCenter restarts 30 seconds later — it begins booting 30 seconds after the DCs started, and by the time vCenter's services are fully online (~3–5 minutes later), the DCs are long available. vCenter's LDAP authentication to AD succeeds on startup rather than needing to retry.

IssuingCA gets a 120-second delay because it needs AD running (to authenticate as a domain member) and ideally vCenter running (for management). Waiting 120 seconds after the restart wave begins means the DCs are likely available before the IssuingCA even starts booting.

These delays are approximate — HA does not have a dependency checker that waits for a service to actually respond before starting the next VM. The delays are time-based best-effort sequencing. For hard dependencies, application-level clustering (Windows Server clustering, database AAG) provides true dependency management.

Simulate host failure — verify HA restarts VMs
The most important test in this guide — confirming HA actually works before a real failure occurs
Plan this test carefully — all VMs on the test host will be briefly unavailable
Testing HA requires actually triggering a host failure event. The safest approach is to simulate it by disconnecting the management network on a host that is carrying non-critical VMs only. Move all critical VMs off the test host first. Perform the test during a maintenance window. Ensure the anti-affinity rules mean no critical VM is on the test host.
