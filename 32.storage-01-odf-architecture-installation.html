STORAGE-01 · Section 01
The NovaCrestBank storage landscape — vSAN, MinIO, and ODF
Before ODF was introduced, NovaCrestBank's OCP cluster used two storage systems: vSAN (via the vSphere CSI driver, providing block storage as PVCs) and MinIO (a lightweight S3-compatible object store deployed as an OCP workload). This guide introduces ODF as the third system, establishes its role, and explains how the three coexist and complement each other without duplication.

The three storage systems and their roles

vSAN (vSphere CSI)

Managed by VMware/VCF. Provides VMDK-backed persistent volumes. Currently serves: vSAN-gold (RAID-6 FTT:2) for GitLab Gitaly and PostgreSQL. vSAN-standard (RAID-5 FTT:1) for Jenkins, Redis. Fast, proven, hardware-backed. Limited to RWO (single-node ReadWriteOnce) access mode — no shared filesystem. Opaque to OCP — volumes are VMDKs managed by vCenter.

MinIO

Deployed as an OCP workload (StatefulSet). Provides S3-compatible object storage for platform observability: LokiStack log chunks, Tempo traces, GitLab CI artifacts, GitLab backups. Lightweight, simple, runs inside the cluster. Not HA — single StatefulSet. Not suitable for production data requiring high durability. STORAGE-02 covers its full installation and configuration.

ODF (this guide)

Ceph running inside OCP. Provides three storage types simultaneously: RWO block (replaces vSAN for OCP-native workloads), RWX shared filesystem (not possible with vSAN), and S3-compatible object via RGW/MCG (enterprise-grade, Ceph-managed redundancy). OCP-native — storage is visible and managed by OCP, not vCenter. Required for: shared filesystems, OpenShift Virtualization boot disks, workloads that need OCP-layer storage management.

Why NovaCrestBank adds ODF rather than expanding vSAN
vSAN already provides redundant block storage and works reliably. Why introduce ODF and its operational complexity rather than simply expanding vSAN capacity?
Four specific capabilities that vSAN cannot provide drive the ODF adoption decision. Each corresponds to a concrete NovaCrestBank requirement.

1. ReadWriteMany (RWX) shared filesystems. vSAN provides ReadWriteOnce (RWO) volumes — one pod reads and writes a given PVC at a time. Some workloads require multiple pods to mount the same volume simultaneously: OpenShift Virtualization live migration (the VM disk must be accessible from both source and destination nodes during migration), Jenkins agent pods sharing a workspace, and any stateful application that scales horizontally with shared state. vSAN cannot provide RWX. ODF's CephFS provides RWX PVCs natively.

2. OCP-native storage lifecycle management. vSAN PVCs are VMDKs managed in vCenter. When a PVC is resized, a vCenter storage policy must be satisfied. When a PVC is deleted, the VMDK is deleted through vCenter. This creates two control planes for storage — vCenter and OCP. With ODF, the entire storage lifecycle (create, resize, snapshot, restore, delete) is managed through OCP's storage API. No vCenter operations for OCP storage changes. This is aligned with the broader goal of reducing vCenter dependency as workloads migrate to OCP.

3. OpenShift Virtualization prerequisite. The next project in NovaCrestBank's roadmap is OpenShift Virtualization for banking workloads that cannot be containerised. ODF (specifically CephRBD) is the recommended storage backend for VM boot disks in OpenShift Virtualization. vSAN can technically be used but does not support the live migration and hot-plug capabilities that Ceph provides for VM storage. Deploying ODF now prepares the storage layer for OpenShift Virtualization.

4. Volume snapshots and clone operations. ODF's CSI driver supports VolumeSnapshots — point-in-time copies of PVC contents — natively through the Kubernetes VolumeSnapshot API. These are used by OADP (OpenShift API for Data Protection, covered in STORAGE-04) for application-consistent backups. vSAN supports snapshots but through vCenter's snapshot mechanism, which is not accessible via the Kubernetes VolumeSnapshot API without additional tooling.
STORAGE-01 · Section 02
What ODF is — Ceph running as an OCP-native operator
OpenShift Data Foundation is Red Hat's enterprise storage platform for OCP. Under the hood it is Ceph — the open-source distributed storage system — packaged, operated, and supported by Red Hat as an OCP Operator. Understanding what ODF is requires understanding what Ceph is and why running it as an OCP operator changes the operational model.
What
Ceph — a distributed storage system
Ceph is a software-defined storage platform that pools raw disks across multiple nodes and presents them as a unified storage cluster. Data is distributed and replicated across disks and nodes — no single disk or node failure causes data loss. Ceph has no single point of failure: its metadata, monitor quorum, and data placement are all distributed. NovaCrestBank's ODF cluster will use the raw disks on the 3 ODF-dedicated nodes, pool them into a Ceph cluster, and present storage through three OCP-facing APIs simultaneously.
How
ODF Operator — Ceph managed as Kubernetes objects
The ODF Operator translates Ceph administration into Kubernetes reconciliation. A StorageCluster CR declares the desired Ceph cluster configuration — which nodes, which disks, how many replicas. The ODF Operator reads this CR and deploys all Ceph daemons as OCP pods (MON pods, OSD pods, MGR pods, MDS pods, RGW pods). When a MON pod crashes, the Operator reschedules it. When a new disk is added to a node, the Operator provisions a new OSD pod for it. Ceph operations that previously required ceph CLI commands are now observable and manageable through oc get and oc describe.
Why
OCP-native means storage and compute share one lifecycle
When storage runs as an OCP Operator, it participates in OCP's upgrade lifecycle (OLM manages ODF version upgrades), monitoring (ODF exposes Prometheus metrics scraped by the cluster monitoring stack — OBS-01), alerting (ODF ships pre-configured PrometheusRules for Ceph health conditions), and RBAC (OCP users with appropriate roles can manage ODF objects). Storage is no longer a separate system operated by a separate team with separate tooling — it is part of the OCP platform managed by the same platform engineering team using the same tools.
ODF licensing — what the subscription covers
ODF requires a Red Hat ODF subscription, which is separate from the OCP subscription. The ODF subscription covers: the ODF Operator support, the Ceph software support, MCG (Multi-Cloud Gateway), and OADP (backup). At NovaCrestBank, ODF is licensed per managed TiB (tebibyte) of raw storage capacity. The 3 dedicated ODF nodes each contribute their raw disk capacity to the Ceph pool — the licensed TiB is based on the raw total before replication factor. With 3× replication, a 30 TiB raw cluster provides approximately 10 TiB of usable capacity.
STORAGE-01 · Section 03
Ceph components — MON, OSD, MGR, MDS, RGW, MCG explained
A Ceph cluster is composed of six distinct daemon types. Each runs as one or more OCP pods on the ODF storage nodes. Understanding what each daemon does explains why Ceph requires a minimum of 3 nodes, why certain daemons are replicated in odd numbers, and what fails if a specific pod is unhealthy.
MON — Monitor

Maintains the cluster map — the authoritative record of which OSDs exist, which are up, and which data placement groups are where. Runs on 3 nodes in quorum (majority vote required). Losing 2 of 3 MONs loses quorum and the cluster becomes read-only. ODF runs 3 MON pods — one per storage node.

OSD — Object Storage Daemon

One OSD per disk. Stores actual data. Handles replication — when data is written to one OSD, it replicates to two others (3× replication). Also runs recovery when an OSD fails. NovaCrestBank: 2 disks per node × 3 nodes = 6 OSDs total. With 3× replication: data survives loss of any 1 OSD without data loss.

MGR — Manager

Runs cluster management modules: the Ceph Dashboard (web UI), Prometheus metrics export, balancer (rebalances data across OSDs), and orchestrator (coordinates OSD provisioning with the ODF Operator). One active + one standby. ODF runs 2 MGR pods for HA.

MDS — Metadata Server

Required only for CephFS (shared filesystem). Manages the filesystem metadata — directory trees, file attributes, and inode mapping. One active + one standby per filesystem. ODF runs 2 MDS pods for each CephFS filesystem. Not needed if only block or object storage is used.

RGW — RADOS Gateway

Provides S3-compatible and Swift object storage API directly from Ceph. An HTTP gateway in front of the Ceph RADOS layer. Used when you want S3-compatible object storage with Ceph-managed redundancy. ODF deploys 1 or 2 RGW pods. Alternative to MCG for simpler S3 use cases.

MCG — Multi-Cloud Gateway

NooBaa-based S3 gateway that abstracts multiple backing stores (Ceph RGW, AWS S3, Azure Blob). Provides namespace buckets, data federation, and lifecycle policies. Used when you need to span cloud and on-premises object storage. ODF deploys MCG as the ObjectBucketClaim provider.

STORAGE-01 · Section 04
Three storage types — block, file, and object — and when each applies
ODF provides all three fundamental storage types simultaneously from the same Ceph cluster. The type you choose for each workload depends on the access pattern — not the data size or importance. Choosing the wrong type is a common mistake that causes performance or functionality problems.
TypeCeph technologyOCP access modeUse whenNovaCrestBank examples
Block (RBD) Ceph RADOS Block Device ReadWriteOnce (RWO) One pod reads/writes a device-like volume. Database files, application state. Highest performance — direct block I/O. PostgreSQL data directory, Prometheus TSDB, Gitaly repository storage, OpenShift Virtualization VM boot disks
File (CephFS) Ceph Filesystem via MDS ReadWriteMany (RWX) Multiple pods must read/write the same volume simultaneously. Shared workspaces, media libraries, content that multiple replicas access. OpenShift Virtualization live migration (VM disk accessible from two nodes during migration), Jenkins agent shared workspace (if multiple agents need the same files), NFS-style shared content for banking report generation
Object (S3) RGW or MCG (NooBaa) ObjectBucketClaim (not a PVC) Unstructured data accessed via S3 API. Log storage, backups, images, CI artifacts. Not mounted as a filesystem — accessed via HTTP PUT/GET. LokiStack log chunks (replaces MinIO for production), Tempo traces, OADP backup storage, OpenShift image registry backing store
The most common ODF mistake — using block storage where file storage is needed
A StatefulSet with 3 replicas, each mounting its own RWO PVC, is correct — each replica has its own exclusive volume. But an application that creates a Deployment (not StatefulSet) with replicas > 1, where all replicas need to write to a shared directory, cannot use RWO. The second pod cannot mount the PVC that the first pod holds exclusively. This results in pods stuck in Pending state with the error "Multi-Attach error for volume: Volume is already exclusively attached to one node." The fix is always to use a CephFS RWX PVC — not to try to work around RWO limitations.

STORAGE-01 · Section 06
ODF MCG vs MinIO — the object storage decision
Both ODF MCG and MinIO provide S3-compatible object storage accessible from OCP workloads. The choice depends on the durability requirements, the operational model, and whether the data must survive a cluster rebuild. NovaCrestBank uses both for different workloads.

MinIO — keep for platform tooling

  • Single StatefulSet, 1–4 replicas backed by vSAN PVCs
  • Data durability depends on vSAN RAID policy (vsan-standard = RAID-5)
  • Simple to operate, minimal resource footprint
  • No Kubernetes VolumeSnapshot integration
  • Appropriate for: LokiStack, Tempo, GitLab CI artifacts, GitLab backups
  • These workloads tolerate data loss — logs are observable data, not source-of-truth

ODF MCG — for production object storage

  • Ceph-backed, distributed redundancy across 3 ODF nodes
  • ObjectBucketClaims provisioned dynamically via OCP API
  • Namespace buckets can span on-premises and AWS S3 (hybrid cloud)
  • OADP backup stores target ODF MCG for application backup consistency
  • Appropriate for: OADP backup storage, OpenShift Image Registry, production data that must survive node failures
  • Higher resource requirements — runs as part of ODF cluster
STORAGE-01 · Section 07
Storage node planning — dedicated nodes vs converged
ODF can run in two deployment modes: dedicated storage nodes (separate nodes that run only ODF daemons) or converged mode (ODF daemons co-located with application workloads on existing nodes). NovaCrestBank uses dedicated storage nodes for the production ODF cluster.
Dedicated storage nodes vs converged — why dedicated for NovaCrestBank
Converged mode avoids adding new nodes — ODF runs on existing worker nodes alongside application pods. This appears operationally simpler. Why does NovaCrestBank use dedicated storage nodes instead?
Three operational reasons justify dedicated storage nodes for a production banking environment, despite the additional hardware cost.

1. Resource isolation and noisy-neighbour prevention. Ceph OSD daemons are I/O-intensive — they perform continuous replication, recovery, and scrubbing operations on the disks they manage. When co-located with application pods, OSD I/O operations compete with application disk I/O on the same node. For banking microservices with latency SLOs (payment-service P99 < 200ms), competing with OSD background scrubbing on the same node is a latency risk. Dedicated nodes eliminate this competition: OSD I/O happens on nodes that no application pods run on.

2. Upgrade isolation. During a cluster upgrade (UPG-02), worker nodes are drained and rebooted. If OSD daemons run on worker nodes, draining a worker node triggers OSD pod eviction — which triggers Ceph to begin data recovery (it sees the OSD as down and starts rebalancing data). Rebalancing during an OCP upgrade creates unnecessary I/O load and extends the upgrade duration. With dedicated storage nodes managed by a separate MachineConfigPool, they are not drained during worker node upgrades — OSD pods remain running, Ceph stays healthy.

3. Disk management clarity. Dedicated storage nodes have disks that belong exclusively to ODF. There is no question about which disks are ODF OSDs and which are application workload disks — the answer is always "all disks on storage nodes are OSD disks." In converged mode, disk assignment is more complex and error-prone.

STORAGE-01 · Section 09
Install the Local Storage Operator — preparing raw disks for ODF
The Local Storage Operator (LSO) discovers raw block devices on nodes and makes them available as PVs (PersistentVolumes) that ODF can claim as OSD backing storage. Without LSO, ODF cannot access the raw NVMe disks. LSO is a prerequisite that must be deployed before the ODF Operator.
LocalVolumeDiscovery and LocalVolumeSet — discovering and claiming NVMe disks
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeDiscovery
metadata:
  name: auto-discover-devices
  namespace: openshift-local-storage
spec:
  nodeSelector:
    nodeSelectorTerms:
      - matchExpressions:
          - key: cluster.ocs.openshift.io/openshift-storage
            operator: Exists
    # Run discovery only on nodes labelled as ODF storage nodes.
    # The LocalVolumeDiscovery daemon discovers all block devices on these nodes
    # and reports them in LocalVolumeDiscoveryResult objects —
    # showing device name, size, vendor, model, and whether they have a filesystem.
---
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeSet
metadata:
  name: local-block
  namespace: openshift-local-storage
spec:
  nodeSelector:
    nodeSelectorTerms:
      - matchExpressions:
          - key: cluster.ocs.openshift.io/openshift-storage
            operator: Exists
  storageClassName: localblock
    # localblock: the StorageClass LSO creates for the raw block devices.
    # ODF's StorageCluster CR references this StorageClass to claim PVs for OSDs.
  volumeMode: Block
    # Block mode (not Filesystem): OSD devices must be raw block devices.
    # Ceph's OSD daemon manages its own filesystem (BlueStore) on the raw block device.
    # Do NOT use Filesystem mode — Ceph requires raw block access.
  deviceInclusionSpec:
    deviceTypes: [disk]
        # disk only: include physical disks. Exclude: part (partitions), loop, rom.
          # This ensures only the raw NVMe disks are claimed — not the OS disk or
          # any other partition on the storage nodes.
    minSize: 100Gi
        # minSize 100 GiB: the OS disk (480 GB SSD) is > 100 GiB but is already
          # partitioned (has a filesystem) — it will be excluded by the deviceTypes
          # filter. The 3.8 TB NVMe disks are unpartitioned and will be included.
    deviceMechanicalProperties: [NonRotational]
        # NonRotational: NVMe SSDs only. Exclude spinning HDDs (Rotational) if any
          # are present on the node. ODF performance is significantly better on SSDs.

Install the ODF Operator via OLM
The ODF Operator is installed from the Red Hat Operators catalog. It installs the ODF operator itself plus its dependencies: the Rook-Ceph operator (manages Ceph daemons as Kubernetes objects), the NooBaa operator (manages MCG), and the OCS operator (manages the StorageCluster CRD).
ODF Operator Subscription
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: odf-operator
  namespace: openshift-storage
    # openshift-storage: the dedicated ODF namespace.
    # All ODF pods (MON, OSD, MGR, MDS, RGW, MCG) run here.
    # This namespace is created automatically by OLM with the Subscription.
spec:
  channel: stable-4.16
    # ODF channel must match the OCP minor version.
    # OCP 4.16 → ODF stable-4.16. This is a hard requirement:
    # ODF 4.16 is certified against OCP 4.16. Using a mismatched channel
    # is unsupported and may cause compatibility issues with the CSI driver.
  name: odf-operator
  source: redhat-operators
    # redhat-operators: the Red Hat certified catalog.
    # ODF is a Red Hat product — not in community-operators.
    # This is the supported catalog for production deployments.
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual

STORAGE-01 · Section 11
The StorageCluster CR — every field explained
The StorageCluster Custom Resource is the single object that declares NovaCrestBank's complete Ceph cluster configuration: which nodes, which disks, how many replicas, which storage types to enable. The ODF Operator reads this CR and deploys all Ceph daemons to match. Every field is annotated.
StorageCluster CR — NovaCrestBank ODF cluster — every field explained
apiVersion: ocs.openshift.io/v1
kind: StorageCluster
metadata:
  name: ocs-storagecluster
  namespace: openshift-storage
    # The name ocs-storagecluster is the standard name expected by ODF tooling.
    # Changing it requires additional configuration — use the default.
spec:
  storageDeviceSets:
    - name: ocs-deviceset-localblock
      count: 1
            # count: how many device sets to create. 1 = one OSD per PV in the pool.
              # With 6 PVs (2 per node × 3 nodes) and count=1: 6 OSDs are created.
              # Each OSD corresponds to exactly one localblock PV (one NVMe disk).
      replica: 3
            # replica: 3 = each piece of data is written to 3 OSDs on 3 different nodes.
              # Tolerance: can lose 1 of 3 storage nodes without data loss.
              # Cannot lose 2 of 3 nodes (below quorum — cluster goes read-only).
              # Standard for production. Do not use replica: 2 in production.
      dataPVCTemplate:
        spec:
          storageClassName: localblock
                    # localblock: the StorageClass created by LSO LocalVolumeSet.
                      # ODF claims PVs from this StorageClass to back OSD pods.
          accessModes: [ReadWriteOnce]
          volumeMode: Block
                    # Block mode required — raw block device for Ceph BlueStore OSD.
          resources:
            requests:
              storage: 1
                            # storage: 1 (not 1Gi — just "1"). This is a special ODF convention:
                              # setting storage: 1 tells ODF to claim the entire available PV.
                              # The actual PV is 3756 GiB — ODF uses all of it for the OSD.
      placement:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
              - matchExpressions:
                  - key: cluster.ocs.openshift.io/openshift-storage
                    operator: Exists
        tolerations:
          - key: node.ocs.openshift.io/storage
            effect: NoSchedule
            operator: Exists
                # OSD pods: run only on storage nodes (via nodeAffinity).
                  # Tolerate the storage taint (so they can schedule there despite NoSchedule).

  monDataDirHostPath: /var/lib/rook
    # MON data directory on the host filesystem.
    # Ceph MON daemons persist their data here for fast restarts.
    # This path exists on the storage nodes' local filesystems.
    # MON data is small (< 1 GiB) but must survive pod restarts.

  multiCloudGateway:
    dbStorageClassName: ocs-storagecluster-ceph-rbd
        # MCG uses a Ceph RBD PVC for its NooBaa database.
          # After the StorageCluster is healthy, ocs-storagecluster-ceph-rbd
          # exists — MCG bootstraps itself using a PVC from this class.
    reconcileStrategy: manage

  managedResources:
    cephBlockPools:
      reconcileStrategy: manage
    cephFilesystems:
      reconcileStrategy: manage
    cephObjectStores:
      reconcileStrategy: manage
        # reconcileStrategy: manage = ODF Operator owns these resources.
          # ODF creates and manages: CephBlockPool, CephFilesystem, CephObjectStore
          # CR objects that define the Ceph storage pools.
          # "manage" means: if you manually edit these, ODF reverts your changes.
          # Do not change pool settings directly — change the StorageCluster CR instead.

STORAGE-01 · Section 16
MCG — Multi-Cloud Gateway and ObjectBucketClaims
The Multi-Cloud Gateway (MCG) is ODF's S3 gateway abstraction layer. An ObjectBucketClaim (OBC) is how applications request an S3 bucket from MCG — the equivalent of a PersistentVolumeClaim for object storage. When an OBC is created, MCG provisions a bucket and creates a ConfigMap and Secret containing the bucket endpoint, access key, and secret key.
ObjectBucketClaim — creating an S3 bucket via MCG for OADP backup
apiVersion: objectbucket.io/v1alpha1
kind: ObjectBucketClaim
metadata:
  name: oadp-backup-bucket
  namespace: openshift-adp
    # OBC is namespace-scoped. The ConfigMap and Secret MCG creates
    # are in the same namespace as the OBC. OADP reads them from openshift-adp.
spec:
  storageClassName: openshift-storage.noobaa.io
    # Use the MCG StorageClass (NooBaa-backed).
    # MCG automatically selects the backing store (Ceph RGW in this case).
  generateBucketName: oadp-backup
    # generateBucketName: MCG creates a bucket with this prefix + a unique suffix.
    # Result: oadp-backup-a1b2c3d4-xxxx
    # Using generateBucketName rather than a fixed name prevents naming conflicts
    # if multiple OBCs request similarly-named buckets.
  additionalConfig:
    maxSize: 2Ti
        # maxSize: soft quota on the bucket. MCG returns 403 (quota exceeded) when
          # the bucket reaches 2 TiB. Prevents runaway backup jobs from filling
          # the entire ODF object store.
    maxObjects: "10000000"

STORAGE-01 · Section 17
Migrating existing PVCs from vSAN to ODF StorageClasses
PVCs cannot change their StorageClass after creation — the StorageClass is immutable. To migrate a workload from vSAN-backed storage to ODF-backed storage, the data must be copied from the old PVC to a new ODF-backed PVC. The cleanest method for stateful applications is a controlled migration using a temporary Job to copy data between PVCs while the application is scaled down.
When NOT to migrate — existing vSAN PVCs are not broken
Migrating all existing PVCs to ODF StorageClasses is not required and not recommended as a bulk operation. Existing vSAN-backed PVCs for GitLab Gitaly, PostgreSQL, and Jenkins are functioning correctly. Migration should be triggered by a specific operational need: the workload requires VolumeSnapshot backup (STORAGE-04), the workload needs RWX access, or the workload is being rebuilt as part of a major version upgrade. Migrating for its own sake adds risk without benefit.
jumphost — migrate Prometheus TSDB PVC from vsan-standard to ODF RBD
# Step 1: Scale down Prometheus StatefulSet to 0 replicas
[svc-ocp@jumphost ~]$ oc scale statefulset prometheus-k8s \
  -n openshift-monitoring --replicas=0
statefulset.apps/prometheus-k8s scaled

# Step 2: Create new ODF-backed PVC with the same size as the old one
[svc-ocp@jumphost ~]$ cat <<'EOF' | oc apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prometheus-k8s-db-prometheus-k8s-0-new
  namespace: openshift-monitoring
spec:
  storageClassName: ocs-storagecluster-ceph-rbd
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 50Gi
EOF
persistentvolumeclaim/prometheus-k8s-db-prometheus-k8s-0-new created

# Step 3: Run a data copy Job — mount both PVCs, rsync data from old to new
[svc-ocp@jumphost ~]$ cat <<'EOF' | oc apply -f -
apiVersion: batch/v1
kind: Job
metadata:
  name: prometheus-pvc-migrate
  namespace: openshift-monitoring
spec:
  template:
    spec:
      containers:
      - name: rsync
        image: harbor.apps.ocp.corp.novacrestbank.com/ubi9/ubi:latest
        command: [rsync, -av, /old/, /new/]
        volumeMounts:
        - mountPath: /old
          name: old-pvc
        - mountPath: /new
          name: new-pvc
      restartPolicy: OnFailure
      volumes:
      - name: old-pvc
        persistentVolumeClaim:
          claimName: prometheus-k8s-db-prometheus-k8s-0
      - name: new-pvc
        persistentVolumeClaim:
          claimName: prometheus-k8s-db-prometheus-k8s-0-new
EOF

# Wait for Job to complete, then update the StatefulSet to reference the new PVC
# (StatefulSet volumeClaimTemplates are immutable — requires deleting and recreating
# the StatefulSet with the new PVC name, or using --cascade=orphan)
[svc-ocp@jumphost ~]$ oc wait job/prometheus-pvc-migrate \
  -n openshift-monitoring \
  --for=condition=complete --timeout=600s
job.batch/prometheus-pvc-migrate condition met
  ↑ Data copy complete ✓ 
    Scale Prometheus back up, update PVC reference, delete old vSAN PVC.
