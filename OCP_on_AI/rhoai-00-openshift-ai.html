OpenShift AI

A complete architectural reference — from what artificial intelligence actually is, through the RHOAI component stack, GPU infrastructure, NovaCrestBank use cases, and the consulting questions that determine whether an AI platform engagement will succeed or fail
How to read this guide

This guide is written for the reader who is technically strong on OpenShift, Kubernetes, and enterprise infrastructure — but has not yet worked with AI/ML platforms. It starts from the very beginning: what artificial intelligence actually is, why it needs its own infrastructure, and what problem OpenShift AI solves. By the end, you should be able to explain the RHOAI component architecture to a customer who has never heard of it, identify the right RHOAI capability for a given customer problem, assess infrastructure readiness before an engagement begins, and know which questions to ask before any implementation work starts. Every concept is explained as if the reader is encountering it for the first time — because that is the correct way to build lasting clarity.

01 — What AI actually is — before any product

Before understanding OpenShift AI, you need a clear mental model of what artificial intelligence is — not the marketing definition, but the working definition that tells you what software is and is not "AI" in a technical context.

The traditional programming model

In traditional software, a human programmer writes rules. The programmer looks at a problem, understands the logic, and encodes that logic into instructions the computer follows. If you want software that detects fraudulent bank transactions, you would write rules: "if the transaction amount is greater than $10,000 AND the account is less than 30 days old AND the destination country has never received a transfer from this account, flag it as suspicious." The computer executes those rules exactly. It does what you told it to do — nothing more, nothing less.

This works well when the rules are knowable. Tax calculation, invoice processing, order routing — the logic is explicit and a human can write it down completely. But some problems have rules that are either too numerous to write, or not fully understood, or both. How do you write a rule for "is this transaction fraudulent"? A fraudster can always adapt to known rules. And the definition of fraud evolves continuously. The rule-based approach hits a wall.

The machine learning alternative

Machine learning inverts the programming model. Instead of a human writing rules for the computer to follow, the computer derives rules from examples. You show the system thousands of past transactions — labelled as either "legitimate" or "fraudulent" by human analysts — and the system finds the patterns in the data that distinguish one from the other. Those patterns become the "rules," expressed not as human-readable if-statements but as mathematical weights inside a model.

The key insight: the patterns the model discovers may be ones that no human analyst would ever articulate. The model might learn that a combination of 47 subtle signals — none individually suspicious — together strongly predict fraud. A human writing rules could not capture that. The model can.

The one-sentence definition that holds up under scrutiny

Artificial intelligence, in the practical software engineering sense, is the use of statistical patterns derived from data to make predictions or decisions — rather than using explicitly programmed rules. Everything else — neural networks, large language models, computer vision, fraud detection — is a specific technique for doing this particular thing at different scales and with different types of data.

02 — Machine learning — the core idea in plain terms

Machine learning is the subfield of AI that focuses on deriving patterns from data automatically. It is not magic. It is applied mathematics — specifically, optimisation. Understanding the basic mechanism removes the mystique and makes you a better advisor to customers who are evaluating AI platforms.

How a model actually learns — the two-minute version

A model starts with random weights — numbers initialised to arbitrary values. You feed it a training example: a transaction that is known to be fraudulent. The model makes a prediction — initially random, because the weights are random. The prediction is almost certainly wrong. The system then calculates how wrong it was (the "loss") and adjusts all the weights slightly in the direction that would have made the prediction less wrong. This adjustment is called a "gradient step." Repeat this millions of times across millions of examples, and the weights converge to values that make consistently accurate predictions on data the model has never seen before.

This process — called gradient descent — is the core of almost all modern machine learning. The mathematical details vary enormously between model types, but the fundamental loop is always: predict, measure error, adjust weights, repeat.

Why model format matters for platform decisions

The format a model is exported in determines which serving runtime can host it. ONNX (Open Neural Network Exchange) is the most portable format — a model trained in PyTorch, TensorFlow, or Scikit-learn can all be exported to ONNX and served by NVIDIA Triton, the default serving runtime in RHOAI. A model exported only as a TensorFlow SavedModel requires TensorFlow Serving. A model in GGUF format (quantised LLM) requires a different runtime (llama.cpp-based). Before any RHOAI engagement, the first question is always: what format is the model in? This single answer determines 30% of the architecture decisions that follow.

03 — MLOps — why models need their own platform

If a model is just a file, and inference is just an API call, why does it need an entire platform? Why can't a data scientist train a model on their laptop, export it, and deploy it as a container the same way any other microservice is deployed? The answer is that models have a lifecycle that is fundamentally different from application code — and that difference creates operational problems that standard DevOps tooling cannot solve.

The four ways models are different from application code

1. Models degrade over time without code changes. An application written correctly continues to work correctly unless someone changes the code or the environment. A model can degrade — silently, with no errors — simply because the world changed. A fraud detection model trained on 2022 transaction data may perform poorly in 2025 because fraud patterns evolved. The model file has not changed. The code that serves it has not changed. But the model is producing worse predictions. This is called "model drift" and it has no equivalent in traditional software. It requires continuous monitoring and periodic retraining — automated processes that standard CI/CD pipelines have no concept of.

2. Training requires different compute than inference. Training a model requires large amounts of GPU memory for long periods — minutes to days for large models. Inference requires low latency — milliseconds per prediction, potentially thousands of times per second. These are opposite compute profiles. A GPU node provisioned for training (large memory, tolerant of batch latency) is different from a GPU node provisioned for inference (fast, real-time, potentially multiple concurrent requests). The platform needs to manage both profiles.

3. Models have data dependencies, not just code dependencies. A model's behaviour is determined by the data it was trained on, not just its architecture. Two model files with identical architecture but trained on different datasets will behave completely differently. Versioning a model requires versioning the training data, the preprocessing steps, the hyperparameters, and the evaluation results — none of which are captured by a standard Git repository or container registry.

4. Reproducing a model requires reproducing a pipeline, not a build. If a security vulnerability is found in an application, the fix is: patch the code, build a new container, deploy. If a model is found to be biased or inaccurate, the fix is: collect better data, retrain with corrected preprocessing, evaluate on held-out data, validate fairness metrics, promote through staging, deploy. This is a pipeline, not a build step. It must be automated, auditable, and repeatable.

MLOps in one sentence

MLOps is the set of practices and tooling that applies DevOps principles — automation, reproducibility, versioning, monitoring — to the machine learning model lifecycle, accounting for the ways that lifecycle differs fundamentally from application software development. OpenShift AI is Red Hat's enterprise MLOps platform built on Kubernetes.

04 — ODH and RHOAI — upstream, downstream, and why it matters

Red Hat OpenShift AI (RHOAI) is built on Open Data Hub (ODH). Understanding the relationship between them prevents confusion about what you are installing, where features come from, and why the release cadence differs between the two.

Open Data Hub — the upstream community project

Open Data Hub is an open-source, community-driven AI/ML platform that runs on Kubernetes. It is the upstream project — the place where new features are developed, experimented with, and proven before being productised. ODH is roughly analogous to CentOS Stream (the upstream for RHEL) — it is where the work happens, it is usable, but it carries no enterprise support SLA, no guaranteed upgrade path, and no commitment to production stability.

ODH assembles existing open-source AI/ML tools into a cohesive platform: JupyterHub for notebooks, KServe for model serving, Kubeflow Pipelines for pipeline orchestration, MLflow for experiment tracking. ODH's contribution is the integration layer — making these independent projects work together on Kubernetes, with a unified UI (the ODH Dashboard) and a unified operator.

Red Hat OpenShift AI — the downstream supported product

RHOAI takes a curated, tested, and supported subset of ODH and packages it as a Red Hat product. The key differences from ODH:

Support
RHOAI: Red Hat enterprise support SLA ODH: Community only — file a GitHub issue
Release cadence
RHOAI: Versioned releases, tested against specific OCP versions ODH: More frequent, less predictable
Certification
RHOAI: Images from registry.redhat.io — CVE scanned, FIPS eligible ODH: Upstream container images — no CVE SLA
Feature set
RHOAI: Curated — only components that passed Red Hat QE ODH: All upstream components, including experimental
Installation
RHOAI operator from Red Hat OLM catalog ODH operator from OperatorHub community catalog
Licensing
RHOAI: Requires Red Hat OpenShift AI subscription ODH: Free — Apache 2.0
Why this matters in a consulting engagement

When a customer asks "can we use the free version instead of buying RHOAI?", the answer requires nuance. ODH is viable for a team of data scientists running experiments in a dev environment. It is not viable for a regulated banking institution running production fraud detection models — because the CVE responsibility, the upgrade path, and the support SLA all fall on the customer's internal team. For NovaCrestBank, the choice is RHOAI: the bank operates under APRA CPS 234 and PCI-DSS requirements that mandate enterprise support for components in production systems. A community-only supported AI platform would not pass a regulatory audit.

05 — Every RHOAI component — what it does and when you need it

RHOAI is not a single application. It is a set of components that can be individually enabled or disabled in the DataScienceCluster CR. Understanding each component independently — what problem it solves, what it does not do, and when a customer genuinely needs it — is essential for scoping an engagement correctly. Installing every component for every customer wastes resources and creates complexity without value.

Dashboard
RHOAI Dashboard
The web UI for RHOAI. Data scientists use it to create DataScienceProjects, launch Workbenches (Jupyter notebooks), deploy models, view pipeline runs, and access the model registry. Administrators use it to manage user permissions per project. Built as an OCP Route — accessible from any browser.
When to enable: Always. It is the primary interface for all users and administrators. There is no scenario where you install RHOAI without the Dashboard.
Workbenches
Workbenches (Jupyter)
A Workbench is a Jupyter notebook server running as a pod on OCP, with a persistent PVC for notebook storage. Data scientists write, run, and iterate on ML code inside Workbenches. RHOAI provides pre-built Workbench images (Standard Data Science, PyTorch, TensorFlow, CUDA-enabled variants for GPU-attached Workbenches). Each Workbench is scoped to a DataScienceProject.
When to enable: When the customer has data scientists who will write model code on the platform. If the customer's data scientists use external tools (SageMaker Studio, Databricks) and only deploy finished models to RHOAI, Workbenches may not be needed.
Model Serving
KServe (Single-Model Serving)
KServe hosts one model per InferenceService pod. Provides a dedicated REST and gRPC endpoint per model with full GPU resource allocation per model. Supports autoscaling including scale-to-zero (the model pod can stop when idle and restart on demand — important for cost control). Used when a model requires exclusive GPU access, consistent low latency, or when each model's endpoint needs to be independently scalable.
When to enable: When the customer has latency-sensitive inference requirements (sub-100ms) or when models require exclusive GPU allocation. NovaCrestBank's Fraud Detection System uses KServe for exactly this reason.
Model Serving
ModelMesh (Multi-Model Serving)
ModelMesh hosts many models inside a shared pool of serving pods. Models are loaded and evicted from GPU/CPU memory dynamically based on demand. Efficient when you have many models but not all are called frequently. One ModelMesh deployment can serve dozens of models using far fewer GPU resources than one KServe deployment per model. Latency is higher than KServe because models may need to be loaded on first request.
When to enable: When the customer has many models with infrequent or unpredictable traffic patterns, or when GPU resources are constrained and must be shared across models. NovaCrestBank's Document Intelligence System uses ModelMesh for its NLP extraction models.
Pipelines
Data Science Pipelines (DSP)
An OCP-native wrapper around Kubeflow Pipelines v2. A pipeline is a directed acyclic graph (DAG) of steps: ingest data, preprocess, train, evaluate, register model, deploy. Each step runs as a separate pod with its own container image, resources, and S3 artifact storage. Pipelines can be triggered manually, on a schedule, or by external events (Alertmanager webhook, S3 event). Pipeline run history is stored and browsable in the RHOAI Dashboard.
When to enable: When the customer needs to automate the model training and deployment process. Essential for production ML platforms where models are retrained periodically. Not needed if the customer only deploys pre-trained models from external sources.
Registry
Model Registry
A versioned catalogue of trained models with metadata: training dataset reference, hyperparameters, evaluation metrics, deployment history, and approval status. Data scientists register models here after successful training. MLOps engineers promote models from "Candidate" to "Production" through the registry. Provides the audit trail that regulated industries require — every model deployed to production has a traceable lineage back to its training run and dataset.
When to enable: When the customer operates in a regulated industry (banking, healthcare, insurance) or when multiple data scientists are producing models and governance of which model version is in production is required. Essential for NovaCrestBank — APRA CPS 230 model risk governance requires traceable model lineage.
TrustyAI
TrustyAI (Explainability & Fairness)
TrustyAI monitors deployed models for two properties: explainability (why did the model produce this prediction?) and fairness (does the model treat different demographic groups differently?). For explainability, it uses SHAP (SHapley Additive exPlanations) to produce per-prediction explanations: "this transaction was flagged because the merchant category had weight 0.42, the time of day had weight 0.31, and the deviation from usual spend pattern had weight 0.27." For fairness, it computes statistical parity metrics across protected attributes (age, gender, ethnicity).
When to enable: In regulated industries where model decisions affect individuals and regulators may require explanation of automated decisions. Also when the organisation has a model risk governance function. For NovaCrestBank — credit scoring and fraud flagging decisions can legally require explanation. Enable for any model whose decisions affect customers directly.
Distributed
Ray / CodeFlare
Ray is a distributed computing framework for Python workloads. In RHOAI, it enables distributed model training across multiple GPU nodes — training a single large model whose parameters do not fit in one GPU's memory by splitting the work across multiple GPUs across multiple nodes. CodeFlare is the Kubernetes operator that manages Ray clusters on OCP. Not needed for inference. Needed only for training very large models.
When to enable: When the customer needs to train large models (LLMs, large vision transformers) that require multi-GPU distributed training. Not needed for inference or for training smaller models that fit on a single GPU. NovaCrestBank's Phase 1 use cases (FDS, KDIS) use pre-trained or smaller models — Ray is not required.

06 — KServe — the model serving engine in depth

KServe is the most important component to understand deeply, because it is where the model meets production traffic. Every inference request that a banking application sends to a model goes through KServe. Getting the KServe configuration right determines whether the AI platform delivers the latency, availability, and scalability the business requires.

How KServe works — the architecture

When you create an InferenceService object in Kubernetes (the KServe CR), KServe does four things automatically. First, it downloads the model from wherever it is stored — an S3-compatible object store (ODF MCG, AWS S3, MinIO), an OCI registry (Harbor), or a PVC. Second, it starts a serving runtime container — a process that loads the model into memory and exposes a REST and gRPC API that accepts inference requests. Third, it creates a Kubernetes Service and an OCP Route that expose the inference endpoint to callers. Fourth, it configures a Horizontal Pod Autoscaler (HPA) that scales the number of serving runtime pods up or down based on request rate — including all the way down to zero when no requests arrive.

The three parts of an InferenceService

An InferenceService has three optional components. Only the predictor is mandatory.

Predictor — the component that actually loads the model and serves predictions. You specify the model format (ONNX, TensorFlow, TorchScript, HuggingFace) and the serving runtime to use. KServe routes inference requests to the predictor. This is the minimum viable InferenceService.

Transformer — an optional pre- and post-processing layer that runs before the predictor receives the request and after it returns the response. Use it when the raw model input/output format differs from what the calling application sends/expects. For example: a banking application sends a JSON payload with a transaction object. The predictor expects a numerical feature vector. The transformer converts between the two. This keeps preprocessing logic out of the calling application and inside the AI platform boundary.

Explainer — an optional component that, when called, returns an explanation of why the predictor produced its output for a specific input. Powered by TrustyAI. Used by the NovaCrestBank model risk team when they need to audit a specific fraud flagging decision.

Serving runtimes — what they are and which to choose

A ServingRuntime is a container that knows how to load a specific model format and serve it via a standardised API. RHOAI ships with pre-configured ClusterServingRuntimes. The choice of runtime is determined by the model format.

07 — Data Science Pipelines — the automation engine in depth

Data Science Pipelines (DSP) is how the model training, evaluation, and deployment process is automated. Without pipelines, a data scientist trains a model manually in a Workbench, manually evaluates it, manually registers it, and manually deploys it. This is viable for a team of two in a proof-of-concept. It is not viable for a production banking platform where models must be retrained when drift is detected, where every training run must be auditable, and where a bad model must be automatically rejected before it reaches production.

What a pipeline looks like — conceptually

A pipeline is a Python function decorated with KFP (Kubeflow Pipelines) SDK annotations. The annotations tell the pipeline engine: this function is a pipeline step, it takes these inputs, it produces these outputs, it should run in this container image, and it should request these compute resources. The pipeline engine translates the decorated Python into a pipeline definition (YAML) that describes the DAG — which steps depend on which other steps, what data flows between them, and what to do when a step fails.

When a pipeline is compiled and uploaded to RHOAI, it appears in the Dashboard. A PipelineRun is an execution of a pipeline — like a CI/CD job run. The RHOAI Dashboard shows all PipelineRuns, their status per step, their logs, and their output artifacts. Every PipelineRun is stored with its inputs, outputs, and the model registry entry it produced — creating a complete audit trail from trigger to deployed model.

Pipeline artifact storage — why ODF MCG is the right choice

Each pipeline step produces artifacts — intermediate datasets, preprocessed feature files, trained model files, evaluation reports. These artifacts are stored in an S3-compatible object store between steps. RHOAI's DSP operator creates an ObjectBucketClaim in ODF MCG automatically when a DataSciencePipelines component is enabled. This is the correct integration for NovaCrestBank: the pipeline artifact store is on the same OCP cluster as the pipeline workloads, under the same ODF storage that backs VM disks and snapshots. No external S3 dependency, no data leaving the cluster boundary — which satisfies APRA CPS 234 data sovereignty requirements for model training data.

Pipelines vs CI/CD — the relationship a platform engineer needs to understand

Data Science Pipelines are not a replacement for Argo CD or Tekton. They solve a different problem. Argo CD manages the deployment of Kubernetes manifests — including InferenceService CRs. DSP manages the process of producing the model that an InferenceService will serve. The two tools work in sequence: DSP trains and registers the model → the Model Registry is updated → Argo CD detects the new model version in the InferenceService CR → Argo CD deploys the updated InferenceService. Neither tool duplicates the other's function.

08 — Why AI needs a GPU — explained from physics up

This is the question that platform engineers ask most often: why can't we run the AI workloads on the same CPUs that run everything else? The answer is not "because GPUs are faster." The answer is more specific — and understanding it correctly prevents expensive mistakes in infrastructure planning.

What a CPU is optimised for

A CPU is designed to execute instructions as quickly as possible, one after another, with low latency. A modern server CPU has between 16 and 128 cores. Each core is a highly complex execution unit with deep out-of-order execution pipelines, large caches, and sophisticated branch prediction — all designed to minimise the time between when an instruction is issued and when its result is available. CPUs excel at tasks where instructions depend heavily on each other (the result of step 3 is needed to compute step 4) and where the sequence of operations branches unpredictably (if/else logic, function calls, event-driven code). This describes most application software — web servers, database engines, operating system kernels.

What a GPU is optimised for

A GPU is designed to execute the same operation on thousands of data points simultaneously — not sequentially. A modern data centre GPU (NVIDIA A100) has 6,912 CUDA cores. Each core is far simpler than a CPU core — it cannot do complex branching, it has minimal cache, and its per-core performance on sequential tasks is much lower than a CPU core. But 6,912 cores all operating in parallel on the same mathematical operation can process data at a throughput that no CPU array can match.

This matters for machine learning because the fundamental operation in both training and inference is matrix multiplication. When a model processes an input, it multiplies the input vector by a large matrix of weights — potentially millions or billions of multiplications that are all independent of each other. Every multiplication can happen simultaneously. A CPU performs these multiplications sequentially across its 64–128 cores. A GPU performs them all at once across its 6,912 cores. For a matrix operation with 1 million multiplications, the GPU is not "faster" — it is operating on a different axis entirely.

The concrete numbers that make the case to a customer

NVIDIA A100 80GB: 312 TFLOPS (Tensor Float 32) for AI workloads. A high-end Intel Xeon Scalable CPU: ~3–5 TFLOPS. For inference throughput on a transformer-based NLP model (BERT-large): a single A100 can process approximately 3,000–5,000 inference requests per second. The same workload on CPU: approximately 50–150 requests per second. The GPU is not "60× faster" — it operates at a completely different scale. For NovaCrestBank's FDS fraud scoring, which must return a risk score within the payment authorisation timeout (typically 300ms round-trip including network), GPU inference makes the latency budget achievable at production transaction volumes. CPU inference does not.

Training vs inference — different GPU requirements

Training requires maximum GPU memory and compute — the model weights, the gradients, and the optimizer state all live in GPU memory simultaneously. An NVIDIA A100 80GB can train a model with up to approximately 10 billion parameters in 16-bit precision. Training runs for hours to days. During training, the GPU is fully utilised continuously.

Inference requires low latency and potentially high throughput — the model weights live in GPU memory, and inference requests are processed as they arrive. An A100 80GB serving a fraud detection model (typically 100M–500M parameters) will have substantial free GPU memory — which is why multiple models can share a GPU in ModelMesh mode, or why a single GPU node can serve multiple concurrent InferenceService deployments if they are small models.

Why the DL385 Gen11 was chosen for this — reference to VIRT-00

This decision was made deliberately in VIRT-00 Section 04 — before OpenShift AI was built. The DL385 Gen11 nodes provide 6 × PCIe 5.0 ×16 slots per node, reserved explicitly for future GPU expansion. The rationale at the time: NVIDIA and AMD data centre GPUs use 16 PCIe 5.0 lanes per card. Nodes with fewer PCIe lanes cannot host multiple GPUs without contention. By choosing DL385 Gen11 for OpenShift Virtualization, NovaCrestBank preserved the option to add GPU cards to existing servers for OpenShift AI — without a hardware refresh. The two nodes selected for GPU expansion in Series VI (bm-virt-04 and bm-virt-05) receive one NVIDIA A100 80GB PCIe card each, using one of their six available PCIe 5.0 ×16 slots. The remaining five PCIe slots on each node remain available for additional GPUs as the AI workload grows.

09 — NVIDIA GPU Operator — every component explained

Installing a GPU card in a server and plugging it into an OCP cluster does not make the GPU usable by Kubernetes pods. Kubernetes has no native awareness of GPU hardware. Making a GPU schedulable as a Kubernetes resource requires software components that span the host OS, the container runtime, and the Kubernetes scheduler. The NVIDIA GPU Operator installs and manages all of these components automatically.

Why you cannot just install the NVIDIA driver the normal way on RHCOS

On a standard RHEL server, you install the NVIDIA driver with dnf install nvidia-driver, reboot, and the GPU is available. On RHCOS (Red Hat CoreOS) — the OS that runs on all OCP nodes — this approach does not work. RHCOS is an immutable operating system. Its root filesystem is read-only and managed by the Machine Config Operator. You cannot install packages with dnf. You cannot modify kernel modules directly. The OS image is baked and updated atomically. The NVIDIA GPU Operator solves this by running the NVIDIA driver inside a privileged container on the node — a "driver container" that mounts the host filesystem and loads the kernel module without permanently modifying the host OS. When the GPU Operator is updated, the driver container is updated, and the new driver is loaded without touching the RHCOS image itself. This is the only supported approach for GPU drivers on RHCOS.

The four components the GPU Operator installs and manages

1. Node Feature Discovery (NFD)

NFD is a Kubernetes operator that scans every node's hardware and labels the node with what it finds. When bm-virt-04 has an NVIDIA A100 PCIe card installed, NFD automatically adds labels including nvidia.com/gpu.present=true, nvidia.com/gpu.product=A100-PCIE-80GB, and nvidia.com/gpu.count=1. These labels are how the GPU Operator knows which nodes need GPU driver software, and how KServe InferenceService pods know which nodes to schedule on. NFD must be installed and healthy before the GPU Operator can do anything useful.

2. GPU Driver Container

The GPU Operator deploys a DaemonSet pod on every node labelled nvidia.com/gpu.present=true. This pod runs the NVIDIA driver installation process: it mounts the host's /run/nvidia directory, compiles and loads the NVIDIA kernel module (nvidia.ko), and makes the GPU device files (/dev/nvidia0, /dev/nvidiactl, /dev/nvidia-uvm) available on the host. After this pod completes successfully, the node can access the GPU at the hardware level. The driver version is set in the ClusterPolicy CR and the GPU Operator manages upgrades — updating the driver container image updates the GPU driver across all GPU nodes without requiring node reboots in most cases.

3. NVIDIA Device Plugin

The Device Plugin is a Kubernetes DaemonSet that registers the GPU as a schedulable resource with the kubelet on each GPU node. After the Device Plugin runs, the node's capacity includes nvidia.com/gpu: 1. This is how Kubernetes knows the node has a GPU and how pods can request GPU resources. A pod that declares resources.requests: nvidia.com/gpu: 1 will only be scheduled on a node where the Device Plugin has registered a GPU. Without the Device Plugin, Kubernetes cannot schedule GPU workloads — it has no awareness that the GPU exists even if the driver is installed.

4. DCGM Exporter (Data Centre GPU Manager)

DCGM is NVIDIA's library for monitoring and managing GPUs at the datacenter level. The DCGM Exporter runs as a DaemonSet on GPU nodes and exposes GPU metrics in Prometheus format on port 9400. The metrics it exports include: GPU utilisation (% of GPU compute cycles being used), GPU memory used/total, GPU temperature, GPU power draw, GPU clock speed, and PCIe bandwidth. These metrics are scraped by the cluster Prometheus and can be displayed in Grafana dashboards or used in PrometheusRules (e.g., alert when GPU temperature exceeds 85°C for more than 2 minutes). For NovaCrestBank: DCGM metrics allow the platform team to know whether the FDS inference workload is GPU-bound, whether the GPU is running hot under sustained load, and whether the A100 cards are being utilised or sitting idle.

10 — Fraud Detection System — architecture decisions

NovaCrestBank's Fraud Detection System (FDS) scores every payment transaction in real time and returns a risk score before the payment is authorised. The banking application vm-banking-app-01 — which runs inside an OCP Virt KubeVirt VM — makes an API call to the FDS model endpoint for every transaction. The FDS must return a score within the payment authorisation timeout. This is a latency-critical, high-throughput inference problem. Every architecture decision below follows from those two constraints.

The business requirement — stated precisely

RequirementValueWhy it matters architecturally
Inference latency (p99)< 80msThe payment authorisation round-trip must complete within 300ms. Network adds ~20ms. Application processing adds ~50ms. The model must score in <80ms at the 99th percentile — not on average. A fast average with a slow tail kills the user experience.
Throughput500 transactions/second peakNovaCrestBank's peak payment volume occurs Friday afternoon. At 500 TPS, the inference endpoint must handle 500 concurrent (or batched) requests per second without degradation.
Availability99.99% (4 nines)Fraud detection is on the critical path of every payment. If the FDS endpoint is down, payments cannot be processed. This is not a best-effort service.
Model formatONNXThe data science team exports models to ONNX from their PyTorch training pipeline. ONNX is the portability choice — it allows the serving runtime to be changed without retraining.
Audit trailEvery decision explainableAPRA CPS 230 and AUSTRAC AML obligations require that automated fraud decisions be explainable to regulators on request. TrustyAI must be enabled for FDS.

Architecture decision 1: KServe single-model serving, not ModelMesh

ModelMesh is more GPU-efficient — it shares GPU resources across models. But ModelMesh has higher latency because models may be evicted from GPU memory when idle and must be reloaded on the next request. The FDS model is called at high frequency continuously during banking hours. It will never be evicted. ModelMesh's efficiency advantage does not apply. KServe single-model serving is the correct choice because it gives the FDS model a dedicated GPU allocation, a dedicated serving runtime, and a dedicated endpoint — with no competing workloads on the same inference pod.

Architecture decision 2: NVIDIA Triton as the ServingRuntime

Triton is chosen over OpenVINO (CPU) and TorchServe because: the model is ONNX format (Triton supports ONNX natively), a GPU is available (Triton uses CUDA directly for maximum GPU utilisation), and Triton provides dynamic batching. Dynamic batching is critical at 500 TPS: instead of processing each of 500 transactions per second individually, Triton accumulates requests arriving within a 2ms window and processes them as a single GPU batch. This reduces the number of GPU kernel launches from 500/second to approximately 50/second, dramatically improving throughput and GPU utilisation.

Architecture decision 3: the Transformer component for preprocessing

The raw transaction payload from vm-banking-app-01 is a JSON object with 30+ fields — merchant name, amount, account ID, timestamp, geolocation, device fingerprint, and so on. The ONNX model expects a fixed-length numerical feature vector of 128 dimensions. The conversion — feature engineering, encoding categorical variables, normalising numerical fields — must happen somewhere. The options are: in the calling application (vm-banking-app-01), or in the KServe Transformer component. The correct answer is the Transformer. Putting feature engineering in the calling application creates a coupling: every time the model's feature set changes, the banking application must be updated. The Transformer keeps this logic in the AI platform boundary — the banking application always sends the same transaction JSON, and the Transformer handles conversion to whatever the model expects.

Architecture decision 4: two replicas minimum, autoscaling to eight

A single InferenceService pod has a single GPU allocation. At 500 TPS with dynamic batching, one A100 is sufficient for the compute load. But availability requires at least two replicas — if one pod is evicted, the other continues serving. At the maximum autoscaling target of eight replicas, the FDS can handle sudden traffic spikes (fraud events often coincide with unusual transaction volumes) by distributing across both GPU nodes (bm-virt-04 and bm-virt-05), with pod anti-affinity ensuring replicas spread across nodes.

The FDS InferenceService — annotated

kind: InferenceService
metadata:
  name: fds-v2-onnx

    # DataScienceProject namespace — scoped to the FDS team.

  # vm-banking-app-01 is in virt-workloads namespace — cross-namespace NetworkPolicy
  # permits the banking VM to reach this endpoint (see Section 12).
  annotations:
    serving.kserve.io/deploymentMode: RawDeployment
        # RawDeployment: KServe deploys a standard Kubernetes Deployment, Service,
    # and HPA — not Knative Serving. RawDeployment is required when:
    # 1. Knative is not installed on the cluster (NovaCrestBank does not use Knative)
    # 2. Scale-to-zero is not desired (FDS must always have at least 2 replicas)
    # Serverless mode uses Knative and enables scale-to-zero — appropriate for
    # low-frequency models, not for FDS which is on the payment critical path.
spec:
  predictor:
    minReplicas: 2
        # minReplicas: 2 — always maintain 2 serving pods.
    # Even at zero traffic, 2 pods are running so the endpoint is always available.
    # Anti-affinity (below) ensures they are on different nodes.
    maxReplicas: 4
        # maxReplicas: 4 — maximum 4 pods during traffic spikes.
    # Limited to 4 because NovaCrestBank has 2 GPU nodes × 1 GPU each = 2 physical GPUs.
    # Using MIG (not configured here) would allow more replicas per GPU.
    # Each replica requests nvidia.com/gpu: 1 — 4 replicas would exceed GPU capacity.
    # In practice: 2 replicas during normal hours, 4 only during extreme spikes
    # where MIG partitioning would be needed. Set conservatively for Phase 1.
    scaleTarget: 80
        # scaleTarget: target GPU utilisation (%) for HPA scaling decisions.
    # When average GPU utilisation across pods exceeds 80%, add a replica.
    # Below 40%, remove a replica (down to minReplicas).
    model:
      modelFormat:
        name: onnx
            # modelFormat: tells KServe what format the model file is in.
      # KServe uses this to select a compatible ServingRuntime automatically
      # (unless runtime is specified explicitly, as it is below).
      runtime: triton-23-10
            # runtime: explicitly specifying NVIDIA Triton as the ServingRuntime.
      # triton-23-10 is the ClusterServingRuntime name pre-configured by RHOAI.
      # RHOAI ships several pre-built ClusterServingRuntimes — you can also
      # create custom ServingRuntime CRs for runtimes RHOAI does not include.
      storageUri: s3://rhoai-models/fds/v2.3/model.onnx

      # s3://rhoai-models: the ODF MCG bucket configured in the RHOAI DataScienceCluster.
      # fds/v2.3/model.onnx: the specific model version path in the bucket.
      # KServe's storage initialiser container downloads this file at pod startup.
      # To deploy a new model version: update this path and apply.
      # The Argo CD GitOps workflow manages this — a model version bump is a Git commit.
      resources:
        limits:
          nvidia.com/gpu: "1"
                    # Request 1 full A100 GPU per replica.
          # The GPU Operator device plugin enforces this — the pod will not
          # schedule unless a node has 1 free nvidia.com/gpu resource.
          # Without this limit: the pod schedules on a CPU node and
          # Triton falls back to CPU inference — 50× slower, latency SLA missed.
          cpu: "4"
          memory: 16Gi
                    # CPU and memory for the Triton process itself (not the GPU).
          # Triton uses CPU for request routing, batching logic, and
          # pre/post-processing outside the GPU kernel. 4 CPU cores and 16Gi
          # is appropriate for 500 TPS with dynamic batching.
      affinity:

                      # Without this: Kubernetes might schedule on a non-GPU node
                      # because the GPU request is already satisfied by the limit.

                  serving.kserve.io/inferenceservice: fds-v2-onnx
              topologyKey: kubernetes.io/hostname
                            # Hard anti-affinity: no two FDS pods on the same node.
              # With minReplicas: 2, this ensures the two pods are always on
              # bm-virt-04 AND bm-virt-05 — never both on the same node.
              # If bm-virt-04 fails, bm-virt-05's pod continues serving.
              # This is how the 99.99% availability requirement is met.
  transformer:

    # to the 128-dimension numerical feature vector the ONNX model expects.
    containers:
      - image: harbor.apps.ocp.novacrestbank.local/ai/fds-transformer:v2.3
                # Custom transformer image built from the FDS team's Python code.
        # Stored in the internal Harbor registry — not pulled from the internet.
        # The transformer implements the KServe v2 inference protocol:
        # it receives the JSON request, preprocesses it to a feature vector,
        # forwards to the predictor, receives the score, and returns it.
        name: fds-transformer
        resources:
          limits:
            cpu: "2"
            memory: 4Gi
                        # Transformer runs on CPU — feature engineering is CPU work.
            # No GPU requested for the transformer component.

11 — Document Intelligence System — architecture decisions

NovaCrestBank's KYC Document Intelligence System (KDIS) processes identity documents and loan application forms submitted by customers — passports, driver's licences, bank statements, payslips. The system extracts structured data (name, date of birth, address, account numbers, income figures) from scanned images and handwritten or printed text, and writes the extracted fields into the loan origination system. Unlike FDS, KDIS is not on a real-time critical path — documents are submitted and results are expected within minutes, not milliseconds. This changes every architecture decision.

Why KDIS is an async pipeline, not a synchronous endpoint

A customer submitting a loan application uploads several documents — passport, bank statements (12 months), payslips (3 months). Processing all of them sequentially in a synchronous API call would hold an HTTP connection open for 30–120 seconds. This is poor user experience and violates typical gateway timeouts. The correct architecture is: customer submits documents → documents are stored in S3 (ODF MCG) → a pipeline is triggered → pipeline steps process each document type in parallel → extracted data is written to the database → customer sees a "processing complete" status update in the banking portal. The pipeline is the unit of work, not the API call.

Why ModelMesh over KServe for KDIS

KDIS uses three different models in sequence: an OCR model (extracts text from images), a document classification model (identifies what type of document was submitted — passport vs bank statement), and an NLP extraction model (parses the extracted text into structured fields). These three models are called as pipeline steps — not in real time per user request. The frequency of each model call is determined by document submission volume, not transaction rate. ModelMesh is the correct choice because: (1) the three models can share GPU resources rather than each requiring a dedicated GPU; (2) when no documents are being processed, the models can be evicted from GPU memory and GPU resources freed for FDS; (3) the latency tolerance of the pipeline (minutes, not milliseconds) makes ModelMesh's eviction latency (seconds to reload a model) irrelevant.

12 — Integration with the existing NovaCrestBank platform

This is the section that distinguishes an architectural reference from a product tutorial. RHOAI does not exist in isolation — it integrates with every platform component built in Series I, II, and III. Understanding these integration points is what a consulting architect brings to an engagement that a data scientist or an RHOAI product specialist cannot.

Harbor — model images and transformer containers

Harbor is already running at harbor.apps.ocp.novacrestbank.local from the Series II platform. For RHOAI, Harbor serves two purposes. First, all custom container images (the FDS Transformer, the KDIS pipeline step containers) are built by the CI/CD pipeline and pushed to Harbor before being referenced in InferenceService and pipeline specs. This keeps all container images in the air-gapped internal registry — no images are pulled from the internet during serving. Second, model files can be stored as OCI artifacts in Harbor (the OCI artifact format allows arbitrary blobs to be stored in a container registry). This means a model version and its transformer container image can be versioned and stored together in Harbor under the same tag, providing a single source of truth for a complete model deployment unit.

Argo CD — GitOps for model deployment

InferenceService CRs are Kubernetes objects. Like all Kubernetes objects in NovaCrestBank's environment, they are managed by Argo CD. The two-repository pattern from VIRT-07 extends to RHOAI: the banking-config Git repository contains the InferenceService YAML for FDS and KDIS. When the data science team promotes a new model version — updating the storageUri in the InferenceService YAML via a Git commit and pull request — Argo CD detects the change and applies the updated InferenceService to the cluster. KServe then downloads the new model version and performs a rolling update of the serving pods with zero downtime. The model deployment is auditable (every version is a Git commit), reviewable (pull requests require approval), and rollback is a Git revert.

ACM — governance policies for the AI platform

ACM Governance policies (from VIRT-09) extend to RHOAI. Two policies are relevant. The first enforces GPU resource quotas per DataScienceProject — preventing a data scientist from accidentally requesting all available GPU resources for a Workbench experiment, starving the FDS production serving pods. The second enforces that TrustyAI is enabled in every DataScienceProject that contains a production InferenceService — ensuring explainability monitoring is never absent from a production model deployment. If a data scientist creates a production InferenceService without TrustyAI, ACM detects the non-compliant state and alerts the platform team.

ACS — model container image scanning

ACS (from VIRT-08) scans all container images in Harbor for CVEs. This includes the RHOAI component images (from registry.redhat.io, scanned on pull), the ServingRuntime images (Triton, TGI), and the custom transformer images. ACS's admission controller can be configured to block deployment of an InferenceService if its transformer container image has unpatched critical CVEs — the same admission control that prevents vulnerable application containers from reaching production prevents vulnerable model transformer containers from reaching production.

AAP — pipeline triggers from Alertmanager

The alert-driven automation pipeline from VIRT-10 extends to model retraining. When Prometheus detects that the FDS model's prediction accuracy has dropped below a threshold (a custom PrometheusRule monitoring a DCGM/TrustyAI metric exported to the cluster Prometheus), Alertmanager fires a webhook to EDA. EDA's Rulebook matches on alertname == "FDSModelDriftDetected" and triggers an AAP Job Template that submits a new KDIS/FDS training PipelineRun via the RHOAI API. The pipeline runs, trains a new model version, evaluates it, registers it in the Model Registry, and if it passes evaluation thresholds, creates a pull request in the banking-config Git repository that updates the InferenceService storageUri. A human approves the pull request, Argo CD applies it, and the new model is in production — with a complete audit trail from the drift alert through the PR to the deployment.

ODF — model artifact storage and pipeline state

ODF's MCG (NooBaa object gateway) provides the S3-compatible object store that backs both the Model Registry artifact storage and the Data Science Pipelines artifact store. NovaCrestBank benefits from using ODF for this rather than AWS S3 or external MinIO: model weights never leave the OCP cluster boundary (data sovereignty for APRA CPS 234 purposes), the same storage platform that backs VM disks and OADP backups backs model artifacts (single operational surface), and ODF's erasure coding and replication provide the same durability guarantees for model artifacts as for application data.

NetworkPolicy — isolating inference traffic

RHOAI InferenceService pods run in DataScienceProject namespaces (fraud-detection, document-intelligence). The banking application VM runs in virt-workloads. Without a NetworkPolicy, cross-namespace traffic is unrestricted on the OVN overlay — any pod in any namespace can reach any other pod. The correct configuration adds one NetworkPolicy in the fraud-detection namespace that permits ingress on port 8080 (REST) and 8081 (gRPC) only from pods in the virt-workloads namespace. This ensures the FDS endpoint is reachable from the banking VM but not from any other namespace — including a compromised Workbench pod in the same DataScienceProject.

13 — Infrastructure prerequisites before an engagement

This is the checklist a consulting architect runs through before any RHOAI work begins at a customer site. Many RHOAI engagements fail not because of RHOAI configuration errors but because the underlying infrastructure was not ready before the AI platform was installed. Every item below corresponds to a real failure mode observed in production deployments.

OCP cluster prerequisites
OCP version compatibility: RHOAI 2.x requires OCP 4.12 or later. Verify the exact compatibility matrix for the RHOAI version being installed — some RHOAI minor versions require specific OCP minor versions. Run oc get clusterversion and cross-check against the RHOAI release notes before installing the operator.
All cluster operators healthy: RHOAI operator installation will appear to succeed even if underlying cluster operators are degraded — but RHOAI components will fail to become ready. Run oc get clusteroperators | grep -v "True.*False.*False" and resolve any degraded operators before proceeding.
Sufficient cluster resource headroom: RHOAI control plane components (the operator, the Dashboard, the DSP controller, the Model Registry) consume approximately 8 CPU cores and 16 GiB of memory across their pods. Verify this headroom exists on non-GPU worker nodes before installation. Installing RHOAI on a cluster at 90% CPU utilisation will cause Pending pods and mysterious failures.
Default storage class: RHOAI's DSP operator creates PVCs automatically. The cluster must have a default StorageClass. Verify: oc get storageclass | grep default. In NovaCrestBank: ocs-storagecluster-ceph-rbd is the default.
S3-compatible object store accessible from OCP: RHOAI requires an S3 bucket for the Model Registry and for DSP pipeline artifact storage. In NovaCrestBank: ODF MCG (NooBaa) is configured and an ObjectBucketClaim is created in the RHOAI namespace. In a customer environment without ODF: an external S3 endpoint (AWS S3, MinIO, Ceph RGW) must be reachable from the OCP cluster network before RHOAI installation.
GPU infrastructure prerequisites
GPU cards physically installed and powered: Confirm via iLO/BMC that the GPU cards are physically seated, PCIe x16 negotiated, and the server firmware can see them. In NovaCrestBank: iLO 6 on bm-virt-04 and bm-virt-05 shows the A100 in the PCIe device inventory.
Node Feature Discovery installed before GPU Operator: NFD must label nodes with GPU hardware before the GPU Operator can target those nodes for driver installation. The order matters. Installing the GPU Operator before NFD results in no DaemonSet pods scheduling on GPU nodes — the operator sees no GPU-labelled nodes.
Kernel version compatible with the GPU driver version: NVIDIA publishes a compatibility matrix of driver version to kernel version. The RHCOS kernel version on OCP nodes is not chosen by the customer — it is set by OCP. Check the kernel version of your OCP nodes (oc debug node/bm-virt-04 -- chroot /host uname -r) and the NVIDIA GPU Operator release notes to confirm the driver version in ClusterPolicy is compatible with that kernel.
GPU node firmware up to date: NVIDIA A100 PCIe cards have firmware that must be compatible with the driver version. Outdated GPU firmware can cause ECC memory errors, driver load failures, or silent compute errors. Verify firmware version via iLO or nvidia-smi -q | grep Firmware after driver installation.
Power budget confirmed: An NVIDIA A100 PCIe 80GB card draws up to 250W under load. Two cards in bm-virt-04 and bm-virt-05 add up to 500W per node at peak, on top of the existing CPU (280W TDP for dual EPYC 9354P) and NVMe (25W × 4). Confirm the node's power supply capacity (the DL385 Gen11 ships with 2 × 1000W or 2 × 1600W PSUs — verify which configuration is installed) and the data centre PDU circuit capacity.
Network and registry prerequisites
Internal container registry reachable from all nodes: RHOAI serving runtime images (Triton, TGI) are large — 10–20 GB each. In an air-gapped environment, these must be mirrored to Harbor before installation. Verify Harbor is reachable from GPU nodes and that the pull-through cache is populated with the required images. A failed image pull on a GPU node during serving deployment is difficult to diagnose if the registry prerequisite was not checked.
DNS resolution for model storage endpoint: The KServe storage initialiser downloads models from the storageUri at pod startup. If DNS resolution for the S3 endpoint (ODF MCG route, or external S3 hostname) fails from the GPU node network, the pod fails to start with an unhelpful "model download failed" error. Verify DNS resolution: oc debug node/bm-virt-04 -- chroot /host nslookup rhoai-models.apps.ocp.novacrestbank.local.

14 — Questions to ask before committing to an implementation approach

These are the questions that determine the architecture before any YAML is written. A consulting architect who asks these questions in the first engagement conversation avoids discovering late-stage blockers during implementation. Each question is paired with the architectural decision it determines.

Q1 — Model format
"What format are your models exported in — ONNX, TensorFlow SavedModel, TorchScript, HuggingFace, or something else?"

This determines the ServingRuntime. ONNX → Triton or OpenVINO. TensorFlow → TF Serving or Triton. HuggingFace Transformers → TGI. Custom/proprietary → custom ServingRuntime required (longer implementation, custom image build). If the customer says "we don't know yet — our data scientists are still deciding" — the architecture cannot be finalised. Do not commit to a serving runtime choice until the model format is confirmed.
Q2 — Latency requirement
"What is the maximum acceptable inference latency at the 99th percentile, and where is that measured — at the model endpoint, or end-to-end from the calling application?"

This determines KServe vs ModelMesh, GPU vs CPU inference, and whether dynamic batching can be used. If the answer is "we're not sure, real-time I suppose" — probe further: "Is the model on the critical path of a user-facing transaction, or is it used for background processing?" Critical path → KServe, GPU, tight latency budget. Background → ModelMesh, CPU may suffice, batch processing acceptable.
Q3 — Throughput requirement
"How many inference requests per second do you need to handle at peak? And what is the peak-to-average ratio?"

This determines replica count, HPA configuration, and whether a single GPU is sufficient. A customer who says "500 requests per second" but whose peak is a 10-second burst once a day needs a different configuration than one whose peak is sustained for 4 hours. Scale-to-zero (Knative Serverless mode) is appropriate for bursty, low-average-throughput workloads. Dedicated replicas (RawDeployment mode) are appropriate for sustained high-throughput workloads.
Q4 — Model size
"How large is the model file — in gigabytes — and how much GPU memory does it consume when loaded?"

This determines GPU memory requirements and whether MIG partitioning is feasible. A 500MB ONNX model uses approximately 1–2 GB GPU memory when loaded by Triton. A 7B parameter LLM in FP16 uses approximately 14 GB GPU memory. An A100 80GB can host approximately 5–6 medium models or 1 large LLM simultaneously. If the customer cannot answer this question, ask for the model's parameter count and precision (FP32 / FP16 / INT8) — GPU memory ≈ (parameters × bytes per parameter) × 1.2 overhead factor.
Q5 — Retraining cadence
"How often do you expect to retrain and redeploy the model? Is retraining triggered by a schedule, by detected drift, or by the data science team on an ad-hoc basis?"

This determines whether Data Science Pipelines is needed. If the answer is "once a quarter, manually" — Pipelines is useful but not critical; the team can run training in a Workbench and manually register the result. If the answer is "automatically when we detect performance degradation" — Pipelines plus the Alertmanager integration (Section 12) is required. If the answer is "we don't retrain — we use a vendor-supplied model that is updated by the vendor" — Pipelines is not needed at all.
Q6 — Explainability and audit requirements
"Are the model's decisions subject to regulatory review or customer-facing explanation requirements? Can a regulator ask 'why was this transaction flagged' or 'why was this loan declined' and expect an answer?"

This determines whether TrustyAI is required. In banking, credit scoring and fraud flagging typically trigger explanation obligations under consumer credit law (NCCP in Australia, FCRA in the US) and AML regulations. If the customer's legal team has not addressed this question, surface it — a production model making customer-affecting decisions without explainability infrastructure is a regulatory risk.
Q7 — Data location constraints
"Are there data sovereignty or data residency requirements for the training data and the model weights? Can model artifacts leave the data centre boundary — for example, can they be stored in a cloud object store?"

This determines the artifact storage architecture. If training data contains personally identifiable information (PII) — which it does in NovaCrestBank's case, since transaction data and KYC documents are used for training — data sovereignty requirements may prohibit storing model artifacts in cloud object stores. ODF MCG (on-cluster, on-premises S3) is the correct choice for NovaCrestBank. A customer without this constraint can use AWS S3, simplifying the storage architecture.

The last row matters most

The most valuable consulting intervention is recognising when a customer is not ready for an AI platform at all. An organisation that does not yet have labelled training data, does not have a defined decision they want to automate, or does not have a clear success metric for model performance is not ready to deploy RHOAI. Deploying the platform before the use case is clear produces an expensive, underutilised infrastructure — and damages the customer's trust in both the technology and the vendor. The correct response to "we want to do AI" is a discovery engagement — not a platform deployment. This is the conversation that separates a consulting architect from a solutions engineer who sells licenses.
