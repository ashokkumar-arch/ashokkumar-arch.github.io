What carries over from Win-03 and Win-04 — the PKI infrastructure already exists
The Win-03 and Win-04 guides built the full PKI hierarchy:
— RootCA (10.70.0.20) — offline Windows Server CA, root of trust for NovaCrestBank

This guide uses the existing PKI to issue certificates for the four VCF components. No new PKI infrastructure is needed.

WIN-VCF-03 · Section 02
Certificate plan — 4 components, all certificate details before clicking
Every certificate field for every component — know the SANs before generating any CSR
Why SANs are critical — the Common Name alone is not enough
A certificate has a Subject field with a Common Name. Is the Common Name sufficient for VCF components?
No. Modern browsers and TLS clients (including NSX Manager, SDDC Manager APIs, and the OCP installer) validate the Subject Alternative Name (SAN) extension — not the Common Name — to verify that the certificate matches the hostname they are connecting to. If the SAN does not include the exact hostname used to connect, certificate validation fails even if the Common Name matches.

For VCF components the following SANs are required per certificate:
— The primary FQDN (e.g., sddc-manager.novacrestbank.local)
— The short hostname (e.g., sddc-manager) — some internal tools use unqualified names
— The management IP address — some NSX internal API calls use IP addresses

NSX Manager cluster certificate additionally requires the VIP FQDN (nsx-mgr.novacrestbank.local) and all three node FQDNs — because clients may connect to any of the four addresses.

StepWhere it happensWhat you get

WIN-VCF-03 · Section 11
Install and apply the certificate to the NSX Manager cluster
Importing the signed certificate into NSX Manager's certificate store and then applying it to the cluster API service — two separate steps
Two steps: import the certificate, then apply it to the cluster
NSX Manager has a Certificates section. Why is importing the certificate not the same as applying it?
Because NSX Manager decouples certificate storage from certificate usage. Importing adds the certificate to the internal store but does not change what the API service presents. After import, the certificate must be explicitly applied to the cluster node (and optionally to the cluster VIP) to take effect. This allows certificate pre-staging — a new certificate can be imported and validated before the service is switched over.

WIN-VCF-03 · Section 12
Generate the Machine SSL CSR for vcenter-mgmt
vCenter has a built-in Certificate Manager accessible from the vSphere Client. The Machine SSL certificate is the one presented to browsers and API clients — it is the certificate to replace.
Machine SSL certificate — what it is and why it is the only one that needs replacing in this guide
vCenter has multiple certificate types (Machine SSL, Solution User certificates, VMCA root). Which one needs to be replaced for browser and API trust?
The Machine SSL certificate is the TLS certificate presented on port 443 when any client (browser, NSX Manager, SDDC Manager, OCP installer) connects to vCenter. It is the one that causes browser warnings and API trust failures when it is self-signed.

The Solution User certificates are used for internal vCenter service-to-service communication — they are not presented to external clients and do not need to be replaced for the purposes of this guide. The VMCA root certificate is vCenter's internal CA — replacing it requires re-issuing all Solution User certificates and is significantly more complex. For NovaCrestBank, replacing only the Machine SSL certificate with an IssuingCA-signed cert provides all required trust without the complexity of a full VMCA replacement.

What OCP-VCF-01 builds next — install-config.yaml

The Windows Server phase is now complete. OCP-VCF-01 creates the install-config.yaml — the configuration file the OCP IPI installer reads to know where to deploy the cluster. For the NovaCrestBank VCF+NSX environment, this file is significantly different from the standalone vSphere deployment used in the previous series:

  • vCenter reference — points to vcenter-ocp.novacrestbank.local (the OCP Workload Domain vCenter, not the Management Domain vCenter)
  • Network segment reference — specifies seg-ocp-nodes (the NSX overlay segment created in NSX-03) instead of a vDS port group
  • CA bundle — includes the NovaCrestBank CA chain (rootca.cer + issuingca.cer concatenated) so the installer trusts vcenter-ocp's certificate
  • apiVIPs and ingressVIPs — 10.50.0.200 and 10.50.0.201 (the NSX LB VIPs created in NSX-04)
  • Node count and sizes — 3 masters + 3 infra + 3 workers, hardware sizing per NovaCrestBank banking workload requirements

OCP-VCF-01 is a pure configuration file guide — it explains every field in install-config.yaml, what is different from the previous series, and what must match values configured in earlier guides (NSX segment names, VIP addresses, DNS records, vCenter FQDN).
