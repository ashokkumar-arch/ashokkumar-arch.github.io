PKI — the infrastructure behind every padlock in your browser
Every time a browser shows a padlock (HTTPS), every time an SSH client verifies a server's identity, every time a machine authenticates to a network — a certificate is involved. A certificate answers one question: "Can I trust that this server is who it claims to be?" Without certificates, all encrypted connections are vulnerable to man-in-the-middle attacks — the connection is encrypted, but you do not know who you are encrypting it with. PKI is the system that issues and manages those certificates.

Why build a private CA — not just use DigiCert or Let's Encrypt for everything
NovaCrestBank already uses DigiCert for its public-facing website. Why build an internal CA at all?
Because public CAs cannot issue certificates for internal hostnames (.local domains and private IP addresses), and using public CAs for internal infrastructure creates unnecessary cost and external dependencies.

DigiCert and Let's Encrypt can only issue certificates for publicly-resolvable hostnames — names that DNS can verify exist on the public internet. vcenter.novacrestbank.local is a private hostname — public CAs will refuse to issue a certificate for it because .local is not a public TLD.

A private CA can issue certificates for any hostname or IP address — including all the internal infrastructure that uses private DNS names. The certificates are trusted by all domain-joined machines (which automatically receive the Root CA certificate from Group Policy) and by any system where you manually install the Root CA certificate (like ESXi hosts).

NovaCrestBank's PKI design: private CA handles all internal infrastructure certificates. DigiCert continues to handle the public banking portal certificate (novacrestbank.com) — which customers' browsers trust without needing the internal Root CA.

How a client verifies a certificate — the chain validation

When a browser connects to vCenter and receives vCenter's TLS certificate, it performs chain validation:

  1. Browser reads vCenter's certificate. It is signed by NovaCrestBank-Issuing-CA.
  2. Browser looks for NovaCrestBank-Issuing-CA's certificate. It was installed on all domain-joined machines via Group Policy. The browser finds it and reads it. It is signed by NovaCrestBank-Root-CA.
  3. Browser looks for NovaCrestBank-Root-CA's certificate. It was also installed via Group Policy. The browser finds it. It is self-signed — it signed itself. The browser checks if this Root CA is in its Trusted Root Certification Authorities store.
  4. The Root CA certificate IS in the Trusted Root store (put there by Group Policy). The browser trusts it.
  5. Since the Root CA is trusted, the Issuing CA certificate (signed by the Root CA) is trusted. Since the Issuing CA is trusted, vCenter's certificate (signed by the Issuing CA) is trusted. The padlock appears.
Why not just use a single CA for everything — why have two tiers
A one-tier PKI (just a Root CA that issues all certificates directly) is simpler. Why does NovaCrestBank use two tiers?
Because the Root CA must be offline for security, and an offline CA cannot issue certificates on demand. The Issuing CA bridges this gap — it stays online to issue certificates while the Root CA stays offline to protect the root of trust.

If a single CA were used as both the root of trust AND the certificate issuer, it would need to be online at all times (to respond to certificate requests and publish CRLs). An online Root CA can be compromised — if an attacker obtains the Root CA's private key, they can issue certificates for any hostname, including impersonating any bank or government website. Browsers would trust these fraudulent certificates.

With two tiers: even if the Issuing CA is compromised, the attacker can only issue certificates that chain to NovaCrestBank's Root CA — which is only trusted by machines that explicitly trust it (NovaCrestBank's own infrastructure). The broader internet is unaffected. And the compromised Issuing CA can be revoked by the offline Root CA — which signs a new CRL that marks the Issuing CA certificate as revoked, causing all systems to immediately stop trusting it.
Windows Server · Win-03 · Section 03
Why the Root CA must be offline — and what "offline" means in practice
The single most important architectural decision in the entire PKI design
The Root CA private key — the most sensitive asset in NovaCrestBank's infrastructure
The Root CA holds a private key that can sign any certificate. If this private key is stolen, an attacker can create certificates for any hostname that all NovaCrestBank systems would trust — vcenter.novacrestbank.local, dc01.novacrestbank.local, any internal system. Every encrypted connection would be vulnerable to man-in-the-middle. Recovering from a compromised Root CA requires rebuilding the entire PKI from scratch and replacing every certificate in the environment — a multi-day outage with significant operational risk. Protecting the Root CA private key is the highest-priority security objective in the PKI design.

What "offline" means for the Root CA VM

The Root CA VM is powered off in vCenter. The power state is "Off." The VM still exists on the datastore — its disk is intact, its configuration is preserved, its certificate database is preserved. "Offline" means:

  • No network exposure — a powered-off VM has no network interface, cannot receive connections, cannot be attacked remotely
  • No running processes — no services, no attack surface
  • Private key not in memory — the private key is only in memory when the CA service is running. When powered off, it is encrypted on disk by Windows DPAPI
  • Accessible when needed — the VM can be powered on in minutes via vCenter or iLO if it needs to sign a new Issuing CA certificate or publish an updated CRL
When the Root CA needs to be powered on — and who authorises it
The Root CA is powered off. When does it need to be turned on again, and what controls access?
The Root CA only needs to be powered on for two reasons, both infrequent:

1. To sign a new Issuing CA certificate — if the Issuing CA certificate expires (in 10 years) or if the Issuing CA is replaced due to compromise. This happens at most once every 10 years under normal circumstances.

2. To publish an updated Root CA CRL — the Root CA's CRL (which can only revoke the Issuing CA certificate) has a long validity period (typically 1 year). Every year, the Root CA must be powered on briefly to publish a new CRL. If the CRL expires, all certificates chaining to the Root CA become invalid — even valid ones. This is a critical operational task that must be scheduled and performed on time.

Access control: The Root CA VM is in vCenter. Access to power it on requires vCenter administrator credentials (administrator@vsphere.local or a domain admin with the vCenter admin role). The change management process should require two-person authorisation for Root CA power-on events — one person initiates, a second approves in CyberArk. This is a banking regulatory expectation for critical PKI operations.

Why the Root CA is NOT joined to the domain
DC01 and DC02 are domain members. The Issuing CA will be a domain member. Why should the Root CA be standalone — not joined to novacrestbank.local?
Because a domain-joined Root CA trusts every domain administrator — and any domain administrator could potentially access the Root CA's private key.

Domain-joined machines allow domain administrators to log in locally and have full local administrator rights. If the Root CA were joined to novacrestbank.local, any NOVACRESTBANK\Domain Admins member could log into the Root CA and potentially access or export the Root CA private key. The Root CA's security depends on restricting who can access it to an absolute minimum — ideally a dedicated PKI administrator account, not the entire domain admin group.

As a standalone machine (not domain-joined), the Root CA only accepts local user accounts. Only the local Administrator account (with the Root CA-specific password) can log in. This completely separates Root CA access from AD domain administration — a domain admin compromise does not automatically compromise the Root CA.

This also means the Root CA cannot be a type "Enterprise CA" (Enterprise CAs require domain membership). It must be a "Standalone CA" — which is fine because the Root CA's only job is to sign the Issuing CA's certificate. It does not need AD integration for certificate templates or auto-enrolment.

Why esxi03 — not esxi01 where DC01 and vCenter run
The Root CA is a high-value security asset. Does its placement on a specific host matter?
Yes — placing the Root CA on a different host from the Domain Controllers provides physical separation and means a single host failure does not take both a DC and the Root CA offline simultaneously.

More importantly: the Root CA should not share a host with DC01. DC01 has network connectivity and active AD services. The Root CA will be powered off — but if it were on the same host as DC01, a host compromise could theoretically provide access to both the Domain Controller and the offline Root CA's disk files. Physical separation reduces the blast radius of any single-host security incident.

Once vSAN is configured (after this guide series), a DRS anti-affinity rule will ensure the Root CA VM never accidentally migrates onto the same host as DC01 or DC02.

Key length 4096 — not 2048 or 8192
The default is 2048 bits. Why use 4096 for the Root CA?
Because the Root CA's key must remain cryptographically secure for the entire lifetime of the PKI — up to 20 years — and 4096-bit RSA provides adequate security margin for that timeframe.

Why not 2048? A 2048-bit RSA key is secure today and will likely remain secure for 10 years. However, the Root CA certificate itself lasts 20 years. As computing power increases (and quantum computing research advances), 2048-bit RSA may become weaker during the Root CA's lifetime. The cost of using 4096 is negligible (slightly slower certificate signing operations — milliseconds). The benefit is an additional security margin for the longest-lived certificate in the PKI.

Why not 8192? 8192-bit RSA is overkill and causes interoperability issues with some older clients and TLS stacks. 4096 is the standard for Root CAs in banking and financial infrastructure.

Why RSA and not ECDSA? ECDSA (Elliptic Curve) with a 256-bit key provides equivalent security to RSA-3072 at much smaller key size. However, ECDSA support in Windows Server AD CS and some older client libraries is less mature than RSA. RSA-4096 is the conservative, well-tested choice for a new banking CA. ECDSA is better for future deployments when the entire ecosystem has matured.
SHA-256 — not SHA-1 or SHA-512
Why SHA-256 — not the older SHA-1 or the newer SHA-512?
SHA-256 is the current industry standard for certificate signing algorithms — universally supported, cryptographically sound, and mandated by all modern browsers and operating systems.

SHA-1 was deprecated by browsers in 2017 — certificates signed with SHA-1 are rejected by all modern browsers with an error. Using SHA-1 would make the entire PKI immediately non-functional in current systems.

SHA-512 is more cryptographically secure than SHA-256 but produces a larger hash (64 bytes vs 32 bytes) and is slower to compute. The additional security of SHA-512 over SHA-256 is not necessary for certificate signing — SHA-256 provides more than adequate collision resistance for PKI use. SHA-256 is universally supported with no compatibility concerns. SHA-512 is increasingly supported but may cause issues with some legacy systems NovaCrestBank might still encounter.

Set: CSP = RSA#Microsoft Software Key Storage Provider, Key length = 4096, Hash algorithm = SHA256. Click Next.

Why "NovaCrestBank-Root-CA" — naming convention decisions
The CA name appears in every certificate ever issued by this PKI. What makes a good CA name?
Three principles applied: organisation prefix, role description, and no version numbers.

Organisation prefix (NovaCrestBank) — When an administrator inspects a certificate chain on any system, the CA name is the first thing that tells them which organisation's PKI issued this certificate. Without the prefix, a name like "Root-CA" is meaningless — it could be any organisation.

Role description (Root-CA) — Clearly distinguishes the Root CA from the Issuing CA in the chain. When viewing a certificate's "Issued by" field, you can immediately see which tier of the PKI signed it: NovaCrestBank-Root-CA signs the Issuing CA certificate. NovaCrestBank-Issuing-CA signs all leaf certificates.

No version numbers — Avoid "NovaCrestBank-Root-CA-2024" or "v1" in the name. If the Root CA is ever rebuilt, the version number in the old CA name creates confusion — which version signed which certificate? A timeless name like "NovaCrestBank-Root-CA" allows for a clean rebuild if needed.

No spaces — Spaces in CA names cause issues in some tools and scripts that parse Distinguished Names. Hyphens are safe everywhere.

Why 20 years for the Root CA
20 years seems very long. Why not 5 years for better security (shorter validity = less time for compromise to go undetected)?
Because the Root CA certificate validity must cover the lifetime of all certificates it will ever sign — plus operational buffer for renewals.

The certificate hierarchy requires: Root CA validity > Issuing CA validity > Leaf cert validity.

If the Root CA certificate expires while any Issuing CA certificate it signed is still valid, the entire PKI breaks immediately — every leaf certificate becomes untrusted because the chain traces to an expired Root CA. The Root CA certificate expiry date is the hard deadline for every certificate in the entire PKI.

NovaCrestBank's chain: Root CA = 20 years → Issuing CA = 10 years → leaf certs = 1–2 years. The 20-year Root CA validity means: if a new Issuing CA is needed at year 9, the new Issuing CA certificate (10 years) will still expire before the Root CA certificate (year 20). There is a comfortable 10-year buffer.

The security concern about long validity is addressed by keeping the Root CA offline — not by shortening its validity period. Windows Server · Win-03 · Section 16
CRL and AIA — what they are and why they must be configured
Certificate revocation and authority information — two extensions that appear in every certificate and must point to reachable locations
What a CRL is and why it matters
What is a Certificate Revocation List (CRL) and what happens if it is not accessible?
A CRL is a signed list of certificate serial numbers that have been revoked before their expiry date. Every certificate contains a URL (the CDP — CRL Distribution Point) pointing to where clients can download the CRL. When a client validates a certificate, it downloads the CRL and checks if the certificate's serial number appears on it. If the CRL is unavailable, clients may refuse to trust any certificate — depending on their configuration.

Why revocation matters: if a private key is stolen (e.g., vCenter's TLS private key is obtained by an attacker), the certificate using that key must be revoked immediately. Simply issuing a new certificate is not enough — the compromised certificate is still valid and an attacker could still use it. Revoking it adds it to the CRL. All clients that check the CRL immediately stop trusting the compromised certificate.

For the Root CA specifically: The Root CA's CRL is very simple — it only contains one type of entry: Issuing CA certificates that have been revoked. If the Issuing CA is ever compromised, its certificate is revoked via the Root CA's CRL. This is the nuclear option that immediately invalidates the entire Issuing CA and everything it signed. The Root CA's CRL must be published to a location accessible even when the Root CA itself is offline — because the Root CA is almost always offline.
What the AIA extension is
What is Authority Information Access (AIA) and why does it need to be configured?
AIA is a certificate extension that contains two pieces of information: (1) where to download the CA certificate (so clients can build the trust chain without having the CA certificate pre-installed), and (2) where the OCSP responder is (if Online Certificate Status Protocol is used instead of CRL — not applicable for NovaCrestBank's design).

The AIA URL in the Root CA's configuration appears in the Issuing CA certificate — it tells clients "if you need to download the NovaCrestBank-Root-CA certificate, go here." This allows clients that do not already have the Root CA certificate in their trust store to obtain it automatically and complete the chain validation.

For NovaCrestBank's internal PKI, the AIA URL points to a web server or file share accessible to all internal clients. The CRL and AIA locations must be accessible even when the Root CA is offline — because clients check these URLs during normal certificate validation, not during certificate issuance.

The publication locations NovaCrestBank uses

Because the Root CA will be offline, its CRL and CA certificate must be published to a location that is always accessible. NovaCrestBank uses a shared folder on DC01 that is accessible as an HTTP URL via IIS (to be configured in a moment) and as a UNC path for the CA to write to:

CRL local write path (UNC): \\dc01\pki\

What a CSR is and what it contains

A Certificate Signing Request (CSR) is a file generated by the Issuing CA containing: (1) the Issuing CA's public key, (2) the Issuing CA's proposed subject name (NovaCrestBank-Issuing-CA), and (3) a digital signature created with the Issuing CA's private key (proving the requester owns the key). The Root CA signs this CSR — adding the Root CA's signature, the validity period, and CA-specific extensions — to produce the Issuing CA's certificate.

How the CSR file reaches the Root CA

Because the Root CA is in a workgroup (not domain-joined), and may not have a permanent network connection to the Issuing CA server, the CSR transfer can be done in several ways. The simplest for NovaCrestBank's build phase (while the Root CA is still powered on and networked):

Why the Root CA issues the Issuing CA certificate manually — not automatically
Why is there a "Pending Requests" step — why does the Root CA not issue the certificate automatically when it receives the CSR?
Because the Root CA is a Standalone CA (not an Enterprise CA), and Standalone CAs require manual approval of all certificate requests.

Enterprise CAs can be configured with certificate templates that automatically approve certain types of requests (based on AD group membership and template settings). This is convenient for auto-enrolment of thousands of machines.

Standalone CAs have no connection to Active Directory — they cannot verify the requester's identity against AD groups or templates. All requests go to Pending status and require a human administrator to review and explicitly approve them. This is the correct behaviour for a Root CA: the administrator should consciously review and approve every Issuing CA certificate request — there should never be more than one or two in the Root CA's lifetime.

Windows Server · Win-03 · Section 23
Why the Root CA is powered off permanently after this guide
The most important operational step in this guide — making the Root CA offline is not optional, it is the entire point of the two-tier PKI design
The Root CA must be powered off — not just disconnected from the network
Simply removing the network cable (or disconnecting the VM's NIC in vCenter) is not sufficient. A VM with its NIC disconnected is still running — it has active processes, the CA service is running, the private key is in memory, and it could potentially be reconnected to the network at any time by anyone with vCenter access. The correct offline state is powered off (VM power state = Off) in vCenter. In this state, there are no running processes, the private key is not in memory, and the VM cannot communicate with anything.
What exactly is protected by powering off the Root CA
An attacker who compromises a vCenter admin account could power on the Root CA. Is powered-off truly secure?
Powered-off provides meaningful security layers even if vCenter is accessible to an attacker, because accessing the Root CA's CA service requires multiple steps that each create detectable audit events.

To compromise a powered-off Root CA, an attacker would need to: (1) Obtain vCenter administrator credentials → logged in vCenter's audit trail. (2) Power on the Root CA VM → logged in vCenter's event log. (3) Log into Windows as local Administrator → requires the Root CA-specific password (separate from domain admin password). (4) Access the CA service → logged in the Windows Security event log and the CA audit log. (5) Export or use the private key → requires additional authorization and is heavily logged.

Each of these steps generates audit events in vCenter, Windows, and the CA logs. In a properly monitored environment (which NovaCrestBank should have — SIEM, log aggregation, alerts on Root CA power events), this chain of actions would trigger alerts before the attacker could complete the compromise.

The powered-off state is not meant to be impenetrable against an attacker who has already compromised vCenter admin and domain admin credentials simultaneously — that attacker already has access to everything. The powered-off state defends against: remote network compromise, accidental access, insider threats with partial access, and ensures that the Root CA private key is not continuously exposed in memory.

Operational reminder — the annual CRL renewal
The Root CA CRL expires in 52 weeks from today. Set a reminder or ITSM recurring task for early April 2027 (2 weeks before expiry) to power on the Root CA and publish a new CRL. If this is missed and the CRL expires, every certificate in NovaCrestBank's PKI becomes untrusted simultaneously — causing vCenter to fail, ESXi host connections to break, and all HTTPS services to show certificate errors. This is one of the most disruptive PKI operational failures and is entirely preventable with a single annual reminder.
