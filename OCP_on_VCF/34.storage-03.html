STORAGE-03 · Section 02
Access modes explained — RWO, RWX, ROX, and RWOP
The access mode of a PVC determines how many pods on how many nodes can mount the volume simultaneously. Choosing the wrong access mode is the most common storage misconfiguration in OCP environments — it typically manifests as pods stuck in Pending with the "Multi-Attach error" message. Understanding access modes is prerequisite to every StorageClass decision.
Access modeFull nameMeaningSupported byUse when
RWO ReadWriteOnce One node mounts the volume read-write. Multiple pods on that same node can access the volume (though this requires the storage driver to support it). No other node can mount it simultaneously. vsan-gold, vsan-standard, ocs-ceph-rbd, ocs-ceph-rbd-virtualization Single-pod stateful applications: databases (PostgreSQL, etcd), application caches (Redis), single-replica StatefulSets (GitLab Gitaly, Prometheus).
RWX ReadWriteMany Multiple nodes can mount the volume simultaneously, all read-write. There is no exclusivity — any number of pods on any number of nodes can mount and write concurrently. ocs-storagecluster-cephfs, ocs-storagecluster-ceph-rbd-virtualization Multi-replica Deployments that share state, OpenShift Virtualization live migration (VM disk accessible from source and destination node simultaneously), any workload that horizontal scales with shared storage.
ROX ReadOnlyMany Multiple nodes can mount the volume simultaneously, all read-only. No pod can write to the volume. ocs-storagecluster-cephfs (with annotation) Shared read-only data: configuration files, static content, model weights loaded by multiple pods. Rarely used in NovaCrestBank's current architecture.
RWOP ReadWriteOncePod Only one pod cluster-wide can mount the volume read-write. Stricter than RWO — even two pods on the same node cannot share the volume. Introduced in Kubernetes 1.29. ocs-storagecluster-ceph-rbd (OCP 4.16+) Critical single-writer workloads where accidental double-mount could cause data corruption: etcd data, financial databases where split-brain is catastrophic.
The Multi-Attach error — the most common access mode mistake
A Deployment with 3 replicas using an RWO PVC works for the first pod but the second and third pods remain Pending with Multi-Attach error: Volume is already exclusively attached to one node. What went wrong and what is the correct fix?
The root cause: a Deployment creates pods that can schedule to any node. With an RWO PVC, the first pod attaches the volume to its node. When the second pod schedules to a different node and attempts to attach the same PVC, the CSI driver rejects the attachment — the PVC is exclusively attached to the first pod's node.

Wrong fix: trying to force all replicas to the same node. Using node affinity to keep all replicas on one node allows them to share an RWO volume, but destroys the availability benefit of multiple replicas — if that node fails, all replicas fail simultaneously. This trades an immediate startup problem for a worse availability problem later.

Wrong fix: using a StatefulSet with one PVC per replica. StatefulSets with volumeClaimTemplates create a separate PVC per pod — each pod has its own independent volume. This is correct for truly independent state (like database replicas with their own data) but incorrect for applications where replicas need to share the same data.

Correct fix: change the PVC to use CephFS RWX. If the application genuinely needs multiple pods to access the same data simultaneously, change the PVC's storageClassName to ocs-storagecluster-cephfs and accessModes: [ReadWriteMany]. All pods on all nodes mount the same CephFS volume. If the application writes, CephFS handles concurrent write coordination. If the application only reads, ROX is also an option.
STORAGE-03 · Section 03
Reclaim policy — Retain vs Delete — what happens when a PVC is deleted
The reclaim policy determines what happens to the underlying storage when the PVC that claimed it is deleted. All 7 NovaCrestBank StorageClasses use the Delete policy. Understanding why, and when Retain is appropriate, prevents accidental data loss and deliberate data accumulation.

Delete (current policy — all StorageClasses)

When a PVC is deleted, the underlying PV and its backing storage (VMDK for vSAN, Ceph RBD image for ODF) are automatically deleted. Storage is reclaimed immediately. No manual cleanup required. Risk: if a PVC is accidentally deleted (kubectl delete pvc, Argo CD prune), the data is gone. There is no recycle bin. GitOps discipline (no manual kubectl in production, Argo CD prune=false for production) is the primary protection against accidental deletion.

Retain (not used — documented for completeness)

When a PVC is deleted, the underlying PV transitions to Released status but the backing storage is NOT deleted. A human must manually decide what to do with the Released PV. Use Retain when: the data must be preserved regardless of PVC lifecycle (compliance archive), or during a planned workload migration where you want to manually rebind the PV to a new PVC. NovaCrestBank uses VolumeSnapshots (Section 14) rather than Retain for data preservation — snapshots are more controlled and manageable than orphaned Released PVs.

Protecting against accidental PVC deletion in production — the three-layer defence
All production PVCs are protected by three layers: (1) GitOps — PVCs in the production namespace are declared in the GitOps config repo; Argo CD has prune: false for production, meaning it will never automatically delete a resource from the cluster even if it disappears from Git. (2) Application PodDisruptionBudgets — ensuring pods (and their PVCs) are not disrupted during maintenance without explicit approval. (3) OADP VolumeSnapshot backups (STORAGE-04) — even if a PVC is accidentally deleted, the most recent snapshot can be used to restore the volume. The Delete reclaim policy is acceptable because these three layers prevent accidental deletion.
STORAGE-03 · Section 04
The decision framework — five questions to select the correct StorageClass
Every storage decision in NovaCrestBank's platform can be made by answering five questions in order. The framework is designed to arrive at a single StorageClass recommendation without requiring the engineer to know all seven StorageClasses and their properties up front.
Question 1
Does the workload need S3-compatible object storage (HTTP GET/PUT) rather than a mounted filesystem?
YES → object storage Continue to Question 5 (MinIO vs MCG).
Examples: LokiStack chunks, Tempo traces, CI artifacts, backups
Question 2
Do multiple pods on different nodes need to mount the same volume simultaneously (ReadWriteMany)?
YES → ocs-storagecluster-cephfs The only RWX option.
Examples: OCP Virtualization live migration, shared workspace, multi-replica shared state
Question 3
Is this a VM boot disk for OpenShift Virtualization?
YES → ocs-storagecluster-ceph-rbd-virtualization Ceph RBD with VM-specific features.
Required for live migration support. Do NOT use for regular pod PVCs.
Question 4a
Is this a new workload (no existing PVC)?
YES → ocs-storagecluster-ceph-rbd Default for all new OCP stateful workloads.
OCP-native lifecycle, VolumeSnapshot support, ODF monitoring integration
Question 4b
Is this an existing vSAN-backed PVC? Does the workload need VolumeSnapshots or RWX?
NO → keep vsan-gold or vsan-standard Leave existing PVCs. No migration needed.
Migrate opportunistically if VolumeSnapshots or RWX become required.
Question 5
(Object storage only) Is this backup data or the image registry — requiring high durability and VolumeSnapshot integration?
YES → openshift-storage.noobaa.io (MCG OBC) Ceph-backed durability.
NO → MinIO (S3 API directly) Observability data, CI artifacts, GitLab backups.

Why vsan-gold exists — the FTT:2 durability argument
vsan-gold uses RAID-6 with Failures To Tolerate set to 2 — the VMDK is striped across enough hosts that two simultaneous disk failures do not cause data loss. This was chosen for GitLab Gitaly (source code) and PostgreSQL (GitLab metadata) because these contain data that cannot be reconstructed: committed source code and audit records. Ceph's 3× replication also tolerates a single node failure, but RAID-6 FTT:2 provides a higher tolerance against simultaneous disk failures within a RAID stripe — a different failure model. For NovaCrestBank's existing GitLab and PostgreSQL PVCs, migrating from vsan-gold to ODF ceph-rbd provides equivalent or better protection and adds VolumeSnapshot support. However, the migration carries risk and is not required until VolumeSnapshot-based backup is specifically needed for those workloads.
STORAGE-03 · Section 06
Shared RWX — CephFS is the only option in NovaCrestBank's cluster
When a workload needs multiple pods on different nodes to mount the same volume simultaneously, ocs-storagecluster-cephfs is the only option. Neither vSAN nor any other StorageClass in the cluster supports ReadWriteMany for filesystem access. This section documents the CephFS StorageClass properties and the workloads that require it.

ocs-storagecluster-cephfs — properties

Supported access modesReadWriteMany (RWX) · ReadWriteOnce (RWO) · ReadOnlyMany (ROX)
Backing technologyCephFS distributed filesystem — managed by 1 active + 1 standby MDS daemon
Concurrent writer supportYes — CephFS handles concurrent writes from multiple pods via distributed locking. Applications must be designed for concurrent access (same as NFS shared mounts).
VolumeSnapshot support✓ CephFS CSI snapshots via Kubernetes VolumeSnapshot API
Performance characteristicsGood throughput for sequential reads/writes. Metadata operations (directory listing, file creation) are slower than local filesystem — MDS is the bottleneck for workloads with high metadata churn.
NovaCrestBank current RWX workloadsOpenShift Virtualization VM disks (live migration) · Future: Jenkins agent shared workspace if needed
CephFS is not always better than per-replica RWO volumes — the correct design choice
CephFS RWX is often requested when the real solution is to redesign the application to not need shared storage. A banking microservice that uses shared storage for cross-replica coordination is likely an anti-pattern — distributed locking on a shared filesystem introduces latency and consistency complexity. The correct design for banking microservices is: stateless application tier (each pod handles any request independently) + external state in a database or message queue (which uses its own RWO PVC per replica via StatefulSet). CephFS RWX is correct for: live VM migration, shared read-only content distribution, and legacy applications that genuinely require shared filesystem access.

STORAGE-03 · Section 09
Banking workloads — payment-service, databases, and stateful applications
Banking application workloads are predominantly stateless microservices (payment-service, account-service, fraud-detection-service) — they use Kubernetes ConfigMaps and ExternalSecrets for configuration but do not have their own PVCs. The stateful components of the banking platform are the downstream databases and the payment processing ledger store.
Why most banking microservices do not have PVCs — the stateless design principle
The payment-service Deployment has no PVCs. Where does it store its state?
Stateless application design: the payment-service stores no state locally. All state is externalised.

Transactional state — the record of each payment — lives in the payment database (PostgreSQL). The payment-service writes to the database via JDBC and reads from it, but the database owns the data. If the payment-service pod is deleted and recreated, it reconnects to the same database and continues processing — no data is lost because no data was in the pod.

Session state — if the payment API has sessions (most banking APIs are sessionless REST) — would live in a shared Redis cache, not in the pod.

Configuration — environment variables injected from Kubernetes Secrets (via ExternalSecrets from CyberArk). No configuration files on local disk.

Why this matters for storage design: A 3-replica payment-service Deployment can use RWO PVCs incorrectly (each pod gets its own PVC with its own local data) or correctly (no PVCs at all, all state in the shared database). The correct design — no PVCs, shared database — is what enables the payment-service to scale horizontally without storage constraints.

STORAGE-03 · Section 11
Setting the default StorageClass — and why the current default needs changing
The default StorageClass is used when a PVC does not specify a storageClassName. Currently, vsan-standard is the default — it was set before ODF was deployed. With ODF available, the correct default for new OCP workloads is ocs-storagecluster-ceph-rbd. This section changes the default and explains the consequences.
Why change the default StorageClass from vsan-standard to ocs-storagecluster-ceph-rbd
vsan-standard works correctly for most workloads. Why change the default at all? What breaks if vsan-standard remains the default?
Two specific problems arise if vsan-standard remains the default after ODF is deployed.

Problem 1 — Operator-created PVCs go to vSAN instead of ODF. Many OCP Operators create PVCs without specifying a StorageClass — they rely on the cluster default. ACS Central's database PVC, if re-deployed, would go to vsan-standard. A new banking PostgreSQL StatefulSet without an explicit storageClassName would go to vsan-standard. These workloads should default to ODF RBD (for VolumeSnapshot backup support) but silently land on vSAN instead. The engineer must audit every PVC to determine if it ended up on the correct StorageClass — this is exactly the kind of implicit, invisible misconfiguration that causes problems months later when OADP backup is configured and half the workloads have no VolumeSnapshot support.

Problem 2 — "No default StorageClass" errors if vsan-standard is removed. If vSAN is ever decommissioned (possible as NovaCrestBank moves to full OCP-native storage), removing vsan-standard while it is still the default causes all PVCs without explicit storageClassName to fail with "no default StorageClass found." Changing the default to ODF RBD now prevents this future problem and correctly reflects the current storage policy: ODF is the preferred storage for new OCP workloads.

What does NOT break when the default changes: existing PVCs are not affected — they already have a storageClassName set. Only future PVCs that do not specify a storageClassName will land on the new default. The change is forward-looking, not retroactive.

STORAGE-03 · Section 13
Volume expansion — which StorageClasses support online resize
Volume expansion (growing a PVC after creation) is supported by most StorageClasses in the NovaCrestBank cluster. The key question is whether the expansion requires a pod restart and whether the filesystem inside the pod automatically grows. STORAGE-03 · Section 15
PVC patterns — StatefulSet volumeClaimTemplates vs Deployment PVCs
Kubernetes provides two patterns for attaching persistent storage to pods: volumeClaimTemplates in a StatefulSet (creates one PVC per pod automatically) and explicit PVCs referenced in a Deployment. Choosing the wrong pattern creates management complexity and availability problems.
Pattern 1
StatefulSet volumeClaimTemplate — one PVC per pod
A StatefulSet with volumeClaimTemplates creates a separate PVC for each pod replica automatically. Pod postgres-0 gets PVC data-postgres-0; pod postgres-1 gets PVC data-postgres-1. When a pod is rescheduled, it reconnects to its own PVC — the data follows the pod identity. Use when: each replica has independent state that must survive pod restarts and rescheduling. Examples: database replicas (each has its own data directory), Prometheus replicas (each scrapes independently), GitLab Gitaly (single replica, but uses StatefulSet for stable PVC binding). All NovaCrestBank stateful platform components use this pattern.
Pattern 2
Deployment with explicit PVC (RWX) — shared volume
A Deployment references a pre-existing PVC directly in the pod spec. All replicas mount the same PVC. This requires the PVC to be RWX (CephFS) — otherwise only one pod can mount it. Use when: all replicas need to share the same data simultaneously. Examples: a content serving Deployment where all replicas read from the same static file directory, Jenkins agents sharing a build workspace, OpenShift Virtualization VMs during live migration. NovaCrestBank uses this pattern for VM storage (STORAGE-04 / OpenShift Virtualization).
Anti-pattern
Deployment with RWO PVC — the Multi-Attach trap
A Deployment with replicas: 3 referencing a single RWO PVC. Works for 1 replica on the node that holds the PVC. Fails for all other replicas that schedule to other nodes. This is the most common storage misconfiguration. The fix is either: (1) convert to StatefulSet with volumeClaimTemplates if each replica needs independent storage, or (2) convert the PVC to CephFS RWX if all replicas genuinely need to share data. Never "fix" this by forcing all replicas to the same node.

STORAGE-03 · Section 17
StorageClass policy in GitOps — enforcing correct selection via Kustomize overlays
The StorageClass assignment table from previous sections is not just documentation — it is policy that should be enforced in the GitOps config repo. Kustomize overlays make the StorageClass explicit for every PVC in every environment. A developer who omits the storageClassName gets the cluster default (ODF RBD) — which is correct. A developer who specifies the wrong StorageClass for their environment will have it reviewed in the production promotion MR.
How the GitOps workflow enforces StorageClass policy
StorageClass selection in NovaCrestBank's GitOps model works at two levels: declarative (every PVC in the config repo explicitly states its StorageClass) and review (the production promotion MR shows the PVC diff — a reviewer can see if a new stateful workload is using vsan-standard when it should use ocs-storagecluster-ceph-rbd). The Kustomize base defines the correct StorageClass for production. The dev overlay might use a lighter StorageClass. The production overlay enforces the production storage policy. A developer cannot accidentally deploy a vSAN-backed PVC to production without a platform engineer seeing it in the MR diff and requesting a change.
Kustomize overlay — dev uses vsan-standard, production uses ODF RBD
