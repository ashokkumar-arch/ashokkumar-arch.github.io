The complete registry picture in a production OCP deployment
OpenShift already uses quay.io and registry.redhat.io. What gap does Harbor fill?
The external Red Hat registries and Harbor serve completely different purposes and are not alternatives to each other — they coexist in every serious enterprise deployment. Understanding what each one serves makes the architecture obvious.

registry.redhat.io — Red Hat's primary product image registry. Requires authentication (the pull secret from OCP-VCF-01). Serves: RHCOS base images, all official Red Hat operator images (OpenShift Pipelines, Logging, ACS), the Prometheus and Alertmanager images used by the monitoring stack, the OCP router image, and the vSphere CSI driver image. These are Red Hat's images — NovaCrestBank has no write access here and cannot push anything to it.

registry.access.redhat.com — Red Hat's public registry, no authentication required. Serves: Universal Base Images (UBI) — the minimal RHEL-based base images used as the FROM layer in NovaCrestBank's own Dockerfiles. ubi9-minimal, ubi9/openjdk-21, ubi9/python-311 come from here.

quay.io — Red Hat's container registry platform (the same technology you can self-host). Used by OpenShift for: OCP release images during install and upgrades, some operator catalog images. Also a general-purpose public registry where many open-source projects publish their images.

Harbor (internal) — NovaCrestBank's own registry, running inside the VCF environment. Serves: every image that NovaCrestBank's own developers build (the payment microservice, the customer portal, the API gateway). It also mirrors Red Hat images locally so nodes never need to reach the internet for any pull. It scans every image for CVEs before it can be deployed. It signs images so OCP can verify provenance before running them.

The flow in production: developer commits code → Jenkins builds an image using a UBI base → Jenkins pushes the image to Harbor → Harbor scans it → Harbor signs it → OCP pulls it from Harbor when a deployment is created. Red Hat's external registries are only contacted for Red Hat's own platform images — and even those can be mirrored into Harbor for a fully air-gapped deployment.

Registry-01 · Section 02
Why NovaCrestBank needs an internal registry — the decision
A structured decision covering the four drivers that make Harbor mandatory in a regulated banking environment, not optional infrastructure.
Decision — internal registry for NovaCrestBank
NovaCrestBank already has access to Red Hat's registries. Is a self-hosted Harbor registry necessary, or is this a nice-to-have?
Harbor is mandatory for NovaCrestBank, not optional. Four independent requirements each individually justify it. Meeting all four with Red Hat's external registries alone is not possible.

Driver 1 — Proprietary code cannot leave the data centre. NovaCrestBank's payment processing microservice, core banking APIs, and customer portal contain proprietary business logic. These images cannot be pushed to a public or Red Hat-controlled registry. A self-hosted Harbor is the only viable destination for images built from NovaCrestBank's source code. This is not a preference — it is a legal and competitive necessity.

Driver 2 — PCI-DSS Req 6.3.3 — all software components must be protected against known vulnerabilities. Every container image deployed in the cardholder data environment must be scanned for CVEs before deployment. The scan must produce an audit trail. Harbor's integrated Trivy scanner provides this scan at the registry layer — before the image reaches OCP — and produces machine-readable scan reports that satisfy audit requirements. Red Hat's external registries do not scan NovaCrestBank's images.

Driver 3 — Image provenance and supply chain integrity. PCI-DSS Req 6.4.3 and NIST SP 800-161 require software supply chain controls. A signed image guarantees that what runs in production is exactly what Jenkins built and tested — no substitution, no tampering. Harbor acts as the signing verification point. An OCP policy can be configured to refuse to run any image that is not signed by NovaCrestBank's cosign key, regardless of which registry it came from.

Driver 4 — Network resilience and pull performance. OCP nodes pulling images from registry.redhat.io over the internet introduce: latency (each pull is a WAN round-trip), external dependency (if Red Hat's CDN is unreachable, node bootstrapping and operator updates stall), and data egress (continuous image pulls consume WAN bandwidth). Mirroring Red Hat images into Harbor means every pull is a LAN pull — sub-second on 10Gbit vSAN-backed storage. Upgrades and operator installs do not require any internet access.

Why Harbor specifically, not another registry
The main alternatives to Harbor in enterprise environments are: Quay (Red Hat's version), JFrog Artifactory, and Nexus Repository. Harbor is chosen for NovaCrestBank because: it is fully open source (CNCF graduated project, no licensing cost), it has native OCI compliance, its Trivy integration is best-in-class for container scanning, it has a clear and well-documented API for automation, and it deploys cleanly as a Helm chart on OCP. Quay is a close alternative — it is what Red Hat uses for quay.io itself — but it is heavier to operate and requires a separate subscription for enterprise features. For NovaCrestBank's scale, Harbor is the right fit.
Registry-01 · Section 03
Harbor concepts — what each component does
Harbor has its own object model. Understanding it before touching the UI prevents confusion when configuring projects, accounts, and policies.
Harbor's object model — the four things you need to understand
Harbor has projects, repositories, robot accounts, and replication rules. What is each one and how do they relate?
Harbor organises images in a hierarchy: Registry → Project → Repository → Image Tag. Each level has its own access controls and policies.

Registry is the Harbor instance itself — harbor.novacrestbank.local. One registry per environment in NovaCrestBank's case. Everything below is contained within it.

Project is a namespace within the registry — analogous to an organisation on Docker Hub. Projects are the primary unit of access control and policy in Harbor. NovaCrestBank will have three: banking-apps (application images built by development teams), base-images (approved UBI and other foundation images), and platform (mirrored Red Hat images). Each project has its own visibility (public or private), CVE scan policy, signing policy, and member permissions.

Repository is a named image within a project — for example, banking-apps/payment-service. A repository contains multiple tagged versions of the same image: :latest, :v1.2.3, :sha256-abc123.

Robot account is a service account — not a human identity — used for machine-to-machine authentication. Jenkins uses a robot account with push permission to push images to banking-apps. OCP nodes use a separate robot account with pull-only permission to pull images. Robot accounts have minimal, specific permissions and their tokens can be rotated without affecting human accounts.

Replication rule is what pulls images from external registries (registry.redhat.io, quay.io) into Harbor automatically on a schedule. This is how the air-gap mirror is maintained — replication rules pull Red Hat images into the platform project nightly.

Harbor component architecture — what runs on OCP

Core
harbor-core
The main Harbor API server. Handles all UI and API requests, user authentication, project management, and policy enforcement. Backed by a PostgreSQL database (persistent storage on vsan-standard). The component developers and admins interact with.
Reg
harbor-registry
The OCI Distribution registry — the actual image store. Implements the OCI Distribution Spec and Docker Registry API v2. This is what receives docker push and responds to docker pull. Backed by a PVC on vsan-gold for image layer storage (this is the largest storage consumer — 500 GB initial allocation).
Scan
harbor-trivy
Trivy vulnerability scanner embedded in Harbor. Triggered automatically on every image push. Scans OS packages (RPM, DEB), language-specific packages (Maven, pip, npm), and application binaries for known CVEs from the NVD and vendor advisories. Produces a structured report per image layer. Can block image promotion based on severity threshold.
GW
harbor-nginx (ingress)
TLS termination and reverse proxy fronting the Harbor UI and registry API. In NovaCrestBank's deployment, the OCP Ingress Router replaces this — Harbor is exposed via an OCP Route with the IssuingCA-signed wildcard certificate. The URL becomes harbor.apps.ocp.novacrestbank.local.
Rep
harbor-jobservice
Background job processor. Handles replication jobs (pulling images from external registries), scan jobs (dispatching images to Trivy), and garbage collection. Runs as a separate pod so heavy replication jobs do not block the API.

Registry-01 · Section 05
Deploy Harbor with Helm on OCP
Harbor is deployed from the official Helm chart with NovaCrestBank-specific values. Every value is annotated with its purpose and the architectural reason behind it.
OCP security context — Harbor Helm chart needs a small adjustment for OpenShift
Harbor's official Helm chart was designed for vanilla Kubernetes where pods can run as any UID. OCP's default SecurityContextConstraints (SCC) enforce that pods run as a non-root UID from the namespace's assigned range. The podSecurityContext override below removes the fixed UID from the Harbor chart so OCP assigns one from the namespace range automatically. Without this, every Harbor pod fails with permission denied on startup.
jumphost.novacrestbank.local — add Helm repo and write values file
# Add the official Harbor Helm repository
[svc-ocp@jumphost ~]$ helm repo add harbor https://helm.goharbor.io && helm repo update
"harbor" has been added to your repositories
Update Complete. Happy Helming!

harbor-values.yaml — complete values with every field explained
expose:
  type: ingress
    # ingress: use an OCP Route / Kubernetes Ingress for TLS termination
    # The OCP Ingress Router handles TLS using the wildcard cert from OCP-VCF-02
    # "clusterIP" or "nodePort" would bypass the NSX LB — not appropriate here
  ingress:
    hosts:
      core: harbor.apps.ocp.novacrestbank.local

          # Developers push to: docker push harbor.apps.ocp.novacrestbank.local/banking-apps/payment-service:v1.0
    annotations:
      route.openshift.io/termination: "edge"
        # edge: TLS terminated at the OCP router (HAProxy on infra nodes)
          # Traffic between router and Harbor pods is plain HTTP inside the cluster
          # This is the standard OCP pattern — TLS at the edge, plain HTTP internally

externalURL: https://harbor.apps.ocp.novacrestbank.local
# externalURL: Harbor uses this in replication rules, webhook notifications,
# and in the Docker login instructions shown in the UI.
# Must match the hostname above exactly including the scheme.

harborAdminPassword: "NovaCrest$HarborAdmin2026!"
# Initial admin password — stored in CyberArk.
# The admin account is used only for initial setup.
# After AD integration (Section 07), AD group "Harbor-Admins" takes over.
# This password is rotated after first login.

persistence:
  enabled: true
  persistentVolumeClaim:
    registry:
      storageClass: vsan-gold
            # vsan-gold: Retain policy, RAID-6 FTT:2, thick provisioned
                # Image layers are the most critical data in Harbor — this is why
                # gold storage is used. A lost image layer cannot be recovered
                # without rebuilding the image from source.
      size: 500Gi
            # 500 GB initial allocation. Size depends on:
                # number of images × average image size × number of tags kept
                # For NovaCrestBank: ~50 banking microservices, ~200 MB each,
                # ~20 tags each = ~200 GB for apps + headroom for mirrored Red Hat images
    database:
      storageClass: vsan-standard
      size: 20Gi
            # PostgreSQL: 20 GB is generous — Harbor's DB is small
                # (image metadata, scan results, accounts, policies)
    redis:
      storageClass: vsan-standard
      size: 5Gi
            # Redis: Harbor uses Redis for job queuing and caching
    jobservice:
      storageClass: vsan-standard
      size: 5Gi

trivy:
  enabled: true
    # Enables the embedded Trivy scanner pod.
    # Trivy downloads its vulnerability database from GitHub on startup.
    # In a fully air-gapped environment, mirror the Trivy DB to a local HTTP server
    # and set trivy.offlineScan: true with a local DB URL.
  gitHubToken: ""
    # Optional: GitHub token to avoid rate limiting on Trivy DB downloads.
    # Trivy pulls its CVE database from GitHub releases. Without a token,
    # GitHub rate-limits unauthenticated requests to 60/hour.
    # For production: create a GitHub PAT with no scopes (read-only public data).

updateStrategy:
  type: Recreate
    # Recreate (not RollingUpdate) for Harbor's stateful components.
    # The registry and database pods use ReadWriteOnce PVCs — only one pod
    # can mount them at a time. RollingUpdate would leave the old pod holding
    # the PVC while the new pod tries to mount it, causing a deadlock.

podSecurityContext:
  runAsNonRoot: true
    # OCP SCC compatibility — do NOT specify runAsUser here.
    # OCP assigns a UID from the namespace's range automatically.
    # Specifying a fixed UID (Harbor chart default: 10000) fails on OCP
    # because OCP's restricted SCC does not allow arbitrary UID selection.

nginx:
  enabled: false
    # Disable Harbor's built-in nginx — OCP Ingress Router handles TLS.
    # Harbor's nginx would add an unnecessary extra proxy hop.

notary:
  enabled: false
    # Disable Notary (Docker Content Trust) — NovaCrestBank uses cosign instead.
    # cosign is the modern, OCI-native signing standard used by Red Hat and CNCF.
    # Notary v1 is legacy — it uses a different signing format not supported by OCP's
    # built-in signature verification policy.

Registry-01 · Section 06
Configure NovaCrestBank-IssuingCA certificate for Harbor
Harbor is already served by the OCP wildcard certificate via the Ingress Route. This section handles the remaining certificate requirement: Docker and OCP nodes must trust Harbor's certificate when pushing and pulling images.
Why nodes need to trust Harbor's certificate separately from OCP
The additionalTrustBundle in OCP-VCF-01 already added our IssuingCA to every RHCOS node. Does that mean nodes already trust Harbor's certificate?
Yes — and this is one of the major benefits of having already done the PKI work in WIN-VCF-03. Because harbor.apps.ocp.novacrestbank.local is served by the OCP wildcard certificate (signed by NovaCrestBank-IssuingCA), and because every RHCOS node already trusts NovaCrestBank-IssuingCA via additionalTrustBundle, every node in the cluster can pull from Harbor without any additional certificate configuration.

What does require action: developer workstations and the Jenkins pod. Any host that was not provisioned with the additionalTrustBundle — such as a developer's laptop or a Jenkins agent that runs outside OCP — will see a certificate error when pushing to Harbor. Those machines need the chain.pem installed into their system trust store. On RHEL/Fedora: cp chain.pem /etc/pki/ca-trust/source/anchors/ && update-ca-trust. On Ubuntu/Debian: cp chain.pem /usr/local/share/ca-certificates/ && update-ca-certificates. On macOS: add to Keychain.

Registry-01 · Section 07
AD LDAP integration — NovaCrestBank domain accounts for Harbor
Configure Harbor to authenticate against DC01. After this, developers and operations engineers log into Harbor with their NovaCrestBank domain credentials — no separate Harbor passwords.
Why AD integration matters beyond convenience
Separate local passwords in Harbor mean: passwords that rotate independently of AD, accounts that survive when a developer leaves NovaCrestBank (their AD account is disabled but their Harbor local account still works), and no connection to the NovaCrestBank identity governance process. LDAP integration means: when HR disables an AD account, Harbor access is revoked automatically. When a developer changes their AD password, it applies to Harbor immediately. PCI-DSS Req 8.2.1 requires all user IDs to be uniquely identified and managed through a single identity system.

Registry-01 · Section 08
Create projects — banking-apps, base-images, and platform
Projects are the primary organisational and policy unit in Harbor. Three projects map to the three categories of images NovaCrestBank manages — each with different access controls and vulnerability policies.
Why three projects — the principle behind the structure
Why not one project for everything? What does the three-project structure achieve?
Different image categories have different trust levels, different teams responsible for them, and different vulnerability tolerance levels. A single project forces the same policy onto images that warrant different treatment.

banking-apps contains NovaCrestBank's own application images — the highest trust requirement because they run in the cardholder data environment. Only the Jenkins robot account can push here. Any critical CVE blocks deployment. Every image must be cosign-signed before OCP will run it.

base-images contains the curated foundation images — UBI base images, approved JDK images, approved Python images. The platform team manages this project. These are the approved FROM layers for all NovaCrestBank Dockerfiles. A developer cannot use a random public image as their base — they must use an image from this project. This is how you prevent log4shell-style vulnerabilities where a dependency pulled from the internet compromises the entire application.

platform contains mirrored Red Hat images — OCP release images, operator images, Prometheus, Grafana. This project is populated by replication rules, not by humans. It is the source of truth for all platform-level pulls in an air-gapped configuration. Access is read-only for all humans and the OCP pull robot account.

Configure OCP to pull from Harbor — adding Harbor to the global pull secret
OCP's global pull secret (the one baked in during OCP-VCF-01) authenticates against Red Hat's registries. Harbor is added to it so every node in the cluster can pull application images from Harbor without any per-namespace secret configuration.
Global pull secret vs per-namespace imagePullSecret — which to use
OCP allows per-namespace imagePullSecrets on ServiceAccounts or per-Pod imagePullSecrets. Why use the global pull secret instead?
The global pull secret on the cluster is applied to every node's CRI-O configuration — it is available cluster-wide without any per-namespace configuration. This is the right approach for an internal registry used by all application teams.

Per-namespace imagePullSecrets are appropriate when: different teams have different registries (team A uses Registry X, team B uses Registry Y), or when you need fine-grained control over which namespaces can access which registries. For NovaCrestBank where Harbor is the single internal registry used by everyone, the global pull secret is simpler and ensures no namespace is accidentally locked out of Harbor because someone forgot to add an imagePullSecret to the ServiceAccount.

Important: the global pull secret is stored in the openshift-config namespace as the Secret pull-secret. Modifying it triggers a rolling restart of all MachineConfig objects — nodes reboot one at a time to pick up the new CRI-O configuration. This takes 10–15 minutes and should be done during a maintenance window.

Registry-01 · Section 11
Trivy scanner — what it scans, what it catches, and how to read the results
Trivy is an open-source vulnerability scanner embedded in Harbor. Understanding what it actually checks — and what it misses — is essential before treating its reports as a compliance artefact.
What Trivy actually scans inside a container image
Trivy says an image is "clean." Does that mean it is safe to deploy in a banking environment?
A clean Trivy scan means: no known CVEs were found in the OS packages and application dependencies at the time of the scan, according to the vulnerability databases Trivy consulted. It does not mean the image is free of security issues — it means it has no known, indexed vulnerabilities that Trivy's databases cover.

What Trivy does scan: OS-level packages (RPM database on UBI, DEB packages on Debian/Ubuntu), language ecosystem package manifests (Maven pom.xml, pip requirements.txt, npm package-lock.json, Go go.sum), and binary analysis for some languages. It checks each package version against: NVD (National Vulnerability Database), Red Hat Security Advisories (RHSA), GitHub Security Advisories, and vendor-specific advisory feeds.

What Trivy does not scan: Custom application code (it is not a SAST/DAST tool — it cannot find SQL injection in NovaCrestBank's payment code), secrets hardcoded in image layers (use a separate secrets scanner for this), misconfiguration (use OPA/Gatekeeper for policy checks), runtime behaviour (use ACS for runtime threat detection — Phase 6, OCP-OPS-03).

For NovaCrestBank's compliance: Trivy results satisfy PCI-DSS Req 6.3.3 (installed software protected against known vulnerabilities) when: scans run automatically on every push, scan results are retained as audit evidence, critical CVEs are blocked, and scan results are reviewed regularly. Trivy alone does not satisfy the full software security requirement — it works alongside ACS runtime scanning and SAST tools.

Registry-01 · Section 12
Scan policy — enforce CVE blocking at the registry and at the cluster
Harbor's project-level vulnerability policy is the first gate. OCP's ClusterImagePolicy (via Sigstore/cosign) is the second gate. A malicious or compromised image must defeat both to run in production.
Two-gate enforcement — why both layers are needed
Harbor's "prevent vulnerable images from running" setting blocks pull requests for images that exceed the CVE threshold. But Harbor only enforces this when the pull comes through Harbor's API correctly. A misconfigured deployment that somehow bypassed Harbor's block would still succeed. The second gate — OCP's ImagePolicy — enforces at the cluster level: OCP will not start a pod running an image that is not signed by NovaCrestBank's cosign key. An unsigned image, regardless of its source, will not run. Together: Harbor ensures images are clean, cosign ensures images are authentic.

The Harbor project vulnerability blocking is already configured in Section 08 ("prevent_vul": "true", "severity": "critical"). Verify it is active:

Registry-01 · Section 13
Why mirror Red Hat registries into Harbor — air-gap, speed, and compliance
Mirroring means copying Red Hat's images from their external registries into the Harbor platform project so OCP nodes never need to contact the internet for any image pull. This is the difference between an internet-dependent cluster and a fully self-contained one.
The case for mirroring in a banking environment
OCP already has a pull secret for Red Hat's registries. Why go through the effort of mirroring everything into Harbor?
Three independent reasons — any one of them alone justifies the effort in a banking environment.

Reason 1 — Operational resilience. Right now, OCP node bootstrapping, operator updates, and cluster upgrades all depend on connectivity to registry.redhat.io and quay.io. If Red Hat's CDN has an outage — which has happened — those operations stall. With mirroring, every pull happens on the LAN. An OCP upgrade proceeds whether the internet is up or not. In banking, "we couldn't upgrade because Red Hat's CDN was down during our change window" is not acceptable.

Reason 2 — Network security compliance. PCI-DSS Req 1.3.2 requires that outbound internet connections from the cardholder data environment are restricted to those necessary for the cardholder data environment to function. OCP nodes pulling images from the internet on every pod restart is hard to justify as "necessary." With Harbor mirroring, the firewall rule allowing OCP nodes to reach registry.redhat.io can be removed — nodes only need to reach harbor.novacrestbank.local, which is internal. This reduces the attack surface significantly.

Reason 3 — Image provenance and auditing. Red Hat's images change over time — an operator image tagged :latest pulled today is not the same image as the same tag pulled last month. With mirroring, NovaCrestBank controls exactly which versions of Red Hat images are in the environment. A specific Red Hat advisory is only applied when NovaCrestBank's operations team deliberately updates the mirror — not automatically because a node happened to pull a new tag. Every change to the platform image inventory is a deliberate, audited action.

Registry-01 · Section 15
ImageDigestMirrorSet — redirect OCP node pulls to Harbor
Mirroring copied the images into Harbor. The ImageDigestMirrorSet tells every OCP node: "when you need an image from registry.redhat.io or quay.io, pull it from Harbor instead." Without this step, nodes still pull from Red Hat's registries even though the images exist in Harbor.
How image redirection works in OCP — IDMS vs the old ICSP
What is an ImageDigestMirrorSet and how is it different from ImageContentSourcePolicy?
ImageDigestMirrorSet (IDMS) is the modern OCP 4.13+ replacement for the deprecated ImageContentSourcePolicy (ICSP). Both do the same thing: they configure CRI-O on every node to check a mirror registry before the original source registry when pulling an image by digest.

The mechanism: CRI-O has a registries.conf file that can specify mirrors for source registries. When a pod spec says image: registry.redhat.io/ubi9/ubi-minimal@sha256:abc123, CRI-O first tries harbor.apps.ocp.novacrestbank.local/platform. If it finds the image there by digest (sha256:abc123), it pulls from Harbor. If not, it falls through to the original source. The IDMS applies this config cluster-wide via the MCO — a rolling node restart applies the CRI-O configuration.

Critical: redirection works by digest, not by tag. :latest tags are not reliably redirectable — two different machines may resolve :latest to different sha256 digests. oc-mirror automatically pins all image references to their sha256 digest in the generated manifests.

Registry-01 · Section 16
Image signing with cosign — why it matters and how it works
A clean Trivy scan proves an image has no known CVEs. A cosign signature proves an image was built by NovaCrestBank's authorised Jenkins pipeline — not replaced, not tampered with, not injected from an unauthorised source.
The threat cosign defends against — and why CVE scanning alone is not enough
We already scan images with Trivy. What does cosign add that Trivy does not cover?
Trivy checks what is in an image. cosign verifies who created it and that it hasn't been altered since. They are completely complementary — a secure supply chain needs both.

Consider this attack scenario without cosign: a developer with push access to Harbor is compromised. The attacker pushes a malicious version of the payment-service image — it passes Trivy's CVE scan because the malicious code is custom, not a known CVE. The image tag :v1.2.3 is overwritten with the malicious image. OCP pulls it on the next deployment. The attacker's code runs in production.

With cosign: the Jenkins pipeline signs every image it builds using NovaCrestBank's private signing key (stored in a Hardware Security Module or OCP Secret). The signature is stored in Harbor alongside the image as a cosign attestation. OCP is configured with a ClusterImagePolicy that refuses to admit any pod running an image that is not signed by NovaCrestBank's signing key. The attacker's overwritten image has no valid signature — OCP refuses to run it.

cosign uses Sigstore — an open standard for signing OCI artifacts. The signature is stored as an OCI artifact in the same Harbor repository as the image, tagged with the format sha256-{digest}.sig. No separate infrastructure needed — Harbor stores both the image and its signature natively. Verification is done by the OCP admission controller (via the ClusterImagePolicy resource) before any pod is scheduled.

Registry-01 · Section 18
Garbage collection and storage management
Harbor stores image layers as blobs. When tags are deleted or overwritten, the blobs are not removed immediately — they remain as unreferenced data until garbage collection runs. Without scheduled GC, the 500 GB vsan-gold PVC fills up over months.
Garbage collection in Harbor requires a registry read-only window
During garbage collection, Harbor puts the registry into read-only mode — pushes are blocked, pulls continue. For NovaCrestBank, schedule GC during a known quiet period (e.g., Sunday 02:00–03:00) when no CI/CD pipelines are active. The typical GC window is 5–15 minutes. Configure in Harbor UI under Administration → Garbage Collection → Set Schedule.
GC scheduleCron: 0 2 * * 0 — every Sunday at 02:00
Delete untagged artifactsEnabled — remove image layers that have no tag pointing to them
Tag retention policy (banking-apps)Retain: last 10 tags per repository. Retain: tags matching v*.*.* (semantic versions) for 180 days. Delete: tags matching latest older than 7 days.
Tag retention policy (platform)Retain: last 5 tags per repository. Mirror jobs update in-place — older mirror tags can be removed.
