VCF-02 · Section 02
From vDS to NSX — what you already know, and what is genuinely new
The candidate knows vDS well. NSX replaces and extends vDS — but the mental model transfers directly

The key insight — NSX moves the network into the hypervisor
Everything in the table above can be summarised in one sentence: NSX moves switching, routing, and firewalling from external hardware appliances into the ESXi hypervisor kernel. The physical network becomes a simple IP fabric that carries encrypted GENEVE tunnels. All the intelligence — routing decisions, firewall policy enforcement, load balancing — happens in software inside the ESXi host, at line rate, for every VM.
VCF-02 · Section 03
TEP — Tunnel Endpoint — the engine of the NSX overlay
The most fundamental new concept for a vDS-familiar administrator — everything else in NSX depends on understanding the TEP
Core concept
🔶 Tunnel Endpoint (TEP)

A TEP is a VMkernel interface on an ESXi host that NSX uses exclusively to send and receive GENEVE-encapsulated overlay packets. Every ESXi host that participates in NSX gets one or more TEP VMkernels automatically when NSX "prepares" the host. The TEP is NSX's point of entry and exit on the physical network.

What the candidate already knows that maps to the TEP

The candidate knows vmk0 (management), vmk1 (vMotion), and vmk2 (vSAN). Each VMkernel has a purpose and a VLAN. The TEP is another VMkernel — with its own purpose and its own VLAN.

vmk0 (management)VLAN 10 · Carries vCenter/SDDC Manager communication with ESXi
vmk1 (vMotion)VLAN 20 · Carries live VM migration traffic
vmk2 (vSAN)VLAN 25 · Carries vSAN inter-host storage traffic
vmk-tep (TEP) — NEWVLAN 100 · Carries all NSX GENEVE overlay traffic — every VM's packets travel through here encapsulated

The TEP's job in one sentence

When VM-A on esxi-ocp-01 sends a packet to VM-B on esxi-ocp-03, the packet does not travel directly from VM-A to VM-B on a shared VLAN. Instead:

The physical Cisco switch sees only a UDP packet on VLAN 100 — it has no idea that inside that UDP packet is an Ethernet frame from VM-A addressed to VM-B. The Cisco switch does not need to know about the VM's IP addresses, MAC addresses, or subnet. It just forwards UDP packets. This is the power of the overlay — the physical network is completely decoupled from the virtual network.

Why the TEP needs its own VLAN — not shared with management traffic
The TEP could theoretically use VLAN 10 (management). Why does NSX require a dedicated VLAN?
Because TEP traffic is high-volume and latency-sensitive — it carries all VM-to-VM traffic across hosts. Mixing it with management traffic on VLAN 10 would cause contention, and more importantly, it would make QoS and traffic engineering impossible.

In practice: a 5-VM OCP cluster with 3 masters, 3 infra nodes, and 3 workers generates constant etcd replication traffic, pod-to-pod communication, and monitoring metrics — all as GENEVE packets on the TEP VLAN. If this shared VLAN 10 with management, a burst of SDDC Manager communication could delay etcd replication. etcd delays cause master nodes to lose quorum. Losing etcd quorum brings the entire OCP cluster down.

The dedicated TEP VLAN (VLAN 100 in NovaCrestBank's plan) is configured with MTU 9000 on the Cisco fabric — jumbo frames to avoid fragmentation of the GENEVE-encapsulated packets. NIST SP 800-53 SC-8 — Transmission Confidentiality: the dedicated TEP VLAN is also where GENEVE encryption (when enabled) protects all VM traffic in transit.
VCF-02 · Section 04
GENEVE encapsulation — exactly how a packet travels from one VM to another
The mechanical detail of what happens inside the ESXi kernel when a VM sends a packet across an NSX overlay network
The postal analogy for GENEVE encapsulation
Imagine VM-A wants to send a letter to VM-B. The letter is the VM's Ethernet frame (addressed to VM-B's MAC address). In a traditional VLAN network, the letter goes directly — the carrier (Cisco switch) reads the address and delivers it. With GENEVE, the letter is put inside an envelope. The envelope is addressed from TEP-A (10.100.0.21) to TEP-B (10.100.0.23) — addresses that the postal system (Cisco fabric) understands. The postal system delivers the envelope without ever looking inside. At TEP-B, the envelope is opened, and the original letter is delivered to VM-B. The postal system (Cisco) only needed to understand physical TEP addresses, not the original letter's VM addresses.

The GENEVE packet structure — from outside to inside

The four layers — and who reads each one

LayerWho creates itWho reads itWhat it contains
Outer Ethernet ESXi TEP NIC Cisco switch TEP MAC addresses. Cisco switches forward based on this — they never look deeper.
Outer IP + UDP ESXi TEP VMkernel Cisco L3 routing Source TEP IP → Destination TEP IP. UDP port 6081. Cisco routes this to the right destination host.
GENEVE header NSX kernel module NSX on destination ESXi VNI (Virtual Network Identifier) — a 24-bit number that identifies which NSX segment this packet belongs to. Like a VLAN ID, but with 16 million possible values.
Inner Ethernet frame VM guest OS Destination VM's OS The original packet from VM-A — unchanged. Source/destination VM MAC and IP addresses. The application data.
The VNI — why GENEVE can support far more networks than VLANs
Physical VLANs are limited to 4094 (12-bit VLAN ID). Why can NSX support so many more networks?
The GENEVE VNI (Virtual Network Identifier) is a 24-bit field — supporting 2^24 = 16,777,216 unique network identifiers. This is why NSX can give every OpenShift namespace its own isolated logical network, every tenant its own segment, without running out of network IDs — the 4094 VLAN limit would be exhausted in days in a large Kubernetes environment.

For NovaCrestBank's OCP deployment with dozens of namespaces: each namespace can have its own NSX segment with its own VNI. The OCP Router maps each Route to the correct segment. The NSX DFW enforces isolation between namespaces using their VNIs. The physical network never needs a new VLAN — all of this is in software.
VCF-02 · Section 05
Transport Zone — the scope of the overlay fabric
The Transport Zone defines which ESXi hosts can participate in a given overlay network — a policy boundary, not a switching construct
Core concept
🟢 Transport Zone (TZ)

A Transport Zone is a named scope that defines which ESXi hosts participate in an NSX overlay fabric. Any segment created within a Transport Zone is automatically reachable from any ESXi host that is a member of that Transport Zone. A host that is not in the Transport Zone cannot participate in its segments — even if it has NSX installed.

The post code district analogy
Think of a Transport Zone like a post code district. Every building (ESXi host) in the district has postal delivery (NSX overlay participation). A letter (a VM packet) addressed within the district can be delivered to any building in it directly. A building outside the district cannot receive district mail through the normal routing — it requires special handling. The Transport Zone is not the post office — it is the definition of the district itself.

The two Transport Zone types

Overlay Transport Zone

Used for GENEVE-encapsulated overlay segments — the NSX logical networks. Segments in this TZ have VNIs, exist entirely in software, and require no physical VLAN. This is the primary TZ for all NovaCrestBank workloads.

NovaCrestBank creates one: NCB-Overlay-TZ — spanning all 9 ESXi hosts (both Management Domain and OCP Workload Domain).

VLAN Transport Zone

Used for VLAN-backed segments — logical constructs that map to physical VLANs. These are needed for the Edge Nodes to access the physical uplink VLANs (the Edge Uplink VLANs 102 and 103). They are the bridge between NSX and the physical world.

NovaCrestBank creates one: NCB-VLAN-TZ — used only by the Edge Nodes.

Why NovaCrestBank needs both Transport Zone types
The Overlay TZ handles all VM traffic in software. Why is the VLAN TZ needed at all?
Because the NSX Tier-0 Gateway (which runs on Edge Nodes) needs to connect to the physical Cisco fabric for BGP peering. That connection uses physical VLAN-backed uplinks — not GENEVE tunnels. The VLAN TZ provides the logical construct that maps the Edge Node's uplink to the physical VLAN.

Think of it this way: everything inside NSX is overlay (GENEVE). But at the border — where NSX meets the physical Cisco switches — there must be a translation point. The Edge Node's VLAN-backed uplink (in the VLAN TZ) is that translation point. Traffic from an OCP pod crosses multiple boundaries: pod → NSX overlay (Overlay TZ) → Edge Node → VLAN-backed uplink (VLAN TZ) → Cisco physical switch → Palo Alto → internet.

NovaCrestBank's Transport Zones

NCB-Overlay-TZType: Overlay · All 9 ESXi hosts (mgmt-01 through 04, ocp-01 through 05) · Used by all workload segments (seg-infra-services, seg-ocp-nodes)
NCB-VLAN-TZType: VLAN · Edge Nodes only · Used for Edge Uplink segments (VLAN 102 and VLAN 103) that connect T0 to Cisco fabric
VCF-02 · Section 06
Segment — NSX's logical switch and what replaces VLANs
A Segment is the fundamental unit of network connectivity in NSX — it is what VMs and OCP nodes connect to
Core concept
🔵 Segment (Logical Switch)

A Segment is a logical Layer-2 network in NSX. VMs connect to segments the same way they connect to vDS port groups — through a virtual NIC setting. But unlike port groups, segments are software-defined and require no physical VLAN configuration. VMs on the same segment can communicate directly. VMs on different segments communicate through a Tier-1 Gateway.

Segment vs vDS Port Group — the practical differences

Routing between segments? Requires external router or firewall Tier-1 Gateway routes between segments — in software, in the hypervisor

How NovaCrestBank's segments look to the VMs

DC01 is connected to segment seg-infra-services. From DC01's perspective, it is on a normal Ethernet network — it has IP 10.70.0.10, it can ping other VMs on 10.70.0.0/24, and it can communicate with OCP nodes (via the T1 and T0 gateways) just like on any other routed network. DC01 does not know or care that its "network" is a GENEVE overlay running on top of a physical VLAN 100 fabric. It sees exactly the same experience it had on VLAN 70 in the previous classic vSphere environment.

The segment looks identical to the VM — the change is invisible to applications
This is a critical operational point: migrating from classic vSphere (VMs on VLAN 70) to VCF+NSX (VMs on seg-infra-services) is transparent to the applications and VMs. The IP addresses stay the same. The gateway IP stays the same. The firewall rules apply via DFW instead of a physical firewall — but the VM's network experience is unchanged. No application reconfiguration is needed.
VCF-02 · Section 07
Overlay segments vs VLAN-backed segments — and when each is used
Not all NSX segments are overlay segments — understanding the two types prevents confusion in the NSX Manager GUI

Overlay Segment

What it is: A GENEVE-encapsulated logical network. Has a VNI (not a VLAN ID). Spans all hosts in the Overlay Transport Zone. No physical VLAN configuration required.

Used for: All workload networks — infrastructure VMs, OCP nodes, OCP application namespaces. 99% of the segments NovaCrestBank creates.

Examples in NovaCrestBank: seg-infra-services, seg-ocp-nodes

VLAN-Backed Segment

What it is: An NSX logical construct that maps to a specific physical VLAN. Has a VLAN ID (not a VNI). Only available in the VLAN Transport Zone. Passes traffic as tagged VLAN packets on the physical fabric — no GENEVE encapsulation.

Used for: Edge Node uplinks — the physical VLAN connections between Edge Nodes and Cisco leaf switches. Also used for VLAN-backed Edge TEP connections.

Examples in NovaCrestBank: seg-edge-uplink-a (VLAN 102), seg-edge-uplink-b (VLAN 103)

In the NSX Manager GUI — how to tell which type you are creating
When creating a segment in NSX Manager, there is a field called "Transport Zone." If you select NCB-Overlay-TZ, the segment becomes an overlay segment (no VLAN ID field appears — NSX assigns a VNI automatically). If you select NCB-VLAN-TZ, a VLAN ID field appears — you enter the physical VLAN number. This distinction appears in NSX-03 when creating the edge uplink segments.
VCF-02 · Section 08
Tier-1 Gateway — the tenant router
The T1 provides routing between segments without requiring any external device — it runs entirely in the ESXi hypervisor kernel
Core concept
🟢 Tier-1 Gateway (T1)

A Tier-1 Gateway is a software router that lives in the NSX distributed routing layer — inside the ESXi kernel on every host. It connects one or more segments and provides routing between them, and it connects upward to a Tier-0 Gateway for access to the physical world. The T1 is the "tenant router" — each logical group of related workloads (infrastructure VMs, OCP nodes) gets its own T1.

The office floor analogy
Think of a large office building with multiple floors. Each floor is a Segment (a separate network). The T1 Gateway is the internal staircase — it connects the floors to each other within the building. The T0 Gateway is the building's main exit — connecting the entire building to the street (physical network). Someone on floor 2 (VM on seg-infra-services) can reach floor 5 (VM on seg-ocp-nodes) via the T1 staircase. To reach outside the building (internet, external services), they must pass through the T0 exit.

Where the T1 actually runs — distributed routing

This is the most surprising aspect of the T1 for a vDS-familiar administrator: the T1 does not run in a single VM or appliance. It runs in a distributed fashion inside the ESXi kernel on every host in the Transport Zone.

The T1 routing decision happens locally on the source ESXi host — the packet is routed by the T1 kernel component on esxi-ocp-01, then encapsulated in GENEVE, and sent to esxi-ocp-03's TEP. The destination ESXi decapsulates and delivers. No traffic goes through any external router or VM appliance for T1 routing — it is all in the kernel. This is why inter-segment routing in NSX is as fast as switching — there is no extra hop.

VCF-02 · Section 09
Tier-0 Gateway — the BGP border between NSX and the physical world
The T0 is where the NSX overlay meets the physical Cisco fabric — it runs on Edge Nodes, not in the hypervisor kernel
Core concept
🔶 Tier-0 Gateway (T0)

The Tier-0 Gateway is a software router that runs on dedicated NSX Edge Node VMs (not distributed across ESXi hosts like the T1). The T0 peers with the physical Cisco leaf switches via BGP, advertising NSX segment routes into the physical network and receiving the default route back. All traffic leaving the NSX overlay for the physical world passes through the T0.

BGP — why the T0 uses it to talk to Cisco

Why BGP — not a static default route from T0 to Cisco
The T0 could be configured with a static default route pointing to the Cisco leaf switches. Why does NovaCrestBank use BGP instead?
Because NovaCrestBank has two Edge Nodes (edge-01 and edge-02) and two Cisco leaf switches (Leaf-01 and Leaf-02). Static routes cannot automatically fail over — if Leaf-01 fails, a static route pointing to Leaf-01 is dead until an administrator manually changes it. BGP detects the peer failure within seconds and automatically uses the surviving path.

More specifically, BGP provides:

1. Automatic failover: T0 peers with both Leaf-01 (on VLAN 102) and Leaf-02 (on VLAN 103). If Leaf-01's BGP session drops, all traffic automatically flows through Leaf-02 — within the BGP hold timer (typically 3-9 seconds).

2. Route advertisement: The T0 advertises NSX segment routes (e.g., 10.50.0.0/24, 10.70.0.0/24) to the Cisco fabric. The Cisco spines then know how to reach OCP node IPs and infrastructure VM IPs without any static route configuration on the switches.

3. ECMP: With Active-Active T0 mode, both Edge Nodes advertise the same routes. Cisco performs ECMP — load-balancing traffic equally across both Edge Nodes.

NIST SP 800-53 CP-7 — Alternate Processing Site: dynamic BGP routing ensures that a single edge device failure does not isolate the NSX environment from the physical network.

The routing hierarchy in NovaCrestBank

Reading the diagram from bottom to top: VMs live on Segments. Segments connect to T1 Gateways. T1 Gateways connect to the T0 Gateway. The T0 Gateway connects to the Cisco fabric via BGP. The Cisco fabric connects to the Palo Alto NGFW. The Palo Alto connects to the internet.

VCF-02 · Section 11
Distributed Firewall — security without appliances
The most powerful security concept in NSX — a stateful firewall that runs inside every ESXi host's kernel, inspecting every packet before it enters the network
Core concept
🔴 NSX Distributed Firewall (DFW)

The NSX Distributed Firewall is a stateful, kernel-level firewall that runs inside every ESXi host. Every packet sent by every VM passes through the DFW before it enters the virtual switch. The DFW enforces policy at the VM's virtual NIC — the most granular possible enforcement point. DFW policy is defined centrally in NSX Manager and automatically distributed to all ESXi hosts.

The security guard in the office analogy
A traditional perimeter firewall is a security guard at the building entrance. Everyone inside the building can interact freely — the guard only checks who enters or leaves. The NSX DFW is a security guard sitting at every desk, checking every piece of paper that passes between colleagues. Even if a colleague walks across the room (VM-to-VM on the same ESXi host), the DFW checks that interaction. The perimeter guard and the desk guard serve different purposes — and NovaCrestBank needs both.

The critical difference — east-west vs north-south

North-South traffic

Traffic entering or leaving the data centre — from internet to VMs, from VMs to external services. This passes through the Palo Alto NGFW at the perimeter. The Palo Alto inspects this traffic.

Example: A user's browser connecting to banking-portal.apps.ocp.novacrestbank.local

East-West traffic

Traffic between VMs inside the data centre — VM to VM, pod to pod, OCP node to DC01. This traffic never passes through the Palo Alto. The DFW is the only firewall that sees it.

Example: An OCP pod querying DC01 on 10.70.0.10 for LDAP authentication — this never leaves the data centre, so the Palo Alto cannot inspect it. DFW does.

PCI-DSS v4.0 Requirement 1.3 — Network access controls between trusted and untrusted zones
Without DFW, east-west traffic inside the data centre is completely uninspected. A compromised VM could communicate freely with any other VM on the same segment — including payment processing VMs, database servers, or domain controllers. PCI-DSS Requirement 1.3 mandates that inbound and outbound traffic be restricted to what is necessary. DFW implements this requirement for all east-west traffic — the Palo Alto implements it for north-south traffic. Together they satisfy the full requirement.

How DFW rules work

DFW rules are defined in NSX Manager and consist of:

  • Source: A group of VMs (by tag, segment, IP, or NSX group). Example: "All VMs tagged as OCP-Nodes"
  • Destination: Another group. Example: "All VMs tagged as Infrastructure-Services"
  • Service: A protocol and port. Example: "TCP 636 (LDAPS)"
  • Action: Allow or Deny

Rules are evaluated in order (top to bottom). The last rule in NovaCrestBank's DFW policy is a default deny — any traffic not matching a preceding Allow rule is silently dropped. PCI Req 1.2.1 — all allowed traffic flows are explicitly permitted; everything else is denied.

The defence-in-depth model — NIST SP 800-53 SC-7 and SC-7(13)

The Palo Alto provides the perimeter boundary. DFW provides internal microsegmentation. Together they implement defence-in-depth — an attacker who breaches the perimeter still faces DFW controls on every VM-to-VM path. An attacker who somehow reaches an OCP worker node cannot laterally move to DC01 without passing a DFW rule that explicitly blocks it.
VCF-02 · Section 13
Gateway Firewall — the T1 north-south control
A second type of firewall in NSX — applied at the T1 Gateway level for traffic entering and leaving a tenant domain
Core concept
🔶 Gateway Firewall

The Gateway Firewall applies at the T1 (and T0) Gateway boundary — it inspects traffic as it crosses from one domain to another (e.g., traffic leaving seg-ocp-nodes heading to seg-infra-services, which must pass through T1-OCP → T0 → T1-Infra). It complements the DFW — DFW is per-VM, the Gateway Firewall is per-domain-boundary.

DFW — per VM enforcement

Rules applied at the VM's virtual NIC. Inspects all traffic from/to a specific VM regardless of source or destination. Ideal for intra-segment rules (VMs on the same segment) and fine-grained VM-level policy.

Gateway Firewall — per domain boundary

Rules applied at the T1 gateway interface. Inspects traffic as it crosses between routing domains (segment → T1 → T0 → T1 → segment). Ideal for domain-level policy — "OCP segment is only allowed to send DNS to infrastructure segment."

In NovaCrestBank's DFW policy (defined in VCF-01 Section 15), the inter-zone rules (e.g., "OCP Nodes → Infrastructure Services on TCP 636 LDAPS only") are implemented as Gateway Firewall rules on NCB-T1-OCP. The intra-segment rules (OCP pod to OCP pod) are implemented as DFW rules. Both are configured in NSX Manager's Security section — covered in the NSX-05 guide.

VCF-02 · Section 14
NSX Manager — the control plane
NSX Manager is the single GUI and API for the entire NSX-T deployment — understanding its role before the first login
NSX Manager is to NSX what vCenter is to vSphere
The candidate knows vCenter: it is the management plane for ESXi hosts. The ESXi hosts do the actual work (running VMs, vSAN, etc.) — vCenter configures and monitors them. NSX Manager plays the same role for the NSX data plane. The ESXi hosts do the actual networking work (routing, switching, DFW enforcement) — NSX Manager configures and monitors them. Just as vCenter does not run VMs itself, NSX Manager does not forward packets itself.

The three-node NSX Manager cluster

NSX Manager is deployed as a cluster of 3 identical VMs. All three run the same active role — there is no primary/standby. Any NSX Manager node can serve the GUI and API. A virtual IP (VIP) — nsx-mgr.novacrestbank.local at 10.10.0.63 — floats between the three nodes and always points to the active node. Administrators always use the VIP, never an individual node IP.

Why 3 nodes (not 1 or 2)etcd consensus — NSX Manager uses etcd (same as OCP) internally. etcd requires a majority quorum: 3 nodes = quorum of 2 = tolerates 1 failure. 2 nodes cannot form quorum. 1 node is a single point of failure.
What NSX Manager controlsTransport Zones · IP Pools · Segments · T0/T1 Gateways · DFW rules · Load Balancer Virtual Servers · Edge Node deployment · Host preparation (NSX VIB installation on ESXi)
What NSX Manager does NOT doForward packets (the ESXi kernel and Edge Nodes do that). Store VM data. Replace vCenter (they have separate roles and separate GUIs).
PCI-DSS v4.0 Requirement 12.3.2 — Targeted risk analysis for critical components
NSX Manager controls the network and firewall policy for the entire VCF environment — including the cardholder data environment. Access to NSX Manager must be restricted to authorised personnel only (implemented via RBAC integration with Active Directory — covered in WIN-VCF-01) and all access must be logged. PCI-DSS requires a targeted risk analysis for critical infrastructure components like NSX Manager, documented and reviewed annually.

Edge Node — where routing and load balancing physically run
The Edge Node is a special-purpose VM that runs the T0 Gateway, handles BGP peering, and hosts the NSX Load Balancer — the bridge between NSX and the physical world
Core concept
🔶 NSX Edge Node

An NSX Edge Node is a large VM (8 vCPU, 32 GB RAM) deployed on ESXi hosts in the Management Domain. It hosts components that require centralised execution: the Tier-0 Gateway (BGP routing), the NSX Load Balancer, and NAT. Unlike the T1 (distributed across all ESXi hosts), the T0 must run in a specific place because it has physical network connections (the BGP uplinks to Cisco) — it cannot be fully distributed.

Why the Edge Node is a "Large" VM — not Medium or Small

Edge Node sizing matters for NSX Load Balancer support
NSX Edge Nodes come in Small, Medium, Large, and Bare Metal sizes. Why does NovaCrestBank use Large?
Because the NSX Load Balancer — required for OCP API VIP and Ingress VIP — is only supported on Large or Bare Metal Edge Nodes. Small and Medium Edge Nodes support basic routing only.

Large Edge Node specifications: 8 vCPU, 32 GB RAM, 200 GB disk. These resources are dedicated to: BGP routing (CPU-intensive for large routing tables), NSX Load Balancer processing (maintaining connection state for thousands of concurrent OCP API and application connections), and SNAT/DNAT for any network address translation rules.

For NovaCrestBank with OCP serving banking application traffic: the NSX LB will maintain connection state for all active user sessions to *.apps.ocp.novacrestbank.local. Large Edge sizing provides sufficient memory for the connection table and CPU headroom for TLS processing at the load balancer.

NSX Load Balancer — Virtual Servers and server pools
The NSX Load Balancer replaces keepalived for OCP VIPs — understanding its components before the NSX-04 GUI guide
The receptionist analogy for load balancing
Imagine calling a large bank's customer service number (the VIP — 10.50.0.200:6443). The receptionist (NSX LB Virtual Server) answers every call and routes it to an available representative (OCP master node in the server pool). If one representative is unavailable (master node fails), the receptionist does not put the caller through to that person — they route to the next available one. The caller only ever dials one number and never knows there are multiple representatives.

Server Pool A group of backend servers (IP + port) that receive traffic. Configured with a load balancing algorithm and health check. NCB-OCP-API-Pool: master-0:6443, master-1:6443, master-2:6443 with TCP health check every 5 seconds

The NSX Load Balancer health monitoring ensures that OCP API and application traffic is never routed to failed or degraded backend nodes. This satisfies PCI-DSS Req 6.4's requirement for automated protection of public-facing applications — the load balancer is the automated mechanism that prevents sessions from reaching unavailable services.

Packet walk — an OCP pod's packet to the internet, step by step
The complete journey of a single packet from an OCP application pod to an external API — through every NSX component
Scenario
An OCP banking application pod (running on worker-0 at 10.50.0.31) makes an HTTPS request to an external payment API at 203.0.113.50. Here is every step of that packet's journey.
1
Pod creates the packet — destination: 203.0.113.50:443
The Linux process inside the OCP pod opens a TCP connection to 203.0.113.50:443. The pod's networking stack creates an Ethernet frame: source IP = pod's IP (e.g., 10.128.0.5), destination IP = 203.0.113.50. The pod has no route to 203.0.113.50 directly — its default gateway is the NSX T1 interface on 10.50.0.1.
2
DFW inspects the packet at worker-0's vNIC
Before the packet leaves the pod's virtual NIC, the NSX Distributed Firewall kernel module on esxi-ocp-05 (which runs worker-0) inspects it. The DFW checks its rules: "OCP application pods are allowed to send TCP 443 to external destinations." The rule matches — the packet is permitted. DFW tracks the connection state (source IP, destination IP, source port, destination port) in its connection table.
3
T1 routing — NCB-T1-OCP routes to the T0
The T1 kernel component on esxi-ocp-05 sees the packet is destined for 203.0.113.50 — which is not on any connected segment. The T1 forwards it to its next hop: NCB-T0-Gateway. This routing decision happens entirely inside the ESXi kernel — no packet leaves the host yet.
4
GENEVE encapsulation — TEP wraps the packet for physical transit
The T1 kernel has determined the packet needs to go to the Edge Node (which runs the T0). The Edge Node's TEP is at 10.101.0.11 on VLAN 101. The ESXi TEP on esxi-ocp-05 (10.100.0.25) wraps the original Ethernet frame in a GENEVE header + UDP header + outer IP header (src=10.100.0.25, dst=10.101.0.11). The encapsulated packet is sent out on VLAN 100 towards the Cisco fabric.
5
Cisco fabric forwards the UDP packet to the Edge Node
The Cisco leaf switch sees a UDP packet on VLAN 100 from 10.100.0.25 to 10.101.0.11. It routes it to VLAN 101 (Edge TEP VLAN). The Edge Node (edge-01) receives the packet on its VLAN 101 interface. The Cisco switch never reads the inner packet — it only knows about TEP IPs.
6
Edge Node decapsulates and applies T0 routing + NAT
Edge-01 receives the GENEVE packet, strips the encapsulation, and examines the inner packet. The T0 gateway on edge-01 routes the packet: destination 203.0.113.50 matches the default route (0.0.0.0/0) received from Cisco via BGP. The T0 applies SNAT (Source NAT) — replacing the pod's source IP with an external-facing IP so return packets can find their way back. The Gateway Firewall on the T0 may also inspect the packet at this point.
7
T0 sends the packet to Cisco via BGP uplink (VLAN 102)
Edge-01 sends the (now SNAT'd) packet out its VLAN 102 interface (10.102.0.2) to Cisco Leaf-01 (10.102.0.1). This is a normal Layer-3 packet — no GENEVE encapsulation. Cisco routes it toward the Palo Alto NGFW.
8
Palo Alto inspects and forwards to the internet
The Palo Alto sees an outbound HTTPS connection from the bank's external IP to 203.0.113.50. Its security policy allows outbound HTTPS from the OCP egress range. It applies App-ID (identifies it as SSL/TLS), logs the connection, and forwards it to the ISP.
✓
The response packet follows the reverse path
The response from 203.0.113.50 arrives at the Palo Alto, is forwarded to Cisco, reaches edge-01 on VLAN 102. The T0 reverses the SNAT (un-NATs the destination back to the pod's IP). The packet is GENEVE-encapsulated and sent via VLAN 100 back to esxi-ocp-05's TEP. The TEP decapsulates it. The DFW checks the return packet (it matches the existing connection state — allowed). The packet is delivered to the pod. The entire round-trip: typically under 10 milliseconds within the data centre.

Terminology reference — every NSX term explained in one place
A reference card for the GUI guides that follow — if the candidate encounters an unfamiliar term in NSX Manager, find it here
NSX TermPlain language explanationvSphere parallel
TEP
Tunnel Endpoint		 A VMkernel interface on each ESXi host dedicated to sending and receiving GENEVE-encapsulated overlay packets. Lives on the TEP VLAN (VLAN 100). Like vmk2 (vSAN VMkernel) — a dedicated VMkernel with a dedicated VLAN and purpose.
GENEVE The tunnel protocol NSX uses to carry overlay traffic. Wraps a VM's Ethernet frame inside a UDP packet so it can traverse the physical network. No direct parallel — vSphere has no equivalent tunnelling. Similar conceptually to iSCSI encapsulation but for all VM traffic.
VNI
Virtual Network Identifier		 A 24-bit number (like a VLAN ID but with 16 million values) in the GENEVE header that identifies which NSX segment a packet belongs to. VLAN ID — but with 16M values instead of 4094.
Transport Zone A named scope defining which ESXi hosts participate in an NSX overlay. Segments in a TZ are reachable from all hosts in that TZ. A vDS spans hosts — the TZ defines which hosts can participate in NSX segments.
Segment An NSX logical Layer-2 network. VMs connect to segments. No physical VLAN required for overlay segments. vDS Port Group — but software-defined, no physical VLAN needed.
Tier-1 Gateway A software router running distributed in the ESXi kernel. Routes between segments within a tenant domain. Does not require BGP or external connectivity. Like having a router inside vCenter without needing any physical router hardware.
Tier-0 Gateway A software router running on Edge Nodes. Connects NSX overlay to the physical network via BGP. The "exit point" of the NSX environment. Replaces the role of a physical border router between the vSphere environment and the physical network.
Edge Node A large VM (8 vCPU, 32 GB) that hosts the T0 Gateway, NSX Load Balancer, and NAT. Runs in the Management Domain. No direct parallel — a specialised infrastructure VM with physical network connections.
Edge Cluster A group of Edge Nodes (NCB-Edge-Cluster = edge-01 + edge-02). T0 Gateways and Load Balancers are attached to an Edge Cluster. Like a vSphere HA cluster — a group of similar components managed together.
Distributed Firewall (DFW) A stateful firewall running in every ESXi host's kernel. Inspects every packet from every VM before it enters the virtual switch. Central policy, distributed enforcement. No direct vSphere parallel. Closest: NSX DFW replaces the need for external firewall appliances for east-west traffic.
Gateway Firewall A firewall applied at the T1 (or T0) gateway boundary. Inspects traffic crossing between routing domains (e.g., OCP segment → infrastructure segment). Like a zone-based firewall on a router — traffic entering/leaving a routing domain is inspected.
NSX Manager A 3-node cluster of VMs that provides the GUI and API for all NSX configuration. The control plane — does not forward packets. vCenter — manages ESXi hosts without running VMs itself. NSX Manager manages NSX components without forwarding packets.
Transport Node An ESXi host (or Edge Node) that has been "prepared" by NSX — NSX VIB installed, TEP configured, and added to a Transport Zone. An ESXi host added to a vDS — prepared and managed by vCenter. Transport Node = NSX-prepared ESXi host.
NSX Load Balancer A software load balancer running on Edge Nodes. Provides Virtual Servers (VIPs) with server pools and health monitors. Replaces keepalived for OCP VIPs. No direct vSphere parallel. Replaces a physical/virtual load balancer appliance for OCP API and Ingress VIPs.
Host Transport Node The specific term NSX uses for an ESXi host that has been prepared as a Transport Node. In NSX Manager, this is the "Prepare Host" step. Adding an ESXi host to vCenter — the NSX equivalent is preparing it as a Host Transport Node.
