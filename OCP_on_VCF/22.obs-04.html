OBS-04 · Section 01
Why tracing — following one customer transaction across 8 services
Metrics tell you that something is slow. Logs tell you that an error occurred. Traces tell you exactly where, in which service, and in which operation within that service the time was spent on a specific request. For banking microservices, traces are the only tool that can answer "why was this customer's specific payment slow?"

The gap that metrics and logs cannot fill
NovaCrestBank already has Prometheus metrics (OBS-02) and Loki logs (OBS-03). A customer calls to complain their payment took 45 seconds. What can metrics and logs tell you, and where do they fall short?
Prometheus metrics for the payment submission endpoint show P99 latency at 44 seconds for a 10-minute window. Logs for the payment ID show the request was received, processed, and completed. But neither tells you which of the 8 services involved in processing the payment was responsible for the 44 seconds of latency.

The payment submission flow at NovaCrestBank involves 8 services called in sequence or parallel: the API gateway validates the request and authenticates the customer, the fraud detection service scores the transaction, the payment service validates the payment details and checks the account, the account service verifies the account balance, the NPP rail adapter submits to the payment network, the notification service prepares the confirmation, the audit service records the event, and the response aggregator assembles the final response. Total call chain: 8 services.

With metrics alone: P99 is 44 seconds. You know the payment was slow. You do not know whether the 44 seconds was all in the NPP rail adapter (slow payment rail), all in the account service (database lock), all in the fraud detection service (model inference time), or spread across all 8 services.

With logs alone: you can find the specific log lines for payment ID NCB-87654321 across all 8 services. But correlating timestamps across 8 services with millisecond precision, accounting for clock skew, and reconstructing the sequence of operations from flat log lines is manual work — 30+ minutes for an experienced engineer, if everything is structured and timestamped correctly.

With a trace: the trace shows a waterfall diagram where each service's contribution is a horizontal bar. The NPP rail adapter bar is 41 seconds long. The other 7 services contributed the remaining 3 seconds. The NPP adapter's spans show that 41 seconds was spent in the HTTP call to the NPP rail's external endpoint — the rail was slow, not the internal service. Total investigation time: 30 seconds in the Jaeger UI.

OBS-04 · Section 02
OpenTelemetry concepts — traces, spans, and context propagation
OpenTelemetry (OTel) is the industry standard for distributed tracing instrumentation — vendor-neutral, supported by every major cloud provider, and built into most frameworks. Understanding its three core concepts makes the instrumentation code in Sections 09–12 self-explanatory.
Trace
The complete request journey
A trace represents the complete end-to-end journey of one request through the entire system — from the API gateway receiving the HTTP request to the final response being sent back to the client. A trace is identified by a globally unique trace ID (128-bit random number, typically represented as a 32-character hex string like 4bf92f3577b34da6a3ce929d0e0e4736). Every span in the trace carries this same trace ID, which is what allows them to be assembled into the waterfall diagram.
Span
One unit of work within a trace
A span represents one discrete operation — one HTTP call, one database query, one message queue publish. It has: a span ID (unique within the trace), a parent span ID (the span that created it — this is what defines the tree structure), a name (the operation being performed, e.g., POST /api/v1/payments/submit), a start time and duration, and optional attributes (key-value metadata) and events (timestamped log-like records within the span). The trace's waterfall diagram is built by rendering each span as a horizontal bar positioned by its start time and duration.
Context propagation
How the trace ID crosses service boundaries
When the api-gateway calls the payment-service over HTTP, the api-gateway injects the trace ID and its span ID into the HTTP request headers. The standard header is traceparent (W3C Trace Context format): 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01. The payment-service extracts this header on arrival and uses the trace ID to associate its spans with the same trace. This propagation happens automatically when the OpenTelemetry Java agent is attached — no application code required. The payment-service does not need to know anything about OpenTelemetry internals to participate in distributed tracing — it just needs the Java agent and the collector endpoint.
Collector
The aggregation and forwarding layer
The OpenTelemetry Collector is a standalone process (running as a Deployment in OCP) that receives spans from instrumented applications, applies processing (batching, sampling, attribute filtering), and forwards them to Tempo for storage. Applications export to the Collector's OTLP (OpenTelemetry Protocol) endpoint — not directly to Tempo. This indirection provides: central sampling configuration (only the Collector needs updating to change sampling rates), attribute filtering (removing sensitive span attributes before they are stored), and multi-backend export (spans can go to both Tempo and a cloud tracing service from one Collector).
OBS-04 · Section 03
Tempo — trace storage without index cost
Tempo is the trace storage backend. Like Loki (for logs), Tempo is designed to store data in object storage cheaply — the index is tiny and the raw trace data is compressed in MinIO.
Why Tempo rather than Jaeger's own storage backend
Jaeger is the standard distributed tracing UI. Jaeger has its own storage backends (Elasticsearch or Cassandra). Why use Tempo as the storage backend rather than Jaeger's native storage?
Jaeger's Elasticsearch backend has the same problem as the OpenShift EFK logging stack — every span attribute is indexed in Elasticsearch, creating significant storage and compute overhead at high trace volumes. A banking API processing 100 transactions per second, with 8 services per transaction and 5 spans per service, generates 4,000 spans per second. Indexed in Elasticsearch, this creates enormous index maintenance overhead.

Tempo takes Loki's approach: store compressed span data in object storage (MinIO), maintain a tiny index only for trace IDs, and scan the raw data for attribute queries. The trade-off is the same as Loki — searching for traces by attribute (find all traces where payment_type=NPP and duration > 500ms) requires scanning object storage rather than querying an index — slower for attribute search, but dramatically cheaper for storage and maintenance.

For NovaCrestBank's use case, this trade-off is correct: most trace queries are "find me the trace for payment ID NCB-12345" (lookup by trace ID — O(1) in Tempo's trace ID index) or "show me the slowest 10 traces in the last hour" (requires scanning, but Tempo can generate RED metrics — Rate, Errors, Duration — from the span stream that allow Grafana to identify slow traces by service without scanning all stored spans).

Jaeger UI works with Tempo as the backend — NovaCrestBank gets the familiar Jaeger interface for trace investigation while using Tempo's cost-effective storage.
OBS-04 · Section 04
Traces, metrics, and logs — the three pillars working together
OBS-01 through OBS-04 build the complete observability stack. Each pillar answers a different class of question. The power comes from cross-signal correlation — starting with a metric anomaly, finding the relevant logs, then viewing the trace.
SignalToolQuestion it answersCardinalityData per event
Metrics Prometheus + Grafana "Is the service working? How fast? How many errors? How full?" — aggregate health over time Low (bounded label sets) Tiny (one float64 per time series per scrape interval)
Logs LokiStack + Grafana "What happened inside the service? What error occurred? What was the value of this variable?" — event detail High (free-form text) Medium (log line per event, typically 100–500 bytes)
Traces Tempo + Jaeger UI "Where did the time go for THIS specific request? Which service is slow?" — per-request timing breakdown Very high (one trace per request) Large (multiple spans per request, each with attributes)
The investigation workflow — always start with metrics, end with traces
The correct investigation pattern: Metrics → Logs → Traces. Metrics identify that something is wrong (P99 latency spike on the Grafana dashboard). Logs narrow down to when and which error occurred (LogQL query for the time window). The trace for a specific affected request shows exactly where the time was spent. This three-step funnel is efficient because metrics are cheap and always available, logs are medium-cost and always retained, traces are expensive and sampled — you start with the cheap signal and drill down only to the expensive one when you know what you are looking for.

OBS-04 · Section 06
Deploy Tempo — the TempoStack CR
The TempoStack Custom Resource deploys the Tempo backend components. Tempo's architecture stores all span data in MinIO and maintains only a trace ID index in local PVCs — minimal on-cluster storage footprint.
TempoStack — every field explained
apiVersion: tempo.grafana.com/v1alpha1
kind: TempoStack
metadata:
  name: tempo-banking
  namespace: openshift-tempo-operator
spec:
  storage:
    secret:
      name: tempo-minio-creds
      type: s3
  storageSize: 10Gi
    # storageSize: the PVC size for Tempo's WAL (write-ahead log) and
    # local block cache. 10 Gi is sufficient — Tempo writes through to MinIO
    # and only keeps a small local cache. Not the primary data store.

  storageClassName: vsan-standard

  retention: 168h
    # 168h = 7 days trace retention in MinIO.
    # Traces are high-cardinality and high-volume — 7 days is the standard
    # operational retention for distributed tracing.
    # Unlike logs (30–90 days), traces are used for immediate incident investigation,
    # not long-term compliance analysis.
    # For a banking cluster sampling 1% of 100 TPS = 1 trace/second:
    # 7 days × 86,400 seconds × 1 trace × ~5 KB per trace ≈ 3 GB — very manageable.

  template:
    compactor:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations: [{key: node-role.kubernetes.io/infra, effect: NoSchedule}]
    distributor:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations: [{key: node-role.kubernetes.io/infra, effect: NoSchedule}]
    ingester:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations: [{key: node-role.kubernetes.io/infra, effect: NoSchedule}]
    querier:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations: [{key: node-role.kubernetes.io/infra, effect: NoSchedule}]
    queryFrontend:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations: [{key: node-role.kubernetes.io/infra, effect: NoSchedule}]
      jaegerQuery:
        enabled: true
                # enabled: Tempo exposes a Jaeger-compatible API endpoint.
                  # This allows the Jaeger UI to connect to Tempo as its backend —
                  # engineers use the familiar Jaeger interface for trace investigation
                  # while Tempo provides the cost-effective MinIO storage.
        ingress:
          type: route
            # Expose Jaeger UI as an OCP Route so engineers can access it from a browser.

  limits:
    global:
      ingestion:
        ingestionRateLimitBytes: 15000000
          # 15 MB/second maximum ingestion rate per tenant.
              # With 1% sampling at 100 TPS and ~5 KB per trace:
              # 1 trace/second × 5 KB = 5 KB/second — well within limit.
              # The limit prevents a misconfigured sampling rate from flooding Tempo.
        maxTracesPerUser: 10000
OBS-04 · Section 07
Deploy the OpenTelemetry Collector
The OpenTelemetry Collector receives spans from banking microservices and forwards them to Tempo. It runs as a Deployment (not a DaemonSet) — one Collector per namespace handles all instrumented pods in that namespace.
OpenTelemetryCollector — pipeline: applications → Tempo
apiVersion: opentelemetry.io/v1alpha1
kind: OpenTelemetryCollector
metadata:
  name: ncb-collector
  namespace: banking-production
    # Collector runs in the same namespace as the banking applications.
    # Applications send spans to the Collector's ClusterIP Service.
    # The Collector forwards to Tempo in openshift-tempo-operator namespace.
spec:
  mode: deployment
    # deployment: one Collector Deployment per namespace.
    # Alternative "daemonset" mode runs a Collector on every node —
    # more resilient but much higher resource cost. One deployment per namespace
    # is appropriate for NovaCrestBank's scale.
  config: |
        receivers:
          otlp:
            # OTLP: the OpenTelemetry native protocol receiver.
              # Accepts spans over gRPC (port 4317) and HTTP (port 4318).
              # Banking microservices with the Java agent use gRPC by default.
            protocols:
              grpc:
                endpoint: "0.0.0.0:4317"
              http:
                endpoint: "0.0.0.0:4318"

        processors:
          batch:
            # Batch processor: collect spans into batches before forwarding to Tempo.
              # Reduces the number of network round trips from Collector to Tempo.
              # send_batch_size: send when 512 spans are accumulated.
              # timeout: send every 5 seconds even if batch is not full.
            send_batch_size: 512
            timeout: 5s

          memory_limiter:
            check_interval: 1s
            limit_mib: 400
            # Memory limiter: if the Collector process exceeds 400 MB,
              # it begins dropping spans rather than OOMKilling.
              # Dropping some spans is preferable to the Collector crashing
              # and losing all spans until it restarts.

          attributes:
            # Attribute processor: add and remove span attributes before storage.
            actions:
              - key: "card.number"
                action: delete
              - key: "cvv"
                action: delete
              - key: "account.password"
                action: delete
              # Delete sensitive attributes before they reach Tempo storage.
                # If a developer accidentally adds a span attribute containing a card number,
                # the attribute processor removes it before storage.
                # This is the tracing equivalent of the CLF prune filter for logs (OBS-03).
              - key: "cluster.name"
                value: "ncb-ocp-cluster"
                action: insert
              # Add the cluster name to every span — useful when Tempo is shared
                # across multiple clusters in future (the cluster label identifies the source).

        exporters:
          otlp/tempo:
            endpoint: "tempo-banking-distributor.openshift-tempo-operator.svc:4317"
            # The Tempo Distributor's gRPC OTLP endpoint.
              # Internal cluster Service — spans never leave the cluster network.
              # TLS is handled by the OCP service CA (both endpoints are in-cluster).
            tls:
              insecure: false

        service:
          pipelines:
            traces:
              receivers: [otlp]
              processors: [memory_limiter, batch, attributes]
              exporters: [otlp/tempo]

OBS-04 · Section 09
The OpenTelemetry Java agent — zero-code automatic instrumentation
The most important section in this guide. The OpenTelemetry Java agent automatically instruments every Spring Boot application without a single line of application code change. The agent instruments Spring MVC, RestTemplate, JDBC, Hibernate, and dozens of other libraries by bytecode injection at JVM startup.
How the Java agent works — bytecode injection at startup
The Java agent requires no application code changes. How does it instrument the payment-service without the payment-service knowing anything about OpenTelemetry?
The Java agent uses the JVM's java.lang.instrument API — a standard JVM mechanism that allows a Java agent JAR to transform class bytecode as it is loaded by the JVM. When the payment-service JVM starts with -javaagent:/path/to/opentelemetry-javaagent.jar, the agent registers itself as a bytecode transformer. When the JVM loads Spring MVC's DispatcherServlet class, the agent transforms it — injecting OpenTelemetry instrumentation code that creates spans, propagates context, and exports to the Collector. The payment-service code never changes — it still calls DispatcherServlet normally. The agent intercepts at the bytecode level, below the application code.

What the Java agent instruments automatically: all incoming HTTP requests (creates a root span per request with HTTP method, URL, status code), all outgoing HTTP calls via RestTemplate or WebClient (creates child spans with the target service URL), all JDBC queries (creates database spans with the SQL statement, sanitised — no bind parameter values are captured), all Spring @Scheduled methods (creates spans for background job execution), and all Spring @Async tasks. For the payment-service, this covers the entire call chain automatically — no manual span creation needed for the standard operations. Custom business logic spans (the payment validation logic, the fraud score retrieval) require the manual instrumentation in Section 11.

Adding the Java agent to the payment-service Deployment

payment-service Deployment — Java agent injected via initContainer
spec:
  template:
    spec:
      initContainers:
        - name: copy-otel-agent
          image: harbor.apps.ocp.novacrestbank.local/platform/otel-java-agent:2.x.x
                    # An image containing only the opentelemetry-javaagent.jar.
                      # Built from: FROM scratch COPY opentelemetry-javaagent.jar /agent/
                      # Mirrored to Harbor. Version-pinned — update in Harbor oc-mirror run.
          command: [cp, /agent/opentelemetry-javaagent.jar, /otel/opentelemetry-javaagent.jar]
          volumeMounts:
            - name: otel-agent
              mountPath: /otel
            # InitContainer pattern: copy the agent JAR from the agent image to
                # a shared emptyDir volume (/otel). The main container then mounts the same
                # volume and the agent JAR is available at /otel/opentelemetry-javaagent.jar.
                # This decouples the agent version from the application image —
                # the agent can be updated by changing the initContainer image version
                # without rebuilding the payment-service image.
      containers:
        - name: payment-service
          image: harbor.apps.ocp.novacrestbank.local/banking-apps/payment-service:v1.2.3
          env:
            - name: JAVA_TOOL_OPTIONS
              value: "-javaagent:/otel/opentelemetry-javaagent.jar"
                # JAVA_TOOL_OPTIONS: automatically picked up by any JVM on startup.
                    # The -javaagent flag attaches the OpenTelemetry Java agent.
                    # No application code change. No Dockerfile change.
                    # The agent reads its configuration from environment variables (Section 10).
          volumeMounts:
            - name: otel-agent
              mountPath: /otel
      volumes:
        - name: otel-agent
          emptyDir: {}
Alternative: OpenTelemetry Operator auto-instrumentation injection
The OpenTelemetry Operator can inject the Java agent automatically by adding one annotation to the Deployment: instrumentation.opentelemetry.io/inject-java: "true". The operator creates the initContainer and JAVA_TOOL_OPTIONS injection automatically. This is operationally cleaner than the manual initContainer pattern above — the Instrumentation CR (configured once per namespace) centralises the agent version and configuration, and updating the agent version updates all instrumented Deployments via one CR change. NovaCrestBank uses this pattern for all 50 banking microservices after the initial payment-service proof of concept.
OBS-04 · Section 10
Configuring the agent via environment variables
The OpenTelemetry Java agent reads its configuration entirely from environment variables. These are set in the Deployment or injected via the OTel Operator's Instrumentation CR. Every environment variable is annotated with its purpose.
payment-service Deployment env — OTel agent configuration
env:
  - name: JAVA_TOOL_OPTIONS
    value: "-javaagent:/otel/opentelemetry-javaagent.jar"

  - name: OTEL_SERVICE_NAME
    value: payment-service
      # The service name that appears in Jaeger UI and Tempo for this application.
        # Use a consistent name that matches the metric labels and log labels —
        # this makes cross-signal correlation possible by service name.

  - name: OTEL_EXPORTER_OTLP_ENDPOINT
    value: http://ncb-collector-collector.banking-production.svc:4317
      # The OTel Collector's gRPC OTLP endpoint (port 4317).
        # In-cluster Service — spans stay within the cluster network.
        # Note: http:// not https:// — the Collector uses plaintext for in-cluster
        # communication. TLS is used from Collector to Tempo (the inter-namespace hop).

  - name: OTEL_RESOURCE_ATTRIBUTES
    value: deployment.environment=production,team=banking-platform,namespace=banking-production
      # Resource attributes: added to every span from this service.
        # deployment.environment: environment name (matches the Prometheus/Loki labels).
        # team: team ownership — for multi-team Tempo instances.
        # namespace: the OCP namespace — not automatically added by the Java agent
        # unless the pod is configured to inject it (the Operator does this automatically).

  - name: OTEL_TRACES_SAMPLER
    value: parentbased_traceidratio
      # Sampling strategy: parentbased_traceidratio.
        # parentbased: respect the sampling decision made by the upstream caller.
        # If the api-gateway decides to sample a request (and injects "sampled=1" in
        # the traceparent header), ALL downstream services must also sample it —
        # otherwise the trace is incomplete.
        # If the api-gateway decides NOT to sample, none of the downstream services
        # create spans either — no partial traces in Tempo.
        # traceidratio: the fallback when there is no upstream sampling decision.

  - name: OTEL_TRACES_SAMPLER_ARG
    value: "0.01"
      # 0.01 = 1% head sampling rate at the payment-service level.
        # This is the fallback rate for requests where no upstream sampling decision exists.
        # Sampling strategy is fully explained in Sections 13-14.
        # The Collector applies tail-based sampling on top of this head sampling.

  - name: OTEL_LOGS_EXPORTER
    value: none
      # none: do NOT export logs via OpenTelemetry.
        # Logs are collected separately by Vector (LokiStack/OBS-03).
        # Exporting logs via OTel would create duplicate log storage —
        # once in Loki (via Vector) and once in Tempo's trace storage.
        # Keep log and trace pipelines separate.

  - name: OTEL_METRICS_EXPORTER
    value: none

Adding custom spans for banking business logic
The Java agent instruments all framework operations automatically. Custom spans are needed only for business-domain operations that are not framework calls — the payment validation logic, the fraud score computation, the NPP submission step.
PaymentService.java — custom OpenTelemetry spans for banking operations
import io.opentelemetry.api.GlobalOpenTelemetry;
import io.opentelemetry.api.trace.Span;
import io.opentelemetry.api.trace.StatusCode;
import io.opentelemetry.api.trace.Tracer;
import io.opentelemetry.context.Scope;

@Service
public class PaymentService {

    private final Tracer tracer = GlobalOpenTelemetry.getTracer("com.novacrestbank.payment");
    // GlobalOpenTelemetry.getTracer(): the Java agent registers itself as the global
    // OpenTelemetry implementation. Calling getTracer() returns the agent's tracer —
    // no OpenTelemetry SDK dependency needed. Zero-dependency instrumentation.

    public PaymentResult submitPayment(PaymentRequest request) {

        // The Java agent automatically created a span for the incoming HTTP request
        // (POST /api/v1/payments/submit). Our custom spans are CHILD spans of that.

        // ── Custom span: payment validation ────────────────────────────────────
        Span validationSpan = tracer.spanBuilder("payment.validate")
            .setAttribute("payment.type", request.getPaymentType())
            .setAttribute("payment.amount.cents", request.getAmountCents())
            // SAFE: amount in cents — not a PAN, not a CVV.
            // NEVER add: customer ID, account number, card number, CVV as span attributes.
            .startSpan();

        try (Scope scope = validationSpan.makeCurrent()) {
            ValidationResult validation = validator.validate(request);
            validationSpan.setAttribute("validation.passed", validation.isPassed());
            validationSpan.setAttribute("validation.failures", validation.getFailureCount());
            if (!validation.isPassed()) {
                validationSpan.setStatus(StatusCode.ERROR, "Validation failed");
                validationSpan.addEvent("validation.failed");
                // addEvent: timestamped log-like record within the span.
                // Appears in Jaeger UI as a point in time on the span's timeline.
                // Use for significant events within a long operation.
            }
        } finally {
            validationSpan.end(); // Always end spans in a finally block
        }

        // ── Custom span: NPP rail submission — the operation most likely to be slow ──
        Span nppSpan = tracer.spanBuilder("npp.submit")
            .setAttribute("npp.endpoint", nppConfig.getEndpointUrl())
            .setAttribute("payment.type", "NPP")
            .startSpan();

        try (Scope scope = nppSpan.makeCurrent()) {
            NppResponse response = nppAdapter.submit(request);
            nppSpan.setAttribute("npp.response.code", response.getCode());
            nppSpan.setAttribute("npp.round_trip_ms", response.getRoundTripMs());
            // Attributes added AFTER the operation: actual measured values.
            // npp.round_trip_ms: the time the NPP rail took to respond.
            // In Jaeger UI: this span's duration IS the NPP round trip time.
            // The attribute provides additional context (the raw API round trip
            // vs the total span duration which includes parsing overhead).
        } catch (NppException e) {
            nppSpan.setStatus(StatusCode.ERROR, e.getMessage());
            nppSpan.recordException(e);
            // recordException: captures the exception stack trace as a span event.
            // Visible in Jaeger UI as an error event on the npp.submit span.
            throw e;
        } finally {
            nppSpan.end();
        }
    }
}
OBS-04 · Section 12
Context propagation — passing paymentId across service boundaries
The trace ID links spans from different services into one trace automatically. But business identifiers like the paymentId need explicit propagation to appear as span attributes in every service. OpenTelemetry Baggage provides this.
Why paymentId must be explicitly propagated — the trace ID is not enough
The trace ID already links all spans from a payment transaction. Why also explicitly propagate the paymentId through every service?
The trace ID is a random UUID — it has no meaning to a business analyst or a fraud investigator. When a customer calls to complain about payment NCB-20260420-87654321, the operations team searches for that payment ID — not for a random trace ID. The trace ID links the technical spans; the paymentId links the business investigation to the technical trace.

Without paymentId propagation: an engineer must look up the paymentId in application logs (OBS-03) to find the timestamp, then search Tempo for traces in that time window from the payment-service, then identify which trace corresponds to that payment. Multiple steps, multiple tools.

With paymentId propagation via Baggage: Tempo indexes the paymentId as a span attribute on every span in the trace. Search Jaeger UI for payment.id=NCB-20260420-87654321 — directly finds the trace. One step, one tool.
Setting and reading Baggage — paymentId across service boundaries
import io.opentelemetry.api.baggage.Baggage;

// In the api-gateway — set the paymentId in Baggage when the request is received
Baggage.current().toBuilder()
    .put("payment.id", paymentRequest.getPaymentId())
    .put("payment.type", paymentRequest.getPaymentType())
    .build()
    .storeInContext(Context.current())
    .makeCurrent();
// Baggage is automatically propagated in the baggage HTTP header to downstream services.
// The OpenTelemetry Java agent reads the baggage header in each downstream service
// and makes the values available via Baggage.current().

// In the payment-service — read the paymentId from Baggage and add to spans
String paymentId = Baggage.current().getEntryValue("payment.id");
Span.current().setAttribute("payment.id", paymentId);
// Setting the paymentId as an attribute on Span.current() means:
// every span in payment-service that is active when this runs carries the paymentId.
// In Jaeger, searching for payment.id=NCB-20260420-87654321 finds all spans
// across all services that processed this payment.

// In the fraud-detection-service — same pattern
// The baggage header arrives automatically — just read and attach
Span.current().setAttribute("payment.id", Baggage.current().getEntryValue("payment.id"));
// One line per service — all 8 services in the payment chain carry the paymentId
// as a span attribute. All 8 appear in the Jaeger UI search for that payment ID.
OBS-04 · Section 13
Why sampling — the cost of tracing every request at banking API volumes
At NovaCrestBank's payment API volume, tracing 100% of requests is technically possible but operationally expensive. Sampling reduces the volume while preserving the traces that matter most for investigation.
The cost calculation — why 100% sampling is impractical at banking scale
If tracing provides complete visibility, why not trace every request? Disk storage is inexpensive.
The cost of 100% sampling is not primarily storage — it is CPU overhead in the instrumented services and network overhead between services and the Collector.

NovaCrestBank's payment API: 100 TPS (transactions per second) × 8 services × 5 spans per service = 4,000 spans per second. Each span requires: the instrumented service to create a span object, populate attributes, and serialise to OTLP format (~ 0.1ms CPU per span). At 4,000 spans/second: 400ms of CPU per second dedicated to tracing — a meaningful fraction of the CPU budget on high-TPS services. For the payment-service running 5 TPS per pod with a 2-second P99 deadline, spending 2ms per request on tracing (5% of the latency budget) is measurable.

The storage consequence at 100% sampling: 4,000 spans/second × 5 KB per span = 20 MB/second of trace data = 1.7 TB per day. MinIO handles this easily. Tempo handles this reasonably. But the Collector becomes a bottleneck and a failure point — if the Collector is overloaded or unavailable, the banking services experience backpressure from the trace export. At 1% sampling: 40 spans/second, 200 KB/second, 17 GB per day — entirely manageable with no impact on the banking services.

The key insight for banking: most requests are identical. A payment that completes normally in 80ms does not produce any new information — you already have thousands of similar traces. What matters is tracing all slow requests (P99 investigation), all errors, and all requests involving specific payment IDs when investigating customer complaints. This is exactly what tail-based sampling provides.
OBS-04 · Section 14
Tail-based sampling — keeping the traces that matter
NovaCrestBank uses a two-layer sampling strategy: head sampling (1% at the service level, configured in Section 10) plus tail-based sampling in the OTel Collector (keep 100% of slow and error traces). This gives comprehensive coverage of problematic requests with minimal overhead for normal requests.
Head sampling vs tail sampling — the fundamental difference
What is the difference between head sampling and tail sampling, and why does NovaCrestBank use both?
Head sampling makes the sampling decision at the start of a request — before anything is known about how the request will proceed. A 1% head sampling rate means 99% of requests are discarded immediately, before any spans are created or exported. Head sampling is CPU-efficient (the JVM never creates spans for unsampled requests) but it samples blindly — it discards 99% including all slow requests and all error requests.

Tail sampling makes the sampling decision at the end of a request — after all spans for the trace have been collected. The Collector sees the complete trace and can decide: "this trace has an error span — keep it. This trace has a duration > 500ms — keep it. This trace is a healthy 80ms request — discard it." Tail sampling keeps exactly the traces that matter for investigation.

The problem with tail sampling alone: the Collector must buffer the complete trace in memory before making the sampling decision. At 100 TPS with 4,000 spans/second, buffering all spans for all in-flight requests requires significant Collector memory. The 1% head sampling at the service level reduces the Collector's buffer to 40 spans/second — manageable with 400 MB of Collector memory.

NovaCrestBank's two-layer strategy: Head sampling at 1% (OTel agent setting) reduces the volume to 1% before it reaches the Collector. Tail-based sampling in the Collector then upgrades sampled traces: keep 100% of traces with errors (regardless of head sampling), keep 100% of traces with duration > 500ms, keep 100% of traces with the priority baggage attribute set (a developer can force-sample a specific payment ID for debugging).

Tail-based sampling configuration in the OTel Collector

processors:

      # Wait up to 10 seconds for all spans in a trace to arrive before deciding.

        # 10 seconds captures most spans while not buffering for too long.
    num_traces: 5000
      # Buffer up to 5,000 in-flight traces waiting for the sampling decision.
    policies:
      - name: always-sample-errors

              # A payment failure is always worth investigating.
              # Error traces are the highest-value traces for incident investigation.

      - name: always-sample-slow

              # Slow traces are exactly the ones needed to diagnose latency violations.
              # At 1% head sampling, some slow traces would be discarded without this policy.
              # This policy retrieves them from the buffer and keeps them.

      - name: always-sample-priority
        type: string_attribute
        string_attribute:
          key: sampling.priority
          values: [force]
            # Keep ALL traces with sampling.priority=force baggage attribute.
                # A developer investigating a specific customer complaint can set:
                # Baggage.put("sampling.priority", "force") in the api-gateway

                # This forces 100% sampling for that specific payment — every call to
                # every service is traced, regardless of head sampling.

      - name: probabilistic-1pct
        type: probabilistic
        probabilistic: {sampling_percentage: 100}
          # 100% of what arrives at the Collector (which is already 1% head-sampled).
              # This is the baseline — the 1% head sampling happens at the service level.
              # The error, latency, and priority policies override this for important traces.

NovaCrestBank sampling strategy — what gets traced

100%
All error traces
Any trace with an ERROR span — payment failures, exceptions, timeout errors. Always kept. The most valuable traces for incident investigation.
100%
All slow traces (≥ 500ms)
Any trace where the total duration exceeds the latency SLO threshold. Essential for diagnosing P99 violations and customer latency complaints.
100%
Force-sampled traces
Traces where a developer sets sampling.priority=force — for investigating a specific customer complaint or testing a new deployment.
1%
Normal healthy traces
Fast, successful requests. 1% is sufficient for baseline behaviour understanding and RED metric generation. The other 99% are discarded — they contain no new information.

Reading a trace — the payment transaction waterfall explained
The Jaeger UI waterfall diagram contains more information than the bar widths suggest. Understanding every element of a trace view makes investigations faster and more complete.
Element in Jaeger UIWhat it meansWhat to look for
Span duration bar widthThe time this span took — from startTimeUnixNano to endTimeUnixNano. The root span's width is the total request duration.Very wide spans that are children of the total — these are the bottlenecks. A bar that is 90% of the root span's width is doing 90% of the work.
Gap between parent and child span startThe time between when the parent span started and when the child span started — serialisation/network overhead for calling the child service, or business logic running in the parent before calling the child.Large gaps indicate processing time in the parent before making the downstream call. A 100ms gap before a database call means 100ms of application code running before the query.
Span attributes (tags in Jaeger)Key-value metadata attached to the span — HTTP method, URL, status code, payment ID, NPP response code, etc.error=true attributes indicate the span errored. http.status_code=503 on a child span means the downstream service returned Service Unavailable.
Span events (logs in Jaeger)Timestamped records within a span — created by span.addEvent() or span.recordException(). Appear as dots on the span timeline.A dot near the start of a long span followed by a long empty bar indicates the work was triggered early but completion was delayed (waiting for a lock, a queue, or an async operation).
Red/error indicator on a spanThe span's status code is ERROR. Set by span.setStatus(StatusCode.ERROR) in the application code or automatically by the Java agent for HTTP 5xx responses.Find the first red span in the trace — that is the origin of the error. Parent spans inherit ERROR status if any child errors, so the root span may also be red even though the error originated in a child.
Spans with the same service name but different operationsMultiple spans from the same service — indicating multiple operations within that service (validation, database query, external call).Which operation within the service is slowest? If payment-service has 5 child spans and one is 10x wider than the others, that operation is the bottleneck within the service.
