OBS-03 · Section 01
The three log types — infrastructure, audit, and application
OpenShift generates three fundamentally different categories of log. Each has a different source, different content, different audience, different retention requirement, and different destination. Treating them identically — sending everything to the same place with the same retention — is the most common logging architecture mistake.

Infrastructure logs

Source: All pods in openshift-*, kube-* namespaces — etcd, API server, controller-manager, OCP router, monitoring stack, cert-manager, ACS.
Content: Operational logs from cluster components — etcd leader elections, API server request processing, router access logs, operator reconciliation cycles.
Audience: Platform engineering team — diagnosing cluster-level incidents.
Retention: 7 days in LokiStack. Not sent to SIEM (too noisy, not a compliance requirement).

Audit logs

Source: The Kubernetes API server audit mechanism — every API call to the OCP cluster is recorded: who, what, when, from where, what happened.
Content: Structured JSON records of every kubectl/oc command, every application API call, every Argo CD sync, every pipeline run, every admin action.
Audience: Security Operations Centre, compliance team, auditors, regulators.
Retention: 12 months minimum (PCI-DSS Req 10). Forwarded to Splunk SIEM — the primary compliance record.

Application logs

Source: All pods in customer namespaces — banking-production, banking-staging, banking-dev. Everything the payment-service, account-service, and other microservices write to stdout/stderr.
Content: Application business logic logs — transaction processing decisions, error stack traces, business rule evaluations, user-visible events.
Audience: Development teams, banking operations.
Retention: 30 days in LokiStack. Certain business events forwarded to Splunk for correlation.

Why audit logs must go to Splunk separately — not just into LokiStack
LokiStack can store all three log types. Why separately forward audit logs to Splunk rather than just querying LokiStack for compliance evidence?
LokiStack provides operational log search for the platform and development teams. Splunk provides the compliance audit record for the security operations centre and regulators. They serve different purposes and have different requirements that cannot be collapsed into a single system.

Immutability: PCI-DSS Req 10 requires that audit logs cannot be modified or deleted — they must be tamper-evident. Splunk Enterprise Security is configured with an append-only index that operators cannot modify, and the audit trail of the Splunk audit log itself is protected. LokiStack, while configurable with retention policies, does not provide the same tamper-evident guarantees that a purpose-built compliance platform does.

Retention duration: PCI-DSS requires 12 months of audit logs with 3 months immediately searchable. Splunk's tiered storage (fast disk + cold storage) handles this retention model efficiently. Storing 12 months of Kubernetes audit logs in LokiStack on vSAN would require significantly more expensive storage than Splunk's compressed cold tier.

SOC tooling: The security operations team works in Splunk — they have their ACS alerts, their NSX-05 firewall logs, and their banking application security events all in one SIEM. Adding Kubernetes audit logs to Splunk gives the SOC analyst a single tool for cross-source correlation: "who ran kubectl delete pod before this ACS alert fired?" An analyst who has to query LokiStack for audit logs and Splunk for everything else cannot perform efficient incident correlation.

FFIEC examination readiness: When bank examiners request audit log evidence, the compliance team produces a Splunk report — a document auditors recognise and trust. Exporting from LokiStack requires custom tooling that examiners may not accept as readily as a report from an established enterprise SIEM platform.
OBS-03 · Section 02
Why Loki — log aggregation without index explosion
Loki is the log aggregation backend deployed by the OpenShift Logging Operator. Understanding what makes Loki architecturally different from Elasticsearch (the traditional choice) explains why it is the right fit for OCP's log volume.
Loki vs Elasticsearch — the indexing philosophy difference
The previous logging architecture used Elasticsearch (the EFK stack — Elasticsearch, Fluentd, Kibana). OpenShift Logging 5.5+ replaced it with LokiStack. Why?
Elasticsearch indexes every field of every log line. This makes full-text search across any field extremely fast — but it makes storage enormous and index maintenance expensive. As log volume grows, Elasticsearch's index size grows proportionally and its resource consumption grows faster than linearly. A large OCP cluster generating 10 GB of logs per day can require 500 GB of Elasticsearch storage (with replicas and indices) within a month.

Loki takes the opposite approach: it indexes only the labels (a small, controlled set of log stream identifiers — namespace, pod name, container name) and stores log lines as compressed raw text. The label index is tiny and constant in size regardless of log volume. Queries against log content use regular expression matching against the compressed log lines — slower for arbitrary full-text search, but sufficient for the structured log queries that platform engineers and security analysts actually need.

For NovaCrestBank's OCP cluster: LokiStack stores logs in MinIO (on-premises S3-compatible object storage on the vSAN cluster) — inexpensive block storage rather than expensive SSD-backed Elasticsearch nodes. The LokiStack Querier and Ingester pods run on infra nodes and their compute footprint is a fraction of what an equivalent Elasticsearch cluster would require. The trade-off is that searching for an arbitrary string across all log lines is slower — but that is exactly what Splunk is for.

LokiStack components running in NovaCrestBank's cluster

OBS-03 · Section 03
ClusterLogForwarder — the routing engine for all log streams
The ClusterLogForwarder (CLF) is the single Kubernetes object that controls where every log stream in the cluster goes. It replaces Fluentd with Vector (the modern, high-performance log collector) and provides a declarative routing model.
inputs
spec.inputs[]
Define which log sources to collect. Three built-in input types: infrastructure (system namespaces), audit (API server audit log), application (all other namespaces). Each input can be further refined with namespace or label selectors — for example, an application input that collects only from banking-production.
filters
spec.filters[]
Transform or drop log records before forwarding. Use for: masking sensitive fields (PAN data in application logs — replace credit card numbers with XXXX before they reach storage), parsing structured JSON logs, dropping health check noise (removing thousands of liveness probe log lines that add volume without value), and adding labels for downstream correlation.
outputs
spec.outputs[]
Define where to send log streams. Multiple outputs can receive the same log stream simultaneously. Output types include: lokiStack (the in-cluster LokiStack), splunk (Splunk HEC endpoint), syslog, http, elasticsearch. Each output has its own authentication and TLS configuration.
pipelines
spec.pipelines[]
Connect inputs to outputs, optionally through filters. A pipeline says: "take log stream X, apply filter Y, send to output Z." The same input can flow to multiple pipelines with different filters and different outputs — NovaCrestBank uses this to send audit logs to both LokiStack (for short-term operational search) and Splunk (for long-term compliance retention) simultaneously.

Prerequisites — object storage, cert-manager, and infra nodes
LokiStack requires S3-compatible object storage for log chunk storage. NovaCrestBank deploys MinIO on the vSAN cluster as an on-premises S3 endpoint. cert-manager (OCP-OPS-02) must be installed — LokiStack uses it for its own TLS certificates.
MinIO must be running and the bucket created before LokiStack deployment
LokiStack connects to MinIO at startup and validates the bucket exists and is accessible. If the bucket does not exist or MinIO is unreachable, LokiStack's Ingester pods enter CrashLoopBackOff immediately. Deploy MinIO first, create the ncb-ocp-logs bucket, then deploy LokiStack.

kind: Subscription
metadata:
  name: cluster-logging
  namespace: openshift-logging
spec:
  channel: stable
  name: cluster-logging
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual
    # Manual: logging updates can change Vector configuration or forwarding behaviour.
    # Every update is reviewed. Automatic updates could silently break SIEM forwarding.

  name: loki-operator
  namespace: openshift-operators-redhat

OBS-03 · Section 06
Deploy LokiStack — the storage and query layer
The LokiStack Custom Resource tells the Loki Operator what size cluster to deploy, which object storage to use, and how long to retain logs. Every field is annotated.
LokiStack — every field explained
apiVersion: loki.grafana.com/v1
kind: LokiStack
metadata:
  name: logging-loki
  namespace: openshift-logging
    # The name "logging-loki" is referenced by the ClusterLogForwarder.
    # Keep this name — it is the convention used by the Logging Operator integration.
spec:
  size: 1x.small
    # LokiStack size determines the number of replicas and resource requests
    # for each component. Options: 1x.extra-small, 1x.small, 1x.medium, 1x.large.
    # 1x.small: suitable for clusters generating up to 50 GB of logs per day.
    # NovaCrestBank's 9-node cluster with 50 banking microservices generates
    # approximately 10–20 GB per day — 1x.small is appropriate.
    # Scale to 1x.medium if daily log volume consistently exceeds 30 GB.

  storage:
    schemas:
      - version: v13
        effectiveDate: "2024-01-01"
          # Schema version v13 is the current LokiStack schema.
              # The effectiveDate determines when this schema takes effect.
              # Older schemas remain readable — schema changes are additive.
    secret:
      name: lokistack-minio-creds
      type: s3
        # References the Secret created in Section 04 containing MinIO credentials.
          # type: s3 — MinIO implements the S3 API, so LokiStack uses the s3 type.
          # For AWS S3: type remains s3 — just change the endpoint in the Secret.

  storageClassName: vsan-standard
    # StorageClass for LokiStack's own PVCs (Ingester WAL, Index Gateway store).
    # vsan-standard: Delete policy, RAID-5. LokiStack's in-cluster PVCs are small
    # (Ingester WAL: 10 Gi) and their data is backed by object storage — they are
    # not the primary data store. vsan-standard is appropriate.

  tenants:
    mode: openshift-logging
      # openshift-logging mode: LokiStack integrates with OCP's RBAC for multi-tenancy.
        # Each OCP namespace becomes a separate Loki tenant.
        # Application teams can query their own namespace logs from Grafana
        # without seeing logs from other namespaces.
        # Platform engineers can query all tenants.
        # This is enforced by the Loki Gateway using OCP service account tokens.

  rules:
    enabled: true
        # Enable the Loki Ruler component — allows writing alert rules against log streams.
          # Used in Section 17 for log-based alerting (alerting on log content, not metrics).
    namespaceSelector:
      matchLabels:
        openshift.io/cluster-monitoring: "true"

  limits:
    global:
      retention:
        days: 30
          # Global retention: delete log chunks from object storage after 30 days.
              # Per-stream retention can be configured to override this:
              # audit logs retained for 90 days (approaching PCI-DSS Req 10's 12-month
              # requirement — the remainder lives in Splunk which retains 12 months).
              # Infrastructure logs: 7 days (operational, not compliance).
              # Application logs: 30 days (debugging window).
      ingestion:
        perStreamRateLimitBurst: 50MB
          # Maximum log burst rate per stream (per namespace per pod) before Loki
              # back-pressures Vector. 50 MB burst handles application log storms
              # (a payment service exception cascade) without dropping logs.

  template:
    distributor:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations:
        - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
    ingester:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations:
        - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
    querier:
      nodeSelector: {node-role.kubernetes.io/infra: ""}
      tolerations:
        - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
        # All LokiStack components run on infra nodes — same isolation pattern
          # as Prometheus (OBS-01) and GitOps (OPS-05). Log ingestion must not
          # compete with banking application workloads on worker nodes.

OBS-03 · Section 08
ClusterLogForwarder — infrastructure and application logs to LokiStack
The first ClusterLogForwarder pipeline routes infrastructure and application logs to LokiStack. Audit logs are configured separately in Section 09 because they have different destinations and filters.
ClusterLogForwarder — infrastructure and application logs
apiVersion: logging.openshift.io/v1
kind: ClusterLogForwarder
metadata:
  name: instance
    # "instance" is the required name — only one ClusterLogForwarder per cluster.
  namespace: openshift-logging
spec:
  outputs:
    - name: lokistack-logs
      type: lokiStack
      lokiStack:
        name: logging-loki
          # References the LokiStack deployed in Section 06
        namespace: openshift-logging
        target:
          name: logging-loki
          namespace: openshift-logging

  filters:
    - name: drop-healthchecks
      type: drop
      drop:
        - test:
          - field: .message
            matches: 'GET /actuator/health'
            # Drop health check probe logs from application pods.
                # OCP probes every pod's /actuator/health every 10 seconds.
                # With 50 pods × 6 probes/minute = 300 log lines/minute of pure noise.
                # These are not errors, not business events — pure volume without value.
        - test:
          - field: .message
            matches: 'GET /metrics'
            # Also drop Prometheus scrape access logs — another high-volume noise source.

    - name: mask-pan-data
      type: prune
      prune:
        in: [.cardNumber, .cvv, .pan]
          # Remove these fields entirely from log records before storage.
              # If a banking application accidentally logs a card number or CVV,
              # the prune filter removes them before the log reaches LokiStack or Splunk.
              # PCI-DSS Req 3.2: CVV must never be stored after authorisation.
              # Pruning at the forwarder is the defence-in-depth safety net.

  pipelines:
    - name: infrastructure-to-loki
      inputRefs: [infrastructure]
      filterRefs: [drop-healthchecks]
      outputRefs: [lokistack-logs]
        # Infrastructure logs → LokiStack only (7-day retention via LokiStack stream rules)

    - name: application-to-loki
      inputRefs: [application]
      filterRefs: [drop-healthchecks, mask-pan-data]
      outputRefs: [lokistack-logs]
        # Application logs → LokiStack (30-day retention)
          # Both filters applied: drop health check noise AND mask any PAN data
          # PAN masking is applied before logs ever reach storage — defence in depth
OBS-03 · Section 09
ClusterLogForwarder — audit logs to Splunk SIEM
Audit logs flow to two destinations simultaneously: LokiStack for operational search (querying recent audit events from Grafana or the OCP console) and Splunk for long-term compliance retention. The CLF pipeline is the mechanism that fans the same stream to two outputs.
Why audit logs go to both LokiStack and Splunk — different purposes, not redundancy
Sending audit logs to both destinations serves two different use cases, not the same one twice. LokiStack receives audit logs so platform engineers can query recent audit events (last 7 days) directly from Grafana without switching tools — "who deleted that Secret in banking-production 2 hours ago?" is answered from the same Grafana instance used for metrics. Splunk receives audit logs for long-term retention (12 months), SOC analyst correlation with other security events, and compliance report generation for PCI-DSS assessors. The operational convenience of LokiStack does not satisfy the compliance requirement — only Splunk's tamper-evident append-only index does.
ClusterLogForwarder — audit log pipelines added to the instance
  # Added to the ClusterLogForwarder spec.outputs[] section:
  outputs:
    - name: lokistack-logs        # (defined in Section 08)
      ...

    - name: splunk-siem
      type: splunk
      splunk:
        url: https://splunk-hec.novacrestbank.local:8088
          # The Splunk HTTP Event Collector endpoint.
              # Same endpoint as ACS uses (OCP-OPS-03 Section 18).
              # Different HEC token — audit logs go to a separate Splunk index.
        authentication:
          token:
            from: secret
            secret:
              name: splunk-audit-hec-token
              key: hecToken
                # Separate HEC token for audit logs.
                    # Splunk index: ncb_ocp_audit_logs
                    # Separate from ncb_acs_security (OCS-OPS-03).
                    # Separate index means: separate retention policy,
                    # separate search scope, separate access controls.
        indexName: ncb_ocp_audit_logs
          # The Splunk index that receives OCP audit events.
              # This index is configured in Splunk with:
              # - Retention: 12 months on fast disk, then archive to cold storage
              # - Access: Security Operations team only (not general analysts)
              # - Integrity checking: Splunk data integrity control enabled
      tls:
        certificate:
          from: configmap
          configmap:
            name: ncb-ca-bundle
            key: ca-bundle.crt
              # The NovaCrestBank CA bundle for verifying Splunk's TLS certificate.
                  # Never use insecureSkipVerify: true — audit log forwarding over
                  # unverified TLS could be silently redirected to an attacker's server.

  # Added to the ClusterLogForwarder spec.filters[] section:
  filters:
    - name: audit-enrich
      type: kubernetesMetadata
        # Enrich audit log records with Kubernetes metadata before sending to Splunk.
          # Adds: cluster name, namespace labels, pod annotations.
          # Makes Splunk records richer for correlation — the analyst sees the full
          # context of what object was affected, not just its name.

  # Added to the ClusterLogForwarder spec.pipelines[] section:
  pipelines:
    - name: audit-to-loki-and-splunk
      inputRefs: [audit]
      filterRefs: [audit-enrich]
      outputRefs: [lokistack-logs, splunk-siem]
        # Audit logs → BOTH LokiStack and Splunk simultaneously.
          # Vector fans the stream to two outputs in parallel.
          # No log record is sent to one destination and not the other.
          # If Splunk HEC is temporarily unreachable, Vector buffers locally
          # and retries — ensuring no audit events are lost during SIEM maintenance.
OBS-03 · Section 10
Audit log anatomy — what every field means
Every API call to the OCP cluster generates a structured JSON audit log record. Understanding each field makes querying and incident investigation effective. This is the primary data source for both operational security and PCI-DSS compliance evidence.
Example audit log record — oc delete pod command
{
  "kind": "Event",
  "apiVersion": "audit.k8s.io/v1",
  "level": "RequestResponse",
    ↑ level: what is logged. "Request" = headers only. "RequestResponse" = headers + body.
     Audit policy controls this — sensitive namespaces log full request bodies.
  "auditID": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    ↑ Unique ID for this event — correlates with request logs in other systems
  "stage": "ResponseComplete",
    ↑ stage: when in the request lifecycle this was recorded.
     "RequestReceived" = logged when API server receives the request (before processing).
     "ResponseComplete" = logged after response sent — includes the outcome (allowed/denied).
  "requestURI": "/api/v1/namespaces/banking-production/pods/payment-service-abc123",
  "verb": "delete",
    ↑ verb: what Kubernetes operation. get, list, watch, create, update, patch, delete.
     "delete" of a production pod is a high-value event for SOC correlation.
  "user": {
    "username": "alice.smith",
    "uid": "alice.smith@novacrestbank.local",
    "groups": ["NovaCrestBank-Banking-Developers", "system:authenticated"]
  },
    ↑ Who performed the action — their AD username and group memberships.
     This is the "who" in the PCI-DSS Req 10 "who, what, when, where, what happened" model.
  "sourceIPs": ["10.70.0.55"],
    ↑ Where the request came from — the jump host IP in this case.
     A sourceIP of a pod IP (10.50.x.x) means a pod is calling the API — possible compromise.
  "objectRef": {
    "resource": "pods",
    "namespace": "banking-production",

LogQL — the Loki query language from first principles
LogQL is how engineers query logs stored in LokiStack. Its syntax is deliberately similar to PromQL but serves a different purpose — finding log lines rather than querying time-series metrics.
LogQL's two expressions — log queries and metric queries
PromQL has one kind of expression — metric queries. LogQL has two. What is the difference?
A LogQL log query finds and returns log lines — the actual text of matching log entries. A LogQL metric query counts, rates, or aggregates those log lines and returns a time series — like a Prometheus metric but computed from log content. Both start with a log stream selector.

Log stream selector: {namespace="banking-production", pod=~"payment-service-.*"} — selects all log streams from payment-service pods using LokiStack's label index. This is the most efficient part of the query — LokiStack uses the label index to find which object storage chunks to search.

Log pipeline (optional filter): appended with |= "ERROR" (contains string), != "DEBUG" (does not contain), |~ "payment_id=\d+" (regex match). These filter the log lines within the selected streams — slower than the stream selector because they scan compressed text.

Metric extraction (optional): | rate(), | count_over_time(), | json (parse JSON and extract fields). These convert log lines into metric values — for example, counting error log lines per minute to create an Alertmanager-compatible alert.
# ── LOG QUERIES (return log lines) ────────────────────────────────────────────

# Select all logs from payment-service in production — no filter
{namespace="banking-production", app="payment-service"}

# Find all ERROR lines from payment-service in the last hour
{namespace="banking-production", app="payment-service"} |= "ERROR"

# Find payment failures — regex matching for Java exception patterns
{namespace="banking-production"} |~ "PaymentException|InsufficientFundsException"

# Parse structured JSON logs and filter by a field value
{namespace="banking-production", app="payment-service"}
  | json                       # parse the log line as JSON, extract fields
  | level = "ERROR"            # filter where the "level" JSON field equals "ERROR"
  | paymentType = "NPP"       # and the paymentType field equals "NPP"
  | line_format "{{.timestamp}} [{{.level}}] {{.message}} payment={{.paymentId}}"
                               # reshape the output — show only relevant fields

# ── METRIC QUERIES (return time series) ───────────────────────────────────────

# Error log rate per minute — useful for Grafana time series panel
rate(
  {namespace="banking-production"} |= "ERROR" [1m]
)
# Returns: log lines per second containing "ERROR" — same shape as a Prometheus metric.
# Use in Grafana Loki panels — shows error log frequency as a time series chart.

# Count payment exceptions by exception type — for investigation dashboards
sum by (exceptionType) (
  count_over_time(
    {namespace="banking-production"}
    | json
    | level = "ERROR"
    [5m]
  )
)
# Returns: count of error logs in the last 5 minutes grouped by exceptionType JSON field.
# Requires structured JSON logging from the application (not just plain text).

OBS-03 · Section 15
NSX-05 DFW event correlation with cluster audit logs
The most powerful security investigations combine two data sources: NSX-05 DFW logs (who is trying to make network connections) and OCP audit logs (who is taking cluster API actions). Together they answer questions neither can answer alone.
The correlation scenario — a compromised pod making lateral movement
How does correlating NSX DFW logs with OCP audit logs reveal a lateral movement attack that neither data source alone would surface?
Consider this attack sequence: a payment-service pod is compromised via a deserialization vulnerability. The attacker running inside the pod tries two things: (1) connect to another banking service's database directly (lateral movement via network), and (2) use the pod's ServiceAccount token to call the Kubernetes API and read Secrets.

NSX-05 DFW log alone: shows a denied network connection from a payment-service pod IP (10.50.0.x) to a database IP (10.50.0.y:5432) — unusual because payment-service connects to its own DB, not account-service's DB. The DFW blocks and logs the connection. But the NSX log does not know this is a compromised pod — it just sees an IP address.

OCP audit log alone: shows a Kubernetes API call to list Secrets in banking-production using the payment-service ServiceAccount token, from source IP 10.50.0.x (the pod's IP). Unusual — normal application code does not list Secrets via the API; it uses environment variables or mounted files. But the audit log does not know the pod is compromised — it just sees the ServiceAccount name.

Correlated in Splunk: the SOC analyst's correlation rule fires when the same source IP (10.50.0.x) appears in both an NSX DFW deny event (lateral movement attempt) and a Kubernetes audit event (Secret access attempt) within the same 5-minute window. This combination is definitive evidence of active exploitation — one anomaly might be a misconfiguration; both anomalies from the same IP in the same 5-minute window is an active incident. The analyst now knows exactly which pod (10.50.0.x → pod name from OCP), exactly what was attempted, and exactly when. Response: isolate the pod, preserve the forensic evidence, begin incident response.

Structured logging is a prerequisite for effective LogQL queries
Application logs that are plain text require regex matching — slow, brittle, and unable to extract field values. NovaCrestBank's banking microservices use Logback with the Logstash encoder to emit structured JSON — every log line is a JSON object with fields like level, logger, message, paymentId, paymentType, duration_ms. This enables the queries below.

{namespace="banking-production", app="payment-service"}

OBS-03 · Section 17
Log-based alerts — Loki ruler and PrometheusRule
Prometheus alerts on metrics (numbers). Loki ruler alerts on log content — specific log patterns that indicate security or operational issues that do not surface as metrics. Both are important; neither replaces the other.
When to use a log-based alert vs a metric-based alert
OBS-02 already has Prometheus alerts for error rate. Why also have Loki ruler alerts on log content?
Prometheus metrics measure quantitative behaviour — how many requests, how fast, how often. Log-based alerts fire on specific textual patterns that either have no metric equivalent or where the log content provides context that a metric cannot. Three specific categories where log alerts add unique value:

1. Security-specific patterns. "A process named 'bash' executed inside the payment-service container" is not a metric — it is an event that either happened or didn't. ACS detects this at the OS level (OPS-03), but if a banking application itself logs a suspicious execution (for example, "executing shell command: bash -c..."), the Loki ruler catches it from the application log stream.

2. Application logic errors that never surface as HTTP errors. A payment that fails internally but returns HTTP 200 (a "silent failure" — the payment is declined but the API returns success) produces an ERROR log line but no 5xx response. The Prometheus error rate metric stays at zero. Only the log-based alert fires.

3. Configuration and security events. "TLS handshake failure" in an application log means something is wrong with the mTLS setup — a certificate may have expired. This produces a specific log line before any metrics change.
Loki ruler alert — suspicious patterns in banking-production application logs
apiVersion: loki.grafana.com/v1
kind: AlertingRule
metadata:
  name: ncb-banking-log-alerts
  namespace: banking-production
spec:
  tenantID: application
    # tenantID: which Loki tenant to query.
    # "application" = the application log tenant (banking-production logs)
  groups:
    - name: ncb.payment-service.log-alerts
      interval: 1m
        # Evaluate this rule group every 1 minute.
      rules:

        - alert: NCBPaymentSilentFailure
          expr: |
                sum(count_over_time(
                  {namespace="banking-production", app="payment-service"}
                  | json
                  | level = "ERROR"
                  | message =~ "Payment processed with status SUCCESS but internal error.*"
                  [5m]
                )) > 0
            # Alert when any log line matches the "silent failure" pattern.
                # count_over_time > 0 means at least one occurrence in the last 5 minutes.
                # This specific message pattern indicates the application returned HTTP 200
                # but internally logged an error — a payment status inconsistency.
          for: 0m
            # for: 0m = fire immediately on first match. No sustained period needed —
                # even one silent failure is immediately alertworthy.
          labels: {severity: critical, team: banking-platform}
          annotations:
            summary: Payment silent failure detected — payment may be in inconsistent state
            description: "A payment returned HTTP 200 (success) but the application logged an internal error. The payment may be in an inconsistent state. Immediate investigation required."

        - alert: NCBTLSHandshakeFailure
          expr: |
                rate(
                  {namespace="banking-production"}
              | json
              | message =~ ".*TLS handshake.*failure|.*SSL.*error|.*certificate.*expired.*"
              [5m]
            ) > 0.1
            # More than 0.1 TLS errors per second — possible certificate expiry cascade.
                # One TLS error is normal noise. 0.1/s sustained means many connections failing.
          for: 2m
          labels: {severity: warning, team: banking-platform}
          annotations:
            summary: TLS handshake failures detected in banking-production
            description: "Elevated TLS handshake failure rate. Check certificate expiry: oc get certificate -n banking-production"
