GITOPS-05
The core principle — why Git is a better source of truth than the cluster itself
NovaCrestBank already deploys to OCP using kubectl apply and Helm from the pipeline. What does GitOps add that those approaches do not provide?
kubectl apply and Helm deployments are push-based: someone runs a command and the cluster changes. The command might come from a pipeline, from a developer's laptop, from a runbook during an incident. After the command runs, there is no record of the intended state — only the current state in the cluster. If the cluster is rebuilt, everything must be redeployed manually. If a developer makes an ad-hoc change to fix a production issue, that change lives only in the cluster — not in any config file. The cluster drifts from the documented intent.

GitOps is pull-based: the cluster's desired state is declared in a Git repository. Argo CD continuously compares the cluster's actual state to the Git repository and reconciles any difference. Nobody runs kubectl apply in production — they commit to Git. The Git repository is the source of truth, not the cluster.

Four properties GitOps provides that push-based deployment cannot:

1. Declarative desired state. The entire configuration of every banking application in every environment is captured in YAML files in a Git repository. To know what should be running in production right now, read the Git repository — not the cluster.

2. Versioned and immutable history. Every change to every application's configuration is a Git commit — with an author, timestamp, commit message, and a diff showing exactly what changed. The complete history of every deployment is in Git. PCI-DSS Req 6.4 requires change management audit trails — Git IS the audit trail.

3. Continuous reconciliation. Argo CD checks the cluster against Git every 3 minutes by default. If someone runs kubectl edit in production, Argo CD detects the drift and alerts (or automatically reverts, depending on the sync policy). The cluster cannot stay out of sync with Git for long.

4. Automated self-healing. If a Deployment is accidentally deleted, Argo CD recreates it from Git within 3 minutes — without any human intervention. If a ConfigMap is corrupted, Argo CD restores it from Git. The cluster heals itself from the desired state in Git.

The GitOps operational change for NovaCrestBank's engineers

Before GitOps — push-based

  • Pipeline runs kubectl apply -f deployment.yaml
  • Developer SSH's to jumphost and runs oc edit deployment payment-service to fix prod
  • Cluster state and Git config diverge — no one knows the "true" desired state
  • Rollback = manual — find the old YAML, re-apply it
  • Audit trail: nothing unless the engineer kept their terminal history

After GitOps — pull-based

  • Pipeline commits the new image tag to the config repo — Argo CD deploys
  • Developer opens a PR, gets it reviewed, merges it — Argo CD deploys in minutes
  • Cluster state always converges to the Git repository
  • Rollback = git revert — Argo CD deploys the reverted state
  • Audit trail: every change is a Git commit with author, timestamp, diff
OCP-OPS-05 · Section 02
Argo CD architecture — Application, AppProject, ApplicationSet, and the three CRDs
OpenShift GitOps installs Argo CD as a set of Kubernetes components. Three CRDs form the object model. Understanding each one before configuring them prevents confusion about scope, permissions, and the relationship between a template and an instance.
1
Application
What it is: The primary Argo CD object. An Application connects a source (a path in a Git repository) to a destination (a namespace in an OCP cluster). It defines what to deploy, where to deploy it, and how to sync (automatically or manually). Argo CD continuously reconciles the Application's destination namespace to match the Git source path.

Key fields: source.repoURL (the Git repository URL), source.path (the directory inside the repo containing Kubernetes YAML or Kustomize overlays), source.targetRevision (the Git branch or tag), destination.namespace (where to deploy), project (which AppProject this Application belongs to — controls what it is allowed to deploy).

NovaCrestBank use: One Application per service per environment — payment-service-dev, payment-service-staging, payment-service-production. These are generated automatically by ApplicationSet.
2
AppProject
What it is: A security boundary that scopes what an Argo CD Application is allowed to do. An AppProject declares: which Git repositories Applications in this project can sync from, which destination clusters and namespaces Applications can deploy to, and which Kubernetes resource types Applications are allowed to create or modify.

Why it matters for NovaCrestBank: Without AppProject restrictions, an Application could theoretically sync from any Git repository (including an attacker-controlled one) and deploy to any namespace (including kube-system). AppProject enforcement means: the banking-apps AppProject only allows syncing from gitlab.novacrestbank.local/gitops/banking-config and only deploying to banking-dev, banking-staging, and banking-production namespaces.

The default project: Applications in the default AppProject have no restrictions. Never use the default project for production workloads.
3
ApplicationSet
What it is: A template generator for Applications. Instead of writing three nearly-identical Application objects for dev, staging, and production, you write one ApplicationSet with a generator that produces all three. The ApplicationSet controller watches the generator definition and creates or deletes Application objects as the generator parameters change.

Generators: The List generator takes an explicit list of parameter sets — NovaCrestBank uses this to enumerate dev, staging, and production environments with their specific values. The Git generator can read a list of directories in the config repo and create an Application for each one automatically.

Why ApplicationSet instead of three Application objects: Changes to the sync policy, the target revision, or the source repo need to be made in one place — the ApplicationSet — and they propagate to all generated Applications automatically. Three separate Applications require three separate edits for the same change.

OCP-OPS-05 · Section 03
The two-repository pattern — why source code and config live in separate Git repos
NovaCrestBank uses two Git repositories per application: the application source code repository and the GitOps configuration repository. Understanding why they are separate — and what lives in each — is the architectural foundation of the entire GitOps model.
Why two repositories rather than one monorepo with code and config
The payment-service source code and its Kubernetes deployment YAML could both live in the same repository. Why separate them?
Combining source code and deployment configuration in one repository creates three specific problems that the two-repository pattern solves.

Problem 1 — The deployment repo would change for every commit. Every code commit triggers a pipeline that builds a new image and updates the image tag in the config. If the config is in the same repo as the code, the pipeline would commit back to the same repo that triggered it — creating a recursive trigger loop. Every image build would trigger another pipeline run.

Problem 2 — Different access controls for different teams. Developers need read-write access to the source code repo to commit features. But developers should not be able to directly commit production deployments — that requires review and approval. With two repos: developers have read-write access to the source code repo. Only senior engineers and the automated pipeline have write access to the production path of the config repo. Environment separation is enforced by Git permissions, not by policy.

Problem 3 — The config repo has a different history purpose. The source code history tells the story of feature development. The config repo history tells the story of what was deployed and when — this is the change management and rollback record. Mixing these histories makes both less useful. The config repo's git log is the deployment audit trail; it should contain only deployment commits, not feature development commits.

NovaCrestBank's two repositories:
gitlab.novacrestbank.local/banking/payment-service — the application source code. Developers commit here. The pipeline reads from here.
gitlab.novacrestbank.local/gitops/banking-config — the Kubernetes configuration. The pipeline commits here (image tag updates). Engineers open PRs here for configuration changes. Argo CD reads from here.

What lives in each repository

Source repo — banking/payment-service

  • src/ — Java source code
  • integration-tests/ — integration test suite
  • Dockerfile — container build instructions
  • Jenkinsfile — pipeline definition
  • pom.xml — Maven build config

Argo CD never reads this repo. The pipeline reads it to build the image.

Config repo — gitops/banking-config

  • apps/payment-service/base/ — base Deployment, Service, ConfigMap YAML
  • apps/payment-service/overlays/dev/ — dev-specific patches
  • apps/payment-service/overlays/staging/ — staging-specific patches + image tag
  • apps/payment-service/overlays/production/ — production patches + approved image tag

Argo CD reads only this repo. Developers never deploy directly from here — they open PRs.

OCP-OPS-05 · Section 04
Kustomize overlays — base configuration with environment-specific patches
Kustomize is the templating approach NovaCrestBank uses in the GitOps config repo. It allows a common base configuration to be shared across dev, staging, and production, with each environment applying only its specific differences — without templating languages or variables.
Why Kustomize rather than Helm for the GitOps config repo
Helm is already used to deploy Harbor and the operators. Why use Kustomize for the banking applications in the GitOps config repo?
Helm is a templating and packaging tool — it renders YAML from templates and values files. Kustomize is a patching tool — it takes existing YAML and overlays specific changes without templating. For the GitOps config repo, Kustomize is the correct choice for three reasons.

First, Kustomize is YAML-native — the base configuration is plain Kubernetes YAML that any engineer can read and understand without knowing Helm's templating syntax. The dev overlay is also plain YAML patches. No {{ .Values.image.tag }} syntax, no template rendering required to understand what will be deployed.

Second, Kustomize diffs cleanly in GitLab pull requests. When the pipeline updates the production image tag, the PR diff shows exactly one line changed in the kustomization.yaml — the image reference. With Helm, a diff might span multiple values files and rendered templates. The GitOps config PR reviewer sees exactly what they need to see: one line, one change.

Third, Kustomize is built into kubectl and Argo CD natively — no Helm chart repository management, no chart versioning, no chart dependency resolution. Kustomize overlays are files in the Git repo — they version with the repo.

OCP-OPS-05 · Section 07
Deploy the Argo CD instance — the ArgoCD Custom Resource
The ArgoCD CR tells the GitOps operator how to deploy the Argo CD instance. Every field is annotated with its purpose and the reasoning behind the NovaCrestBank-specific values. This CR deploys all six Argo CD components into the gitops namespace.
Why a dedicated gitops namespace and not openshift-gitops
OpenShift GitOps deploys Argo CD into openshift-gitops by default. Why create a separate namespace?
The default openshift-gitops namespace is visible and accessible to all OCP users by default — it has a ClusterRoleBinding that grants the Argo CD ServiceAccount cluster-admin. For NovaCrestBank's banking environment, this is too broad. A dedicated gitops namespace allows: tighter NetworkPolicy (only the GitOps namespace can reach the config repo, only the Argo CD server receives user connections), a custom ServiceAccount with scoped permissions (deploy only to banking-dev, banking-staging, banking-production — not to kube-system or any other namespace), and cleaner RBAC where the Argo CD admin role maps only to AD groups that should manage deployments.
ArgoCD Custom Resource — every field explained
apiVersion: argoproj.io/v1alpha1
kind: ArgoCD
metadata:
  name: ncb-argocd
  namespace: gitops
    # Dedicated namespace — not openshift-gitops.
    # All Argo CD pods, Secrets, and ServiceAccounts live here.
spec:
  server:
    route:
      enabled: true
      tls:
        termination: reencrypt
                # reencrypt: TLS from the client to the OCP router,
                  # then a NEW TLS connection from the router to the Argo CD server.
                  # This means Argo CD's own certificate is validated end-to-end —
                  # not just at the router edge. This is the correct setting for
                  # applications with their own TLS (as Argo CD does via cert-manager).
    insecure: false
        # false: Argo CD server enforces HTTPS. Never set true in production.

  applicationSet:
    enabled: true
        # Enable the ApplicationSet controller — required for Section 12.

  dex:
    openShiftOAuth: false
        # false: do not use OCP OAuth for Argo CD authentication.
          # true would use OpenShift's OAuth and OCP user accounts.
          # NovaCrestBank uses Active Directory LDAP directly (Section 09) —
          # this gives more granular group mapping than OCP OAuth allows.

  rbac:
    defaultPolicy: role:readonly
        # Any authenticated user who does not match a specific RBAC policy gets
        # read-only access by default. This means:
        # - All NovaCrestBank AD users who can authenticate to Argo CD can VIEW
        #   application status, sync history, and resource trees.
        # - No AD user gets more than read-only access unless explicitly mapped
        #   to a higher role via the policy field below.
        # - Developers see the state of their applications without being able to modify.
    policy: |
            g, NovaCrestBank-GitOps-Admins, role:admin
            # AD group "NovaCrestBank-GitOps-Admins" → Argo CD admin role.
              # Can create/edit/delete Applications, AppProjects, repositories.
              # Platform Engineering team members only.
            g, NovaCrestBank-GitOps-Operators, role:readonly
            # AD group "NovaCrestBank-GitOps-Operators" → read-only.
              # Can view all Applications and trigger manual syncs on approved apps.
              # Operations team members.
    scopes: '[groups]'
        # Argo CD reads the AD user's group memberships from the LDAP token
        # and uses them for RBAC policy matching (the "g" lines above).

  redis:
    autotls: openshift
        # OCP-managed TLS for the Redis pod. Uses the cluster's service CA
        # to secure the Argo CD → Redis connection. Appropriate for in-cluster Redis.

  repo:
    autotls: openshift
        # OCP-managed TLS for the repo-server. Secures the
        # application-controller → repo-server communication.

  ha:
    enabled: false
        # false: single-instance Argo CD. HA runs 3x replicas of all components.
        # For NovaCrestBank's single-cluster deployment, HA is not justified —
        # Argo CD is not in the critical path for running applications.
        # Running applications continue working even if Argo CD is temporarily down.
        # Enable HA only if zero-downtime GitOps reconciliation is a hard requirement.

  resourceExclusions: |
        - apiGroups:
            - tekton.dev
          kinds:
            - TaskRun
            - PipelineRun
          clusters:
            - "*"
      # Exclude Tekton TaskRun and PipelineRun objects from Argo CD's resource tracking.
      # These objects are created and deleted frequently by the pipeline — if Argo CD
      # tracked them, it would constantly show resources as OutOfSync because the
      # cluster has PipelineRun objects that are not in the config Git repo.

OCP-OPS-05 · Section 09
Active Directory authentication for Argo CD
Argo CD's Dex component handles authentication. Configure it to use NovaCrestBank's Active Directory via LDAP — the same LDAP integration used by OCP OAuth (OCP-VCF-02) and Harbor (Registry-01).
Why AD authentication matters beyond convenience — RBAC group mapping
When a NovaCrestBank engineer logs in to Argo CD with their AD credentials, Dex performs an LDAP bind, authenticates the user, and reads their AD group memberships. Those group memberships are included in the OIDC token. Argo CD's RBAC engine reads the groups from the token and applies the policy rules from the ArgoCD CR — members of NovaCrestBank-GitOps-Admins get admin, everyone else gets read-only. When a developer leaves the company and their AD account is disabled, their Argo CD access is revoked automatically on the next login attempt. No separate Argo CD user management is required.

OCP-OPS-05 · Section 11
AppProject — scoping what Argo CD is permitted to deploy
The AppProject is the security boundary for Argo CD Applications. It restricts which repositories Applications can sync from, which namespaces they can deploy to, and which Kubernetes resource types they are allowed to create. Every banking application belongs to this project.
Why AppProject restrictions matter — preventing scope creep in Argo CD
Applications in the default AppProject can deploy to any namespace. Why create a separate banking-apps project with restrictions?
Without AppProject restrictions, a misconfigured Application could deploy to kube-system, to the stackrox namespace, or to any other namespace in the cluster. It could also sync from any Git repository — including one controlled by an attacker if the Argo CD API were somehow compromised or a malicious Application object were created.

The banking-apps AppProject declares exactly: Applications in this project may only sync from gitlab.novacrestbank.local/gitops/banking-config. They may only deploy to banking-dev, banking-staging, and banking-production namespaces. They may only create resource types from the approved list — not ClusterRoles, not ClusterRoleBindings, not custom operator CRDs that could elevate privilege.

These restrictions are enforced by Argo CD's admission webhook — an Application that violates them is rejected when created, not when it syncs. This is a preventive control that bounds the blast radius of any misconfiguration or compromise of the GitOps system.

OCP-OPS-05 · Section 12
ApplicationSet — deploying to dev, staging, and production from one definition
The ApplicationSet is the most powerful Argo CD feature for multi-environment deployments. One ApplicationSet object generates three Application objects — dev, staging, and production — each pointing at the correct Kustomize overlay and using the correct sync policy for that environment.
Why different sync policies for different environments
Why does dev use automatic sync but production uses manual sync? Why not make everything automatic for speed?
The sync policy should match the risk profile of the environment, not the preference for speed.

Development (banking-dev): Auto-sync, no prune, self-heal. A developer commits a change, Argo CD deploys it within 3 minutes. If something is wrong, the developer is right there to notice and fix it. No customers are affected. Speed matters more than caution here.

Staging (banking-staging): Auto-sync, with prune, self-heal. The pipeline commits the new image tag after the ACS gate passes and integration tests run on it — staging should always reflect the pipeline output. Prune is enabled so Argo CD removes resources that are no longer in the config repo (prevents stale resources accumulating in staging). No customer impact.

Production (banking-production): Manual sync only. The production overlay is only updated via an approved merge request (from the pipeline's Stage 8 PR gate). Argo CD detects the change and shows the Application as OutOfSync, but does not apply anything. An operations engineer reviews the diff in the Argo CD UI — confirming it is exactly the expected change — and clicks Sync during an approved change window. Manual sync means two humans have reviewed the production change: the PR approver and the operations engineer who triggers the sync.
ApplicationSet — payment-service across dev, staging, and production

OCP-OPS-05 · Section 14
Sync policies — what automated, prune, and self-heal actually do
The three sync policy options — automated sync, prune, and self-heal — have specific and independent effects. Choosing the wrong combination causes either missed deployments or unexpected resource deletions.
OptionWhat it doesNovaCrestBank settingWhy
automated Argo CD automatically syncs the Application when it detects a diff between the cluster state and the Git state. Without automated, Argo CD only reports OutOfSync — it never applies changes without a human triggering a sync. Dev: Yes · Staging: Yes · Production: No Dev and staging: speed and automation are the priority. Production: manual sync for the final human review before production changes are applied.
prune When a resource exists in the cluster but has been removed from the Git config (for example, a ConfigMap was deleted from the repo), Argo CD deletes it from the cluster during the next sync. Without prune, Argo CD only creates and updates — it never deletes. Dev: No · Staging: Yes · Production: No Staging needs prune to keep the environment clean as config evolves. Dev: no prune because developers sometimes create ad-hoc resources for testing. Production: no prune because deleting a production resource is a human decision that should never be automatic.
selfHeal If someone manually modifies a resource in the cluster that Argo CD manages (kubectl edit, oc patch), Argo CD detects the drift and reverts it to match Git. Without self-heal, Argo CD only syncs when Git changes — it does not revert manual changes. Dev: Yes · Staging: Yes · Production: No Dev and staging: self-heal enforces that those environments always reflect Git — developers cannot leave undocumented changes. Production: self-heal disabled because emergency changes during incidents (a quick memory limit increase to stop an OOM crash) should survive until properly committed to Git.
The danger of enabling prune and self-heal on production
Enabling both prune and self-heal on the production Application creates a scenario where: an engineer makes an emergency change (kubectl edit) to stabilise a production issue, and Argo CD automatically reverts it within 3 minutes, reintroducing the instability. This is the worst possible interaction between GitOps automation and incident response. Production must have selfHeal: false — emergency changes survive until properly committed to Git during the post-incident review.
OCP-OPS-05 · Section 15
Sync waves and hooks — controlling deployment order within a sync
When Argo CD syncs an Application with multiple resources, it applies all resources simultaneously by default. Sync waves allow ordering — for example, ensuring a database migration Job completes before the application Deployment starts.
Why order matters for banking microservice deployments
Why can't Argo CD just apply all resources at once? What goes wrong without sync waves?
Most stateless applications can be updated without ordering — new pods come up, old ones go down, the Service routes traffic. But banking microservices that have a database schema often cannot update this way. If the payment-service v1.2.3 uses a new database column that does not exist yet, and the new pods start before the database migration has run, the pods crash immediately with a SQL column-not-found error. Every transaction attempt fails until the migration completes.

Sync waves assign a wave number (an integer annotation) to each resource. Argo CD applies wave 0 first, waits for all wave-0 resources to be healthy, then applies wave 1, waits for health, then wave 2. For the payment-service: the database migration Job runs in wave 0. When the Job reaches Completed status (health check), Argo CD proceeds to wave 1 which contains the Deployment. The new payment-service pods start after the database schema is ready — no crash, no failed transactions.
Sync wave annotations in the payment-service base resources
# In the database migration Job (base/db-migration.yaml):
apiVersion: batch/v1
kind: Job
metadata:
  name: payment-service-db-migrate
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
        # Wave -1: runs BEFORE wave 0.
          # Argo CD applies this Job first and waits for Job completion (all pods succeeded)
          # before proceeding to wave 0 resources (ConfigMap, Service)
          # and then wave 1 (Deployment).
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
        # PreSync hook: runs before any other resource in the sync.
          # hook-delete-policy: BeforeHookCreation: deletes the old Job before creating the
          # new one. Without this, re-running a sync fails because the Job already exists.

# In the ConfigMap and Service (base/configmap.yaml, base/service.yaml):
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "0"
        # Wave 0: ConfigMap and Service apply after migration completes.
          # Service is needed before the Deployment so pods have a Service to register with.

# In the Deployment (base/deployment.yaml):
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "1"
        # Wave 1: Deployment applies last — after migration and Service are ready.
          # New pods start into a cluster where:
          # - the database schema is current (migration completed in wave -1)
          # - the Service exists (applied in wave 0)
          # Zero downtime, zero SQL errors on startup.
OCP-OPS-05 · Section 16
Health checks — what Synced and Healthy mean for each resource type
An Argo CD Application has two independent status dimensions: Sync Status and Health Status. Understanding both — and what they mean for different Kubernetes resource types — is essential for correctly interpreting the Argo CD dashboard.

Sync status — does the cluster match Git?

Synced — the cluster state exactly matches the Git repository. Every resource the Application manages exists in the cluster and has the same spec as in Git.

OutOfSync — the cluster state differs from Git. Either: a new commit changed the Git state and the cluster has not been updated yet, or someone manually changed a resource in the cluster.

Unknown — Argo CD cannot determine sync status. Usually means the repo-server cannot clone the repository, or the Application is newly created and has not completed its first comparison.

Health status — are the resources working?

Healthy — all resources are operating normally. For Deployments: all desired replicas are available. For Jobs: completed. For Services: endpoints exist.

Progressing — Argo CD is applying changes or waiting for resources to become ready. A rolling Deployment update shows Progressing until the new pods are Available. Normal and transient — if it stays Progressing for more than 10 minutes, investigate.

Degraded — something is wrong. Pods in CrashLoopBackOff, a Deployment with unavailable replicas, a StatefulSet with a failed pod. The Argo CD UI shows which resource is degraded and why.

OCP-OPS-05 · Section 17
Drift detection — when the cluster diverges from Git
Drift is when the cluster's actual state differs from the Git repository. Argo CD detects drift continuously and can either alert, self-heal, or wait for manual action depending on the sync policy.
Drift detection is one of GitOps's most valuable security properties
Without GitOps, configuration drift is invisible — the cluster changes over time through ad-hoc interventions and the documented configuration diverges from reality. With Argo CD, any deviation from Git immediately shows as OutOfSync in the dashboard and triggers a notification. A developer who accidentally runs kubectl edit deployment payment-service -n banking-production and changes the replica count from 3 to 1 will see the production Application go OutOfSync within 3 minutes. The ops team sees the alert in Splunk. With self-heal enabled (dev/staging), it reverts automatically. With self-heal disabled (production), the ops team investigates and decides whether to sync (revert to Git) or update Git to match the intended change.

OCP-OPS-05 · Section 18
Rollback — reverting a bad production deployment
A deployment that causes a production issue needs to be reverted quickly. GitOps rollback is a Git operation — reverting a commit causes Argo CD to deploy the previous state. This is faster, cleaner, and more auditable than a manual kubectl rollout undo.
Git revert vs kubectl rollout undo — why Git is the correct rollback mechanism
OCP has kubectl rollout undo deployment/payment-service which reverts to the previous pod template. Why not use that for emergency rollback?
kubectl rollout undo rolls back the Deployment's pod template to the previous ReplicaSet. It is fast and does not require touching Git. But it creates a divergence: the cluster is now running the previous image but the Git config repo still shows the new image. Argo CD immediately detects this as OutOfSync. If self-heal is enabled, it reverts the rollback — redeploying the bad version. If self-heal is disabled, the cluster stays on the old version but Git says it should be on the new version — causing confusion about what the "correct" state is and leaving a permanent OutOfSync that will be synced by the next person who triggers a sync.

The correct rollback is a Git revert in the banking-config repository: revert the commit that updated the production overlay image tag. This makes Git the source of truth again. Argo CD detects the config repo change and syncs the cluster to the reverted state — exactly like a normal deployment, but backwards. The Git history shows: commit A (previous version), commit B (bad version), commit C (revert of B). The audit trail is complete. No OutOfSync confusion. The correct rollback flow for production is: open a GitLab MR to revert the production overlay commit, get a fast-track approval (production incident = P1 = immediate approver availability), merge, Argo CD detects and deploys the revert.

RBAC for developers and operations checklist
The complete RBAC model for NovaCrestBank's Argo CD — who can do what, and why those boundaries exist. Followed by the full handover checklist confirming OpenShift GitOps is production-ready.

The NovaCrestBank Argo CD RBAC model

The design principle — developers see everything, change nothing directly
The GitOps model requires that developers understand the state of their application across all environments — they should be able to answer "is my latest commit running in production?" by looking at the Argo CD dashboard. But they should not be able to trigger production syncs, modify Application definitions, or add repositories. The RBAC model enforces this: all authenticated AD users get read-only access by default (via defaultPolicy: role:readonly). Elevated access requires membership in a specific AD group. No individual account is granted special Argo CD permissions — group membership controls everything.
AD GroupArgo CD RoleWhat they can doWho
NovaCrestBank-GitOps-Admins role:admin Full Argo CD administration: create/edit/delete Applications, AppProjects, and cluster credentials. Configure repositories. Override sync policies. Should have at most 5 members — the principal platform engineers. Platform Engineering leads
NovaCrestBank-GitOps-Operators role:readonly + manual sync on banking-apps View all Applications. Trigger manual syncs on production Applications. Cannot create or modify Applications, AppProjects, or repositories. The role for operations engineers who execute production deployments. Operations team
NovaCrestBank-Banking-Developers proj:banking-apps:developer-readonly View all Applications in the banking-apps AppProject. See sync status, health status, resource trees, sync history, and logs. Cannot trigger syncs, cannot modify anything. The role for development teams. Banking application development teams
All other authenticated AD users role:readonly (defaultPolicy) View all Applications across all projects. Read-only — cannot trigger syncs or modify anything. Gives security team, auditors, and executives visibility without any write access. Anyone who can authenticate with AD

What "view" actually allows developers to see

A developer with developer-readonly access can see everything needed to understand their application's deployment state:

Development

Current sync status, pod count and health, ConfigMap contents, running image digest, recent sync history and which commits triggered each sync, live pod logs.

Staging

All of the above for staging. The developer can see exactly which pipeline run image is deployed, whether it is the latest, and the full resource tree including NetworkPolicy and Service endpoints.

Production

Full visibility: current image digest running in production, replica count, last sync time and who triggered it, sync history going back to the Application's creation. Cannot trigger sync.
