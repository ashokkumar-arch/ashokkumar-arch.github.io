The core concept — what an operator actually is
The word "operator" appears everywhere in OpenShift. What does it actually mean?
An operator is a piece of software that automates the operational knowledge a human expert would otherwise need to apply manually — encoded once and then executed continuously, forever, by a control loop running inside the cluster.

To understand this properly, start from something familiar. Kubernetes already has built-in controllers: the ReplicaSet controller watches ReplicaSet objects and ensures the right number of pods are running. The Deployment controller watches Deployment objects and handles rolling updates. These controllers are the foundation of how Kubernetes works — they implement the reconciliation loop: observe current state, compare to desired state, take action to close the gap.

An operator extends this pattern to application-specific knowledge. Consider what a human database administrator does when asked to set up a production PostgreSQL cluster: they create multiple pods, configure replication, set up backup schedules, tune memory parameters, handle failover, and rotate certificates. This is operational knowledge that lives in a person's head. An operator encodes that same knowledge into a controller that runs continuously. You tell it "I want a PostgreSQL cluster" by creating a PostgresCluster object — a Custom Resource — and the operator's controller loop does everything the DBA would do, continuously, responding to failures and changes automatically.

The three-word definition: An operator is a domain-specific Kubernetes controller.

The reconciliation loop — the engine behind every operator

Every operator — without exception — is built on the same loop. Understanding this loop means you understand what every operator does at its core, regardless of what it manages.

Observe
Watch for changes
The operator registers watchers on specific Kubernetes objects — both its own Custom Resources (Certificate, Pipeline, ArgoCD) and standard objects (Pods, Services, Secrets). When any watched object changes, Kubernetes puts a reconciliation request on the operator's work queue. The operator does not poll — it is event-driven.
Desired
Read desired state
When the reconcile function runs, it reads the Custom Resource that triggered it — for example, a Certificate object that says "I want a TLS certificate for harbor.apps.ocp.novacrestbank.local, signed by NovaCrestBank-IssuingCA, renewed 30 days before expiry." This is the desired state.
Actual
Observe actual state
The operator then looks at what actually exists in the cluster — is there already a Kubernetes Secret with a valid certificate for that hostname? When does it expire? Is the certificate signed by the right CA? This is the actual state.
Compare
Compute the diff
The operator compares desired state to actual state and determines what actions are needed. No action needed → the reconcile function returns successfully and nothing happens. Action needed → proceed to the next step.
Act
Take corrective action
The operator calls the appropriate APIs to close the gap. Cert-Manager requests a new certificate from the IssuingCA. The Logging Operator creates a Fluentd DaemonSet on every node. ACS deploys its sensor pods to a new cluster. The action is exactly what a human operator would do — but done in milliseconds, every time, without error.
Status
Update status
After acting, the operator updates the .status field of the Custom Resource to reflect what happened — "certificate issued, expires 2028-04-20, renewal scheduled for 2028-03-21." This is how you observe what an operator has done: oc describe certificate shows the status the operator wrote.
Why this loop runs forever — and why that is the point
The reconciliation loop does not run once and stop. It runs continuously, triggered by any change to any watched object. If someone manually deletes a certificate Secret that an operator manages, the operator detects the deletion within seconds (because it watches Secrets) and recreates it. If a node crashes and takes an operator-managed pod with it, the operator detects the missing pod and recreates it elsewhere. This self-healing property is what makes operators operationally powerful — the human configuration intent is maintained automatically regardless of what happens to the cluster.
OCP-OPS-01 · Section 02
Custom Resource Definitions — extending the Kubernetes vocabulary
An operator needs something to reconcile. CRDs are how operators add new nouns to Kubernetes — new object types that carry configuration the operator understands and acts on.
What a CRD is — and why it matters to NovaCrestBank's cluster
What is a Custom Resource Definition and how does it relate to operators?
Kubernetes ships with built-in resource types: Pod, Deployment, Service, ConfigMap, Secret, PersistentVolumeClaim. These cover general-purpose workloads. A Custom Resource Definition (CRD) is a schema that teaches Kubernetes about a new resource type — one that Kubernetes did not ship with but that an operator understands.

When you install Cert-Manager, it registers three new CRDs: Certificate, Issuer, and ClusterIssuer. After those CRDs are installed, oc get certificate works just like oc get pod — Kubernetes now knows what a Certificate is, stores it in etcd, and enforces the schema. Cert-Manager's controller loop watches for Certificate objects and acts on them.

When you install OpenShift Pipelines, it registers: Pipeline, Task, PipelineRun, TaskRun, Workspace. When you install ACS, it registers: Central, SecuredCluster. Every operator brings its own CRDs — its own vocabulary — and then watches for objects of those types.

The practical consequence for NovaCrestBank: installing an operator does nothing visible to application teams until someone creates a Custom Resource of the type the operator manages. Installing Cert-Manager does not create any certificates. Creating a Certificate object causes Cert-Manager to issue that certificate. The CR is the declaration of intent; the operator is the engine that fulfils it.

CRDs persist in etcd — they are cluster data, not just operator config
When an operator is uninstalled, its CRDs and any Custom Resources created from them remain in etcd unless explicitly deleted. This is intentional — it prevents accidental data loss if an operator is removed for maintenance. It also means: deleting an operator does not delete the Certificates, Pipelines, or Applications it was managing. The objects survive; the controller that acted on them is gone. Always clean up CRs before uninstalling an operator, or re-install the operator to let it handle its own cleanup.
OCP-OPS-01 · Section 03
Why operators instead of Helm charts or manual YAML
Helm charts and manual YAML are not wrong — they are the right tool for deploying stateless applications. Operators are the right tool when operational knowledge needs to be encoded and executed continuously. Understanding the distinction prevents misusing either.
Helm vs Operator — choosing the right tool
We already use Helm to deploy Harbor. Why not use Helm to deploy everything — Prometheus, Argo CD, ACS?
Helm is a templating and packaging tool. It renders YAML templates with values and applies them to the cluster. Once applied, Helm's job is done — it does not watch the cluster, does not respond to failures, does not handle day-2 operations like certificate rotation or version upgrades. Helm is excellent for deploying an application once; it is not designed to operate an application continuously.

Consider Prometheus. Deploying Prometheus with Helm is straightforward — a Helm chart renders the StatefulSet, Services, and ConfigMaps. But running Prometheus in production involves: adding new scrape targets when a new application is deployed, adjusting alert rules when the team adds new SLOs, upgrading Prometheus when a new version is released while maintaining the existing configuration, and recovering from storage corruption. With Helm, each of these requires a human to run helm upgrade with the right values. With the Cluster Monitoring Operator, each of these is handled automatically — new scrape targets are discovered via ServiceMonitor objects, alert rules are added via PrometheusRule objects, and upgrades are handled by the operator's reconciliation loop.

The practical boundary:
Use Helm (or plain YAML) for: stateless applications, one-time deployments, simple services that do not require ongoing operational decisions, anything where the desired state is static and does not need to respond to cluster events.
Use operators for: databases, observability stacks, certificate management, security tools, CI/CD engines — anything where the desired state is dynamic and operational decisions need to be made continuously based on what is happening in the cluster.

A concrete comparison — deploying Prometheus three ways

Way 1
Manual YAML
Create StatefulSet, Service, ConfigMap manually. No upgrade path. Adding a new scrape target requires editing the ConfigMap and restarting. Certificate rotation requires manually updating a Secret. Node failure recovery is manual. This is how Prometheus was managed before operators existed — it works, but the operational burden is entirely on the human.
Way 2
Helm chart
Better than manual YAML — a chart packages the components, provides configurable values, and supports helm upgrade. But adding a new scrape target still requires a helm upgrade with updated values. The chart does not watch for new Service objects and auto-discover targets. It does not automatically renew certificates. Operational knowledge still lives outside the cluster, in the team's runbooks.
Way 3
Operator (Prometheus Operator)
You create a Prometheus Custom Resource and ServiceMonitor objects. The operator handles everything else — creating the StatefulSet, mounting the right config, discovering new ServiceMonitors as they appear (any team can add their own ServiceMonitor to add a scrape target without touching the Prometheus config), handling storage, managing upgrades. The operational knowledge is encoded in the operator and runs continuously.

Anatomy of an operator — what physically runs in the cluster
An operator is ultimately a set of Kubernetes objects. Knowing what those objects are means you can find, inspect, and diagnose any operator using standard OCP tools.
Every operator resolves to the same set of components
Regardless of what an operator manages — certificates, pipelines, log forwarding — its physical presence in the cluster is always the same structure: a Deployment running the controller binary, RBAC permissions allowing it to watch and modify cluster objects, and the CRDs that define its vocabulary. When you know this, you know how to inspect any operator.
Pod
Controller pod
The operator's executable runs as a Deployment (usually 1 replica, sometimes 2 for HA) in a dedicated namespace. This pod contains the reconciliation loop code. It watches the API server for changes to its CRDs and standard objects, and calls the Kubernetes API to act on them. If this pod crashes, the operator stops reconciling — existing resources continue to run but are no longer managed until the pod recovers. Find it: oc get pods -n <operator-namespace>.
RBAC
ServiceAccount + ClusterRole + ClusterRoleBinding
Every operator has a dedicated ServiceAccount that its controller pod runs as. A ClusterRole grants it exactly the permissions it needs — for example, Cert-Manager needs permission to create Secrets (to store issued certificates), read ConfigMaps (to find CA configuration), and create CertificateRequests. The ClusterRoleBinding connects the ServiceAccount to the ClusterRole. Find it: oc get clusterrole | grep <operator-name>.
CRDs
CustomResourceDefinitions
The schemas the operator understands. Installed cluster-wide (not namespaced). Once installed, users can create objects of these types in any namespace. The CRDs include validation — if you create a Certificate object with an invalid field, the API server rejects it at create time based on the CRD's schema, before the operator even sees it. Find them: oc get crd | grep <operator-domain>.
Watcher
Informers (inside the pod)
Inside the controller pod, the operator establishes informers — cached watches — on every object type it cares about. When a Certificate is created, when a Pod crashes, when a Secret is deleted — these events are delivered to the reconcile queue. The informer cache means the operator does not need to query the API server on every reconciliation loop iteration; it reads from an in-memory cache that is kept in sync by a background watcher.
Status
Status subresource on Custom Resources
The operator writes its findings and actions back to the .status field of the Custom Resource it manages. Status conditions follow the same pattern as cluster operators: Type: Ready, Status: True/False/Unknown, Reason, Message. This is how you know what the operator has done and whether it succeeded. oc describe certificate <name> → look at the Conditions section.

OCP-OPS-01 · Section 05
OLM — the Operator Lifecycle Manager — the operator that manages operators
OLM is built into every OpenShift cluster. It is responsible for installing, updating, and removing operators. Every operator you install through OperatorHub or via a Subscription object goes through OLM.
What OLM does — and why it exists
Operators are just Kubernetes Deployments and CRDs — why do you need a separate system (OLM) to manage them? Why not just kubectl apply the operator's manifests?
You can install operators manually with kubectl apply — and for simple operators in development, this works fine. OLM exists because production environments need to answer four questions that manual installation does not answer: How do you know which version of an operator is compatible with this OCP version? How do you handle updates when a new operator version is released? How do you prevent two operators from installing conflicting CRDs? How do you declare "I always want the latest stable version of this operator" without running manual checks every week?

OLM answers all four. It maintains a catalog of operators (a list of every available operator, its versions, its dependencies, and its compatibility matrix). When you create a Subscription object for an operator, OLM resolves which version to install, checks for conflicts with already-installed operators, installs it, and then watches for new versions in the catalog. When a new version appears that satisfies your update policy, OLM either installs it automatically (if approval is set to Automatic) or creates an InstallPlan object waiting for your approval (if set to Manual).

For NovaCrestBank, every operator uses Manual approval. This means operator updates do not happen automatically — an OCP administrator must review the InstallPlan and approve it. In a banking environment where change management requires planned, reviewed updates, automatic operator updates would violate change control procedures.

OLM components that run in every OCP cluster

OLM
olm-operator pod
The core OLM controller. Watches Subscription objects and InstallPlan objects. When a Subscription is created, it queries the catalog, resolves the correct operator version, and creates an InstallPlan. When an InstallPlan is approved, it applies the operator's manifests (CRDs, Deployment, RBAC) to the cluster.
Cat
catalog-operator pod
Watches CatalogSource objects and Subscription objects for new versions. Periodically polls each CatalogSource for new operator versions. When a new version appears in a channel that a Subscription is tracking, it creates a new InstallPlan. This is what drives operator updates — the catalog-operator is the engine behind "new version available" notifications in OperatorHub.
Cat
CatalogSource pod (one per catalog)
A CatalogSource is a database of operators — it lists every available operator, its versions, its channels, its dependencies, and its install manifests. OCP ships with four built-in CatalogSources pointing at Red Hat's catalogs on quay.io. After Registry-01's mirroring work, these are redirected to Harbor's platform project via the ImageDigestMirrorSet.
OCP-OPS-01 · Section 06
OLM objects — CatalogSource, Subscription, InstallPlan, CSV
Four OLM objects are involved every time an operator is installed. Understanding what each one is and what it does makes the installation process transparent rather than a black box.
1
CatalogSource
What it is: A pointer to a registry image that contains an operator catalog database. OCP ships with redhat-operators, certified-operators, community-operators, and redhat-marketplace by default. After Registry-01, NovaCrestBank's redhat-operators CatalogSource points to the mirrored image in Harbor.
Find it: oc get catalogsource -n openshift-marketplace
2
Subscription
What it is: Your declaration of intent — "I want this operator from this catalog, on this channel, with this update approval mode." Creating a Subscription triggers OLM to resolve and install the operator. It also keeps the subscription active for future updates.
Find it: oc get subscription -n <operator-namespace>
Key fields: channel (which release stream to follow), installPlanApproval (Manual for NovaCrestBank — all operator updates require human approval)
3
InstallPlan
What it is: OLM's resolved plan for installing a specific operator version — a list of all the CRDs, Deployments, ServiceAccounts, and RBAC objects that will be applied to the cluster. With Manual approval, the InstallPlan sits in Approved: false state until an administrator reviews and approves it. This is the change control gate for operator installs and updates in NovaCrestBank.
Find it: oc get installplan -n <operator-namespace>
Approve it: oc patch installplan <name> -n <ns> --type merge -p '{"spec":{"approved":true}}'
4
ClusterServiceVersion (CSV)
What it is: The operator's "manifest of everything it is" — its version, the CRDs it owns, the permissions it requires, its owned and required APIs, and metadata for the OperatorHub UI. The CSV is what OLM actually applies when installing an operator. One CSV per operator version — upgrading an operator means OLM creates a new CSV and removes the old one.
Find it: oc get csv -n <operator-namespace>
Key field: phase: Succeeded means the operator is installed and running correctly

OCP-OPS-01 · Section 07
OperatorHub — the catalog interface, air-gap behaviour, and Harbor
OperatorHub is the web UI in the OCP console that displays available operators from all CatalogSources. After Registry-01's mirroring, OperatorHub shows operators from Harbor — not the internet.
OperatorHub after air-gap mirroring — what changes and what does not
After applying the ImageDigestMirrorSet in Registry-01, does OperatorHub still work? How does it know to use Harbor instead of Red Hat's catalog?
OperatorHub continues to work exactly as before, but the images it uses to display and install operators now come from Harbor instead of Red Hat's registries. The CatalogSource objects were updated by oc-mirror's output to point at Harbor's platform project, and the ImageDigestMirrorSet redirects any remaining digest-based pulls from quay.io to Harbor automatically.

What you see in the UI is identical — the same operator names, descriptions, logos, and versions. What changed is where the catalog database pod pulls its image from: Harbor's platform project instead of quay.io. When an admin installs an operator from OperatorHub, OLM creates an InstallPlan, the catalog resolves the operator's image references to Harbor-mirrored digests, and all pods that are part of the operator installation are pulled from Harbor.

One important consequence for NovaCrestBank: only operators that were included in the oc-mirror ImageSetConfiguration are available in OperatorHub. If an operator was not mirrored, it appears in OperatorHub as available but fails to install because its image is not in Harbor and there is no route to the internet. Before installing any new operator, verify it is included in the ImageSetConfiguration and re-run oc-mirror to pull it in.
Disable the default internet-pointing CatalogSources after air-gap mirroring
OCP ships with four default CatalogSources that point at Red Hat's registries. After mirroring, these should be disabled — OLM will try them if the Harbor-based catalog does not have a requested operator, and the internet-pointing pull will fail loudly (or succeed if the network allows, bypassing your air-gap controls). Disable in the OCP console: Administration → Cluster Settings → Configuration → OperatorHub → Sources → uncheck unwanted sources. Or via CLI: oc patch operatorhub cluster --type json -p '[{"op":"add","path":"/spec/disableAllDefaultSources","value":true}]'
OCP-OPS-01 · Section 08
Channels and update approval — controlling when operators update
Channels and approval modes are how NovaCrestBank controls the pace of operator updates. The wrong setting here means operators update automatically during production hours, which violates change management policy.
What a channel is — and which one to choose
When installing an operator, you specify a channel. What is a channel and how does NovaCrestBank choose?
A channel is a release stream for an operator — similar to a Linux package repository stream or a software support tier. Different channels receive updates at different cadences and have different stability guarantees.

Common channel names and what they mean:

stable — the recommended channel for production. Updates are tested and stable before being promoted here. Red Hat guarantees support for operators on the stable channel. This is the correct choice for NovaCrestBank for most operators.

latest — more recent than stable, may include new features that are still maturing. Some operators (like OpenShift Pipelines) only publish to latest. For those operators, latest is the production channel.

stable-x.y — a version-pinned channel (e.g., stable-5.9 for OpenShift Logging). This means "only receive updates within the 5.9.x minor release family." Choosing a version-pinned channel prevents major version updates that might require changes to NovaCrestBank's Logging configuration. This is the right choice for operators where major version updates have breaking changes that need planning.

NovaCrestBank default rule: use stable when available. Use version-pinned channels (stable-5.9) for operators with significant configuration — Logging, Loki, ACS. Use the operator documentation to understand what a major version upgrade entails before subscribing to an unpinned channel.

Manual vs Automatic approval — NovaCrestBank's policy

Automatic approval

OLM applies new operator versions immediately when they appear in the catalog. No human review. Fast, but an operator update can break running workloads without warning or a change management record. Not appropriate for NovaCrestBank production.

Manual approval — NovaCrestBank standard

A new operator version creates an InstallPlan with Approved: false. An OCP administrator reviews the change, raises a change request in the NovaCrestBank change management system, receives approval, then approves the InstallPlan during the change window. Every operator update is a documented, reviewed change.

Monitoring for pending InstallPlans — the operational routine
With Manual approval, operator updates accumulate as unapproved InstallPlans. Include this check in the weekly OCP operations review: oc get installplan -A | grep -v Approved. For each unapproved InstallPlan, review the CSV changelog (available on the Red Hat operator documentation page for that operator version), assess the impact, raise a change request, and approve during the next available change window. Never approve an InstallPlan without reading the operator changelog for that version.

OCP-OPS-01 · Section 10
Cert-Manager — automatic certificate lifecycle for every service
A concept introduction before OCP-OPS-02 installs it. Understanding what Cert-Manager does and why NovaCrestBank needs it at scale makes the installation decisions in OCP-OPS-02 obvious rather than arbitrary.
The certificate management problem at NovaCrestBank's scale
We already replaced the OCP platform certificates manually in OCP-VCF-02. We did the same for NSX and vCenter in WIN-VCF-03. Why is an operator needed for certificates?
Manual certificate management works for a small, fixed set of infrastructure components. It becomes a serious operational risk when the number of certificate-bearing services grows beyond what a human can reliably track.

In OCP-VCF-02 and WIN-VCF-03, NovaCrestBank manually managed certificates for: SDDC Manager, NSX Manager cluster (3 nodes), vcenter-mgmt, vcenter-ocp, OCP API, OCP Ingress wildcard. That is roughly 8 certificates, each with a 2-year lifetime, each requiring a manual CSR, manual IssuingCA submission, manual Secret update, and manual service restart. Manageable with a calendar reminder.

Now add the banking application layer: 50 microservices, each with its own HTTPS endpoint inside the cluster for mutual TLS between services. Each service needs a certificate. Each certificate needs to be renewed before it expires. A certificate that expires unnoticed causes a service outage — and in a banking environment, "payment service unreachable because TLS cert expired" is a P1 incident with regulatory implications.

Cert-Manager automates the entire lifecycle: you create a Certificate object (or annotate an Ingress, which triggers automatic certificate creation), Cert-Manager requests the certificate from NovaCrestBank-IssuingCA, stores it as a Kubernetes Secret, and monitors the expiry date. 30 days before expiry, Cert-Manager automatically requests a renewal. The application never sees the renewal — the Secret is updated in place and the pod gets the new certificate without restart (for most applications).
Cert-Manager
cert-manager.io · CNCF Graduated · OCP-OPS-02
What it does: Manages the full lifecycle of X.509 certificates inside the cluster — issuance, storage as Kubernetes Secrets, renewal 30 days before expiry, and revocation. OCP-OPS-01 · Section 11
Red Hat Advanced Cluster Security — runtime security and compliance
ACS is the security layer that watches what is actually happening in the cluster at runtime — not just what the configuration says should happen. It is the difference between configuration security and operational security.
What ACS does that no other tool in the stack does
We already have: NSX-05 DFW for network security, Harbor Trivy for vulnerability scanning, OCP RBAC for access control. What gap does ACS fill that these tools do not cover?
NSX-05, Trivy, and RBAC are all preventive controls — they try to stop bad things from entering the cluster. ACS is a detective control — it watches what is actually happening inside the cluster at runtime and alerts on suspicious behaviour, regardless of how it got past the preventive controls.

A concrete example: a banking application pod that has never executed curl or any network tool suddenly starts doing curl https://suspicious-domain.com. Trivy does not catch this — the image was clean when it was scanned. NSX-05 DFW may or may not have a rule blocking this specific egress. RBAC has no visibility into what processes run inside pods. ACS, however, has a baseline of "normal" behaviour for this pod (built from observing it over time) and alerts immediately when an unexpected process executes or an unexpected network connection is attempted.

Four things ACS does specifically:
1. Runtime process monitoring — alerts when a pod executes a process that was not in the original container image (for example, a shell spawned by an exploited web application).
2. Cluster-wide vulnerability management — continuous scan of all running images, not just images on push. If a new CVE is published that affects an image already running in production, ACS surfaces it immediately without waiting for the next Harbor push.
3. Network policy assistance — observes actual network connections between pods and generates recommended OCP Network Policies based on observed traffic. This is how NovaCrestBank generates its initial Network Policy baseline without guessing.
4. Compliance reporting — runs automated checks against CIS Kubernetes Benchmark and PCI-DSS controls, producing reports that can be submitted directly as audit evidence.
Red Hat Advanced Cluster Security for Kubernetes
rhacs-operator · Red Hat · OCP-OPS-03
Architecture: Central (management UI, policy engine, vulnerability database — one instance per environment) + SecuredCluster (sensor deployed to each OCP cluster being monitored — installs a DaemonSet collector on every node). In NovaCrestBank's deployment, Central runs on the OCP cluster it is protecting (a valid configuration for single-cluster environments).

Key CRDs: Central (the ACS management plane — one per environment), SecuredCluster (registers an OCP cluster with Central for monitoring).

NovaCrestBank compliance controls: PCI-DSS Req 6.3.3 (runtime vulnerability management), Req 11.3 (intrusion detection — ACS runtime alerts), Req 11.5 (file integrity monitoring via process execution monitoring).
Channel: stable Namespace: stackrox Approval: Manual

OpenShift Pipelines — Tekton-based CI/CD inside the cluster
Pipelines provides the build and test infrastructure that runs entirely inside OCP — no external CI server required. Each build step runs as an ephemeral pod, with its own RBAC, on the cluster's own resources.
OpenShift Pipelines and Jenkins — how they work together
Phase 9 (GitOps with Jenkins) also involves pipelines. What is the relationship between OpenShift Pipelines (Tekton) and Jenkins?
Jenkins and OpenShift Pipelines are complementary, not alternatives. Jenkins is the orchestrator — it receives webhooks from Git, makes decisions about what to build and when, handles branch and PR logic, and provides the familiar Jenkins UI for developers. OpenShift Pipelines provides the execution engine — the actual build steps run as Tekton TaskRuns (pods) inside OCP.

In NovaCrestBank's architecture: the Jenkins pipeline (the Jenkinsfile) defines the overall workflow. For the actual build steps — compile the Java code, run unit tests, build the container image with Buildah — Jenkins triggers a Tekton PipelineRun. The Tekton tasks run as ephemeral pods, each step isolated in its own container. When the Tekton run completes, Jenkins receives the result and decides what happens next (promote to staging, open a PR, notify Slack).

The reason for this split: Jenkins is excellent at orchestration and has decades of plugin ecosystem for notification, quality gates, and workflow control. Tekton is excellent at secure, isolated, Kubernetes-native build execution. Using both gives NovaCrestBank the strengths of each without the weaknesses of either.
OpenShift Pipelines
openshift-pipelines-operator-rh · Red Hat · OCP-OPS-04
What it installs: The Tekton Pipelines controller, the Tekton Triggers controller (for webhook-driven PipelineRuns), the Tekton Chains controller (for supply chain attestation — records what was built and how, signed with cosign), and the Pipelines as Code operator (for Tekton pipelines defined in the application's Git repository).

Key CRDs: Pipeline (the workflow definition — an ordered sequence of Tasks), Task (a single build step — build, test, push, sign), PipelineRun (a specific execution instance of a Pipeline — creates pods for each Task step), TektonConfig (cluster-level Pipelines configuration).

Security model: Each PipelineRun pod runs with the ServiceAccount of the namespace it is in — not as a privileged cluster account. Buildah (container image building) runs without Docker daemon, without host path mounts, as a non-root user. This is fundamentally more secure than Jenkins building images with Docker-in-Docker.

OCP-OPS-01 · Section 13
OpenShift GitOps — Argo CD for declarative, auditable deployment
GitOps replaces kubectl apply in production with Git commits. Every application deployment is a reviewed, approved change in a Git repository — with a full diff, a reviewer, and an immutable audit trail.
Why GitOps is the correct deployment model for a regulated banking environment
We can already deploy applications with oc apply or Helm. Why add GitOps on top?
PCI-DSS Req 6.4 requires that all changes to system components go through a formal change management process — reviewed, tested, and approved before deployment. With manual kubectl apply or ad-hoc Helm releases, the only record of what changed is in a person's terminal history (if they kept it). With GitOps, every change to every deployed application is a Git commit — with a diff showing exactly what changed, a reviewer who approved it, a timestamp, and an author. The Git history is the change log.

Beyond compliance: GitOps eliminates configuration drift. In a manually managed cluster, the actual running state and the intended state diverge over time — a developer patches a ConfigMap to fix a production issue and forgets to update the YAML file. Argo CD continuously compares the Git repository (desired state) to the cluster (actual state) and alerts on — or automatically corrects — any deviation. What is in Git is what runs in production, always.
OpenShift GitOps (Argo CD)
openshift-gitops-operator · Red Hat · OCP-OPS-05
What it installs: An Argo CD instance in the openshift-gitops namespace (managed by the operator), the Argo CD web UI (accessible at openshift-gitops-server.apps.ocp.novacrestbank.local), and an ApplicationSet controller for template-driven multi-environment deployments.

Key CRDs: ArgoCD (the Argo CD instance configuration), Application (a connection between a Git repository path and a cluster namespace — "deploy what is in this Git folder to this namespace"), ApplicationSet (a template that generates multiple Applications — used for deploying the same application to dev/staging/production from a single definition).

NovaCrestBank deployment model: Two Git repositories: the application source repo (where developers commit code — read by Jenkins/Pipelines) and the GitOps config repo (where the Kubernetes YAML for each application lives — Argo CD syncs from here to the cluster). Developers never directly edit the cluster — they edit YAML files in the config repo via pull requests.

OCP-OPS-01 · Section 15
External Secrets Operator — CyberArk integration for automated secret rotation
Kubernetes Secrets stored as base64-encoded YAML in etcd are not secrets in any meaningful security sense — they are accessible to anyone with the right RBAC. External Secrets Operator pulls the actual secret values from CyberArk at runtime, keeping credentials out of Git and out of the cluster's configuration layer.
The GitOps secret problem — and how ESO solves it
GitOps says everything is in Git. But database passwords cannot go in Git. How does ESO resolve this contradiction?
ESO separates the secret reference (which is safe to put in Git) from the secret value (which is not). In your GitOps config repo, you commit an ExternalSecret object that says "the value for this Kubernetes Secret comes from CyberArk at path /novacrestbank/prod/payment-db-password". The ESO controller reads this reference, authenticates to CyberArk, fetches the actual password value, and creates a standard Kubernetes Secret with that value. The Git repository contains only the reference path — never the password itself.

The rotation benefit: when CyberArk rotates the database password (which it does automatically on a schedule), ESO detects that the CyberArk value has changed (it polls on a configurable interval, typically every hour). It updates the Kubernetes Secret with the new value. If the application is configured to watch the Secret (via a volume mount with inotify) or if a rolling restart is triggered by an annotation, the application picks up the new password without any manual intervention. PCI-DSS Req 8.6 requires automated management of interactive and non-interactive accounts — ESO with CyberArk satisfies this for database credentials, API keys, and service account tokens.
External Secrets Operator
external-secrets · external-secrets.io · CNCF Sandbox · GITOPS-05
What it installs: The ESO controller pod, the ESO webhook (validates ExternalSecret objects on creation), and the cert-controller (manages ESO's own internal certificates — requires Cert-Manager to be installed first).

Key CRDs: ClusterSecretStore (one cluster-wide connection to CyberArk — the authentication configuration and endpoint), ExternalSecret (per-application: which CyberArk path maps to which Kubernetes Secret key).

CyberArk integration: ESO connects to CyberArk Conjur (or CyberArk Vault) using a Kubernetes ServiceAccount JWT token for authentication. No static credentials — CyberArk trusts OCP's identity. The ClusterSecretStore configures this trust relationship once; all ExternalSecrets in all namespaces use it.

OCP-OPS-01 · Section 16
The standard installation flow — what happens every time an operator is installed
Every operator installation in NovaCrestBank follows the same four-step flow. Knowing this flow means you can install any operator — even ones not in this guide — by applying the same pattern.
  • 1
    Create the namespace and OperatorGroup

    Most operators install into their own namespace. An OperatorGroup object tells OLM which namespaces this operator should watch for Custom Resources. For operators that manage cluster-wide resources (Cert-Manager, ACS, Logging), the OperatorGroup targets all namespaces. For operators scoped to one team's namespace, the OperatorGroup targets only that namespace.

  • 2
    Create the Subscription — with Manual approval

    The Subscription specifies: which operator (package name from the catalog), which catalog (sourceNamespace: openshift-marketplace), which channel, and installPlanApproval: Manual. Creating the Subscription triggers OLM to query the catalog and create an InstallPlan.

  • 3
    Review and approve the InstallPlan

    OLM creates an InstallPlan with spec.approved: false. Review what the InstallPlan will install (oc describe installplan <name> -n <ns> shows the CSV and all CRDs it will apply). Verify it matches expectations. Then approve: oc patch installplan <name> -n <ns> --type merge -p '{"spec":{"approved":true}}'

  • 4
    Verify the CSV reaches Succeeded phase

    After approval, OLM installs the operator's CRDs, Deployment, and RBAC. Watch the CSV: oc get csv -n <ns> -w. The CSV progresses through phases: Pending → Installing → Succeeded. Succeeded means the operator pod is running and ready to reconcile Custom Resources. Any other final state is an error — describe the CSV for diagnostic information.

Troubleshooting operators — what to check first
Operator problems follow predictable patterns. This section covers the five most common issues and the exact commands to diagnose them — applicable to every operator in the NovaCrestBank cluster.

The five most common operator problems

SymptomMost likely causeFirst command to run
CSV stuck in Installing phase Controller pod cannot start: image pull failure (not mirrored in Harbor), resource quota exceeded on the namespace, or PVC not bound (StorageClass issue) oc describe csv <name> -n <ns> — look at the Status.Conditions section for the specific reason. Then oc get pods -n <ns> — look for ImagePullBackOff or Pending.
CSV shows Succeeded but Custom Resources stay in unknown state The operator's controller pod is running but its reconciliation is failing for that specific CR — wrong configuration in the CR, RBAC missing, or the operator cannot reach an external dependency oc describe <crd-name> <cr-name> -n <ns> — read the Status.Conditions that the operator wrote. Then oc logs -n <operator-ns> <controller-pod> — search for ERROR lines mentioning the CR name.
Operator update available but InstallPlan not approved Expected behaviour with Manual approval — this is a feature, not a bug. An admin must review and approve the InstallPlan during a change window. oc get installplan -A | grep false — lists all pending approvals. Review each one's CSV changelog before approving.
CatalogSource pod in CrashLoopBackOff The operator catalog image cannot be pulled from Harbor — either the mirror is stale (oc-mirror needs to be re-run) or the robot-ocp pull credentials have expired oc describe catalogsource <name> -n openshift-marketplace — check the image reference. Operator was working, now the controller pod is crashing repeatedly Usually a failed update — the new CSV version has a bug, or a CRD schema change is incompatible with existing Custom Resources in the cluster oc logs -n <ns> <pod> --previous — the previous pod's logs often contain the crash reason. If the crash is due to a CRD schema change, existing CRs may need to be migrated. Check the operator's release notes for the version that was installed.
