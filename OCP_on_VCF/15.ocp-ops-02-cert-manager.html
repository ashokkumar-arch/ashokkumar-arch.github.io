The scale problem — why manual works for 8 certs but fails at 80
We successfully managed certificates manually in WIN-VCF-03 and OCP-VCF-02. Why change the approach for application certificates?
Manual certificate management has a linear cost: each certificate requires a human action at creation and at every renewal. At 8 certificates with 2-year lifetimes, this means roughly 4 manual renewals per year — manageable with a calendar reminder. At banking application scale, the numbers change fundamentally.

Consider NovaCrestBank's application layer at production scale: 50 banking microservices, each exposing an HTTPS endpoint for internal service-to-service communication (mutual TLS — PCI-DSS Req 4.2.1 requires encryption of cardholder data in transit, which includes internal service calls). Each service needs its own certificate. That is 50 certificates. If the certificate lifetime is 1 year, that is 50 manual renewals per year. If it is 90 days (the modern best practice), it is 200 renewals per year.

Each manual renewal requires: generating a CSR on the jump host, submitting it to the IssuingCA web enrollment (WIN-VCF-03 procedure), downloading the signed certificate, updating the Kubernetes Secret, restarting the affected pods. Conservatively 15 minutes per certificate. 200 renewals × 15 minutes = 50 hours of manual work per year. And that is if nothing goes wrong, no one is on leave, and no calendar reminder is ever missed.

The failure mode that makes this unacceptable for banking: a certificate that expires unnoticed causes an immediate TLS handshake failure. If the payment processing service's mTLS certificate expires, calls from the API gateway to the payment service fail with x509: certificate has expired or is not yet valid. No payment can be processed until the certificate is manually renewed. In banking, this is not a technical incident — it is a service outage with potential regulatory consequences.

cert-manager eliminates this class of failure entirely. Once a Certificate object is created, the renewal happens automatically, 30 days before expiry, without human intervention. The only human action is the initial object creation — after that, cert-manager owns the lifecycle.

OCP-OPS-02 · Section 02
How cert-manager works — the complete lifecycle from object to running pod
Following one certificate from the moment its Certificate object is created through to the moment it is served by a running pod — and through its first automatic renewal — makes every subsequent configuration decision obvious.
The complete certificate journey — what happens inside cert-manager
When I create a Certificate object, what happens between that moment and the certificate appearing as a Kubernetes Secret?
cert-manager's controller watches for Certificate objects. When it sees a new one, it creates a CertificateRequest object, fulfils it by communicating with the configured Issuer, and stores the result as a Kubernetes TLS Secret. The entire process is typically complete in under 10 seconds for an internal ADCS issuer.

The exact sequence in NovaCrestBank's environment:

1. An engineer creates a Certificate object: "I want a cert for payment-service.banking-apps.svc.cluster.local, signed by novacrestbank-issuer, stored as Secret payment-service-tls, valid for 1 year."
2. cert-manager's controller detects the new Certificate and checks whether the named Secret already exists with a valid, non-expiring certificate. It does not — no Secret exists yet.
3. cert-manager generates a private key and a CSR for the requested hostname.
4. cert-manager creates a CertificateRequest object containing the CSR and a reference to the ClusterIssuer.
5. The ADCS issuer plugin reads the CertificateRequest, authenticates to issuingca.novacrestbank.local using the stored credential Secret, and submits the CSR to the IssuingCA's web enrollment endpoint (/certsrv).
6. The IssuingCA signs the certificate using the NovaCrestBank-WebServer template and returns it.
7. cert-manager stores the signed certificate and private key together as a Kubernetes TLS Secret (payment-service-tls) in the correct namespace.
8. cert-manager updates the Certificate's .status.conditions to Ready: True with the expiry date and renewal time.
9. The pod that mounts payment-service-tls picks up the new Secret at its next start — or immediately if the application watches the Secret file path for changes.

The automatic renewal cycle

Day 0
Certificate issued — Secret created, cert valid for 365 days. cert-manager calculates the renewal time: 30 days before expiry (day 335). A timer is set.
Day 335
Renewal triggered automatically — cert-manager detects the certificate enters its renewal window (30 days before expiry). It creates a new CertificateRequest and submits to IssuingCA. No human action required.
Day 335 + 10s
New certificate issued by IssuingCA — The ADCS issuer returns the newly signed certificate. cert-manager updates the Secret in place — the old cert/key pair is replaced with the new one atomically.
Day 335 + ~60s
Application picks up new certificate — Pods mounting the Secret as a volume see the updated file within the kubelet sync period. Applications that hot-reload detect the change and reload TLS without restarting. Applications that read cert only at startup need a rolling restart via Reloader annotation.
Day 365
Old certificate expires — irrelevant — Renewal already happened 30 days ago. The running certificate is valid for another 335 days from day 335. TLS connections continue without interruption throughout.
Why 30 days before expiry — the reasoning behind the renewal window
The 30-day default renewal window exists because certificate operations can fail — the IssuingCA might be temporarily unreachable, a network policy might block the ADCS call, the credential password might have expired. A 30-day window gives cert-manager 30 attempts (it retries on exponential backoff) to complete the renewal before the certificate actually expires. Even if the IssuingCA is down for a week, the renewal will succeed with 3 weeks to spare. For short-lived certificates (90 days), cert-manager uses 2/3 of the certificate's lifetime as the renewal window, not a fixed 30 days — it adjusts automatically based on spec.renewBefore.
OCP-OPS-02 · Section 03
The four CRDs — ClusterIssuer, Issuer, Certificate, and CertificateRequest
cert-manager installs four CRDs. Each represents a distinct concept. Understanding all four before touching the cluster means every configuration choice in the sections that follow has a clear reason.
1
ClusterIssuer
What it is: A cluster-scoped connection to a certificate authority. Cluster-scoped means it is not in any namespace — any Certificate object in any namespace can reference it. For NovaCrestBank, one ClusterIssuer named novacrestbank-issuer connects to NovaCrestBank-IssuingCA. All namespaces — banking-apps, monitoring, logging, gitops — use this same ClusterIssuer to issue certificates from the internal PKI.

When to use ClusterIssuer vs Issuer: Use ClusterIssuer when the same CA serves the entire cluster. Use a namespace-scoped Issuer only when different namespaces need certificates from different CAs. NovaCrestBank uses only ClusterIssuers — one PKI for the entire cluster.
2
Issuer
What it is: A namespace-scoped connection to a CA — identical to ClusterIssuer in every way except it only serves Certificates in the same namespace. A Certificate in namespace banking-apps can only reference an Issuer in namespace banking-apps, never one in monitoring.

NovaCrestBank usage: Not used at the cluster level — all Issuers are ClusterIssuers. However, cert-manager internally creates one short-lived bootstrap Issuer per namespace when processing ADCS requests — this is an internal implementation detail that does not appear in normal operations.
3
Certificate
What it is: The declaration of a desired certificate — what hostname(s) it covers, how long it should be valid, when to renew it, which Issuer to use, and which Kubernetes Secret to store the result in. This is the primary object that engineers and application teams create. Once created, cert-manager owns the lifecycle — issuance, monitoring, and renewal — without further human involvement.

Key fields: spec.secretName (where the cert+key pair is stored as a TLS Secret), spec.dnsNames (the SANs the certificate covers), spec.issuerRef (which ClusterIssuer to use), spec.duration (how long the certificate is valid), spec.renewBefore (how far in advance to renew).
4
CertificateRequest
What it is: An intermediate object created automatically by cert-manager — never created manually by engineers. When cert-manager needs to issue or renew a Certificate, it creates a CertificateRequest containing the CSR and a reference to the Issuer. The Issuer reads the CertificateRequest, obtains the signed certificate from the CA, and marks it Approved and Issued.

Why it matters to engineers: When a Certificate is stuck — Ready: False — the first diagnostic step is to look at the CertificateRequest objects in the same namespace. The CertificateRequest's status conditions contain the specific error message from the CA or from the issuer plugin. This is where you find out why a certificate failed to issue: wrong template name, wrong credentials, CA unreachable, DNS name not permitted by CA policy.
What you create vs what cert-manager creates on your behalf
As an engineer you create two types of objects: ClusterIssuer (once, during setup) and Certificate (once per service that needs a certificate). cert-manager creates CertificateRequest automatically when processing a Certificate, and creates the Kubernetes TLS Secret automatically when the CA returns the signed certificate. You never create CertificateRequests or manage TLS Secrets manually once cert-manager is configured.

  targetNamespaces: []

    # cert-manager must watch ALL namespaces because Certificate objects
    # can be created in any namespace — banking-apps, monitoring, logging, etc.
    # A namespace-scoped OperatorGroup prevents cert-manager from reconciling
    # Certificates outside its own namespace, breaking all other teams.
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: cert-manager
  namespace: cert-manager
spec:
  channel: stable
    # stable: production-grade channel — updates are tested before appearing here.
  name: cert-manager
    # Package name exactly as it appears in the catalog packagemanifest.
  source: redhat-operators
    # The Harbor-mirrored CatalogSource from Registry-01.
    # Points at harbor.apps.ocp.novacrestbank.local/platform — not the internet.
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual
    # Manual: OLM creates an InstallPlan but does NOT apply it until an admin
    # explicitly approves it. NovaCrestBank's standard for all operators.
    # Every install and update is a reviewed, approved change.
    # Automatic would install new versions immediately — violates
    # NovaCrestBank's change management policy (PCI-DSS Req 6.4).

OCP-OPS-02 · Section 07
How cert-manager connects to Windows ADCS — the integration architecture
cert-manager has built-in support for ACME (Let's Encrypt), Vault, and Venafi. Active Directory Certificate Services — what NovaCrestBank uses — requires the ADCS issuer plugin. Understanding how it works explains every decision in the ClusterIssuer configuration.
What ADCS is and how the plugin bridges cert-manager to it
cert-manager has built-in issuers for Vault and ACME. Why does NovaCrestBank need a separate ADCS issuer plugin, and how does it work?
cert-manager's built-in issuers speak ACME, CFSSL, Vault, and Venafi. None of them speak the Microsoft ADCS protocol. NovaCrestBank's PKI runs on Windows Server ADCS — the same infrastructure used in WIN-VCF-03. ADCS exposes certificate enrollment through its web enrollment interface (/certsrv). The ADCS issuer plugin is a separate controller that bridges cert-manager's CertificateRequest objects to the ADCS web enrollment interface.

When the ADCS issuer plugin receives a CertificateRequest, it does exactly what a human engineer did in WIN-VCF-03: it authenticates to https://issuingca.novacrestbank.local/certsrv using stored credentials, submits the CSR using the NovaCrestBank-WebServer certificate template, receives the signed certificate, and returns it to cert-manager. The plugin runs as a separate Deployment alongside the main cert-manager controller — they communicate via the CertificateRequest API.

The credential model: The ADCS plugin needs to authenticate to the IssuingCA web enrollment. It reads credentials from a Kubernetes Secret in the cert-manager namespace. This Secret contains the username and password for svc-certmanager-enroll — a dedicated AD service account with permission to enroll certificates from the IssuingCA and no other rights.

OCP-OPS-02 · Section 09
Create the ClusterIssuer — every field explained
The ClusterIssuer connects cert-manager to NovaCrestBank-IssuingCA. Once this object exists and is Ready, any Certificate object in any namespace can reference it to obtain an IssuingCA-signed certificate.
Why one ClusterIssuer for the entire cluster — not one per namespace
Should NovaCrestBank have a separate Issuer per namespace, or one ClusterIssuer for everything?
One ClusterIssuer is the correct architecture for NovaCrestBank because the entire cluster uses the same PKI — every service in every namespace should be issued a certificate from NovaCrestBank-IssuingCA. A namespace-scoped Issuer must be created in every namespace that needs certificates, which creates management overhead and the risk of forgetting to create the Issuer before deploying a service that needs a certificate.

A namespace-scoped Issuer is appropriate when different teams own different CAs. NovaCrestBank has a single, unified PKI. One ClusterIssuer is correct.
ClusterIssuer — every field explained
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
# ClusterIssuer is cluster-scoped — no namespace field.
# Any Certificate in any namespace can reference this issuer.
metadata:
  name: novacrestbank-issuer
    # The name all Certificate objects reference in spec.issuerRef.name.
    # Choose a stable, meaningful name — this string appears in every
    # Certificate YAML across the entire cluster. Changing it requires
    # updating every Certificate object that references it.
spec:
  adcs:
    # adcs: instructs cert-manager to use the ADCS issuer plugin.
    # This key is registered by the ADCS issuer plugin's CRD installation.
    # If the ADCS plugin pod is not running, this issuer stays NotReady.
    caBundle: BASE64_ENCODED_CHAIN_PEM
        # The CA bundle used to verify the IssuingCA's HTTPS certificate.
          # chain.pem (IssuingCA + RootCA) base64-encoded.
          # Without this, the ADCS plugin rejects the TLS connection to /certsrv
          # with "x509: certificate signed by unknown authority".
          # Generate: base64 -w0 ~/ocp-kit/certs/chain.pem
    credentialsRef:
      name: adcs-credentials
        # References the Secret created in Section 08.
          # Must be in the cert-manager namespace.
    statusCheckInterval: 6h
      # How often the plugin polls the IssuingCA for certificate status.
      # ADCS sometimes processes requests asynchronously — the plugin submits
      # the CSR and polls until the certificate is issued. 6h avoids flooding.
    retryInterval: 1h
      # How long to wait before retrying a failed request.
      # If the IssuingCA is temporarily unreachable, the plugin retries every 1h —
      # well within the 30-day renewal window.
    url: https://issuingca.novacrestbank.local/certsrv
      # The IssuingCA web enrollment URL.
      # Must use HTTPS — credentials are sent in the request body.
      # Must be reachable from inside the cluster. Test from a pod if uncertain:
      # oc run -it --rm curl-test --image=curlimages/curl -- \
      #   curl -sk https://issuingca.novacrestbank.local/certsrv/
    templateName: NovaCrestBank-WebServer
      # The ADCS certificate template used for all requests.
      # Must exist on the IssuingCA and be configured to:
      # - Allow SANs (Subject Alternative Names)
      # - Allow the svc-certmanager-enroll account to enroll
      # - Issue certs with: Digital Signature, Key Encipherment,
      #   TLS Web Server Auth, TLS Web Client Auth key usages
      # Name must match exactly the template display name in the CA console — case-sensitive.

The Certificate CRD — every field explained with its architectural reason
The Certificate object is what engineers and application teams create when they need a certificate. Every field has a purpose and a reason — understanding all of them means you can write a Certificate for any service without guessing.
Certificate — fully annotated reference YAML
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: payment-service-tls
    # Convention: {service-name}-tls
  namespace: banking-apps
    # The namespace where the TLS Secret will be created.
    # The Secret can only be used by pods in this namespace.
    # Certificate must be in the same namespace as the pods that will mount it.
spec:
  secretName: payment-service-tls
    # The Kubernetes TLS Secret cert-manager creates and manages.
    # Contains two keys:
    #   tls.crt  the signed certificate (PEM)
    #   tls.key  the private key (PEM)
    # Pods reference this Secret in their volumeMounts.
    # Convention: same name as the Certificate object for clarity.
    # cert-manager OWNS this Secret — do not manually edit it.

  commonName: payment-service.banking-apps.svc.cluster.local
    # The CN (Common Name) for the certificate subject.
    # For internal cluster services: always use the full Kubernetes Service FQDN.
    # This ensures validity for direct service-to-service calls within the cluster
    # regardless of which namespace the caller is in.

  dnsNames:
    - payment-service
          # Short name — valid when caller is in the same namespace
    - payment-service.banking-apps
          # Namespace-qualified — valid cross-namespace in short form
    - payment-service.banking-apps.svc
    - payment-service.banking-apps.svc.cluster.local
          # Full FQDN — always works regardless of calling context.
              # Best practice: include all four forms so the cert is valid
              # regardless of how DNS resolves the name in different callers.

  usages:
    - digital signature
    - key encipherment
    - server auth
      # server auth: payment-service presents this cert TO callers
      # required for any TLS server
    - client auth
      # client auth: payment-service presents this cert WHEN CALLING other services.
      # Required for mutual TLS (mTLS).
      # A cert with only server auth cannot be used as a client cert —
      # the handshake fails with "certificate not authorised for client auth".
      # For services that only receive calls (never initiate), client auth is optional.
      # For banking microservices using mTLS: always include both.

  duration: 8760h
    # 8760h = 365 days = 1 year.
    # Shorter lifetimes (90 days) are better security practice.
    # cert-manager makes 90-day certs operationally equivalent to 1-year
    # because renewal is automatic. NovaCrestBank uses 1-year to match
    # the IssuingCA template default and reduce ADCS enrollment volume.

  renewBefore: 720h
    # 720h = 30 days.
    # cert-manager triggers renewal 30 days before expiry.
    # Provides 30 days of buffer if the IssuingCA is temporarily unreachable.
    # Rule of thumb: at least 1/4 of the duration. For 365d, 30 days is safe.

  privateKey:
    algorithm: RSA
    size: 2048
        # RSA-2048. NovaCrestBank-IssuingCA signs RSA-2048 keys.
          # The private key is generated by cert-manager on the OCP cluster —
          # it NEVER leaves the cluster. Only the CSR (public key) is sent to the CA.
    rotationPolicy: Always
        # Always: generate a NEW private key on every renewal.
          # A compromised old key provides no value after renewal when a new
          # key is generated. Never reuse private keys across certificate lifetimes.
          # "Never" would reuse the same key — do not use that setting.

  issuerRef:
    name: novacrestbank-issuer
      # Must exactly match the ClusterIssuer name from Section 09.
    kind: ClusterIssuer
      # ClusterIssuer (cluster-scoped).
      # If this were "Issuer", cert-manager would look for a namespace-scoped
      # Issuer in the banking-apps namespace — not find novacrestbank-issuer —
      # and the Certificate would stay in Pending indefinitely.
    group: cert-manager.io
      # Required for external issuers. cert-manager.io is correct for ClusterIssuer.

OCP-OPS-02 · Section 13
Automatic certificates via Ingress annotation — developer self-service HTTPS
The Ingress annotation is the implicit way to request a certificate. Add one annotation to any OCP Route and cert-manager automatically creates and manages the Certificate object on your behalf. Development teams get HTTPS without needing to understand PKI or cert-manager.
Ingress annotation vs explicit Certificate object — when to use each
We just created a Certificate explicitly. What does the Ingress annotation add and when should each approach be used?
The Ingress annotation and the Certificate CRD are two interfaces to the same underlying mechanism — cert-manager watches both and creates the same TLS Secret either way. The choice is about who is responsible and what level of control is needed.

Use an explicit Certificate object when: you need specific SANs (multiple DNS names for service mesh routing), specific key usage extensions (mTLS requires client auth in addition to server auth), specific key algorithm or rotation policy, or the certificate is for an internal cluster service rather than an externally-accessible route. The Certificate CRD gives full control over every parameter.

Use the Ingress annotation when: the certificate is for an OCP Route exposing a single hostname, the default settings (RSA-2048, 1-year, server auth only) are acceptable, and the team creating the Route should not need PKI knowledge. The annotation is designed for developer self-service — a developer adds one annotation to their Route YAML and receives a valid IssuingCA-signed certificate, automatically renewed, with zero further effort.
OCP Route with cert-manager annotation — one annotation, automatic certificate
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: payment-portal
  namespace: banking-apps
  annotations:
    cert-manager.io/cluster-issuer: novacrestbank-issuer
        # THIS SINGLE ANNOTATION is all cert-manager needs.
          # cert-manager's route-shim watches for Routes with this annotation,
          # creates a Certificate object automatically with the Route's hostname
          # as the single SAN, and injects the resulting TLS certificate and key
          # directly into the Route's spec.tls fields.
          #
          # The ClusterIssuer name must match exactly — novacrestbank-issuer.
          #
          # Use cert-manager.io/cluster-issuer for ClusterIssuers (correct for NovaCrestBank).
          # Use cert-manager.io/issuer only for namespace-scoped Issuers.
spec:
  host: payment-portal.apps.ocp.novacrestbank.local
    # This hostname becomes the certificate's CN and SAN automatically.
  to:
    kind: Service
    name: payment-portal-svc
  tls:
    termination: edge
        # edge: TLS terminated at the OCP router (HAProxy on infra nodes).
          # cert-manager injects the cert into spec.tls.certificate
          # and spec.tls.key automatically after issuance.
          # Do not fill in these fields manually — cert-manager owns them.
    insecureEdgeTerminationPolicy: Redirect
        # Redirect HTTP → HTTPS. Never use "Allow" for banking applications —
          # Allow would serve the application over plain HTTP to any client
          # that does not follow the redirect.
OCP-OPS-02 · Section 14
Mutual TLS between banking microservices — why it is required and how cert-manager enables it
One-way TLS verifies the server's identity to the client. Mutual TLS adds the reverse — the server also verifies the client's identity. For banking microservices carrying cardholder data, mTLS is the correct baseline for internal service-to-service communication.
Why mTLS is necessary on top of NSX-05 DFW
The payment-service and API gateway communicate inside the cluster on a private network. NSX-05 DFW already restricts this traffic by IP and port. Why is mutual TLS also necessary?
NSX-05 DFW operates at the network layer — it allows or denies traffic between IP addresses and ports. It cannot verify the identity of the application making the connection. A compromised pod on a permitted source IP can call the payment-service and the DFW allows it, because the DFW sees only "permitted IP:port pair". mTLS operates at the application layer — every connection requires the caller to present a valid certificate signed by NovaCrestBank-IssuingCA. A compromised pod without a valid certificate cannot complete the TLS handshake regardless of what the DFW permits.

This is defence in depth: NSX-05 DFW is the network layer gate (necessary but not sufficient). mTLS is the application layer gate (adds cryptographic identity verification the network layer cannot provide). PCI-DSS Req 4.2.1 requires strong cryptography for all transmissions of cardholder data — in banking environments this extends to internal service communication that carries cardholder data.

With cert-manager, enabling mTLS for a service requires only two things: a Certificate object with both server auth and client auth usages, and the application configured to request the client certificate during the TLS handshake. cert-manager handles all certificate management complexity — the application team only mounts the TLS Secret and enables mTLS in their TLS server configuration.
mTLS Certificate — payment-service with client auth enabled for mutual TLS

      # the TLS library rejects it with "certificate not authorised for client auth".

How the application uses this certificate — the volume mount pattern
The Secret payment-service-mtls-tls is mounted as a volume into the pod at /etc/tls/. The pod sees two files: /etc/tls/tls.crt (the certificate) and /etc/tls/tls.key (the private key). The TLS server configuration points to these file paths. When cert-manager renews the certificate it updates the Secret in place — Kubernetes propagates the updated file content to all mounting pods within the kubelet sync period (~60 seconds). Applications with hot-reload TLS detect the file change and reload the certificate without restarting. Applications that read the cert only at startup need the Reloader annotation (Section 17).
OCP-OPS-02 · Section 15
How automatic renewal works — every step from trigger to running application
Renewal is the most operationally important feature of cert-manager. Understanding exactly how it works means you can diagnose a stuck renewal, know when to intervene, and explain to a PCI-DSS auditor how certificate validity is maintained without manual management.
1
Expiry detection
cert-manager continuously monitors every Certificate's status.notAfter field and calculates the renewal trigger time as notAfter - renewBefore. For a 365-day certificate with a 30-day renewBefore, the trigger fires on day 335. cert-manager uses a precise timer set to this exact moment — not a cron job that might fire hours late. The renewal begins exactly when needed.
2
New private key generated on the cluster
cert-manager generates a new RSA-2048 private key entirely on the OCP cluster. The private key never leaves the cluster — only the CSR (which contains the public key but not the private key) is sent to the IssuingCA. With rotationPolicy: Always, a completely new key pair is created on every renewal — a compromised old key provides zero value once the new certificate is issued.
3
CertificateRequest created and fulfilled
cert-manager creates a new CertificateRequest with the fresh CSR. The ADCS issuer plugin picks it up, authenticates to issuingca.novacrestbank.local using the adcs-credentials Secret, submits the CSR with the NovaCrestBank-WebServer template, receives the newly signed certificate, and marks the CertificateRequest Issued. The complete ADCS round-trip typically takes under 10 seconds.
4
TLS Secret updated atomically
cert-manager replaces the tls.crt and tls.key values in the Kubernetes Secret in a single atomic update operation. At no point is the Secret empty or missing. Any client reading the Secret at any moment during the renewal finds a valid, non-expired certificate — either the old one (before the update) or the new one (after). There is no gap.
5
Application picks up the new certificate
Pods mounting the Secret as a projected volume see the file content updated within the kubelet sync period (default: 60 seconds). Applications using file-based TLS with hot-reload (Go tls.Config.GetCertificate, nginx, HAProxy) detect the file change and reload without pod restart. Applications reading the cert only at startup require a rolling restart via the Reloader operator. cert-manager itself does not trigger any pod restarts — this is the application's responsibility.
OCP-OPS-02 · Section 16
Monitoring certificate expiry — Prometheus alerts for the operations team
cert-manager exposes Prometheus metrics natively. Two alert rules provide advance warning when certificates approach expiry — a safety net for the cases where automatic renewal has failed silently.
Why monitoring is still needed when renewal is automatic
cert-manager's automatic renewal is reliable but not infallible. It can fail silently if: the IssuingCA is unreachable for the entire 30-day renewal window, the ADCS issuer pod crashes and is not restarted, the adcs-credentials Secret is accidentally deleted, or the svc-certmanager-enroll account password expires in AD. In any of these cases, cert-manager keeps retrying but the certificate keeps approaching expiry. Without a Prometheus alert, the first indication of failure is TLS handshake errors in production. The 14-day warning alert fires while there is still time to diagnose and fix the root cause before any service is affected.
PrometheusRule — two-tier certificate expiry alerts
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: cert-manager-expiry-alerts
  namespace: cert-manager
  labels:
    role: alert-rules
      # The Prometheus Operator (part of the cluster monitoring stack) watches
      # for PrometheusRule objects with this label and loads them into Prometheus
      # automatically. No manual Prometheus configuration needed.
spec:
  groups:
    - name: cert-manager.certificates
      rules:

        - alert: CertificateExpiringSoon
          expr: |
            certmanager_certificate_expiration_timestamp_seconds - time() < 14 * 24 * 3600
            # Fires when any cert-manager Certificate expires within 14 days.
            # certmanager_certificate_expiration_timestamp_seconds is a Prometheus gauge
            # cert-manager exposes for EVERY Certificate object it manages.
            # "expiration_timestamp - now() < 14 days" = 14 days or less remain.
            #
            # This fires AFTER automatic renewal should have already succeeded
            # (renewal runs at day 335 = 30 days before expiry).
            # If this fires, the renewal has been failing for ~16 days and
            # needs immediate investigation.
          for: 1h
            # Must be consistently near-expiry for 1 hour before firing.
            # Prevents alert flapping during cert-manager pod restarts.
          labels: {severity: warning}
          annotations:
            summary: "Certificate {{ $labels.name }} in {{ $labels.namespace }} expires in {{ $value | humanizeDuration }}"
            description: "Automatic renewal has not succeeded. Check CertificateRequest status and ADCS issuer pod logs in the cert-manager namespace."
            runbook: "https://wiki.novacrestbank.local/runbooks/cert-manager-renewal-failure"

        - alert: CertificateExpiryCritical
          expr: |
            certmanager_certificate_expiration_timestamp_seconds - time() < 3 * 24 * 3600
            # Fires within 3 days of expiry — P1 incident trigger.
            # At this point manual intervention is required immediately.
            # Force re-issuance: delete the Certificate object and re-apply it.
            # cert-manager immediately creates a new CertificateRequest.
          for: 5m
          labels: {severity: critical}
          annotations:
            summary: "CRITICAL: Certificate {{ $labels.name }} expires in {{ $value | humanizeDuration }}"
            description: "TLS failure is imminent. Delete and recreate the Certificate object to force immediate re-issuance."
            runbook: "https://wiki.novacrestbank.local/runbooks/cert-emergency-renewal"
OCP-OPS-02 · Section 17
Zero-downtime rotation — how applications receive new certificates
cert-manager updates the TLS Secret atomically. Whether the running application picks up the new certificate without a pod restart depends entirely on how it reads its certificate. Three patterns — and NovaCrestBank's recommendation for each.

Pattern 1 — Hot-reload (recommended)

The application uses filesystem watches (inotify) to detect when the mounted certificate file changes and reloads the TLS configuration without interrupting existing connections. Supported by nginx, Go services using tls.Config.GetCertificate, and HAProxy. Zero downtime, zero intervention. The gold standard for banking microservices.

Pattern 2 — Reloader-triggered rolling restart

The application reads the certificate only at startup. The Stakater Reloader operator detects the Secret change and triggers a rolling pod restart. Zero downtime because rolling restarts maintain availability — old pods serve traffic until new pods are Ready. Adds a ~60-second delay between cert rotation and the new cert being served.

Pattern 3 — Manual restart (avoid in production)

The application reads the cert only at startup and there is no Reloader. cert-manager renews the Secret but the running pod continues serving the old certificate until manually restarted. The old certificate stays in service — which defeats the security benefit of short-lived certificates and could leave an expired cert in service if the team forgets to restart. Avoid this pattern. If an application cannot hot-reload, add the Reloader annotation.

jumphost — Reloader annotation for applications that cannot hot-reload
# Add to any Deployment whose pods read the certificate only at startup
[svc-ocp@jumphost ~]$ oc annotate deployment payment-service \
  -n banking-apps \
  reloader.stakater.com/auto="true"

    # Reloader detects the Secret change and triggers:
    #   oc rollout restart deployment/payment-service -n banking-apps
    # Rolling restart: pods replaced one at a time — no payment-service downtime.
