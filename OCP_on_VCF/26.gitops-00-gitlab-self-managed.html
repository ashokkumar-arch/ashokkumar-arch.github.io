GitLab self-managed vs Bitbucket Data Center vs GitHub Enterprise Server
Three enterprise Git platforms exist for on-premises deployment: GitLab self-managed, Atlassian Bitbucket Data Center, and GitHub Enterprise Server. Which is correct for NovaCrestBank's OCP-centric environment and why?
GitLab self-managed is the correct choice. The decision rests on four factors that are specific to NovaCrestBank's architecture and operating model.

1. Kubernetes-native deployment via an Operator. GitLab publishes an official GitLab Operator for OCP, installable via OLM. The Operator manages GitLab's lifecycle — upgrades, configuration changes, scaling — as Kubernetes-native objects (GitLab CR). Bitbucket Data Center is a Java application packaged as a Helm chart with no OCP Operator and requires manual lifecycle management. GitHub Enterprise Server runs only as a dedicated VM appliance, not as an OCP workload at all. GitLab is the only choice deployable with OCP's operator-driven lifecycle model, which keeps it consistent with Harbor, cert-manager, ACS, and every other component in this architecture.

2. Air-gapped installation support. GitLab publishes its complete component image list for each release — every image can be mirrored to Harbor in advance. The GitLab Operator respects OCP's ImageContentSourcePolicy for image mirrors. Bitbucket and GitHub Enterprise have less mature air-gapped installation documentation and do not integrate with OCP's mirror infrastructure.

3. Single-platform CI/CD consolidation. GitLab includes GitLab CI (used in development/staging environments for lighter workloads) alongside the webhook integration with Jenkins that NovaCrestBank uses for production pipeline orchestration. One platform provides both. Bitbucket requires a separate CI platform (Bamboo, Jenkins) without native parity.

4. AD/LDAP integration maturity. GitLab's LDAP integration for Active Directory is the most mature of the three — it supports group sync (AD groups map to GitLab groups automatically), LDAP-based access control on repositories, and the same bind DN / search base pattern used by OCP OAuth and Harbor in this series. The LDAP configuration in Section 09 follows the identical pattern as Harbor Registry-01 Section 11 — no new credentials, no new service accounts.

What GitLab replaces and what it adds to NovaCrestBank's platform

Before GitLab

  • No internal Git server — developers would need to use GitHub.com or a shared NFS filesystem with bare Git repos
  • No merge request / code review workflow
  • No protected branch enforcement — anyone can push directly to main
  • No webhook source for Jenkins build triggers
  • Argo CD config repos have no access control beyond filesystem permissions
  • No audit trail for "who committed what to which repo when"

After GitLab

  • Internal Git hosting at gitlab.corp.novacrestbank.com — no traffic leaves the network
  • Merge request workflow with mandatory peer review and approval gates
  • Protected branches — direct push to main blocked; all changes require MR
  • Webhooks trigger Jenkins pipeline on every push or MR merge
  • Argo CD reads GitOps config repos via deploy key (read-only, scoped per repo)
  • GitLab audit events satisfy PCI-DSS Req 10 for source code change management
GITOPS-00 · Section 02
GitLab architecture on OCP — components and storage planning
GitLab is not a single container — it is a distributed application composed of multiple services. The GitLab Operator deploys and manages all of them. Understanding each component's storage requirements before deploying prevents the most common post-installation problem: running out of storage on a critical GitLab component.
Web
gitlab-webservice
The main GitLab Rails application — serves the web UI, REST API, and Git over HTTPS. Stateless — multiple replicas can run simultaneously. NovaCrestBank runs 2 replicas for availability. No persistent storage required (state is in PostgreSQL and Gitaly). The OCP Route points here. All browser traffic goes to webservice, which coordinates with other components for data.
Git
gitaly
The Git repository storage service. Every Git operation (clone, push, fetch) is handled by Gitaly, which manages the actual repository files on disk. This is GitLab's most important persistent storage component. NovaCrestBank stores all source code here. StorageClass: vsan-gold (high-durability RAID-6 FTT:2) — source code is production data, not reconstructible. PVC size: 200 Gi initial allocation, expandable.
DB
postgresql
GitLab's relational database. Stores: users, groups, projects, merge requests, issues, pipeline records, audit events, and all GitLab metadata. Second most critical component — losing the database loses all GitLab metadata (repositories survive in Gitaly, but project membership, MR history, audit logs, and settings are in PostgreSQL). StorageClass: vsan-gold. PVC size: 50 Gi.
Cache
redis
In-memory cache and session store. Used for: web session state, Sidekiq job queues, rate limiting counters, and temporary data. Losing Redis causes: session logout for all users, Sidekiq job queue drain, and temporary degraded performance until queues refill. Recoverable within minutes — not a critical data store. StorageClass: vsan-standard. PVC size: 10 Gi.
Worker
sidekiq
GitLab's background job processor. Handles: email notifications, CI/CD pipeline triggers, group membership sync (including AD/LDAP group sync), repository housekeeping, and webhook delivery. If Sidekiq is unhealthy, webhooks stop firing (Jenkins builds stop triggering) and LDAP group sync stops. No persistent storage — processes jobs from Redis queues.
Obj
MinIO (object storage)
GitLab uses object storage for: CI/CD artifacts, container registry layers, LFS (Large File Storage) objects, backups, and uploaded files. NovaCrestBank reuses the same MinIO instance used by LokiStack (OBS-03) and Tempo (OBS-04) — a separate bucket (ncb-gitlab) is created. This avoids deploying a second object storage service and leverages existing MinIO capacity on vSAN.

GITOPS-00 · Section 03
Domain and naming — moving to corp.novacrestbank.com
Phase 9 introduces the domain change from novacrestbank.local to corp.novacrestbank.com. This section explains the architectural reason for the change and the naming conventions that apply from this guide forward.
Why corp.novacrestbank.com — the Microsoft recommendation and its banking implications
The earlier guides in this series used novacrestbank.local for the internal AD domain and service hostnames. Why move to corp.novacrestbank.com for Phase 9?
The .local suffix conflicts with mDNS (Multicast DNS / Bonjour, RFC 6762), which reserves the .local domain for link-local service discovery. In modern enterprise environments, three specific problems arise from using .local for Active Directory.

1. mDNS resolution conflicts. macOS and Linux workstations using Bonjour/Avahi will attempt mDNS resolution before DNS for any .local hostname. A developer's laptop trying to reach gitlab.novacrestbank.local may send mDNS queries that never reach the corporate DNS server, resulting in intermittent resolution failures. This is not a theoretical problem — it causes real support tickets in enterprise environments where engineers use macOS.

2. TLS certificate complications. Public CAs will not issue certificates for .local domain names (per CA/Browser Forum Baseline Requirements since 2015). NovaCrestBank's internal CA (NovaCrestBank-IssuingCA) can still issue them, but if NovaCrestBank ever needs a publicly trusted certificate for any internal service (for regulatory reporting tools, for partner access, for auditor-facing portals), .local certificates are categorically invalid.

3. Microsoft's own recommendation. Microsoft has deprecated .local for Active Directory since Windows Server 2008 documentation and formally recommends a subdomain of a registered public domain — corp.novacrestbank.com where novacrestbank.com is NovaCrestBank's registered public domain. The corp subdomain is the most common enterprise convention (alternatives: internal, ad, intranet).

Practical approach for this guide series: Guides 1 through UPG-03 used novacrestbank.local — those guides are internally consistent and reflect real historical enterprise practice. Phase 9 onward uses corp.novacrestbank.com — the modern correct pattern. When the portfolio website references the series, both conventions are acknowledged as deliberate: the classic guides use the convention that was standard when those environments were built; the GitOps guides reflect current best practice.

Install the GitLab Operator via OLM
The GitLab Operator is installed via OLM from the Red Hat Certified catalog. In NovaCrestBank's air-gapped environment, the GitLab Operator image and all GitLab component images were mirrored to Harbor during the oc-mirror run that precedes this installation.
GitLab Operator Subscription — OLM installation
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: gitlab-operator-kubernetes
  namespace: gitlab
    # GitLab Operator installs into the application namespace (gitlab),
    # not into openshift-operators. This is the namespace-scoped operator pattern —
    # the operator only manages GitLab instances in the same namespace.
    # This is safer for a banking environment: the operator cannot accidentally
    # affect other namespaces.
spec:
  channel: stable
  name: gitlab-operator-kubernetes
  source: certified-operators
    # certified-operators: the Red Hat-certified operator catalog.
    # GitLab is in the certified catalog (not community-operators) —
    # it has been validated against OpenShift compatibility requirements.
    # This is the appropriate catalog for a production banking environment.
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual
    # Manual: GitLab operator updates can change the managed GitLab configuration.
    # Every update requires review and approval — the same policy as all other
    # OLM operators in this environment.

GITOPS-00 · Section 06
The GitLab CR — every field explained
The GitLab Custom Resource is the single object that declares the entire GitLab deployment — version, storage, object storage, TLS, ingress, and initial configuration. The GitLab Operator reads this CR and reconciles all GitLab component Deployments and StatefulSets to match.
GitLab CR — NovaCrestBank GitLab instance — every field explained
apiVersion: apps.gitlab.com/v1beta1
kind: GitLab
metadata:
  name: gitlab
  namespace: gitlab
spec:
  chart:
    version: 8.x.x
      # GitLab Helm chart version. The Operator maps chart versions to GitLab
        # application versions. Chart 8.x corresponds to GitLab 17.x.
        # Check the GitLab Operator compatibility matrix before specifying.
    values:
      global:
        hosts:
          domain: corp.novacrestbank.com
            # The base domain. GitLab generates subdomains from this:
                # gitlab.corp.novacrestbank.com  (main GitLab UI)
                # registry.corp.novacrestbank.com  (GitLab container registry — NOT used;
                #   Harbor is the container registry. Disable below.)
                # pages.corp.novacrestbank.com  (GitLab Pages — disabled for banking)
          https: true
          gitlab:
            name: gitlab.corp.novacrestbank.com
          registry:
            enabled: false
              # false: disable the built-in GitLab container registry.
                  # NovaCrestBank uses Harbor (Registry-01) as the authoritative
                  # container registry. Running two registries creates confusion
                  # about which is the source of truth for images.
                  # All pipeline builds push to Harbor, not GitLab's registry.

        ingress:
          class: openshift-default
          annotations:
            cert-manager.io/cluster-issuer: novacrestbank-issuer
              # cert-manager will automatically provision a TLS certificate
                  # for gitlab.corp.novacrestbank.com from the NovaCrestBank-IssuingCA.
                  # The same pattern used for Harbor, Argo CD, and Grafana.

        edition: ee
          # ee: Enterprise Edition. Required for:
            # - LDAP group synchronisation (maps AD groups to GitLab groups automatically)
            # - Branch protection with multiple approval rules
            # - Audit event streaming (export to Splunk)
            # - Compliance framework enforcement on repositories
            # NovaCrestBank holds a GitLab EE self-managed subscription.

        time_zone: Australia/Sydney
          # Timezone for GitLab's UI timestamps, scheduled jobs, and LDAP sync schedule.
            # NovaCrestBank headquarters: Sydney. Audit events are stored in UTC
            # internally but displayed in AEST/AEDT in the UI.

        psql:
          password:
            secret: gitlab-postgresql-password
            key: postgresql-password

        redis:
          auth:
            enabled: true
            secret: gitlab-redis-secret
            key: redis-password

        object_store:
          enabled: true
          proxy_download: true
            # proxy_download: GitLab webservice proxies object storage downloads
                # through itself rather than redirecting clients to MinIO directly.
                # This is required in an air-gapped environment — developer workstations
                # cannot reach MinIO directly (it is cluster-internal).
                # GitLab retrieves the object from MinIO and streams it to the client.
          connection:
            secret: gitlab-minio-creds
            key: connection

        initialRootPassword:
          secret: gitlab-initial-root-password
          key: password
            # The initial GitLab root (admin) password. Created as a Secret before
                # deploying. After initial setup and LDAP integration is verified,
                # the root account is disabled — all access goes through AD accounts.
                # The root password is stored in CyberArk for emergency break-glass access.

      gitlab:
        webservice:
          replicaCount: 2
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
        sidekiq:
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
        gitaly:
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
          persistence:
            storageClass: vsan-gold
            size: 200Gi
              # vsan-gold: RAID-6 FTT:2. Source code is irreplaceable production data.
                  # 200 Gi: initial allocation for all banking microservice repos,
                  # GitOps config repos, infrastructure-as-code, and runbook sources.
                  # vSAN storage is expandable — start with 200 Gi and monitor growth.

      postgresql:
        primary:
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
          persistence:
            storageClass: vsan-gold
            size: 50Gi

      redis:
        master:
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}
          persistence:
            storageClass: vsan-standard
            size: 10Gi

GITOPS-00 · Section 09
AD/LDAP authentication — the same pattern as OCP OAuth and Harbor
GitLab's LDAP configuration uses the same Active Directory bind account, LDAP base DN, and group search structure used by OCP OAuth and Harbor. No new service accounts are required — svc-ocp-ldap already has the minimum read permissions needed for all three systems.
Why LDAP and not SAML or OIDC for GitLab authentication
NovaCrestBank's Active Directory does not have ADFS (Active Directory Federation Services) configured, which is required for SAML and OIDC federation. LDAP bind authentication is the direct AD integration path that works with any AD deployment without additional infrastructure. The same svc-ocp-ldap bind account that serves OCP OAuth and Harbor now also serves GitLab — three systems, one service account, one set of LDAP policies to manage. If ADFS is added to the environment in future, GitLab can be migrated to SAML without losing user accounts (GitLab supports both simultaneously during a migration window).

Group and project structure — banking and gitops groups
GitLab's group hierarchy organises repositories and applies access control at scale. NovaCrestBank uses two top-level groups that mirror the two-repository pattern from GitOps architecture: banking for application source code and gitops for deployment configuration.
Why two top-level groups — the access control separation
The banking source code and the GitOps config repos have different access control requirements. All banking developers need read-write access to application source code repos so they can commit features. But only senior engineers and the automated pipeline service account should write to the GitOps config repos — a developer committing directly to the production GitOps overlay would bypass the review process that satisfies PCI-DSS Req 6.4. By putting them in separate top-level groups, different membership policies apply: banking developers have Maintainer role in the banking group; they have only Reporter (read-only) in the gitops group. GITOPS-00 · Section 13
Deploy keys for Argo CD read access
Argo CD needs read access to the GitOps config repos to sync cluster state. A deploy key — an SSH key pair where the public key is registered in GitLab as a project-level read-only key — provides scoped access without creating a full GitLab user account for Argo CD.
Deploy key vs personal access token vs service account user for Argo CD access
Three options exist for giving Argo CD access to GitLab repos: an SSH deploy key, a personal access token, or a dedicated GitLab user account acting as a service account. Which is correct for a banking environment?
An SSH deploy key is correct. The three options have different security properties.

Personal access token (PAT): A token belonging to a specific user. If that user's account is disabled (employee leaves, password rotation), the token becomes invalid and Argo CD loses access to all repos. PATs expire and require manual rotation. A PAT has access to everything the user has access to — not scoped to specific repos. Never use a human user's PAT for automated systems.

Service account user: A dedicated GitLab account (e.g., ncb-argocd-svc) with a PAT. Better than a human user's PAT but still requires managing a GitLab account and its password/PAT lifecycle. The account consumes a GitLab EE license seat.

SSH deploy key (correct choice): An SSH key pair generated specifically for Argo CD. The public key is registered in GitLab as a deploy key on the specific repos Argo CD needs to read. It grants read-only access to those specific repos — nothing else. It does not expire unless deliberately rotated. It does not belong to a user account — disabling an AD account does not affect it. The private key is stored as a Kubernetes Secret in the gitops namespace (OPS-05 registered it in the Argo CD repository configuration). This is the minimum-privilege, lifecycle-independent approach.

GITOPS-00 · Section 16
Push rules and branch protection for PCI-DSS compliance
GitLab push rules enforce code quality and compliance requirements at the repository level — before code reaches a review. These rules satisfy PCI-DSS Req 6.3.3 (all security components protected from tampering) and contribute to the Req 6.4 change management evidence chain.
GitLab push rules — applied to all projects in the banking group via API
# Applied via: Settings → Repository → Push rules (UI)
# Or via API for all projects in the banking group

# Rule 1: Reject unsigned commits
reject_unsigned_commits: false
  # false: NovaCrestBank does not yet require GPG-signed commits.
  # This is a future hardening target — when enforced, every developer commit
  # must be GPG-signed with a key registered in their GitLab profile.
  # Set to true after the developer GPG key enrollment process is complete.

# Rule 2: Commit author email must be from NovaCrestBank domain
author_email_regex: ^[a-zA-Z0-9._%+\-]+@novacrestbank\.com$
  # Rejects commits where the author email is not an @novacrestbank.com address.
  # Prevents accidental commits made with personal email addresses configured
  # in developers' local git config. Ensures every commit is attributable to
  # a specific NovaCrestBank employee for audit purposes.

# Rule 3: No merge request may be merged without resolving all discussion threads
all_discussions_must_be_resolved_before_merge: true
  # A reviewer's comment on a merge request creates a discussion thread.
  # A thread can only be resolved by the reviewer who created it (or an admin).
  # This ensures reviewers' concerns are addressed — not silently skipped.

# Rule 4: Prevent file names that expose secrets or environment data
file_name_regex: (\.(env|pem|key|p12|pfx|jks)|application-production\.yaml)$
  # Reject pushes containing files matching these patterns:
  # .env files: typically contain environment variables including passwords
  # .pem, .key, .p12, .pfx, .jks: private keys and keystores
  # application-production.yaml: could contain hardcoded production credentials
  # This is a defence-in-depth measure — the primary secret protection is
  # External Secrets Operator (GITOPS-05), but file-level rejection catches
  # accidental secret commits before they reach the repo history.

# Rule 5: Prevent secrets in commit messages or file content (regex scan)
commit_message_negative_regex: (?i)(password|secret|token|credential)\s*[:=]
  # Reject commits where the commit MESSAGE itself contains patterns like
  # "password = abc123" or "token: xyz789".
  # Developers occasionally include credentials in commit messages when
  # documenting a "quick fix" — this rule catches it before it reaches the repo.

GitLab backup strategy on OCP
GitLab's data lives in two places: the PostgreSQL database (metadata, audit logs, user accounts) and Gitaly (the actual Git repositories). A complete backup captures both. In NovaCrestBank's OCP environment, GitLab's built-in backup tool is run as a Kubernetes CronJob.
GitLab backup CronJob — daily backup to MinIO
apiVersion: batch/v1
kind: CronJob
metadata:
  name: gitlab-backup
  namespace: gitlab
spec:
  schedule: "0 2 * * *"
    # 2:00 AM daily — low-traffic period for NovaCrestBank.
    # GitLab backup creates a brief maintenance mode — schedule outside business hours.
  concurrencyPolicy: Forbid
    # Forbid: do not start a new backup if the previous one is still running.
    # A GitLab backup with 200 Gi of repositories can take 30+ minutes.
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: gitlab-backup
              image: harbor.apps.ocp.corp.novacrestbank.com/platform/gitlab-toolbox:17.x
                # The gitlab-toolbox image contains the gitlab-backup-cli and rake tasks.
              command:
                - /bin/bash
                - -c
                - |
                    gitlab-backup create BACKUP=daily STRATEGY=copy 2>&1
                    echo "Exit code: $?"
                    # BACKUP=daily: prefix for the backup archive name
                      # STRATEGY=copy: copy files rather than streaming (safer for large repos)
                      # Output goes to the configured backup_path (MinIO ncb-gitlab bucket)
          restartPolicy: OnFailure
          nodeSelector: {node-role.kubernetes.io/infra: ""}
          tolerations:
            - {key: node-role.kubernetes.io/infra, effect: NoSchedule}

Backup destinationMinIO ncb-gitlab bucket — backups/ prefix
Backup retention30 days in MinIO — MinIO lifecycle rule deletes backups older than 30 days
What is backed upPostgreSQL database dump + all Gitaly repositories + uploads + CI artifacts
What is NOT in the backupTLS certificates (managed by cert-manager, re-issuable) · GitLab secrets/config (stored in OCP Secrets, backed up with etcd)
