The four properties of GitOps — and why each one matters specifically in banking
GitOps is described in many ways. What is the shortest precise definition, and what are the properties that make it appropriate for a regulated banking environment rather than just a convenient operational practice?
GitOps has one core rule and four derived properties. The core rule: the desired state of every deployed system is stored in Git, and an automated agent continuously reconciles the actual state of the system to match it. The four properties follow from this rule.

Property 1 — Declarative configuration. The entire state of NovaCrestBank's production environment — every Deployment, every Service, every ConfigMap, every NetworkPolicy — is declared in YAML files in a Git repository. To know what should be running in production right now, read the Git repository. This is auditable in a way that "run this sequence of commands" is not. A PCI-DSS assessor can read the Git repository and verify that the production environment matches what was documented in the change request. This is not possible in an imperative deployment model.

Property 2 — Versioned and immutable history. Every change to every deployed system is a Git commit with an author, a timestamp, and a diff. The Git history is the deployment history. This is the audit trail that PCI-DSS Req 10 and FFIEC examination require — not just "something changed at this time" but "this specific person changed this specific configuration value from X to Y at this exact time, and it was reviewed and approved by this other person." No other deployment model provides this level of audit detail without additional tooling.

Property 3 — Automated reconciliation. Argo CD continuously compares the cluster's actual state to the Git repository and corrects any drift — automatically for dev and staging environments, with human approval for production. If a developer runs kubectl edit in production and changes a resource limit, Argo CD detects the drift within 3 minutes and alerts (or reverts, depending on configuration). Configuration drift — a pervasive problem in environments without GitOps — becomes immediately visible and alertable. This satisfies the "continuous monitoring" requirement of NIST SP 800-53 and the configuration management controls in FFIEC CAT.

Property 4 — Separation of concerns between building and deploying. In NovaCrestBank's GitOps model, Jenkins builds the application image (CI) and Argo CD deploys it (CD). These are separate systems with separate credentials and separate concerns. Jenkins never has credentials to modify the production cluster. Argo CD never has credentials to push code. If Jenkins is compromised, the attacker gains the ability to build images — but the images must still pass through the GitOps promotion process (code review, ACS scan, merge request approval) before reaching production. This defence-in-depth separation directly addresses the supply chain attack vectors that PCI-DSS DSS v4 increasingly focuses on.
GITOPS-01 · Section 02
Git as the source of truth — what this means operationally
"Git is the source of truth" is the foundational statement of GitOps. It has a precise operational meaning that is different from "we store our configuration in Git." The difference determines whether a team is doing GitOps or just using Git as a documentation tool.

Git as documentation (not GitOps)

  • YAML files exist in a Git repo but are applied manually with kubectl apply -f
  • An engineer makes a change in the cluster directly; updates Git "to keep it in sync" later (or doesn't)
  • The cluster is the truth — Git is a backup or reference copy
  • Configuration drift is invisible until someone compares the cluster to the repo
  • Rollback means finding the old YAML and re-applying it
  • Audit trail: the Git commit history, if the engineer remembered to commit after the change

Git as source of truth (GitOps)

  • The Git repository is the authoritative definition of what should exist in the cluster
  • Changes to the cluster happen only via Git commits — no direct kubectl apply in production
  • Git is the truth — the cluster is reconciled to match Git, not the reverse
  • Configuration drift is detected automatically within minutes and creates an observable alert
  • Rollback is git revert — Argo CD deploys the reverted state automatically
  • Audit trail: every change is a commit with author, timestamp, reviewer, and diff — before the change is deployed
The operational test — "if the cluster were destroyed right now, what would you do?"
In a GitOps model: recreate the cluster, point Argo CD at the Git repository, and the cluster self-heals to the desired state automatically. Every Deployment, every Service, every NetworkPolicy, every ConfigMap is in Git. The cluster is reconstructible from Git. In a non-GitOps model: the cluster needs to be manually rebuilt from memory, runbooks, and partially-current YAML files that may not reflect what was actually running before the disaster. The difference in recovery time and reliability is measured in hours vs days. For NovaCrestBank's banking environment, "reconstructible from Git" means RTO (Recovery Time Objective) for the application tier is measured in tens of minutes rather than hours — a significant contribution to the disaster recovery posture that APRA CPS 234 and FFIEC BCP requirements mandate.
GITOPS-01 · Section 03
Pull-based deployment vs push-based deployment
GitOps uses a pull-based deployment model. The deployment agent (Argo CD) runs inside the cluster and pulls configuration from Git. This is architecturally opposite to push-based deployment (a CI system pushing changes into the cluster from outside). The distinction has specific security consequences for a banking environment.
Why pull-based deployment has a stronger security posture for banking
Jenkins could deploy to production directly by running kubectl apply after a successful build — this is the traditional CI/CD push model. Why does NovaCrestBank use a pull-based model (Argo CD inside the cluster pulling from Git) instead?
The push model creates a specific security risk that the pull model eliminates: external write access to the production cluster.

In a push-based model, Jenkins needs a kubeconfig or ServiceAccount token with sufficient permissions to create and modify resources in the production namespace. This credential must live outside the cluster — in Jenkins, in a CI/CD secrets vault, or in the pipeline itself. If Jenkins is compromised, the attacker inherits production cluster write access. If the credential leaks (in a pipeline log, in a Jenkinsfile committed to Git, in a Jenkins plugin vulnerability), an attacker gains production cluster access. The credential has to exist outside the cluster, which means it is a remote code execution away from being misused.

In the pull-based model: Argo CD runs inside the production cluster and has the necessary ServiceAccount permissions to apply manifests — but those permissions never leave the cluster. Jenkins has no credentials to write to the production cluster. Jenkins' credentials are: write to Harbor (push images), write to GitLab (commit image tag updates). Neither of these credentials gives Argo CD production cluster access. An attacker who compromises Jenkins can push a malicious image to Harbor — but that image must still pass through Argo CD's reconciliation (which checks that the image was signed with the cosign signing key — Section 16) and cannot reach production without a signed, reviewed merge request into the production GitOps config repo.

The network diagram difference: Push model — Jenkins has an outbound connection to the Kubernetes API server (external → internal, needs firewall rule). Pull model — Argo CD has an outbound connection to GitLab (internal → internal, stays inside the network). No external system needs firewall access to the Kubernetes API server. This is a meaningful reduction in the cluster's attack surface.

GITOPS-01 · Section 04
No manual kubectl in production — the operational discipline
The most difficult aspect of GitOps adoption is cultural, not technical. Engineers who have spent years running kubectl apply and oc edit must change their workflow. This section explains what the rule means, why it is necessary, and what the correct alternative is for every situation where an engineer would previously have reached for kubectl.
The rule — no manual kubectl apply, oc edit, or oc patch in any managed production namespace
Any resource in a namespace managed by Argo CD that is modified directly — via kubectl, oc, the OCP web console, or any other mechanism other than a Git commit — creates drift between the cluster and Git. Argo CD detects this drift within 3 minutes. If self-heal is enabled (dev and staging environments), Argo CD reverts the change. In production (where self-heal is disabled), the drift persists as an OutOfSync alert until it is either codified in Git or reverted via a sync. The drift creates an audit gap: the change happened outside Git and therefore outside the audit trail. For PCI-DSS, an untracked configuration change in a production namespace is a compliance finding.

The correct GitOps alternative for each traditional kubectl operation

Old workflow (direct kubectl)GitOps workflowWhy
kubectl apply -f new-deployment.yaml Add the file to the GitOps config repo, open a merge request, get it reviewed and merged, Argo CD deploys it Every new resource must have a review record and be in Git before it exists in production
oc edit deployment payment-service (change replica count) Update the replica count in the Kustomize overlay for the environment, commit, MR, merge The replica count is configuration — it belongs in Git. Argo CD would revert an oc edit within 3 minutes.
kubectl scale deployment payment-service --replicas=5 (emergency scale-up) During an incident: update the HorizontalPodAutoscaler min replicas in Git with an expedited MR (fast-track approval, not bypass). If immediate: update the overlay via Git directly with emergency approval. Even emergency changes need an audit record. An expedited MR takes 5 minutes and creates the required change evidence.

kubectl delete pod payment-service-abc123 (force restart a pod) This is the one exception: deleting a specific pod instance (not a Deployment or resource definition) is permitted. Argo CD manages Deployments, not individual pod instances — deleting a pod causes the Deployment to create a replacement, which is expected behaviour. Pod deletion is an operational action that does not change desired state — it triggers the existing desired state (replica count) to be satisfied with a new pod. No drift created.
GITOPS-01 · Section 05
Why two repositories — the problems a single repository creates
The two-repository pattern is one of the defining architectural choices of GitOps. Application source code lives in one repository; deployment configuration lives in another. This is not a matter of organisation preference — it solves three specific problems that arise when they are combined.
The three problems a monorepo creates for GitOps
Keeping source code and deployment configuration in the same repository is simpler — one repo, one history, one set of access controls. What specific problems does this create?
Each problem is a real operational failure mode, not a theoretical concern.

Problem 1 — The recursive trigger loop. When the pipeline builds a new image and commits the updated image tag into the config repo, that commit triggers the pipeline again (because the pipeline is triggered by commits to the repo). If source code and config are in the same repo, the pipeline triggers itself on every image tag update. Mitigations exist (ignore commits from the CI service account) but they are fragile and create subtle edge cases. The two-repo pattern eliminates the loop entirely: the pipeline triggers on commits to the source repo; image tag updates go to the config repo, which has no pipeline trigger.

Problem 2 — Different review requirements for different content. A feature branch in the source repo containing a new payment API endpoint needs a review from a developer peer who understands the payment logic. A change to the production overlay in the config repo (increasing replica count, updating a resource limit, promoting an image to production) needs a review from a senior platform engineer who understands the production environment. These are different audiences with different expertise. Protected branch rules, approval policies, and CODEOWNERS files need to express different requirements for each repo. A monorepo with one set of branch protection rules cannot express "this directory needs approval from this team and that directory needs approval from that team" cleanly. Two repos with two separate access control models makes this natural.

Problem 3 — The signal-to-noise problem in Git history. The source repo's history should tell the story of feature development: commits like "Add NPP payment rail adapter" or "Fix: handle timeout in fraud scoring". The config repo's history should tell the story of deployments: commits like "production: promote payment-service to v1.4.2@sha256:abc123" or "production: increase replicas to 5 for end-of-month load". These are different audit trails with different consumers. A PCI-DSS assessor reviewing change management evidence wants to read the config repo history — a sequence of deployment commits, each with a linked merge request showing approval. Mixing this with feature development commits from the source repo makes the compliance evidence harder to read and verify.

GITOPS-01 · Section 07
The GitOps config repository — the deployment record for the entire banking platform
The config repository at gitlab.corp.novacrestbank.com/gitops/banking-config is the single source of truth for what runs in every environment. Its Git history is the deployment audit trail. Its structure is designed to make the Kustomize base-plus-overlay pattern work cleanly across dev, staging, and production.
Why the config repo history IS the change management audit trail
PCI-DSS Req 6.4 requires documented, reviewed, tested, and approved changes for all production deployments. The config repo's Git history provides all four: documented (the commit shows exactly what changed in the YAML), reviewed (the merge request shows who reviewed it and what comments were made), tested (the merge request description links to the Jenkins build log and ACS scan result), approved (the merge request shows who approved it and when, timestamped and immutable). This is not a separate audit system on top of the deployment process — the Git history itself is the Req 6.4 evidence. When a PCI-DSS assessor asks "show me the change management records for the payment-service deployments in the last 90 days," the answer is: here is the config repo's git log filtered to the production overlay, with links to each merge request.

GITOPS-01 · Section 09
PCI-DSS Req 6.4 — change management through merge requests
PCI-DSS Requirement 6.4 mandates that all system components are protected from known vulnerabilities through documented, authorised change management processes. GitOps with the two-repository pattern satisfies this requirement structurally — not as an additional control bolted on, but as a consequence of how deployments work.
PCI-DSS Req 6.4.1

Security vulnerabilities are identified and addressed through a defined vulnerability management process.

GitOps satisfies this through: Every image pushed to Harbor passes through an ACS vulnerability scan in the Jenkins pipeline (GITOPS-03 Stage 6). The scan result is linked from the merge request that promotes the image to production. If the scan finds critical vulnerabilities, the pipeline fails and the MR is blocked from merging — the image cannot reach production.

PCI-DSS Req 6.4.2

All changes to system components in the production environment are managed and authorised through a formal change control process.

GitOps satisfies this through: Every production deployment is a merge request in the GitOps config repo. Merge requests are reviewed by at least one platform engineer (CODEOWNERS enforcement). The merge request approval is timestamped, attributed to a specific AD-authenticated user, and immutable in the GitLab audit log. This is the formal change control record.

PCI-DSS Req 6.4.3

All production changes include a defined rollback plan.

GitOps satisfies this through: The rollback plan for every GitOps deployment is implicit and identical: git revert the deployment commit in the config repo, merge the revert, Argo CD deploys the previous state automatically. No separate rollback plan document is needed because the mechanism is always the same and always available.

The merge request as change management evidence — what it contains

What changedThe diff in the MR shows exactly one changed line: the image digest reference in overlays/production/kustomization.yaml. The diff is the complete description of the change — there is no ambiguity about what will be deployed.
Why it changedThe MR description (written by the Jenkins pipeline automatically) includes: the source commit SHA that triggered the build, a link to the Jenkins build log, the ACS vulnerability scan result, and the integration test pass/fail status.
Who reviewed itThe GitLab MR approval section shows which platform engineer approved the MR, from which IP, at what exact time — immutable in the GitLab audit event log.
When it was deployedThe Argo CD sync history shows when the sync was triggered (automatically after merge) and which Git commit SHA it synced. Linked from the MR via Argo CD's GitLab integration.
Rollback evidenceIf a rollback occurred: a subsequent MR shows Revert "pipeline: promote payment-service to v1.4.2" — the rollback decision and its review are also in Git history.
GITOPS-01 · Section 10
PCI-DSS Req 10 — the complete audit trail from commit to running pod
PCI-DSS Requirement 10 mandates audit logs that record who did what, when, from where. GitOps creates a multi-layered audit trail — each layer captures a different phase of the deployment journey. Together they provide the most complete audit record possible for a software deployment.
Layer 1
GitLab source repo commit
Records: which developer committed which code change, when, to which branch. Every commit is attributed to an AD-authenticated user (GitLab requires @novacrestbank.com author email — enforced by push rule). The commit hash is cryptographically linked to the content — it cannot be altered without producing a different hash. Req 10 element satisfied: identity of user initiating the change, date/time of change, what was changed (diff).
Layer 2
Jenkins build log
Records: every stage of the build pipeline — test results, SonarQube scan, Buildah image build, cosign signing, Harbor push, ACS vulnerability scan verdict. The build log is stored in Jenkins and linked from the GitLab MR. Req 10 element satisfied: evidence that the change was tested and scanned before deployment, and that the image reaching production is exactly the output of this specific build.
Layer 3
GitLab config repo MR
Records: who approved the production promotion, when, the complete diff of what changed in the config repo, and all reviewer comments. Immutable in the GitLab audit event log (streamed to Splunk — GITOPS-00 Section 17). Req 10 element satisfied: authorisation of the change (reviewer approval), identity of the authorising person, time of authorisation, the exact change authorised.
Layer 4
Argo CD sync history
Records: when the sync was triggered, which Git commit SHA was applied, which resources were created or modified, and whether the sync succeeded. Accessible in the Argo CD UI and forwarded to Splunk. Req 10 element satisfied: date/time of deployment to production, the specific state deployed (commit SHA), success or failure of the deployment.
Layer 5
OCP API server audit log
Records: every Kubernetes API call that Argo CD made to create or modify resources — which ServiceAccount made the call, to which resource, at what time, with what result. Forwarded to Splunk via the CLF pipeline (OBS-03 Section 09). Req 10 element satisfied: system-level record of what changed in production infrastructure, independent of any application-level logging, tamper-evident (append-only Splunk index).
The combined audit trail answers every PCI-DSS Req 10 question
PCI-DSS Req 10 asks: who (Layer 1, 3), what (Layer 2, 3, 4), when (Layer 1, 3, 4, 5), from where (Layer 3, 5), what happened (Layer 4, 5). No single layer answers all questions. Together, the five layers provide a complete, correlated, tamper-evident record from developer keystroke to running pod — a record that no manual deployment process can produce as reliably or as completely.
GITOPS-01 · Section 11
NIST SI-2 — flaw remediation through controlled deployment
NIST SP 800-53 Control SI-2 (Flaw Remediation) requires organisations to identify, report, and correct information system flaws — including installing security-relevant software updates within organisation-defined timeframes. GitOps provides the mechanism for doing this in a controlled, auditable way.
How GitOps specifically satisfies SI-2 — the flaw remediation workflow
NovaCrestBank's security team receives a CVE advisory for a library used by the payment-service. NIST SI-2 requires a documented, controlled, timely remediation. How does the GitOps model provide this?
The GitOps flaw remediation workflow for a library CVE has four steps, each with audit evidence.

Step 1 — Detection. The ACS vulnerability scanner (running continuously against images in Harbor) detects that the current production image of payment-service uses a library version with the CVE. ACS creates an alert in the ACS console and the SOC's Splunk dashboard. The detection is timestamped — NIST SI-2 requires demonstrating that flaws are identified in a timely manner. The detection time is the Splunk alert timestamp.

Step 2 — Remediation in source code. A developer updates the dependency in pom.xml, creates a feature branch in the source repo, opens a merge request. The MR description references the CVE number (e.g., CVE-2026-12345). The Jenkins pipeline runs automatically, builds a new image with the updated library, runs the ACS scan — which now shows the CVE is resolved. The developer merges the MR.

Step 3 — Controlled promotion through environments. The Jenkins pipeline automatically promotes the new image to staging, runs integration tests, then opens the production promotion MR in the config repo. The MR description says "Security fix: CVE-2026-12345 remediation." A platform engineer reviews and approves. The production overlay is updated.

Step 4 — Audit evidence for NIST SI-2. The complete evidence chain: the ACS alert (detection), the source repo MR (remediation decision), the Jenkins build log (proof the fix was tested), the config repo production MR with approval (controlled deployment), the Argo CD sync (deployment confirmation). All timestamped, all in Splunk. NIST SI-2 assessment answer: "Here is the CVE, here is when it was detected, here is the approved change that remediated it, here is when it was deployed to production." Demonstrable in under 5 minutes from audit evidence alone.

GITOPS-01 · Section 14
Jenkins' role — build, test, sign, and hand off to Git
Jenkins is the CI half of the pipeline. Its responsibility ends with a commit to the GitOps config repo. Jenkins never touches the production cluster directly — it has no kubeconfig for production, no OCP ServiceAccount token for production, and no awareness of what Argo CD does after the commit.
Stage 1
Checkout — clone source repo
Jenkins clones gitlab.corp.novacrestbank.com/banking/payment-service at the commit SHA that triggered the webhook. The checkout verifies the commit is on a protected branch (main) — feature branch pushes do not trigger the production pipeline. The exact commit SHA is recorded in every subsequent artefact for traceability.
Stage 2
Build and unit test
Maven runs the build (mvn clean verify), compiling the Java code, running all unit tests, and producing the application JAR. If any unit test fails, the pipeline stops here. SonarQube quality gate runs — code coverage, security hotspots, and maintainability score must meet the configured thresholds. A failed quality gate blocks the pipeline.
Stage 3
Container image build with Buildah
Buildah builds the container image from the Dockerfile in the source repo. Buildah runs as a non-root user in an ephemeral Jenkins agent pod — no Docker daemon, no docker.sock socket mount, no privileged container required. This is a key security property: the build environment cannot escape its container. The image is built but not yet pushed.
Stage 4
Push to Harbor and sign with cosign
The image is pushed to harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service:vX.Y.Z. Immediately after push, cosign signs the image using NovaCrestBank's signing key (stored in a Kubernetes Secret in the Jenkins namespace — accessed by the Jenkins ServiceAccount). The signature is pushed to Harbor alongside the image. The image digest (sha256:...) is recorded for all subsequent pipeline stages.
Stage 5–8
ACS gate, staging deploy, integration tests, production MR
ACS scans the image via the Harbor registry API — if critical CVEs are found, the pipeline fails. Jenkins commits the new image digest to the staging overlay in the config repo and waits for Argo CD to sync staging. Integration tests run against staging. If all pass, Jenkins opens a merge request against the production overlay in the config repo — this MR is the handoff from Jenkins to the human approval process. Jenkins' job is done.
GITOPS-01 · Section 15
Argo CD's role — deploy, reconcile, and maintain desired state
Argo CD is the CD half of the pipeline. It has no knowledge of Jenkins, no awareness of what built the images, and no direct interaction with developers. It has one job: ensure the cluster's actual state matches the Git repository's declared state.
Argo CD's responsibilities and non-responsibilities — the boundary is important
Argo CD could in theory do more — it could trigger builds, validate images, or run tests. Why is its role scoped to "read Git and apply to cluster"?
Keeping Argo CD's responsibility scoped to deployment and reconciliation is a deliberate design choice that makes the system easier to reason about and more resilient.

Single responsibility. Argo CD is extremely good at one thing: continuously reconciling cluster state to match a Git repository. If it also triggered builds, managed CI pipelines, and ran tests, a failure in any of those additional functions could compromise its ability to do the one critical thing — deploy and maintain production state. The pipeline fails fast in Jenkins; the deployment never fails in Argo CD because its job is simple.

Credential minimisation. Argo CD's ServiceAccount needs permissions to create and modify Kubernetes resources in the banking namespaces. That is all. It does not need Harbor credentials, Jenkins API tokens, or GitLab write access. The credential surface area for the deployment system is minimised to exactly what is needed.

Failure domain isolation. If Jenkins is down, builds stop — but Argo CD continues reconciling production to match the current config repo state. Production does not degrade because the CI system is unhealthy. If Argo CD is down, the cluster continues running its last deployed state — applications do not fail because the deployment system is temporarily unavailable. The two systems can fail independently without cascading.
GITOPS-01 · Section 16
Image signing with cosign — the trust chain from code to container
Image signing with cosign establishes a cryptographic link between the image in Harbor and the Jenkins build that produced it. It closes a specific supply chain attack vector: an attacker pushing a malicious image to Harbor cannot get it into production because unsigned images are rejected by the admission controller.
What cosign signing prevents — the specific attack it closes
Harbor already has role-based access control — only authorised service accounts can push images. If Harbor access is controlled, what additional security does cosign signing provide?
Harbor RBAC controls who can push to Harbor. cosign signing controls which images that were pushed to Harbor are permitted to run in production. These are different controls addressing different attack vectors.

The attack that RBAC alone does not prevent: Harbor's ncb-pipeline-svc account has push access to the banking-apps project. If an attacker compromises the Jenkins server (through a vulnerability in Jenkins itself, a malicious Jenkins plugin, or a compromised agent pod), they inherit the ncb-pipeline-svc Harbor credentials. They can push a malicious image to harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service:latest. Without cosign, if that image tag is referenced in the config repo, Argo CD would deploy it to production. With cosign, the malicious image cannot be signed (the signing key is in a Kubernetes Secret that the attacker does not have), so the ACS admission controller rejects the unsigned image when Argo CD attempts to deploy it.

The trust chain cosign creates: Only images signed by the NovaCrestBank signing key can run in production namespaces. The signing key is only accessible to the Jenkins ServiceAccount in the jenkins namespace. An image signed by this key was built by an authorised Jenkins pipeline that ran the full test and scan sequence. An unsigned image — regardless of where it came from — is rejected. This satisfies NIST SI-7 (Software, Firmware, and Information Integrity) and is increasingly referenced in PCI-DSS DSS v4 software supply chain requirements.

GITOPS-01 · Section 17
The developer's daily workflow in a GitOps environment
A developer on the banking platform team works within the GitOps model every day. This section makes the workflow concrete — what a developer does from writing code to seeing it run in production, and what they never need to do.
1
Write code, push to feature branch
Developer writes the feature, runs unit tests locally, pushes the feature branch to banking/payment-service on GitLab. A branch pipeline may run (a lighter-weight Jenkins job that runs tests only, without building and pushing an image). When ready: opens a merge request to main. A peer developer reviews and approves the MR. The developer merges.
2
Watch the pipeline run automatically
Within 30 seconds of merging to main, Jenkins starts the production pipeline (triggered by the GitLab webhook). The developer can watch the pipeline progress in the Jenkins UI (accessible at jenkins.apps.ocp.corp.novacrestbank.com) — each stage showing its status. A failed stage notifies the developer via email. A successful pipeline posts a comment on the GitLab MR with links to the build log and ACS scan result.
3
Confirm staging deployment via Argo CD
After the pipeline succeeds, the developer can see the staging Application in the Argo CD UI (argocd-server-gitops.apps.ocp.corp.novacrestbank.com — read-only access for developers, as configured in OPS-05 Section 19). The staging Application shows Synced and Healthy with the new image digest. The developer can see their pod running. They can query the payment API in staging to verify the feature works end-to-end.
4
Production merge request — the one human gate
Jenkins opened a production promotion MR in the config repo automatically. The developer sees this MR linked from the Jenkins build output. They cannot merge it — they have Reporter access (read-only) in the config repo. A platform engineer reviews the MR (the diff is exactly one line: the updated image digest), verifies the ACS scan and integration test results are attached, and approves the merge. Argo CD syncs production within 3 minutes of the merge. The developer sees the production Application turn Synced and Healthy in the Argo CD UI.

What a developer never does in a GitOps environment

  • Runs kubectl apply or oc apply in the production namespace
  • SSHes to a production node or container
  • Manually edits a Kubernetes resource in the production cluster
  • Pushes directly to the main branch of the GitOps config repo
  • Bypasses the Jenkins pipeline to push an image directly to Harbor
  • Manually triggers an Argo CD sync for production without a corresponding config repo change
GITOPS-01 · Section 18
Emergency change workflow — GitOps during an incident
The most common objection to GitOps in banking operations is: "What about emergencies? We cannot wait for a merge request when the payment service is down." This section addresses that objection directly and explains why GitOps does not slow down incident response — it makes it faster and safer.
The emergency change workflow — maintaining GitOps discipline under time pressure
At 2:30 AM on a Saturday, the payment-service is returning 503 errors due to an OOM condition. The on-call engineer needs to increase the memory limit immediately. The normal MR approval process takes 30 minutes. What does the GitOps model provide?
The GitOps model for emergencies is not "bypass the process" — it is "use the expedited process." The process takes 5–10 minutes rather than the normal 30–60 minutes. The audit trail is still complete. The rollback is still a git revert.

Step 1 (2 minutes): Create and commit the fix. The on-call engineer opens the config repo, navigates to apps/payment-service/overlays/production/kustomization.yaml, changes the memory limit from 512Mi to 1Gi, commits to a new branch directly from the GitLab web UI (no laptop required — a phone browser works), and opens a merge request marked "URGENT: P1 incident CR-2026-04-20-P1."

Step 2 (5 minutes): Expedited approval. The on-call engineer's incident management process includes an "emergency approver" — a senior platform engineer on the secondary on-call rotation. They are called (phone, not Slack — Slack may be down during an incident). The secondary on-call engineer reviews the one-line diff in the MR, approves it, and merges it. GitLab records the approval timestamp, approver identity, and IP address automatically.

Step 3 (3 minutes): Argo CD deploys. Argo CD detects the merge, syncs the production namespace, updates the Deployment with the new memory limit. OCP performs a rolling update. The OOM condition is resolved within 8–10 minutes of the engineer waking up.

Why this is faster than the alternative. The alternative (kubectl edit in production) appears faster but creates two problems: (1) Argo CD detects the drift and shows production as OutOfSync — another alert fires and the on-call engineer must also update Git to resolve the drift, adding work to an already stressful incident. (2) There is no audit record of who changed what during the incident — a compliance gap that must be reconstructed manually from memory and terminal history after the fact. The GitOps emergency process takes 10 minutes and creates a perfect audit record automatically. The kubectl alternative takes 5 minutes and creates 30 minutes of post-incident cleanup work.
