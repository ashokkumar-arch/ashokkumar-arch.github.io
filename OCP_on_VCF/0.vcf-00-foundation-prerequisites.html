Foundation & Prerequisites

Everything the VCF + NSX + OCP series assumes — made explicit, with the why behind every decision

00 — Why this guide exists

The NovaCrestBank VCF + NSX + OCP series is a standalone series. You do not need to have read the Classic vSphere series to follow it from start to finish.

This guide is the foundation. It establishes every design decision, every IP address, every VLAN, every FQDN, and every certificate plan that all subsequent guides reference. Before any NSX guide begins, this document must be understood. The most common reason NSX configurations fail is that the network planning was not precise — TEP pool overlap, BGP ASN conflict, or missing DNS records. This guide eliminates all of those risks by making the design explicit before any tool is opened.

How to use this guide Read it fully before starting NSX-01. Every table in this guide is the authoritative reference for the corresponding configuration guide — the NSX VLAN IDs used in NSX-01 come from the VLAN plan in Section 05. The IP addresses in NSX-02 come from the IP plan in Section 06. The DNS records in WIN-VCF-01 come from Section 08. Nothing in this series is designed from scratch mid-guide — everything is designed here first.
Relationship to the Classic vSphere series The NovaCrestBank Classic vSphere series covered building this same physical environment from the ground up — Cisco leaf-spine fabric, NX-OS configuration, ESXi installation, vCenter deployment, Windows AD/PKI, and OpenShift on classic vSphere. If you want the deep-dive on any of those foundational layers, the Classic series provides it. This VCF series begins where VCF itself begins: the physical fabric is already built, and now we are deploying VMware Cloud Foundation on top of it.

01 — What VCF changes from classic vSphere — the single most important mental shift

In classic vSphere, networking worked like this: physical VLAN → vDS port group → VM NIC. Every new application network required a VLAN provisioned on the Cisco switches and a port group created on the vDS. The network team and the virtualisation team both had to be involved for every new network segment.

In VCF with NSX, the physical network team provisions a small, fixed set of VLANs once during VCF bring-up — and then steps away. Every network segment that OCP, banking applications, or any workload needs is created by the virtualisation/cloud team in the NSX Manager GUI in seconds, with no physical switch changes required.

🔴 Application Layer — OpenShift

OCP API, Ingress, pod networks, namespaces, banking application workloads. All live as logical constructs managed by NSX and OCP jointly. The OCP installer creates NSX segments automatically during installation when NSX-T integration is configured.

🟣 NSX Overlay — Software-Defined Network

Tier-0 Gateway (BGP border), Tier-1 Gateways (tenant routing), Segments (logical switches), Load Balancer (API VIP + Ingress VIP), Distributed Firewall (microsegmentation). All created in NSX Manager GUI — zero physical switch changes required after initial VCF bring-up.

🔵 VCF Underlay — Physical VLANs

A small, fixed set of VLANs on the Cisco fabric: VCF management traffic, NSX TEP tunnel traffic, and Edge uplink traffic. Provisioned once during VCF bring-up and rarely changed afterwards. VLANs 10, 20, 25, 40, 100, 101, 102, 103 — defined in full in Section 05.

⚫ Physical Fabric — Cisco Nexus Leaf-Spine

Three Cisco Nexus 93180YC-FX leaf switches and two Cisco Nexus 9504 spine switches. Same hardware used in the Classic series. In VCF mode it carries fewer VLANs but at higher bandwidth — all NSX overlay traffic runs as GENEVE-encapsulated UDP on top of the physical links. MTU 9000 end-to-end is required.

The key mental shift — VLANs are for infrastructure, not for applications In the Classic series, every application had its own VLAN: VLAN 50 for OCP nodes, VLAN 60 for production VMs, VLAN 70 for Windows servers. In VCF + NSX, physical VLANs exist only for infrastructure (VCF management, NSX tunnels, Edge uplinks). All application networks live on NSX logical segments — which look and behave exactly like VLANs to the VMs and pods, but are software-defined and require no physical VLAN provisioning.

02 — Physical fabric — same hardware, completely different job

NovaCrestBank's physical data centre fabric is a Cisco Nexus leaf-spine architecture:

  • Spine layer: 2 × Cisco Nexus 9504 (100 GbE uplinks, pure Layer 3, BGP/ECMP)
  • Leaf layer: 3 × Cisco Nexus 93180YC-FX (48 × 25 GbE downlinks to servers, 100 GbE uplinks to spines)
  • Edge routing: 2 × Cisco ASR 1002-X (BGP dual-ISP, upstream to Palo Alto NGFW)
  • OOB management: Cisco Catalyst 9200 (iLO on all servers, switch management)
  • Server NICs: HP ProLiant DL380 Gen11 — 6 NICs per host (vmnic0/1: management LACP, vmnic2/3: VM/NSX LACP, vmnic4/5: vSAN LACP)

The fabric gets simpler because application complexity moves into software. The NAS VLAN (30) is also not present in the VCF underlay — NetApp NFS access is handled via the NSX overlay in VCF mode.

03 — Two workload domains — the VCF topology

VCF organises ESXi hosts into Workload Domains. NovaCrestBank uses two:

Management Domain

Created automatically during VCF bring-up. Cannot be deleted. Runs VCF's own management plane. What runs here:

  • SDDC Manager (sddc-manager.novacrestbank.local) — VCF's orchestrator for the entire VCF stack
  • Management vCenter (vcenter-mgmt.novacrestbank.local) — manages Management Domain ESXi hosts
  • NSX Manager cluster (3 VMs: nsx-mgr-01/02/03) — shared NSX instance for both domains
  • NSX Edge cluster (edge-01, edge-02) — runs T0 gateway and BGP routing
  • DC01 and DC02 — Active Directory domain controllers
  • Root CA and Issuing CA — NovaCrestBank internal PKI
  • Infrastructure service VMs (DHCP, jump host)

ESXi hosts required: 4 minimum for a VMware production-supported configuration (esxi-mgmt-01 through esxi-mgmt-04).

OpenShift VI Workload Domain

Created after the Management Domain is healthy. Has its own vCenter instance. Uses shared NSX from the Management Domain. What runs here:

  • OCP Workload vCenter (vcenter-ocp.novacrestbank.local)
  • OpenShift master nodes (3), infrastructure nodes (3), worker nodes (3+)
  • Future banking application workloads

ESXi hosts: 5 (esxi-ocp-01 through esxi-ocp-05).

Why shared NSX — not per-domain NSX VCF allows each Workload Domain to have its own NSX instance or share the Management Domain's NSX. NovaCrestBank uses shared NSX. Per-domain NSX means two 3-node NSX Manager clusters to maintain, two sets of certificates, two control planes to monitor. At NovaCrestBank's scale (9 ESXi hosts total), one NSX instance is the correct and simpler architecture — one pane of glass for all networking across both domains.
PCI-DSS v4.0 Req 1.3.2

Having OpenShift as a separate VCF Workload Domain — with its own vCenter and distinct NSX segment namespace — provides the administrative and logical separation between the management plane and the cardholder data environment required by PCI-DSS network segmentation controls.

Why two Edge Nodes — not one An Edge Node runs the Tier-0 Gateway that holds the BGP adjacency to the Cisco fabric. If that BGP adjacency drops, every VM and OCP pod that needs to reach anything outside the NSX overlay loses connectivity. One Edge Node is a single point of failure for all external routing. Two Edge Nodes in Active-Active HA means if edge-01 fails, edge-02 continues forwarding immediately — no manual intervention, no VIP migration delay. NIST SP 800-53 SC-5NIST CP-7

Why two T1 gateways — not one The two T1s represent two separate security domains with different trust levels. Traffic between OCP nodes and infrastructure services (e.g., OCP pods querying DC01 for LDAP) must traverse T0 → T1-Infra → seg-infra-services. This inter-T1 path creates a natural policy enforcement point. NSX DFW gateway firewall rules on NCB-T1-OCP explicitly allow only specific OCP→infrastructure traffic (LDAPS port 636, DNS port 53) and deny everything else. With one T1, that boundary disappears.
