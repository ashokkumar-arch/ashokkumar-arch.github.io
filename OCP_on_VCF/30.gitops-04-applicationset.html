GITOPS-04 · Section 01
ApplicationSet vs Application — when you need one definition for many environments
An Argo CD Application is a one-to-one relationship: one Application object deploys one application to one namespace. An ApplicationSet is a controller that generates multiple Application objects from a single template. NovaCrestBank uses an ApplicationSet because the same banking microservices must be deployed to three environments — dev, staging, and production — with consistent configuration and consistent sync policy, without maintaining three separate Application objects per microservice.

AppSet
ApplicationSet vs three separate Application objects — the maintenance problem
NovaCrestBank has 5 banking microservices, each deployed to 3 environments. That is 15 Application objects if using individual Applications, or 5 ApplicationSet objects. Why does the ApplicationSet approach win?
The maintenance arithmetic is only part of the answer. The structural argument is more important: an ApplicationSet guarantees that the same sync policy, the same source repo path pattern, and the same destination cluster are applied consistently to every generated Application. Individual Application objects can drift from each other as engineers modify them independently.

The maintenance problem: 15 Application objects for 5 microservices × 3 environments. When the GitOps config repo URL changes (domain migration, GITOPS-00 Section 03), every Application object must be updated. When NovaCrestBank adds a fourth environment (disaster recovery), 5 new Application objects must be created. With ApplicationSet: update one template, all 15 Applications are regenerated. Add one entry to the generator list, 5 new Applications appear automatically.

The consistency problem: Individual Application objects are created once and then mutated independently by engineers over time. One environment's Application might have syncPolicy.automated.selfHeal: true while another has it disabled — a configuration drift that causes inconsistent behaviour that is invisible until an incident occurs. ApplicationSet templates enforce consistency: every generated Application has identical sync policy because they all derive from the same template.

The GitOps problem: An ApplicationSet object is itself a Kubernetes resource stored in the GitOps config repo and managed by Argo CD. Adding a new environment is a Git commit. With individual Application objects, adding an environment might be done via the Argo CD UI (not in Git — a GitOps violation) or via a manual kubectl apply (also not the right approach). ApplicationSet keeps the "what applications exist" question in Git, not just the "what state applications are in" question.

Individual Application objects (not used)

  • 15 objects for 5 microservices × 3 environments
  • Each object maintained separately — drift risk over time
  • Adding an environment = 5 new objects created manually
  • Sync policy can diverge between environments
  • No enforcement that dev/staging/prod use the same structure

ApplicationSet (NovaCrestBank's approach)

  • 5 ApplicationSet objects — one per microservice
  • All 15 Applications generated from consistent templates
  • Adding an environment = one line in the generator list
  • Sync policy identical across all generated Applications
  • Structure enforced: every Application uses the same overlay pattern
GITOPS-04 · Section 02
The Kustomize base-plus-overlay model — how environments diverge from a single definition
Kustomize is the templating engine Argo CD uses to render the Kubernetes manifests for each environment. NovaCrestBank uses Kustomize's base-plus-overlay pattern: a base/ directory contains the canonical Kubernetes resources, and each environment's overlay/ directory patches only what is different for that environment. Argo CD runs kustomize build and applies the output to the cluster.
Base
base/ — the canonical definition
The base directory contains complete, valid Kubernetes manifests: a Deployment with 1 replica and placeholder image, a Service, a HorizontalPodAutoscaler, a NetworkPolicy, and a ServiceMonitor. These are the resources that exist in every environment. The base is valid on its own but is never deployed directly — it is always referenced through an overlay. Argo CD never points at the base directory. If you can understand the base, you understand 80% of what runs in production.
Overlay
overlays/<env>/ — what is different for this environment
Each environment overlay contains a kustomization.yaml that: (1) references the base with resources: [../../base], (2) applies strategic merge patches to change specific fields (replica count, resource limits, image reference), and (3) may add environment-specific resources not in the base (PodDisruptionBudget for production, lax resource limits for dev). Argo CD points at the overlay directory. Kustomize merges base + patches and produces the final manifests.
Output
kustomize build — what Argo CD actually applies
When Argo CD syncs, it runs kustomize build overlays/production/ (for the production Application). Kustomize reads the base Deployment, applies the production replica count patch (3 replicas), applies the production resource limit patch, and substitutes the signed image digest. The output is a complete, environment-specific set of Kubernetes manifests — no templating variables, no Helm values — just plain Kubernetes YAML. Argo CD compares this output to the cluster's current state to determine if a sync is needed.
Why Kustomize and not Helm for NovaCrestBank's application delivery
Helm uses Go templating — YAML with {{ .Values.replicaCount }} placeholders that must be evaluated against a values file to produce actual manifests. Argo CD supports Helm, but the output of helm template is less predictable: the same values file on a different Helm version can produce different output. Kustomize patches are applied to the base YAML using strategic merge — the output is deterministic given the same inputs, regardless of Kustomize version. For a regulated banking environment where "exactly what is deployed must be auditable from the config repo," Kustomize's deterministic, patch-based approach is more appropriate than Helm's template-evaluation approach. The diff in a GitLab MR for a Kustomize change shows exactly what the deployed manifest will change. The diff for a Helm values change requires running helm template to understand the manifest impact.
GITOPS-04 · Section 03
ApplicationSet generators — list, git directory, and cluster
An ApplicationSet generator defines how the ApplicationSet controller creates Application objects. NovaCrestBank uses the List generator — the simplest and most explicit generator — which creates one Application per entry in the list. The list entries are the environments: dev, staging, and production.
Why the List generator and not the Git Directory or Cluster generator
Argo CD's Git Directory generator automatically discovers overlay directories in the config repo and creates Applications for each one. This is more automated than the List generator. Why does NovaCrestBank use the List generator instead?
The Git Directory generator creates Applications for every directory it discovers that matches a pattern. This is powerful for self-service platforms where teams add their own application directories. It is inappropriate for NovaCrestBank's banking environment for one specific reason: it makes adding a new deployment environment a filesystem operation rather than an explicit configuration decision.

If NovaCrestBank used the Git Directory generator and a developer accidentally created a directory named overlays/dr-test/ in the config repo for testing purposes, the ApplicationSet controller would immediately create a new Argo CD Application pointing at that directory and attempt to deploy it. In a banking production cluster, creating unintended deployments from filesystem artifacts is a configuration management risk — it violates the principle that every production deployment must be explicitly authorised.

The List generator is explicit: the list of environments is declared in the ApplicationSet YAML. Adding a new environment (e.g., a disaster recovery environment) requires a deliberate change to the ApplicationSet object — a Git commit, a merge request, an approval, and Argo CD applying the change. The new Application is created only when the platform team explicitly decides to create it. This is the correct model for a regulated environment.

GITOPS-04 · Section 04
The payment-service ApplicationSet — complete YAML annotated
The complete ApplicationSet for the payment-service microservice. This object lives in gitops/banking-config/platform/appsets/payment-service-appset.yaml. Argo CD manages it (it is itself in the config repo, managed by a bootstrap Application). Every field is annotated.
platform/appsets/payment-service-appset.yaml — complete ApplicationSet
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: payment-service
  namespace: gitops
    # ApplicationSet lives in the gitops namespace alongside Argo CD.
    # The name "payment-service" prefixes all generated Application names:
    # payment-service-dev, payment-service-staging, payment-service-production.
spec:
  goTemplate: true
    # goTemplate: true enables Go templating in the ApplicationSet template.
    # This allows {{ .environment }} and {{ .namespace }} variable substitution.
    # Without goTemplate: true, variables use the older {{ `{{environment}}` }} syntax.

  generators:
    - list:
        elements:
          - environment: dev
            namespace: banking-dev
            overlayPath: apps/payment-service/overlays/dev
            syncPolicy: automated
            selfHeal: true
            prune: true
                        # dev: fully automated — any drift is immediately corrected.
                          # selfHeal: true — if a developer manually edits a resource in
                          # banking-dev, Argo CD reverts it within 3 minutes.
                          # prune: true — if a resource is removed from the config repo,
                          # Argo CD deletes it from the cluster. Safe for dev.

          - environment: staging
            namespace: banking-staging
            overlayPath: apps/payment-service/overlays/staging
            syncPolicy: automated
            selfHeal: true
            prune: true
                        # staging: also automated — the Jenkins pipeline directly commits
                          # to the staging overlay, and Argo CD deploys immediately.
                          # This is the automated CD half of the pipeline (GITOPS-03 Stage 6).

          - environment: production
            namespace: banking-production
            overlayPath: apps/payment-service/overlays/production
            syncPolicy: manual
            selfHeal: false
            prune: false
                        # production: MANUAL sync policy.
                          # selfHeal: false — Argo CD detects drift and alerts (OutOfSync status)
                          # but does NOT automatically revert it. A production engineer makes
                          # the decision to sync, not the automation.
                          # prune: false — resources are never automatically deleted from
                          # production. A resource removal requires an explicit sync with
                          # prune enabled — an additional human confirmation step.
                          # This is a critical safety property: a misconfigured config repo
                          # deletion cannot automatically remove a production resource.

  template:
    metadata:
      name: payment-service-{{ .environment }}
      namespace: gitops
      labels:
        app.kubernetes.io/name: payment-service
        environment: {{ .environment }}
      annotations:
        notifications.argoproj.io/subscribe.on-sync-failed.gitlab: payment-service
                # Argo CD Notifications: send a GitLab comment on sync failure.
                  # The annotation subscribes the "on-sync-failed" trigger to
                  # the "gitlab" notification template, targeting the payment-service project.
                  # Platform engineers and developers see sync failures in GitLab
                  # without needing to monitor the Argo CD UI continuously.

    spec:
      project: banking-apps
            # project: the Argo CD AppProject that scopes this Application.
              # banking-apps AppProject (configured in OPS-05) restricts:
              # - Source repos: only gitlab.corp.novacrestbank.com/gitops/*
              # - Destination namespaces: only banking-dev, banking-staging, banking-production
              # - Cluster resources: cannot create ClusterRoles or ClusterRoleBindings
              # A misconfigured ApplicationSet cannot accidentally deploy to openshift-monitoring.

      source:
        repoURL: git@gitlab.corp.novacrestbank.com:gitops/banking-config.git
        targetRevision: HEAD
                # HEAD: always use the latest commit on the default branch (main).
                  # The staging overlay is updated by the Jenkins pipeline commit (GITOPS-03 Stage 6).
                  # The production overlay is updated by the production MR merge.
                  # Both changes land on main and Argo CD picks them up at HEAD.
        path: {{ .overlayPath }}
                # {{ .overlayPath }} resolves to the environment-specific overlay path.
                  # dev: apps/payment-service/overlays/dev
                  # staging: apps/payment-service/overlays/staging
                  # production: apps/payment-service/overlays/production
        kustomize:
          version: v5.3.0
                    # Pin the Kustomize version to ensure deterministic builds.
                      # A Kustomize version change can produce different output from the
                      # same kustomization.yaml. Pinning ensures the same version that
                      # was tested in the dev environment runs in production.

      destination:
        server: https://kubernetes.default.svc
                # In-cluster deployment — Argo CD uses its own ServiceAccount to apply
                  # manifests to the local cluster (not a remote cluster). Single cluster
                  # in NovaCrestBank's current architecture.
        namespace: {{ .namespace }}

      syncPolicy:
        automated:
          selfHeal: {{ .selfHeal }}
          prune: {{ .prune }}
        syncOptions:
          - CreateNamespace=false
                    # false: do not auto-create the namespace. Namespaces are
                        # pre-created with specific labels and annotations (NSX segment
                        # binding, monitoring labels). Auto-creating a namespace would
                        # miss these labels and break network policy and monitoring.
          - PrunePropagationPolicy=foreground
                    # Foreground deletion: when Argo CD prunes a resource, it waits
                        # for the resource to actually be deleted before considering the
                        # sync complete. Background deletion can leave dangling child
                        # resources (e.g., Pods from a deleted Deployment) until the
                        # garbage collector runs. Foreground is safer for production.
          - RespectIgnoreDifferences=true
      ignoreDifferences:
        - group: apps
          kind: Deployment
          jsonPointers:
            - /spec/replicas
                    # ignoreDifferences on /spec/replicas: when the HPA (HorizontalPodAutoscaler)
                        # scales the Deployment to a different replica count than the config repo
                        # specifies, Argo CD would normally see this as drift and mark the
                        # Application OutOfSync. ignoreDifferences tells Argo CD to ignore
                        # the replica count field when computing drift.
                        # Without this: every HPA scale event causes an OutOfSync alert, and
                        # if selfHeal is enabled, Argo CD reverts the HPA's scaling decision —
                        # fighting the HPA in a continuous loop.
GITOPS-04 · Section 05
How dev, staging, and production differ — what each overlay patches
The three environments share the same base but differ in replica count, resource limits, image reference, and which additional resources exist. Understanding the differences explains why a separate overlay per environment is necessary rather than using Helm values files or environment variables.
Dev — banking-dev
  • Replicas: 1
  • Image: :latest tag (developers push directly)
  • CPU request: 100m · limit: 500m
  • Memory request: 256Mi · limit: 512Mi
  • No PodDisruptionBudget
  • No HPA (manual scaling only)
  • NetworkPolicy: permissive (within namespace)
  • Sync: automated, selfHeal, prune
  • Log level: DEBUG
Staging — banking-staging
  • Replicas: 2 (production-like availability)
  • Image: @sha256:... digest (Jenkins pipeline commits)
  • CPU request: 250m · limit: 1
  • Memory request: 512Mi · limit: 1Gi
  • No PodDisruptionBudget (2 replicas but staging can tolerate brief disruption)
  • HPA: min 2, max 4
  • NetworkPolicy: same rules as production
  • Sync: automated, selfHeal, prune
  • Log level: INFO
Production — banking-production
  • Replicas: 3 (UPG-02 PDB requirements)
  • Image: @sha256:... digest (production MR merge only)
  • CPU request: 500m · limit: 2
  • Memory request: 1Gi · limit: 3Gi
  • PodDisruptionBudget: minAvailable: 2
  • HPA: min 3, max 8
  • NetworkPolicy: strict (defined ingress/egress only)
  • Sync: manual (human approval required)
  • Log level: WARN
GITOPS-04 · Section 06
The base — shared resources across all environments
The base directory contains the resources that every environment needs. It is the canonical definition of the payment-service from Kubernetes' perspective — what a new engineer reads to understand what the payment-service is and how it runs.
apps/payment-service/base/kustomization.yaml — base index
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
metadata:
  name: payment-service-base

commonLabels:
  app.kubernetes.io/name: payment-service
  app.kubernetes.io/part-of: banking-platform
  app.kubernetes.io/managed-by: argocd
    # commonLabels: applied to ALL resources in the base.
    # The managed-by: argocd label is used by the NSX-05 DFW rules
    # to identify Argo CD-managed workloads — a secondary validation
    # that the workload entered production through the GitOps pipeline.

resources:
  - deployment.yaml
  - service.yaml
  - hpa.yaml
  - networkpolicy.yaml
  - servicemonitor.yaml
    # Five resources in the base:
    # deployment.yaml: Deployment with placeholder image and 1 replica
    # service.yaml: ClusterIP Service on port 8080
    # hpa.yaml: HorizontalPodAutoscaler (min/max overridden by each overlay)
    # networkpolicy.yaml: NetworkPolicy (rules overridden by each overlay)
    # servicemonitor.yaml: Prometheus ServiceMonitor (identical across all envs)

images:
  - name: harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service
    newTag: placeholder
      # placeholder tag: the base never references a real image tag.
        # Every overlay uses "kustomize edit set image" to replace this
        # with the environment-appropriate image reference.
        # Kustomize would fail to build without this images entry.
GITOPS-04 · Section 07
Dev overlay — single replica, relaxed limits, latest image
The dev overlay is the most permissive. Developers push images directly to Harbor with the :latest tag and the dev Application syncs automatically — no pipeline, no approval, no digest pinning. This environment is for rapid iteration, not stability.
apps/payment-service/overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

images:
  - name: harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service
    newTag: latest
      # :latest tag for dev — developers push to this tag directly for quick iteration.
        # ACS vulnerability scan is NOT enforced in the dev namespace —
        # developers need to test even pre-hardened builds.
        # cosign signature is NOT enforced in dev — same reason.

patches:
  - patch: |-
          apiVersion: apps/v1
          kind: Deployment
          metadata:
            name: payment-service
          spec:
            replicas: 1
            template:
              spec:
                containers:
                - name: payment-service
                  resources:
                    requests: {cpu: 100m, memory: 256Mi}
                    limits: {cpu: 500m, memory: 512Mi}
                  env:
                  - name: LOG_LEVEL
                    value: DEBUG
                  - name: ENVIRONMENT
                    value: development
      # Inline strategic merge patch: set 1 replica, low resource limits, DEBUG logging.
        # "strategic merge" means: only the specified fields are changed.
        # Everything not mentioned (ports, liveness probes, volume mounts) is
        # unchanged from the base Deployment. This is not a full replacement.
GITOPS-04 · Section 08
Staging overlay — production-like, pipeline-updated image digest
The staging overlay mirrors production configuration as closely as possible — same resource limits, same NetworkPolicy rules, same log level — with two differences: 2 replicas instead of 3, and no PodDisruptionBudget. The image reference is a digest, updated by the Jenkins pipeline (GITOPS-03 Stage 6) after every successful build.
apps/payment-service/overlays/staging/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

images:
  - name: harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service
    newName: harbor.apps.ocp.corp.novacrestbank.com/banking-apps/payment-service
    digest: sha256:abc123def456789...
      # This digest line is what the Jenkins pipeline updates (kustomize edit set image).
        # Before pipeline run: sha256:old-digest (the previous build)
        # After pipeline run: sha256:new-digest (the current build)
        # Git diff: exactly one line changed — the digest value.
        # This one-line diff is the complete staging deployment change record.

patches:
  - patch: |-
          apiVersion: apps/v1
          kind: Deployment
          metadata:
            name: payment-service
          spec:
            replicas: 2
            template:
              spec:
                containers:
                - name: payment-service
                  resources:
                    requests: {cpu: 250m, memory: 512Mi}
                    limits: {cpu: "1", memory: 1Gi}
                  env:
                  - name: LOG_LEVEL
                    value: INFO
                  - name: ENVIRONMENT
                    value: staging

apps/payment-service/overlays/production/pdb.yaml — PodDisruptionBudget
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: payment-service-pdb
  namespace: banking-production
spec:
  minAvailable: 2
    # minAvailable: 2 — at least 2 payment-service replicas must be available
    # at all times. The 3rd replica can be disrupted during node drains
    # (OCP upgrades — UPG-02 Section 10) without violating this PDB.
    # If only 2 replicas are Running (one has crashed), the PDB ALLOWED-DISRUPTIONS
    # drops to 0 — node drains are blocked until the 3rd replica recovers.
    # This is the safety interlock: a degraded payment-service blocks maintenance
    # that would make it more degraded.
  selector:
    matchLabels:
      app.kubernetes.io/name: payment-service
GITOPS-04 · Section 10
NetworkPolicy per overlay — NSX DFW alignment
NovaCrestBank's network security uses two complementary controls: the NSX-05 Distributed Firewall (applied at the vSphere layer — every VM's traffic filtered before it reaches the network) and OCP NetworkPolicies (applied at the Kubernetes layer — pod-to-pod traffic within the cluster). Each overlay defines the NetworkPolicy appropriate for that environment's risk profile.
Why NetworkPolicy in the config repo and not just NSX DFW rules
NSX-05 DFW already controls traffic between VLANs and segments. Why do the Kustomize overlays also include Kubernetes NetworkPolicy objects? Is this redundant?
Defence in depth: NSX DFW and Kubernetes NetworkPolicy operate at different layers and address different threats. They are complementary, not redundant.

NSX DFW operates at the VM (node) level. It controls traffic between OCP nodes — for example, between the worker VLAN and the infra VLAN, or between the OCP overlay segment and the management segment. NSX DFW does not have visibility into individual Kubernetes pod identities (pod labels, namespace, ServiceAccount). It sees: "traffic from this worker node's IP to this other worker node's IP on this port." If two pods on the same worker node communicate, NSX DFW cannot distinguish between them — the traffic never leaves the node's virtual switch.

Kubernetes NetworkPolicy operates at the pod level. It controls traffic based on pod label selectors and namespace selectors — "allow traffic from pods with label app=fraud-detection to pods with label app=payment-service on port 8080." This is intra-node pod-to-pod control that NSX DFW cannot provide. A compromised pod on the same worker node as the payment-service could attempt lateral movement to the payment-service API without NSX DFW seeing it. NetworkPolicy blocks this.

The overlay model enables environment-specific policies: dev uses a permissive NetworkPolicy (easier debugging), staging and production use strict policies (same rules — consistency). The NetworkPolicy is in the config repo and managed by Argo CD — changes require the same MR and approval process as any other production change, providing an audit trail for network policy changes that NSX DFW policy changes alone might not provide.
apps/payment-service/base/networkpolicy.yaml — production-grade NetworkPolicy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-service-netpol
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: payment-service
  policyTypes: [Ingress, Egress]

  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: openshift-ingress
      ports: [{port: 8080}]
            # Allow: OCP Router (in openshift-ingress namespace) → payment-service on 8080.
              # This is how HTTPS traffic from external clients reaches the payment-service.
              # The OCP Router terminates TLS at the Route and forwards plain HTTP to port 8080.

    - from:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: fraud-detection-service
      ports: [{port: 8080}]
            # Allow: fraud-detection-service → payment-service.
              # Fraud detection needs to call the payment-service API to enrich
              # fraud signals with payment context. Scoped to the fraud-detection pod label.

    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: openshift-monitoring
      ports: [{port: 8080, protocol: TCP}]
            # Allow: Prometheus (in openshift-monitoring) → payment-service on 8080.
              # Prometheus scrapes the /actuator/prometheus endpoint for metrics.
              # Without this, the ServiceMonitor would create scrape targets that
              # Prometheus cannot reach — silent metric collection failure.

  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: banking-production
          podSelector:
            matchLabels:
              app.kubernetes.io/name: account-service
      ports: [{port: 8080}]
            # Allow: payment-service → account-service (same namespace, pod label scoped).
              # Payment processing calls the account-service to validate account status.

    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: openshift-dns
      ports: [{port: 53, protocol: UDP}]
            # Allow: payment-service → CoreDNS for service discovery.
              # Without DNS egress, the payment-service cannot resolve

GITOPS-04 · Section 11
Sync waves — controlling deployment order within a sync operation
When Argo CD syncs an Application, by default it applies all resources simultaneously. Sync waves introduce ordering: resources in wave 0 must be Healthy before wave 1 begins, wave 1 before wave 2, and so on. NovaCrestBank uses sync waves to ensure database schema migrations complete before new application pods start.
The database migration problem — why sync waves solve what init containers cannot
When the payment-service is updated to a new version that requires a database schema migration, what is the risk if both the migration job and the new Deployment pods start simultaneously?
The risk is a race condition: if a new payment-service pod starts and connects to the database before the schema migration has added the required new column or table, the pod fails with a database error — "column 'payment_rail' does not exist." This is a production outage during a deployment.

Init containers: An init container on the payment-service pod that runs the migration before the application starts would solve the per-pod sequencing — but has two problems. First, if 3 replicas start simultaneously, all 3 run the migration in parallel — causing database lock contention. Second, the migration runs as part of the Deployment rollout, not as a separate, auditable Job — the migration is harder to observe, retry independently, and report on for change management purposes.

Sync waves: The migration Job is in wave 0. The Deployment is in wave 1. Argo CD applies wave 0 (the migration Job), waits for the Job to report Succeeded, then applies wave 1 (the Deployment). The migration runs exactly once, to completion, before any application pod starts. If the migration fails (database error, timeout), wave 1 never starts — the current production Deployment remains running with the old schema and old application version. The deployment fails cleanly without a partial state.

When sync waves are NOT needed: If a release has no schema changes, the migration Job does not exist in the config repo (no migration, no Job). Argo CD has nothing to wait for. The Deployment wave-1 annotation is still present but wave 0 is empty — Argo CD proceeds directly to wave 1. Sync waves have zero overhead when the wave 0 resources are absent.
GITOPS-04 · Section 12
Payment-service sync wave configuration — database migration first
The payment-service production sync is ordered across four waves. Namespaced infrastructure (ConfigMaps, Secrets managed by External Secrets) must exist before the migration runs. The migration must complete before the application Deployment starts. The HPA is applied last because it depends on the Deployment existing.
Wave -1
ExternalSecret
External Secrets Operator populates the database credentials Secret from CyberArk. Must exist before the migration Job or application pods attempt to connect to the database. Wave -1 (negative) ensures this runs before anything else, including any wave 0 resources.
Wave 0
Job (db-migrate)
The database migration Job runs Flyway (or Liquibase) to apply any pending schema migrations. Argo CD waits for this Job to reach status.conditions[type=Complete] before starting wave 1. If the Job fails (migration error), Argo CD marks the sync as Failed and alerts. Wave 0 timing: typically 10–60 seconds for migration jobs.
Wave 1
Deployment, Service, NetworkPolicy, PDB
The new application Deployment starts after the migration completes. Argo CD performs a rolling update — new pods start using the updated schema, old pods (on the old schema, still compatible due to backward-compatible migration design) continue running until the rolling update completes. Service and NetworkPolicy exist before new pods receive traffic. PDB ensures the rolling update respects minimum availability.
Wave 2
HorizontalPodAutoscaler
The HPA is applied after the Deployment exists and is Healthy. Applying the HPA before the Deployment exists would cause an HPA validation error (HPA references a Deployment that does not yet exist). Wave 2 ensures the target resource exists before the HPA that scales it is created.

GITOPS-04 · Section 13
Health checks — what Argo CD considers Healthy and why it matters
Argo CD evaluates the health of each resource it manages. A resource is Healthy only when it meets specific conditions. The Application's overall health is the worst health of all its resources. An OutOfSync or Degraded Application is immediately visible in the Argo CD UI and triggers a notification.
Resource typeArgo CD health conditionDegraded when
Deployment Healthy when availableReplicas >= desiredReplicas Any replica is in CrashLoopBackOff, ImagePullBackOff, or Pending (resource exhaustion). Progressing during a rolling update — not Degraded, but not Healthy until the rollout completes.
Job (migration) Healthy when status.succeeded >= 1 Job failed (all retries exhausted). Argo CD marks the sync as Failed and stops wave progression. The Application shows Degraded and the sync failure triggers the GitLab notification webhook.
HorizontalPodAutoscaler Healthy when currentReplicas >= minReplicas HPA cannot scale (target resource not found, metrics server unavailable). This is a common source of false Degraded status after a fresh deploy — the HPA may briefly show Degraded before the metrics server has seen the new pods.
PodDisruptionBudget Always Healthy (PDB is a policy object, not a controller) N/A — the PDB's disruptionsAllowed field is informational. Argo CD does not mark a PDB Degraded even when disruptionsAllowed=0. The OCP upgrade controller respects the PDB regardless.
ExternalSecret Healthy when the Secret it manages exists and was recently synced CyberArk unreachable, secret path not found, or sync interval exceeded. A Degraded ExternalSecret means the Kubernetes Secret may be stale or missing — the application pods may fail to start if they depend on this Secret.
GITOPS-04 · Section 14
Custom health checks for banking workloads
Argo CD's built-in health checks cover standard Kubernetes resources. For NovaCrestBank's banking workloads, a custom health check evaluates the payment-service's Spring Boot Actuator health endpoint — ensuring Argo CD considers the Application Healthy only when the application itself reports healthy, not just when Kubernetes says the pod is Running.
Why a Running pod does not always mean a Healthy application
A Kubernetes pod transitions to Running when its containers start — when the process starts, not when the application inside it is ready to serve requests. A Spring Boot application starting with a large classpath and database connection pool initialisation takes 30–60 seconds from container start to first API response. Argo CD's default Deployment health check considers the Deployment Healthy when availableReplicas = desiredReplicas — which counts Running pods, not Ready pods. With a custom health check that queries the actuator endpoint, Argo CD waits until the application actually reports {"status":"UP"} before marking the Application Healthy. This is the correct signal for banking applications where "Healthy in Argo CD" should mean "ready to process payments," not "container process started."
argocd-cm ConfigMap patch — custom health check for payment-service Deployment
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: gitops
data:
  resource.customizations.health.apps_Deployment: |
        hs = {}
        hs.status = "Progressing"
        hs.message = "Waiting for deployment rollout"
        if obj.status ~= nil then
          if obj.status.availableReplicas ~= nil then
            if obj.status.availableReplicas == obj.spec.replicas then
              if obj.status.updatedReplicas == obj.spec.replicas then
                hs.status = "Healthy"
                hs.message = "All replicas updated and available"
              end
            end
          end
        end
        return hs
      # Lua script: Argo CD evaluates this Lua script to determine Deployment health.
        # Healthy only when: availableReplicas = spec.replicas AND updatedReplicas = spec.replicas.
        # This means: all replicas are running the new version AND all replicas are available.
        # During a rolling update: updatedReplicas < spec.replicas → Progressing (not Healthy).
        # This prevents Argo CD from marking the Application Healthy mid-rollout.

  resource.customizations.ignoreDifferences: |
        - group: apps
          kind: Deployment
          jsonPointers:
          - /spec/replicas
      # Global ignoreDifferences matching the ApplicationSet template setting.

GITOPS-04 · Section 17
Sync policy — automated vs manual, self-heal, and prune explained
The ApplicationSet generator list sets different sync policies for each environment. The combination of automated, selfHeal, and prune determines how aggressively Argo CD enforces the config repo's desired state against the cluster's actual state.
SettingdevstagingproductionWhat it controls
automated true true false (manual) Whether Argo CD automatically syncs when it detects the config repo has changed. If false, Argo CD detects the change and shows OutOfSync but waits for a human to trigger the sync.
selfHeal true true false Whether Argo CD automatically reverts cluster-side changes that differ from the config repo. If a developer kubectl edits a resource in dev, Argo CD reverts it within 3 minutes. In production, selfHeal=false means drift is detected and alerted — not auto-reverted.
prune true true false Whether Argo CD deletes cluster resources that no longer exist in the config repo. If prune=false in production and a ConfigMap is removed from the config repo, Argo CD marks the Application OutOfSync but does NOT delete the ConfigMap until a manual sync with prune explicitly enabled is performed.
Never enable automated sync with prune=true in production
If automated: true and prune: true are set for the production Application, and a developer accidentally deletes a resource from the production overlay in the config repo (or commits a rename that Kustomize interprets as a deletion), Argo CD automatically deletes that resource from the production cluster — immediately, without human confirmation. For a banking production environment, this is an unacceptable risk. Production sync must be manual: a human reviews the diff (including any resource deletions), confirms the intent, and explicitly triggers the sync.

Argo CD RBAC — who can sync, who can delete, who reads only
Argo CD has its own RBAC system, separate from Kubernetes RBAC. It controls who can view Applications, who can trigger syncs, who can delete Applications, and who can modify Argo CD settings. NovaCrestBank maps AD groups to Argo CD roles.
argocd-rbac-cm ConfigMap — Argo CD RBAC policy
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: gitops
data:
  policy.default: role:readonly
    # Default role for all authenticated users: readonly.
    # Any AD user who authenticates to Argo CD (via OCP OAuth) gets read access
    # to all Applications by default. They can see what is deployed but cannot
    # trigger syncs or delete anything.

  policy.csv: |
        # Platform engineers: full admin on all banking-apps project Applications
        p, role:platform-engineer, applications, *, banking-apps/*, allow
        p, role:platform-engineer, repositories, *, *, allow
        p, role:platform-engineer, clusters, get, *, allow
        # applications, * : create, get, update, delete, sync, override, action/*
        # Platform engineers can trigger manual syncs for production.

        # Banking developers: read-only on banking-apps Applications
        p, role:banking-developer, applications, get, banking-apps/*, allow
        p, role:banking-developer, applications, sync, banking-apps/*-dev, allow
        # Developers can view all Applications and trigger syncs ONLY for dev Applications.
          # They cannot sync staging or production — even staging syncs are automated
          # by the pipeline. A developer cannot force-sync production.

        # AD group to Argo CD role mapping
        g, NovaCrestBank-Platform-Engineers, role:platform-engineer
        g, NovaCrestBank-Banking-Developers, role:banking-developer
        # Argo CD reads group membership from the OIDC/OAuth token claims.
          # OCP OAuth includes AD group memberships in the token.
          # Platform engineers who are in NovaCrestBank-Platform-Engineers
          # automatically receive the role:platform-engineer Argo CD role.
