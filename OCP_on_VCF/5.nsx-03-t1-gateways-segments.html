NSX-03 · Section 02
Why two T1 Gateways — the security boundary design decision
Understanding this design choice before touching the GUI — it is the single most important architectural decision in this guide
One T1 vs two T1s — the security boundary argument
Both the infrastructure VMs (DC01, DC02) and the OCP nodes could be connected to a single T1 Gateway. Why does NovaCrestBank use two separate T1 Gateways?
Because traffic between two segments on the same T1 is routed internally within the T1 — it never crosses the T0, never passes through a gateway firewall policy point, and is only controlled by DFW. Traffic between segments on different T1s must pass through the T0, which is the natural enforcement point for inter-domain Gateway Firewall rules.

With one T1 (everything together):
— An OCP pod querying DC01 on 10.70.0.10 for LDAPS: the T1 routes it internally. DFW can still inspect this, but there is no Gateway Firewall boundary reflecting the different trust levels of the two domains.

With two T1s (separate domains):
— An OCP pod querying DC01: traffic goes OCP pod → seg-ocp-nodes → NCB-T1-OCP → NCB-T0-Gateway → NCB-T1-Infra → seg-infra-services → DC01. The T0 crossing is the enforcement point. The Gateway Firewall on NCB-T1-OCP explicitly allows only LDAPS (TCP 636) and DNS (TCP/UDP 53) toward the Infrastructure domain — everything else is denied at the T1 boundary.

The two-T1 design encodes the trust boundary in the routing topology, not just in firewall rules. Even if the DFW were misconfigured, an OCP pod could not reach the DC01 admin share or RDP port because the Gateway Firewall on NCB-T1-OCP explicitly denies those. Defence in depth at the network architecture level.

PCI-DSS v4.0 Req 1.3.2 — Restrict inbound and outbound traffic to that which is necessary for the cardholder data environment. The two-T1 design enforces this at the routing topology level, not just at the firewall rule level.
Why both T1s are Active-Standby — not Active-Active
The T0 is Active-Active. Why are the T1 Gateways Active-Standby?
Because T1 Gateways run distributed in the ESXi kernel for east-west (intra-segment and inter-segment) routing — but they have a "service router" component that runs on an Edge Node for north-south traffic (traffic that crosses from T1 to T0). Active-Standby for the T1 service router means one Edge Node is the primary handler for T0-bound traffic from this T1.

In practice for NovaCrestBank, the T1 HA mode has minimal operational impact because:
1. Most VM-to-VM traffic within a segment is handled by the distributed T1 kernel component — no Edge Node involved at all.
2. Traffic leaving the T1 to reach the T0 (e.g., OCP node accessing the internet) does pass through the T1 service router on the Edge Node — but stateful sessions survive failover because NSX synchronises connection state between Edge Nodes.

Active-Standby for T1 is the simpler and most common configuration for tenant gateways. Active-Active T1 requires ECMP configuration on the T1 and is generally used for very high-throughput workloads.

What "Linked Tier-0 Gateway" physically creates
When the T1 is linked to the T0, NSX creates an internal transit link between the two gateways — a point-to-point connection using a reserved link-local subnet (100.64.x.x/31) that is not visible in any GUI. This transit link carries routing information from the T1 to the T0. Because T1 Connected route redistribution was already enabled on the T0 in NSX-02, any segment connected to this T1 will have its prefix automatically propagated to the T0 and included in BGP advertisements to Cisco. The administrator does not need to configure any routing between T1 and T0 — the link is the entire routing configuration.
NSX-03 · Section 05
Enable route advertisement on NCB-T1-Infra
Telling the T1 which of its routes to share upward to the T0 — a separate control from the T0's route redistribution
T1 route advertisement vs T0 route redistribution — two separate controls
The T0 already has "T1 Connected" route redistribution enabled. Does the T1 also need route advertisement configured separately?
Yes — there are two separate controls:

T0 Route Redistribution (configured in NSX-02): tells the T0 "when you receive route information from a linked T1, redistribute it into BGP." This is the outbound control — once the T0 has the routes, it puts them into BGP.

T1 Route Advertisement (configured here): tells the T1 "which of your routes to share upward to the linked T0." By default, a T1 shares nothing — even if the T0 is listening. Without enabling T1 route advertisement, the T0 never receives the T1's segment prefixes and cannot include them in BGP.

Think of it as two gates on the same road: the T1 advertisement gate (does the T1 open its gate to share routes upward?) and the T0 redistribution gate (does the T0 take those routes and put them into BGP?). Both gates must be open for segment prefixes to reach Cisco via BGP.

NSX-03 · Section 14
How to connect existing VMs to NSX segments
DC01, DC02, and IssuingCA currently have their virtual NICs connected to a vDS port group. Moving them to the NSX overlay segment seg-infra-services is a single vNIC setting change — transparent to the VM
What changes on the VM when its NIC is moved to an NSX segment
DC01 currently has its NIC connected to a vDS port group on VLAN 70. When the NIC is changed to seg-infra-services, what physically changes — and what does DC01 experience?
From DC01's perspective: nothing changes. The VM sees a normal Ethernet network with the same IP address, the same gateway (10.70.0.1 — now served by the T1 instead of a physical router), the same subnet. DC01 does not know or care whether its network is backed by a physical VLAN or an NSX GENEVE overlay.

What physically changes:
1. The VM's virtual NIC port is moved from the vDS port group (which maps to physical VLAN 70) to the NSX segment port (which uses GENEVE encapsulation on VLAN 100).
2. Traffic from DC01 now enters the NSX overlay — it is GENEVE-encapsulated by the ESXi host's TEP and forwarded to the destination via the overlay tunnel.
3. The gateway IP 10.70.0.1 is now served by the NCB-T1-Infra distributed router in the ESXi kernel — not by any physical device. DC01's default gateway still responds to ARP for 10.70.0.1 but the response comes from the NSX kernel module, not a physical router.

The IP address does not change. The subnet does not change. Running applications on DC01 experience no interruption — the NIC migration causes a brief link-down/link-up event (approximately 1 second) but the IP stack reconnects immediately.

PCI Req 1.2.1 — This migration implements the documented network design. The VM's effective network position (subnet, gateway, access controls) is the same. The enforcement mechanism changes from physical VLAN-based access control to NSX DFW — which provides more granular control at the VM level.
Migrate DC01 and DC02 one at a time — not simultaneously
DC01 and DC02 are Active Directory Domain Controllers. If both are migrated simultaneously and either experiences a connectivity interruption during the migration, all domain authentication across the environment may fail for that brief period. Migrate DC01 first, verify DNS and LDAP are responding from DC01 after migration, then migrate DC02. The brief link-down during NIC migration (approximately 1 second) is tolerated by AD replication which has a built-in retry mechanism.

Verify reachability — NSX Manager Traceflow
Using NSX Manager's built-in Traceflow tool to verify that overlay traffic flows correctly between segments — without needing SSH access to VMs
What Traceflow is and why it is more useful than a ping for NSX troubleshooting
A ping verifies IP reachability. What does NSX Traceflow add?
Traceflow sends a synthetic packet through the NSX data plane and reports exactly what happens at each hop — which ESXi host processed it, which TEP tunnel it traversed, whether DFW allowed or blocked it, which T1 gateway forwarded it, and whether it arrived at the destination VM's virtual NIC. A ping only reports success or failure. Traceflow reports the complete packet path through the NSX overlay.

For NovaCrestBank: a ping from the management workstation to DC01 (10.70.0.10) proves DC01 is reachable from the physical network via BGP routing. A Traceflow from DC01's virtual port to the IssuingCA virtual port proves the NSX overlay is correctly routing traffic within the seg-infra-services segment. These are different tests — both are needed.

What NSX-04 builds next — NSX Load Balancer

NSX-03 built the tenant networks — VMs can now communicate through the NSX overlay. NSX-04 builds the load balancing — the two Virtual IPs (VIPs) that OCP requires:

  • OCP API VIP — 10.50.0.200 on port 6443. Load balances across the 3 OCP master nodes. The VIP that api.ocp.novacrestbank.local resolves to in DNS.
  • OCP Ingress VIP — 10.50.0.201 on ports 443 and 80. Load balances across the 3 OCP infrastructure nodes. The VIP that *.apps.ocp.novacrestbank.local resolves to.

NSX-04 configures these as NSX Load Balancer Virtual Servers attached to NCB-T1-OCP on NCB-Edge-Cluster. Both VIPs are created before OCP installation begins — the OCP installer expects them to already be reachable at the VIP addresses specified in install-config.yaml.

Because "All LB VIP IPs" advertisement was already enabled on NCB-T1-OCP in Section 08, the VIP /32 host routes (10.50.0.200/32 and 10.50.0.201/32) will automatically be advertised to Cisco via BGP as soon as the Virtual Servers are created in NSX-04.
