The key insight — what makes Tekton architecturally different from Jenkins or GitHub Actions
NovaCrestBank already uses Jenkins. What does Tekton add, and why is it not just another CI/CD system?
Traditional CI/CD systems (Jenkins, GitHub Actions, GitLab CI) run build jobs on dedicated servers — a Jenkins controller node, a GitHub-hosted runner, a GitLab runner. These servers persist between builds. Every build shares the same environment: the same filesystem, the same installed tools, the same network access, the same credentials in environment variables. A build that borrows a Docker socket to build images is running Docker with root access on the same host as every other build. A compromised build step can access credentials left behind by a previous build.

Tekton runs every Task as an ephemeral Kubernetes Pod. When a PipelineRun begins, Tekton creates a new Pod for each Task — with exactly the tools needed for that Task, with a ServiceAccount that has only the permissions that Task requires, with Workspaces providing its storage, with Secrets injected from OCP at runtime. When the Task finishes, the Pod is deleted. The next run creates a completely fresh Pod. There is no persistent build environment that accumulates state, installed tools, or cached credentials.

For NovaCrestBank's banking environment, this matters in three specific ways:
1. Blast radius isolation. A compromised build step cannot affect other builds — each step runs in its own container with its own security context. If the unit test step is somehow exploited, it runs as a ServiceAccount with permission only to read the workspace PVC. It cannot push to Harbor, cannot access production secrets, cannot reach the database.
2. Rootless builds. Tekton uses Buildah for container image building — not Docker. Buildah builds OCI images without a Docker daemon, without root access, without mounting the host Docker socket. Every banking microservice image is built by a non-root process with no host privileges.
3. Audit trail. Every PipelineRun is a Kubernetes object in etcd. Every TaskRun is a Kubernetes object with start time, end time, exit code, and logs. The complete audit trail of every build that produced every image in Harbor is in the OCP cluster's etcd and in Splunk (via the OpenShift Logging audit stream). PCI-DSS Req 6.4 — change management controls — is satisfied by this immutable record.

How Jenkins and Tekton work together in NovaCrestBank's architecture

The two-layer model — Jenkins orchestrates, Tekton executes
Jenkins and Tekton are complementary, not alternatives. Jenkins is the orchestration layer: it receives Git webhooks, evaluates branch strategy (feature branch vs main vs release), decides what to build and when, handles retry logic, sends Slack notifications, and provides the familiar Jenkins UI that development teams use daily. Tekton is the execution layer: the actual build steps — compile, test, containerise, scan, sign — run as isolated Tekton TaskRun pods inside OCP. Jenkins triggers a Tekton PipelineRun via the OCP API, waits for it to complete, and reads the result. This gives NovaCrestBank the orchestration power of Jenkins with the security isolation of Kubernetes-native execution.
OCP-OPS-04 · Section 02
Pipeline and Task — the workflow definition and its steps
Pipeline and Task are the two objects that define what a CI/CD workflow does. They are reusable templates — they describe steps, they do not execute them. Execution happens when a PipelineRun or TaskRun is created.
Pipeline vs Task — the right level of abstraction for each concern
When should something be a separate Task rather than a step inside a single Task?
A Task is the unit of isolation — everything inside a single Task shares one Pod, one ServiceAccount, one set of environment variables, and one set of mounted Secrets. Different Tasks run in different Pods with potentially different ServiceAccounts and different access rights.

The correct boundary is: if two operations need different security contexts, different tools, or different access rights, they belong in separate Tasks. A unit test step and an image signing step should be separate Tasks because: the unit test needs access to Maven and the source code but should not have access to the cosign signing key. The image signing step needs the cosign key but does not need Maven or the source code. Running them in separate Tasks enforces this isolation at the Kubernetes level — not by policy or by code review, but by the fact that the signing key Secret is only mounted in the signing Task's Pod.

Inside a single Task, steps share the Pod's filesystem and environment but run sequentially in separate containers. Use multiple steps within a single Task when operations are tightly coupled (compile → package → unit test all operate on the same build output) and when the isolation overhead of a separate Pod is not warranted.

The Task object — what it defines

steps
steps[]
An ordered list of containers to run sequentially. Each step specifies a container image (the tool), a command or script, and optionally environment variables. Steps share the Task Pod's filesystem — the output of step N is visible to step N+1 in the same directory. Steps run in the order declared and stop at the first failure.
params
params[]
Input parameters the Task receives when it runs. Typed (string, array, object). Referenced in step commands and scripts with $(params.name). Params decouple the Task definition (reusable template) from the specific values of a particular run — the same build Task can build any microservice by passing different IMAGE and APP_NAME params.
workspaces
workspaces[]
Named storage volumes the Task expects to have mounted. A workspace is declared by name in the Task spec — the actual PVC or Secret that backs it is provided when the TaskRun is created. This decouples the Task definition from storage provisioning. The source workspace contains the Git clone. The maven-settings workspace contains Maven configuration. The cosign-key workspace contains the signing key.
results
results[]
Named string outputs the Task produces. A Task step writes a value to /tekton/results/result-name. Tekton captures this and makes it available to subsequent Tasks in the Pipeline via $(tasks.task-name.results.result-name). The image digest produced by the Buildah Task is a result consumed by the cosign signing Task — this is how the exact image digest being signed is passed between Tasks without a shared filesystem.

OCP-OPS-04 · Section 03
PipelineRun and TaskRun — how pipeline definitions become running pods
Pipeline and Task are templates — they define what should happen. PipelineRun and TaskRun are instances — they cause it to actually happen. Understanding the relationship between templates and instances is essential for debugging pipeline failures.
The template vs instance relationship — why this design matters
Why are Pipeline/Task and PipelineRun/TaskRun separate objects? Why not just have one object that both defines and runs?
The separation between template and instance is the same design principle as Kubernetes Deployment vs Pod, or MachineSet vs Machine. The template defines the desired shape. The instance represents one particular execution.

A Pipeline definition for the payment-service build is written once and lives permanently in the cluster. Every time a developer pushes a commit, a new PipelineRun is created — a new instance of that same Pipeline definition, with specific values for this particular run (the Git commit SHA, the image tag to build, the target environment). The Pipeline definition does not change. The PipelineRun records: when it started, which commit it built, each TaskRun's outcome, the logs from each step, and when it finished.

This separation enables: rerunning a failed pipeline by creating a new PipelineRun from the same definition with the same parameters, auditing every historical build by inspecting old PipelineRun objects, and seeing which pipelines are currently running without needing to look at Pod names. It also means a developer cannot accidentally modify the Pipeline definition by triggering a run — they create a PipelineRun, not edit the Pipeline.

What Tekton creates when a PipelineRun starts

1
PipelineRun object created
The top-level run record. Contains: which Pipeline it runs, param values for this run, workspace bindings (which PVC backs the source workspace), the ServiceAccount it runs as, and its current status (Running/Succeeded/Failed). The PipelineRun is the audit record — it persists after the run completes. Find with: oc get pipelineruns -n banking-apps
2
TaskRun objects created — one per Task
For each Task in the Pipeline, Tekton creates a TaskRun object. Each TaskRun records: which Task it executes, the specific params passed to this Task in this run, and its outcome. TaskRuns are created in the order the Pipeline's runAfter dependencies allow — Tasks without dependencies start immediately; Tasks with runAfter: [previous-task] wait for their predecessor's TaskRun to succeed. Find with: oc get taskruns -n banking-apps
3
Pods created — one per TaskRun
Each TaskRun creates exactly one Pod. The Pod runs the Task's steps as a sequence of init containers (for setup) and regular containers (for each step). The Pod is named {taskrun-name}-pod. When the TaskRun completes, Tekton keeps the Pod in Completed state (not deleted) so logs remain accessible. Pods are garbage collected after a retention period (default: 96 hours for completed PipelineRuns).

OCP-OPS-04 · Section 04
Workspaces — how Tasks share data without sharing a Pod
Workspaces are the data-passing mechanism between Tasks. Each Task runs in its own Pod, so Tasks cannot share a local filesystem. Workspaces provide shared PVCs, ConfigMaps, or Secrets that multiple Tasks mount — the output of one Task becomes the input to the next through a shared workspace PVC.
How the source code flows from git-clone to buildah without copying it
The git-clone Task runs in one Pod and clones the source code. The Maven build Task runs in a completely different Pod. How does Maven see the source code that git-clone wrote?
The source workspace is a PVC. Both the git-clone Task and the Maven build Task mount the same PVC at the same path (/workspace/source). When git-clone runs in its Pod and writes source files to /workspace/source, it is writing to the PVC. When the git-clone Pod finishes and the Maven build Pod starts, it mounts the same PVC and sees exactly what git-clone wrote. The PVC is the data bus between Tasks.

This design has an important security consequence: different Tasks can mount different workspaces. The cosign signing Task only needs the cosign key workspace — it does not need the source code workspace. The ACS scan Task only needs the image name (passed as a param) — it does not need the source workspace or the signing key. By selectively mounting only the workspaces each Task legitimately needs, the pipeline enforces least-privilege at the data level: the unit test Task cannot access the cosign key because that workspace is not mounted in its Pod.

The five workspaces in NovaCrestBank's pipeline

OCP-OPS-04 · Section 05
Why Buildah and not Docker — rootless builds in OCP
Every NovaCrestBank banking microservice image is built with Buildah, not Docker. This is not a preference — it is a security requirement. Understanding why makes the Buildah Task configuration self-explanatory.
The Docker socket problem — why Docker is inappropriate for CI/CD in OCP
Jenkins agents traditionally mount the Docker socket (/var/run/docker.sock) to build container images. What is wrong with this approach in OCP?
The Docker socket is a UNIX socket that provides root access to the Docker daemon running on the host. A process that can write to the Docker socket can: run any container with any privileges, mount any host filesystem path into a container, extract any image layer (including layers with embedded secrets), and escape the container to the host. Mounting the Docker socket into a CI/CD Pod is equivalent to giving the CI/CD process root on the node.

In OCP, RHCOS's security model does not run a Docker daemon at all. OCP uses CRI-O for container runtime. There is no Docker socket to mount. Additionally, OCP's SecurityContextConstraints prevent privileged Pod mounts by default — a pipeline Pod trying to mount the host Docker socket would be rejected at admission.

Buildah is an OCI image builder that does not require a daemon, does not require root, and does not mount anything from the host. It builds images by manipulating OCI image layers directly in userspace. A Buildah process running as a non-root user (UID 1000) inside a container can build a fully functional OCI image, push it to Harbor, and sign it with cosign — all without any elevated privileges. This is the only correct approach for building container images in a secure Kubernetes environment.

The resulting images are functionally identical to Docker-built images — they are standard OCI format, stored in Harbor, pulled by CRI-O. The difference is entirely in how they are built, not in what they produce.

Docker socket (do not use)

  • Mounts /var/run/docker.sock — root access to the host
  • Requires privileged: true in Pod spec
  • OCP SCC rejects this by default
  • A compromised build step owns the node
  • Does not work on RHCOS (no Docker daemon)

Buildah (correct approach)

  • No daemon, no socket, no host mounts
  • Runs as non-root UID 1000 inside the container
  • Uses OCP's privileged SCC only for user namespace mapping (fuse-overlayfs)
  • A compromised build step is contained to the Pod
  • Fully supported on RHCOS / CRI-O

OCP-OPS-04 · Section 08
Pipeline namespace, RBAC, and service accounts
The NovaCrestBank banking pipeline runs in the banking-apps namespace — the same namespace as the banking application deployments. The pipeline ServiceAccount needs carefully scoped permissions: enough to do its job, no more.
Why the pipeline ServiceAccount's permissions matter — the principle of least privilege in CI/CD
The pipeline ServiceAccount is the identity the PipelineRun Pods use to interact with the Kubernetes API. If this ServiceAccount has cluster-admin permissions (a common lazy pattern), a compromised build step gains cluster-admin access. The NovaCrestBank pipeline ServiceAccount is granted exactly what it needs: push to Harbor (via Secret), deploy to the banking-apps and banking-staging namespaces (via Role binding), read Secrets in the pipeline namespace (for cosign key and Harbor credentials), and trigger Argo CD sync (via ArgoCD ServiceAccount). It cannot read Secrets in other namespaces, cannot modify cluster-level objects, and cannot reach production namespaces directly.

Stage 3 — Build the OCI image with Buildah
Buildah builds the NovaCrestBank banking microservice OCI image from the compiled JAR. Critically: no Docker daemon, no root, no host mounts. Every image built in NovaCrestBank's pipeline is a rootless Buildah build.
Task — ncb-buildah-build
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: ncb-buildah-build
  namespace: banking-apps
spec:
  params:
    - name: IMAGE
        # Full image reference: harbor.apps.ocp.novacrestbank.local/banking-apps/payment-service:v1.2.3
    - name: DOCKERFILE
      default: ./Dockerfile
    - name: CONTEXT
      default: .
  workspaces:
    - name: source
    - name: harbor-auth
        # Secret containing Harbor robot-jenkins credentials as Docker config JSON.
          # Buildah reads this to authenticate pushes to Harbor.
          # Mounted read-only — this workspace is NOT shared with test Tasks.
  results:
    - name: IMAGE_DIGEST
        # The SHA-256 digest of the built image.
          # Format: sha256:abc123def456...
          # This result is consumed by the cosign Task to sign the EXACT digest —
          # not just the tag. Tags are mutable; digests are immutable.
  steps:
    - name: build-and-push
      image: harbor.apps.ocp.novacrestbank.local/platform/buildah:latest
            # Buildah image mirrored to Harbor. Red Hat's official Buildah image.

          set -eux

            -f $(params.DOCKERFILE) \
            -t $(params.IMAGE) \

          buildah push \

            $(params.IMAGE)

          cat /tmp/image.digest > /tekton/results/IMAGE_DIGEST
          echo "Image pushed: $(params.IMAGE)@$(cat /tmp/image.digest)"

              # This is the ONLY Task in the pipeline with privileged: true.
              # It is required for rootless builds — the privilege is needed for
              # user namespace setup (uid_map/gid_map), not for actual root access.
              # The OCP SCC "pipelines-scc" grants this specifically to pipeline-sa
              # and only in the banking-apps namespace — not cluster-wide.
              # This is much safer than Docker-in-Docker which requires true root.

OCP-OPS-04 · Section 13
Stage 4 — Push to Harbor and sign with cosign
After Buildah pushes the image to Harbor, the cosign Task signs the image digest using NovaCrestBank's private signing key. The signature is stored in Harbor alongside the image. OCP's ClusterImagePolicy (configured in Registry-01 Section 17) refuses to run any unsigned image from the banking-apps scope.
Why sign the digest rather than the tag
The image is pushed as payment-service:v1.2.3. Why does cosign sign the SHA-256 digest rather than the tag?
Image tags are mutable. The tag :v1.2.3 can be reassigned to point to a different image layer by anyone with push access to Harbor. If the pipeline signs the tag and the tag is later reassigned to a malicious image, the signature now appears to cover the malicious image. The SHA-256 digest is immutable — it is a cryptographic hash of the image content. A different image has a different digest. Signing the digest means: the signature covers exactly the bytes that were built and tested, and any substitution of the image content (even with the same tag) produces a different digest that the signature does not cover. OCP's ClusterImagePolicy verifies the signature against the digest at runtime — not against the tag.

This is why Buildah writes the image digest to a TaskRun result in Stage 3, and why the cosign Task receives the digest as a param from that result: $(tasks.buildah-build.results.IMAGE_DIGEST).
Task — ncb-push-and-sign
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: ncb-push-and-sign
  namespace: banking-apps
spec:
  params:
    - name: IMAGE
        # Full image reference
    - name: IMAGE_DIGEST
        # SHA-256 digest from Stage 3 result: $(tasks.buildah-build.results.IMAGE_DIGEST)
  workspaces:
    - name: cosign-key
        # The NovaCrestBank cosign private key Secret.
          # ONLY this Task mounts this workspace.
          # No other Task in the pipeline can access the signing key.
          # This is the Kubernetes-level enforcement of least-privilege:
          # the signing key Secret is not in the environment — it is only
          # accessible to the specific Pod that needs it.
    - name: harbor-auth
        # Harbor credentials — cosign needs to push the signature to Harbor
  results:
    - name: SIGNED_IMAGE_REF
        # Full image reference including digest — passed to ACS scan and GitOps Tasks
  steps:
    - name: sign-image
      image: harbor.apps.ocp.novacrestbank.local/platform/cosign:latest
      script: |
          #!/usr/bin/env bash
          set -eux
          export REGISTRY_AUTH_FILE=$(workspaces.harbor-auth.path)/config.json
          export COSIGN_PASSWORD=""  # Key password injected via Secret in a real deployment
          FULL_REF="$(params.IMAGE)@$(params.IMAGE_DIGEST)"
          # Sign the exact digest — not the tag
          cosign sign \
            --key $(workspaces.cosign-key.path)/cosign.key \
            --tlog-upload=false \
            "${FULL_REF}"
              # --tlog-upload=false: do not upload to Sigstore's public transparency log
                # (Rekor). NovaCrestBank keeps signing events internal. The signature
                # is stored in Harbor as an OCI artifact alongside the image.
          echo -n "${FULL_REF}" > /tekton/results/SIGNED_IMAGE_REF
          echo "Signed: ${FULL_REF}"
          # Verify the signature was stored correctly in Harbor
          cosign verify \
            --key $(workspaces.cosign-key.path)/cosign.pub \
            --insecure-ignore-tlog \
            "${FULL_REF}" \
            && echo "Signature verified ✓"
      securityContext:
        runAsNonRoot: true
        runAsUser: 65532

Stage 5 — ACS vulnerability gate
The ACS scan Task calls the ACS Central REST API to scan the newly pushed image. If any CRITICAL CVE is found, the pipeline fails at this stage. The image remains in Harbor but is not deployed anywhere. This is the hard security gate of the NovaCrestBank pipeline.
Why the ACS gate in the pipeline supplements Harbor's Trivy scan
Harbor's Trivy scans the image on push and blocks pulls if a CRITICAL CVE is found. The ACS pipeline gate adds two things Trivy does not provide. First, ACS uses Red Hat's RHSA (Red Hat Security Advisory) vulnerability database in addition to the NVD — Red Hat's database has faster advisory publication for RHEL/UBI-based images and often identifies CVEs with mitigations that Trivy marks as CRITICAL but Red Hat has rated as moderate in context. Second, the ACS gate result is recorded as a Tekton TaskRun with the full CVE report as an artefact — this is machine-readable audit evidence that the image was scanned at this exact pipeline run, against this exact commit, and the result was pass or fail. Harbor's Trivy block is a runtime control; the ACS pipeline gate is an audit control.
Task — ncb-acs-image-scan
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: ncb-acs-image-scan
  namespace: banking-apps
spec:
  params:
    - name: IMAGE_REF
        # Full image reference with digest from Stage 4 result
    - name: ACS_ENDPOINT
      default: central.stackrox.svc:443
            # Internal cluster Service — no Route hop, lower latency
    - name: FAIL_ON_SEVERITY
      default: CRITICAL
            # Fail the pipeline if any CVE at this severity or higher is found.
                # CRITICAL: hard stop. HIGH: allowed (must be remediated within 30 days).
  workspaces:
    - name: acs-credentials
        # Secret containing the ACS API token for the pipeline service account.
          # Not the admin password — a dedicated read-only API token created in ACS
          # Central → API tokens → New token → Role: "Continuous Integration"
    - name: results
        # Workspace where the CVE report is written for the promotion stage
  results:
    - name: SCAN_RESULT
        # "PASS" or "FAIL" — read by the staging deploy Task to gate deployment
  steps:
    - name: scan-image
      image: harbor.apps.ocp.novacrestbank.local/platform/ubi9-minimal:latest
      script: |
          #!/usr/bin/env bash
          set -eux
          ACS_TOKEN=$(cat $(workspaces.acs-credentials.path)/token)
          IMAGE_REF="$(params.IMAGE_REF)"
          # Trigger image scan via ACS Central REST API
          SCAN_RESPONSE=$(curl -sk \
            -H "Authorization: Bearer ${ACS_TOKEN}" \
            "https://$(params.ACS_ENDPOINT)/v1/imagescan?image=${IMAGE_REF}&force=true")
          # Count CRITICAL CVEs in the scan response
          CRITICAL_COUNT=$(echo "$SCAN_RESPONSE" | \
            python3 -c "import json,sys; d=json.load(sys.stdin); \
              print(d.get('result',{}).get('summary',{}).get('CRITICAL',0))")
          echo "Critical CVEs found: $CRITICAL_COUNT"
          echo "$SCAN_RESPONSE" > $(workspaces.results.path)/acs-scan-report.json
          if [ "$CRITICAL_COUNT" -gt "0" ] && \
             [ "$(params.FAIL_ON_SEVERITY)" = "CRITICAL" ]; then
            echo -n "FAIL" > /tekton/results/SCAN_RESULT
            echo "PIPELINE GATE FAILED: $CRITICAL_COUNT critical CVEs found in $(params.IMAGE_REF)"
            echo "Remediation required before this image can be deployed."
            exit 1
          fi
          echo -n "PASS" > /tekton/results/SCAN_RESULT
          echo "ACS scan gate PASSED. Image approved for staging deployment."
      securityContext:
        runAsNonRoot: true
        runAsUser: 65532
OCP-OPS-04 · Section 15
Stage 6 — Deploy to staging via GitOps
After the ACS gate passes, the pipeline commits the new image tag to the GitOps staging config repository. Argo CD detects the commit and deploys to the banking-staging namespace. The pipeline does not deploy directly — it updates Git.
Why the pipeline commits to Git rather than running kubectl apply directly
The pipeline ServiceAccount has permission to create Deployments in banking-staging. Why not just run kubectl apply directly in the pipeline instead of going through Git?
Running kubectl apply directly from a pipeline creates state that exists in the cluster but not in Git. If the cluster is rebuilt, the deployment is gone. If someone needs to audit what version was running on a specific date, there is no record. If the staging deployment needs to be rolled back, there is no canonical state to return to. The cluster becomes the source of truth — not Git.

Committing to Git and letting Argo CD deploy means: every staging deployment is a Git commit — author, timestamp, diff, and the CI system that created it. Rolling back is a Git revert. Auditing historical deployments is a Git log. If the cluster is rebuilt, Argo CD resyncs from Git and the deployment is restored exactly. The pipeline's job is to build and verify the image; Argo CD's job is to deploy it. Separating these concerns gives NovaCrestBank a complete, immutable audit trail for PCI-DSS Req 6.4.

Stage 8 — Promote to production
After integration tests pass, the pipeline opens a pull request on the GitOps production config repository. The PR requires approval from the technical lead or CISO. Only after approval and merge does Argo CD deploy to production. The pipeline never touches production directly.
Why production requires a pull request rather than an automatic commit
Staging deployment is automated because staging has no cardholder data and no customer impact. Production deployment carries regulatory risk — a failed production deployment affects real banking customers and may trigger PCI-DSS incident reporting. The pull request provides: a mandatory human review of exactly what is about to change in production (the diff shows the new image digest), an approval audit trail in GitLab (who approved it, when, from which IP), a window for the approver to check the integration test results and the ACS scan report in the pipeline dashboard, and the ability to delay the deployment to an approved change window. This directly satisfies PCI-DSS Req 6.4 (change management controls) and FFIEC change management examination expectations.
