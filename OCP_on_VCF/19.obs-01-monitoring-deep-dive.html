The key insight — why Prometheus pulls rather than applications pushing metrics
Traditional monitoring systems (Nagios, Zabbix) have agents that push metrics to a central server. Prometheus does the opposite — it pulls metrics from targets on a schedule. Why?
The pull model has three properties that make it fundamentally better suited for Kubernetes environments than the push model.

1. Prometheus controls the scrape rate. In a push model, if 200 pods all push metrics simultaneously during a load spike, the monitoring server receives 200 simultaneous writes and can be overwhelmed. In Prometheus's pull model, Prometheus decides when to scrape each target and at what interval. A load spike in the application does not cause a load spike in the monitoring system.

2. Prometheus can detect target failure. In a push model, if a target stops pushing, the monitoring server has no way to know whether the target crashed or is just quiet. In Prometheus's pull model, if a target stops responding to scrape requests, Prometheus immediately knows — the scrape fails with a connection refused or timeout. Target down is an observable, alertable condition.

3. No monitoring agent to manage. Applications expose metrics as a simple HTTP endpoint — typically /metrics — and Prometheus calls that endpoint periodically. The application does not need a monitoring agent installed or configured. For NovaCrestBank's 50 banking microservices, this means: add one dependency to the Java application (Micrometer for Spring Boot metrics), expose the /metrics endpoint, create a ServiceMonitor — done. No per-service agent configuration, no firewall rule for the agent, no agent updates.

What Prometheus stores: Time-series data. A time series is a sequence of (timestamp, float64 value) pairs identified by a unique combination of a metric name and a set of label key-value pairs. For example: jvm_memory_used_bytes{area="heap", pod="payment-service-abc123", namespace="banking-production"} is one time series. Prometheus stores every scraped sample in its local TSDB (Time Series Database) on disk. Default retention: 15 days. Thanos extends this to months or years.

OBS-01 · Section 02
How Prometheus finds targets — ServiceMonitor and PodMonitor CRDs
Prometheus does not scrape all Kubernetes pods by default. It scrapes only what is explicitly declared via ServiceMonitor or PodMonitor objects. Understanding how target discovery works from ServiceMonitor → Service → Pod explains why each field in a ServiceMonitor is necessary.
The target discovery chain — how Prometheus gets from ServiceMonitor to pod IP
A ServiceMonitor specifies a namespace and a label selector. How does Prometheus turn that into an actual IP address and port to scrape?
The discovery chain is: ServiceMonitor selects Services → Service selects Pods → Pods expose the /metrics endpoint. Prometheus does not scrape Services directly — Services are a virtual abstraction. It resolves the Service to its Endpoint objects (which contain the actual pod IPs) and scrapes each pod individually.

Step by step: The Prometheus Operator watches for ServiceMonitor objects in namespaces it is configured to monitor. When it finds one, it reads the selector.matchLabels field to identify which Services to watch. It finds all Service objects in the specified namespaceSelector that match those labels. For each matching Service, Prometheus resolves the Service's Endpoints to get the list of current pod IPs and ports. Prometheus then scrapes each pod IP directly at the configured path (typically /metrics) and port.

Why this is important for NovaCrestBank: When the payment-service is scaled from 3 to 5 replicas, Prometheus automatically discovers the 2 new pods within the next service discovery refresh cycle (typically 30 seconds). No manual Prometheus configuration change is needed. The ServiceMonitor is the static declaration; the Endpoint list is dynamic. The pod scaling event propagates automatically to Prometheus's scrape target list.

The complete ServiceMonitor → Service → Pod chain for payment-service

1
ServiceMonitor object
Created in the banking-production namespace (or the monitoring namespace, depending on configuration). Declares: which namespaces to look in for Services (namespaceSelector), which label to use to identify the correct Service (selector.matchLabels: app: payment-service), which port on the Service carries the metrics (endpoints[0].port: metrics — the port name, not number), the scrape interval, and any relabelling rules. The Prometheus Operator watches all ServiceMonitor objects and generates Prometheus scrape configuration from them automatically.
2
Kubernetes Service object
The payment-service Service has labels app: payment-service — matching the ServiceMonitor selector. It defines a port named metrics on port 8080 (or a dedicated metrics port). The Service is the bridge between the static ServiceMonitor declaration and the dynamic pod list. As pods come and go, the Service's Endpoint list updates automatically.
3
Kubernetes Endpoints object
Created and maintained automatically by Kubernetes when a Service is created. Contains the current list of pod IPs and ports for all pods matching the Service's selector that are Ready. Prometheus resolves each Service to its Endpoints to get the actual IP:port pairs to scrape. This is the dynamic layer — it changes when pods are created, destroyed, or fail readiness checks.
4
Pod /metrics HTTP endpoint
Each payment-service pod exposes a /metrics HTTP endpoint on the configured port. Prometheus calls this endpoint every 30 seconds (the default scrape interval). The response is in Prometheus exposition format — plain text, one metric per line, with labels. Prometheus parses each line, attaches additional labels from the ServiceMonitor relabelling rules (namespace, pod name, job), and stores the resulting time series in its TSDB.
OBS-01 · Section 03
PromQL — reading and writing Prometheus queries from first principles
PromQL (Prometheus Query Language) is how engineers query metrics, write alert expressions, and build Grafana dashboards. Understanding its three core concepts — instant vectors, range vectors, and functions — makes every query readable rather than cryptic.
The three data types in PromQL — what each one is and when each is used
PromQL queries return different kinds of data depending on the query form. What are the differences and when does each appear?
An instant vector is the most common PromQL result — it is a set of time series, each with one value at the current moment. A range vector is a set of time series, each with multiple values over a time window. A scalar is a single numeric value with no labels.

Instant vector: jvm_memory_used_bytes{namespace="banking-production"} — returns one value per matching time series right now. Used for dashboards and current-state queries.

Range vector: jvm_memory_used_bytes{namespace="banking-production"}[5m] — returns 5 minutes of values per time series. The square brackets make it a range vector. Range vectors are inputs to functions like rate(), increase(), and changes() — you never display a range vector directly.

The key rule: a Counter must always be wrapped in rate(counter[5m]) before being displayed or used in an alert. A raw counter value is meaningless — it grows monotonically until the process restarts. The rate() function computes the per-second increase rate over the last 5 minutes, accounting for counter resets automatically. This is why alert expressions for HTTP error rates always look like rate(http_requests_total{status="500"}[5m]) rather than http_requests_total{status="500"}.

PromQL building blocks — the essential functions for banking monitoring

# ── 1. Rate — per-second rate of a counter over a time window ─────────────────
rate(http_requests_total{namespace="banking-production", status=~"5.."}[5m])
# Returns: per-second rate of 5xx responses over the last 5 minutes.
# status=~"5.." is a regex match — matches 500, 501, 502, 503, etc.
# The [5m] range tells Prometheus to use the last 5 minutes of data to compute the rate.
# Longer window = smoother (less noise) but slower to respond to spikes.
# 5 minutes is the standard for alert expressions.

# ── 2. Sum — aggregate across labels ──────────────────────────────────────────
sum(rate(http_requests_total{namespace="banking-production"}[5m])) by (pod)
# Aggregates request rate across all status codes, grouped by pod name.
# "by (pod)" = keep the pod label, discard all others.
# Result: one time series per pod showing its total request rate.

# ── 3. histogram_quantile — compute latency percentiles ───────────────────────
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket{namespace="banking-production"}[5m])) by (le, pod))
# P99 latency (the 99th percentile response time) per pod over the last 5 minutes.
# "le" = "less than or equal" — the histogram bucket label, must be kept in the "by".
# 0.99 = 99th percentile. Use 0.50 for median, 0.95 for P95, 0.999 for P99.9.
# This is the standard expression for SLO latency monitoring in banking APIs.

# ── 4. increase — total increase of a counter over a window ───────────────────
increase(http_requests_total{namespace="banking-production", status="500"}[1h])
# Total number of 500 errors in the last 1 hour (not per-second rate).
# Useful for alert conditions like "more than 10 errors in the last hour".

# ── 5. predict_linear — forecast metric trend ─────────────────────────────────
predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[6h], 4 * 3600)
# Linear extrapolation: if disk usage continues at the current 6-hour trend,
# what will the available bytes be in 4 hours?
# Use in alerts: "if disk fills up in less than 4 hours, alert now."
# This is the standard pattern for disk space alerts — alert before the disk is full,
# not when it is full. Filling at 2 AM with 4 hours of lead time wakes the on-call engineer
# with time to act. Alerting when full at 2 AM means the damage is already done.

# ── 6. absent — alert when a metric disappears ────────────────────────────────
absent(up{job="payment-service", namespace="banking-production"})
# Returns a value when NO time series matching the selector exist.
# Used to alert when a scrape target has completely disappeared from discovery.
# If all payment-service pods are down and no endpoints exist, Prometheus stops
# seeing the "up" metric entirely — absent() fires.

Alertmanager — routing, grouping, silencing, and inhibition explained
Prometheus evaluates alert rules and fires alerts. Alertmanager receives those alerts and decides what to do with them — which team to notify, on which channel, grouped how, and when to silence or suppress. Understanding Alertmanager's four features makes alert fatigue avoidable in production.
Alertmanager's job — the difference between firing an alert and sending a notification
Prometheus evaluates rules and fires alerts. What does Alertmanager add? Why is it not enough to have Prometheus send emails directly?
Prometheus fires an alert when an expression evaluates to true for a configured duration. Without Alertmanager, a separate notification would be sent for every alert — every 30 seconds while the condition persists. A single incident (for example, 3 nodes under memory pressure) could generate thousands of email notifications before anyone acknowledges it. Alertmanager exists to make alerts useful rather than overwhelming.

Grouping: Alertmanager groups related alerts into a single notification. If all three masters and two workers simultaneously have high memory pressure (likely caused by a single event), Alertmanager sends one notification: "5 nodes are under memory pressure" rather than 5 separate notifications. The grouping key is configurable — NovaCrestBank groups by alertname and namespace.

Routing: Different alerts go to different destinations. etcd quorum loss → PagerDuty P1 (wakes the on-call engineer). Certificate expiring in 14 days → email to the platform team. Node memory pressure → PagerDuty P2. The routing tree in Alertmanager's configuration determines the destination and urgency for each alert based on its labels.

Silencing: During a planned maintenance window (OCP upgrade, vSphere maintenance), alerts that are expected fire and would be noisy. A silence tells Alertmanager: "from 2026-04-20 22:00 to 23:00, suppress all alerts with alertname=NodeNotReady." The underlying alert still fires; Alertmanager just does not send the notification while the silence is active.

Inhibition: If a cluster-level alert is firing (for example, "all masters are unreachable"), the many downstream alerts it causes (etcd health failure, API server unavailable, etc.) are redundant — they are all symptoms of the same root cause. Inhibition rules tell Alertmanager: "if alert X is firing, suppress alert Y." This prevents notification floods where the root cause arrives with 50 symptom alerts attached.

The alert lifecycle — from expression to notification

Inactive
Alert expression evaluates to false
The rule's expression does not match any time series, or the resulting value is 0 or NaN. No alert is firing. Prometheus evaluates the expression every evaluationInterval (default: 30 seconds) and moves the alert to Pending when the expression first evaluates to true.
Pending
Expression true but for duration not yet met
The expression evaluated to true but the alert has a for: 5m clause — it must remain true for 5 continuous minutes before firing. This prevents false alarms from momentary spikes. A payment-service pod with a 3-second CPU spike at 100% would trigger the expression but not reach Pending→Firing if the CPU drops before the for duration expires.
Firing
Expression true for the full for duration
Prometheus sends the alert to Alertmanager. Alertmanager groups it with related firing alerts, waits for the group_wait period (time to collect more alerts into the same group before sending the first notification), then sends the notification. While the alert remains firing, Alertmanager sends repeat notifications at the repeat_interval frequency until the alert resolves or is silenced.
Resolved
Expression evaluates to false — alert resolved
Prometheus marks the alert as resolved and sends the resolved state to Alertmanager. Alertmanager sends a resolution notification ("Alert resolved: NodeMemoryPressure on infra-0 has recovered") — this closes the PagerDuty incident and notifies the team that the condition is gone.

Thanos — long-term metric storage and global query
OCP's built-in Prometheus retains metrics for 15 days by default. Thanos extends this to months or years by uploading Prometheus TSDB blocks to object storage and providing a unified query interface. For NovaCrestBank, Thanos satisfies the banking requirement to retain operational metrics for audit and trend analysis.
What Thanos adds — and what it does not change
Prometheus already stores metrics locally. Why add Thanos? And does it replace Prometheus or sit alongside it?
Thanos sits alongside Prometheus — it does not replace it. Prometheus continues to scrape targets, evaluate alert rules, and serve real-time queries. Thanos adds three capabilities Prometheus cannot provide alone.

1. Long-term storage. Prometheus's local TSDB has a configurable retention period — OCP's default is 15 days, and extending it indefinitely requires an ever-growing vSAN PVC. Thanos Sidecar runs alongside each Prometheus pod and uploads TSDB blocks (2-hour chunks of compacted metric data) to object storage (S3-compatible, such as AWS S3 or MinIO on-premises). Once uploaded, Prometheus can delete the local blocks — the vSAN PVC stays bounded while Thanos queries object storage for historical data.

2. Long-term retention for compliance. NovaCrestBank needs operational metrics retained for 90 days minimum for FFIEC examination readiness — being able to show that the cluster was healthy and performing within SLOs during the examination period. Thanos provides queryable access to these historical metrics without keeping 90 days of data on expensive vSAN storage.

3. Unified query interface. The Thanos Querier component provides a single PromQL endpoint that queries both live Prometheus data (from the last 15 days) and historical data from object storage (older than 15 days) transparently. The engineer makes one query — Thanos routes it to Prometheus for recent data and to object storage for older data, deduplicates the results, and returns a unified response.

For NovaCrestBank's current single-cluster deployment: the immediate value is long-term storage and compliance retention. The global query capability becomes valuable when NovaCrestBank adds more OCP clusters.

The cluster-monitoring-config ConfigMap (Section 08) enables and configures Thanos Sidecar on the built-in Prometheus pods. What NovaCrestBank adds in this guide is: enabling Thanos long-term storage by configuring an object storage backend, which is beyond the default OCP monitoring configuration.

The OCP monitoring stack — what is installed and what manages what
OpenShift's monitoring stack is not a single tool — it is a collection of components managed by the Cluster Monitoring Operator (CMO). Understanding the division between the platform stack and the user workload stack prevents the most common monitoring configuration mistakes.
Two Prometheus instances — platform metrics vs application metrics
OCP appears to have two Prometheus instances. What is the difference and why are they separate?
OCP installs two separate Prometheus stacks, managed by the same Cluster Monitoring Operator but serving different purposes with different access controls.

Platform Prometheus (in the openshift-monitoring namespace) scrapes all OCP platform components: etcd, kube-apiserver, kube-controller-manager, the OCP router, the vSphere CSI driver, node metrics (CPU, memory, disk), and all other OCP internal components. Only cluster administrators can query this Prometheus. It contains sensitive operational data about the cluster infrastructure. It runs on infra nodes and is managed entirely by the CMO — engineers do not create ServiceMonitors in openshift-monitoring.

User Workload Prometheus (in the openshift-user-workload-monitoring namespace) scrapes banking application metrics — whatever ServiceMonitors and PodMonitors application teams create in their own namespaces. Application teams can query their own namespace's metrics without accessing the platform Prometheus. Developers can create ServiceMonitors in their application namespaces and see their metrics in Prometheus without any cluster-admin intervention.

The key rule: never create ServiceMonitors in the openshift-monitoring namespace. They are ignored by the CMO. Create ServiceMonitors in the application namespace (banking-production) and the User Workload Prometheus picks them up automatically.

Why monitoring lives on infra nodes — the isolation architecture
The monitoring stack was moved to infra nodes in OCP-VCF-02 Section 10. This section explains precisely why — what goes wrong if monitoring runs on worker nodes, and how infra node isolation protects monitoring reliability in a banking environment.
What happens when monitoring and application workloads share the same nodes
The monitoring stack was moved to infra nodes in OCP-VCF-02. Why is this isolation architecturally important rather than just a nice-to-have?
Prometheus is the most memory-hungry component in a production OCP cluster. A Prometheus instance monitoring a banking cluster with 50 microservices, each with 20 metrics, scraping every 30 seconds, retains 15 days of data — this easily consumes 30–80 GB of RAM. If Prometheus runs on a worker node shared with banking application pods, two specific failure modes emerge.

Failure mode 1 — application load starves monitoring. During a banking load spike (end of month batch processing, a flash sale), banking application pods consume increased CPU and memory. If Prometheus is on the same node, it competes for resources. The kubelet begins OOM-killing pods. Prometheus — which is a constant-memory process that does not scale back during application spikes — gets killed first, because it often has a higher memory footprint than any individual application pod. The cluster loses monitoring exactly when the load spike is most likely to cause an incident that needs monitoring to diagnose.

Failure mode 2 — monitoring load affects banking applications. Prometheus's periodic bulk reads from its TSDB (for rule evaluation, for Grafana dashboard queries, for Thanos block uploads) cause I/O spikes. If this I/O competes with a banking application's database I/O on the same vSAN datastore path, banking transaction latency increases during routine monitoring operations.

Infra node isolation eliminates both failure modes. Infra nodes run only monitoring (Prometheus, Alertmanager, Grafana), logging (LokiStack), and Ingress Router. Banking application pods cannot be scheduled on infra nodes (NoSchedule taint). Monitoring pods cannot be displaced by banking application load. The two workload categories have independent resource pools with independent scheduling decisions. If a load spike causes an infra node resource constraint, the only consequence is degraded monitoring — which is acceptable. The banking applications on worker nodes are unaffected.

OBS-01 · Section 08
cluster-monitoring-config — the ConfigMap that controls the entire monitoring stack
The cluster-monitoring-config ConfigMap in openshift-monitoring is the single point of control for the entire OCP monitoring stack. Every configuration choice — infra node placement, storage, retention, Thanos, user workload monitoring — is expressed here. Every field is explained.
Why a ConfigMap controls the monitoring stack instead of direct CRD configuration
The Cluster Monitoring Operator reads this ConfigMap and reconciles all monitoring components to match the declared configuration. Engineers never directly edit Prometheus's StatefulSet, Alertmanager's Deployment, or Grafana's ConfigMap — the CMO owns those objects and would revert any direct edits. The ConfigMap is the correct interface. This is the same pattern as the Argo CD ArgoCD CR and the cert-manager ClusterIssuer — you configure the operator, not its outputs.
cluster-monitoring-config ConfigMap — every field explained
apiVersion: v1
kind: ConfigMap
metadata:
  name: cluster-monitoring-config
    # This exact name is required — the CMO watches for this specific ConfigMap.
    # Any other name is ignored. Misspelling means no configuration is applied.
  namespace: openshift-monitoring
    # Must be in openshift-monitoring — the CMO only reads from this namespace.
data:
  config.yaml: |
        # Every key under config.yaml targets a specific monitoring component.
        # The key is the component name; the value is its configuration.

        prometheusK8s:                ← the main platform Prometheus
          nodeSelector:
            node-role.kubernetes.io/infra: ""
            # Schedule Prometheus pods on infra nodes — nodes with this label.
              # This label is set by the MachineSet in OCP-VCF-02.
          tolerations:
            - key: node-role.kubernetes.io/infra
              effect: NoSchedule
            # Toleration for the infra node's NoSchedule taint.
              # Without this, Prometheus pods cannot be scheduled on infra nodes
              # even with the correct nodeSelector — the taint blocks them.
          retention: 15d

              # 15d is the OCP default. Reduce to 7d if Thanos is configured for
              # long-term storage (Thanos Sidecar uploads blocks to object storage

              # Works together with retention: whichever limit is hit first wins.
              # 40 GB is appropriate for a banking cluster with 50+ services scraping
              # ~1000 metrics each at 30-second intervals with 15-day retention.
          volumeClaimTemplate:
            spec:
              storageClassName: vsan-standard
                  # vsan-standard: Delete policy, RAID-5 FTT:1.
                    # Prometheus TSDB data can be rebuilt from re-scraping if lost —
                    # it is not permanent record data like ACS alerts or application data.
                    # vsan-standard (not vsan-gold) is appropriate: faster, less overhead,
                    # and the data is reconstructible.
              resources:
                requests:
                  storage: 50Gi
                  # 50 GB per Prometheus pod (2 pods = 100 GB total vSAN).
                    # For 15d retention with 40 GB limit: 50 GB provides headroom.
                    # Size conservatively: Prometheus compresses data well but
                    # a cluster under heavy load generates more samples per interval.

        alertmanagerMain:               ← Alertmanager cluster (3 pods)

            node-role.kubernetes.io/infra: ""

            - key: node-role.kubernetes.io/infra
              effect: NoSchedule
            # Same infra node placement as Prometheus.
              # Alertmanager runs as 3-pod cluster (HA). If monitoring is needed
              # during an infra node failure, the remaining 2 pods maintain quorum.
          volumeClaimTemplate:
            spec:
              storageClassName: vsan-standard
              resources:
                requests:
                  storage: 2Gi
                  # Alertmanager's PVC stores: silences, notification logs, and
                    # alert correlation state. 2 GB is generous — typical usage is
                    # under 100 MB. But use a PVC rather than emptyDir so silences
                    # survive pod restarts (a silence created during an incident
                    # should persist across the Alertmanager rolling restart).

            - key: node-role.kubernetes.io/infra
              effect: NoSchedule

          # Port names (not numbers) are used here because: if the port number changes

          # The payment-service Service must have a port named "metrics".
      path: /actuator/prometheus

      interval: 30s

          # For high-frequency trading or payment processing latency monitoring,

          # For most banking microservices, 30s is appropriate.
      scheme: https

          # The payment-service uses mTLS (cert-manager Certificate from OCP-OPS-02).
          # Scraping over plain HTTP would expose metric data in transit — unacceptable
          # for a banking environment. The cert-manager-issued certificate on the

      tlsConfig:

OBS-01 · Section 11
PodMonitor CRD — when to use it instead of ServiceMonitor
ServiceMonitor requires a Kubernetes Service. PodMonitor scrapes pods directly without a Service intermediary. Understanding when to use each prevents confusion when a service does not have a Kubernetes Service object.
ServiceMonitor vs PodMonitor — the correct choice for each scenario
ServiceMonitor is used for most applications. When does a PodMonitor make more sense?
Use ServiceMonitor for 90% of cases — any application that has a Kubernetes Service object (which is almost everything in a banking microservices environment). Use PodMonitor in three specific situations.

1. The application exposes metrics on a port that is not in the Service. Some applications expose a dedicated metrics port (for example, port 9090 for metrics, separate from port 8080 for the API). If the Service only exposes port 8080, a ServiceMonitor cannot reach port 9090. A PodMonitor scrapes pods directly and can specify any port regardless of what the Service exposes.

2. Each pod exposes different metrics that should not be aggregated. In a StatefulSet where pod-0 is the primary and pod-1 is a replica, each pod has a distinct role and distinct metrics. A ServiceMonitor routes through the Service load balancer — scrapes land on random pods. A PodMonitor discovers each pod individually with its unique pod labels, enabling per-pod dashboard rows.

3. No Service exists. DaemonSets running infrastructure agents (node exporters, log collectors) often have no Service — they run directly on nodes. A PodMonitor can discover these pods by their labels without requiring a Service to exist.

For NovaCrestBank's banking microservices, ServiceMonitor is always correct. PodMonitor would be used for the OCP node exporter, the vSphere CSI metrics, or custom DaemonSet-based monitoring agents.

OBS-01 · Section 14
etcd quorum loss alerts — the cluster's most critical signal
etcd is the database that stores all Kubernetes state. etcd quorum loss means the API server stops accepting writes — no deployments, no ConfigMap updates, no Secret changes. For NovaCrestBank, this means no banking application deployments can proceed. This is the cluster's highest-priority alert.
Understanding etcd quorum — why 2-of-3 is the critical threshold
The alert fires when 2 etcd members lose their leader. Why 2, not 1 or 3?
etcd uses the Raft consensus algorithm, which requires a majority of members to agree before committing any write. In a 3-member etcd cluster, the majority is 2. If 2 members can agree, quorum is maintained and writes proceed. If only 1 member is available, it cannot form a majority by itself — quorum is lost and the cluster stops accepting writes to prevent split-brain scenarios.

The scenarios:
1 member unreachable → quorum maintained (2 of 3 still agree) → API writes continue → alert at WARNING severity, not critical. The system degrades gracefully.
2 members unreachable → quorum lost (only 1 of 3 available, no majority) → API server enters read-only mode → NO new deployments, no OCP operator reconciliation, no scaling. The cluster is effectively frozen. This is the CRITICAL alert that wakes the on-call engineer at 2 AM regardless of day.

For NovaCrestBank, this translates directly to banking operations: if etcd loses quorum during business hours, no application can be deployed, no Kubernetes Secrets can be rotated, no OCP operators can reconcile — including cert-manager (certificate renewals stall) and the Logging Operator (log forwarding configuration stalls). The banking applications already running continue running — their pods are not affected — but any operational change to the cluster is impossible.

Node memory and disk pressure alerts
Node-level resource exhaustion is a leading cause of production incidents. The following rules alert in advance — before the kubelet starts OOM-killing pods or before the node root filesystem fills up and the node becomes unhealthy.
PrometheusRule — node memory, disk, and pressure alerts
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: ncb-node-alerts
  namespace: openshift-monitoring
  labels: {role: alert-rules}
spec:
  groups:
    - name: ncb.nodes
      rules:

        - alert: NCBNodeMemoryHigh
          expr: |
                (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
              / node_memory_MemTotal_bytes > 0.85
            # (total - available) / total = used fraction.
                # > 0.85 = more than 85% of node memory is in use.
                # 85% is the warning threshold — not 90% or 95%.
                # By the time a node reaches 90%, the kubelet is already applying
                # memory eviction pressure and may start killing pods.
                # 85% gives 10+ minutes to investigate and take action.
                # node_memory_MemAvailable_bytes is more accurate than MemFree —
                # it includes reclaimable memory (cache, buffers) as "available".
          for: 10m
          labels: {severity: warning, team: platform}
          annotations:
            summary: Node {{ $labels.node }} memory utilisation is {{ $value | humanizePercentage }}
            description: "Node {{ $labels.instance }} is using {{ $value | humanizePercentage }} of memory. Consider draining and investigating for memory leaks in pods."

        - alert: NCBNodeMemoryCritical
          expr: |
                (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
              / node_memory_MemTotal_bytes > 0.93
            # > 93%: the kubelet's eviction threshold on most OCP configurations.
                # Above this, the kubelet begins OOM-killing pods.
                # This alert fires CONCURRENTLY with NodeMemoryHigh — they share the same
                # node but have different severity levels and different routing (critical
                # routes to PagerDuty P1, warning routes to email).
          for: 5m
          labels: {severity: critical, team: platform}
          annotations:
            summary: CRITICAL — Node {{ $labels.node }} OOM eviction imminent
            description: "Node {{ $labels.instance }} is at {{ $value | humanizePercentage }} memory. The kubelet is OOM-killing pods. Drain the node immediately: oc adm drain {{ $labels.node }} --ignore-daemonsets --delete-emptydir-data"

        - alert: NCBNodeDiskFillingUp
          expr: |
                predict_linear(
                  node_filesystem_avail_bytes{mountpoint="/", fstype!="tmpfs"}[6h],
                  4 * 3600
                ) < 0
            # predict_linear: "if disk usage continues at the current 6-hour trend,
                # will the disk fill up within 4 hours?"
                # < 0 means: the linear prediction goes below zero within 4 hours.
                # This fires BEFORE the disk is full — with 4 hours of lead time.
                # Alerting when disk is FULL at 2 AM is useless.
                # Alerting at 10 PM that the disk will fill by 2 AM gives the on-call
                # engineer time to either clear space or trigger a maintenance response.
          for: 30m
          labels: {severity: warning, team: platform}
          annotations:
            summary: Node {{ $labels.instance }} root filesystem will fill within 4 hours
            description: "Based on the last 6 hours of trend, node {{ $labels.instance }} root filesystem ({{ $labels.device }}) will be full within 4 hours. Current available: {{ $labels.mountpoint }} available bytes. Clean up container image layers or large log files."

        - alert: NCBNodeNotReady
          expr: |
            kube_node_status_condition{condition="Ready", status="true"} == 0
            # A node is no longer Ready — the kubelet is not reporting healthy to the API server.
                # This fires for any node: master, infra, or worker.
                # Note: uses kube_node_status_condition from kube-state-metrics — NOT
                # from the node_exporter. kube-state-metrics reads from the Kubernetes API,
                # which is authoritative for node status. The node_exporter reads from the
                # node itself — but if a node is unreachable, node_exporter data disappears.
          for: 5m
          labels: {severity: critical, team: platform}
          annotations:
            summary: Node {{ $labels.node }} is NotReady
            description: "Kubernetes node {{ $labels.node }} has been NotReady for 5 minutes. Pods may be evicted. Check node status: oc get node {{ $labels.node }} and oc describe node {{ $labels.node }}"
OBS-01 · Section 16
Certificate expiry alerts — 30-day advance warning
cert-manager automatically renews certificates 30 days before expiry (OCP-OPS-02). These Prometheus alerts are the safety net — if cert-manager's renewal fails silently, these alerts fire while there is still time to fix the root cause manually.
Why alert at 30 days when cert-manager renews at 30 days
cert-manager is configured to renew certificates at 30 days before expiry. Why also have a Prometheus alert at 30 days? What situation does the alert catch that cert-manager does not?
cert-manager renews certificates automatically — but if cert-manager itself is broken, the renewal does not happen. The Prometheus alert fires independently of cert-manager, watching the actual certificate expiry from the certmanager_certificate_expiration_timestamp_seconds metric. If cert-manager's ADCS issuer pod crashes and stays down for two weeks, cert-manager will not renew any certificates — but Prometheus will see the expiry timestamps approaching and fire the alert at 14 days with another critical alert at 3 days.

The two are different types of controls: cert-manager renewal is a preventive control (tries to prevent expiry from ever being reached). The Prometheus alert is a detective control (detects when the preventive control has failed and expiry is genuinely approaching). Both are necessary for a banking environment where an expired TLS certificate causes immediate service failures.
PrometheusRule — certificate expiry alerts (complement to cert-manager renewal)
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: ncb-certificate-alerts
  namespace: cert-manager
    # This PrometheusRule is in the cert-manager namespace.
    # The cert-manager ServiceMonitor (from OCP-OPS-02 Section 16) exposes
    # certmanager_* metrics to the User Workload Prometheus.
    # These rules are evaluated by the User Workload Prometheus.
  labels: {role: alert-rules}
spec:
  groups:
    - name: ncb.certificates
      rules:

        - alert: NCBCertificateExpiringSoon
          expr: |
                certmanager_certificate_expiration_timestamp_seconds
              - time() < 14 * 24 * 3600
            # certmanager_certificate_expiration_timestamp_seconds: the Unix timestamp
                # when each cert-manager-managed certificate expires. Exposed per Certificate object.
                # - time(): subtract the current Unix timestamp → seconds until expiry.
                # < 14 * 24 * 3600: fire when less than 14 days (1,209,600 seconds) remain.
                # 14 days: well after cert-manager should have renewed (30 days).
                # If this fires, cert-manager's renewal has been failing for ~16 days.
                # Still enough time to investigate and fix before the certificate expires.
          for: 1h
          labels: {severity: warning, team: platform}
          annotations:
            summary: Certificate {{ $labels.name }} in {{ $labels.namespace }} expires in {{ $value | humanizeDuration }}
            description: "cert-manager automatic renewal has not succeeded for certificate {{ $labels.name }}. Check the CertificateRequest status and the ADCS issuer pod logs in the cert-manager namespace."
            runbook: https://wiki.novacrestbank.local/runbooks/cert-manager-renewal-failure

        - alert: NCBCertificateExpiryCritical
          expr: |
                certmanager_certificate_expiration_timestamp_seconds
              - time() < 3 * 24 * 3600
            # < 3 days: P1 incident trigger. TLS service failure is imminent.
                # At this point, manual certificate renewal is required immediately.
                # Force immediate renewal: delete the Certificate object and re-apply it.
                # cert-manager creates a new CertificateRequest immediately on creation.
          for: 5m
          labels: {severity: critical, team: platform}
          annotations:
            summary: CRITICAL — Certificate {{ $labels.name }} expires in {{ $value | humanizeDuration }}
            description: "Immediate manual renewal required. Service TLS failures will occur when the certificate expires. Force renewal: oc delete certificate {{ $labels.name }} -n {{ $labels.namespace }} and re-apply the Certificate object."
            runbook: https://wiki.novacrestbank.local/runbooks/cert-emergency-renewal

        - alert: NCBCertificateReadyFalse
          expr: |
            certmanager_certificate_ready_status{condition="False"} == 1
            # A cert-manager Certificate object has Ready=False — the issuer failed
                # to issue the certificate. This fires immediately when a Certificate
                # transitions to the failed state — before any expiry threat.
                # Useful for catching new certificate issuance failures (not just renewals).
          for: 15m
          labels: {severity: warning, team: platform}
          annotations:
            summary: Certificate {{ $labels.name }} in {{ $labels.namespace }} is not Ready
            description: "Certificate {{ $labels.name }} has been in Ready=False state for 15 minutes. The ADCS issuer is failing to issue this certificate. Check: oc describe certificate {{ $labels.name }} -n {{ $labels.namespace }}"

Alertmanager's routing tree determines who gets notified and on which channel for each alert. NovaCrestBank routes critical alerts to PagerDuty (wakes the on-call engineer) and warning alerts to email (reviewed during business hours).

global:
  resolve_timeout: 5m
    # How long Alertmanager waits after an alert stops firing before
    # sending a "resolved" notification. 5 minutes prevents premature
    # resolution notifications for flapping alerts.
  pagerduty_url: https://events.pagerduty.com/v2/enqueue
  smtp_smarthost: smtp-relay.novacrestbank.local:587
  smtp_from: ocp-monitoring@novacrestbank.local
  smtp_require_tls: true

route:
  group_by: [alertname, namespace]
    # Group alerts that share the same alertname AND namespace into one notification.
    # This prevents: 5 separate notifications for NodeMemoryHigh on 5 nodes.
    # Instead: one notification "NodeMemoryHigh on 5 nodes in openshift-monitoring".
  group_wait: 30s
    # After the first alert in a new group, wait 30s before sending the notification.
    # This collects related alerts that fire within 30 seconds of each other
    # into a single grouped notification rather than sending one immediately.
  group_interval: 5m
    # Minimum time between notifications for the same group.
    # If new alerts join the group, wait 5 minutes before sending another notification.
  repeat_interval: 4h
    # If an alert is still firing after 4 hours, send another notification.
    # This reminds the team that an unresolved alert is still active.
    # For critical alerts, a 4-hour repeat is appropriate (not every 30 minutes —
    # that causes alert fatigue).
  receiver: platform-email
    # Default receiver — any alert not matching a more specific route goes here.
  routes:
    - match:
        severity: critical
          # Match: any alert with severity=critical label
      receiver: pagerduty-p1
          # Route to PagerDuty P1 — wakes the on-call engineer via mobile alert.
            # Critical alerts: etcd quorum loss, node OOM, certificate expiry <3 days.
      group_wait: 10s
          # Critical alerts: shorter group wait (10s not 30s) — send faster.
      repeat_interval: 1h
          # Repeat every 1 hour for critical alerts — more aggressive reminders.
    - match:
        severity: warning
        team: platform
      receiver: platform-email
          # Platform warning alerts → email to platform engineering team.
            # Reviewed during business hours. Not waking anyone up at night.

Grafana — dashboards for the NovaCrestBank operations team
Grafana provides the visual layer on top of Prometheus. The OCP console includes a basic Grafana-compatible dashboard view. For production banking monitoring, a dedicated Grafana deployment provides richer dashboards for the operations team.
OCP built-in dashboards vs a dedicated Grafana deployment
OCP has an Observe → Dashboards section in the console. Why would NovaCrestBank also deploy a dedicated Grafana instance?
The OCP console's built-in dashboards are read-only — they are shipped by Red Hat and cannot be customised. Engineers cannot add a dashboard showing payment-service P99 latency, transaction throughput, or the four golden signals for the banking microservices. The built-in dashboards cover platform metrics (etcd, API server, node CPU/memory) but not application-level metrics.

A dedicated Grafana deployment (OCP 4.13+ uses GrafanaOperator from OperatorHub) provides: custom dashboards for banking application SLOs (P99 latency, error rate, transaction throughput), composite dashboards correlating application performance with infrastructure health (is the payment-service latency spike correlated with node memory pressure?), and dashboards for the operations team's weekly SLO review. Grafana connects to the Thanos Querier endpoint as its data source, giving access to both platform and application metrics from a single query interface.
