OCP-VCF-02 · Section 02
Launch the installer
One command starts everything. Once it runs, VMs are created in vcenter-ocp within seconds. Keep Terminal 1 open for the full 35–45 minutes — never close it during the install.
What the installer does the moment you run it — the sequence
The installer reads install-config.yaml, then deletes it (this is why the backup is mandatory), connects to vcenter-ocp using the certThumbprint, uploads the RHCOS OVA to vsanDatastore if not cached, creates the bootstrap VM on seg-ocp-nodes, then creates the 3 master VMs. All VM NICs are attached to seg-ocp-nodes. The bootstrap VM boots, pulls its ignition config from the installer, starts a temporary API server, and waits for masters to join etcd. The installer then waits — up to 30 minutes — for the Kubernetes API to respond on 10.50.0.200:6443 (the NSX LB VIP).

OCP-VCF-02 · Section 03
Bootstrap phase — what actually happens and how to watch it
The bootstrap phase is the most opaque part of an OpenShift install. This section explains exactly what is happening inside the bootstrap VM and how to verify progress in real time.
The key insight — what the bootstrap VM actually does
The bootstrap VM is a temporary control plane. It exists for one purpose: to help the three permanent master nodes form their own etcd cluster and take over the Kubernetes API. The bootstrap VM runs a temporary etcd, a temporary kube-apiserver, and a temporary Machine Config Server. The masters boot, download their ignition config from the bootstrap, start their own etcd instances, and once all three masters have joined etcd quorum, the bootstrap declares its job done and is destroyed. From that point forward, the three masters own the cluster permanently.

Phase timeline — what is happening at each minute

T+0
Bootstrap boots RHCOS
Bootstrap VM powers on, RHCOS boots from the VMDK. The bootstrap ignition config (written by the installer before it was deleted) is baked into the VM via vApp properties. RHCOS reads this config and starts bootkube.service.
T+3
Bootstrap starts temporary etcd + API
bootkube.service starts a single-node temporary etcd and a temporary kube-apiserver. This is not the permanent cluster — it is a scaffold. The bootstrap also starts the Machine Config Server on port 22623, which masters will contact to get their ignition configs.
T+5
Masters boot and contact bootstrap
The three master VMs power on. Each master's RHCOS boots, contacts api-int.ocp.novacrestbank.local:22623 (the Machine Config Server on the bootstrap) to download its ignition config. This is why the api-int DNS record is mandatory — masters use it, not api.
T+10
etcd quorum forms — the critical milestone
Each master starts its own etcd instance and joins the cluster. Once all three masters have joined, etcd has a 3-member quorum. This is the point of no return — the cluster is now self-sustaining and will survive the loss of the bootstrap VM.
T+22
Bootstrap complete — deleted
Masters have taken over the API. The installer destroys the bootstrap VM in vcenter-ocp. The NSX LB pool NCB-Pool-OCP-API now points only to the three masters. The bootstrap VM is gone forever.

OCP-VCF-02 · Section 04
Masters coming up — monitoring the permanent control plane
After etcd quorum forms, the permanent cluster API takes over from the bootstrap. Monitor master health and approve any pending CSRs before workers can join.
Why CSR approval matters — and when it applies
Every node that joins an OpenShift cluster submits a Certificate Signing Request to prove its identity. The cluster issues each node a client certificate, which it uses to communicate with the API server securely. In IPI installs, the machine-approver controller handles this automatically — but it may not be running yet during the early master phase. The rule: always run oc get csr every few minutes and approve anything Pending. A worker stuck in Pending has an unapproved CSR 90% of the time.

OCP-VCF-02 · Section 07
MachineSet — what it is, what it does, and why you need it
Before creating infra nodes, understand the object that manages them. MachineSet is one of the most important OpenShift concepts for infrastructure teams — and one of the most skipped in documentation.
The key insight — MachineSet is OpenShift's version of "desired state" for VMs
In traditional VMware administration, you create VMs manually in vCenter. In OpenShift on VCF, you declare the desired number of nodes in a MachineSet object, and the Machine API Operator creates, manages, and deletes VMs in vcenter-ocp on your behalf. If a VM is accidentally deleted in vCenter, the Machine API Operator recreates it. If you change the replica count from 3 to 5, two new VMs appear in vCenter. This is infrastructure-as-code for your cluster nodes — the VMs are a consequence of the Kubernetes object, not the other way around.

The four objects in the Machine API — what each one does

ObjectWhat it isWho creates itAnalogy
MachineSet A template + desired replica count for a group of nodes. Defines the VM spec (vCPU, RAM, disk, network, datacenter, datastore) and how many nodes of that spec to maintain. The controller watches the actual count and reconciles to the desired count. Infrastructure engineer (you) A ReplicaSet for VMs — like how a Deployment maintains pod count, a MachineSet maintains VM count
Machine Represents one individual node VM. The MachineSet creates one Machine object per replica. Each Machine object tracks the lifecycle: Provisioning → Running → Deleting. One Machine = one VM in vcenter-ocp = one node in the cluster. Created automatically by MachineSet A single Pod created by a ReplicaSet
Node The Kubernetes node object. Created automatically when a Machine's VM boots RHCOS and the kubelet registers with the API server. The Machine and Node are linked — deleting the Machine deletes the Node and the VM. Created automatically when Machine's VM joins The running container from the pod perspective
providerSpec The vSphere-specific configuration inside a MachineSet — VM hardware settings (numCPUs, memoryMiB, diskGiB), vCenter topology (datacenter, cluster, datastore, resourcePool), and the network (networkName). This is what tells the Machine API "go create this VM in vcenter-ocp on this segment." Specified by infrastructure engineer inside MachineSet The VM hardware spec sheet

What workspace means inside providerSpec

workspace — the vCenter topology that locates where to create the VM
The workspace field in providerSpec is how the Machine API Operator knows which vCenter, which datacenter, which cluster, and which datastore to create the VM in. In VCF, this always points to vcenter-ocp (not vcenter-mgmt) because OCP VMs belong to the OCP Workload Domain. Each field in workspace maps directly to an object path in vcenter-ocp: datacenter is the vCenter datacenter object, folder is the VM folder where VMs will appear, datastore is the full inventory path to vsanDatastore, resourcePool is the cluster's default resource pool, and server is the vcenter-ocp FQDN. Without a correct workspace, the Machine API cannot find where to create VMs and every Machine will stay in Provisioning state indefinitely.

What networkName means in the VCF+NSX context

The networkName field inside network.devices is the name of the vCenter network object to attach the VM's NIC to. In standalone vSphere this would be a vDS portgroup name. In VCF+NSX this is the name of the NSX overlay segment — seg-ocp-nodes. The Machine API Operator attaches each new node VM's NIC to this segment in vcenter-ocp, putting it on the 10.50.0.0/24 subnet where the NSX LB, DFW, and routing are already configured.

The infra node role — why infra nodes exist

OpenShift platform components — the Ingress Router (HAProxy), the internal image registry, and the monitoring stack (Prometheus, Alertmanager) — run as pods. By default they run on worker nodes, competing with application pods for CPU and memory. Infra nodes are worker nodes with the label node-role.kubernetes.io/infra="" and a NoSchedule taint. The label allows platform components (which have a matching nodeSelector) to schedule there. The taint prevents application pods (which have no matching toleration) from landing there. The result: platform components get dedicated nodes, application pods get clean workers, and Red Hat's licensing model does not charge subscription cores for infra nodes running only platform workloads.

Before creating StorageClasses, understand what each field controls. A StorageClass is a template — every PersistentVolume provisioned from it inherits its properties.
The key insight — a StorageClass is the developer's storage menu
The vSphere CSI driver is already running in the cluster — the IPI installer deployed it automatically. It also created one default StorageClass called thin-csi. But one StorageClass is not enough for a banking environment. A database holding transaction records needs different properties than a CI/CD build workspace. StorageClasses let the infrastructure team codify those different properties once, and developers simply request storage by name — they never see vSAN policies, vCenter objects, or provisioner configuration.

FieldWhat it controlsOptions and meaning
provisioner Which CSI driver handles provisioning. This routes the PVC to the right storage system. csi.vsphere.volume = vSphere CSI driver (creates VMDKs on vSAN). Every StorageClass backed by vSAN must use this provisioner.
reclaimPolicy What happens to the underlying VMDK when the PVC is deleted. Delete: the VMDK is deleted from vSAN immediately when the PVC is removed. Use for non-critical, ephemeral data. Verify dynamic provisioning — end-to-end test
Create a test PVC for each StorageClass and confirm a vSAN VMDK is provisioned in vcenter-ocp. Watch the PVC lifecycle from Pending through Bound.
Why you see Pending before Bound with WaitForFirstConsumer
When you create a PVC with WaitForFirstConsumer, the status stays Pending — not because anything is wrong, but because the CSI driver is waiting to find out which node the pod will be scheduled on. The VMDK placement on vSAN needs to be co-located with the node's ESXi host for performance. Once you create a Pod that references the PVC, the scheduler places the Pod on a node, tells the CSI driver the node's location, and the CSI driver creates the VMDK on the appropriate vSAN datastores. The PVC status then moves from Pending to Bound. This is by design — Immediate binding could create VMDKs on ESXi hosts far from the pod's node.
