Why MinIO was chosen for platform observability storage — not ODF MCG, not a cloud provider
When LokiStack, Tempo, and GitLab each need S3-compatible object storage, ODF MCG is available (deployed in STORAGE-01). Why was MinIO chosen instead of reusing ODF MCG for these workloads?
MinIO was deployed before ODF existed in NovaCrestBank's environment. When Phase 7 (Observability) was being built, ODF had not yet been deployed — vSAN was the only storage available. MinIO was chosen as a self-contained, S3-compatible object store that runs as a standard OCP workload with vSAN-backed PVCs, requiring no additional infrastructure. This is the correct sequencing for a real production environment: observability infrastructure is deployed early in the platform lifecycle, before enterprise storage is available.

The reasons MinIO remains the correct choice for platform observability workloads even now that ODF MCG is available:

1. Operational simplicity for non-critical data. Log chunks (LokiStack), trace data (Tempo), and GitLab CI artifacts are all observable data — they are derived from running systems and can be regenerated if lost. They are not source-of-truth data. Using ODF MCG for these workloads would add Ceph operational complexity (MCG reconciliation, NooBaa status monitoring, Ceph pool health) to what should be a lightweight platform service. MinIO is a single StatefulSet: troubleshooting reduces to "is the pod Running and is the PVC Bound."

2. Deployment ordering. Observability (Phase 7) is deployed before storage (this guide) in a real production build sequence. An observability stack that depends on ODF would mean ODF must be deployed before monitoring — which is backwards. Monitoring should be available from early in the platform lifecycle, including to observe the ODF deployment itself. MinIO enables this independence.

3. Acceptable durability for the data classification. Logs with 30-day retention on vsan-standard (RAID-5 FTT:1) have acceptable durability — log data is supplementary to the primary audit trail (Splunk), not the sole copy. If the MinIO PVC were to be lost, the impact is: LokiStack query history unavailable, Tempo traces unavailable, GitLab CI artifacts unavailable. Source code (Gitaly), database (PostgreSQL), and audit records (Splunk) are unaffected. This risk is accepted as appropriate for the data classification.

When to migrate from MinIO to ODF MCG: if the durability requirement changes (regulatory audit requires log retention with Ceph-level redundancy), or if NovaCrestBank adds hybrid cloud object tiering (MCG can tier cold data to AWS S3 automatically).

MinIO's role in the NovaCrestBank platform

ncb-ocp-logs

LokiStack log chunks — all infrastructure, audit, and application logs. Retention: 30 days. Written by LokiStack Distributor, read by LokiStack Querier. Covered in OBS-03.

ncb-ocp-traces

Tempo distributed trace data — payment service, account service, and fraud detection traces. Retention: 7 days. Written and read by Tempo Distributor and Querier. Covered in OBS-04.

ncb-gitlab

GitLab CI/CD artifacts, LFS objects, uploads, and daily backups. Retention: 30 days for artifacts, 30 days for backups. Configured in GITOPS-00 Section 04.

STORAGE-02 · Section 02
MinIO architecture on OCP — StatefulSet, PVCs, Service
MinIO on OCP runs as a single-node StatefulSet. It is not clustered (multi-node distributed MinIO would require 4 nodes minimum — unnecessary overhead for NovaCrestBank's observability use case). The single-node deployment provides: one MinIO pod with one PVC for persistent data storage, one ClusterIP Service for internal S3 API access, one Route for the MinIO console, and optional TLS termination.
Pod
MinIO StatefulSet — single replica on infra nodes
A single-replica StatefulSet named minio in the minio namespace. StatefulSet (not Deployment) because: stable pod identity (minio-0), ordered restart on upgrade, and persistent volume claim that follows the pod. The MinIO container listens on two ports: 9000 (S3 API) and 9090 (MinIO Console web UI). Runs on infra nodes — MinIO is platform infrastructure, not a banking workload. Resource requirements are modest: 512Mi memory request for the MinIO process itself; actual memory grows with concurrent connections and open files.
Storage
volumeClaimTemplate — vsan-standard PVC
The StatefulSet volumeClaimTemplate creates a 200 GiB PVC on vsan-standard (RAID-5 FTT:1). The PVC is mounted at /data inside the MinIO container — MinIO stores all bucket data here. The PVC name follows the StatefulSet pattern: minio-data-minio-0. If the MinIO pod is rescheduled to a different infra node, the PVC remains bound and the new pod inherits all data — StatefulSet semantics guarantee this.
Access
Two Services — S3 API (ClusterIP) and Console (Route)
Two Kubernetes Services: minio (ClusterIP, port 9000) for internal S3 API access by LokiStack, Tempo, and GitLab, and minio-console (port 9090) exposed via an OCP Route at minio-console.apps.ocp.corp.novacrestbank.com for the web administration UI. Internal workloads use the ClusterIP Service URL (http://minio.minio.svc:9000) — no Route required for S3 API calls within the cluster. TLS is terminated at the OCP Route for the console; S3 API traffic within the cluster is plain HTTP (internal cluster network).

STORAGE-02 · Section 03
MinIO vs ODF MCG — the complete decision for NovaCrestBank workloads
Now that both MinIO (this guide) and ODF MCG (STORAGE-01) are available, every object storage workload in NovaCrestBank has two options. This section provides the definitive assignment — which workloads use MinIO and which use ODF MCG — with the reasoning for each.
WorkloadObject storeBucketReason
LokiStack log chunks MinIO ncb-ocp-logs Observability data. 30-day retention. Acceptable to lose if MinIO PVC is lost (Splunk holds the authoritative audit trail). Deployed before ODF existed.
Tempo trace data MinIO ncb-ocp-traces 7-day retention. Traces are diagnostic, not compliance-critical. Lightweight and well-served by a single MinIO instance.
GitLab CI artifacts and LFS MinIO ncb-gitlab CI artifacts are reproducible (re-running the pipeline regenerates them). LFS objects are referenced from source code — loss is inconvenient but not a compliance event.
GitLab backups MinIO ncb-gitlab (backups/ prefix) Daily backups with 30-day retention. Secondary backup — the primary DR strategy (STORAGE-04) covers GitLab via OADP. MinIO backup is an additional recovery option.
OADP application backups ODF MCG oadp-backup-* (OBC) Backup data must survive any single component failure. Ceph 3× replication required. This is the primary DR recovery store.
OpenShift Image Registry ODF MCG image-registry-* (OBC) Container images are critical infrastructure — losing the image registry means no new pod starts. Ceph-backed durability required.
Future: LokiStack (if durability upgraded) Migrate to ODF MCG OBC If regulatory requirements mandate Ceph-level durability for log retention, migrate LokiStack to an ODF MCG OBC. Procedure: create new OBC, reconfigure LokiStack, copy existing data if needed.

TLS certificate from cert-manager before deployment
The MinIO console is exposed via an OCP Route at minio-console.apps.ocp.corp.novacrestbank.com. The Route requires a TLS certificate. cert-manager (OPS-02) provisions this certificate from the NovaCrestBank IssuingCA. The certificate must exist before the Route is created — cert-manager creates it from a Certificate object applied to the minio namespace.
Certificate object — MinIO console TLS from NovaCrestBank IssuingCA
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: minio-console-tls
  namespace: minio
spec:
  secretName: minio-console-tls
  issuerRef:
    name: novacrestbank-issuer
    kind: ClusterIssuer
  dnsNames:
    - minio-console.apps.ocp.corp.novacrestbank.com
    - minio.minio.svc
    - minio.minio.svc.cluster.local
    # Three SANs:
    # minio-console.apps.ocp.corp.novacrestbank.com — the external console Route
    # minio.minio.svc — the internal ClusterIP Service hostname
    # minio.minio.svc.cluster.local — the fully-qualified internal hostname
    # Including both internal and external SANs in one certificate avoids
    # certificate warnings whether accessed from inside or outside the cluster.
  duration: 8760h
  renewBefore: 720h

STORAGE-02 · Section 07
MinIO StatefulSet — every field explained
The MinIO StatefulSet is the complete deployment specification. Every annotation, environment variable, probe, and resource setting is explained — this is the installation that OBS-03, OBS-04, and GITOPS-00 assumed existed as a prerequisite.
MinIO StatefulSet — complete and fully annotated
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: minio
  namespace: minio
spec:
  serviceName: minio
  replicas: 1
    # 1 replica: single-node MinIO.
    # NovaCrestBank's observability data does not require multi-node MinIO (which needs 4+ nodes).
    # Single-node: simpler, lower resource cost, still HA within OCP (pod restarts automatically).
    # If MinIO is down: LokiStack and Tempo cannot write new data but continue serving
    # cached/queried data. Impact is observability degradation, not banking outage.
  selector:
    matchLabels:
      app: minio
  template:
    metadata:
      labels:
        app: minio
        app.kubernetes.io/name: minio
        app.kubernetes.io/part-of: platform-storage
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "9000"
        prometheus.io/path: /minio/health/cluster
                # Prometheus annotations: allow Prometheus to scrape MinIO metrics.
                  # MinIO exposes /minio/v2/metrics/cluster on port 9000 for Prometheus.
                  # The health endpoint is simpler for basic up/down monitoring.
    spec:
      serviceAccountName: minio
      nodeSelector:
        node-role.kubernetes.io/infra: ""
      tolerations:
        - key: node-role.kubernetes.io/infra
          effect: NoSchedule
      securityContext:
        fsGroup: 1000
                # fsGroup: 1000 — all files created in the /data volume are owned by GID 1000.
                  # MinIO runs as UID 1000 (non-root). This ensures MinIO can read and write
                  # to the /data volume without permission errors.
      containers:
        - name: minio
          image: harbor.apps.ocp.corp.novacrestbank.com/platform/minio:RELEASE.2025-latest
          args:
            - server
            - /data
            - --console-address
            - :9090
                        # server /data: MinIO stores all bucket data under /data.
                          # --console-address :9090: bind the web console to port 9090.
                          # Without this flag, MinIO chooses a random console port — the
                          # Service cannot target a stable port.
          env:
            - name: MINIO_ROOT_USER
              valueFrom:
                secretKeyRef:
                  name: minio-root-credentials
                  key: MINIO_ROOT_USER
            - name: MINIO_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: minio-root-credentials
                  key: MINIO_ROOT_PASSWORD
            - name: MINIO_PROMETHEUS_AUTH_TYPE
              value: public
                            # public: Prometheus can scrape MinIO metrics without authentication.
                              # The metrics endpoint is on the internal ClusterIP Service —
                              # not accessible outside the cluster. "public" within a private
                              # cluster is acceptable for a metrics endpoint.
            - name: MINIO_BROWSER_REDIRECT_URL
              value: https://minio-console.apps.ocp.corp.novacrestbank.com
                            # BROWSER_REDIRECT_URL: the external console URL.
                              # MinIO uses this for OAuth2 callback URLs and CORS.
                              # Without this, the console may redirect to the internal pod IP.
          ports:
            - name: s3
              containerPort: 9000
            - name: console
              containerPort: 9090
          resources:
            requests:
              cpu: 250m
              memory: 512Mi
            limits:
              cpu: 2
              memory: 2Gi
                            # 2 Gi memory limit: MinIO's memory grows with concurrent operations.
                              # Under LokiStack write load (multiple Distributors writing chunks
                              # simultaneously), MinIO can use 1–1.5 Gi. 2 Gi provides headroom.
          volumeMounts:
            - name: minio-data
              mountPath: /data
          livenessProbe:
            httpGet:
              path: /minio/health/live
              port: 9000
            initialDelaySeconds: 30
            periodSeconds: 20
          readinessProbe:
            httpGet:
              path: /minio/health/ready
              port: 9000
            initialDelaySeconds: 15
            periodSeconds: 10
                        # /minio/health/live: returns 200 if MinIO is alive (process running).
                          # /minio/health/ready: returns 200 if MinIO is ready to accept requests
                          #   (storage backend accessible, no ongoing heal that blocks writes).
                          # Using both probes: OCP restarts MinIO if it's hung (liveness), and
                          # removes it from Service endpoints if it's not ready (readiness).

  volumeClaimTemplates:
    - metadata:
        name: minio-data
      spec:
        storageClassName: vsan-standard
        accessModes: [ReadWriteOnce]
        resources:
          requests:
            storage: 200Gi
                        # 200 GiB initial allocation.
                          # Sizing rationale:
                          # ncb-ocp-logs (30-day LokiStack): ~80 GB compressed log data/month
                          # ncb-ocp-traces (7-day Tempo): ~15 GB trace data/week
                          # ncb-gitlab (artifacts + backups): ~30 GB
                          # Total estimated: ~125 GB steady-state usage.
                          # 200 GiB provides comfortable headroom.
                          # vSAN PVCs are expandable — see Section 18 for resize procedure.
STORAGE-02 · Section 08
Services and Route — S3 API endpoint and console access
Two Services expose MinIO within the cluster, and one Route exposes the console externally. The S3 API Service uses a ClusterIP — internal workloads connect directly without leaving the cluster network. The console Service is exposed via a TLS-terminating Route for platform engineers to administer MinIO through a browser.
MinIO Services and Route
apiVersion: v1
kind: Service
metadata:
  name: minio
  namespace: minio
  labels:
    app: minio
spec:
  selector:
    app: minio
  ports:
    - name: s3
      port: 9000
      targetPort: 9000
    # ClusterIP Service on port 9000.
    # Internal workloads use: http://minio.minio.svc:9000
    # (service-name.namespace.svc:port — standard Kubernetes DNS pattern)
    # LokiStack: http://minio.minio.svc:9000 in the LokiStack CR objectStorage.endpoint
    # Tempo: http://minio.minio.svc:9000 in the TempoStack CR
    # GitLab: http://minio.minio.svc:9000 in the GitLab CR object_store.connection
---
apiVersion: v1
kind: Service
metadata:
  name: minio-console
  namespace: minio
spec:
  selector:
    app: minio
  ports:
    - name: console
      port: 9090
      targetPort: 9090
---
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: minio-console
  namespace: minio
  annotations:
    cert-manager.io/cluster-issuer: novacrestbank-issuer
spec:
  host: minio-console.apps.ocp.corp.novacrestbank.com
  to:
    kind: Service
    name: minio-console
  port:
    targetPort: console
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
    certificate: # injected by cert-manager annotation
    key: # injected by cert-manager annotation
    # edge termination: OCP Router terminates TLS, forwards plain HTTP to MinIO console.
    # The S3 API (port 9000) does NOT have a Route — it is internal-only.
    # S3 traffic within the cluster stays on plain HTTP over the cluster network.
    # This is acceptable: the cluster network is isolated within the NSX overlay segment.

STORAGE-02 · Section 10
The mc CLI — MinIO's administration tool
The mc (MinIO Client) CLI is the primary tool for administering MinIO: creating buckets, managing access keys, setting lifecycle policies, and monitoring storage usage. It runs as a temporary OCP pod using the mc container image from Harbor. There is no local mc binary on the jumphost — all mc operations run inside the cluster against the internal Service endpoint.
jumphost — configure mc alias and verify connectivity
# Run mc as a temporary pod inside the cluster
# This reaches MinIO via the ClusterIP Service — no Route needed
[svc-ocp@jumphost ~]$ MINIO_USER=$(oc get secret minio-root-credentials -n minio \
  -o jsonpath='{.data.MINIO_ROOT_USER}' | base64 -d)
MINIO_PASS=$(oc get secret minio-root-credentials -n minio \
  -o jsonpath='{.data.MINIO_ROOT_PASSWORD}' | base64 -d)

[svc-ocp@jumphost ~]$ oc run mc-admin --rm -i --restart=Never \

Why mc runs as an OCP pod rather than from the jumphost directly
The MinIO S3 API Service (minio.minio.svc:9000) is a ClusterIP Service — it is only accessible from within the OCP cluster's pod network. The jumphost is outside the cluster network. Running mc as an OCP pod in the minio namespace allows it to reach the ClusterIP Service directly without a Route. This pattern — running administrative tools as temporary pods — is also used for etcd CLI (UPG-02 Section 06) and Ceph tools (STORAGE-01 Section 12). It avoids exposing administrative APIs to external networks.

STORAGE-02 · Section 14
Creating scoped access keys per workload — the security model
Each workload that accesses MinIO has its own dedicated access key with permissions scoped to the bucket it needs. No workload uses the root MinIO credentials. This follows the same minimum-privilege model applied to every other credential in the NovaCrestBank platform.
Why per-workload access keys rather than a single shared key
Using a single access key for all workloads (LokiStack, Tempo, GitLab) simplifies credential management — one key to rotate, one Secret to manage. Why use separate keys per workload?
The blast radius argument: if one workload's credentials are compromised, the damage is limited to that workload's bucket.

If LokiStack's NCB_LOKI_ACCESS_KEY is somehow exposed (in a pod log, in a debug session, in a CI pipeline output), the attacker can read and write the ncb-ocp-logs bucket — containing compressed log chunks of OCP operational data. They cannot read ncb-gitlab (source code LFS objects and CI artifacts), and they cannot read ncb-ocp-traces (payment service trace data).

With a shared key, the same exposure gives access to all three buckets simultaneously. The GitLab bucket contains CI artifacts that could reveal application deployment details. The trace bucket contains payment transaction traces with timing and call patterns.

Per-workload keys also enable independent rotation: if a workload is decommissioned, its access key is revoked without affecting other workloads. If a key needs emergency rotation (suspected compromise), only the affected workload is disrupted — not all three.

NCB_LOKI_ACCESS_KEYreadwrite on ncb-ocp-logs only. Used by: LokiStack Distributor (write) and Querier (read). Injected into openshift-logging namespace Secret via ESO.
NCB_TEMPO_ACCESS_KEYreadwrite on ncb-ocp-traces only. Used by: Tempo Distributor and Querier. Injected into openshift-tempo-operator namespace Secret via ESO.

STORAGE-02 · Section 17
MinIO monitoring — Prometheus metrics and alerts
MinIO exposes Prometheus metrics at /minio/v2/metrics/cluster on port 9000. A ServiceMonitor object configures Prometheus to scrape these metrics. Two alert rules cover the most critical MinIO failure modes: disk capacity nearing limits and MinIO pod being unavailable.
ServiceMonitor and PrometheusRule for MinIO monitoring
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: minio
  namespace: minio
spec:
  selector:
    matchLabels:
      app: minio
  endpoints:
    - port: s3
      path: /minio/v2/metrics/cluster
      interval: 30s
---
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: minio-alerts
  namespace: minio
spec:
  groups:
    - name: minio
      rules:
        - alert: MinIODiskCapacityWarning
          expr: (minio_disk_storage_used_bytes / minio_disk_storage_total_bytes) > 0.75
          for: 15m
          labels:
            severity: warning
          annotations:
            summary: MinIO disk usage above 75% — expand PVC before reaching limit
                        # Fires at 75% to give lead time for PVC expansion (Section 18).
                          # At 100%, MinIO returns 503 to all write requests —
                          # LokiStack and Tempo write failures cascade to log and trace loss.

        - alert: MinIODown
          expr: up{job="minio"} == 0
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: MinIO is unreachable — LokiStack and Tempo writes are failing
STORAGE-02 · Section 18
Capacity sizing and PVC expansion
The 200 GiB initial allocation is sized based on expected steady-state usage. This section provides the data behind that sizing calculation and the procedure for expanding the PVC when the MinIODiskCapacityWarning alert fires at 75% usage (150 GiB used).

Sizing calculation — steady-state storage requirements

ncb-ocp-logs (LokiStack, 30-day retention)~2.5 GB/day compressed log data × 30 days = ~75 GB steady-state. LokiStack compresses logs aggressively (~10:1 ratio from raw log volume). Raw log volume ~250 MB/day for the 9-node cluster.
ncb-ocp-traces (Tempo, 7-day retention)~2 GB/day trace data × 7 days = ~14 GB steady-state. Trace data is sampled (GITOPS-04 sampling strategy): 100% errors, 1% normal transactions.
ncb-gitlab (artifacts, LFS, uploads)~1 GB/week CI artifacts (5 microservices × ~200 MB each) × 4 weeks = ~4 GB artifacts. LFS objects: ~5 GB (binaries, diagram files). Uploads: ~2 GB. Total: ~11 GB.
ncb-gitlab (backups, 30-day retention)~700 MB/day backup × 30 days = ~21 GB. GitLab backup includes Gitaly repositories + PostgreSQL dump. Compressed.
Total estimated steady-state~121 GB. 200 GiB allocation provides ~65% headroom above steady-state.
Alert threshold (75%)~150 GiB. Alert fires ~30 GB before the PVC is full — sufficient time to expand.
