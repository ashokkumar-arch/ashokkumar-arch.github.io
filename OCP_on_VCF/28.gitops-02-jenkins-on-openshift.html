Jenkins Operator vs Helm chart — four specific advantages for NovaCrestBank's OCP environment
The Jenkins Helm chart is widely documented and used. The Jenkins Operator is specific to OpenShift and Kubernetes environments. What are the concrete advantages of the Operator over the Helm chart for NovaCrestBank's situation?
The Operator model has four specific advantages that matter in NovaCrestBank's environment. None of them are "it looks nicer" — they are each operational properties that reduce maintenance burden and increase consistency with every other component in the platform.

1. OLM lifecycle management — the same model as ACS, GitLab, cert-manager, and every other operator in this series. The Helm chart requires an engineer to manually run helm upgrade for every Jenkins update — download the new chart, verify values compatibility, execute the upgrade, verify the result. The Operator manages Jenkins updates through OLM's InstallPlan and approval workflow — the same workflow NovaCrestBank uses for every other operator. Jenkins updates are treated identically to ACS updates or Logging Operator updates: an InstallPlan appears, the platform team reviews and approves it, OLM applies it. No special-casing for Jenkins.

2. The JenkinsInstance CR is GitOps-friendly. The Helm chart's configuration lives in a values.yaml file that is applied imperatively with helm upgrade. The Jenkins Operator's configuration lives in a JenkinsInstance Custom Resource — a Kubernetes object that can be stored in the platform/ directory of the GitOps config repo and managed by Argo CD. The entire Jenkins configuration — its plugins, its agent pod templates, its resource limits — is a YAML file in Git. Changes to Jenkins are made via merge request, reviewed, approved, and applied by Argo CD. This is consistent with GitOps-01's principle: everything is in Git.

3. Agent pod configuration as Kubernetes-native objects, not Jenkins XML. Traditional Jenkins agent configuration is stored in Jenkins' internal XML configuration files — not visible in Git, not reviewable, only editable via the Jenkins UI or JCasC (Jenkins Configuration as Code). The Operator's JenkinsInstance CR includes the Kubernetes agent pod template configuration directly in the CR spec — it is YAML, it is in Git, it is reviewed with the same process as any other configuration change. There is no "go into the Jenkins UI and click through agent template configuration."

4. Air-gapped installation support. The Jenkins Operator and all required component images have well-documented image lists. They are mirrored to Harbor during the oc-mirror run. The Helm chart has less predictable image references (especially for the Jenkins agent images, which vary by use case), making air-gapped installation documentation harder to follow consistently.
GITOPS-02 · Section 02
Jenkins on OCP architecture — controller and dynamic agents
Jenkins on OpenShift runs as two kinds of workload: the Jenkins controller (the long-running StatefulSet) and Jenkins agent pods (ephemeral pods created on demand for each build). Understanding both explains every configuration decision in this guide.
Controller
Jenkins controller — the long-running process
The Jenkins controller is a single-replica StatefulSet (not a Deployment — it needs a stable network identity and persistent storage). It hosts: the Jenkins web UI, the pipeline scheduler, the build queue, the plugin system, credentials storage, and job configuration. It does NOT run build steps itself — all actual CI work happens in agent pods. The controller's PVC on vsan-standard stores: job history, plugin binaries, build logs, credentials, and workspace (for fallback only). Size: 50 Gi. Location: infra nodes.
Agent
Jenkins agent pods — ephemeral, one per build stage
When a pipeline stage starts, Jenkins creates a new Kubernetes pod in the jenkins namespace using the agent pod template (Section 15). The pod contains the build tools (Maven, JDK, Buildah, cosign, kubectl). The stage runs inside this pod, builds the image, pushes to Harbor, signs with cosign, and exits. When the pipeline stage completes, the pod is terminated and deleted automatically. No agent pods exist when no builds are running — zero idle resource cost. This is the primary advantage of Kubernetes-native Jenkins agents over traditional static agents.
Communication
Controller ↔ agent communication via JNLP
When Jenkins creates an agent pod, it injects a JNLP (Java Network Launch Protocol) sidecar container into the pod. The JNLP container establishes a connection back to the Jenkins controller's internal Service. Pipeline steps execute in the named containers within the pod (Maven build in the maven container, Buildah in the buildah container). The JNLP sidecar coordinates the step execution. When all steps complete, the pod's main process exits and Kubernetes terminates the pod.

GITOPS-02 · Section 03
Dynamic agent pods — ephemeral, one per build, zero idle cost
The dynamic agent pod model is architecturally different from traditional Jenkins agents (fixed VMs or containers that run permanently). Understanding the difference explains why it is the correct model for an OCP environment and why it improves both security and cost.
Why ephemeral agents are superior to static agents for banking CI/CD
Traditional Jenkins deployments use a small number of persistent agent nodes (VMs or containers) that stay running all the time. The Kubernetes plugin creates a new pod for each build. Why is the ephemeral model preferable for NovaCrestBank?
Four properties of ephemeral agents make them superior to static agents in NovaCrestBank's environment.

1. Build isolation eliminates contamination between builds. Each build starts from a clean container image. A previous build cannot leave behind temporary files, modified environment variables, cached credentials, or corrupted build state that affects the next build. With static agents, a build that leaks credentials into the agent's filesystem leaves them there for the next build. With ephemeral agents, every build starts from the exact image defined in the agent template — identical, clean, and known.

2. Zero idle resource consumption. Static agents consume CPU and memory whether a build is running or not. For NovaCrestBank's build frequency (tens of builds per day, not thousands), having three permanent agent VMs would waste infra node resources during the 90% of the day when no builds are running. Ephemeral agents consume zero resources when idle — the infra nodes' capacity is available for monitoring, logging, and GitLab when no builds are in progress.

3. Agent configuration is version-controlled and auditable. The agent pod template is in the JenkinsInstance CR (in Git, reviewed via MR). If a tool version in the agent needs updating — for example, upgrading Maven from 3.8 to 3.9 — the change is a Git commit with a diff showing exactly what changed in the agent definition, reviewed and approved by the platform team. With static agents, changing the tool version means updating every agent VM manually — with no audit trail and high risk of missing a node.

4. Automatic scaling to build demand. Jenkins can run up to 5 concurrent builds (the configured maximum) by creating up to 5 agent pods simultaneously. If demand is low, 1 pod is created. If 5 builds queue up simultaneously (end-of-sprint push by multiple developers), 5 pods are created in parallel. The infra nodes absorb this burst and return to normal capacity when the builds finish. Static agents cannot scale beyond their pre-provisioned count without human intervention.

  name: jenkins-operator
  namespace: jenkins

  channel: alpha
    # "alpha" is the current stable channel for the Jenkins Operator community edition.

  name: jenkins-operator

GITOPS-02 · Section 06
The Jenkins CR — every field explained
The JenkinsInstance Custom Resource declares the complete Jenkins deployment. The Operator reads this CR and reconciles the Jenkins controller StatefulSet, Services, Routes, and RBAC objects. Every field is annotated.

JenkinsInstance CR — NovaCrestBank Jenkins — every field explained
apiVersion: jenkins.io/v1alpha2
kind: Jenkins
metadata:
  name: ncb-jenkins
  namespace: jenkins
spec:
  configurationAsCode:
    enabled: true
      # true: Jenkins Configuration as Code (JCasC) plugin is enabled.
        # JCasC reads a YAML configuration file and applies it to Jenkins on startup.
        # This allows the entire Jenkins configuration (security, LDAP, jobs) to be
        # declared in the JenkinsInstance CR rather than configured via the Jenkins UI.
        # Without JCasC, every Jenkins restart requires manual UI reconfiguration.
    secret:
      name: jenkins-configuration-secret
        # Secret containing the JCasC YAML configuration file.
          # Contains: LDAP bind password reference, admin credentials reference,
          # GitLab API token reference. Sensitive values are referenced from Secrets
          # via JCasC variable interpolation — not hardcoded in the CR.

  master:
      # master: configuration for the Jenkins controller StatefulSet
    annotations:
      cert-manager.io/cluster-issuer: novacrestbank-issuer
    nodeSelector:
      node-role.kubernetes.io/infra: ""
    tolerations:
      - key: node-role.kubernetes.io/infra
        effect: NoSchedule
        # Jenkins controller runs on infra nodes — consistent with GitLab, Prometheus,
          # LokiStack, and Tempo. Jenkins is platform infrastructure, not a banking workload.
    securityContext:
      runAsUser: 1000
      fsGroup: 1000
        # Non-root user. OCP's default SCC (restricted-v2) requires non-root.
            # Jenkins Operator sets this automatically — shown here for transparency.
    volumes:
      - name: jenkins-home
        persistentVolumeClaim:
          claimName: jenkins-home
    volumeMounts:
      - name: jenkins-home
        mountPath: /var/jenkins_home
    resources:
      requests:
        cpu: 500m
        memory: 1Gi
      limits:
        cpu: 2
        memory: 4Gi
          # 4 Gi memory: the Jenkins controller JVM needs 2–3 Gi heap under normal load.
              # With 50 pipelines and build history, 4 Gi provides comfortable headroom.
              # CPU: bursts to 2 cores during build scheduling; 500m baseline is sufficient.

  serviceAccount:
    annotations:
      jenkins.io/service-account-type: controller
      # The Jenkins Operator creates a ServiceAccount for the Jenkins controller.
        # The controller SA needs: ability to create/delete agent pods in the jenkins namespace.
        # It does NOT need: cluster-admin, cluster-wide resource access, or access to
        # any namespace other than jenkins. Section 09 defines the exact RBAC.

  plugins:
      # Plugins declared here are installed and managed by the Operator.
        # No manual plugin installation via the Jenkins UI — changes go through Git and MR.
        # Full plugin list is in Section 17.
    - name: kubernetes
      version: "4205.v430230a_b_b_b_b"
            # Kubernetes plugin: enables dynamic agent pod creation.
                # This is the most critical plugin — without it, agents cannot be created.
    - name: gitlab-plugin
      version: "1.8.1"
    - name: ldap
      version: "736.vea_f7c4ac8b_4d"
    - name: pipeline-model-definition
      version: "2.2214.vb_b_34b_2ea_9b_83"
    - name: configuration-as-code
      version: "1810.v9b_c30a_249a_4c"

  restore:
    enabled: false
      # false: no automated restore on startup. Manual restore from backup if needed.
        # The Jenkins PVC is on vsan-standard — it survives pod restarts.
        # A restore is only needed after a PVC loss, which triggers a manual process.

GITOPS-02 · Section 08
Persistent storage for Jenkins home — vsan-standard StorageClass
The Jenkins controller's /var/jenkins_home directory holds everything Jenkins needs to survive a pod restart: build history, plugin binaries, job configurations, and credentials. The PVC must be provisioned before the JenkinsInstance CR is applied.
Why vsan-standard and not vsan-gold for Jenkins home
Gitaly and PostgreSQL (in GitLab) use vsan-gold (RAID-6 FTT:2). Why does Jenkins use vsan-standard (RAID-5 FTT:1) instead?
The storage tier is chosen based on the recoverability of the data, not its importance to operations.

Gitaly holds source code repositories — data that cannot be reconstructed if lost (developer commits that have not been pushed elsewhere). vsan-gold is correct: losing Gitaly data means losing code permanently.

Jenkins home holds: build history and logs (reproducible — the pipeline can be re-run), plugin binaries (downloadable from Harbor mirror), and job configurations (declared in the JenkinsInstance CR and in Jenkinsfiles in Git — fully reconstructible from Git). The most sensitive data in Jenkins home is the credentials store (LDAP bind password, Harbor push credentials, GitLab API token). These are all stored as OCP Secrets and referenced by Jenkins — they are not irreplaceable data that only Jenkins holds.

If the vsan-standard PVC for Jenkins home is lost: restore the JenkinsInstance CR from Git (Argo CD will recreate everything), re-provision the PVC, and Jenkins starts fresh. Build history is lost, but no source code, no images, and no deployment configuration is lost. The pipeline is re-runnable. Recovery time: 30 minutes. This is acceptable for a CI system and does not warrant the higher cost of vsan-gold storage.
PersistentVolumeClaim — Jenkins home on vsan-standard
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: jenkins-home
  namespace: jenkins
spec:
  storageClassName: vsan-standard
    # vsan-standard: RAID-5 FTT:1. Appropriate for CI build data.
    # Jenkins data is reproducible — pipeline can be re-run to regenerate artifacts.
  accessModes: [ReadWriteOnce]
    # ReadWriteOnce: Jenkins controller is single-replica — one pod reads/writes.
    # If Jenkins HA is implemented in future, this must change to ReadWriteMany
    # with an appropriate StorageClass (ODF or NFS-backed).
  resources:
    requests:
      storage: 50Gi
        # 50 Gi: Jenkins home with 50 banking pipelines, 30-day build history,
            # plugins (~500 MB), and workspace (~5 GB).
            # Monitor: oc exec jenkins-ncb-jenkins -- df -h /var/jenkins_home
            # Expand if usage exceeds 70%.
GITOPS-02 · Section 09
RBAC design — create agent pods, not cluster config
Jenkins needs specific Kubernetes permissions to create and manage agent pods. It must have exactly these permissions and nothing more. This section defines the minimum-privilege RBAC that allows dynamic agents while preventing Jenkins from modifying cluster configuration.
Why minimum-privilege RBAC for Jenkins is a security requirement, not a best practice
Jenkins, if compromised through a malicious plugin, a Groovy script injection vulnerability, or a pipeline script that escapes its sandbox, would inherit all permissions granted to its ServiceAccount. If Jenkins has cluster-admin — a configuration seen in some older tutorials — a compromised Jenkins instance can read all Secrets across all namespaces, modify any Deployment, or create new privileged pods anywhere in the cluster. NovaCrestBank's Jenkins ServiceAccount has exactly three permissions: create and delete Pods in the jenkins namespace, get and list Pods in the jenkins namespace, and get Secrets in the jenkins namespace (to mount credentials into agent pods). Nothing else. If Jenkins is compromised, the blast radius is limited to: ability to create pods in the jenkins namespace. This is a significant security boundary.
Jenkins ServiceAccount, Role, and RoleBinding — minimum privilege
apiVersion: v1
kind: ServiceAccount
metadata:
  name: jenkins
  namespace: jenkins
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: jenkins-agent-role
  namespace: jenkins
rules:
  - apiGroups: [""]
    resources: [pods]
    verbs: [create, delete, get, list, watch]
      # create/delete pods: required to create and clean up agent pods.
        # get/list/watch: required to monitor agent pod status during builds.
        # This is the ONLY resource Jenkins can create. It cannot create Deployments,
        # Services, ConfigMaps, or any other resource type in any namespace.
  - apiGroups: [""]
    resources: [pods/exec]
    verbs: [create]
      # pods/exec: required for the Kubernetes plugin to execute pipeline steps
        # inside agent container processes via the Kubernetes API exec mechanism.
  - apiGroups: [""]
    resources: [pods/log]
    verbs: [get]
      # pods/log: required to stream build step output back to the Jenkins controller
        # for display in the Jenkins UI and logging to /var/jenkins_home.
  - apiGroups: [""]
    resources: [secrets]
    verbs: [get]
      # secrets/get: required to mount credentials into agent pods.
        # Jenkins mounts the cosign signing key, Harbor push credentials, and
        # GitLab API token as Secrets into agent pods. Without get on secrets,
        # the Kubernetes plugin cannot resolve Secret mounts.
        # SCOPED: only in the jenkins namespace — cannot read Secrets in
        # banking-production, openshift-monitoring, or any other namespace.
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: jenkins-agent-rolebinding
  namespace: jenkins
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: jenkins-agent-role
subjects:
  - kind: ServiceAccount
    name: jenkins
    namespace: jenkins

GITOPS-02 · Section 11
LDAP plugin — the same AD pattern as OCP OAuth, Harbor, and GitLab
Jenkins LDAP authentication uses the same Active Directory bind account and search configuration as every other system in this series. The configuration is delivered via JCasC (Jenkins Configuration as Code) — not via the Jenkins UI — so it is version-controlled in Git.

Fourth system using svc-ocp-ldap — the value of a consistent identity model
NovaCrestBank now has four systems authenticating against Active Directory using the same service account: OCP OAuth (ocp-vcf-02), Harbor (registry-01), GitLab (gitops-00), and Jenkins (this guide). One bind account, one set of LDAP policies, one place to audit AD service account usage. When the svc-ocp-ldap password is rotated (per NovaCrestBank's service account password policy), the new password must be updated in four Kubernetes Secrets across four namespaces — a known, finite, documented process. This is far better than four different bind accounts with four different expiry dates, four different permission sets, and four different rotation schedules.

GITOPS-02 · Section 13
GitLab plugin — webhook receiver and build trigger
The GitLab plugin enables two capabilities: receiving webhooks from GitLab (so Jenkins starts a build when a push occurs) and reading pipeline status back to GitLab (so the GitLab merge request UI shows whether the Jenkins pipeline passed). Both are configured via JCasC.
→
GitLab → Jenkins: webhook trigger
When a developer merges a merge request to main in the banking/payment-service repo, GitLab sends an HTTP POST to https://jenkins.apps.ocp.corp.novacrestbank.com/project/banking/payment-service. The GitLab plugin receives the webhook, validates the HMAC secret token (configured in GITOPS-00 Section 14), and triggers the corresponding Jenkins pipeline job. The payload includes the commit SHA, branch name, and author — Jenkins uses these to check out the exact commit that triggered the build.
←
Jenkins → GitLab: pipeline status reporting
As the pipeline progresses through stages, Jenkins updates the GitLab merge request's pipeline status (the ✓ or ✗ indicator shown in the GitLab MR UI). When the pipeline completes, Jenkins posts a comment on the MR summarising the build result — including a link to the build log, the ACS scan verdict, and the integration test result. This closes the feedback loop: developers see the build result in the same GitLab MR where they made the commit, without needing to go to Jenkins separately.

GITOPS-02 · Section 15
Kubernetes agent pod template — the build environment
The agent pod template defines what every build environment looks like: which containers are available, what resources are allocated, which Secrets are mounted, and where builds run. This is the heart of NovaCrestBank's CI build environment — declared in Git, reviewed via MR, version-controlled.
JCasC — Kubernetes agent pod template (in jenkins.yaml)
jenkins:
  clouds:
    - kubernetes:
        name: openshift
        serverUrl: https://kubernetes.default.svc
          # kubernetes.default.svc: the Kubernetes API server's internal Service.
            # Jenkins is running inside OCP — it uses the in-cluster API endpoint.
            # No external kubeconfig needed — the Jenkins ServiceAccount token is used.
        credentialsId: ""
          # Empty: use the Jenkins ServiceAccount's token (mounted automatically).
            # The ServiceAccount is the jenkins SA with the minimum-privilege Role (Section 09).
        namespace: jenkins
        jenkinsUrl: http://jenkins-ncb-jenkins.jenkins.svc:8080
          # The internal Service URL for JNLP agent-to-controller communication.
            # Agents connect back to the controller on port 8080 (plain HTTP, internal).
            # External HTTPS is only for the OCP Route used by browsers.
        maxRequestsPerHost: 64
        podLabels:
          - key: app
            value: jenkins-agent
        templates:
          - name: ncb-banking-agent
              # Template name referenced in Jenkinsfile: agent { kubernetes { ... label 'ncb-banking-agent' } }
            namespace: jenkins
            nodeSelector: node-role.kubernetes.io/worker=
              # Agent pods run on WORKER nodes — not infra nodes.
                  # Build workloads (Maven compilation, Buildah image build) are
                  # CPU and memory intensive. They should not compete with the
                  # monitoring stack (Prometheus, Loki) on infra nodes.
                  # This is the one case where a platform workload intentionally
                  # goes to worker nodes: the agents are transient, bounded,
                  # and protected by resource limits.
            containers:
              - name: jnlp
                image: harbor.apps.ocp.corp.novacrestbank.com/jenkins/inbound-agent:latest-jdk21
                resourceRequestCpu: 100m
                resourceRequestMemory: 256Mi

              - name: maven
                image: harbor.apps.ocp.corp.novacrestbank.com/jenkins/maven:3.9-eclipse-temurin-21
                command: sleep
                args: infinity
                  # command: sleep infinity — the container stays alive for the pipeline duration.
                      # The JNLP container executes pipeline steps by running commands inside
                      # this container via exec. Without sleep infinity, the container exits immediately.
                resourceRequestCpu: 500m
                resourceLimitCpu: 2
                resourceRequestMemory: 1Gi
                resourceLimitMemory: 3Gi
                  # 3 Gi memory limit: Maven builds with Spring Boot dependencies require
                      # 1–2 Gi heap during compilation. 3 Gi provides headroom for
                      # large multi-module builds without OOMKilling the agent.

              - name: buildah
                image: harbor.apps.ocp.corp.novacrestbank.com/jenkins/buildah:latest
                  # Buildah image: runs as non-root user. Builds OCI images without
                      # requiring a Docker daemon or a privileged container.
                      # Buildah uses user-namespace isolation for rootless builds.
                command: sleep
                args: infinity
                securityContext:
                  privileged: false
                  runAsUser: 1000
                    # privileged: false — Buildah does NOT need privileged mode.
                          # This is a key security property of Buildah over Docker:
                          # Docker-in-Docker requires privileged containers (full root access
                          # to the host). Buildah's rootless mode works without it.
                resourceRequestCpu: 500m
                resourceLimitCpu: 2
                resourceRequestMemory: 512Mi
                resourceLimitMemory: 2Gi

            volumes:
              - secretVolume:
                  secretName: jenkins-cosign-signing-key
                  mountPath: /etc/cosign
                  # Mount the cosign signing key into the agent pod at /etc/cosign.
                      # The buildah container signs images using:
                      # cosign sign --key /etc/cosign/cosign.key harbor.../payment-service@sha256:...

            podRetention: never
              # never: delete the agent pod immediately when the pipeline completes.
                  # "default" retains pods for a period — wastes resources.
                  # "never" ensures zero agent pods exist when no builds are running.
GITOPS-02 · Section 16
Agent tools — Maven, Buildah, cosign, and kubectl in the agent pod
Each tool in the agent pod is in a separate container, mirrored to Harbor, and pinned to a specific version in the JenkinsInstance CR. Updating a tool version is a Git commit with a diff, not a manual agent node update.
ToolContainerHarbor imagePurpose in pipeline
Maven 3.9 + JDK 21 maven harbor.../jenkins/maven:3.9-eclipse-temurin-21 Compile Java source, run unit tests, generate JAR. mvn clean verify in Stages 1–2.
Buildah buildah harbor.../jenkins/buildah:latest Build OCI container image from Dockerfile without Docker daemon. Non-root, no privileged mode. Stage 3.
cosign buildah (installed alongside Buildah) Included in buildah image Sign the image after push to Harbor. Uses /etc/cosign/cosign.key mounted Secret. Stage 4.

Why Buildah and not Docker-in-Docker for container image builds
Docker-in-Docker (DinD) requires a privileged container — the agent pod runs with full root access to the underlying host. This means a compromised build can escape the container and access the host node. Buildah builds OCI-compliant images using Linux user namespaces for isolation, without a daemon and without privileged mode. The image it produces is identical to what Docker would produce — it is an OCI standard container image that Harbor, OCP, and any other OCI-compliant system can use. NovaCrestBank's security policy prohibits privileged containers in the jenkins namespace — Buildah is the only viable build tool.
GITOPS-02 · Section 17
Plugin management via the Operator — no manual UI installs
The Jenkins Operator manages all plugin installations through the spec.plugins list in the JenkinsInstance CR. No plugins are installed via the Jenkins UI. Adding a plugin requires a Git commit to the JenkinsInstance CR, a MR, an approval, and Argo CD applying the change.

Never install Jenkins plugins through the Jenkins UI in NovaCrestBank's environment
A plugin installed via the Jenkins UI exists only in the /var/jenkins_home/plugins directory. It is not in the JenkinsInstance CR. If the Jenkins pod is restarted and the PVC contents are lost, the plugin disappears. If a new Jenkins instance is deployed from the CR (disaster recovery), the plugin is not present. Most critically: a plugin installed via the UI bypasses the code review and change management process — a malicious actor with Jenkins admin access could install a credential-stealing plugin without any audit trail. All plugins must be in the JenkinsInstance CR, in Git, reviewed and approved.

Security hardening — CSRF protection, TLS, and job isolation
Jenkins has historically been a high-value target for attackers due to its access to build credentials and code. These hardening settings are applied via JCasC and are non-negotiable in NovaCrestBank's PCI-DSS-scoped environment.

JCasC security hardening settings (in jenkins.yaml)
jenkins:
  crumbIssuer:
    standard:
      excludeClientIPFromCrumb: false
      # CSRF protection: enabled. A CSRF crumb is required for all POST requests
        # to the Jenkins API. Without this, an attacker who tricks a Jenkins admin
        # into visiting a malicious page can trigger builds or modify configuration
        # via cross-site request forgery.
        # excludeClientIPFromCrumb: false — include the client IP in the crumb.
        # This adds a second validation dimension: the crumb is only valid from the
        # IP that requested it.

  slaveAgentPort: 50000
    # Agent JNLP port. Fixed at 50000 — allows a specific NetworkPolicy rule
    # to permit only jenkins-agent pods to connect on this port (Section 09 RBAC).

  agentProtocols:
    - JNLP4-connect
      # JNLP4-connect: the only enabled agent protocol. Provides encrypted
        # communication between agents and the controller.
        # Older protocols (JNLP, JNLP2, JNLP3) have known vulnerabilities — disabled.

  markupFormatter:
    rawHtml:
      disableSyntaxHighlighting: false
    # rawHtml: allow HTML in build descriptions (for Blue Ocean pipeline display).
    # Alternative: safeHtml — strips potentially malicious HTML.
    # rawHtml is acceptable because only authenticated users with job-read permission
    # can view build descriptions. Access is controlled at the auth layer.

security:
  scriptApproval:
    approvedSignatures: []
      # Empty: no pre-approved Groovy script signatures.
        # The Groovy script sandbox is enabled — scripts must be approved by an admin
        # before running. This prevents Jenkinsfile scripts from executing arbitrary
        # system commands that could exfiltrate credentials or modify the cluster.

  globalJobDslSecurityConfiguration:
    useScriptSecurity: true
      # true: Job DSL scripts require admin approval before execution.
        # Prevents pipeline-as-code from self-modifying Jenkins job configuration
        # without review.
