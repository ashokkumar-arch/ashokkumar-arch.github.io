UPG-03 · Section 01
What a MachineConfig is — declarative RHCOS node configuration
A MachineConfig (MC) is a Kubernetes Custom Resource that declares the desired state of an RHCOS node's operating system configuration — kernel arguments, sysctl parameters, files on disk, and systemd units. Instead of SSHing to nodes to make configuration changes, you apply a MachineConfig object and the MCO applies it across the correct nodes automatically.

Why MachineConfig — the problem with SSH-based node configuration
RHCOS nodes are regular Linux systems. Why not SSH to a node and run sysctl -w net.core.somaxconn=65535 or edit /etc/sysctl.d/99-custom.conf directly?
Three specific problems arise with SSH-based node configuration in an OCP cluster, each of which MachineConfig eliminates.

1. The change does not survive a node rebuild. RHCOS is an immutable operating system — the root filesystem is read-only and managed by rpm-ostree. Changes made directly to RHCOS via SSH to the rootfs are overwritten the next time the MCO applies a MachineConfig (for example, during a cluster upgrade). After an upgrade, the node reboots with a fresh RHCOS image and the manually applied sysctl is gone. The sysctl must be re-applied, but now nobody knows it was there because it was never documented.

2. The change is not replicated to other nodes. If you SSH to worker-0 and set a sysctl, worker-1 and worker-2 still have the default value. If a new worker node is added (via MachineSet replica scaling — Section 17), it also has the default value. Keeping N nodes in sync via SSH is a configuration management problem that MachineConfig solves by design — one MC object, all matching nodes converge to the same state.

3. There is no audit trail. An SSH session to a node leaves no Kubernetes audit record (the OCP API server audit log does not see SSH commands). A MachineConfig applied via oc apply creates a Kubernetes API audit event — who applied it, when, from what IP. The node configuration is version-controlled in Git (if using GitOps), reviewed via PR, and auditable. For NovaCrestBank's PCI-DSS environment, SSH-based configuration changes to production nodes are a compliance gap; MachineConfig changes are not.

What a MachineConfig can and cannot configure

Can configure via MachineConfig

  • Kernel arguments (kargs) — added at boot via the bootloader
  • Sysctl parameters — written to /etc/sysctl.d/ and applied at boot
  • Files on the filesystem — placed at specified paths with specified permissions
  • systemd units — created, enabled, or disabled
  • CRI-O container runtime configuration
  • SSH authorised keys for the core user
  • Network manager configuration (static routes, DNS)

Cannot configure via MachineConfig

  • Kubernetes/OCP component configuration (use cluster operators or CRs instead)
  • Pod-level settings (use SecurityContext, LimitRange, or operator config)
  • Application software installation (use container images)
  • Dynamic runtime values (MachineConfig is applied at boot — not live)
  • Anything that requires the kubelet to be restarted mid-operation
UPG-03 · Section 02
What a MachineConfigPool is — why there are three separate pools
A MachineConfigPool (MCP) is a group of nodes that share the same desired MachineConfig. NovaCrestBank's cluster has three pools: master, infra, and worker. Each pool can have different configuration applied to it, and each pool updates its nodes independently.
master pool

Nodes: master-0, master-1, master-2
Selector: node-role.kubernetes.io/master: ""
Updates first during cluster upgrades. Tainted NoSchedule for workloads — only control plane pods run here. Configuration should never differ from Red Hat defaults unless required by a specific security hardening requirement.

infra pool

Nodes: infra-0, infra-1, infra-2
Selector: node-role.kubernetes.io/infra: ""
Runs the monitoring stack, logging, router. NovaCrestBank applies additional sysctl parameters here (Section 07) to support the high-throughput Loki log ingestion and Prometheus TSDB write workloads. Custom kernel arguments for infra workload tuning live here, not on the worker pool.

worker pool

Nodes: worker-0, worker-1, worker-2
Selector: node-role.kubernetes.io/worker: ""
Runs banking application pods. Updates after master during cluster upgrades. Custom kernel arguments for network throughput tuning are applied here for the payment service workload. New nodes added via MachineSet (Section 17) join this pool automatically.

Why three pools rather than one — the separation of concerns argument
A single MachineConfigPool targeting all nodes with a single set of configurations would be simpler to manage. Why have three separate pools?
Different node roles require different configurations, and pooling them separately prevents accidental misconfiguration and provides independent update control.

Different configuration requirements: The infra pool needs elevated fs.inotify.max_user_instances because Prometheus and Loki each watch thousands of filesystem events (inotify). The worker pool needs elevated net.core.somaxconn because banking API pods accept many simultaneous connections. The master pool should have neither — applying application-tuning parameters to control plane nodes risks destabilising etcd and kube-apiserver by competing for the same kernel resources.

Independent update sequencing: During a cluster upgrade, the master MCP must complete before worker and infra MCPs are allowed to proceed (UPG-02). If all nodes were in one pool, the MCO could not enforce this ordering — it would drain and reboot all nodes in arbitrary order, potentially rebooting a worker before all masters were updated. Three pools with explicit ordering give the MCO the dependency graph it needs.

Independent pause control: NovaCrestBank's upgrade strategy (UPG-02 Section 02) pauses the worker and infra MCPs before triggering the upgrade, then un-pauses them after verifying the control plane. This is only possible because worker and infra are separate pools from master.
UPG-03 · Section 03
The rendered MachineConfig — how source MachineConfigs are merged
Multiple MachineConfig objects can target the same pool. The MCO merges them into a single rendered MachineConfig — the actual configuration applied to each node. Understanding the rendered MC explains why oc get machineconfigs shows many more objects than you created.
What a rendered MachineConfig is and why it matters
Running oc get machineconfigs shows 15 MachineConfig objects but only 2 were manually created. What are the others and what is rendered-worker-abc123?
The cluster has two categories of MachineConfig objects: source MachineConfigs (the individual configs, each targeting a pool via the machineconfiguration.openshift.io/role label) and one rendered MachineConfig per pool (the merged result). The source configs come from OCP itself (each cluster operator contributes MachineConfigs defining their required node configuration), from NovaCrestBank's custom configs, and from any operator that needs node-level configuration.

When any source MachineConfig changes — whether from an OCP upgrade delivering new configs or from an engineer applying a custom sysctl — the MCO recomputes the rendered MachineConfig for that pool. The rendered MC is a hash of all source configs merged together. The name rendered-worker-abc123def456 where abc123def456 is a hash of all contributing source configs.

Why this matters operationally: The MachineConfigPool's status field spec.configuration.name shows the rendered MC that nodes should be running. The status.observedGeneration and machineCount vs updatedMachineCount comparison shows how many nodes are running the current rendered MC vs an older one. Whenever you see a node "Updating" in the MCP, it means that node has an older rendered MC hash than the current desired one — the MCO is bringing it up to date.

UPG-03 · Section 04
The Machine Config Daemon — the per-node agent that applies configurations
The Machine Config Daemon (MCD) is a DaemonSet pod running on every node. It is the agent that receives configuration changes from the MCO, applies them to the node, and reports status back. Understanding the MCD explains what is actually happening inside a node during a config update.
1
MCO computes new rendered MC
When a source MachineConfig changes (new custom sysctl applied, or OCP upgrade delivers new base configs), the MCO recomputes the rendered MachineConfig for the affected pool and updates the pool's spec.configuration.name to the new rendered MC hash. The pool's updatedMachineCount drops — it now knows some nodes are behind.
2
MCD on the node detects the change
The MCD pod on each node watches the MachineConfigPool and the rendered MachineConfig. When it detects that the desired rendered MC hash differs from what the node is running, it begins the update process. The MCD reads the new rendered MC and determines what needs to change: which files need to be written, which kernel arguments need to be added, which systemd units need to be enabled.
3
MCD applies the changes
For changes that can be applied without a reboot (writing a file, enabling a systemd unit that does not require a kernel-level change): the MCD applies them in-place and marks the node updated without a reboot. For changes that require a reboot (kernel arguments, most sysctl parameters, RHCOS version updates): the MCD cordons the node, drains it (respecting PodDisruptionBudgets), applies the changes via rpm-ostree (for RHCOS changes) or writes config files and reboots.
4
MCD reports status
After successful application, the MCD annotates the node with the new rendered MC hash. The MCO sees the annotation and increments updatedMachineCount for the pool. When updatedMachineCount equals machineCount, the pool reports UPDATED=True. The MCD also reports any failures — if a node fails to apply a config, the pool reports DEGRADED=True and the MCD pod logs contain the specific error.

UPG-03 · Section 07
Applying a custom sysctl to infra nodes
NovaCrestBank's infra nodes run Prometheus and LokiStack — both involve high-frequency file descriptor and inotify operations. Two kernel parameters need to be elevated beyond RHCOS defaults. This section is the complete walkthrough: write the MachineConfig, apply it, and watch the rollout.
Which sysctl parameters and why infra nodes specifically need them
Why do infra nodes need elevated fs.inotify.max_user_instances and fs.inotify.max_user_watches, and why not apply these to all nodes?
Prometheus watches thousands of files through inotify — its TSDB storage directory, rule files, and config maps are all monitored for changes. LokiStack's Ingester similarly watches its chunk write-ahead log directory. The default RHCOS values (max_user_instances=128 and max_user_watches=8192) are appropriate for normal application pods. They are too low for Prometheus and Loki, which each require hundreds of instances and tens of thousands of watches.

The symptom when the limit is hit: the Prometheus pod logs show too many open files errors, rule evaluation begins failing, and the cluster-monitoring-config CMO reports Prometheus as Degraded. The error is non-obvious because "too many open files" typically implies file descriptor limits, not inotify limits — but inotify watches consume file descriptors counted against the same per-process limit.

Why not apply to all nodes: Elevating inotify limits consumes kernel memory proportionally to the maximum configured — even if the limit is never reached, the kernel reserves memory for the theoretical maximum. Worker nodes running banking application pods do not need elevated inotify limits. Applying this sysctl only to the infra pool keeps worker nodes at the default (lower memory reservation) and applies the elevated limit only where it is genuinely needed.
99-ncb-infra-sysctl — custom sysctl for infra nodes — fully annotated
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 99-ncb-infra-sysctl
    # Naming convention: 99- prefix ensures this sorts AFTER all OCP base configs
    # (00- through 01-). If there is a conflict on the same parameter, the
    # last-sorted config wins. Custom configs must use 99- to take precedence.
  labels:
    machineconfiguration.openshift.io/role: infra
      # This label is what assigns this MachineConfig to the infra pool.
        # ONLY nodes in a MachineConfigPool whose machineConfigSelector matches
        # this label will receive this configuration.
        # The infra pool's machineConfigSelector matches: role=infra.
        # Worker nodes will never receive this config — different label.
spec:
  config:
    ignition:
      version: 3.4.0
        # Ignition version 3.4.0 is required for RHCOS 4.16. Do not change this.
          # Ignition is the RHCOS first-boot configuration system — even for post-install
          # MachineConfigs, the spec uses the Ignition format for file declarations.
    storage:
      files:
        - path: /etc/sysctl.d/99-ncb-infra-tuning.conf
            # Write this file to the RHCOS filesystem.
                # /etc/sysctl.d/*.conf files are read by systemd-sysctl at boot
                # and applied before any services start. This is the correct location
                # for persistent sysctl parameters — NOT /proc/sys/ (which is runtime-only
                # and does not survive reboots).
          mode: 0644
            # File permissions: owner=read/write, group=read, other=read.
                # Sysctl config files should be world-readable — no secrets here.
          contents:
            inline: |
                            # NovaCrestBank infra node kernel tuning
                            # Applied for Prometheus TSDB and LokiStack inotify requirements
                            # Change control: CR-2026-0420-SYSCTL-INFRA

                            fs.inotify.max_user_instances = 8192
                            # Default: 128. Elevated to 8192.
                              # Prometheus and LokiStack each create hundreds of inotify
                              # instances for TSDB directory watching, config map watching,
                              # and rule file monitoring. 128 is exhausted by Prometheus alone.

                            fs.inotify.max_user_watches = 524288
                            # Default: 8192. Elevated to 524288 (512K).
                              # Each watched file or directory consumes one watch.
                              # Prometheus TSDB with 15d retention watches thousands of
                              # block files. LokiStack watches the chunk WAL directory.
                              # 8192 is exhausted. 524288 provides substantial headroom.

                            vm.max_map_count = 262144
                            # Default: 65530. Elevated to 262144.
                              # Elasticsearch-compatible indices (used by LokiStack's
                              # index gateway) require a higher max_map_count.
                              # 65530 causes "max virtual memory areas vm.max_map_count
                              # is too low" errors in Loki index-gateway logs.
  kernelArguments: []
    # No kernel arguments for this MachineConfig — only sysctl file changes.
    # An empty kernelArguments list is explicit and avoids confusion.
    # Kernel argument changes require the same node reboot that sysctl changes need,
    # so combining them in the same MC is valid and reduces total reboot count.

UPG-03 · Section 08
Applying kernel arguments to worker nodes
Kernel arguments are set via the bootloader and take effect at the next node reboot. NovaCrestBank adds transparent_hugepage=madvise to worker nodes — this provides better memory performance for the JVM-based banking microservices without the unpredictable latency spikes that always mode causes.
Why transparent hugepage settings matter for JVM banking applications
By default, RHCOS uses transparent_hugepage=always — the kernel automatically promotes 4K pages to 2MB hugepages aggressively. For JVM applications (the payment-service, account-service), this causes periodic latency spikes: the kernel's khugepaged daemon consolidates pages in the background, which stalls memory allocation for the JVM during GC cycles. madvise mode allows the JVM to explicitly request hugepages for its heap (which it does) while preventing aggressive kernel-side promotion for the rest of memory. The result is consistent P99 latency without GC-correlated spikes.
99-ncb-worker-kernel-args — transparent hugepage mode for worker nodes
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 99-ncb-worker-kernel-args
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  kernelArguments:
    - transparent_hugepage=madvise
      # transparent_hugepage=madvise: hugepages are only promoted when the process
        # explicitly requests them (via madvise(MADV_HUGEPAGE)).
        # The JVM (used by all NovaCrestBank banking microservices) calls madvise
        # for its heap — it gets hugepages for the heap and benefits from the
        # reduced TLB pressure. Other processes get normal 4K pages.
        # Alternative values:
        # "always": aggressive hugepage promotion — causes GC latency spikes (avoid).
        # "never": no hugepages — loses JVM heap performance benefit (avoid).
        # "madvise": the correct setting for mixed JVM + general workload nodes.
    - net.ifnames=0
      # Keep predictable network interface names if needed. Usually not required
        # on VCF VMs where the interface name is fixed by vSphere.
        # Shown here as an example of combining multiple kernel arguments
        # in the same MachineConfig — they all take effect at the same reboot.
  config:
    ignition:
      version: 3.4.0
Kernel argument changes always require a node reboot — plan accordingly
Unlike some file-based MachineConfig changes that can be applied without a reboot, kernel argument changes require a full node reboot to take effect. The MCO will drain and reboot every worker node in sequence when this MachineConfig is applied. For NovaCrestBank's 3-worker cluster, this means ~36 minutes of rolling reboots (12 minutes per worker × 3 workers, sequential). Apply this during a maintenance window or pause the worker MCP and un-pause during the next scheduled maintenance window (Section 12).
UPG-03 · Section 09
Deploying a custom file to all nodes via MachineConfig
The MachineConfig storage.files section can deploy any file to any path on the RHCOS filesystem. This is used for custom CRI-O configuration, container registry mirrors, SSH known_hosts, and any other file that must exist on every node.
99-ncb-worker-registry-mirror — Harbor registry mirror for all worker nodes
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 99-ncb-worker-registry-mirror
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  config:
    ignition:
      version: 3.4.0
    storage:
      files:
        - path: /etc/containers/registries.conf.d/99-ncb-harbor-mirror.conf
            # /etc/containers/registries.conf.d/ is the directory where CRI-O
                # reads additional registry configuration files.
                # Files in this directory are automatically included — no need
                # to modify the main registries.conf file (which OCP manages).
          mode: 0644
          contents:
            inline: |
                            [[registry]]
                            prefix = "quay.io/openshift-release-dev"
                            location = "harbor.apps.ocp.novacrestbank.local/openshift-release-dev"
                            insecure = false

                            [[registry]]
                            prefix = "registry.redhat.io"
                            location = "harbor.apps.ocp.novacrestbank.local/redhat"
                            insecure = false
                            # Mirror registry.redhat.io pulls through Harbor.
                              # In air-gapped environments this is required —
                              # nodes cannot reach registry.redhat.io directly.
                              # The MachineConfig ensures this is on every worker
                              # (and applied as a matching config to master/infra pools).

        - path: /etc/pki/ca-trust/source/anchors/ncb-ca-bundle.crt

UPG-03 · Section 12
Pausing a MachineConfigPool — batching configuration changes safely
If multiple MachineConfig changes need to be applied to the same pool, pausing the pool before applying them causes all changes to be batched into a single rolling restart rather than triggering separate restarts for each change.
When to pause a pool — the batching advantage
NovaCrestBank needs to apply both the kernel argument change (Section 08) and the sysctl change (Section 07 equivalent for workers) to worker nodes. Why pause the pool first rather than applying them sequentially?
Without pausing: applying the kernel argument MC triggers the MCO to begin a rolling restart of all 3 workers (~36 minutes). Then applying the sysctl MC triggers another rolling restart of all 3 workers (~36 minutes). Total: 72 minutes of rolling reboots, each worker rebooting twice. Banking applications endure two separate periods of reduced replica count (2 of 3 replicas during each drain).

With pausing: pause the worker MCP, apply both MachineConfigs, then un-pause. The MCO computes one new rendered MC that incorporates both changes, then performs one rolling restart. Total: ~36 minutes, each worker reboots once. Banking applications endure one period of reduced replica count.

The pause-apply-unpause pattern is the correct approach whenever multiple MC changes are intended for the same pool. It is also used during cluster upgrades (UPG-02) to control when worker nodes reboot relative to the control plane upgrade.
jumphost — pause pool, apply multiple MCs, un-pause for single rollout

UPG-03 · Section 14
Draining a node safely — the correct procedure
Node draining is required before taking a node out of service for maintenance, hardware replacement, or manual intervention. Done incorrectly, a drain can violate PodDisruptionBudgets and cause banking application downtime. Done correctly, it is non-disruptive.
The two drain flags that matter most for a banking environment
oc adm drain has many flags. Which ones does NovaCrestBank use, and why are --ignore-daemonsets and --delete-emptydir-data always included?
Every production drain at NovaCrestBank uses both flags — but for specific reasons that matter.

--ignore-daemonsets: DaemonSet pods (Vector log collector, machine-config-daemon, ovnkube-node, node-exporter) cannot be drained — they are deliberately pinned to specific nodes. Without this flag, the drain command refuses to proceed because it cannot evict DaemonSet pods. With this flag, DaemonSet pods are left running on the node (they will terminate when the node reboots) and the drain proceeds with all other pods. Always include this flag — the DaemonSet pods being present on a draining node is safe and expected.

--delete-emptydir-data: Pods that use emptyDir volumes (temporary scratch space) are not evicted by default because evicting them would lose the data in the emptyDir. Most banking application pods do not use emptyDir — but OCP internal components (such as some operator pods) do. Without this flag, the drain hangs waiting for these pods to be evicted with their emptyDir data intact. With this flag, the drain evicts them anyway (the emptyDir data is lost — which is acceptable for scratch space by definition). Include this flag for production drains.

--timeout=300s: Set a timeout so the drain does not hang indefinitely if a pod refuses to evict (due to a PDB violation or a stuck finalizer). 300 seconds (5 minutes) is generous for most pods while still failing clearly if something is genuinely stuck.

UPG-03 · Section 15
Cordon, uncordon, and when to use each without a full drain
Cordoning marks a node unschedulable without evicting running pods. It is the lighter alternative to a full drain — used when you want to prevent new pods from landing on a node without disturbing what is already running.
OperationWhat it doesExisting podsNew pod schedulingWhen to use
cordon Marks node unschedulable=true Continue running undisturbed Blocked — scheduler ignores this node Preparing a node for maintenance. Running an investigation. Preventing new pods during a suspected node instability.
drain Cordons + evicts all evictable pods (respecting PDBs) Evicted to other nodes Blocked Before rebooting/shutting down a node. Before hardware maintenance. Before removing a node from the cluster.
uncordon Marks node unschedulable=false No change — pods stay where they are Restored — node is schedulable again After completing maintenance. After MCO finishes updating the node.
The MCO cordons and uncordons nodes automatically — do not interfere
During a MachineConfig rollout, the MCO cordons each node before applying changes and uncordons it after the node reboots and rejoins. Do not manually cordon nodes in a pool that is currently updating — the MCO tracks cordon state as part of its update process and a manually cordoned node may confuse the MCO's progress tracking. Only manually cordon nodes when the MCO is not running a rollout on that pool.

UPG-03 · Section 17
MachineSet — adding a new worker node via replica scaling
A MachineSet is the Kubernetes object that manages the lifecycle of OCP nodes — similar to how a Deployment manages pods. Scaling a MachineSet's replica count provisions a new node VM on vSphere, installs RHCOS, and joins it to the cluster automatically. This is the correct way to add capacity, not manual VM creation.
Why MachineSet scaling rather than manually creating a VM and running the OCP installer
A new VM could be created in vSphere, RHCOS installed manually, and the node joined to the cluster using the ignition config. Why use MachineSet instead?
Manual VM creation and node joining works but has four operational problems that MachineSet solves.

1. Configuration consistency: A MachineSet is a template — every node it creates has exactly the same vSphere configuration (CPU, memory, disk, network, datastore, folder) as every other node in the set. Manual VM creation relies on the engineer getting every parameter right. A MachineSet-created node is guaranteed to match the other nodes in the pool because they were all created from the same template.

2. Automatic MachineConfig application: When a MachineSet creates a node, the node is labelled with the pool's role label (e.g., node-role.kubernetes.io/worker: ""). The MachineConfigPool's machineConfigSelector automatically picks up the new node and the MCO applies the current rendered MachineConfig — including NovaCrestBank's custom sysctl and kernel arguments — to the new node automatically. No manual steps.

3. Lifecycle management: Scaling the MachineSet down removes the node cleanly (drain, delete, VM deletion). Manual nodes must be manually removed. A node that was manually added but is not in a MachineSet is an "unmanaged" node — the Machine API does not track it and it cannot be scaled down automatically.

4. Automatic RBAC and certificate joining: MachineSet-created nodes use a Machine Approval Controller that automatically approves their certificate signing requests. Manually added nodes require manual CSR approval.

UPG-03 · Section 18
What happens when a new node joins — the complete automated sequence
From MachineSet scale-up to the node being schedulable and running NovaCrestBank's custom configuration, the process is fully automated. Understanding each step explains why it takes ~15 minutes and what to watch if it stalls.
T+0
MachineSet creates the Machine object
The Machine API controller reads the MachineSet template and creates a Machine object in openshift-machine-api. The vSphere Machine provider reads the template (vCenter server, datacenter, cluster, datastore, network, VM template) and begins creating the vSphere VM. Phase: Provisioning.
T+2m
vSphere VM created, RHCOS first boot
The vSphere VM is powered on. RHCOS boots from the VM template. On first boot, Ignition runs: it downloads the worker ignition config from the MCS (Machine Config Server) running in the cluster, applies it (writes certificates, configures the kubelet, sets up CRI-O), and reboots. Phase: Provisioned.
T+5m
kubelet starts, node registers with API server
After the Ignition reboot, the kubelet starts and attempts to register with the API server. The kubelet generates a certificate signing request (CSR). The Machine Approver controller automatically approves the CSR (because the node's Machine object exists in the cluster). The node appears in oc get nodes with status NotReady.
T+8m
Node becomes Ready, MCO applies rendered MC
OVN-Kubernetes configures networking for the new node (NSX-T TEP tunnel established, overlay segment connectivity confirmed). The node transitions to Ready. The MachineConfigPool detects the new node (it has the worker role label) and the MCO schedules the rendered worker MachineConfig for application to this node. The MCD pod starts on the new node.
T+12m
Custom MachineConfig applied — node fully configured
The MCD on the new node detects it needs to apply the current rendered MC (which includes NovaCrestBank's sysctl and kernel arguments). It writes the sysctl config files and stages the kernel arguments via rpm-ostree, then reboots the node. After this final reboot (~3 minutes), the node comes up with all NovaCrestBank custom configuration applied. MCP UPDATEDMACHINECOUNT increments. The node is fully schedulable and identical in configuration to the other workers.
