NSX-04 · Section 03
NSX LB vs keepalived — what changes for OpenShift
The candidate built OCP with keepalived in the previous vSphere series — understanding what the NSX LB replaces and what stays the same
Exactly what keepalived did — and why NSX LB does it better
In the previous OCP deployment, keepalived ran inside the OCP master nodes and used VRRP to claim the API VIP. What exactly did it do and why does the NSX LB replace it?
keepalived broadcast VRRP announcements from the master nodes to claim ownership of the VIP IP address. Whichever master was the VRRP master held the VIP. If that master failed, a new VRRP election occurred among surviving masters and one of them claimed the VIP. This worked but had three limitations:

1. The VIP had to be on the same subnet as the masters. keepalived uses gratuitous ARP to claim the VIP — ARP is a Layer-2 protocol and only works within the same broadcast domain. The API VIP had to be in 10.50.0.0/24 for the same reason the masters were in 10.50.0.0/24.

2. Failover was slow — 3–10 seconds. VRRP uses advertisement intervals and dead timers. During that window, new connections to the API VIP failed. Existing connections dropped.

3. Traffic was not load-balanced. One master held the VIP and received all API traffic. The other two masters received no direct API traffic except when the primary master failed.

The NSX LB resolves all three:
1. The VIP (10.50.0.200) does not need to be on the same subnet as the masters — it is a floating IP managed by NSX on the Edge cluster. It can be any IP in any NSX-routable range.
2. Failover is sub-second — NSX LB detects a dead member within 2 consecutive failed probes (10 seconds with 5s interval) and stops sending traffic to it immediately. The health check window is tunable.
3. Traffic is genuinely load-balanced — Round Robin across all healthy master nodes simultaneously.

PCI-DSS v4.0 Req 6.4 — Automated technical solutions detect failures and route traffic away from failed components. The NSX LB satisfies this with active health monitoring and automated pool member removal.

NSX-04 · Section 04
Create the Load Balancer service — NCB-OCP-LB
The container object that all pools and virtual servers belong to — attached to NCB-T1-OCP on NCB-Edge-Cluster
Why the Load Balancer attaches to NCB-T1-OCP — not directly to the Edge Cluster
The Load Balancer runs on the Edge Nodes in NCB-Edge-Cluster. Why is it attached to NCB-T1-OCP instead of directly to the Edge Cluster?
Because the Load Balancer VIPs are addresses on the OCP network domain — the VIPs (10.50.0.200 and 10.50.0.201) are within the 10.50.0.0/24 segment that NCB-T1-OCP owns. Attaching the LB to NCB-T1-OCP places the VIPs in the correct routing context — when a client connects to 10.50.0.200, the T1 OCP gateway handles the forwarding to the Edge Nodes where the LB Virtual Server processes it.

Additionally, "All LB VIP IPs" route advertisement was enabled on NCB-T1-OCP in NSX-03. Attaching the LB to NCB-T1-OCP is what causes the VIP /32 host routes to be advertised via that T1. If the LB were attached to NCB-T1-Infra instead, the VIP routes would be advertised via the infrastructure T1 — wrong routing domain, and the Infra T1 has no route advertisement for LB VIPs enabled.

NSX-04 · Section 05
What health monitors do — and why they matter more than the load balancing algorithm
The health monitor is the automated failure detection mechanism — without it, the LB continues sending traffic to failed OCP nodes
What happens without a health monitor — why it is not optional
If no health monitor is configured, the Server Pool still distributes traffic via Round Robin. What is the consequence when a master node fails?
Without a health monitor, the Server Pool has no mechanism to detect that a pool member has failed. It continues sending new connections to that member according to the load balancing algorithm — Round Robin would send every third request to the dead master node. The clients (kubectl, the OCP web console, the installer) receive TCP connection timeouts for those requests. The LB has no visibility into backend health.

With a health monitor: the NSX LB probes each pool member every 5 seconds on the configured port (TCP 6443 for the API pool). If a probe fails twice consecutively (10 seconds total), the member is marked DOWN and removed from the active rotation. All subsequent traffic goes to the two surviving masters. When the failed master recovers and two consecutive probes succeed, it is marked UP and added back to the rotation.

For NovaCrestBank's banking environment: the OCP API handles authentication for all banking applications and cluster operations. A dead master node silently receiving API traffic causes 33% of API requests to timeout — banking operations fail intermittently, which is worse than a clean failure. Active health monitoring eliminates this class of failure.

PCI-DSS v4.0 Req 6.4 — Automated technical solutions detect when data has been tampered with, is absent, or when other anomalies affecting security of the application are detected. Active health monitoring is the mechanism that detects absent backend services.

NovaCrestBank health monitor parameters

ProtocolTCP — the simplest and most reliable probe for OCP. An HTTP probe would also work but TCP is sufficient since we only need to verify port availability, not HTTP response content.
Interval5 seconds — probe sent every 5 seconds to each pool member
Timeout5 seconds — if the TCP handshake does not complete within 5 seconds, the probe is counted as failed
Rise count (mark UP)3 — after 3 consecutive successful probes, a DOWN member is marked UP and re-enters the rotation
Fall count (mark DOWN)2 — after 2 consecutive failed probes, a member is marked DOWN and removed from rotation (10 seconds to detect failure)

NSX-04 · Section 08
What a server pool is and how it distributes traffic
The pool is the group of backend servers — understanding its key settings before creating the OCP pools
Pool settingWhat it controlsNovaCrestBank choice
Algorithm How incoming connections are distributed across pool members. Round Robin sends each new connection to the next member in sequence. Least Connection sends to the member with fewest active connections. IP Hash sends a given client IP always to the same member. Round Robin — appropriate for OCP API (stateless from the LB perspective, TLS sessions handle stickiness) and Ingress (OCP Router handles session affinity per Route object).
Members The backend server IPs and ports. Each member entry is: IP address + port + weight. Weight controls the relative share of traffic — equal weight = equal distribution. API pool: master-0/1/2 at 10.50.0.11–.13:6443. Ingress pool: infra-0/1/2 at 10.50.0.21–.23:443.
Health Monitor Which health monitor to use for this pool. The monitor probes each member on its configured port. API pool → NCB-HM-OCP-API. Ingress pool → NCB-HM-OCP-Ingress.
SNAT mode Whether the LB replaces the client source IP with an LB IP (SNAT) or passes the original client IP through (Transparent). For OCP API, Transparent is preferred so master nodes see the actual client IP in their audit logs. Transparent for both pools — OCP audit logs preserve original client IP addresses. PCI Req 10.2.1 — audit logs capture all access with the originating IP.
TCP multiplexing Whether the LB reuses TCP connections to backends (multiplexing) or creates a new TCP connection per client connection (passthrough). For OCP API with TLS, passthrough is required — TLS handshake must terminate at the master node for mutual TLS to work. Disabled — OCP API uses mutual TLS. TCP passthrough preserves the full TLS connection end-to-end.

NSX-04 · Section 11
What a Virtual Server is — the VIP that clients connect to
The Virtual Server is the public face of the load balancer — the IP address and port that DNS resolves to
Why three Virtual Servers for two VIP addresses
NovaCrestBank has two VIPs: 10.50.0.200 (API) and 10.50.0.201 (Ingress). Why are three Virtual Servers needed?
Because the Ingress VIP serves two different ports (443 for HTTPS and 80 for HTTP) that need separate Virtual Server configurations. NSX Load Balancer creates one Virtual Server per IP+port combination. Therefore:

NCB-VS-OCP-API: 10.50.0.200 port 6443 → NCB-Pool-OCP-API (masters)
NCB-VS-OCP-HTTPS: 10.50.0.201 port 443 → NCB-Pool-OCP-Ingress (infra nodes)
NCB-VS-OCP-HTTP: 10.50.0.201 port 80 → NCB-Pool-OCP-Ingress (same infra nodes)

Both HTTPS and HTTP Virtual Servers use the same backend pool (NCB-Pool-OCP-Ingress) because the same infra nodes serve both. The OCP Router (HAProxy) on the infra nodes handles both ports. Separating them into two Virtual Servers allows port 80 traffic to be forwarded independently — the OCP Router typically redirects HTTP (port 80) to HTTPS (port 443), but the redirect happens at the application layer, not the LB layer.

Virtual Server key fields

FieldWhat it controls
VIP AddressThe IP the Virtual Server listens on — the IP that DNS resolves to. Must be a free IP not assigned to any VM or other device.
PortThe TCP port the Virtual Server listens on. Incoming connections on this IP:Port are forwarded to the pool.
Load BalancerWhich LB service this Virtual Server belongs to — NCB-OCP-LB.
Server PoolWhich pool receives the forwarded traffic. Each Virtual Server forwards to exactly one pool.
ProtocolTCP for all OCP VIPs — Layer 4 passthrough. The LB does not inspect or modify the payload. HTTP/HTTPS protocols would enable Layer 7 features but require TLS termination at the LB, which would break OCP's end-to-end TLS.

What NSX-05 builds next — Distributed Firewall

NSX-04 completed the networking and load balancing. NSX-05 adds the security layer — the NSX Distributed Firewall (DFW) that enforces microsegmentation between the security zones defined in VCF-01 Section 15.

NSX-05 screen-by-screen steps:

  • Create Security Groups (groups of VMs by segment or tag) — one group per security zone
  • Create a Default-Deny policy — the last rule in the DFW that blocks all traffic not explicitly permitted
  • Create the inter-zone allow rules: OCP → Infra on TCP 636 (LDAPS), TCP/UDP 53 (DNS), TCP 5696 (KMIP/HSM)
  • Create intra-zone rules: OCP node-to-node communication (required for etcd, cluster networking, and pod scheduling)
  • Verify with Traceflow that allowed traffic passes and denied traffic is blocked

After NSX-05, the complete NSX configuration for NovaCrestBank VCF + OCP is done. The next phase moves to Windows Server (WIN-VCF-01 — DNS records for VCF) and then the OCP installation itself.
