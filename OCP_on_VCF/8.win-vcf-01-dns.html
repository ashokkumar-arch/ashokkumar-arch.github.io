PCI-DSS v4.0 Req 12.3.2 — DNS records as component inventory
PCI-DSS requires maintaining a current and accurate inventory of all system components in scope. DNS records serve as that inventory — each record names, identifies, and locates a component. The PTR records add bidirectional verification: a hostname resolves to an IP (A record) and that IP resolves back to the hostname (PTR). SDDC Manager validates PTR records during VCF bring-up as a pre-check. Missing PTRs fail the pre-check and block deployment.

Why PTR records are required — not optional — for VCF
PTR records resolve an IP address back to a hostname. For VCF, are they truly required or just good practice?
Required. SDDC Manager runs a DNS pre-check during bring-up that performs a reverse lookup on each ESXi host's management IP. It expects the PTR to return the correct FQDN. If any PTR is missing or mismatched, VCF bring-up fails at the pre-check stage with a "DNS validation error" for that host — before any deployment begins.

NSX Manager also validates reverse DNS for its node IPs during cluster formation. The OCP IPI installer validates forward and reverse DNS for all node FQDNs before placing any VM. A missing PTR record causes a "machine DNS validation failed" error during bootstrapping.

The Windows DNS Manager checkbox "Create associated pointer (PTR) record" automates PTR creation at the same time as the A record. Always tick it. It is faster and eliminates the chance of a typo in a manually created PTR.

WIN-VCF-01 · Section 07
How to create the wildcard A record for OCP Ingress
The *.apps.ocp record is the only wildcard in this guide — it needs special handling in the Name field
What a wildcard DNS record does and why OCP needs one
OCP creates Route objects for every application — banking-portal.apps.ocp.novacrestbank.local, console-openshift-console.apps.ocp.novacrestbank.local, and hundreds more. Is a separate DNS record needed for each?
No — one wildcard record covers all of them. A DNS wildcard record with the name *.apps.ocp.novacrestbank.local matches any query for [anything].apps.ocp.novacrestbank.local that does not have a more specific record. All queries return the Ingress VIP (10.50.0.201).

When a user navigates to banking-portal.apps.ocp.novacrestbank.local: DNS has no specific record for "banking-portal" but finds the wildcard. It returns 10.50.0.201. The request reaches the NSX LB Ingress Virtual Server, which forwards to an OCP infra node HAProxy, which routes to the correct OCP Route for banking-portal.

In the DNS Manager New Host dialog, the Name field accepts an asterisk. Enter *.apps.ocp exactly — the asterisk followed by the sub-labels. No PTR record is created for the wildcard because a wildcard is not assigned to a single physical host.
  • 1
    In novacrestbank.local zone — right-click → New Host (A or AAAA)…

    Same as any other A record — right-click empty space in the novacrestbank.local right panel → New Host (A or AAAA)…

WIN-VCF-01 · Section 13
OCP API VIP, API-int VIP, and Ingress wildcard record
Three records that the OCP installer validates before placing any node VM. All three must exist before the installer runs.
These three records are the minimum DNS required before OCP installation begins
The OCP IPI installer performs DNS validation during its pre-flight checks. It resolves api.ocp.novacrestbank.local, api-int.ocp.novacrestbank.local, and a test subdomain under apps.ocp.novacrestbank.local. If any of these fail to resolve, the installer exits before placing any VM. These records are the gate that must be cleared before OCP-VCF-01.
Critical naming rule — OCP records use sub-labels, not just a hostname

WIN-VCF-01 · Section 16
Force AD replication to DC02
DNS records created on DC01 are stored in Active Directory and replicate to DC02 automatically. Forcing immediate replication ensures DC02 has all records before VCF bring-up begins.
Why force replication rather than waiting for the automatic interval
Active Directory replication is automatic — records created on DC01 will eventually reach DC02. Why force it manually?
The default AD replication interval is 15 minutes within a site. During VCF bring-up or OCP installation, components may send DNS queries to DC02 (the secondary DNS server) before automatic replication completes. If the records are not yet on DC02 and the component receives a NXDOMAIN response from DC02, the deployment can fail even though DC01 has the correct record.

Forcing replication immediately after completing all DNS records eliminates this race condition. VCF and OCP deployments can take 30–90 minutes — in that window, normal replication would have occurred anyway, but forcing it now removes all uncertainty.
