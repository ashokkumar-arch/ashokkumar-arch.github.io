What this guide builds — in order

  1. IP Address Pools — NCB-Host-TEP-Pool and NCB-Edge-TEP-Pool. NSX automatically assigns TEP IPs from these pools when hosts are prepared.
  2. Transport Zones — NCB-Overlay-TZ (for all workload segments) and NCB-VLAN-TZ (for Edge Node uplinks). Every segment created later will reference one of these TZs.
  3. Uplink Profiles — NCB-Host-Uplink-Profile and NCB-Edge-Uplink-Profile. These tell NSX which physical NIC(s) to use on each ESXi host for TEP traffic and which VLAN to tag TEP packets with.
  4. Host Transport Nodes — NSX "prepares" each ESXi host: installs the NSX VIB (kernel module), creates the TEP VMkernel using the IP pool, and joins the host to the Transport Zone. After this step, all 9 ESXi hosts become Transport Nodes and can participate in the NSX overlay.
Why this order — each step depends on the previous one
The IP Pool must exist before the Transport Node is prepared (NSX needs the pool to assign a TEP IP). The Transport Zone must exist before the Transport Node is prepared (NSX needs to know which TZ to add the host to). The Uplink Profile must exist before the Transport Node is prepared (NSX needs to know which physical NIC and VLAN to use for the TEP). This is why these seemingly simple objects are created first — they are the inputs to the host preparation step that actually builds the overlay fabric.

NSX-01 · Section 03
Why IP address pools are created first
Understanding the role of IP pools before touching the screen — so every field makes sense when it appears
What the IP pool does when a host is prepared
When NSX prepares an ESXi host as a Transport Node, it needs to create a TEP VMkernel interface on that host. Where does the TEP IP address come from?
From the IP Address Pool. When NSX prepares esxi-ocp-01, it looks at the IP pool referenced in the Transport Node Profile, takes the next available IP (10.100.0.21), assigns it to the TEP VMkernel it creates on that host, and marks 10.100.0.21 as used in the pool. When esxi-ocp-02 is prepared, NSX takes the next IP (10.100.0.22), and so on.

This is automatic — the administrator never manually configures a TEP IP on any ESXi host. The IP pool is the mechanism that makes this automation possible. Without a pre-defined pool, NSX cannot assign TEP IPs and the host preparation fails.

One pool per TEP VLAN: NovaCrestBank has two TEP VLANs — VLAN 100 for ESXi hosts and VLAN 101 for Edge Nodes. These are on different subnets (10.100.0.0/24 and 10.101.0.0/24), so they need two separate pools: NCB-Host-TEP-Pool and NCB-Edge-TEP-Pool.

The two IP pools NovaCrestBank needs

NCB-Host-TEP-PoolSubnet: 10.100.0.0/24 · Range: 10.100.0.11 – 10.100.0.30 · No gateway (TEPs communicate at L2 within VLAN 100) · Used by: all 9 ESXi hosts
NCB-Edge-TEP-PoolSubnet: 10.101.0.0/24 · Range: 10.101.0.11 – 10.101.0.20 · No gateway · Used by: Edge Nodes (configured in NSX-02)
Navigation path for IP Address Pools
In NSX Manager: Networking → IP Management → IP Address Pools. This is under the Networking tab, not the System tab — because IP Pools are considered a networking resource (they are used to assign IPs to network interfaces).

NSX-01 · Section 06
What the Transport Zone screens look like — before creating anything
Understanding what you will see in the NSX Manager GUI when working with Transport Zones
Navigation path for Transport Zones
In NSX Manager: System → Fabric → Transport Zones. Transport Zones are in the System tab because they are infrastructure-level fabric configuration — not day-to-day networking objects. VCF may have already created default Transport Zones during bring-up. If they exist, verify their names and types. If they match NovaCrestBank's requirements, they can be used. If not, create new ones.
What to do if VCF already created Transport Zones
VCF bring-up sometimes creates default Transport Zones with names like "nsx-overlay-transportzone" and "nsx-vlan-transportzone." Should these be used or should NovaCrestBank create new ones?
If VCF created default Transport Zones, NovaCrestBank can either use them (renaming them to match the NCB naming convention) or create new ones. The important thing is that the Management Domain's hosts are already prepared using the default TZs — do not delete them.

Recommended approach: if default TZs exist and the Management Domain hosts are already using them, create additional Transport Zones specifically for the OCP Workload Domain with NovaCrestBank's naming convention. If no default TZs exist (VCF did not create them, or this is a fresh NSX-T deployment), create both NCB-Overlay-TZ and NCB-VLAN-TZ from scratch.

For this guide, we proceed as if creating both Transport Zones fresh — the GUI steps are identical whether they are the first or additional TZs.

NSX-01 · Section 10
What an Uplink Profile is — before creating one
The Uplink Profile answers three questions NSX has about every ESXi host: which physical NIC to use for TEP traffic, which VLAN to tag TEP packets with, and how to team multiple NICs together
The three questions an Uplink Profile answers
When NSX prepares an ESXi host, it needs to create a TEP VMkernel. But where should that VMkernel send its packets? Which physical NIC? Which VLAN? How should traffic be distributed across multiple NICs?
The Uplink Profile answers all three:

1. Which physical NIC (Uplink)? The Uplink Profile defines uplink names (e.g., "uplink-1" and "uplink-2"). When the Transport Node Profile is created later, it maps these uplink names to actual physical NICs (e.g., vmnic2 and vmnic3) on each ESXi host.

2. Which VLAN? The Uplink Profile has a "Transport VLAN" field — the VLAN ID that NSX tags TEP VMkernel packets with. For NovaCrestBank's ESXi hosts, this is 100 (Host TEP VLAN). For Edge Nodes, it is 101 (Edge TEP VLAN).

3. How to team multiple NICs? The Uplink Profile defines a teaming policy — Active-Active (LACP or load-balancing across both NICs) or Active-Standby (one NIC active, one standby). For NovaCrestBank's LACP bonded NICs, Active-Active is used.

Why two Uplink Profiles? ESXi hosts use VLAN 100 for TEP traffic (Host TEP VLAN), while Edge Nodes use VLAN 101 (Edge TEP VLAN). Because the TEP VLAN is different, the Uplink Profile is different — even though the teaming policy and uplink count are the same.
Navigation path for Uplink Profiles
In NSX Manager: System → Fabric → Profiles → Uplink Profiles. Click the System tab → Fabric → Profiles. Inside Profiles, select the Uplink Profiles tab.

NSX-01 · Section 13
What "preparing a host" means — before touching the screen
Host Transport Node preparation is the step where NSX actively reaches into each ESXi host and installs the overlay networking capability — understanding what happens makes every field meaningful
What happens on the ESXi host when NSX prepares it
When NSX Manager "prepares" an ESXi host as a Host Transport Node, what physically changes on that ESXi host?
NSX Manager connects to the ESXi host via the management VMkernel (vmk0) and performs the following sequence — automatically, without administrator interaction on the ESXi host:

1. Installs the NSX VIB (Vibrational Installation Bundle) — a kernel module that adds overlay networking capabilities to the ESXi kernel. This is the DFW enforcement engine, the GENEVE encapsulation/decapsulation engine, and the TEP VMkernel driver. Installing the VIB does not require a host reboot in most cases (NSX 4.x supports live VIB installation).

2. Creates the TEP VMkernel interface — NSX creates a new VMkernel (vmk-tep or similar) on the host, assigns it an IP from the TEP IP pool (e.g., 10.100.0.21 from NCB-Host-TEP-Pool), tags it with the Transport VLAN from the Uplink Profile (VLAN 100), and maps it to the physical NIC specified in the Uplink mapping.

3. Joins the host to the Transport Zone — the host registers itself with NSX Manager as a member of NCB-Overlay-TZ. NSX Manager records that this host can participate in Overlay segments.

4. Establishes the BFD session — NSX Manager starts monitoring the host's TEP reachability using BFD (Bidirectional Forwarding Detection). If the TEP becomes unreachable, NSX Manager is notified within milliseconds.

The end state: The ESXi host now has an NSX kernel module, a TEP VMkernel with an IP address from the pool, and membership in the Transport Zone. It is a Transport Node. Any overlay segment created in NCB-Overlay-TZ is automatically reachable from this host.

The two ways to prepare hosts in NSX Manager

Method 1 — Prepare individual hosts

System → Fabric → Transport Nodes → Host Transport Nodes → Add. Add each ESXi host one by one. More control, more work. Good for troubleshooting a specific host.

Method 2 — Prepare a vCenter cluster (used in this guide)

System → Fabric → Transport Nodes → Host Transport Nodes → Configure NSX for this cluster. Applies a Transport Node Profile to all hosts in a vCenter cluster simultaneously. NSX prepares all hosts in parallel. This is the correct method for NovaCrestBank.

Transport Node Profile — the template for cluster preparation
When preparing a cluster, NSX uses a "Transport Node Profile" — a saved configuration that specifies: which Transport Zone to join, which Uplink Profile to use, and how to map physical NICs to uplink names. Once the profile is applied to a cluster, every host in that cluster (and every future host added to that cluster) is prepared automatically using the same profile. This is the VCF-integrated workflow — SDDC Manager can also trigger this via its own UI, but the NSX Manager GUI gives the most visibility into what is happening.

Understanding the two status columns

Node Status — is the host reachable?

Up — NSX Manager can communicate with the NSX agent on this host. The NSX VIB is installed and running. The TEP VMkernel is up.

Degraded — NSX can reach the host but some component has a problem. Investigate immediately.

Down — NSX cannot reach the host. Check vmk0 management connectivity first, then NSX VIB installation status.

Config State — did configuration apply correctly?

Success — the Transport Zone membership, TEP configuration, and Uplink mapping were all applied correctly to this host.

Partial Success — some configuration applied but not all. Click the host name to see which step failed.

Failed — configuration did not apply. Common causes: wrong vmnic mapping (NIC not connected to the right VLAN), VLAN 100 not trunked to this host's port-channel, MTU mismatch.

If any host shows Failed Config State — do not proceed
A host with Failed configuration state has no working TEP VMkernel. Any overlay segment will show that host as unavailable. VMs placed on that host will not have overlay network connectivity. Fix the failure on that specific host before proceeding to NSX-02. Click the host name → view the configuration details → the failed step is shown with an error message. Common fix: verify vmnic mapping and VLAN trunk configuration on the Cisco switch port-channel for that host.

What BFD Status means and why it matters more than a simple ping
The table shows "BFD Status" not "Ping Status." What is BFD and why does NSX use it instead of ping to verify TEP connectivity?
BFD (Bidirectional Forwarding Detection) is a lightweight, fast-failure-detection protocol that NSX uses to monitor TEP-to-TEP connectivity. BFD sends hello packets every 300ms and declares a link failure if 3 consecutive hellos are missed — total detection time 900ms. A ping would take 3-5 seconds to detect a failure.

Why this matters for NovaCrestBank: if a host's TEP becomes unreachable (NIC failure, switch port failure, VLAN misconfiguration), NSX Manager knows within 1 second via BFD. It immediately removes that host's TEP from the overlay fabric and stops sending overlay traffic to it. Without BFD, NSX would only discover the failure when a VM tried to communicate through the broken TEP — causing packet loss until the next connectivity check.

NIST SP 800-53 SI-2 — Flaw Remediation: rapid fault detection (BFD sub-second failure detection) ensures that network path failures are identified and traffic is rerouted before applications experience sustained outage.

What NSX-02 builds next — Edge Transport Nodes
