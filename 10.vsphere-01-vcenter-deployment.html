What vCenter is — and why the entire cluster depends on it

vCenter Server — the management brain of the vSphere cluster
Each ESXi host you installed in ESXi-01 can independently run virtual machines — but it has no knowledge of the other four hosts. vCenter Server is the centralised platform that sees all five hosts simultaneously and provides the features that require cross-host coordination: vMotion (live VM migration between hosts), HA (automatic VM restart if a host fails), DRS (automatic workload balancing), vSAN (distributed storage), and the Distributed Virtual Switch (vDS). Without vCenter, none of these features exist. With vCenter, the five hosts become a single unified cluster with shared resources.

vCenter is itself a VM — deployed onto esxi01

vCenter Server ships as a pre-built Linux virtual appliance called the VCSA (vCenter Server Appliance). You do not install it from scratch — VMware provides a finished VM image. You run an installer application on your laptop, which connects to esxi01 over the network, uploads the VCSA image, creates a new VM, and configures it. The VCSA VM then runs permanently on esxi01 — or later on any host in the cluster.

Where vCenter runs in NovaCrestBank's environment
vCenter will be deployed as a VM on esxi01, using a temporary standard vSwitch for network access during the initial deployment. After the vDS is configured (vSphere-02), vCenter's VM network adapter will be migrated to the vDS port group. vCenter's IP address will be 10.10.0.100 on VLAN 10 (the ESXi management network) — configured in the deployment wizard and permanent from that point forward.

VCSA architecture — embedded PSC vs external PSC
Understanding the deployment model before the installer asks — so you know exactly what to select and why
The deployment model question — answered before the installer asks
The VCSA installer will ask you to choose a deployment model. In older versions of vCenter (6.x and earlier), there were complex choices about Platform Services Controllers (PSC). In vCenter 7.0 and later, VMware simplified this: the PSC is always embedded inside the vCenter appliance — you cannot separate them anymore. For NovaCrestBank running vCenter 8.0, there is only one choice: vCenter Server with an embedded Platform Services Controller. This is selected automatically. You do not need to worry about external PSC architecture.

What the embedded PSC means

The Platform Services Controller (PSC) handles identity and authentication for vCenter — specifically the Single Sign-On (SSO) service that manages the vsphere.local SSO domain. In the embedded model, the PSC runs inside the same VCSA virtual machine as vCenter itself. This is the correct and recommended architecture for NovaCrestBank because:

  • NovaCrestBank has a single vCenter instance — no multi-site topology requiring a separate PSC
  • The embedded model has fewer components to manage, fewer failure points, and is easier to backup and restore
  • VMware deprecated the external PSC in vCenter 7.0 — it no longer exists as an option in vCenter 8.0

SSO domain — vsphere.local

During setup, vCenter creates a private SSO domain named vsphere.local by default. This is NOT the same as Active Directory — it is vCenter's own internal identity store. The SSO domain contains one built-in administrator account: administrator@vsphere.local. This account is what you use to log into vCenter immediately after deployment — before AD integration is configured. After the Windows DC and AD integration are set up (in a later guide), you will add AD groups to vCenter roles and use your normal AD accounts for daily operations. The administrator@vsphere.local account becomes a break-glass account used only when AD is unavailable.

The temporary network path — why this works before the vDS exists
ESXi creates a default standard vSwitch (vSwitch0) with vmnic0 and a port group called "VM Network" during installation. This port group carries untagged traffic on the native VLAN. The Cisco leaf switch trunk port (Po101) has native VLAN 1 — which means untagged traffic from the VM Network port group lands on VLAN 1. VLAN 1 has no SVI and no routing. This means vCenter's VM cannot reach the internet or other VLANs — but that is fine. vCenter only needs to reach the ESXi management network (VLAN 10, 10.10.0.0/24) for deployment. However, there is a problem: the VM Network port group does not carry VLAN 10 because it is untagged. We need to fix this before deploying vCenter.

The required fix — create a VLAN 10 port group on esxi01 before deploying vCenter

Before running the VCSA installer, you must create a port group on esxi01's vSwitch0 that carries VLAN 10. vCenter's network configuration during the wizard will specify this port group — and vCenter's management VMkernel must reach 10.10.0.0/24. Without this port group, vCenter deploys but is unreachable on the network.

Downloading and mounting the VCSA installer
The VCSA installer is a large ISO file that contains both the appliance image and the installer application — here is how to get it and run it
  • 1
    Download the VCSA ISO from VMware Customer Connect

    Go to customerconnect.vmware.com. Log in. Navigate to Products → VMware vSphere → VMware vCenter Server → Download. Select version 8.0 Update 3. Download the file labelled VMware vCenter Server Appliance — the filename looks like:
    VMware-VCSA-all-8.0.3-24022515.iso
    This file is approximately 9.5 GB. Download it to your laptop.

  • 2
    Mount the ISO as a virtual drive on your laptop

    On Windows 10/11: Right-click the ISO file in File Explorer → Mount. Windows mounts it as a drive letter (e.g., D: or E:). A File Explorer window opens showing the ISO contents.

    On macOS: Double-click the ISO file. macOS mounts it and shows it in Finder under Locations. A new volume appears with the VMware name.

  • 3
    Navigate to the installer folder

    Inside the mounted ISO, open the folder vcsa-ui-installer. Inside that folder you will see three subfolders: win32, mac, and lin64. Open the folder matching your laptop's operating system.

  • 4
    Run the installer application

    On Windows: Inside the win32 folder, double-click installer.exe. A Windows security dialog may appear asking "Do you want to allow this app to make changes to your device?" — click Yes.

    On macOS: Inside the mac folder, double-click Installer.app. macOS may show a Gatekeeper warning: "This app cannot be opened because it is from an unidentified developer." If this appears, go to System Preferences → Security & Privacy → General and click Open Anyway next to the VMware Installer entry.

    The VCSA installer application window opens — a dark blue interface with the VMware vCenter Server logo. This is the installer you will use for the next 30 minutes.

This warning appears because esxi01 uses a self-signed TLS certificate — the same reason the browser warned you in the ESXi-01 guide. The certificate is legitimate (it is the one ESXi generated during installation). Click Yes to accept and continue.

Note the SHA1 fingerprint — this is how you verify authenticity
A production-grade check: before clicking Yes, verify the SHA1 fingerprint shown here matches the fingerprint you can retrieve from esxi01 directly. You will see one datastore: datastore1 — the VMFS volume that ESXi created on the M.2 RAID-1 array during installation. It has approximately 930 GB free. Click it to select it (it may already be selected). Tick Enable thin disk mode.

Thin provisioning — why it is correct here
Thin provisioning means the 415 GB storage allocation for the VCSA is not immediately consumed on disk. The VCSA VM starts by using only the actual data written (perhaps 50–80 GB initially) and grows toward 415 GB over time as vCenter accumulates logs, statistics, and configuration data. This is the correct choice for the initial deployment — it leaves more space on datastore1 for the Windows Server VMs (DC1, DC2, Root CA, Issuing CA) that will be created before vCenter is moved to vSAN.

What happens during Stage 1 deployment

  • 1
    VM is created on esxi01 (seconds)

    The installer connects to esxi01's API and creates a new VM with the specified hardware configuration (2 vCPUs, 12 GB RAM). The VM appears in the esxi01 Host Client inventory almost immediately.

  • 2
    The VCSA disk image is uploaded (15–20 minutes)

    The installer uploads the 9.2 GB VCSA disk image from your laptop to esxi01 over the network. This is the longest step — and the speed depends entirely on your network connection to esxi01. On a gigabit connection with low latency it takes about 5–8 minutes. On a slower connection or VPN it may take 20+ minutes. Do not close the installer window or let your laptop sleep.

  • 3
    VM is powered on and boots (2–3 minutes)

    After the upload, the installer powers on the VCSA VM. The VM boots a Linux OS (Photon OS — VMware's purpose-built Linux), initialises its services, and applies the network configuration (IP, hostname, gateway, DNS) that you specified. The progress log shows "Power on VM" completing.

ℹYou can open a browser tab to https://10.10.0.11/ui (esxi01 Host Client) and watch the VCSA VM appear in the Virtual Machines list. You will see it power on and begin consuming CPU and memory as it boots. This is a good sanity check that the deployment is progressing correctly.

If Stage 1 fails — what to do
If the deployment fails mid-way, the installer shows an error message with a description. Common causes: (1) No VLAN 10 port group on vSwitch0 — the VM deployed but cannot get a network connection. Fix: create the port group, then delete the failed VCSA VM from esxi01's Host Client and restart the installer. (2) Not enough disk space — the datastore was full. Fix: check available space on datastore1 in the Host Client. (3) Network timeout — your laptop connection to esxi01 was interrupted during upload. Fix: check your network path to esxi01 and restart the installer. In all cases, delete the partially-created VM from esxi01 before restarting.

Setting up the vCenter inventory hierarchy — Datacenter contains the Cluster, Cluster contains the hosts

The vCenter inventory hierarchy

vCenter organises everything in a tree: vCenter → Datacenter → Cluster → Host → VM. Before you can add ESXi hosts, you must create a Datacenter object and a Cluster object to put them in.

Step 1: Create the Datacenter

  • 2
    Name the Datacenter

    Datacenter name: NovaCrestBank-DC1

    Click OK. The Datacenter object appears in the inventory tree as a folder icon labelled NovaCrestBank-DC1.

Step 2: Create the Cluster inside the Datacenter

  • 3
    Right-click NovaCrestBank-DC1 → New Cluster

    Right-click NovaCrestBank-DC1 in the inventory tree. Click New Cluster. The New Cluster wizard opens.

  • 4
    The New Cluster wizard — what you see and what to configure

    The wizard has a name field at the top and toggle switches for cluster features. Configure as follows:

    Cluster name: NovaCrestBank-Cluster01
    vSphere DRS: OFF for now — DRS requires shared storage (vSAN), which does not exist yet. Enable after vSAN is configured.
    vSphere HA: OFF for now — HA also requires shared storage. Enable after vSAN.
    vSAN: OFF for now — vSAN is configured in its own dedicated guide after the vDS is set up.
    vSphere Lifecycle Manager (vLCM): Leave default — handles ESXi patch management. Configure after initial setup.

    Click OK. The cluster appears in the inventory tree inside the Datacenter.

Why DRS and HA are left off — the dependency chain
DRS automatically migrates VMs between hosts to balance load. HA automatically restarts VMs on surviving hosts when a host fails. Both features require that VMs can be migrated between hosts — which requires shared storage. vSAN is NovaCrestBank's shared storage. Until vSAN is configured and VMs are stored on vSAN, there is nothing to migrate. Enabling DRS and HA now would generate warnings about missing shared storage and attempt to activate features that cannot work. Enable them in the correct order: vSAN first, then HA, then DRS.

What comes next — vSphere-02: vDS Phase 1

vCenter is running but the network is still using temporary standard vSwitches on each ESXi host. The next guide creates a Distributed Virtual Switch (vDS) — a single virtual switch that spans all five hosts. The vDS is what makes consistent networking possible across the cluster: one configuration, consistent port groups on every host, LACP for the ESXi-to-leaf trunk ports. The vDS migration is done in phases to avoid losing connectivity to any host during the switchover.
