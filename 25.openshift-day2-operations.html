Guide 6 · Foundation · Section 01
Operations mindset — what Day-2 actually means
The shift from "getting it running" to "keeping it running safely for years" — the real work of an OpenShift engineer
The real-world truth about Day-2
The install took 45 minutes. Day-2 operations will occupy the next 3–5 years. Cluster upgrades happen every 4–6 weeks. Certificates expire on a schedule. New nodes are added as workloads grow. etcd must be backed up before every upgrade. Alerts fire at 2 AM. Knowing how to handle all of this — calmly and correctly, the first time — is what separates a senior OpenShift engineer from someone who installed OCP once. This guide gives you exactly that.

The three things that cause the most Day-2 incidents in banking OCP clusters
1. Certificate expiry — not noticed until it causes an outage. Solve with proactive alerting (Section 14).
2. Upgrade started without etcd backup — if the upgrade fails mid-way and etcd is corrupted, recovery is much harder without a recent backup. Always backup before upgrading.
3. Node added without anti-affinity awareness — adding a worker to the wrong ESXi host violates the anti-affinity rules and puts two master replicas on the same physical host.

Guide 6 · Cluster Upgrades · Section 03
How OCP upgrades work — the mechanism
What actually happens inside the cluster during an upgrade — so you know what is normal and what is a problem
The key insight — OCP upgrades itself
OpenShift uses the Cluster Version Operator (CVO) to manage its own upgrade. You give it a target version. It downloads the release image, updates each cluster operator one at a time, drains and updates each node one at a time. You do not manually update anything. The entire process is automated, rolling, and zero-downtime — as long as your workloads are properly configured with replicas and PodDisruptionBudgets.

Upgrade version channels — which one to use

ChannelExampleWhat it meansNovaCrestBank
stable-4.15 4.15.12 → 4.15.15 z-stream patches only. Bug fixes and CVE patches. No new features. Minimal risk. Monthly — use this
fast-4.15 4.15.12 → 4.15.15 Same as stable but available 1–2 weeks earlier. Less soak time. Not for banking — wait for stable
candidate-4.16 4.15.x → 4.16.0 Minor version upgrade. New features. Requires testing in staging first. Quarterly — staging first, then prod
stable-4.16 4.15.x → 4.16.x Minor version, stable channel. Safest path for minor version upgrades. Quarterly after staging validation

What happens during an upgrade — the sequence

1
T+0:00 — you run the upgrade command
CVO downloads the release image
The Cluster Version Operator fetches the target release image from quay.io. Verifies signature. Extracts manifests for all operators. The cluster keeps running normally — nothing changes yet.
2
T+5:00 — cluster operators update
Each cluster operator updates one at a time
CVO updates operators in dependency order. kube-apiserver → kube-controller-manager → kube-scheduler → etcd → machine-config-operator → others. Each operator does a rolling restart of its pods. You see PROGRESSING=True on each operator in turn. Takes 15–30 minutes.
3
T+30:00 — machine config rollout begins
Nodes update one at a time — the longest phase
The Machine Config Operator (MCO) applies the new RHCOS version to each node. It drains the node (evicts all pods), applies the new OS config, reboots the node, waits for it to come back Ready, then moves to the next node. Masters first, then workers. One node at a time — never two simultaneously.
4
T+60:00–T+180:00 — node-by-node
Each node: drain → reboot → rejoin
Each node takes 8–15 minutes: drain (2–5 min), reboot (3–5 min), come back Ready (2–3 min). With 9 nodes: 9 × 12 min ≈ 108 minutes for all nodes. Your workloads keep running — they move to other nodes while each is being updated.
5
T+120:00–T+200:00 — complete
Upgrade complete — all nodes on new version
CVO reports the cluster is now on the new version. All operators show Available=True. All nodes show the new Kubernetes version. Zero downtime — every workload ran continuously throughout.
What can cause downtime during an upgrade — and how to prevent it
Upgrades are zero-downtime only if workloads are prepared. Single-replica pods WILL have downtime when their node is drained. Ensure all production workloads in banking namespaces have: replicas: 2+, a PodDisruptionBudget allowing at least one pod to be evicted, and podAntiAffinity so replicas are not on the same node. Check this before every upgrade.

Guide 6 · Cluster Upgrades · Section 07
If things go wrong — upgrade rollback
What to do when an upgrade stalls, an operator goes Degraded, or a node fails to come back
Important: OCP upgrades cannot be "undone" in the traditional sense
OpenShift does not have a single "rollback" button. Once nodes start updating, the OS is already changed. The correct recovery is: if the upgrade stalls completely and the cluster is non-functional → restore from the etcd backup you took before the upgrade (Section 16). If only one operator is failing → investigate and fix that operator, do not try to downgrade the whole cluster.

Guide 6 · Node Scaling · Section 08
When and why to add worker nodes
The signals that tell you it is time to add a node — and how to make that decision with data

The three signals that mean you need more nodes

Memory pressure

Node memory consistently above 85% for 3+ days. Pods are being OOM-killed or evicted. New pods cannot schedule because no node has enough free memory. oc adm top nodes shows memory% in red zone.

CPU throttling

Application response times increasing. container_cpu_throttled_seconds_total metric rising in Prometheus. oc adm top nodes shows CPU% above 80% consistently. Pods have CPU limits set and are being throttled.

Pending pods

[svc-ocp@jumphost ~]$ cat ~/ocp-kit/bin/check-cert-expiry.sh
#!/bin/bash
# NovaCrestBank OCP Certificate Expiry Scanner
WARN_DAYS=60
CRIT_DAYS=30

echo "=== OCP Certificate Expiry Report ==="
echo "Date: $(date -u)"
echo "Warning threshold: ${WARN_DAYS} days"
echo "Critical threshold: ${CRIT_DAYS} days"
echo ""

check_cert() {
  local name="$1"
  local pem="$2"
  local expiry=$(echo "$pem" | openssl x509 -noout -enddate 2>/dev/null | cut -d= -f2)
  if [ -z "$expiry" ]; then return; fi
  local days=$(( ($(date -d "$expiry" +%s) - $(date +%s)) / 86400 ))
  if [ "$days" -lt "$CRIT_DAYS" ]; then
    echo "CRITICAL ($days days): $name — expires $expiry"
  elif [ "$days" -lt "$WARN_DAYS" ]; then
    echo "WARNING  ($days days): $name — expires $expiry"
  else
    echo "OK       ($days days): $name"
  fi
}

# Check API server cert
api_pem=$(echo | openssl s_client -connect api.ocp.novacrestbank.local:6443 2>/dev/null | \
  openssl x509 2>/dev/null)
check_cert "API server cert" "$api_pem"

# Check ingress wildcard cert
ingress_pem=$(echo | openssl s_client \
  -connect console-openshift-console.apps.ocp.novacrestbank.local:443 \
  -servername console-openshift-console.apps.ocp.novacrestbank.local 2>/dev/null | \
  openssl x509 2>/dev/null)
check_cert "Ingress wildcard cert" "$ingress_pem"

# Check all TLS secrets in key namespaces
for ns in openshift-config openshift-ingress banking-backend kyc-service; do
  while IFS=$'\t' read name b64; do
    pem=$(echo "$b64" | base64 -d 2>/dev/null)
    check_cert "$ns/$name" "$pem"
  done < <(oc get secret -n "$ns" --field-selector type=kubernetes.io/tls \
    -o go-template='{{range .items}}{{.metadata.name}}{{"\t"}}{{index .data "tls.crt"}}{{"\n"}}{{end}}' 2>/dev/null)
done

              KEYS=$(nsenter --target 1 --mount -- etcdutl snapshot status "$LATEST" \

              if [ "$KEYS" -lt 1000 ]; then
                echo "ERROR: backup has only $KEYS keys — possible corruption"
                exit 1

              echo "Backup verified: $KEYS keys in $LATEST"
              # Remove backups older than 7 days
              find "$BACKUP_DIR" -name "snapshot_*.db" -mtime +7 -delete
            securityContext:
              privileged: true
            volumeMounts:
            - name: host
              mountPath: /host
          volumes:
          - name: host

              path: /

Guide 6 · etcd Backup & Recovery · Section 16
etcd restore — full recovery procedure
Step-by-step etcd restoration when master nodes are lost — the most critical recovery procedure in OCP operations
When to use this procedure
Use this ONLY when the control plane is completely non-functional — all three masters lost, etcd quorum gone, the cluster is completely unresponsive. Do NOT use this for a single master failure (OCP recovers automatically from one master loss). The restore procedure replaces etcd on all masters with the backup snapshot. All cluster state reverts to the backup time — workloads created after the backup will be lost.
Scenario: all three masters lost — etcd must be restored from backup
