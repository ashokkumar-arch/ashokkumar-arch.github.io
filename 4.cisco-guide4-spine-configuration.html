The spine switch's single purpose — stated plainly
A spine switch in a leaf-spine fabric is a pure IP router. It does not know what VLANs exist. It does not hold MAC address tables for servers. It does not run spanning tree. It does not participate in any Layer 2 protocol. Every single interface on a spine switch is a routed Layer 3 interface with an IP address. The spine switch's only job is to look at an incoming IP packet, check its routing table, and forward it out the correct interface as fast as possible. That is it.

What the spine has — and what it does not have

What the spine DOES have

  • Routed (Layer 3) interfaces — every port has an IP address
  • BGP routing table — knows how to reach every leaf subnet
  • ECMP — multiple equal-cost paths to each destination
  • Management VRF — mgmt0 connected to OOB for management
  • Loopback interfaces — for BGP router-ID and future VTEP use
  • Route policies — controls what gets advertised to whom

What the spine does NOT have

  • VLANs — the spine has no VLAN database
  • MAC address table — the spine does not learn MAC addresses
  • Spanning Tree — no STP on routed interfaces (impossible)
  • Access ports — every port is a routed interface
  • SVIs — no VLAN interfaces (SVIs live only on leafs)
  • Server connections — servers connect to leafs, not spines

BGP concepts for spine-leaf underlay
Why BGP — not OSPF — is used in modern data center fabrics, and exactly what it does in NovaCrestBank's spine-leaf design
Your CCNA taught you OSPF — why are we using BGP?
OSPF was designed for traditional campus/WAN routing. BGP was designed for large-scale internet routing between autonomous systems. In modern data centers, BGP is preferred for spine-leaf because: (1) BGP scales better — each leaf only needs to know routes from its directly connected spines, not the entire topology, (2) BGP's AS-path mechanism naturally prevents routing loops without spanning tree, (3) BGP allows fine-grained control over what is advertised via policy, and (4) BGP is the same protocol used for VXLAN/EVPN overlays (which NovaCrestBank will use for NSX-T if needed later). OSPF would work for NovaCrestBank's size, but BGP is the industry standard for this type of deployment.

BGP fundamentals — the concepts you need for this guide

BGP conceptWhat it meansNovaCrestBank's value
Autonomous System (AS) A number that identifies a routing domain under single administrative control. Each "entity" in BGP has an AS number. Spines: AS 65000. Each leaf: its own AS (65001, 65002, 65003). This structure is called eBGP (external BGP — different AS numbers between peers).
eBGP vs iBGP eBGP = BGP between routers in different AS numbers. iBGP = BGP between routers in the same AS number. All spine-leaf BGP sessions are eBGP — spines are AS 65000, leafs are different AS numbers. This is standard for leaf-spine designs.
BGP neighbour A BGP peer — another router you have established a BGP session with. BGP must be explicitly configured — it does not auto-discover like OSPF. spine01 has 3 neighbours: leaf01 (10.0.11.2), leaf02 (10.0.12.2), leaf03 (10.0.13.2). spine02 has 3 neighbours: leaf01 (10.0.21.2), leaf02 (10.0.22.2), leaf03 (10.0.23.2).
BGP session states BGP goes through states: Idle → Active → OpenSent → OpenConfirm → Established. Only when Established are routes exchanged. After this guide, all 6 spine-leaf BGP sessions should show Established. Any other state = problem to diagnose.
Network statement Tells BGP which networks to advertise to neighbours. Only networks explicitly listed are advertised. Spines advertise their loopback prefixes. Leafs advertise their SVI subnets (VLAN 10, 20, 25, etc.) in Guide 5.
Prefix list / route policy Filter that controls which routes are accepted from or sent to a BGP neighbour. Prevents unwanted routes from entering or leaving the routing table. Spines use prefix lists to: accept only leaf SVI subnets from leafs, reject default routes from leafs, and control what is sent to Palo Alto.

Why each leaf gets its own AS number
If leaf01 and leaf02 shared the same AS number, and spine01 learned a route from leaf01, then advertised it to leaf02 — leaf02 would see its own AS in the BGP path and reject the route (BGP loop prevention). By giving each leaf its own AS, routes flow freely. This is the standard design for leaf-spine BGP underlay.

ECMP — equal-cost multipath load balancing
How two spine switches become one load-balanced fabric — and why this eliminates the spanning tree problem entirely
The problem ECMP solves
In traditional networking (with spanning tree), if you have two paths between two points, one is active and one is blocked. You pay for two switches and two sets of cables but only use half the capacity. ECMP (Equal-Cost Multi-Path) allows multiple paths to the same destination to be used simultaneously — all paths are active, all paths carry traffic, and if one fails the others absorb the load instantly.

How ECMP works in NovaCrestBank's fabric

When leaf01 wants to send a packet to leaf02, it has two equally-good paths in its BGP routing table:

  • Path A: leaf01 → spine01 → leaf02 (via 10.0.11.1 and 10.0.12.1)
  • Path B: leaf01 → spine02 → leaf02 (via 10.0.21.1 and 10.0.22.1)

Both paths are the same BGP cost (same number of AS hops, same MED value). With ECMP enabled (maximum-paths 2 in BGP), NX-OS installs both paths in the routing table and distributes traffic between them using a hash of the packet's source IP, destination IP, and protocol — called per-flow load balancing.

Per-flow hashing — what this means in practice

What it does

Each individual TCP or UDP flow (identified by its 5-tuple: source IP, destination IP, source port, destination port, protocol) is consistently assigned to one path. All packets of the same flow take the same path. This ensures TCP ordering is maintained — packets cannot arrive out of order.

What it means for monitoring

You will not see 50%/50% on each spine interface for every second. Different flows go different ways. Over time and across many flows (like in a production vSAN or OpenShift cluster) the aggregate load balances well. A single high-bandwidth flow always goes to one spine. Two parallel flows of the same size go to different spines.

ECMP failure behaviour — why redundancy is instant

If spine01 goes offline:

  1. All BGP sessions to spine01 drop (detected within seconds via BGP keepalive timeouts, or faster with BFD)
  2. All routes learned from spine01 are withdrawn from the leaf routing tables
  3. ECMP now has only one path (via spine02) — traffic automatically shifts 100% to spine02
  4. No spanning tree convergence, no STP topology change, no 30-second blocking wait. Traffic resumes in seconds.

Why /30 subnets — and why they all come from the 10.0.0.0/16 block
Each spine-to-leaf link needs exactly 2 usable IP addresses — one for each end of the link. A /30 subnet gives exactly 2 usable host addresses (network address + broadcast + 2 hosts = 4 addresses). Allocating from a dedicated block (10.0.0.0/16) that is never used for production VLANs means these underlay links are clearly identifiable in the routing table, never conflict with server subnets, and are easy to summarise in prefix lists.

Port channel to Palo Alto — spine01 inside interface
Connecting the firewall to the fabric — a port channel (LACP bond) from spine01 to the Palo Alto PA-5250 inside zone
Why Palo Alto connects to the spine — not the leaf
The Palo Alto PA-5250 is the trust zone boundary — it sits between the internet/DMZ and the internal network. All traffic entering or leaving NovaCrestBank's data center passes through it. The PA connects directly to the spine switches (not the leaf) because: (1) it is a Layer 3 device that routes between zones — it belongs at the Layer 3 level, and (2) connecting to the spine gives it the full routing table of the fabric with full redundancy — both spines, not just one leaf's local VLANs.

Port channel concept — why we bond two links

A port channel (also called LAG — Link Aggregation Group, or EtherChannel in IOS terminology) combines multiple physical links into a single logical link. Benefits: (1) bandwidth aggregation — two 100G links = 200G logical throughput, (2) redundancy — if one physical link fails, the port channel stays up and carries traffic on the remaining link, (3) single point of management — one interface to configure instead of multiple.

We use LACP (Link Aggregation Control Protocol) to dynamically negotiate the port channel between spine01 and the PA-5250. LACP is the industry standard and is preferred over static port channel (which requires manual coordination and does not detect misconfigurations automatically).

Spine peer link — spine01 ↔ spine02
The direct link between the two spines — used for BGP route exchange and as a backup path when a spine-to-leaf link fails
Why spines need a direct link to each other
spine01 and spine02 share AS 65000. They need a BGP session between themselves to synchronise routing information. Without this peer link, if leaf01 sends a route to spine01 only, spine02 would not know about it. The peer link also provides an alternate path: if the spine02→leaf01 link fails, spine02 can still reach leaf01's networks via the peer link to spine01, which still has a working link to leaf01.

spine01 BGP configuration — the complete routing engine
Every BGP statement for spine01 — neighbours, ECMP, timers, address families — with line-by-line explanation of what each command does and why
spine01 — complete BGP configuration 
spine01(config)# router bgp 65000
↑ Enter BGP configuration for AS 65000 (spine AS).
  If you type "router bgp 65001" here by mistake, you create a different AS —
  always double-check the AS number before pressing Enter.

spine01(config-router)# router-id 10.0.0.1
↑ Explicitly set the BGP router-ID to the loopback IP.
  If not set, NX-OS picks the highest IP on any interface — which could change.
  Always set it explicitly for consistency and predictability.

spine01(config-router)# log-neighbor-changes
↑ Log every BGP neighbour state change (Established, Idle, etc.) to syslog.
  This is critical for troubleshooting — you need to know WHEN a neighbour went down.

spine01(config-router)# address-family ipv4 unicast
spine01(config-router-af)# maximum-paths 6
↑ Allow up to 6 equal-cost paths to the same destination (ECMP).
  In NovaCrestBank: spine01 can have up to 6 paths (2 spines × 3 leafs).
  Setting this on the spine means the spine will install multiple paths in its routing table.
spine01(config-router-af)# maximum-paths ibgp 6
↑ Same for iBGP paths (routes learned from spine02 via the peer session)
spine01(config-router-af)# exit

# ── BGP timers — faster convergence ───────────────────────────────
spine01(config-router)# timers bgp 3 9
↑ keepalive=3 seconds, hold-time=9 seconds.
  Default is 60/180. With 3/9: a failed link is detected in 9 seconds (3 missed keepalives).
  This is a good balance — fast enough for production, not so aggressive it causes flapping.

# ── Neighbour: leaf01 (eBGP — AS 65001) ──────────────────────────
spine01(config-router)# neighbor 10.0.11.2 remote-as 65001
↑ Define leaf01 as a BGP neighbour.
  10.0.11.2 = leaf01's IP on the spine01→leaf01 /30 link.
  remote-as 65001 = leaf01 is in AS 65001.
  Because spine01 is AS 65000 and leaf01 is AS 65001: this is eBGP.

spine01(config-router)# neighbor 10.0.11.2 description leaf01 BGP peer

spine01(config-router)# neighbor 10.0.11.2 timers 3 9
↑ Apply same fast timers to this specific neighbour

spine01(config-router)# neighbor 10.0.11.2 password 0 NovaCrest$BGP2026!
↑ MD5 authentication on the BGP session.
  Both sides must have the same password or the session will not establish.
  "0" means the password is entered in clear text (NX-OS encrypts it in the config file).

spine01(config-router)# address-family ipv4 unicast
spine01(config-router-af)# neighbor 10.0.11.2 activate
↑ Enable this neighbour for IPv4 unicast address family (standard IPv4 routing).
  In NX-OS, neighbours must be explicitly activated per address family.
spine01(config-router-af)# neighbor 10.0.11.2 route-map LEAF-IN in
↑ Apply a route policy to routes RECEIVED from leaf01.
  LEAF-IN = accept only the leaf's SVI subnets. Reject everything else.
  (Route map configured in Section 09)
spine01(config-router-af)# neighbor 10.0.11.2 route-map SPINE-OUT out
↑ Apply a route policy to routes SENT to leaf01.
  SPINE-OUT = send leaf01 only the routes it needs: all other leaf subnets
  and the default route (if we are advertising one toward the fabric)
spine01(config-router-af)# neighbor 10.0.11.2 soft-reconfiguration inbound
↑ Store a copy of all received routes before policy filtering.

Prefix lists and route policies
The filters that control exactly which routes enter and leave each BGP session — critical for security and predictable routing
Why route policies matter in a banking data center
Without route policies, a misconfigured leaf or Palo Alto could accidentally advertise a route that disrupts traffic for the entire fabric. For example: a leaf advertising a default route (0.0.0.0/0) would pull all internet-bound traffic through that leaf instead of through the Palo Alto NGFW — bypassing the firewall entirely. Prefix lists prevent this. They are not optional — they are a security control.
