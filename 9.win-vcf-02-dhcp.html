WIN-VCF-02 · Section 02
Why VCF needs DHCP and exactly what it uses it for
Understanding what VCF uses DHCP for — and what it does not use it for — prevents over-provisioning the scopes and explains why exclusion ranges are the most important configuration in this guide
VCF's DHCP usage pattern — bootstrap only, then static
VCF and NSX use static IPs for all named components. Why does DHCP need to exist if everything ends up with a static IP?
VCF bring-up uses DHCP as a bootstrap mechanism. When SDDC Manager deploys a new ESXi host into a domain, the host initially boots and discovers its network configuration via DHCP. SDDC Manager then records the DHCP lease, assigns the correct static IP from its inventory, and reconfigures the VMkernel adapter to use the static IP. The DHCP lease is released. The host retains its static IP permanently after that point.

Specifically for NovaCrestBank:
— VLAN 10 (Management): ESXi management VMkernels use DHCP during initial PXE boot and bring-up. SDDC Manager then assigns the static IP (10.10.0.11–.25) and the DHCP lease is not renewed.
— VLAN 20 (vMotion) and VLAN 25 (vSAN): SDDC Manager configures these VMkernels using DHCP discovery during host commissioning, then converts them to the static IPs in its plan.
— VLAN 100 (Host TEP) and VLAN 101 (Edge TEP): NSX TEP VMkernels are configured from the static IP pools (NCB-Host-TEP-Pool 10.100.0.11–.30, NCB-Edge-TEP-Pool 10.101.0.11–.20) defined in NSX-01. However, DHCP must be available on these VLANs because NSX requires a DHCP relay for internal discovery during the initial Transport Node preparation phase.

The most critical part of DHCP configuration is the exclusion ranges — the static IPs assigned to named components must be excluded from the DHCP pool so the DHCP server never hands them out to a different device.
PCI-DSS v4.0 Req 1.2.1 — DHCP scope boundaries as network security controls
DHCP scopes with exclusions ensure that static IPs for named infrastructure components (SDDC Manager, vCenters, ESXi hosts, NSX Managers) are never assigned dynamically to unexpected devices. An unexpected device receiving one of these IPs would cause an IP conflict that disrupts critical banking infrastructure. The exclusion ranges are the security control. PCI Req 12.3.2 — component identity integrity is maintained by ensuring IP addresses are uniquely assigned.

WIN-VCF-02 · Section 13
Create scope NCB-VLAN100-HostTEP — 10.100.0.0/24
The NSX Host TEP network. The exclusion range for the NSX static TEP pool (10.100.0.11–.30) is the most important exclusion in this scope — it must precisely match what was configured in NSX-01.
Why the NSX TEP DHCP scope needs to exist even though NSX uses static IP pools
NSX-01 configured the NCB-Host-TEP-Pool as a static IP pool (10.100.0.11–.30). NSX Manager assigns TEP IPs from this pool directly — not via DHCP. However, during Transport Node preparation, NSX sends a bootstrap configuration to the ESXi host that requires DHCP to be available on the TEP VLAN for internal VMkernel initialisation. The ESXi host's TEP VMkernel requests a DHCP lease during preparation, NSX immediately replaces that lease with the static pool IP, and releases the lease. Without DHCP present on VLAN 100, Transport Node preparation fails with a "TEP VMkernel initialization error."

DHCP relay must be configured on the Cisco fabric — Windows Server DHCP alone is not enough
The Windows DHCP server sits on VLAN 70 (the infrastructure segment, 10.70.0.0/24). DHCP requests from ESXi hosts on VLAN 10, 20, 25, 100, and 101 do not automatically reach DC01 — DHCP broadcast packets do not cross VLAN boundaries. The Cisco leaf switch SVIs must have the ip helper-address command pointing to DC01 (10.70.0.10) on each SVI:

interface Vlan10
 ip helper-address 10.70.0.10

Repeat for Vlan20, Vlan25, Vlan100, and Vlan101. Without this, ESXi hosts on those VLANs will never receive DHCP responses from DC01, even though the scope is active. This is a Cisco configuration item — outside Windows Server, but required for DHCP to function.
WIN-VCF-02 · Section 19
Configure DHCP failover to DC02
Replicating DHCP scope configuration to DC02 so leases continue if DC01 is unavailable — Windows Server DHCP Failover provides hot-standby redundancy
Why DHCP failover matters for VCF
VCF uses DHCP only for bootstrap — why configure failover for a temporary-use service?
Because if DC01 is unavailable during VCF bring-up or NSX Transport Node preparation, the ESXi host's bootstrap DHCP request gets no response, and the deployment phase fails or stalls — waiting for a DHCP lease that never arrives. With DHCP failover to DC02, if DC01 is unreachable, DC02 takes over lease responses automatically. The failover configuration also synchronises all scope data (exclusions, options, active leases) to DC02, giving a complete standby ready to operate.

PCI-DSS v4.0 Req 12.3.2 — critical infrastructure services must have documented redundancy. NIST SP 800-53 CP-7 — Alternate Processing: network services needed for recovery must themselves be redundant.

What WIN-VCF-03 builds next — PKI and certificate replacement

WIN-VCF-03 replaces the self-signed certificates on VCF and NSX components with certificates signed by NovaCrestBank's IssuingCA. This is required before VCF bring-up completes because:

  • SDDC Manager — replace the default self-signed TLS certificate with an IssuingCA-signed cert. SDDC Manager's FQDN (sddc-manager.novacrestbank.local) must match the cert Subject Alternative Name.
  • vCenter (both domains) — replace the Machine SSL certificate. vCenter certificates affect vSphere Client browser trust and are required for NSX Manager to trust the vCenter API endpoint.
  • NSX Manager cluster — replace the cluster certificate on all three nodes and the cluster VIP. NSX-signed certificates are self-signed by default; browsers and API clients reject them. The IssuingCA-signed certificate is also required for OCP to trust the NSX API during installation.

WIN-VCF-03 covers the IssuingCA web enrollment process (requesting certificates via http://issuingca.novacrestbank.local/certsrv), certificate export from Windows, and the upload-and-replace procedures in the SDDC Manager, vCenter, and NSX Manager GUIs. All GUI only, screen by screen.
