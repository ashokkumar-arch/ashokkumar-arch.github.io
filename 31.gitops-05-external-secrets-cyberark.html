GITOPS-05 · Section 01
The GitOps secret problem — why you cannot put secrets in Git
GitOps says everything is in Git. But GitOps also means that Git is visible to everyone with repository access — developers, auditors, CI systems, and anyone with a compromised developer credential. Database passwords, API keys, and signing certificates cannot live in Git. This is the central tension that External Secrets Operator resolves.

The three approaches to secrets in GitOps — and why only one is correct for banking
A banking application needs a database password to start. Kubernetes needs that password as a Secret. GitOps says everything is in Git. What are the options, and which is appropriate?
Three approaches exist. Two are wrong for NovaCrestBank. One is correct.

Approach 1 — Store the raw secret in Git: categorically wrong. A database password committed to a Git repository — even a private one — is a security incident waiting to happen. Git history is permanent: even if the secret is later deleted, it exists in every clone, every CI cache, and every git log that ran before the deletion. Any developer with read access to the banking/payment-service repo would have the production database password. Any GitLab OAuth token compromise exposes production credentials. PCI-DSS DSS v4 Req 3.4 prohibits storing authentication credentials in cleartext in any location accessible to non-authorised personnel. A developer-accessible Git repo does not satisfy "authorised personnel only" for production database passwords.

Approach 2 — Store encrypted secrets in Git (e.g., Sealed Secrets or SOPS): partially correct but with limitations. Tools like Bitnami Sealed Secrets encrypt the secret with the cluster's public key — only the cluster can decrypt it. SOPS encrypts with a KMS key. Both store ciphertext in Git. This is better than plaintext, but introduces key management complexity (what happens if the cluster's keypair is lost? the secrets cannot be decrypted), and does not support rotation (changing the secret requires re-encrypting the ciphertext and committing again). For NovaCrestBank's requirement of automatic rotation integrated with CyberArk, encrypted-in-Git approaches are inadequate.

Approach 3 — Reference secrets from an external vault (External Secrets Operator + CyberArk): correct. Git stores only a reference to the secret — its path in CyberArk, the Kubernetes Secret it should create, and the key mappings. The reference contains no sensitive data. ESO reads the actual secret value from CyberArk at runtime and creates the Kubernetes Secret in the cluster. When CyberArk rotates the password, ESO automatically fetches the new value and updates the Kubernetes Secret. No secret value is ever in Git. This is the correct model for banking.

What you cannot store in Git

  • Database passwords (payment-service-db-password)
  • API keys (fraud detection service, notification gateway)
  • Private keys (TLS private keys, cosign signing key)
  • Harbor pull secrets (ncb-pipeline-svc credentials)
  • LDAP bind passwords (svc-ocp-ldap)
  • Webhook shared secrets (Jenkins HMAC token)
  • Any value that would cause a security incident if read by a developer

What does live in Git (the ExternalSecret object)

  • The CyberArk path where the secret is stored
  • The name of the Kubernetes Secret to create
  • The key mappings (CyberArk key → Kubernetes Secret key)
  • The refresh interval for automatic re-sync
  • The SecretStore reference (which ESO provider to use)
  • Labels and annotations on the resulting Kubernetes Secret
  • No secret values — only references and metadata
GITOPS-05 · Section 02
The ESO model — how External Secrets bridges Git and CyberArk
External Secrets Operator runs as an OCP workload. It watches ExternalSecret objects (which live in Git and are deployed by Argo CD), fetches the actual secret values from CyberArk Conjur, and creates standard Kubernetes Secret objects in the target namespace. The application pods consume the Kubernetes Secret exactly as if it had been created manually — ESO is invisible to the application.
CyberArk Conjur ncb-bank/banking/payment-service/db-password
"actual-db-password-value"
←
External Secrets Operator Fetches secret every
refreshInterval (1h)
→
Kubernetes Secret payment-service-db-creds
data: {password: base64...}
→
Pod payment-service
env: DB_PASSWORD from Secret
CRD 1
SecretStore / ClusterSecretStore
Defines how ESO connects to an external secret provider — in NovaCrestBank's case, CyberArk Conjur. Contains: the Conjur API endpoint URL, the authentication method (JWT-based workload identity), and the credentials ESO uses to authenticate to Conjur. A ClusterSecretStore is cluster-scoped — it can be referenced by ExternalSecret objects in any namespace. NovaCrestBank uses one ClusterSecretStore named cyberark-conjur for the entire cluster.
CRD 2
ExternalSecret
A namespace-scoped object that declares: which ClusterSecretStore to use, which secret to fetch from CyberArk (the path), which Kubernetes Secret to create, how to map CyberArk keys to Kubernetes Secret keys, and how often to refresh. This is the object that lives in Git (in the Kustomize overlays). Argo CD deploys ExternalSecret objects. ESO reconciles them into Kubernetes Secrets.
Output
Kubernetes Secret (managed by ESO)
A standard Kubernetes Secret created and owned by ESO. Its data content comes from CyberArk. Its lifecycle is managed by the ExternalSecret — when the ExternalSecret is deleted, the Kubernetes Secret is deleted. When ESO fetches a new value from CyberArk (after rotation), it updates the Kubernetes Secret's data field. ESO sets an annotation on the Secret recording the last successful sync time and the provider source.
GITOPS-05 · Section 03
CyberArk Conjur vs HashiCorp Vault — the banking choice
External Secrets Operator supports many secret providers: AWS Secrets Manager, Azure Key Vault, HashiCorp Vault, GCP Secret Manager, CyberArk Conjur, and others. NovaCrestBank's choice of CyberArk Conjur is not a technical preference — it reflects CyberArk's dominant position in the financial services sector and NovaCrestBank's existing CyberArk PAM (Privileged Access Management) deployment.
Why CyberArk Conjur and not HashiCorp Vault for NovaCrestBank
HashiCorp Vault is the most widely documented secret management solution in Kubernetes environments. Why does NovaCrestBank use CyberArk Conjur instead?
NovaCrestBank already runs CyberArk Privileged Access Manager (PAM) for privileged account management — SSH keys, database administrator credentials, Windows service account passwords, and break-glass access. CyberArk Conjur is the DevOps-oriented extension of the same CyberArk platform, designed specifically for machine-to-machine secret management in CI/CD and Kubernetes environments. The decision to use Conjur rather than Vault is a consolidation decision: one vendor, one audit trail, one compliance posture for all secret management across the enterprise.

The specific advantages of Conjur in NovaCrestBank's context:
1. Unified secret storage with CyberArk PAM. The database password stored in CyberArk PAM (rotated by CyberArk's CPM — Central Policy Manager) is the same secret that Conjur surfaces to Kubernetes via ESO. There is no separate "Vault secrets" vs "PAM secrets" synchronisation problem. When the DBA changes a database password in CyberArk PAM, that change flows through Conjur to ESO to the Kubernetes Secret automatically — the entire chain is within one platform.
2. APRA and FFIEC examination familiarity. Australian financial regulators and FFIEC examiners are familiar with CyberArk as the market-leading PAM solution for financial services. Demonstrating CyberArk compliance evidence is straightforward for NovaCrestBank's compliance team. HashiCorp Vault requires more explanation to examiners unfamiliar with it.
3. CyberArk's Kubernetes authentication. CyberArk Conjur's Kubernetes authenticator uses the workload's OCP ServiceAccount JWT token to authenticate — no long-lived credentials needed in the cluster. ESO authenticates to Conjur by presenting the ESO ServiceAccount token. Conjur validates the token against the OCP API server and grants access. This is the same zero-credential-in-cluster model used by cloud providers' workload identity solutions.

GITOPS-05 · Section 05
CyberArk Conjur prerequisites — policy and host identity
Before configuring the ESO ClusterSecretStore, three objects must exist in CyberArk Conjur: a policy branch for NovaCrestBank's banking secrets, a host identity representing the ESO workload, and grants giving that host identity read access to the secrets it needs to surface.
1
Conjur policy branch — ncb-bank/banking
CyberArk Conjur organises secrets in a policy tree. NovaCrestBank's banking secrets live under the branch ncb-bank/banking/. Each microservice has a sub-branch: ncb-bank/banking/payment-service/, ncb-bank/banking/account-service/, etc. Each secret is a variable within its branch: ncb-bank/banking/payment-service/db-password. The policy branch is defined in a YAML-like Conjur policy language and applied by the CyberArk administrator.
2
Host identity — the ESO workload's Conjur identity
Every workload that authenticates to Conjur needs a host identity in the Conjur policy. ESO authenticates using the OCP Kubernetes authenticator: ESO presents its ServiceAccount JWT token, Conjur validates it against the OCP API server, and grants access as the host host/ocp-cluster/ncb-ocp/external-secrets. No password or API key is stored in the cluster — the ServiceAccount token IS the credential. The host identity is created by the CyberArk admin with the annotation authn-k8s/namespace: external-secrets and authn-k8s/service-account: external-secrets.
3
Grants — read access to specific secret variables
The ESO host identity is granted read execute on the secret variables it needs to surface. Minimum-privilege: ESO can read ncb-bank/banking/*/db-password and ncb-bank/banking/*/db-username for all microservices, but cannot read ncb-bank/platform/ (infrastructure secrets) or ncb-bank/cyberark-admin/ (administrative secrets). If ESO's credentials are somehow compromised, the blast radius is limited to the banking application secrets — not the entire CyberArk vault.

Conjur account namencb-bank

GITOPS-05 · Section 06
ClusterSecretStore — connecting ESO to CyberArk Conjur
The ClusterSecretStore is the bridge between ESO and CyberArk Conjur. It is cluster-scoped — one object serves all namespaces. Every ExternalSecret in every banking namespace references this ClusterSecretStore. Creating it requires first storing the Conjur CA certificate and the ESO host identity token in a Kubernetes Secret in the external-secrets namespace.
ClusterSecretStore — cyberark-conjur — every field explained
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: cyberark-conjur
    # Name referenced by all ExternalSecret objects.
    # ExternalSecrets in any namespace use:
    #   secretStoreRef:
    #     name: cyberark-conjur
    #     kind: ClusterSecretStore
spec:
  provider:
    conjur:
      url: https://conjur.corp.novacrestbank.com
            # Internal Conjur URL — accessible from infra nodes where ESO runs.
              # Conjur's TLS certificate is signed by NovaCrestBank-IssuingCA.
              # ESO verifies this certificate using the CA bundle in caBundle below.

      caBundle: LS0tLS1CRUdJTi... (base64-encoded NovaCrestBank CA chain)
            # Base64-encoded NovaCrestBank CA certificate chain (RootCA + IssuingCA).
              # ESO uses this to verify Conjur's TLS certificate.
              # Without caBundle: ESO would reject Conjur's self-signed or
              # internally-signed certificate with x509 validation errors.

      auth:
        apikey:
          account: ncb-bank
                    # Conjur account name. Every Conjur resource path starts with
                      # this account: ncb-bank:variable:ncb-bank/banking/payment-service/db-password
          userRef:
            name: conjur-eso-host-credentials
            namespace: external-secrets
            key: hostId
                        # The Conjur host identity: "host/ocp-cluster/ncb-ocp/external-secrets"
                          # Stored in a Kubernetes Secret (conjur-eso-host-credentials)
                          # in the external-secrets namespace.
                          # Only the external-secrets namespace has this Secret —
                          # no other namespace can see the ESO credentials.
          apiKeyRef:
            name: conjur-eso-host-credentials
            namespace: external-secrets
            key: apiKey
                        # The Conjur API key for the ESO host identity.
                          # Generated by Conjur when the host identity is created.
                          # Used to authenticate ESO to Conjur's API.
                          # This API key never leaves the external-secrets namespace.

GITOPS-05 · Section 08
ExternalSecret anatomy — every field explained
An ExternalSecret object declares the complete specification for a Kubernetes Secret — where to get the values, how often to refresh them, and what the resulting Kubernetes Secret should look like. This object is what lives in Git and is deployed by Argo CD.
ExternalSecret — annotated skeleton showing every field and its purpose
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-service-db-credentials
  namespace: banking-production
    # Namespace: the ExternalSecret lives in the same namespace as the
    # application pods that need the Secret. A banking-production ExternalSecret
    # creates a Kubernetes Secret in banking-production. ESO never creates
    # Secrets in different namespaces from the ExternalSecret — namespace isolation.
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
        # Sync wave -1: this ExternalSecret must be reconciled (Secret created)
          # before the database migration Job (wave 0) or application Deployment
          # (wave 1) start. Without the Kubernetes Secret, the migration Job
          # cannot connect to the database. Section 12 of GITOPS-04 established
          # this dependency — this annotation implements it.

spec:
  refreshInterval: 1h
    # refreshInterval: how often ESO fetches the current value from Conjur.
    # Every hour: ESO calls the Conjur API, gets the current db-password value,
    # and compares it to the current Kubernetes Secret data.
    # If different (rotation occurred): ESO updates the Kubernetes Secret.
    # 1h: a reasonable balance — not so frequent that Conjur is overloaded
    # with API calls, not so infrequent that a rotated secret takes too long
    # to propagate. For highly sensitive secrets: use 15m.

  secretStoreRef:
    name: cyberark-conjur
    kind: ClusterSecretStore
        # Which SecretStore ESO uses to fetch secrets.
          # ClusterSecretStore (cluster-scoped): referenced by any ExternalSecret
          # in any namespace without requiring a namespace-local SecretStore.
          # Alternative: SecretStore (namespace-scoped) — used when different
          # namespaces need different Conjur credentials. Not used here —
          # one ClusterSecretStore serves all banking namespaces.

  target:
    name: payment-service-db-creds
        # The name of the Kubernetes Secret ESO creates in banking-production.
          # Application pods reference this exact name in their env/volumes.
    creationPolicy: Owner
        # Owner: ESO owns the Kubernetes Secret it creates.
          # - If the ExternalSecret is deleted, the Kubernetes Secret is deleted.
          # - ESO sets ownerReference on the Secret pointing to the ExternalSecret.
          # - Argo CD sees the Secret as owned by ESO (not by Argo CD) and does
          #   not attempt to manage or delete it separately.
          # Alternative: Merge — ESO updates an existing Secret rather than owning it.
          # Alternative: None — ESO creates the Secret but does not own it.
    template:
      type: Opaque
      metadata:
        annotations:
          reloader.stakater.com/match: "true"
                    # Stakater Reloader annotation: triggers rolling restarts of Deployments
                      # that reference this Secret when the Secret is updated.
                      # Section 16 explains how rolling restarts are triggered on rotation.

  data:
      # List of individual secret keys to fetch from Conjur
    - secretKey: db-password
            # secretKey: the key name in the resulting Kubernetes Secret's data map.
              # The pod references: secretKeyRef.key: db-password
      remoteRef:
        key: ncb-bank/banking/payment-service/db-password
                # The full Conjur variable path for this secret value.
    - secretKey: db-username
      remoteRef:
        key: ncb-bank/banking/payment-service/db-username
    - secretKey: db-url
      remoteRef:
        key: ncb-bank/banking/payment-service/db-url
                # db-url: the full JDBC connection string.
                  # Storing the URL as a secret (rather than a ConfigMap) because it
                  # contains the hostname of the internal database server —
                  # internal network topology information that should not be in Git.

Payment-service database credentials ExternalSecret
The complete ExternalSecret for the payment-service's database credentials. This file lives in apps/payment-service/overlays/production/externalsecret-db.yaml in the GitOps config repo. Argo CD deploys it in wave -1, ESO fetches the values from Conjur, and the Kubernetes Secret is created before the migration Job starts.
apps/payment-service/overlays/production/externalsecret-db.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: payment-service-db-credentials
  namespace: banking-production
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: cyberark-conjur
    kind: ClusterSecretStore
  target:
    name: payment-service-db-creds
    creationPolicy: Owner
    template:
      type: Opaque
      metadata:
        labels:
          app.kubernetes.io/name: payment-service
          managed-by: external-secrets
        annotations:
          reloader.stakater.com/match: "true"
          external-secrets.io/conjur-path: ncb-bank/banking/payment-service
                    # Custom annotation documenting the Conjur path for auditability.
                      # CyberArk administrators can correlate Kubernetes Secrets to
                      # Conjur paths without reading the ExternalSecret object.
  data:
    - secretKey: db-password
      remoteRef:
        key: ncb-bank/banking/payment-service/db-password
    - secretKey: db-username
      remoteRef:
        key: ncb-bank/banking/payment-service/db-username
    - secretKey: db-url
      remoteRef:
        key: ncb-bank/banking/payment-service/db-url
    - secretKey: npp-api-key
      remoteRef:
        key: ncb-bank/banking/payment-service/npp-api-key
                # NPP (New Payments Platform) API key for the payment rail integration.
                  # Rotated annually by the NPP integration team via CyberArk PAM.
                  # ESO propagates the rotation to the Kubernetes Secret automatically.
    - secretKey: encryption-key
      remoteRef:
        key: ncb-bank/banking/payment-service/at-rest-encryption-key
                # AES-256 key for encrypting sensitive payment fields at rest in the DB.
                  # Rotated annually. Rotation of this key requires re-encrypting
                  # existing database records — a separate migration process.
                  # refreshInterval for this ExternalSecret is 24h to avoid
                  # accidentally triggering an encryption key rotation mid-day.
GITOPS-05 · Section 10
Harbor pull secret ExternalSecret — image pull credentials for every namespace
Every namespace that runs banking application pods must have a Harbor pull secret — the credentials Kubernetes uses to authenticate to Harbor when pulling container images. Rather than manually creating this Secret in each namespace, ESO creates it automatically from a single Conjur variable. A ClusterExternalSecret propagates it to all banking namespaces simultaneously.
ClusterExternalSecret — harbor-pull-secret propagated to all banking namespaces
apiVersion: external-secrets.io/v1beta1
kind: ClusterExternalSecret
  # ClusterExternalSecret: a cluster-scoped object that creates an ExternalSecret
  # in every namespace matching the namespaceSelector.
  # Result: one ClusterExternalSecret creates harbor-pull-secret in
  # banking-dev, banking-staging, AND banking-production simultaneously.
  # When a new banking namespace is added, the ClusterExternalSecret
  # automatically creates the pull secret there too — no manual step.
metadata:
  name: harbor-pull-secret
spec:
  namespaceSelector:
    matchLabels:
      app.kubernetes.io/part-of: banking-platform
    # namespaceSelector: target all namespaces labelled banking-platform.
    # banking-dev, banking-staging, banking-production all have this label.
    # Adding a new banking namespace with this label automatically gets the pull secret.

  externalSecretSpec:
    refreshInterval: 24h
        # Harbor pull credentials rotate infrequently — 24h refresh is sufficient.
    secretStoreRef:
      name: cyberark-conjur
      kind: ClusterSecretStore
    target:
      name: harbor-pull-secret
      creationPolicy: Owner
      template:
        type: kubernetes.io/dockerconfigjson
                # type: dockerconfigjson — the Kubernetes pull secret type.
                  # Kubernetes uses this type to authenticate to container registries.
                  # Deployments reference this Secret in imagePullSecrets.
        data:
          .dockerconfigjson: |
                        {
                          "auths": {
                            "harbor.apps.ocp.corp.novacrestbank.com": {
                              "username": "{{ .username }}",
                              "password": "{{ .password }}",

GITOPS-05 · Section 12
How rotation works — the complete automatic cycle
Automatic secret rotation is the operational property that makes ESO + CyberArk the correct solution for PCI-DSS Req 8.6. When CyberArk's CPM (Central Policy Manager) rotates a password on schedule, the new value propagates to the running pods automatically — without any human intervention, without any pipeline run, without any downtime.
The complete rotation chain — from CyberArk CPM scheduling to pod using new credentials
CyberArk's Central Policy Manager rotates the payment-service database password at 02:00 on the first Sunday of every month. Exactly what happens from that point until the running payment-service pods are using the new password?
Six events, all automatic, no human action required, typical total duration: 90 minutes from CyberArk rotation to all pods using the new password.

T+0 — CyberArk CPM rotates the password. The CPM executes its configured password rotation policy for the ncb-bank/banking/payment-service/db-password variable. It generates a new 32-character random password, updates the database user's password via the database platform plugin, and stores the new password value in Conjur. The old password is retired. The Conjur variable now holds the new value.

T+0 to T+60 — ESO refreshInterval window. The payment-service-db-credentials ExternalSecret has refreshInterval: 1h. ESO fetches the Conjur variable value every hour. At the next scheduled refresh cycle after the rotation (at most 60 minutes later), ESO calls Conjur's API, gets the new password value, and compares it to the current Kubernetes Secret's db-password key.

T+60 — ESO detects the change and updates the Kubernetes Secret. ESO writes the new password value to the payment-service-db-creds Kubernetes Secret's db-password key. The Kubernetes Secret's resourceVersion increments — this is the event that triggers the next step.

T+60 — Stakater Reloader detects the Secret change. Stakater Reloader (Section 16) is watching all Secrets annotated with reloader.stakater.com/match: "true". When the payment-service-db-creds Secret's resourceVersion changes, Reloader triggers a rolling restart of all Deployments that reference this Secret. The payment-service Deployment begins a rolling restart.

T+60 to T+90 — Rolling restart of payment-service pods. OCP performs a rolling update: new pods start with the new database password from the updated Secret, old pods drain their in-flight requests and terminate. The PodDisruptionBudget ensures at least 2 pods are Running at all times during the restart. Rolling restart of 3 pods takes approximately 5–10 minutes.

T+90 — All pods using new password. All 3 payment-service pods are Running with the new database password in their DB_PASSWORD environment variable. The rotation is complete. No downtime. No manual action. No pipeline run.

Why environment variables and not volume mounts for database credentials
Both options work with Kubernetes Secrets. NovaCrestBank uses environment variables for database credentials because: (1) Spring Boot's @Value("${DB_PASSWORD}") and application.properties datasource configuration natively reads environment variables — no code change required; (2) environment variables are injected at pod startup and immutable for the pod's lifetime — if the Secret is updated mid-lifecycle, the running pod continues using the old value until it restarts (which Reloader triggers). Volume-mounted Secrets update live without pod restart — desirable for some use cases, but for database connection pool credentials, a live update would require the connection pool to detect and reconnect with new credentials, which requires application-level support. The rolling restart model (Reloader triggers restart, new pods start with new credentials) is simpler and more reliable.
GITOPS-05 · Section 16
Triggering rolling restarts on secret rotation — Stakater Reloader
When ESO updates a Kubernetes Secret (because CyberArk rotated the value), running pods still have the old value in their environment variables. The pods must restart to pick up the new value. Stakater Reloader is a lightweight controller that watches for Secret updates and triggers rolling restarts of Deployments that reference the updated Secret.
Why Stakater Reloader and not a custom annotation or a CronJob to force restarts
When a Kubernetes Secret is updated, running pods continue using the old environment variable values because environment variables are set at pod start and are immutable for the pod's lifetime. What is the correct mechanism to restart pods when a Secret rotates?
Three approaches exist. Stakater Reloader is the correct one for NovaCrestBank's architecture.

Approach 1 — Manual restart: oc rollout restart deployment/payment-service. Requires human action. Not automatic. Does not satisfy PCI-DSS Req 8.6's requirement that credential rotation is an automated process. Rejected.

Approach 2 — CronJob that forces periodic restarts: A CronJob runs kubectl rollout restart on all banking Deployments every hour. Pods restart regardless of whether the Secret actually changed — causing unnecessary disruption. If a pod restarts 5 minutes before the next CronJob run, it is unnecessarily restarted again. Rejected.

Approach 3 — Stakater Reloader (correct): Reloader watches Kubernetes Secrets for resourceVersion changes. When the Secret changes — and only when it changes — Reloader adds a last-restarted annotation to the Deployment, which OCP's rollout controller interprets as a trigger for a rolling restart. Pods restart exactly once per rotation, immediately when the Secret is updated, not on a polling schedule. Reloader is a 5 MB binary with minimal resource requirements. It is deployed on infra nodes as a Deployment managed by Argo CD.
Reloader Deployment annotation — opt-in per Deployment
# In the Deployment metadata (base/deployment.yaml):
metadata:
  annotations:
    reloader.stakater.com/auto: "true"
      # auto: "true" — Reloader watches ALL Secrets and ConfigMaps referenced
        # by this Deployment. If any referenced Secret updates, this Deployment
        # gets a rolling restart.
        # Alternative: reloader.stakater.com/search: "true" — only watches
        # Secrets/ConfigMaps annotated with reloader.stakater.com/match: "true"
        # (the annotation ESO puts on the Secrets it manages).
        # NovaCrestBank uses "auto" because all Secrets in the Deployment spec
        # are ESO-managed and should trigger restarts on update.

GITOPS-05 · Section 18
Audit trail — secret access logging in CyberArk and OCP
Every secret access — ESO fetching from Conjur, ESO updating a Kubernetes Secret, a pod reading an environment variable injected from a Secret — generates audit events in CyberArk and in the OCP API server audit log. Together these provide the complete secret access audit trail required by PCI-DSS Req 10.2.7 (all secrets and credentials are included in audit logs).
Layer 1
CyberArk Conjur audit log
Every Conjur API call from ESO generates a Conjur audit event: the requestor identity (host/ocp-cluster/ncb-ocp/external-secrets), the variable accessed (ncb-bank/banking/payment-service/db-password), the action (retrieve), the timestamp, and the result (success or deny). Conjur audit events are exported to Splunk via CyberArk's syslog output. Retention: 12 months in Splunk (PCI-DSS Req 10.5.1). This log answers: "who accessed the database password variable in Conjur and when."
Layer 2
OCP API server audit log (Secret operations)
When ESO updates a Kubernetes Secret (on rotation), the OCP API server records: the ESO ServiceAccount identity, the action (update on Secret payment-service-db-creds in namespace banking-production), and the timestamp. When a pod is created and its environment variables are injected from the Secret, the pod creation event is recorded. Forwarded to Splunk via ClusterLogForwarder (OBS-03). This log answers: "when was the Kubernetes Secret for the payment-service database credentials last updated."
Layer 3
ExternalSecret status — ESO's own sync record
The ExternalSecret's status.refreshTime field records the last time ESO successfully fetched from Conjur and updated the Kubernetes Secret. Argo CD also records the ExternalSecret object's state in its sync history. This is not a tamper-evident external log, but it provides a quick in-cluster audit reference: oc get externalsecret -A -o custom-columns=NAME:.metadata.name,SYNCED:.status.refreshTime,READY:.status.conditions[0].status shows all ExternalSecrets and their last sync time.
