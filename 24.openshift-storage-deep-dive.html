Guide 4 · Foundation · Section 01
How OpenShift storage actually works
The full mental model — from a developer's PVC request to a physical disk on the NetApp or vSAN — before touching a single command
The key insight that makes everything clear
OpenShift storage has three layers of abstraction. A developer never sees the NetApp or the vSAN. They see only a PersistentVolumeClaim — a request that says "I need 50 GB, readable by multiple pods at the same time." The infrastructure team configures what happens underneath. This separation is intentional and powerful.

The three-layer model

L1
Developer layer
Developer creates a PersistentVolumeClaim (PVC). Specifies: how much storage, what access mode, which StorageClass. Knows nothing about NetApp, vSAN, NFS, or Fibre Channel.
PVC
L2
Kubernetes / OpenShift layer
OpenShift reads the PVC, looks at the requested StorageClass, calls the matching CSI driver (Trident for NetApp, vSphere CSI for vSAN) to provision a volume. Binds the resulting PersistentVolume (PV) to the PVC.
PV + StorageClass
L3
Infrastructure layer
The CSI driver calls the actual storage system API. For NetApp Trident: creates an NFS volume on the AFF A400, returns the NFS mount path. For vSphere CSI: creates a VMDK on the vSAN datastore, attaches it to the correct ESXi host.
NetApp / vSAN

Guide 4 · Foundation · Section 02
Two storage systems — what each one does
vSphere CSI (vSAN) and NetApp Trident (NFS) — how they differ, when to use each, what they can and cannot do
NovaCrestBank has both — they serve different purposes
Your environment has a vSAN datastore (built from local NVMe/SSD across all 5 ESXi hosts) and a NetApp AFF A400 (NFS over 25GbE to the leaf switch). They are not interchangeable — each has a specific set of jobs. Understanding which to use for which workload is a senior engineer decision.

vSphere CSI — vSAN

ProtocolVMDK (virtual disk) — block storage
Access modeReadWriteOnce ONLY — one pod on one node
PerformanceVery high — NVMe/SSD local to ESXi hosts
Best forDatabases, single-instance stateful apps, monitoring (Prometheus data), etcd
Cannot doReadWriteMany — multiple pods on different nodes cannot mount the same VMDK simultaneously
Already configured?Yes — IPI auto-installed vSphere CSI and created thin-csi StorageClass

NetApp Trident — NFS

ProtocolNFS v3/v4.1 — file storage over network
Access modeReadWriteMany — many pods on many nodes, all at once
PerformanceHigh — 25GbE to leaf, all-NVMe AFF A400
Best forShared content (KYC docs, PDF statements), internal image registry, logging, any app needing shared file access
Cannot doBlock semantics — not suitable for databases needing direct disk access (use vSAN for those)
Already configured?No — needs Trident installation (this guide)

Quick decision guide — which storage for which workload?

WorkloadAccess neededUseWhy
PostgreSQL / OracleRWOvSphere CSIDatabases need block semantics and exclusive access. High IOPS from NVMe vSAN.
Internal image registryRWXTrident NFSMultiple registry pods on different nodes must read/write the same image layers.
Prometheus monitoring dataRWOvSphere CSISingle Prometheus pod writes time-series data. Block storage is more efficient.
KYC document storageRWXTrident NFSMultiple application pods across nodes need to read the same customer document files.
PDF statement generationRWXTrident NFSGenerator pods write; reader pods read — shared access needed.
Application logs (Loki)RWO or RWXTrident NFSLog aggregators often use shared storage for their index.
Redis / session cacheRWOvSphere CSISingle-instance in-memory store with persistence. Block is better.
CI/CD build workspaceRWXTrident NFSMultiple build pods often share a workspace or artifact cache.
Guide 4 · Foundation · Section 03
PVC lifecycle — end to end
Every state a PVC passes through, what each state means, and what to do if it gets stuck

The five PVC states

1
Pending
PVC has been created but no storage has been provisioned yet. For WaitForFirstConsumer StorageClasses (like thin-csi), this is the normal state until a pod is created that uses the PVC. For Immediate StorageClasses (like Trident NFS), Pending means the CSI driver is actively provisioning — should resolve in seconds.
Normal or stuck
2
Bound
A PersistentVolume has been created and linked to this PVC. The storage is ready to use. A pod can now mount it. This is the healthy steady state.
Healthy
3
Lost
The PV that was bound to this PVC has been deleted or is unreachable. The data may still exist on the storage array, but the Kubernetes binding is broken. Requires manual intervention to recover.
Needs attention
4
Released
The PVC has been deleted, but the PV still exists (because reclaim policy is Retain). The volume's data is safe. The PV will not be rebound to a new PVC until an admin manually releases it. Used in data-sensitive banking environments.
Awaiting admin
5
Terminating
The PVC is being deleted. It will stay in this state until all pods that mounted it have stopped. If a pod is stuck or if the CSI driver cannot reach the storage to clean up, the PVC will hang in Terminating. Requires finalizer removal to force-delete.
May need help

Force-delete a stuck Terminating PVC

Only use this when the PVC is genuinely stuck
This removes the Kubernetes finalizer, which forces deletion of the PVC object even if the underlying storage cleanup has not completed. Only use this if the PVC has been in Terminating for more than 10 minutes and you are certain no pod is still using it.
jumphost.novacrestbank.local — force-delete stuck PVC 
# PVC stuck in Terminating
[svc-ocp@jumphost ~]$ oc get pvc -n banking-app
NAME          STATUS        VOLUME       CAPACITY   ACCESS MODES   STORAGECLASS   AGE
kyc-docs-pvc  Terminating   pvc-a3f8b2   50Gi       RWX            nfs-standard   8m  ← stuck

# Check if any pod still has it mounted
[svc-ocp@jumphost ~]$ oc get pods -n banking-app | grep -v Completed
No resources found in banking-app namespace.   ← no pods using it

# Remove the kubernetes finalizer to force deletion
[svc-ocp@jumphost ~]$ oc patch pvc kyc-docs-pvc -n banking-app \
  -p '{"metadata":{"finalizers":[]}}' \
  --type=merge
persistentvolumeclaim/kyc-docs-pvc patched

[svc-ocp@jumphost ~]$ oc get pvc -n banking-app
No resources found in banking-app namespace.   ← PVC is gone ✓
Guide 4 · vSphere CSI · Section 04
vSphere CSI — how it works inside your environment
What each component does, how a VMDK ends up mounted inside a pod running on RHCOS
The vSphere CSI driver — already installed by IPI
When you did the IPI install, the installer detected your vSphere platform and automatically deployed the vSphere CSI driver and created the thin-csi StorageClass. This is one of the advantages of IPI over UPI — you did not have to install this manually.

Component breakdown

ComponentWhere it runsWhat it does
vsphere-csi-controller Single pod on any node (control plane) Watches for new PVCs. Calls the vCenter API to create a VMDK on the vSAN datastore. Creates the PV object in Kubernetes. Handles deletion when PVC is removed.
vsphere-csi-node DaemonSet — one pod per every node When a pod is scheduled to a node, this component attaches the VMDK to the correct ESXi host and mounts it into the pod's filesystem. Handles unmount and detach on pod termination.
thin-csi StorageClass Kubernetes object (no pod) The template for vSAN volumes. diskformat: thin = thin-provisioned VMDK. WaitForFirstConsumer = don't create the VMDK until a pod is actually scheduled, so it lands on the right ESXi host.

Why WaitForFirstConsumer matters in your environment

This is the most misunderstood vSphere CSI behaviour
Your ESXi hosts are distributed across 3 racks (leaf switches). A VMDK on vSAN has a "home" — it lives primarily on the ESXi hosts it was created on. If OpenShift creates the VMDK before knowing which pod will use it (Immediate binding), the VMDK might end up on ESXi hosts 1–3 while the pod gets scheduled to ESXi hosts 4–5. vSphere CSI would then have to move the VMDK across the vSAN network before mounting — or the mount fails. WaitForFirstConsumer solves this: the VMDK is only created after the pod is placed, and it is created on the ESXi hosts where the pod landed.

Guide 4 · vSphere CSI · Section 06
StorageClasses for vSAN — creating a tiered catalogue
Adding a second StorageClass with Retain policy for production databases — so data is never accidentally deleted
The reclaim policy problem in banking environments
The default thin-csi StorageClass uses ReclaimPolicy: Delete. This means if a developer deletes their PVC — by accident, during a cleanup, or when removing a namespace — the underlying VMDK and all its data is permanently deleted immediately. For a test environment that is fine. For Oracle database volumes or anything holding customer data, you want ReclaimPolicy: Retain so data is never automatically deleted.
jumphost.novacrestbank.local — create tiered vSAN StorageClasses 
# Create a Retain-policy StorageClass for production databases
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: thin-csi-retain               # name is descriptive: tech + policy
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: csi.vsphere.volume
parameters:
  datastoreurl: "ds:///vmfs/volumes/vsan:52a3f8b2c1d94e7/"
  storagepolicyname: "vSAN Default Storage Policy"
reclaimPolicy: Retain                 # DATA NEVER AUTO-DELETED
allowVolumeExpansion: true
volumeBindingMode: WaitForFirstConsumer
EOF
storageclass.storage.k8s.io/thin-csi-retain created

# Verify both StorageClasses exist
[svc-ocp@jumphost ~]$ oc get storageclass
NAME                  PROVISIONER          RECLAIMPOLICY   VOLUMEBINDINGMODE    ALLOWVOLUMEEXPANSION   AGE
thin-csi (default)    csi.vsphere.volume   Delete          WaitForFirstConsumer true                   3h
thin-csi-retain       csi.vsphere.volume   Retain          WaitForFirstConsumer true                   10s
                                           ↑ Retain = data safe even if PVC deleted
ℹTell your developers: use thin-csi for test/dev workloads. Use thin-csi-retain for anything holding production data. When a PVC using Retain is deleted, the PV moves to Released state — an admin must manually delete it after confirming the data has been backed up or migrated.

Guide 4 · NetApp Trident · Section 08
NetApp AFF A400 and Trident — the architecture
How Trident connects OpenShift to your NetApp, what an SVM is, and how NFS volumes are created on demand
Your NetApp AFF A400 in the draw.io diagram
In your NovaCrestBank environment: NetApp AFF A400 running ONTAP, connected via 25GbE to your Nexus leaf switches on VLAN 30 (10.30.0.0/24), with SnapMirror replication to DR site. It currently serves KYC docs and PDF statements as NFS/SMB shares. We will add OpenShift NFS volumes to this same array on a dedicated SVM (Storage Virtual Machine).

Trident component breakdown

ComponentWhere it runsWhat it does
trident-controller Single pod (on infra node) The brain of Trident. Watches for PVC events. Calls the NetApp ONTAP REST API to create NFS volumes (qtrees or FlexVolumes). Creates the PV object in Kubernetes. Handles deletion and resize.
trident-node DaemonSet — one pod per node When a pod is scheduled to a node, this component mounts the NFS volume into the pod's namespace. Handles unmount when the pod stops.
TridentBackendConfig Kubernetes CRD (object) Tells Trident: which NetApp ONTAP cluster, which SVM, which credentials, which NFS export policy. This is the connection configuration — one per storage system or SVM.
TridentStoragePool Kubernetes CRD (object) Represents a pool of storage capacity on the backend (an aggregate or set of aggregates on the NetApp). Trident provisions volumes from these pools.
StorageClass Kubernetes object The developer-facing catalogue entry. References Trident as the provisioner and specifies parameters (backend, media type, snapshot policy).

What is an SVM (Storage Virtual Machine)?

Key ONTAP concept you must understand
An SVM is a logical partition of the NetApp ONTAP system. Think of it like a vCenter resource pool for storage. Your NetApp AFF A400 might have one SVM for existing bank applications and a separate SVM for OpenShift. Each SVM has its own: IP addresses (NFS data LIFs), credentials, NFS export policies, volume namespace, and quota limits. Separating OCP storage into its own SVM gives you isolation, independent administration, and independent SnapMirror replication. You will create svm-ocp specifically for OpenShift.

# Step 2: Create the TridentBackendConfig
# This tells Trident which NetApp SVM to use and how to provision volumes
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: trident.netapp.io/v1
kind: TridentBackendConfig
metadata:
  name: backend-ontap-nas-svm-ocp
  namespace: trident
spec:
  version: 1
  storageDriverName: ontap-nas         # NFS driver (not ontap-san which is iSCSI/FC)
  managementLIF: 10.30.0.51            # SVM management IP
  dataLIF: 10.30.0.50                  # NFS data LIF — nodes will mount FROM this IP
  svm: svm-ocp                         # the SVM name you created
  credentials:
    name: netapp-svm-ocp-creds         # references the secret above
    type: secret
  defaults:
    spaceReserve: none                 # thin provisioning
    snapshotPolicy: default            # use ONTAP default snapshot schedule
    exportPolicy: ocp-node-access      # the export policy you created
    snapshotDir: "false"               # hide the .snapshot directory from pods
  limitAggregateUsage: "80%"           # never use more than 80% of aggregate capacity
  limitVolumeSize: "1Ti"               # maximum single volume size = 1 TB
  nfsMountOptions: "vers=4.1"          # force NFS v4.1 (better than v3 for OCP)
EOF

Guide 4 · NetApp Trident · Section 12
StorageClasses for NetApp NFS
Creating two StorageClasses — standard NFS and a Retain-policy variant for sensitive data
jumphost.novacrestbank.local — create Trident NFS StorageClasses 
# StorageClass 1: standard NFS (Delete reclaim — for shared/temp data)
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-standard
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: csi.trident.netapp.io
parameters:
  backendType: "ontap-nas"
  fsType: "nfs"
  provisioningType: "thin"
reclaimPolicy: Delete               # volume deleted when PVC deleted
allowVolumeExpansion: true
volumeBindingMode: Immediate        # NFS: create volume immediately (no node dependency)
mountOptions:
  - vers=4.1
  - rsize=1048576
  - wsize=1048576
  - hard
  - timeo=600
EOF
storageclass.storage.k8s.io/nfs-standard created

# StorageClass 2: NFS with Retain policy (for KYC docs, customer data)
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: nfs-retain
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: csi.trident.netapp.io
parameters:
  backendType: "ontap-nas"
  fsType: "nfs"
  provisioningType: "thin"
  snapshotPolicy: "default"          # enable ONTAP snapshots — automatic recovery points
reclaimPolicy: Retain               # DATA NEVER AUTO-DELETED
allowVolumeExpansion: true
volumeBindingMode: Immediate
mountOptions:
  - vers=4.1
  - rsize=1048576
  - wsize=1048576
  - hard
  - timeo=600
EOF
storageclass.storage.k8s.io/nfs-retain created

Live demo — ReadWriteMany NFS shared volume
Three pods on three different nodes all writing to the same NFS volume simultaneously — the RWX use case vSAN cannot do
The demo scenario — KYC document service
NovaCrestBank's KYC (Know Your Customer) document service runs 3 replicas across 3 worker nodes for high availability. All 3 replicas must be able to read and write to the same document storage simultaneously. This is impossible with vSAN (RWO only). This is exactly what NFS RWX was designed for.
jumphost.novacrestbank.local — RWX NFS demo: 3 pods, 1 shared volume 
[svc-ocp@jumphost ~]$ oc new-project kyc-service
Now using project "kyc-service" on server "https://api.ocp.novacrestbank.local:6443".

# Create the RWX PVC — ReadWriteMany = multiple pods on multiple nodes
[svc-ocp@jumphost ~]$ cat << 'EOF' | oc apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: kyc-documents
  namespace: kyc-service
spec:
  accessModes:
    - ReadWriteMany          # ← this is the key: multiple pods can mount simultaneously
  storageClassName: nfs-retain    # Retain policy + snapshots for customer documents
  resources:
    requests:
      storage: 500Gi         # 500 GB for KYC document storage
EOF

Guide 4 · Decisions · Section 14
Which storage for which workload — the decision guide
The mental model a senior engineer uses when a developer asks "what StorageClass should I use?"

The decision tree — four questions

  • 1
    Does this workload need multiple pods on different nodes to access the same data simultaneously?

    Yes → You must use NFS (RWX). vSAN cannot do this. Use nfs-standard or nfs-retain.
    No → Either storage works. Go to question 2.

  • 2
    Is this a database or anything requiring direct block-level disk access?

    Yes → Use vSAN (thin-csi or thin-csi-retain). Databases (PostgreSQL, MySQL, MongoDB, Redis) work better on block storage — lower latency, no NFS protocol overhead, no locking issues.
    No → Either works. Go to question 3.

  • 3
    Does this data belong to a customer, or is it required for audit/compliance?

    Yes → Use the Retain variant (thin-csi-retain or nfs-retain). Data must not be automatically deleted when a PVC is removed. An admin must explicitly approve deletion after confirming backup.
    No → Delete reclaim policy is acceptable. Go to question 4.

  • 4
    Do you need ONTAP snapshots (instant recovery points)?

    Yes → Use nfs-retain (configured with snapshotPolicy: default). ONTAP snapshots give you point-in-time recovery of the volume without backup tools.
    No → nfs-standard is fine.
