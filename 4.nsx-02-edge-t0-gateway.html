What this guide builds — in order

  1. Edge Node VMs — edge-01 and edge-02 deployed by NSX Manager onto esxi-mgmt-01 and esxi-mgmt-02. Large form factor (8 vCPU, 32 GB) required for NSX Load Balancer support.
  2. Edge Cluster NCB-Edge-Cluster — groups both Edge Nodes. The Tier-0 Gateway attaches to the cluster, not individual nodes, enabling HA.
  3. Tier-0 Gateway NCB-T0-Gateway — Active-Active mode, four uplink interfaces, BGP ASN 65010, four BGP sessions to Cisco Leaf-01 and Leaf-02, route redistribution of Connected routes, and HA VIP.
Edge Nodes are deployed by NSX Manager — not through vCenter directly
Unlike regular VMs deployed through vCenter, Edge Nodes are deployed entirely through the NSX Manager GUI. NSX Manager calls the vCenter API on your behalf. You will see edge-01 and edge-02 appear as VMs in vCenter after NSX deploys them — do not create them manually there.

Why uplink IPs are set on the T0 interface — not on the Edge Node itself
The Edge Node deployment wizard only configures the management IP and the TEP IP. The uplink IPs (10.102.0.2, 10.103.0.2, etc.) are configured later as T0 Gateway Interfaces — because those IPs belong to the routing function of the Tier-0, not to the Edge Node hardware itself. The Edge Node is a platform; the T0 is the software router running on it. Separating the two makes it possible to change routing configuration without redeploying the Edge Node VM.

NSX-02 · Section 06
Deploy edge-01 · Step 4 — Configure NSX switch and uplinks
The step that maps edge-01's two fast-path NICs to the VLAN-backed uplink segments — these become the physical BGP connections to Cisco
What Step 4 physically configures on the Edge Node
The Edge Node VM has four virtual NICs. Steps 1–3 configured the management NIC and the TEP NIC. What does Step 4 configure for the remaining two?
Step 4 creates an NSX virtual switch (N-VDS) on the Edge Node and maps its two remaining virtual NICs — called fp-eth1 and fp-eth2 (fast-path Ethernet) — to VLAN-backed segments on NCB-VLAN-TZ.

fp-eth1 will carry VLAN 102 traffic → physical connection to Cisco Leaf-01.
fp-eth2 will carry VLAN 103 traffic → physical connection to Cisco Leaf-02.

These interfaces are what the Tier-0 Gateway uses for its BGP uplinks. Without this step, the Edge Node has no way to reach the physical network — it can participate in GENEVE tunnels with other hosts but has no physical network exit.

Why anti-affinity — edge-01 on esxi-mgmt-01 and edge-02 on esxi-mgmt-02
Both Edge Nodes could be placed on the same ESXi host. Why does NovaCrestBank place them on different hosts?
Because both Edge Nodes run the Tier-0 Gateway simultaneously in Active-Active mode. If both Edge Nodes are on the same ESXi host and that host fails, both Edge Nodes go down simultaneously — all four BGP sessions drop, all external routing for the entire NSX environment stops. Every OCP pod loses internet connectivity. Every management API call fails. Every DNS query from the overlay fails to reach DC01.

With edge-01 on esxi-mgmt-01 and edge-02 on esxi-mgmt-02: if esxi-mgmt-01 fails, edge-01 goes down but edge-02 continues running on esxi-mgmt-02. The BGP sessions through edge-02 (to both Leaf-01 and Leaf-02) remain established. Traffic continues flowing through the surviving Edge Node with no manual intervention.

NIST SP 800-53 CP-7 — Alternate Processing Site: redundant components must not share a single point of failure. NIST SP 800-53 SC-5 — Denial of Service Protection.

NSX-02 · Section 09
What the Edge Cluster is and why the T0 attaches to it — not to individual Edge Nodes
A short but important concept before creating the cluster
Edge Cluster as the HA abstraction — why the T0 connects to a cluster not a node
There are two Edge Nodes. The Tier-0 Gateway could be attached to edge-01 specifically. Why attach it to an Edge Cluster instead?
Because the Edge Cluster is the HA construct. When the T0 is attached to NCB-Edge-Cluster, NSX distributes the T0 across both Edge Nodes automatically. In Active-Active mode both Edge Nodes simultaneously run the T0 and forward traffic. If edge-01 fails, edge-02 continues all routing with no manual intervention. NSX Manager detects the failure via BFD within approximately 1.5 seconds and withdraws routes through the failed node.

If the T0 were attached to edge-01 directly: there is no HA. edge-01 failure = T0 failure = all external routing for the entire NSX environment stops. The Edge Cluster abstraction is what enables resilience — attaching the T0 to the cluster is the entire configuration required to get Active-Active failover.

NSX-02 · Section 13
Create the Edge uplink segments first
The T0 Interfaces require VLAN-backed segments to attach to — these must exist before adding interfaces to the T0
Why VLAN-backed segments must exist before T0 interfaces can be created
The T0 external interface needs to be "connected to" a segment. Why can the interface not be created without a pre-existing segment?
Because in NSX's data model, every interface lives on a segment — the segment is the logical network that the interface is attached to. For the T0's uplink interfaces, the segments are VLAN-backed (using NCB-VLAN-TZ with VLAN IDs 102 and 103). These segments represent the physical VLAN connections between the Edge Node and Cisco. Without the segments, there is no logical network for the T0 interface to connect to — the field in the interface form will have nothing to select.

NSX-02 · Section 14
Add T0 external interfaces — four uplink IPs across both Edge Nodes
Four interfaces: each Edge Node gets one IP on VLAN 102 (toward Leaf-01) and one IP on VLAN 103 (toward Leaf-02)
Why four interfaces — not two
In Active-Active mode each Edge Node independently peers with both Cisco leaf switches. edge-01 has its own IP on VLAN 102 (10.102.0.2) and its own IP on VLAN 103 (10.103.0.2). edge-02 has its own IP on VLAN 102 (10.102.0.3) and on VLAN 103 (10.103.0.3). This gives 4 BGP sessions total — each Edge Node to each Cisco leaf. If either Edge Node fails, the Cisco leaves still have BGP sessions to the surviving Edge Node. Full dual-fabric redundancy is maintained.

Enable BGP — ASN 65010 and router ID

  • 2
    Fill in the BGP configuration
    NSX
    NCB-T0-Gateway → BGP Configuration
    BGP Status
    BGP is disabled by default. This toggle activates the BGP process on the T0. Without this, the BGP Neighbors section will be greyed out.
    Local AS Number *
    NovaCrestBank's NSX ASN defined in VCF-01. Must match the remote-as value configured on both Cisco leaf switches for their BGP peer entries pointing to the NSX Edge Node IPs. A mismatch prevents BGP sessions from establishing.
    Router ID
    A unique 32-bit number in IP format identifying this BGP speaker. Using edge-01's management IP is standard practice — it is always up when the Edge Node is functioning and is already documented. The Router ID is an identifier; it does not need to be reachable by BGP peers.
    ECMP
    ECMP (Equal-Cost Multi-Path) is required for Active-Active mode. It allows BGP to install multiple equal-cost paths simultaneously and distribute traffic across both Edge Nodes. Without ECMP, only one BGP path is used even though both Edge Nodes are active.
    Graceful Restart
    Leave Disabled for initial deployment. Graceful Restart allows forwarding to continue during a BGP restart — useful for planned maintenance but adds complexity. Enable only after the environment is stable and the Cisco side supports it.

NSX-02 · Section 17
Route redistribution — telling the T0 what to advertise into BGP
Without route redistribution, the BGP sessions form but the Cisco fabric never learns about the NSX segment prefixes — VMs are unreachable from the physical network
Why route redistribution is a separate step — not automatic
The BGP sessions are now configured. Why does the Cisco fabric not automatically learn about NSX segment routes like 10.70.0.0/24 and 10.50.0.0/24?
Because NSX separates BGP configuration from route policy. The BGP sessions define who to peer with. Route redistribution defines what to put into BGP. Without explicitly enabling route redistribution, NSX will establish BGP sessions but advertise zero routes — the Cisco fabric will have no idea that 10.70.0.0/24 (infrastructure segment) or 10.50.0.0/24 (OCP nodes) exist.

NovaCrestBank enables redistribution of "Connected" routes. In NSX terminology, Connected routes are the prefixes of segments directly connected to the T0 or T1 gateways — including seg-infra-services (10.70.0.0/24) and seg-ocp-nodes (10.50.0.0/24), which are connected to T1 gateways that are in turn connected to the T0. These are the routes the Cisco fabric needs to reach NSX-hosted VMs and OCP nodes from anywhere on the physical network.

PCI Req 1.2.1 — Only explicitly permitted routes are redistributed into BGP. No other route types (Static, BGP, or NAT) are redistributed unless specifically needed — this prevents accidental route leakage.

NSX-02 · Section 18
HA VIP — the floating IP for T0 Gateway failover
An optional but recommended HA VIP provides a single stable IP address for management and monitoring tools that need to reach the T0 regardless of which Edge Node is active
What the HA VIP is and when it matters for Active-Active mode
Active-Active mode means both Edge Nodes are always active. If both are always running, why configure an HA VIP at all?
In Active-Active mode, routing traffic is distributed across both Edge Nodes via ECMP — no VIP is needed for routing. The HA VIP serves a different purpose: it provides a single stable IP address for management connections to the T0's control plane — used by monitoring tools, SNMP, and any service that needs to query the T0's operational state regardless of which Edge Node is currently more "primary."

Additionally, some NSX services (NAT, certain inspection services) operate in Active-Standby even within an Active-Active T0 — the HA VIP tells those services which Edge Node to treat as the primary. For NovaCrestBank this is not critical for initial deployment, but configuring it now avoids needing to add it later.

The HA VIP is a floating IP that NSX assigns to the "primary" Edge Node in the cluster. It lives on the uplink segment and is reachable from the physical network.

If any session shows Active, Idle, or Connect instead of Established
Active: NSX is actively trying to connect but Cisco has not responded. Check: Is the BGP password the same on both sides? Is Cisco's remote-as set to 65010? Is VLAN 102/103 reachable from the Edge Node?

Idle: NSX is not attempting the connection. Check: Is BGP enabled on the T0 (toggle in BGP global settings)? Is the source interface assigned correctly?

What NSX-03 builds next — Tier-1 Gateways and Segments

NSX-02 built the north-south exit from the NSX overlay — the T0 Gateway connects NSX to the physical Cisco fabric via BGP.

NSX-03 builds the tenant routing and the logical networks that VMs and OCP nodes actually connect to:

  • Create NCB-T1-Infra — the Tier-1 Gateway for infrastructure service VMs (DC01, DC02, IssuingCA). Connect it to NCB-T0-Gateway.
  • Create NCB-T1-OCP — the Tier-1 Gateway for OpenShift nodes and pods. Connect it to NCB-T0-Gateway.
  • Create seg-infra-services (10.70.0.0/24) — connect to NCB-T1-Infra. Migrate DC01, DC02, IssuingCA VMs onto this segment.
  • Create seg-ocp-nodes (10.50.0.0/24) — connect to NCB-T1-OCP. OCP node VMs will connect to this segment during installation.

After NSX-03 is complete, those segment prefixes (10.70.0.0/24 and 10.50.0.0/24) will automatically appear in the T0 route redistribution — and will be advertised to the Cisco fabric via BGP. The Cisco leaf switches will know how to route traffic to all NSX-hosted VMs.
