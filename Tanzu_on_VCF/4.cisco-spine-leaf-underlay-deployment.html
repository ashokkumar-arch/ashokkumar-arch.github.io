1. Overview

This guide deploys the enterprise Cisco Nexus spine-leaf underlay fabric that becomes the physical networking foundation for the NovaCrestBank VMware Cloud Foundation and Tanzu platform.

This underlay fabric is responsible for:

  • Providing Layer-3 routed connectivity
  • Supporting NSX overlay networking
  • Providing low-latency east-west traffic flow
  • Supporting enterprise Kubernetes networking
  • Providing scalable BGP route distribution
  • Supporting future AI and GPU workload scaling
Important:

VMware Cloud Foundation, NSX, Kubernetes, observability, GitOps, AI workloads, and future hybrid-cloud networking all depend on the physical fabric being designed correctly from the beginning.

This guide focuses on production-grade enterprise networking principles rather than simple lab-only deployment shortcuts.

4. Initial Cisco Nexus Bootstrap

4.1 Initial Login

After physically cabling the switches and connecting console access, perform the initial Cisco Nexus bootstrap configuration. 

switch> enable
switch# configure terminal

4.2 Configure Hostname

hostname mum-spine-01

4.3 Configure Management Interface

interface mgmt0
  ip address 10.10.10.11/24
  no shutdown

4.4 Configure Default Route

ip route 0.0.0.0/0 10.10.10.1

4.5 Configure NTP

ntp server 10.10.70.40

4.6 Configure DNS

ip domain-lookup
ip name-server 10.10.70.40

4.7 Configure Local Administrative User

username netadmin password STRONG_PASSWORD role network-admin
Important:

Local administrator accounts are initially required during bootstrap.

Later guides integrate Cisco infrastructure with:

• Active Directory
• TACACS+
• Centralized authentication
• Enterprise RBAC policies

5. Spine Switch Configuration

5.1 Enable Required Features

feature bgp
feature interface-vlan
feature ospf
feature lacp
feature nv overlay
feature vn-segment-vlan-based

5.2 Configure Spine Uplink Interfaces

interface ethernet1/1
  description TO-LEAF-01
  no switchport
  ip address 172.16.0.1/31
  no shutdown

interface ethernet1/2
  description TO-LEAF-02
  no switchport
  ip address 172.16.0.3/31
  no shutdown

5.3 Configure BGP on Spine

router bgp 65100

  router-id 10.255.0.1

  neighbor 172.16.0.0 remote-as 65101
  neighbor 172.16.0.2 remote-as 65102

  address-family ipv4 unicast

6.1 Configure Leaf Hostname

hostname mum-leaf-01

6.2 Enable Features

  name VMOTION

6.5 Configure Trunk Port Toward ESXi

interface ethernet1/10

  switchport
  switchport mode trunk

  switchport trunk allowed vlan 20,30,40,50,60,70,80,90

  spanning-tree port type edge trunk

  no shutdown

7. vPC Configuration

7.1 Why vPC?

vPC (Virtual Port Channel) allows ESXi hosts to connect redundantly to two separate leaf switches while appearing as a single logical switch.

This eliminates:

  • Single switch dependency
  • STP blocking issues
  • Bandwidth inefficiency

7.2 Configure vPC Domain

vpc domain 10

  peer-switch
  peer-keepalive destination 10.10.10.12

7.3 Configure Peer Link

interface port-channel1

  switchport
  switchport mode trunk
  vpc peer-link

interface ethernet1/47
  channel-group 1 mode active

interface ethernet1/48
  channel-group 1 mode active

7.4 Configure ESXi vPC Port-Channel

interface port-channel10

  description ESXI-HOST-01

  switchport
  switchport mode trunk

  vpc 10

8. BGP Underlay Deployment

8.1 Configure BGP on Leaf

router bgp 65101

  router-id 10.255.1.1

  neighbor 172.16.0.1 remote-as 65100
  neighbor 172.16.0.5 remote-as 65100

  address-family ipv4 unicast

show bgp sessions

Important:

NSX overlay networking requires jumbo frames.

Ensure MTU consistency across:

• Spine switches
• Leaf switches
• ESXi vDS uplinks
• NSX TEP interfaces
