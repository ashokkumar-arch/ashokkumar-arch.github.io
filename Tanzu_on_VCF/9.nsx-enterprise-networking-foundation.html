2. What Exactly is NSX?

A reader may immediately get a question:

Question:

Is NSX simply another VLAN solution?

Or is it a firewall?

Or is it SDN?

2.1 Real Enterprise Explanation

NSX is VMware’s software-defined networking and security platform.

It virtualizes networking in the same way ESXi virtualized compute.

2.2 What NSX Actually Provides

Capability Purpose
Overlay Networking Logical networking independent of physical VLAN limits
Distributed Routing Routing directly inside hypervisors
Microsegmentation East-west security enforcement
Kubernetes Networking Tanzu and container networking integration
Distributed Firewall Kernel-level workload security

5. Overlay Networking Explained

A reader may immediately ask:

Question:

What exactly is “overlay networking”?

Why not directly use VLANs everywhere?

5.1 Real Enterprise Explanation

Overlay networking allows logical networks to run on top of physical infrastructure without depending entirely on physical VLAN scalability.

5.2 Real-World Example

Physical fabric: 

VLAN 50 --> TEP Network

Inside NSX: 

Segment: prod-web-segment
Segment: prod-app-segment
Segment: prod-db-segment
Segment: tanzu-supervisor-segment
Segment: vks-workload-segment

All these logical networks can exist on top of a smaller physical VLAN footprint.

6. Tunnel Endpoints (TEPs)

A reader may get another question:

Question:

What exactly are TEPs?

Why does NSX require them?

6.1 Real Enterprise Explanation

TEP means Tunnel Endpoint.

TEP interfaces create Geneve tunnels between ESXi hosts and Edge nodes.

6.2 TEP VLAN Planning

Purpose VLAN Subnet
NSX TEP Network 50 10.10.50.0/24

6.3 Why Dedicated TEP Networks Matter

  • Overlay traffic becomes isolated
  • MTU tuning becomes easier
  • East-west traffic avoids bottlenecks
  • Kubernetes overlay networking scales better

7. NSX Transport Zones

7.1 What is a Transport Zone?

Another common question:

Question:

What exactly is a transport zone?

Why does NSX need this abstraction?

7.2 Real Enterprise Explanation

A transport zone defines where NSX networking can exist.

It determines which hosts participate in specific NSX networks.

7.3 Enterprise Transport Zones

Transport Zone Purpose
tz-overlay-prod Kubernetes and overlay workloads
tz-vlan-edge North-south VLAN uplinks

8. NSX Edge Cluster Deployment

8.1 Why Edge Nodes Exist

A reader may get another question:

Question:

If routing already exists inside ESXi hosts, why do we still need NSX Edge nodes?

8.2 Real Enterprise Explanation

Edge nodes provide:

  • North-south routing
  • BGP connectivity
  • Load balancing
  • NAT services
  • External connectivity

8.3 Enterprise Edge Cluster Design

Edge Node Purpose
edge-mum-01 Primary
edge-mum-02 Redundant
Important:

Real enterprise environments always deploy NSX Edge nodes redundantly.

Single edge-node designs create a network single point of failure.

11. NSX Microsegmentation and Zero Trust

11.1 What is Microsegmentation?

A reader may immediately get another question:

Question:

Traditional firewalls already exist.

Why is microsegmentation required?

11.2 Real Enterprise Explanation

Traditional firewalls mainly protect north-south traffic.

Modern attacks move east-west between workloads.

NSX Distributed Firewall provides:

  • Kernel-level enforcement
  • Per-workload security
  • Kubernetes namespace isolation
  • Dynamic policy enforcement

12. Enterprise MTU Planning

12.1 Why MTU Matters in NSX

Critical:

Overlay networking adds encapsulation overhead.

If MTU is incorrectly configured:

• Kubernetes traffic fragmentation occurs
• Tanzu networking becomes unstable
• NSX tunnels flap
• Performance degrades severely

12.2 Enterprise MTU Design

Network MTU
Physical Fabric 9000
NSX Overlay 8900
