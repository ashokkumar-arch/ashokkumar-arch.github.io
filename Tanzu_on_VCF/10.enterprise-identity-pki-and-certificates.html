1. Overview

In enterprise production environments, identity and certificate management are foundational security requirements.

Before Tanzu, Kubernetes, GitOps, CI/CD, AI, hybrid cloud, or application deployments exist, enterprise identity architecture must already be correctly designed.

Important:

Most enterprise outages and security incidents do not happen because Kubernetes failed.

They happen because:

• Certificates expired
• DNS failed
• Authentication failed
• Incorrect RBAC permissions existed
• Trust chains broke
• LDAP integration failed
• Service accounts were misconfigured

  • Enterprise Active Directory architecture
  • Enterprise DNS integration
  • Enterprise PKI hierarchy
  • Certificate trust architecture
  • VCF authentication integration

  • Tanzu identity integration
  • Enterprise RBAC design

2. Why Identity Architecture Matters

A reader may immediately get a question:

Question:

Why not simply use local administrator accounts everywhere?

Why build such a large identity architecture?

2.1 Real Enterprise Explanation

Enterprises cannot operate production infrastructure using disconnected local accounts.

Real enterprise environments require:

  • Centralized authentication
  • Centralized authorization
  • Auditing
  • Compliance
  • Certificate trust
  • Identity lifecycle management

3.2 Enterprise Domain Naming

Active Directory Domain:
corp.novacrestbank.internal

3.3 Why We Use .internal

A reader may get another important question:

Question:

Why are we using:

corp.novacrestbank.internal

instead of:

.local or .corp?

3.4 Real Enterprise Explanation

  • .local conflicts with mDNS/Bonjour
  • .corp created historical enterprise conflicts
  • .internal clearly indicates private internal infrastructure
  • .internal avoids public DNS overlap

4. Enterprise DNS Architecture

4.1 Why DNS is Critical

Critical:

Kubernetes, NSX, VCF, Tanzu, vCenter, and PKI all depend heavily on DNS.

If DNS fails:

• NSX services fail

5. Enterprise PKI Architecture

5.1 What is PKI?

A reader may immediately ask:

Question:

Why do enterprises create their own certificate authorities?

Why not use self-signed certificates everywhere?

5.2 Real Enterprise Explanation

Self-signed certificates create trust problems.

Enterprise PKI provides:

  • Centralized trust
  • Certificate lifecycle management
  • Automated certificate issuance
  • Revocation support
  • Enterprise compliance

5.3 Enterprise PKI Hierarchy

+----------------------------------------------------+
|                Offline Root CA                     |
|                mum-rootca01                        |
+----------------------------------------------------+
                      |
                      |
          Signs subordinate CA certificate
                      |
                      |
+----------------------------------------------------+
|                Issuing CA                          |
|                mum-issuingca01                     |
+----------------------------------------------------+
                      |
        +-------------+-------------+
        |             |             |
   vCenter Cert   NSX Cert     Kubernetes Certs
Important:

Real enterprise environments keep the Root CA offline.

Only the Issuing CA remains online for daily operations.
