The standard vSwitch — what it is and why it falls short
Every ESXi host currently has a standard vSwitch (vSwitch0) created automatically during installation. A standard vSwitch is a per-host construct — it exists only on the single ESXi host where it was created, has its own configuration, and is managed separately from every other host's vSwitch. In a five-host cluster, you have five completely separate vSwitches. If you need to add a port group, you must add it to all five hosts individually. If you make a mistake on one host, it differs silently from the others. There is no consistency enforcement.

The standard vSwitch vs vDS — a direct comparison

Network I/O Control (NIOC) Not available Available — reserves bandwidth per traffic type (vSAN, vMotion, VM)

vMotion requirement Port groups must be manually identical on all hosts Port groups are identical by design — vMotion always works

The LACP requirement — why this is non-negotiable for NovaCrestBank

The Cisco leaf switches have the ESXi ports configured as LACP port channels (Po101 through Po106 in Guide 5). LACP is a dynamic link aggregation protocol — both sides must agree to form the bond. The standard vSwitch does not support LACP at all. It can do static port channel (where the switch side runs LACP but the ESXi side uses a static "IP hash" policy) — but this creates an imbalance: the switch LACP state machine detects the ESXi side as non-negotiating and falls back to Individual mode. The port channels on the Cisco switch show (I) Individual — not (P) Active. Traffic works but without the link-level negotiation that catches cable errors and prevents forwarding loops. The vDS supports LACP end-to-end — ESXi sends and receives LACP PDUs, the Cisco switch responds, and the port channel reaches full (P) status. This is the correct enterprise configuration.

The decisions this guide makes — and the reasoning behind each one
Every major vDS design decision stated as a question and answered with the reason — the mental model for making these decisions in any environment
Decision 1
Why one vDS for the entire cluster — not one vDS per host or one vDS per traffic type?
One vDS is the simplest correct architecture. A single vDS spans all hosts and contains all port groups. All port groups are visible and consistent on every host. vMotion can move a VM between any two hosts without compatibility issues because the destination port group always exists. Multiple vDS instances add operational complexity (you must manage two switch objects, track which host is on which switch, and ensure both are always consistent) with no meaningful benefit for NovaCrestBank's five-host environment. VMware recommends one vDS per cluster except in specific multi-tenant or regulatory isolation scenarios that do not apply here.
Decision 2
Why 6 uplinks on the vDS — not 2 or 4?
Because each ESXi host has 6 physical NICs, each with a distinct purpose. The vDS needs one uplink slot per physical NIC so it can be precisely controlled which vmnic carries which traffic type. With 6 uplinks: uplink1 and uplink2 carry management and Windows traffic (vmnic0/1, VLAN 10 and 70); uplink3 and uplink4 carry VM, vMotion, and OCP traffic (vmnic2/3, VLAN 20/50/60); uplink5 and uplink6 carry vSAN and NFS storage traffic (vmnic4/5, VLAN 25 and 30). If only 2 uplinks were defined, all 6 NICs would have to share two slots — losing the ability to separate storage traffic from management traffic at the NIC level. The vDS uplink count is set at creation and changing it later requires a complex reconfiguration.
Decision 3
Why use LACP Enhanced mode — not LACP Basic?
Because Enhanced mode supports multiple LACP Link Aggregation Groups (LAGs) per vDS — one per NIC pair. NovaCrestBank has three NIC pairs per host: management pair (vmnic0/1), VM/OCP pair (vmnic2/3), and vSAN pair (vmnic4/5). Each pair needs its own LACP bond — they connect to different Cisco port channels (Po101, Po102, Po103 respectively) and carry different VLANs. LACP Basic mode only supports a single LAG per vDS. LACP Enhanced mode supports multiple LAGs — one for each NIC pair. This is why Enhanced mode is mandatory for NovaCrestBank's design.
Decision 4
Why is Network I/O Control (NIOC) enabled?
Because banking-grade reliability requires guaranteed bandwidth for storage and management traffic even during VM traffic bursts. NIOC reserves shares of physical NIC bandwidth per traffic type. Without NIOC: a VM generating a bulk data transfer at 25G could starve vSAN replication traffic or management connectivity on the same NIC. With NIOC: vSAN traffic gets a guaranteed bandwidth share, management traffic gets a guaranteed share, and VM traffic gets the remainder. In a banking environment, vSAN latency directly affects VM disk I/O which directly affects transaction throughput. NIOC is the network-layer mechanism that prevents one workload from degrading another.
Decision 5
Why does each port group carry only specific VLANs — not all VLANs?
The same least-privilege principle that was applied to the Cisco trunk ports in Guide 6 applies here. A VM connected to PG-VLAN60-VM-Production should not be able to send traffic on VLAN 25 (vSAN) or VLAN 10 (management). By assigning each port group a specific VLAN ID, ESXi tags all traffic from that port group with that VLAN. A VM on PG-VLAN60-VM-Production can only send tagged VLAN 60 frames — it physically cannot inject traffic into other VLANs without being moved to a different port group. This is a security control, not just an organisational preference.
Decision 6
Why is MTU 9000 set only on the vSAN and NFS port groups — not all port groups?
Because jumbo frames must be consistent end-to-end on every hop, and only the storage paths support it consistently. The Cisco leaf SVIs for VLAN 25 and VLAN 30 are configured at MTU 9000. The SVIs for VLANs 10, 20, 50, 60, and 70 are at MTU 1500. If a management port group (VLAN 10) were set to MTU 9000 but the VLAN 10 SVI on the leaf were still at 1500, the result is silent packet drops for any management frame exceeding 1500 bytes. Consistent end-to-end MTU is more important than using the largest possible MTU. Only the paths verified as end-to-end 9000 (vSAN and NFS) get jumbo frames.

Why Phase 1 only — the migration risk and how to manage it
Moving ESXi hosts from a standard vSwitch to the vDS is the most operationally dangerous step in the VMware build — here is why, and how Phase 1 manages the risk
The single most dangerous operation in the VMware build
Migrating a management VMkernel adapter from a standard vSwitch to the vDS severs and re-establishes the host's management network connection in one operation. If anything goes wrong — wrong port group, wrong VLAN, wrong uplink assignment — the host becomes unreachable via the management network. vCenter loses connection to the host. You cannot fix it remotely because your remote access path (vCenter API, SSH to vmk0) is gone. You must use iLO to access the DCUI and manually restore the management network configuration. The risk is real and manageable — but only if you understand it before you start.

What Phase 1 covers — and what it defers

Phase 1 — this guide

  • Create the vDS with 6 uplinks
  • Create all 7 VLAN port groups
  • Add all 5 hosts to the vDS
  • Migrate vmnic0 (management) to vDS uplink1
  • Migrate vmk0 (management VMkernel) to vDS PG-VLAN10
  • Configure LACP LAG for vmnic0/vmnic1 pair
  • Add vmnic1 to the LACP LAG (completing the management bond)

Phase 2 — after Windows DC and AD CS

  • Configure vMotion VMkernel (vmk1) on VLAN 20
  • Configure vSAN VMkernel (vmk2) on VLAN 25
  • Configure NFS VMkernel (vmk3) on VLAN 30
  • Migrate remaining NIC pairs (vmnic2/3 and vmnic4/5) to vDS
  • Configure remaining LACP LAGs
  • Remove old standard vSwitch from all hosts

Why Phase 1 stops at management only

Phase 1 establishes that the vDS is working correctly for the management path — the path you need to continue configuring everything else. If the management migration fails, you detect it immediately because vCenter loses the host — and you fix it before touching any other VMkernel adapters. If you tried to migrate all VMkernel adapters in one operation and it failed, you would have no management path, no vMotion path, and no storage path simultaneously.

Phase 2 is deferred until after the Windows Domain Controllers and AD CS are deployed (because certificate replacement uses the AD CS PKI, and stable management networking is required throughout that process). The vDS will be partially configured during Phase 1 — some uplinks unused, vMotion/vSAN/NFS VMkernel adapters not yet created. This is the correct and intentional state.

The safety net — iLO is always there

If the management VMkernel migration fails and a host becomes unreachable, the recovery procedure is:

  1. Open iLO for that host (10.40.0.31–35) — this is independent of the production network
  2. Launch the HTML5 remote console
  3. Press F2 in the DCUI, authenticate as root
  4. Navigate to Configure Management Network
  5. Restore the correct VLAN (10), IP address (10.10.0.11), and gateway (10.10.0.1)
  6. Press Escape → apply changes
  7. The host becomes reachable again within seconds

This is exactly why iLO was configured first, in Cisco Guide 3. You always have a path in, regardless of what happens to the production network configuration.

Why this name
Why "NCB-vDS-01" and not "dvSwitch" or "vDS-Management" or "NovaCrestBankDistributedSwitch"?
Three naming principles applied simultaneously:

1. Organisation prefix (NCB). If this vCenter ever manages multiple organisations' resources (e.g., a tenant model), the NCB prefix immediately identifies this as a NovaCrestBank object — not a shared or tenant object.

2. Object type in the name (vDS). The name tells you what kind of object this is without having to look at the icon. In a large vCenter with hundreds of objects, searchable, self-describing names save time.

3. Sequence number (01). This is the first vDS. If a second vDS is ever added (for a dedicated storage network, for example), it becomes NCB-vDS-02. The sequence number prevents naming collisions and makes ordering obvious.

The alternative ("dvSwitch" — vCenter's default) tells you nothing. You cannot tell which organisation owns it, what it is for, or how many there are. Never accept default names for production infrastructure objects.

Type NCB-vDS-01 in the Name field. The Location should already show NovaCrestBank-DC1. Click Next.

Why version 8.0.3 — the latest
Should you choose the latest vDS version or an older compatible version?
Always choose the latest version that all your hosts support — which is 8.0.3 because all five hosts run ESXi 8.0.3.

The temptation to choose an older version "for compatibility" is a mistake in NovaCrestBank's environment. All five hosts are identical, all run ESXi 8.0.3, and none will ever be downgraded. Older vDS versions lack features that are needed: vDS 6.x does not support LACP Enhanced mode. vDS 7.x lacks some NIOC v3 capabilities. vDS 8.0.3 has everything.

The only valid reason to choose an older vDS version is if you have a mixed environment where some hosts run an older ESXi version — then you must pick the version that all hosts support. NovaCrestBank does not have this problem.

Select 8.0.3 (latest). Click Next.

The three settings — what to enter and why

Number of uplinks — set to 6
Why exactly 6 uplinks — and what happens if you set it wrong?
One uplink per physical NIC, because each NIC has a distinct role and must be individually assignable.

The 6 uplinks will be assigned as: Uplink1=vmnic0, Uplink2=vmnic1, Uplink3=vmnic2, Uplink4=vmnic3, Uplink5=vmnic4, Uplink6=vmnic5. This one-to-one mapping makes the configuration explicit and auditable. When a network engineer looks at the vDS uplink configuration, they can immediately see which physical NIC is assigned to which uplink slot and which LACP LAG it belongs to.

If you set fewer uplinks (e.g., 2): you would have to share uplink slots among multiple NICs — losing the ability to assign specific NICs to specific port groups. If you set more uplinks (e.g., 8): you have empty uplink slots with no NICs assigned — wasteful and potentially confusing. 6 is exactly right.

If you set the wrong number and want to change it later: you can add uplinks to a vDS after creation (Actions → Edit Settings → change the count). You cannot reduce the number of uplinks without first ensuring no NICs or LAGs are assigned to the slots being removed.
Network I/O Control — enable it
NIOC sounds complex. Do you actually need it for a 5-host cluster?
Yes — because NovaCrestBank has storage (vSAN), infrastructure (management, vMotion), and workload (VMs) traffic all sharing the same physical NICs via LACP bonds.

The scenario NIOC prevents: esxi01 has a VM running a large batch job that generates 20 Gbps of VM-production traffic. The LACP bond for that NIC pair is 50 Gbps (2 × 25G). Without NIOC, the VM's 20 Gbps plus vSAN replication traffic plus vMotion traffic can collectively saturate the bond — causing vSAN heartbeat timeouts that trigger unnecessarily, or vMotion migrations that stall mid-transfer. With NIOC, vSAN is guaranteed a minimum share (typically 25%), vMotion is guaranteed a share, and VM traffic gets the remainder. The VM batch job slows down — not the storage layer.

Disabling NIOC now saves no time (it is one toggle) but creates a problem you will only discover during peak load — which in a bank is typically end-of-day transaction processing. Enable it now.
Default port group — leave unchecked
Why not let vCenter create a default port group automatically?
Because the default port group has a generic name ("DPortGroup") and no VLAN assignment — it is a trap.

If you accept the default port group, it appears in the vDS with VLAN 0 (untagged) and the name "DPortGroup". Someone will eventually connect a VM to it thinking it is a valid port group. Traffic will arrive at the switch untagged and land on native VLAN 1 — which has no SVI and no routing. The VM will have no network access but no obvious error either. This wastes 30 minutes of troubleshooting. Creating all port groups manually ensures every port group has a meaningful name, the correct VLAN, and is created intentionally.

Enter 6 for uplinks. Set NIOC to Enabled (version 3). Leave the default port group checkbox unchecked. Click Next.

Traffic typeShares settingReservationWhy this setting
vSAN High (100) 0 Mbps vSAN storage I/O is the most latency-sensitive workload — disk writes from VMs transit vSAN. High shares ensure vSAN is served before VM traffic during contention.

Why shares not reservations for vSAN and NFS?
Why use "High shares" for vSAN rather than a fixed Mbps reservation?
Because shares are proportional and adapt to what is actually being used — fixed reservations waste bandwidth.

A fixed reservation of 10,000 Mbps for vSAN would hold 10 Gbps reserved even when no vSAN traffic is flowing (during off-hours, for example). That reserved bandwidth sits unused. High shares (100 vs VM's 50) means: when there IS contention, vSAN gets roughly 2× the proportion of VM traffic. When there is no contention, all traffic flows freely at full speed with no artificial limits. Shares are the right model for a dynamic workload like a vSphere cluster. Fixed reservations are for services that need absolute guarantees regardless of load — like the management traffic 100 Mbps reservation.

Port group design philosophy
The naming convention, VLAN assignment rules, and teaming policy decisions that apply to all 7 port groups — decided once, applied consistently
Port group naming convention
Why the format "PG-VLAN10-Management" — what does each part mean?
PG (object type) — VLAN10 (the VLAN ID) — Management (the traffic purpose).

This three-part format makes every port group self-documenting. When a VM administrator connects a VM to a port group, the name immediately tells them: (1) it is a port group (not a vDS or host), (2) it carries VLAN 10 traffic (they can cross-reference the VLAN plan from Guide 6), (3) it is for management traffic (not VM workloads). There is no ambiguity about what a VM connected to this port group will experience. The alternative — naming it "MGMT" or "Network1" — requires a separate lookup to understand what VLAN it carries and what traffic belongs there.
Teaming and failover policy per port group
Why are different teaming policies applied to different port groups?
Because different traffic types use different physical NIC pairs — and LACP LAGs must know which physical NICs to use.

Each LACP LAG is associated with specific uplinks. PG-VLAN10-Management uses Uplink1 and Uplink2 (vmnic0/1 — the management NIC pair). PG-VLAN20-vMotion and PG-VLAN50-OCP and PG-VLAN60-VM use Uplink3 and Uplink4 (vmnic2/3). PG-VLAN25-vSAN and PG-VLAN30-NFS use Uplink5 and Uplink6 (vmnic4/5). If a VM port group were accidentally configured to use all 6 uplinks, its traffic could end up on the vSAN NIC pair — consuming storage bandwidth with VM traffic. Explicit uplink assignment is a security and performance control, not just organisation.

VLAN type — "VLAN" not "VLAN Trunking" not "Private VLAN"
The VLAN type dropdown has four options: None, VLAN, VLAN Trunking, and Private VLAN. Which to choose and why?
Select "VLAN" for all NovaCrestBank port groups except those carrying VM trunks.

None: Sends untagged frames — correct only if the switch port is an access port. NovaCrestBank uses trunk ports on all ESXi-facing leaf switch ports. Untagged frames land on native VLAN 1 which has no routing. Never use "None" here.

VLAN (selected): Tags all traffic from this port group with the specified VLAN ID. This is what a trunk port expects. The leaf switch receives VLAN 10-tagged frames and routes them to the VLAN 10 SVI. This is correct.

VLAN Trunking: Passes multiple VLAN tags through to VMs — used when a VM itself is a router or firewall that needs to handle multiple VLANs. Not needed for ESXi VMkernel adapters or workload VMs.

Private VLAN: Isolates VMs from each other within the same VLAN using PVLAN. Not used in NovaCrestBank's design.

Why vMotion is on Uplink3/Uplink4 — not the management uplinks
vMotion and management are both infrastructure traffic. Why not put them on the same NIC pair?
Because vMotion generates enormous bandwidth bursts that would crowd out management traffic on a shared NIC.

A vMotion of a 64 GB RAM VM generates approximately 64 GB of network transfer in roughly 20 seconds — approaching 25 Gbps (the full line rate of a single 25G NIC). If vMotion and management shared vmnic0/1, a vMotion event would saturate the management NIC. vCenter loses reliable communication with the host. SSH sessions drop. If an administrator is making a configuration change during the vMotion, the session dies mid-command. By putting vMotion on its own dedicated NIC pair (vmnic2/3), vMotion can burst to full line rate without touching the management path. Management always has dedicated bandwidth regardless of vMotion activity.

MTU 9000 — the complete chain that must be consistent
The port group MTU is set to 9000. Is that sufficient for jumbo frames to work end-to-end?
No — MTU 9000 must be set at every single hop in the path.

Setting MTU 9000 on the vDS port group is one step in a five-step chain. The complete chain that must all be at MTU 9000:
1. ESXi VMkernel adapter (vmk2) — set when configuring the vSAN VMkernel in Phase 2
2. vDS port group (PG-VLAN25-vSAN) — set here
3. Leaf switch port channel (Po103, physical ESXi-facing port) — set in Cisco Guide 5 with mtu 9000
4. Leaf SVI (VLAN 25) — set in Cisco Guide 5 with mtu 9000 on interface vlan25
5. Spine interfaces (Eth1/1-3) — set in Cisco Guide 5 with mtu 9000

If any single hop is at 1500, oversized frames are silently dropped at that hop. The VMkernel sends a 9000-byte frame, it arrives at a 1500-byte interface, and the interface drops it without an ICMP "fragmentation needed" response (because the DF bit is set). vSAN degrades silently — slower writes, more retries — with no obvious error in the vCenter UI. The MTU verification test (vmkping -d -s 8972) is what catches this.

Why NFS uses the same uplinks as vSAN (Uplink5/6)
vSAN and NFS are both storage traffic. Why not put NFS on Uplink3/4 with vMotion to spread the load?
Because vSAN and NFS both require MTU 9000 end-to-end, and the uplinks carrying MTU 9000 frames must connect to switch ports configured for MTU 9000.

On the Cisco leaf switches, Uplink5 and Uplink6 (vmnic4/5) connect to port channels (Po103/Po106) whose member ports are configured at MTU 9000. The port channels for Uplink3/4 (Po102/Po105) are at standard MTU 1500 — they carry vMotion and VM traffic. If NFS were placed on Uplink3/4, its 9000-byte frames would hit 1500-byte switch ports and be dropped.

The MTU configuration on the Cisco switch and the uplink assignment in the vDS must be aligned. This is the exact design from Guide 5: storage NICs (vmnic4/5) connect to the storage LACP bond with MTU 9000; infrastructure NICs (vmnic2/3) connect to the VM/vMotion bond at MTU 1500.

Why OCP nodes share uplinks with vMotion and VM production
OpenShift generates significant east-west traffic (pod-to-pod OVN tunnels). Should it have dedicated NICs?
Not at the initial deployment scale — the Uplink3/4 pair (vmnic2/3) at 2×25G provides 50 Gbps aggregate capacity, which comfortably handles vMotion, OCP overlay traffic, and VM production traffic combined for a 9-node OCP cluster.

OCP's pod-to-pod traffic uses Geneve tunnels encapsulated in UDP. The tunnel overhead adds ~50 bytes per packet. At 1500-byte MTU the effective payload per packet is about 1430 bytes — a small efficiency loss. If this becomes a bottleneck (measurable as high CPU for OVN overlay processing or sustained link utilisation above 80%), the remediation is: (1) increase MTU to 9000 on the OCP port group and VLAN 50 SVIs (reduces fragmentation overhead), or (2) dedicate a fourth NIC pair specifically for OCP. Neither change requires redesigning the vDS — just modifying port group settings and potentially the LACP LAG assignments. The current design is deliberately flexible for future growth.

Why 128 ports on every port group
NovaCrestBank has a 5-host cluster and maybe 20 VMs initially. Why not set the port count lower — say 32?
Because the cost of an unused port is zero, but the cost of running out of ports mid-operation is significant.

vDS ports are not physical resources — they are software constructs that consume negligible memory. There is no performance or cost penalty for having 128 ports when only 20 are used. However, if you set 32 ports and later add a vMotion event that temporarily binds 40 ports, the operation fails with an error: "No free ports on distributed port group." Running out of ports causes VM power-on failures and vMotion failures — at the worst possible time (during a high-load period when you are migrating VMs). 128 is the VMware-recommended starting point for enterprise environments. You can always increase the port count later, but doing so requires a brief port group reconfiguration that generates a change-management event.

Why Windows infrastructure shares uplinks with management (Uplink1/2)
Domain Controllers are high-value targets. Should they have dedicated NICs separate from management traffic?
No — and the reasoning reveals an important principle: security isolation through VLANs is more effective than physical NIC separation for the Windows tier.

Giving the DCs dedicated NICs (vmnic6/7 — which do not exist on NovaCrestBank's hardware) would only add physical separation. The security control that matters is VLAN isolation — the fact that VLAN 70 traffic is separated from VLAN 60 (VM production) at the network layer, enforced by the Cisco leaf switches and the Palo Alto firewall. Physical NIC separation adds no additional security if both NICs are on the same trunk port and VLAN separation is already enforced by the switch.

Sharing Uplink1/2 with management means the management VMkernel and Windows VMs share the same physical NIC pair. This is intentional: management traffic is small (SSH sessions, API calls — rarely more than a few Mbps) and Windows VM traffic for 2 DCs is also small. There is no contention risk. The NIOC management traffic reservation (100 Mbps guaranteed) ensures the management VMkernel always gets through regardless of DC traffic. The right decision is: share the NIC pair, rely on VLAN isolation for security.

The management VMkernel migration on each host takes approximately 30 seconds. During those 30 seconds, vCenter loses the management connection to that host. This is expected and normal. The correct order of operations for management migration

Before migrating: confirm LACP LAG is configured (Section 21 must be done first)
The management VMkernel migration should happen AFTER the LACP LAG for Uplink1/Uplink2 is configured (Section 21) but BEFORE vmnic0 is removed from vSwitch0 (Section 22). Here is why: after migration, vmk0 is on the vDS port group. If vmnic0 is not yet assigned to a vDS uplink, vmk0 has no physical path and the host is immediately unreachable. The correct sequence: (1) Configure LACP LAG, (2) Add vmnic0 to vDS Uplink1 while it is still ALSO on vSwitch0 — brief dual membership, (3) Migrate vmk0 from vSwitch0 to vDS PG-VLAN10, (4) Remove vmnic0 from vSwitch0 (the old path is now gone, vDS is the only path). This guide presents them in teaching order (LACP in Section 21, NIC assignment in Section 22) but the execution order within each host is: LAG → NIC to vDS (keep on vSwitch0 too) → VMkernel migrate → remove from vSwitch0.

LACP mode Active on both sides — why not Passive?
LACP has two modes: Active (initiates negotiation) and Passive (waits for the other side to initiate). The Cisco switch is configured with "mode active". Should ESXi also be Active?
Yes — both sides Active is acceptable and is the recommended configuration for VMware vDS.

LACP requires that at least one side is Active. Two Passive sides never form a LAG — both wait forever for the other to initiate. Two Active sides both initiate, exchange LACP PDUs, and form the LAG — this is correct behaviour. With the Cisco switch at Active and ESXi at Active, the LAG negotiates immediately when the port is enabled. The result is the same as one Active and one Passive — the LAG forms. Active/Active is VMware's default recommendation for vDS LACP because it eliminates any risk of two-Passive no-formation.

The I→P transition — what it proves
Before Phase 1, the Cisco switch showed Eth1/1(I) — Individual. After Phase 1 it shows Eth1/1(P) — in port channel. What exactly changed?
LACP PDU (Protocol Data Unit) exchange began between ESXi and the Cisco switch — and both sides agreed to form the logical bond.

In the Individual state: the Cisco switch was sending LACP PDUs on Eth1/1 but ESXi (using the standard vSwitch) was not responding — the standard vSwitch does not speak LACP. So the Cisco switch left Eth1/1 in Individual mode: the port carries traffic but without LACP negotiation. There is no partner to coordinate with.

In the P (in port channel) state: ESXi's vDS LACP agent is now active on vmnic0. It sends LACP PDUs to the Cisco switch. The switch receives them, validates the system ID and port priority, and both sides agree to form Po101. The port transitions from Individual to P (active member of the port channel). The LAG is now a proper LACP bond — fault detection, load balancing, and flow hashing all work correctly. If vmnic0 fails or the cable is unplugged, the Cisco switch immediately detects the LACP partner has gone, withdraws the port from the channel, and generates a system log event. This level of fault detection is impossible in Individual mode.
