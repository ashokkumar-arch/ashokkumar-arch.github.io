Why a hardware HSM — not software key storage
vSAN encryption can store its keys in a software-based KMS or in a file. Why does NovaCrestBank require a dedicated hardware HSM?
Because the security of every byte stored on vSAN depends entirely on the security of the encryption key. If the key is stored in software — in a database, a configuration file, or even an in-memory vault — an attacker who compromises the server that holds the key can decrypt all vSAN data offline. A hardware HSM makes this physically impossible.

A Hardware Security Module is a dedicated tamper-resistant computing device that performs cryptographic operations and stores keys. Its defining security properties:

Keys never leave the HSM in plaintext. When vCenter needs to encrypt a vSAN key, it sends a request to the HSM. The HSM performs the encryption internally and returns the result. The plaintext key material never travels across the network or exists in server memory outside the HSM's secure boundary.

Physical tamper evidence and resistance. If someone tries to physically open the Thales Luna Network HSM 7, it detects the intrusion and automatically zeroises (erases) all key material. The data on vSAN becomes permanently unrecoverable. This is the correct response to physical theft — stealing the hardware gains nothing because the keys self-destruct.

Regulatory compliance. PCI-DSS v4.0 Requirement 3.7 mandates that encryption keys used for cardholder data protection be stored in a FIPS 140-3 Level 3 validated hardware cryptographic module. The Thales Luna Network HSM 7 is FIPS 140-3 Level 3 validated. A software vault is not. For a bank processing card transactions, the HSM is not optional.

HSM · Section 02
KMIP — how vCenter communicates with the HSM
Key Management Interoperability Protocol — the standard that allows vCenter to request key operations from the Thales Luna without vendor lock-in
What KMIP is and what operations vCenter performs through it
vCenter needs to store, retrieve, and use encryption keys. What does KMIP provide and how does the conversation between vCenter and the HSM actually work?
KMIP (Key Management Interoperability Protocol) is an OASIS standard protocol that defines how clients (like vCenter) request key management operations from a key manager (like the Thales Luna HSM). It runs over TLS on TCP port 5696.

The key operations vCenter performs via KMIP:

CREATE: When vSAN encryption is enabled, vCenter sends a CREATE request to the HSM asking it to generate a new Key Encryption Key (KEK). The HSM generates the key entirely inside its secure boundary, stores it, and returns a key identifier (a UUID) to vCenter. The actual key material never leaves the HSM.

GET: When an ESXi host needs to decrypt vSAN data (at host boot, after HA restart, during normal I/O), vCenter requests the KEK from the HSM by its UUID. The HSM returns the key (encrypted in transit via the KMIP TLS connection). vCenter distributes it to the ESXi host, which uses it to decrypt the vSAN Data Encryption Key, which in turn decrypts the actual data.

LOCATE and DESCRIBE: Key inventory operations — vCenter can list and describe keys in the HSM partition.

The TLS mutual authentication: The KMIP connection uses mutual TLS — both vCenter (as the client) and the HSM (as the server) present certificates and verify each other. This means: even if an attacker intercepts the KMIP traffic, they cannot inject fake key responses (the HSM's certificate was not issued by the attacker's CA) and cannot impersonate vCenter (they do not have vCenter's KMIP client certificate private key).
HSM · Section 03
Thales Luna Network HSM 7 architecture — appliance, partitions, and clients
Understanding the three layers of the Luna HSM before touching the device — appliance-level administration, partition-level key storage, and client-level access
🔒 Luna HSM Appliance (Hardware level)
The physical Thales Luna Network HSM 7 device. Administered by the Security Officer (SO) using the SO password. Contains the Hardware Security Module chip. FIPS 140-3 Level 3 validated. NovaCrestBank has two appliances (HSM-01 at 10.40.0.61, HSM-02 at 10.40.0.62) in HA mode.
↓ contains
🗝 Luna HSM Partition (Logical container)
A named logical container within the HSM that holds cryptographic keys. Think of it as a vault inside the HSM. NovaCrestBank creates one partition: NovaCrestBank-vCenter-KMS. The partition has its own Partition SO password and a separate Crypto Officer (CO) password used by clients to perform key operations.
↓ accessed by
💻 KMIP Client (vCenter)
The vCenter Server VM is registered as a KMIP client. It connects to the HSM partition over the network using NTLS (Network Trust Link Service) — a mutual TLS connection secured by certificates exchanged during the registration process. vCenter's client certificate is registered on the HSM; the HSM's server certificate is trusted by vCenter.
Why two HSM appliances — not one
A single Thales Luna HSM 7 costs tens of thousands of pounds. Why have two?
Because a single HSM is a single point of failure for all encrypted data. If the HSM goes offline — hardware failure, network outage, maintenance — every ESXi host loses access to its vSAN Key Encryption Key. vSAN becomes read-only (it cannot perform new writes that require key access). Virtual machines that need key material for new operations stall. For a banking production environment, this is unacceptable.

The Thales Luna HSM HA group replicates key material between both appliances synchronously. When vCenter makes a KMIP request, it connects to the HA group virtual address — the Luna client library automatically routes to whichever HSM is available. If HSM-01 fails, HSM-02 handles all KMIP requests without any intervention.

Both HSMs are in the OOB management network (VLAN 40, 10.40.0.61 and 10.40.0.62) — accessible from the management network but isolated from the production data VLANs.

HSM credential governance — two-person rule
The HSM Security Officer (SO) password is the master credential for the HSM appliance. If it is forgotten, there is no recovery — the HSM must be reset and all key material is destroyed (meaning all vSAN data becomes permanently unrecoverable). NovaCrestBank enforces a two-person rule for HSM SO access: the password is split — one half known to the CISO, one half known to the Head of Infrastructure. Neither can access the HSM alone. This is documented in the CyberArk entry with two separate vaults.

Initialise the HSM — set the Security Officer password and cloning domain
The most consequential operation on the HSM — initialisation is irreversible and establishes the security officer credential and the cloning domain used for HA synchronisation
Initialisation is irreversible — a zeroised HSM loses all key material
Running hsm init wipes the HSM to a known-clean state and sets the Security Officer (SO) password and cloning domain. If the HSM has previously been used and has existing key material, hsm init permanently destroys it. For NovaCrestBank's new HSM appliances this is not a concern — they are factory-fresh. The cloning domain is a shared secret between both HSM appliances that enables HA key synchronisation — both HSMs must use the same cloning domain.

HSM · Section 10
Establish NTLS trust — exchange certificates between HSM and vCenter
The mutual certificate exchange that creates the secure channel between the Luna client (vCenter) and both HSM appliances
What NTLS is and why mutual certificate exchange is required
What is NTLS and why does it require exchanging certificates in both directions?
NTLS (Network Trust Link Service) is Thales Luna's proprietary protocol for secure communication between the Luna client and the HSM. It uses mutual TLS — both sides must present certificates that the other side has previously trusted. This prevents both impersonation of the HSM (a rogue device cannot intercept key material) and impersonation of the client (an unauthorised host cannot request key operations).

The exchange: (1) The HSM has a server certificate generated during initialisation. The client must install this certificate in its trust store. (2) The client generates its own certificate. The HSM administrator must register this certificate on the HSM, allowing that specific client to connect. After this exchange, only the specific client whose certificate is registered on the HSM can open an NTLS connection — any other host attempting to connect is rejected at the TLS handshake stage, even if they know the partition password.

HSM · Section 17
Initial key generation and background re-encryption
What happens immediately after enabling encryption — the key hierarchy created by the HSM and the re-encryption progress
The vSAN encryption key hierarchy — DEK, KEK, and the HSM's role
vSAN creates "keys" — but what exactly gets generated and where does each piece live?
vSAN uses a two-layer key hierarchy: a Key Encryption Key (KEK) stored on the HSM, and a Data Encryption Key (DEK) stored encrypted on each ESXi host. Only the KEK lives on the HSM — the DEK is stored locally on ESXi but is useless without the KEK to decrypt it.

The exact hierarchy:

KEK (Key Encryption Key): Generated by the Thales Luna HSM when vSAN encryption is enabled. Stored entirely within the HSM's secure boundary. The HSM returns a key identifier (UUID) to vCenter — the actual key bytes never leave the HSM. This KEK is used to wrap (encrypt) the DEK.

DEK (Data Encryption Key): Generated by vCenter for each vSAN host/disk group. The DEK is what actually encrypts the data on the NVMe disks. The DEK is stored encrypted on the ESXi host's disk — encrypted using the KEK. When an ESXi host boots, it requests the KEK from vCenter (which gets it from the HSM), decrypts the DEK with the KEK, and uses the DEK to mount the vSAN disk group.

Why this two-layer approach: The DEK can be stored on the ESXi host because it is encrypted — it is useless without the KEK. The KEK never leaves the HSM. To compromise vSAN encrypted data, an attacker would need to: steal the physical NVMe disks from the servers AND compromise the HSM simultaneously — two independent physical security requirements that are extremely difficult to satisfy simultaneously.

HSM · Section 19
Key rotation — shallow rekey vs deep rekey
Periodic key rotation is a regulatory requirement and operational best practice — understanding the two types before they are needed
Shallow rekey vs deep rekey — when to use each
vSAN supports two types of key rotation: "shallow rekey" and "deep rekey." What is the difference and which should be used for routine rotation?
Shallow rekey replaces the KEK (Key Encryption Key on the HSM) and re-encrypts all DEKs with the new KEK — but does not re-encrypt the actual vSAN data. Deep rekey replaces both the KEK and DEKs and re-encrypts all data.

Shallow rekey: Used for routine periodic key rotation. The HSM generates a new KEK. All DEKs are wrapped with the new KEK. The old KEK is destroyed on the HSM. This takes seconds to minutes — no data movement required. The actual NVMe disk data is not touched — only the key wrapping layer changes. This satisfies PCI-DSS and GDPR key rotation requirements at minimal operational cost.

Deep rekey: Used when there is reason to believe the DEK itself may have been compromised (e.g., a disk was briefly accessible to an untrusted party). Generates new DEKs and re-encrypts all vSAN data. This takes hours and consumes significant I/O — identical to the initial encryption in Section 17. Use sparingly.

NovaCrestBank rotation schedule: Shallow rekey every 90 days (quarterly), aligned with PCI-DSS key rotation expectations. Deep rekey only if a security incident warrants it.

HSM · Section 20
HSM backup and key recovery procedures
The most critical operational procedure in the entire NovaCrestBank infrastructure — losing HSM key material without a backup means all vSAN data is permanently unrecoverable
Without HSM backup, a zeroised HSM means all encrypted data is permanently unrecoverable
If both HSM appliances are simultaneously destroyed (fire, flood, theft) or both are zeroised (tamper event, forgetting both SO passwords), the KEK is permanently lost. The encrypted vSAN data on the NVMe disks is theoretically still there — but without the KEK to decrypt the DEKs, it is computationally impossible to recover. This is the nuclear scenario that backup prevents. The HSM backup is the single most important backup in NovaCrestBank's infrastructure — more critical than any VM backup.
