Why OpenShift rather than vanilla Kubernetes
Kubernetes is open source and free. Why does NovaCrestBank pay for Red Hat OpenShift?
Because OpenShift is Kubernetes plus everything a production banking environment needs that vanilla Kubernetes does not include — and the total cost of building those capabilities from scratch exceeds the subscription cost many times over.

What OCP adds over vanilla Kubernetes that matters for NovaCrestBank:

Built-in security: Every OCP node runs Red Hat CoreOS (RHCOS), an immutable, container-optimised OS managed entirely by the Machine Config Operator. Nodes cannot be SSH'd into directly and their configurations cannot drift — every change goes through the MCO with audit trail. Vanilla Kubernetes has no opinion on the underlying OS.

Integrated certificate management: OCP has a cluster-wide certificate authority and certificate rotation. The API server, etcd, kubelet, and service mesh certificates are all managed and auto-rotated. In vanilla Kubernetes, you manage all of this manually.

Operator Framework: OCP is managed by operators — self-healing controllers that continuously reconcile actual state with desired state. The Cluster Version Operator manages OCP upgrades. The Machine Config Operator manages node configurations. The Monitoring Operator deploys and manages Prometheus and Grafana. These replace hundreds of hours of manual operational work per year.

Integrated monitoring and logging: Prometheus, Grafana, Alertmanager, and the Loki/OpenShift Logging stack are built-in and pre-configured. In vanilla Kubernetes, each of these is a separate installation and configuration project.

FCA/PRA alignment: Red Hat publishes detailed security compliance documentation for OCP aligned with CIS Kubernetes Benchmark, NIST 800-53, and PCI-DSS. Having a supported, documented security baseline reduces the audit burden significantly compared to documenting a custom vanilla Kubernetes hardening effort.

Why 3 master nodes — not 1 or 5
A single master would be simpler. Why require exactly 3?
Because etcd, the distributed key-value store at the heart of the Kubernetes control plane, requires a quorum (majority) to function. 3 nodes = quorum of 2, tolerating 1 failure. 1 node = single point of failure. 5 nodes = tolerates 2 failures but doubles cost with marginal benefit for NovaCrestBank's cluster size.

With 3 masters: if master-0 fails, master-1 and master-2 still have a quorum (2 out of 3). etcd continues serving reads and writes. The cluster keeps scheduling workloads. The API server continues accepting requests. HA restarts master-0 on another host, it rejoins etcd, quorum is restored.

With 1 master: if it fails, etcd has no quorum — no reads, no writes. The cluster stops accepting new pod placements. Running pods continue (they are on worker nodes, not the master) but cannot be rescheduled if they fail. This is not acceptable for a banking platform.

With 5 masters: tolerates 2 failures but requires 2 additional VMs and 2 additional vSphere anti-affinity rule placements. For NovaCrestBank's initial deployment with 5 ESXi hosts and a 9-node OCP cluster, 3 masters is the correct balance of resilience and resource efficiency.
OpenShift · Section 03
IPI vs UPI — why Installer-Provisioned Infrastructure on vSphere
The most fundamental deployment decision — understanding what IPI does and what it means for day-2 operations
IPI — Installer-Provisioned Infrastructure
What is IPI and what does the installer do automatically?
IPI means the openshift-install binary provisions all the infrastructure (VMs, DHCP reservations, DNS records, load balancer configuration) automatically as part of the installation process — you provide credentials and a configuration file, and the installer creates everything.

With IPI on vSphere, the installer: (1) connects to vCenter using the service account credentials in install-config.yaml, (2) creates a folder in the specified datacenter/cluster, (3) deploys 9 VMs (3 masters, 3 infra, 3 workers) from RHCOS OVA templates into the correct vSphere network port group, (4) configures each node with its ignition configuration (which bootstraps the RHCOS OS and joins the cluster), (5) monitors the cluster installation progress until all operators are available.

What IPI gives you after installation: Cluster API (CAPI) machine management — MachineSets that describe desired VM configurations. When you scale a MachineSet by increasing its replica count, OCP automatically provisions new VMs in vCenter to meet the desired state. When a node becomes unhealthy, the Machine Health Check controller can automatically replace it. The cluster manages its own infrastructure.
UPI vs IPI — why not UPI for NovaCrestBank
UPI (User-Provisioned Infrastructure) gives more control — why choose IPI?
Because IPI provides Machine API integration for automated node lifecycle management — a critical operational feature for a banking cluster that must scale and self-heal. UPI clusters do not have Machine API by default and require manual VM management for scaling and replacement.

With UPI: if a worker node VM fails, an administrator must manually create a new VM, boot it with the correct ignition config, and add it to the cluster. This is a manual process that takes 20–30 minutes and requires administrator intervention at potentially inconvenient times.

With IPI (Machine API): if a worker node VM fails, the Machine Health Check detects it, the Machine controller provisions a replacement VM in vCenter automatically, the node joins the cluster, and workloads reschedule onto it — without any administrator action.

UPI is appropriate when: the organisation has strict infrastructure provisioning workflows that cannot be automated, when the target platform doesn't support IPI, or when extremely tight control over every VM parameter is required. For vSphere with a standard vCenter, IPI is the correct choice.

The API VIP and Ingress VIP — what these addresses do
What is a VIP (Virtual IP) in the context of OCP, and why are two separate VIPs needed?
A VIP is a single IP address that routes to multiple backend nodes via a software load balancer — providing a stable entry point even when individual nodes fail or are replaced.

API VIP (10.50.0.200): The API server endpoint. Every interaction with the cluster — kubectl commands, the OCP web console, the installer, CI/CD pipelines — goes to this address. It is load-balanced across all 3 master nodes (each of which runs kube-apiserver). If master-0 fails, the API VIP still works via master-1 and master-2. DNS: api.ocp.novacrestbank.local → 10.50.0.200.

Ingress VIP (10.50.0.201): The ingress router endpoint. All HTTP/HTTPS traffic to applications running in OCP arrives at this address. The OCP Router (based on HAProxy) load-balances across the infra nodes that run the router pods. All application Routes (e.g., banking-portal.apps.ocp.novacrestbank.local) resolve to this single IP. DNS: *.apps.ocp.novacrestbank.local → 10.50.0.201 (wildcard DNS record).

With IPI on vSphere, OCP uses its built-in keepalived implementation to manage these VIPs — no external load balancer hardware is required. The VIPs are configured in install-config.yaml and managed automatically by OCP after installation.

What each platform.vsphere field maps to in vCenter
The install-config.yaml has vcenter, datacenter, defaultDatastore, folder, resourcePool, and network fields. Where does each of these come from in vCenter?
vcenter: The FQDN of vCenter — vcenter.novacrestbank.local. Must be resolvable from the installer host and match the TLS certificate subject. Do not use the IP address — certificate validation will fail.

OpenShift · Section 11
Networking section — cluster CIDR, service CIDR, and machine CIDR
The three network ranges in install-config.yaml — understanding why each must be chosen carefully to avoid conflicts with existing infrastructure
Choosing the three networking CIDRs — overlap is fatal
install-config.yaml has clusterNetwork (pod CIDR), serviceNetwork (service CIDR), and machineNetwork (node CIDR). What are the constraints on choosing these?
The three CIDRs must not overlap with each other or with any existing network in NovaCrestBank's infrastructure — otherwise pod-to-external-service routing breaks in hard-to-diagnose ways.

machineNetwork (10.50.0.0/24): Must match VLAN 50 exactly — this is the physical network the OCP nodes use. Non-negotiable: this is determined by the VLAN planning done in the Cisco guides.

clusterNetwork (10.128.0.0/14): The overlay network for pod-to-pod communication. 10.128.0.0/14 is OCP's default and does not conflict with any of NovaCrestBank's physical networks (10.10.x, 10.20.x, 10.25.x, 10.30.x, 10.40.x, 10.50.x, 10.60.x, 10.70.x). If any existing infrastructure used the 10.128-10.131 range, this would need changing.

serviceNetwork (172.30.0.0/16): Virtual service IPs — only exist within the OVN overlay. Nothing in NovaCrestBank's physical network uses 172.30.x.x. This OCP default is safe.

The most common installation mistake: using an RFC-1918 range for pod or service CIDRs that overlaps with the physical network. A pod trying to reach 10.50.0.100 (a vSphere host) but having an OVN route that intercepts 10.50.x.x as pod traffic — the connection goes nowhere. Always verify against the full IP address plan before installation.
OpenShift · Section 12
Control plane and compute replicas — node sizing decisions
Why the master and worker node sizes were chosen — and the one thing you must never do with control plane replicas
Why controlPlane replicas must be 3 — never change this to 2 or 4
The install-config.yaml says controlPlane.replicas: 3. Can this be set to 2 to save resources?
No — 3 is the only valid production value. 2 masters cannot form an etcd quorum (quorum requires majority: 2 out of 2 = 100% required, no failure tolerance). Setting replicas to 2 creates a cluster where a single master failure breaks the entire control plane permanently until the failed node is recovered.

etcd quorum formula: quorum = (n/2) + 1 (rounded down). For n=3: quorum = 2. For n=2: quorum = 2. With 2 nodes, you need both nodes available for quorum — any failure breaks etcd. With 3 nodes, you only need 2 — one node can fail and the cluster continues.

4 masters: technically valid but wasteful and asymmetric. etcd with 4 nodes still only tolerates 1 failure (quorum = 3, any 2 failures break it). 5 masters would tolerate 2 failures. For NovaCrestBank, 3 masters is the correct balance of resilience and resource usage.

Also: do not change controlPlane.replicas after installation. Adding or removing master nodes post-installation requires careful etcd member management that is operationally complex. The replica count set here is the value the cluster is built with — scale workers instead.

OpenShift · Section 14
What the bootstrap node does — the invisible temporary VM
The bootstrap VM is created automatically, does its job, and disappears — understanding it prevents confusion when watching installation progress
The bootstrap VM — what it does and why it exists
During installation, a 10th VM called "bootstrap" appears in vCenter. What is it for and why is it needed?
The bootstrap VM is a temporary single-purpose VM that performs a chicken-and-egg task: it starts a temporary OCP API server that the master nodes use to bootstrap themselves and form the cluster. Once the masters have formed a functioning cluster, they no longer need the bootstrap VM and it is destroyed.

The bootstrap sequence:
1. Installer creates the bootstrap VM (from RHCOS) and 3 master VMs simultaneously in vCenter
2. Bootstrap VM boots RHCOS and starts a temporary Kubernetes API server
3. Master VMs boot RHCOS and connect to the bootstrap API server for their initial cluster configuration
4. Masters receive their etcd configuration from the bootstrap and form the etcd cluster among themselves
5. Masters' own API server becomes functional — they have a working 3-node control plane
6. Bootstrap VM transfers control to the 3-node cluster and shuts itself down
7. Installer detects the bootstrap is complete ("It is now safe to remove the bootstrap resources") and destroys the bootstrap VM

Why this complexity: RHCOS nodes cannot bootstrap themselves in isolation — they need an existing API server to get their initial configuration. The bootstrap VM provides that initial API server as a self-contained temporary resource. This design means the 3 master nodes can be identical — none of them is the "first master" that has special configuration. The bootstrap VM handles the first-node bootstrapping problem.

OpenShift · Section 24
Create infrastructure nodes — move router and registry off workers
Infra nodes are worker nodes labelled to run platform workloads — they are excluded from subscription counting and dedicated to OCP infrastructure components
Why infrastructure nodes matter for subscription cost
Workers and infra nodes are both Kubernetes nodes. Why distinguish them?
Because OCP subscription pricing counts worker nodes — not infrastructure nodes. Running the OCP Router, image registry, and monitoring stack on worker nodes consumes subscription entitlements for pure infrastructure overhead. Moving these workloads to dedicated infrastructure nodes means NovaCrestBank's 3 worker subscriptions are fully available for banking application workloads, not OCP's own components.

Infrastructure nodes are created by adding a specific label to existing worker nodes and configuring tolerations in the infrastructure workloads' scheduler settings. The installer creates all initial nodes as workers — the infra designation is applied post-install through MachineSet configuration.
