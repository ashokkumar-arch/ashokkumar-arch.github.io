01 — Why the fabric must be configured first — and what happens if you skip it

The temptation in any infrastructure project is to start with the most visible component — rack the servers, boot them, and figure out the network as you go. That approach works in a greenfield environment with no existing production traffic. NovaCrestBank is not that environment. The leaf switches that will carry VLAN 110 bare-metal node traffic also carry VLAN 10 ESXi management traffic, VLAN 50 OCP node traffic, and VLAN 100/101 NSX TEP traffic — all carrying live production workloads.

The rule is: the network must be ready before a single server is powered on. Here is exactly what goes wrong when that order is violated.

What breaks if you power on servers before fabric is configured
The server's NIC sends DHCP discovery packets on VLAN 110. The leaf switch port is either unconfigured (native VLAN 1 — untagged) or has VLAN 110 blocked. The DHCP packet never reaches DC01. The server gets no IP. The Assisted Installer agent cannot register with the cluster. The node appears in the iLO console but never shows up in the Assisted Installer UI. Troubleshooting this takes an hour of confusion when the real fix is a two-minute switch configuration.
If the port-channel is configured but the vPC peer-link does not have VLAN 110 allowed, the two leaf switches disagree about VLAN 110 membership. vPC consistency check fails. The port-channel flaps every 30 seconds. The server sees an intermittent link and the DHCP discovery is never reliably transmitted.
If MTU 9000 is configured on the ODF replication VLAN 120 SVI but not on the physical port-channel interface, all frames larger than 1500 bytes are silently dropped. ODF will appear to install successfully but Ceph OSD replication will fail with inexplicable I/O errors. The symptom — slow storage — points nowhere near the root cause — MTU mismatch on a trunk port.

None of the above problems is difficult to fix. All of them waste time that could be avoided by doing the fabric first. This guide configures everything the five bare-metal DL385 nodes need before a single server is powered on for provisioning.

The additive principle — never modify existing configuration

Every command in this guide adds to the existing switch configuration — it never modifies or removes anything. The VLANs, SVIs, and port-channels for the existing VCF environment (VLANs 10, 20, 25, 40, 50, 70, 100–103) are not touched. If you are ever uncertain whether a command modifies existing configuration, use show run | include <keyword> to see what is already there before changing anything. When in doubt, read before you write.

Why the spines require zero configuration changes

This is worth understanding explicitly. The Cisco leaf-spine design used in NovaCrestBank is a pure Layer 3 fabric — the spines do not participate in any VLAN. VLANs exist only at the leaf layer. The spines learn routes via BGP from the leaves. When leaf-01 adds a BGP network 10.110.0.0/24 statement, the spine automatically learns this prefix via the existing BGP session. No spine configuration is touched. This is one of the core operational advantages of a leaf-spine architecture — adding a new subnet is a leaf-only operation.

03 — VLAN design — why four new VLANs and not one

Before configuring anything, understand why four VLANs are needed instead of one. The Classic series (Guide 6) established the framework: every VLAN must exist for a specific reason. Apply that framework here.

Why four VLANs — could one VLAN serve all bare-metal node traffic?

Technically, you could put all bare-metal traffic — node management, ODF replication, and VM workloads — on a single VLAN. Every device would be reachable. Ceph would replicate. VMs would get IPs. It would work in a lab. But it creates three production problems that a bank cannot accept.

Performance problem: ODF replication during a Ceph PG rebalance event (triggered by a node failure or deliberate rebalancing) can generate sustained 25 Gbps of storage traffic for 10–30 minutes. On a single VLAN, this saturates the same NIC that carries node management traffic — SSH sessions to troubleshoot the rebalancing event itself become unreliable at the exact moment you need them most. With VLAN 120 isolated on a dedicated NIC pair, ODF replication and node management never compete.

Security problem: VM workloads on VLAN 130 are the systems being migrated from the hypervisor environment — they include banking application VMs that may carry cardholder data. If these VMs share a VLAN with the bare-metal node network (VLAN 110), a compromised VM has direct Layer 2 access to the OCP cluster node IPs — bypassing any routing-layer firewall inspection. Separate VLANs force all VM-to-node traffic through the SVI and the NSX gateway firewall.

Compliance problem: PCI-DSS Requirement 1.3.2 requires explicit documentation of all network security controls with business justification. A single-VLAN design cannot satisfy this — there is no demonstrable segmentation between cardholder-data-carrying VMs and infrastructure management traffic. Four VLANs with documented purpose and distinct firewall policies satisfy this requirement cleanly.

MTU 9000 applies only to VLAN 120 — here is why

VLAN 120 (ODF replication) requires MTU 9000 — jumbo frames — because Ceph OSD replication uses large I/O block sizes that are fragmented inefficiently at 1500-byte MTU. VLANs 110, 130, and 140 use standard 1500-byte MTU. RHCOS nodes communicate with the OCP API server using standard TCP — 1500-byte MTU is fine. VM workloads use standard Ethernet frames. The MTU configuration on each interface must match the traffic type — configuring jumbo frames everywhere "just in case" is not correct practice and can mask misconfiguration.

04 — VLAN 110 — bare-metal node network

110
New · OpenShift Virtualization
bm-node-network · 10.110.0.0/24 · Gateway 10.110.0.3 (HSRP) · MTU 1500
The primary network for the five HP ProLiant DL385 Gen11 bare-metal nodes. Every OCP cluster communication packet, Assisted Installer agent registration call, SSH management session, and OVN-Kubernetes GENEVE tunnel between bare-metal nodes traverses this VLAN.

Why this VLAN must exist — and why it cannot reuse VLAN 50

The existing OCP nodes (master-0/1/2, worker-0) are on VLAN 50 — but they reach VLAN 50 through the NSX overlay: the ESXi hypervisor encapsulates their traffic in GENEVE tunnels on VLAN 100 (the Host TEP VLAN), and NSX delivers it to the logical segment seg-ocp-nodes. The VMs themselves think they are on 10.50.0.0/24, but the physical fabric never sees a VLAN 50 tagged frame from those VMs — it sees VLAN 100 TEP frames instead.

Bare-metal nodes running RHCOS do not have ESXi underneath them. There is no NSX host TEP, no GENEVE encapsulation. The RHCOS node's NIC connects directly to the leaf switch. There is no overlay. RHCOS needs a real, physical 802.1Q VLAN — not a logical NSX segment.

Could we use VLAN 50 directly on the leaf switch for bare-metal nodes? In principle yes — both ESXi-hosted OCP nodes and bare-metal nodes would be on 10.50.0.0/24. But this would mean VLAN 50 frames from bare-metal nodes appear on the Cisco fabric, while VLAN 50 traffic from ESXi-hosted nodes is carried inside GENEVE on VLAN 100. Mixed traffic types on the same logical subnet carried by physically different VLANs creates confusion and makes troubleshooting ambiguous. VLAN 110 is a clean, dedicated physical VLAN for bare-metal nodes — no ambiguity, no mixing.

DHCP relay — why VLAN 110 needs it

During the Assisted Installer discovery phase (VIRT-02), each bare-metal node boots from the agent ISO and immediately sends a DHCP discovery broadcast on VLAN 110 to get a temporary IP address. The DHCP server is DC01 at 10.70.0.10 — it is on a different subnet (VLAN 70, via NSX overlay at 10.70.0.0/24).

A DHCP discovery packet is a Layer 2 broadcast. Broadcasts do not cross router boundaries. Without a DHCP relay agent on the leaf switch, DC01 never receives the discovery packet. The bare-metal node gets no IP. The Assisted Installer agent cannot register. The node is invisible to the provisioning system.

The DHCP relay agent (ip helper-address on the SVI) converts the Layer 2 broadcast into a unicast UDP packet addressed to DC01's IP. The leaf switch — not the server — is responsible for this conversion. This is why ip helper-address 10.70.0.10 is mandatory on the VLAN 110 SVI.

05 — VLAN 120 — ODF replication

120
New · OpenShift Virtualization
odf-replication · 10.120.0.0/24 · No gateway needed · MTU 9000 (jumbo frames mandatory)
Dedicated network for Ceph OSD-to-OSD replication between bare-metal nodes. Isolated from all other traffic. Carries no pod, VM, or management traffic — only Ceph internal communication.

Why ODF replication is identical in principle to vSAN (VLAN 25)

In the Classic series, Guide 6 Section 06 explains why vSAN needs VLAN 25 separate from ESXi management (VLAN 10): a vSAN disk rebuild saturates the replication NIC and must never compete with management traffic. Exactly the same reasoning applies here.

Ceph — the storage engine behind ODF — replicates every write to multiple OSD nodes simultaneously. In a replicated storage system with replication factor 3, every 1 GiB written by a VM results in approximately 2 GiB of replication traffic between OSD nodes (the original write goes to the primary OSD, which then replicates to two replica OSDs). Under heavy VM write workload — for example, a banking batch process writing transaction logs — this replication traffic can saturate a 25G NIC for sustained periods.

If this replication traffic shares VLAN 110 (the node network), it competes with Kubernetes pod traffic, OCP API calls, and SSH management sessions — all of which use the same NIC. The result is unpredictable: sometimes OCP API calls time out, sometimes SSH sessions stall, sometimes the kubelet misses its liveness probe and a pod is unnecessarily restarted. All of these symptoms look like application problems but are actually caused by a network design choice.

VLAN 120 on a dedicated NIC pair (separate from the VLAN 110 NIC pair) ensures Ceph replication has its own physical path. Replication saturation cannot affect node management. This is the same principle as vSAN on dedicated NICs — but for ODF.

Why VLAN 120 has no gateway (and why that is correct)

Every other new VLAN (110, 130, 140) has a gateway IP on the leaf SVI — traffic is routed between these VLANs and the rest of the network. VLAN 120 has no gateway. Why?

Ceph OSD replication is purely internal — an OSD on bm-virt-01 replicates to an OSD on bm-virt-02 and bm-virt-03. All five bare-metal nodes are in the same /24 subnet (10.120.0.0/24) and can reach each other at Layer 2 without routing. No traffic on VLAN 120 ever needs to leave the 10.120.0.0/24 subnet. Creating a gateway SVI would add an unnecessary routing hop and would advertise 10.120.0.0/24 into BGP — making a purely internal storage network visible to the broader routing domain. That is both unnecessary and a minor security concern (the ODF replication network should not be reachable from outside the bare-metal nodes).

The leaf switch needs to carry VLAN 120 on its trunk ports to the bare-metal nodes — but it does not need an SVI for it.

06 — VLAN 130 — VM workload primary

130
New · OpenShift Virtualization
vm-workload-primary · 10.130.0.0/24 · Gateway 10.130.0.3 (HSRP) · MTU 1500
The network for VMs migrated from the hypervisor environment. Every migrated banking application VM gets a real, routable IP on this subnet — reachable by name from anywhere in the data centre. This is where DC01, monitoring agents, and the Palo Alto firewall see migrated VMs as network entities.

The critical difference between VLAN 130 and a pod overlay network

When a container workload runs in OCP, it gets an IP in the pod overlay network (10.128.0.0/14 — the OVN-Kubernetes overlay). This IP is not directly reachable from outside the cluster. Other systems reach the container via a Service, a Route, or an Ingress — not directly via the pod IP. This is correct and intentional for containers: the overlay provides network isolation, and Services provide the stable endpoint.

Virtual machines have fundamentally different requirements. A banking application VM — for example, a Windows Server running a payment processing service — has been reachable at a specific IP for years. Other systems have that IP in their configuration files, firewall rules, and DNS records. When that VM is migrated from the hypervisor environment to OpenShift Virtualization, its network presence must be preserved: same VLAN, same subnet, same IP (or at minimum a new IP in the same subnet with an updated DNS record). An overlay IP in 10.128.0.0/14 is not a substitute — it is unreachable from the rest of the data centre.

VLAN 130 provides what VMs need: a real, physical VLAN with a routable subnet, reachable from anywhere in the data centre that has a route to 10.130.0.0/24. The migrated VM gets an IP from this subnet, a DNS A record on DC01, and network presence identical to what it had on VMware. No other system in the data centre needs to know that the VM is now running on KVM instead of ESXi.

DHCP relay — VMs need IPs during discovery too

During VM creation (VIRT-04 and VIRT-05), newly created VMs can use DHCP to get a temporary IP from the 10.130.0.0/24 range before a static IP is assigned via cloud-init. The DHCP relay on the VLAN 130 SVI (ip helper-address 10.70.0.10) serves the same purpose as on VLAN 110 — DC01 is on a different subnet and DHCP broadcasts need to be relayed.

07 — VLAN 140 — VM workload DMZ

140
New · OpenShift Virtualization
vm-workload-dmz · 10.140.0.0/24 · Gateway 10.140.0.3 (HSRP) · MTU 1500
Secondary VM workload network for VMs that require stricter separation from the primary VM workload — for example, VMs that communicate with external partners, VMs hosting pre-production or testing environments, or VMs designated as DMZ-adjacent by the security team.

Why VLAN 140 exists — the compliance argument

Not every migrated VM belongs in the same network zone. A VM running an internal payment processing service (talks to the core banking system, writes to the Oracle database, never touches the internet) and a VM running an API gateway that accepts external partner connections have fundamentally different security profiles. Placing them on the same subnet creates a flat network within the VM workload tier — exactly the problem that separate VLANs exist to solve.

VLAN 140 provides a second isolation zone. The NSX gateway firewall (Model A) or Kubernetes NetworkPolicy (Model B) can enforce strict rules between VLAN 130 and VLAN 140 — for example, a VLAN 140 API gateway VM can call services in VLAN 130 on specific ports, but VLAN 130 VMs cannot initiate connections back to VLAN 140. This enforces a unidirectional trust model between the zones.

PCI-DSS Requirement 1.3.1 requires that inbound traffic to the cardholder data environment be restricted to only what is necessary. If cardholder-data VMs are in VLAN 130 and internet-facing VMs are in VLAN 140, the firewall rule between them is the PCI compliance control. Without the VLAN separation, there is no firewall boundary between these two categories — and no way to satisfy the requirement.

Step 1 — Add new VLANs to the VLAN database

The first step is always to create the VLANs in the database before referencing them anywhere. NX-OS will reject any interface configuration that references a VLAN that does not exist in the database. This is different from IOS — in IOS, configuring a switchport for a VLAN implicitly creates it. NX-OS does not. Attempting to add VLAN 110 to a trunk before creating it with the vlan 110 command results in the trunk port rejecting the VLAN silently — frames tagged with VLAN 110 are dropped, not an error message.

Step 2 — Create SVIs for the three routed VLANs

An SVI (Switched Virtual Interface) is a Layer 3 interface associated with a VLAN. Creating a SVI on the leaf switch makes the leaf the default router for devices in that VLAN. Without an SVI, VLAN 110 is a Layer 2 island — bare-metal nodes can reach each other within the VLAN but cannot reach anything outside it (not DC01 for DNS, not the OCP API VIP, not the jump host).

VLAN 120 (ODF replication) does NOT get an SVI — as explained in Section 05, Ceph replication is entirely within the /24 subnet and needs no routing. All other VLANs get SVIs.

The HSRP (Hot Standby Router Protocol) virtual IP is the gateway address that bare-metal nodes and VMs use as their default route. HSRP allows leaf-01 and leaf-02 to present a single shared IP (10.110.0.3, 10.130.0.3, 10.140.0.3) to devices in each VLAN — if leaf-01 fails, leaf-02 takes over the gateway function within 3 seconds without requiring any change on the connected devices.

VLAN 120 does not have an SVI, but the physical trunk ports that carry it must support 9000-byte frames. MTU 9000 is configured on the port-channel interfaces in Section 13. 09 — leaf-02 full configuration

leaf-02 is the vPC peer of leaf-01. Its configuration is nearly identical — the difference is in IP addresses (.2 instead of .1) and HSRP priority (100 instead of 110, making leaf-01 the HSRP active and leaf-02 the standby for all new VLANs). The VLAN database, port-channel configuration, and BGP additions are identical.

Why the IP and HSRP differences matter

leaf-01 uses .1 IPs (10.110.0.1, 10.130.0.1, 10.140.0.1) and HSRP priority 110. leaf-02 uses .2 IPs (10.110.0.2, 10.130.0.2, 10.140.0.2) and HSRP priority 100. The shared HSRP virtual IP (.3) is the same on both switches. This means: under normal conditions, leaf-01 is the active gateway — all routed traffic exits through leaf-01. If leaf-01 fails, HSRP detects the failure within 3 seconds (hello timer × 3) and leaf-02 takes over the .3 VIP automatically. Connected devices never change their gateway — 10.110.0.3 remains valid regardless of which physical switch is active.

11 — vPC peer-link — adding new VLANs to the allowed list

The vPC peer-link (port-channel Po1 on both leaf-01 and leaf-02) is the inter-switch link that carries all Layer 2 traffic between the two leaf switches. For a VLAN to traverse the peer-link — which it must for vPC to work correctly — that VLAN must be in the peer-link's allowed VLAN list. When a new VLAN is added to server port-channels but forgotten on the peer-link, the vPC consistency check fails, the port-channel is marked inconsistent, and frames are dropped.

Always add new VLANs to the peer-link before the server port-channels

If a VLAN is in a server port-channel's allowed list but not in the peer-link's allowed list, NX-OS will report a vPC consistency error and the port-channel will not forward traffic for that VLAN. The error appears in show vpc consistency-parameters global — but only after you have already configured the server port-channels. It is cleaner to add VLANs to the peer-link first, then configure server port-channels.

12 — BGP — advertising the new subnets to the fabric

The new subnets (10.110.0.0/24, 10.120.0.0/24, 10.130.0.0/24, 10.140.0.0/24) must be advertised via BGP so that the rest of the network — the spines, the NSX T0 gateway, the Palo Alto NGFW — can route traffic to these subnets. Without BGP advertisements, a bare-metal node at 10.110.0.11 can reach its leaf SVI gateway (10.110.0.1) but nothing beyond it. The OCP API server at 10.50.0.200 cannot route back to 10.110.0.11 — the return path does not exist.

How BGP route advertisement works in a leaf-spine fabric

In NovaCrestBank's leaf-spine design, each leaf switch runs BGP and advertises the local subnets it owns to its BGP peers — the two spine switches. The spine switches learn these routes and re-advertise them to all other leaf switches and to any external BGP peers (in this case, the NSX T0 gateway). When bm-virt-01 at 10.110.0.11 sends a packet to dc01 at 10.70.0.10, the path is: bm-virt-01 → leaf-01/02 SVI (10.110.0.1/2) → spine (BGP routes 10.110.0.0/24 via leaf-01/02 and 10.70.0.0/24 via NSX T0) → NSX T0 → seg-infra-services → DC01. The BGP advertisement is what makes the spine know that 10.110.0.0/24 is reachable via leaf-01/02.

VLAN 120 is the ODF Ceph replication network. It is a private, internal network — Ceph OSDs communicate with each other on this subnet, and nothing else. 13 — Port-channels for bare-metal servers — one per server

Each HP ProLiant DL385 Gen11 connects to both leaf-01 and leaf-02 via a vPC port-channel — the same dual-homing pattern used for the existing ESXi servers. vPC port-channels provide two benefits: 50 Gbps aggregate bandwidth (2 × 25G active links via LACP) and full redundancy (if either leaf fails, the port-channel continues on the surviving leaf).

What LACP is and why it is used here

LACP (Link Aggregation Control Protocol, IEEE 802.3ad) is the protocol that bundles multiple physical NIC ports into a single logical link. Without LACP, the switch sees four individual 25G ports from bm-virt-01 — it does not know they belong to the same server and would need STP to prevent loops. With LACP, the switch treats all four ports as one logical port-channel — no STP involvement, no blocked ports, full bandwidth utilisation.

In NX-OS, LACP is configured with channel-group <number> mode active on each physical port. "Active" means the switch sends LACP PDUs (Protocol Data Units) to negotiate the bundle. The server's NIC (Mellanox ConnectX-7 on the DL385) must also be configured for LACP — this is done by the NMState configuration in VIRT-02 when RHCOS is installed.

Port-channel numbering convention

Existing ESXi server port-channels use Po101–Po109 (one per ESXi host). Bare-metal virtualization nodes use Po211–Po215 (one per DL385 node). The gap between 109 and 211 is intentional — it makes it immediately clear from a show vpc output which port-channels belong to which environment. Any network engineer reviewing the switch configuration six months from now can identify the environment from the port-channel number without reading descriptions.

This section requires physical cabling to be complete first

Port-channel configuration on the switch does not depend on the server being cabled — you can pre-configure the port-channel, and it will come up as "waiting for the connected device" in show vpc. However, you must know which physical switch ports each server will connect to before configuring the port-channel. The port assignments below assume: Rack 3 servers connecting to Ethernet1/25–Ethernet1/34 on each leaf (the next available ports after existing ESXi connections). 14 — DHCP relay — understanding what was configured and verifying it works

The ip helper-address 10.70.0.10 command was applied to the VLAN 110 and VLAN 130/140 SVIs in Sections 08 and 09. This section explains what that command does at the packet level — so that when DHCP fails during bare-metal provisioning (and at some point it will), you know exactly where to look.

What happens when a bare-metal node sends a DHCP request

When bm-virt-01 boots from the Assisted Installer ISO for the first time, RHCOS sends a DHCP DISCOVER packet. Let us trace exactly what happens at every network hop.

  1. The DHCP DISCOVER packet leaves bm-virt-01's NIC tagged with VLAN 110. The destination MAC address is ff:ff:ff:ff:ff:ff (broadcast). The source IP is 0.0.0.0 (the node has no IP yet). The destination IP is 255.255.255.255 (limited broadcast).
  2. The packet arrives at leaf-01 on Po211 (the LACP port-channel for bm-virt-01). The switch receives the frame, strips the VLAN tag, and processes it in VLAN 110. The destination is a broadcast — the switch floods it to all VLAN 110 ports. This includes the VLAN 110 SVI (Vlan110 interface on the switch itself).
  3. The Vlan110 SVI receives the DHCP DISCOVER. The switch's IP routing stack checks: does this interface have an ip helper-address configured? Yes — 10.70.0.10. The switch converts the broadcast DHCP DISCOVER into a unicast DHCP packet, changes the destination from 255.255.255.255 to 10.70.0.10 (DC01), and adds a "giaddr" (gateway IP address) field set to 10.110.0.1 (the SVI's IP). This tells DC01 which scope to use for the DHCP lease.
  4. The relayed packet is routed to DC01 at 10.70.0.10. The routing table on leaf-01 shows 10.70.0.0/24 is reachable via the NSX T0 gateway (learned via BGP). The packet is forwarded to the T0, which routes it to the seg-infra-services segment, which delivers it to DC01.
  5. DC01 receives the DHCP DISCOVER with giaddr=10.110.0.1. It identifies the VLAN110-BM-Discovery scope (which covers 10.110.0.0/24) because giaddr falls within that scope's network. DC01 sends a DHCP OFFER back to the relay agent (leaf-01 SVI at 10.110.0.1).
  6. leaf-01 receives the DHCP OFFER from DC01. It converts it back to a Layer 2 frame and broadcasts it on VLAN 110. bm-virt-01 receives the OFFER, accepts it with a DHCP REQUEST, and leaf-01 relays the REQUEST back to DC01. DC01 sends DHCP ACK. bm-virt-01 now has an IP address.

15 — MTU 9000 end-to-end verification

MTU mismatches are the most common and most difficult-to-diagnose problem in storage networks. The symptom — poor ODF performance, Ceph OSD heartbeat timeouts, seemingly random I/O errors — looks like a software problem but is caused by a network configuration oversight. The verification in this section must be completed before VIRT-03 (ODF installation) begins.

What end-to-end MTU means

For a 9000-byte frame to travel from bm-virt-01's ODF replication interface (on VLAN 120) to bm-virt-02's ODF replication interface, every device and every interface in the path must support 9000-byte frames:

  1. bm-virt-01's RHCOS NIC bond sub-interface (bond0.120) — MTU 9000
  2. The LACP port-channel on leaf-01 (Po211) — MTU 9216
  3. The vPC peer-link between leaf-01 and leaf-02 (Po1) — MTU 9216
  4. The LACP port-channel on leaf-02 (Po212) — MTU 9216
  5. bm-virt-02's RHCOS NIC bond sub-interface (bond0.120) — MTU 9000

The switch uses MTU 9216 rather than 9000 because switch interfaces need headroom for encapsulation headers (VLAN tag = 4 bytes, Ethernet header = 18 bytes). Setting switch interfaces to 9216 guarantees 9000-byte frames from servers are never fragmented in the switch.

MTU verification with an actual test frame — do this after servers are cabled

After bare-metal nodes are provisioned in VIRT-02, verify MTU end-to-end with a ping test between nodes on VLAN 120. From bm-virt-01's ODF replication interface, run: ping -M do -s 8972 10.120.0.12. The -M do flag sets the "Don't Fragment" bit. The size 8972 + 28 bytes (IP header + ICMP header) = 9000 bytes total. If this ping succeeds, MTU 9000 is confirmed end-to-end. If it fails with "Frag needed" or simply times out, there is a MTU mismatch somewhere in the path.
