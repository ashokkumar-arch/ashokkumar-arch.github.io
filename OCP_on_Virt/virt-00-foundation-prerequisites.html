00 — Why this guide exists

Every guide in the OpenShift Virtualization series references decisions, IP addresses, VLANs, and infrastructure that are established here. Before reading VIRT-01, this document must be understood in full. The most common reason complex deployments fail — or produce untraceable problems — is that the planning phase was skipped or compressed. This guide is that planning phase, made explicit.

This guide also establishes all prerequisites — not just lists them. Each prerequisite is either verified to already exist, or built here step by step, before the guide that depends on it references it. No VIRT guide will tell you to run a command against infrastructure that has not already been built and verified in this guide or a prior one.

How to use this guide

Read it fully, in order, before starting VIRT-01. Use it as a reference document throughout the series — every IP address, every VLAN ID, every FQDN used in VIRT-01 through VIRT-05 is defined here. When something does not work in a later guide, the first check is always: does the baseline documented here match the actual environment state?

02 — Why OpenShift Virtualization — the business case

Every architectural decision has a business rationale. Understanding the rationale ensures that the technical choices made in VIRT-01 through VIRT-05 serve the actual business objective — not just a technology preference.

The platform consolidation argument

NovaCrestBank currently operates two distinct compute platforms: a hypervisor platform running virtualised workloads, and OpenShift running containerised applications. Each platform has its own operational tooling, its own licensing costs, its own upgrade cycles, and its own team expertise requirements. Running two platforms in parallel means duplicating effort and cost across both dimensions.

OpenShift Virtualization introduces the ability to run virtual machines as native Kubernetes objects on the same OCP cluster that already runs containerised workloads. A VM in OpenShift Virtualization is not a second-class citizen — it is managed by the same Argo CD that manages application deployments, monitored by the same Prometheus stack, protected by the same cert-manager PKI, and governed by the same RBAC that controls access to everything else on the cluster.

The consolidation argument is straightforward: if virtual machines can be managed on the same platform as containers, the organisation operates one platform instead of two. One operational stack. One monitoring system. One GitOps workflow. One set of skills to develop and maintain.

The cost and TCO dimension

Hypervisor licensing is a significant line item in any enterprise IT budget. Organisations that rely on dedicated hypervisor platforms pay licensing costs that scale with the number of physical hosts, sockets, or virtual machines — depending on the vendor's model. As the virtualised workload footprint grows, so does the licensing cost.

OpenShift Virtualization runs on RHCOS (Red Hat CoreOS) bare-metal nodes. There is no separate hypervisor license. The KVM/QEMU stack that powers the virtualisation is part of the Linux kernel — it is not a separately licensed component. For organisations that already hold a Red Hat OpenShift subscription, adding OpenShift Virtualization does not add a new license category — it extends the existing subscription's capabilities.

The Total Cost of Ownership (TCO) reduction comes from two directions simultaneously: lower licensing cost for the virtualisation layer, and lower operational cost from consolidating two platforms into one. For NovaCrestBank's finance and technology leadership, this is a compelling argument that aligns technology decisions with budget objectives.

The strategic exit path

Platform consolidation is not a one-time event — it is a direction. NovaCrestBank's VIRT series is structured around a phased migration: existing VMs are migrated to OpenShift Virtualization over time (VIRT-03), the operational model converges on a single platform (VIRT-04), and the legacy hypervisor infrastructure is eventually decommissioned (VIRT-05). At the end of that journey, NovaCrestBank operates a single platform — OpenShift — for both containerised and virtualised workloads, on bare-metal RHCOS nodes that it owns and controls.

What this means for the architect

Every technical decision in the VIRT series must serve this business objective. When VIRT-02 documents two networking models side by side (Model A for the transition period, Model B for the post-consolidation state), it is because the architect must plan for both — the transition model cannot create technical debt that prevents the final state from being reached. When VIRT-05 documents the full decommission procedure, it is because the exit path must be planned from the beginning, not improvised at the end.

03 — Architecture decisions

These decisions are recorded formally so that future engineers understand not just what was chosen but what was considered and explicitly rejected. An undocumented decision is a decision that will be relitigated — often at the worst possible time.

ADR-VIRT-01 — Cluster topology: join existing vs new cluster

Decision: The new bare-metal nodes join the existing OCP cluster (from Series II) as additional worker nodes.

Rejected: Forming a new, separate OCP cluster for OpenShift Virtualization workloads.

Rationale: A separate cluster would require duplicating the entire platform layer — a second Argo CD, a second cert-manager, a second External Secrets integration with CyberArk, a second Prometheus/Loki/Tempo stack, a second Harbor, and a second set of operator subscriptions. It would also require Advanced Cluster Management (ACM) to federate the two clusters, which is a significantly more complex architecture. The isolation that a separate cluster provides — preventing container workloads from landing on VM nodes — is achieved more simply within a single cluster using taints and tolerations. The separate cluster approach is the correct architecture for a multi-region or multi-tenant strategy, but for NovaCrestBank's data centre consolidation objective, it introduces complexity without benefit.

ADR-VIRT-02 — Bare-metal node OS: RHCOS vs RHEL

Decision: RHCOS (Red Hat CoreOS).

Rejected: RHEL 9 worker nodes.

Rationale: RHCOS is the only OS for which the KubeVirt stack (virt-handler, QEMU binaries, KVM module configuration) is tested and supported. The exact kernel version, KVM module parameters, and QEMU binary that ship with a given OCP version are validated together as a unit. RHEL workers require manual OS lifecycle management (upgrades, patches, configuration drift) — RHCOS nodes are managed declaratively through MachineConfig and upgraded automatically with the cluster. For a banking production environment, operational consistency and auditability are requirements, not preferences.

ADR-VIRT-03 — Node addition method: Assisted Installer Add Hosts

Decision: Assisted Installer "Add Hosts" flow — generating a discovery agent ISO from the existing cluster and booting each bare-metal node from it.

Rejected: Manual coreos-installer method (downloading RHCOS live ISO, booting, running coreos-installer with ignition config).

Rationale: The Assisted Installer Add Hosts flow is the current recommended approach for adding bare-metal nodes to an existing OCP cluster. It automates hardware discovery, performs pre-installation validation (CPU, memory, disk, network — before a single byte is written to disk), and handles the join process including CSR approval coordination. The manual method requires the engineer to manage the ignition config retrieval, coreos-installer arguments, and CSR approval sequence independently — more steps with more opportunities for error. The Assisted Installer flow also produces installation logs that serve as an audit trail, which matters for BFSI change management requirements.

ADR-VIRT-04 — ODF topology: converged vs dedicated storage nodes

Decision: Converged topology — all five bare-metal nodes run both VM workloads and ODF OSD (object storage daemon) processes.

Rejected: Dedicated ODF storage nodes (separate from compute nodes).

Rationale: Dedicated storage nodes provide stronger isolation between storage I/O and compute workloads, but require additional hardware. In the converged model, each node contributes NVMe disks to ODF while also running VM workloads. NovaCrestBank's ODF storage I/O profile (primarily sequential reads and writes from VM disks) does not create the kind of bursty, unpredictable I/O that makes converged topology problematic. ODF resource limits ensure that OSD processes do not starve VM workloads of CPU or memory. Converged topology with five nodes is also more resilient than dedicated storage with three nodes — ODF can tolerate two node failures instead of one. NIST SP 800-53 CP-7

ADR-VIRT-05 — VM networking: OVN-Kubernetes localnet (dual-model)

Decision: OVN-Kubernetes localnet topology for VM secondary networks, with a documented dual-model approach: Model A while NSX-T is present, Model B after NSX-T decommission.

Rejected: Linux bridge CNI for VM secondary networks.

Rationale: OVN-Kubernetes localnet is the recommended CNI for VM secondary networks in current OCP releases. It provides better integration with the OVN control plane, supports live migration MAC mobility natively, and does not require per-node bridge configuration. Linux bridge CNI is simpler but requires manual bridge setup on each node via MachineConfig and does not integrate with OVN for traffic policy. The dual-model documentation ensures NovaCrestBank engineers have a reference for both the current state (NSX-T enforces inter-zone policy) and the target state (NetworkPolicy enforces it after NSX-T exits).

ADR-VIRT-06 — ISO hosting: internal HTTP server

Decision: Agent ISO downloaded once from Red Hat CDN, SHA256-verified, and served internally from an HTTP server on the jump host. No bare-metal node contacts the internet during provisioning.

Rejected: Direct internet download at provision time from each node or from the jump host per provisioning run.

Rationale: PCI-DSS v4.0 Requirement 1.3.2 requires that inbound and outbound traffic be restricted to only what is necessary. An internet-facing download during provisioning opens a transient external access path that is not necessary — the ISO is a known, versioned artifact that can be downloaded once, verified, and reused. Internal hosting also eliminates dependency on Red Hat CDN availability during the provisioning window, which may be a change-controlled time slot. PCI-DSS v4.0 Req 1.3.2

04 — New hardware — HP ProLiant DL385 Gen11

Why DL385 Gen11 and not DL380 Gen11

The existing VCF environment uses HP ProLiant DL380 Gen11 — Intel Xeon-based servers. The new bare-metal nodes use HP ProLiant DL385 Gen11 — AMD EPYC 9004 series-based servers. The difference matters and the rationale must be documented.

The DL380 Gen11 is an excellent general-purpose server and is the correct choice for the VCF environment where ESXi hosts run management and OCP workload VMs. For the OpenShift Virtualization bare-metal nodes, three factors make the DL385 Gen11 the better choice.

Factor 1 — Memory bandwidth

Virtual machines are more memory-bandwidth-sensitive than containerised workloads. A container's memory access pattern is typically bursty — it needs memory in short bursts during request processing. A VM's memory access pattern is continuous — the guest OS and all its processes are constantly reading and writing VM RAM. AMD EPYC 9004 (Genoa) provides significantly higher memory bandwidth than Intel Xeon Scalable — EPYC Genoa supports 12 DDR5 memory channels per socket versus Intel's 8 channels. For VM-dense deployments where multiple VMs share a node, this bandwidth advantage reduces memory contention and improves VM-to-VM isolation.

Factor 2 — PCIe lane count and GPU readiness

The DL385 Gen11 with EPYC Genoa provides up to 160 PCIe 5.0 lanes per socket. The DL380 Gen11 provides fewer PCIe lanes. Why does this matter for OpenShift Virtualization? Two reasons.

First, each bare-metal node uses multiple PCIe slots simultaneously: NVMe drives for ODF OSDs (typically NVMe U.2 in PCIe slots), dual-port 25G NICs for the LACP bond, and potentially SR-IOV NICs for high-performance VM networking. With fewer PCIe lanes, the DL380 Gen11 can become lane-constrained when multiple high-bandwidth PCIe devices are installed simultaneously.

Second — and critically — the NovaCrestBank roadmap includes OpenShift AI (Series VI). AI inference and training workloads require GPU cards, and current generation NVIDIA and AMD data centre GPUs (H100, H200, MI300X) use 16 PCIe 5.0 lanes per GPU. A server with 160 PCIe lanes can host multiple GPUs alongside NVMe and NIC devices without contention. A server with fewer lanes cannot. Choosing the DL385 Gen11 now means the same physical servers that run OpenShift Virtualization workloads today can host GPU-accelerated AI inference workloads in the future — no hardware refresh required for the AI series.

Factor 3 — NUMA topology for VM placement

EPYC Genoa's NUMA architecture provides predictable memory locality for VM CPU pinning workloads. For NovaCrestBank's latency-sensitive financial application VMs (payment processing, real-time fraud detection), CPU pinning with NUMA awareness ensures a VM's vCPUs are always mapped to physical cores within the same NUMA domain as its memory — eliminating cross-NUMA memory latency spikes. This is documented in VIRT-01's PerformanceProfile section.

07a — Pull secret retrieval

What a pull secret is and why it is needed

Red Hat container images — including the RHCOS image, the OpenShift operator images, and the Assisted Installer images — are hosted in Red Hat's container registries (registry.redhat.io, quay.io). These registries require authentication. A pull secret is a JSON file containing base64-encoded credentials that grant access to Red Hat's registries for a specific subscription.

The pull secret is needed in two places in the VIRT series: here, when downloading the agent ISO (Section 07c), and in VIRT-01 when the Assisted Installer generates the discovery ISO for node provisioning. Without a valid pull secret, both operations fail with an authentication error.

The pull secret is tied to the Red Hat account that holds the OpenShift subscription. It is not a password — it is a time-limited credential that can be rotated from the Red Hat Hybrid Cloud Console. Treat it as a secret: do not commit it to Git, do not embed it in scripts without protection, and do not share it across organisations.

07b — Internal HTTP server setup on the jump host

Why an internal HTTP server is needed

During the Assisted Installer Add Hosts flow (VIRT-01), each bare-metal node must boot from an ISO image. The ISO is mounted via iLO virtual media, which supports two delivery methods: mount from a local file path on the iLO host, or mount from an HTTP/HTTPS URL. Mounting via HTTP is the correct approach for automated provisioning — it allows the same ISO to be mounted to all five nodes simultaneously without manually copying files, and the URL is a stable reference that can be embedded in automation scripts.

The HTTP server runs on the jump host, which is on VLAN 40 (OOB management, 10.40.0.0/24). Since iLO interfaces are also on VLAN 40, the iLO of each bare-metal node can reach the HTTP server directly over the OOB network — no routing through the production fabric is required. This is the correct security architecture: ISO serving and iLO communication both stay on the OOB network.

What HTTP server to use and why

Python's built-in http.server module is sufficient for this purpose. It requires no installation, no configuration files, and no service management — it is a single command. For a production-grade alternative, nginx is appropriate but not necessary for this use case. The ISO is a static file being served to five nodes during a one-time provisioning event — Python's HTTP server handles this correctly.

Set up the directory structure and start the server

# Create the directory that will serve ISOs
sudo mkdir -p /opt/novacrest/isos
sudo chown admin:admin /opt/novacrest/isos

# Start the HTTP server as a background service — port 8080 on VLAN 40 interface
# First, identify the jump host's VLAN 40 IP
ip addr show | grep "10.40."
inet 10.40.0.61/24 brd 10.40.0.255 scope global eth0

# Start Python HTTP server — serves /opt/novacrest/isos on port 8080
cd /opt/novacrest/isos
python3 -m http.server 8080 --bind 10.40.0.61 &
Serving HTTP on 10.40.0.61 port 8080 (http://10.40.0.61:8080/) ...

Make the HTTP server persistent across jump host reboots

A background process started with & dies when the shell session ends or the jump host reboots. For a provisioning server that may be referenced over a multi-day provisioning window, it must survive reboots. Create a systemd unit for it.

# Create the systemd unit file

07c — Agent ISO download and SHA256 verification

What the agent ISO is — and why it is different from the RHCOS live ISO

There are two different ISO types used in OpenShift bare-metal deployments. It is important to understand the difference before downloading anything.

The RHCOS live ISO is a general-purpose bootable image that launches a minimal RHCOS environment. It does not know anything about your OCP cluster. If you boot from it, you get a live shell — you then have to manually run coreos-installer with the correct ignition config arguments to install RHCOS to disk and configure the node. This is the manual provisioning method.

The agent ISO (also called the discovery ISO or the Assisted Installer ISO) is generated specifically for your cluster. It contains the Assisted Installer agent pre-configured with your cluster's API endpoint and pull secret. When a node boots from it, the agent automatically registers the node with the Assisted Installer, performs hardware discovery (CPU cores, memory, disk inventory, NIC detection), runs pre-installation validation, and waits for approval to proceed. This is the agent-based method — it is the correct method for NovaCrestBank's Add Hosts workflow.

The agent ISO is generated from the OCP API — it is not a generic Red Hat download. Section 07c downloads the openshift-install binary that generates it. The actual ISO generation happens in VIRT-01 once the Assisted Installer is configured with the cluster details.

07d — DNS and DHCP additions on DC01

Why DNS must exist before node provisioning begins

The Assisted Installer agent running on each bare-metal node performs a reverse DNS lookup of its own IP during the discovery phase. If the PTR record for 10.110.0.11 does not resolve to bm-virt-01.novacrestbank.local, the node fails discovery validation with a hostname mismatch error. DNS A and PTR records must exist before any node is powered on.

DHCP is required for the Assisted Installer's initial discovery phase. When a node first boots from the agent ISO, it does not yet have a static IP configured — it requests an IP via DHCP on VLAN 110. The Assisted Installer assigns the static IP (10.110.0.11–.15) during the installation phase. After the node joins the cluster, RHCOS configures the static IP from the NMState definition in the MachineSet. DHCP is only needed for the initial discovery boot.

07f — iLO initial configuration on all five servers

What iLO is — and why it must be configured before anything else

HPE iLO (Integrated Lights-Out) is the out-of-band management controller embedded in every HP ProLiant server. It is a separate, independent computer on a dedicated management chip — it has its own CPU, its own network port, and its own firmware. It is always powered on when the server is connected to power, even when the server's main OS is off.

iLO serves three functions in the VIRT series. First, it is the only way to interact with the server before an OS is installed — the BIOS configuration, virtual media mounting, and console access during RHCOS installation all go through iLO. Second, it provides out-of-band access to the server even when the OS is crashed or unreachable — critical for troubleshooting provisioning failures. Third, via the Redfish API, iLO can be automated — the prerequisite verification and ISO mounting steps in VIRT-01 use Redfish calls rather than manual GUI interaction.

Step 1 — Set iLO static IP via BIOS on each server (physical access required once)

The HP DL385 Gen11 ships with iLO in DHCP mode. The first step is to assign a static IP on VLAN 40 (OOB management network, 10.40.0.0/24). This requires brief physical console access during the server's first power-on.
