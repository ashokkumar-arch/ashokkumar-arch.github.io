01 — Why manage VMs as GitOps objects

After VIRT-06, banking application VMs are running on OCP Virt bare-metal nodes. They were created by MTV — each VM has a VirtualMachine CR in the virt-workloads namespace. At this point, those CRs exist only in the cluster — they are not tracked in any Git repository. If a VM's configuration needs to change (more memory, a different network interface, an updated cloud-init script), an engineer applies the change directly with oc apply. That change is not reviewed, not versioned, and not reproducible. If the cluster needs to be rebuilt, the VM configuration is lost.

Applying GitOps to VMs changes this: VirtualMachine CRs live in a Git repository. Every configuration change is a reviewed commit. Argo CD continuously reconciles the cluster to match Git. A VM's desired state is always readable from Git — not just inferable from the cluster.

What GitOps adds for VM workloads specifically

Reproducibility: The Git repository is the complete specification for every VM. Recreating a VM after cluster failure or disaster recovery is an Argo CD sync — not a manual kubectl recreation exercise from memory or from an out-of-date runbook.

Environment parity: Dev VMs and production VMs share the same base VirtualMachine specification. The differences — CPU pinning, hugepages, explicit MAC addresses, larger memory — are isolated in Kustomize overlays. When the base VM specification changes (a new cloud-init script, a different network policy annotation), the change applies to both environments through a single base commit.

Change audit trail: Every VM configuration change is a Git commit with an author, timestamp, and diff. PCI-DSS Req 6.4 requires that all changes to production systems are documented, reviewed, and traceable. A Git commit satisfies this — an ad-hoc oc apply does not. PCI-DSS v4.0 Req 6.4

Drift detection: If a VM's configuration in the cluster diverges from Git (an engineer edited the VirtualMachine CR directly to test something and forgot to revert), Argo CD detects the drift and alerts. With self-heal enabled for dev and disabled for production, the correct behaviour applies in each environment.

What Argo CD can and cannot manage for VMs

Can manage: The VirtualMachine CR (the definition — CPU, memory, disk references, network interfaces, runStrategy, evictionStrategy). The cloud-init Secret (userData for hostname, SSH keys, static IP). NetworkAttachmentDefinitions and NodeNetworkConfigurationPolicies. Any ConfigMap or annotation the VM depends on.

Cannot manage: The VM's PVC data content (the disk data itself — this is managed by ODF and backed up by OADP, not by Argo CD). The running VMI (VirtualMachineInstance) — this is a live runtime object. The snapshot objects (these are point-in-time captures, not desired-state declarations).

Argo CD manages the VM definition — not the VM's disk data or its running state. This is the same relationship as Argo CD managing a Deployment (the definition) but not the Pods (the runtime instances).

02 — The two-repository pattern for VM configuration

NovaCrestBank's GitOps architecture uses two Git repositories — inherited from the OCP-OPS-05 GitOps guide that established this pattern for containerised banking applications. The same pattern applies to VM workloads.

Why source code and VM configuration live in separate repositories

For containerised applications, the reason for separation is clear: the application source code repo triggers builds; the config repo holds the deployment YAML updated by the pipeline. For VMs, there is no build pipeline — but the separation still serves two critical purposes.

Access control: Application teams should be able to propose VM configuration changes (requesting more memory, a different cloud-init script) through a pull request. But they should not have direct write access to the production VM configuration. The two-repo pattern enforces this — developers PR against the config repo, senior engineers and platform admins approve changes that affect production VMs.

History clarity: The VM config repo's git log is the complete operational history of every VM — every resize, every network change, every cloud-init update. This history must be clean and purposeful. Mixing it with application source code commits would make the audit trail difficult to read.

Config repo structure for NovaCrestBank VMs

Repository: gitlab.novacrestbank.local/gitops/banking-config — the same config repo used for containerised banking application deployment (OCP-OPS-05). VMs are new top-level directories alongside the existing apps/ directory tree.

Structure: vms/banking-app-vm/ with base/ (shared VirtualMachine CR) and overlays/dev/, overlays/production/. Mirrors the Kustomize overlay structure already established for containerised applications. Engineers familiar with the existing config repo structure immediately understand the VM overlay layout.

banking-config/ ├── apps/ ← existing containerised app overlays (OCP-OPS-05) │ └── payment-service/ │ ├── base/ │ └── overlays/ dev/ staging/ production/ └── vms/ ← VM workload overlays — new in VIRT-07 └── banking-app-vm/ ├── base/ │ ├── kustomization.yaml ← lists VirtualMachine, Secret, NAD ref │ ├── virtualmachine.yaml ← shared VM spec (no env-specific fields) │ └── cloudinit-secret.yaml ← base cloud-init userData └── overlays/ ├── dev/ │ ├── kustomization.yaml ← shared CPU, 8GiB RAM, no hugepages │ └── vm-patch.yaml ← patches for dev resource profile └── production/ ├── kustomization.yaml ← CPU pinning, 32GiB hugepages, VLAN 130 └── vm-patch.yaml ← patches for production resource profile

03 — OpenShift GitOps installation

OpenShift GitOps is installed via OLM — the same pattern used for HCO, ODF, and MTV. The key difference from other operators: the default Argo CD instance created by the operator is disabled, and NovaCrestBank's own ArgoCD CR is applied instead. The default instance has cluster-admin privileges and deploys into openshift-gitops — too broad for a banking environment.

gitops-subscription.yaml — OpenShift GitOps Operator Subscription
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: openshift-gitops-operator
  namespace: openshift-operators
spec:
  channel: latest
    # latest is the only stable production channel for OpenShift GitOps.
  name: openshift-gitops-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual
  config:
    env:
      - name: DISABLE_DEFAULT_ARGOCD_INSTANCE
        value: "true"
                # true: do not create the default openshift-gitops ArgoCD instance.
        # The default instance deploys with cluster-admin ServiceAccount — too broad
        # for NovaCrestBank. Section 04 creates a scoped instance in the gitops namespace.

04 — ArgoCD CR — NovaCrestBank configuration

The ArgoCD CR tells the GitOps operator how to deploy the Argo CD instance. Every field is annotated with its purpose and the NovaCrestBank-specific reasoning. This instance deploys into the gitops namespace — a dedicated namespace separate from openshift-gitops.

argocd-ncb.yaml — ArgoCD CR — every field annotated
apiVersion: argoproj.io/v1alpha1
kind: ArgoCD
metadata:
  name: ncb-argocd
  namespace: gitops
    # gitops: dedicated namespace — not openshift-gitops.
  # Tighter NetworkPolicy, custom RBAC, scoped ServiceAccount.
  # All Argo CD pods, Secrets, and ServiceAccounts live here.
spec:
  server:
    route:
      enabled: true
      tls:
        termination: reencrypt
                # reencrypt: TLS from client to OCP router, then a new TLS connection
        # from the router to the Argo CD server pod. Argo CD's own certificate
        # (issued by cert-manager from NovaCrestBank-IssuingCA) is validated
        # end-to-end — not just at the router edge.
    insecure: false
  applicationSet:
    enabled: true
        # Enable the ApplicationSet controller — required for Section 10.
  dex:
    openShiftOAuth: false
        # false: use AD LDAP directly via Dex (configured below).
    # Not OCP OAuth — AD group-to-Argo CD role mapping requires LDAP groups
    # which OCP OAuth does not expose at the group granularity needed.
  rbac:
    defaultPolicy: role:readonly
        # Any authenticated user who does not match a specific policy gets read-only.
    # Platform engineers, developers, and auditors can all VIEW application status
    # and sync history without being able to modify anything.
    policy: |
      g, NovaCrestBank-GitOps-Admins, role:admin
      g, NovaCrestBank-GitOps-Operators, role:readonly
        # NovaCrestBank-GitOps-Admins → Argo CD admin (platform engineering leads only).
    # NovaCrestBank-GitOps-Operators → read-only + manual sync trigger on approved apps.
    scopes: '[groups]'
        # Argo CD reads AD group memberships from the Dex LDAP token for RBAC matching.
  redis:
    autotls: openshift
  repo:
    autotls: openshift
  ha:
    enabled: false
        # Single-instance Argo CD. Running applications continue working even if
    # Argo CD is temporarily down — it is not in the critical path for running VMs.
  resourceExclusions: |
      - apiGroups:
        - kubevirt.io
        kinds:
        - VirtualMachineInstance
        - VirtualMachineInstanceMigration
        clusters:
        - "*"
    # Exclude VMI and VMIM objects from Argo CD's resource tracking.
  # VMIs are runtime objects — they are created and destroyed by KubeVirt
  # as VMs start and stop. They are not in the Git config repo.
  # If Argo CD tracked them, every VM start/stop would show as OutOfSync.
  # Argo CD manages the VirtualMachine CR (the definition), not the VMI (the instance).
  # VMIM (live migration) objects are ephemeral — also excluded.

05 — AppProject — scoping what Argo CD may deploy for VM workloads

The AppProject is the security boundary for Argo CD Applications. Without it, an Application could deploy to any namespace and create any resource type — including ClusterRoles that elevate privilege. The virt-workloads AppProject restricts VM Applications to exactly the namespaces and resource types they need.

appproject-virt-workloads.yaml — AppProject for VM workloads
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: virt-workloads
  namespace: gitops
spec:
  description: NovaCrestBank VM workloads — dev and production VirtualMachine CRs
  sourceRepos:
    - git@gitlab.novacrestbank.local:gitops/banking-config.git
        # Only this repository. Applications in this project cannot sync from
    # any other source — not GitHub, not an external Helm chart repository.
  destinations:
    - namespace: virt-workloads-dev
      server: https://kubernetes.default.svc
    - namespace: virt-workloads
      server: https://kubernetes.default.svc
            # Two namespaces: dev (smaller VMs, shared CPU) and production (the VIRT-05/06 namespace).
      # Applications in this project cannot deploy to kube-system, openshift-cnv,
      # or openshift-storage — only to these two namespaces.
  clusterResourceWhitelist: []
    # Empty: no cluster-scoped resources allowed.
  # VM Applications cannot create ClusterRoles, ClusterRoleBindings, or CRDs.
  namespaceResourceWhitelist:
    - group: kubevirt.io
      kind: VirtualMachine
            # The core resource: VirtualMachine CRs are the primary GitOps object here.
    - group: "*"
      kind: Secret
            # cloud-init Secrets (userData for hostname, SSH keys, network config).
    - group: "*"
      kind: ConfigMap
    - group: k8s.cni.cncf.io
      kind: NetworkAttachmentDefinition
            # NADs for VM secondary interfaces (VLAN 130/140) — managed alongside VMs.
    - group: networking.k8s.io
      kind: NetworkPolicy
            # NetworkPolicy for intra-namespace VM traffic control (Model B, VIRT-11).

06 — Config repo structure for VM workloads

Before showing the YAML, understand what Kustomize does for VM CRs and why the base/overlay split is the correct structure. Kustomize takes a base VirtualMachine CR that is correct for all environments and applies environment-specific patches on top. The base VM has conservative resources. Each overlay patches exactly the fields that differ for that environment — nothing more.

Dev overlay
  • No CPU pinning — shared CPU pool
  • No hugepages — standard 4 KiB pages
  • 4 vCPU, 8 GiB RAM
  • masquerade primary interface only
  • runStrategy: Manual (start/stop on demand)
  • No explicit macAddress (random — no live migration needed)
Staging (future)
  • CPU pinning enabled
  • 1 GiB hugepages, 16 GiB
  • VLAN 130 secondary interface
  • Static IP on staging VLAN subnet
  • runStrategy: Always
  • Mirror of production resources at 50%
Production overlay
  • dedicatedCpuPlacement: true
  • 1 GiB hugepages, 32 GiB RAM
  • VLAN 130 secondary, explicit MAC
  • Static IP 10.130.0.50, gateway .3
  • runStrategy: Always
  • evictionStrategy: LiveMigrate

07 — Base VirtualMachine CR — the shared specification

The base VirtualMachine CR contains everything that is identical across all environments: the guest OS type, the disk interface bus, the primary masquerade network interface, the cloud-init volume reference, and the node selector that restricts scheduling to bare-metal nodes. Fields that differ between environments are absent from the base — they are added or patched by each overlay.

vms/banking-app-vm/base/virtualmachine.yaml — base VM specification
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: banking-app-vm
    # Base name. Each overlay patches the name to match the environment:
  # dev overlay → banking-app-vm-dev
  # production overlay → vm-banking-app-01 (matches DNS and MAC convention)
  labels:
    app: banking-app
    managed-by: argocd
        # managed-by: argocd label allows Argo CD to track this resource.
    # Argo CD uses this label when computing diff and health status.
    # Without this label, Argo CD may not correctly associate the resource
    # with the Application that manages it.
spec:
  runStrategy: Manual
    # Base default: Manual. Each overlay patches to the correct runStrategy.
  # Manual is the safe base default — the VM does not auto-start when the CR
  # is first applied. Production overlay patches to Always.

  template:
    spec:
      domain:
        cpu:
          cores: 4
          threads: 1
          model: host-passthrough
                    # host-passthrough in the base — all environments use the bare-metal EPYC CPU.
          # dedicatedCpuPlacement is absent from the base. Production overlay adds it.
        memory:
          guest: 8Gi
                    # Conservative base: 8 GiB. Production overlay patches to 32 GiB.
          # hugepages absent from base — dev does not use hugepages.
          # Production overlay adds hugepages.pageSize: "1Gi".
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - masquerade: {}
              name: default
                            # Primary interface: masquerade (OVN overlay). Present in ALL environments.
              # Provides cluster-internal access: OCP API, monitoring, cert-manager.
              # Production overlay adds a secondary VLAN 130 bridge interface.
        resources:
          requests:
            memory: 8Gi

      networks:
        - name: default
          pod: {}

      volumes:
        - name: rootdisk
          dataVolume:
            name: banking-app-vm-disk
        - name: cloudinitdisk
          cloudInitNoCloud:
            secretRef:
              name: banking-app-vm-cloudinit
                        # Reference a Secret for cloud-init rather than inline userData.
            # The Secret contains the userData (hostname, SSH keys, network config).
            # Using a Secret allows the cloud-init data to be managed separately
            # from the VirtualMachine CR — the Secret can be rotated without
            # changing the VM definition, and it is not visible in git blame
            # for the VM YAML (separation of concerns: VM spec vs cloud-init config).

      nodeSelector:
        workload-type: virtualization
                # Present in ALL environments — VMs always schedule on bare-metal nodes.

      tolerations:
        - key: kubevirt.io/virt
          operator: Exists
          effect: NoSchedule
                    # Required in all environments — bare-metal nodes carry this taint.

08 — Dev overlay — shared CPU, standard pages, smaller memory

The dev overlay patches the base VM to a resource profile appropriate for development and functional testing. No CPU pinning — the VM shares physical cores with other workloads. No hugepages — standard 4 KiB pages. Smaller memory. runStrategy: Manual — dev VMs are started and stopped on demand, not kept running continuously. No secondary VLAN interface — the dev VM uses only the masquerade primary interface for OCP cluster access.

vms/banking-app-vm/overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
    # Inherit the complete base VirtualMachine spec.
  # Only the patches below change anything. Everything else is identical to base.
namePrefix: dev-
  # dev-: adds "dev-" prefix to all resource names.
  # banking-app-vm → dev-banking-app-vm
  # Ensures dev and production VMs can coexist in the cluster without name collision.
patches:
  - target:
      kind: VirtualMachine
      name: banking-app-vm
    patch: |
      - op: replace
        path: /spec/runStrategy
        value: Manual
      - op: replace
        path: /spec/template/spec/domain/cpu/cores
        value: 4
      - op: replace
        path: /spec/template/spec/domain/memory/guest
        value: "8Gi"
      - op: replace
        path: /spec/template/spec/domain/resources/requests/memory
        value: "8Gi"
      - op: add
        path: /spec/template/spec/domain/cpu/dedicatedCpuPlacement
        value: false
            # dedicatedCpuPlacement: false — dev VM uses shared CPU pool.
      # The base does not have this field — adding it explicitly here ensures
      # the dev overlay is unambiguous and will not inherit a production overlay
      # if base ever gains a default value for this field.

09 — Production overlay — CPU pinning, hugepages, VLAN 130 secondary interface

The production overlay patches the base VM with the full performance and networking configuration required for banking application workloads. Every field added or changed from the base is annotated.

vms/banking-app-vm/overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namespace: virt-workloads
  # Production VMs go into the virt-workloads namespace (the VIRT-05/06 namespace).
  # Dev VMs go into virt-workloads-dev (separate namespace, separate NetworkPolicy).
patches:
  - target:
      kind: VirtualMachine
      name: banking-app-vm
    patch: |
      - op: replace
        path: /metadata/name
        value: vm-banking-app-01
            # Rename to match the existing DNS record and MAC convention from VIRT-05.
      # vm-banking-app-01.novacrestbank.local → 10.130.0.50 (DNS pre-created in VIRT-02b).
      # This overlay manages the SAME VM that MTV migrated in VIRT-06 —
      # it brings the migrated VM under GitOps control.
      - op: replace
        path: /spec/runStrategy
        value: Always
            # Always: production banking VMs must stay running.
      # KubeVirt restarts the VM if it crashes or the guest OS shuts down unexpectedly.
      - op: add
        path: /spec/template/spec/domain/cpu/dedicatedCpuPlacement
        value: true
            # CPU pinning — dedicated physical cores from the isolated pool (4–63).
      # Required for sub-millisecond P99 latency SLO on payment processing workload.
      - op: replace
        path: /spec/template/spec/domain/cpu/cores
        value: 8
            # 8 dedicated cores for the payment processing VM.
      # Production needs more parallelism than dev (4 cores).
      - op: replace
        path: /spec/template/spec/domain/memory/guest
        value: "32Gi"
      - op: add
        path: /spec/template/spec/domain/memory/hugepages
        value:
          pageSize: "1Gi"
            # 32 GiB backed by 1 GiB hugepages — TLB pressure eliminated for
      # the payment processing VM's full address space (VIRT-04 Section 11).
      - op: replace
        path: /spec/template/spec/domain/resources/requests/memory
        value: "32Gi"
      - op: add
        path: /spec/template/spec/domain/devices/interfaces/1
        value:
          bridge: {}
          name: vlan130-nic
          macAddress: "02:00:00:0a:82:32"
            # Secondary VLAN 130 interface — the production IP interface.
      # macAddress: explicit — required for live migration MAC mobility (VIRT-05 Section 16).
      # 02:00:00:0a:82:32 = 02:00:00 prefix + 10.130.0.50 in hex.
      # This MAC was already assigned to this VM during VIRT-05/06 — the GitOps CR
      # records it explicitly so the MAC is committed to Git and auditable.
      - op: add
        path: /spec/template/spec/networks/1
        value:
          name: vlan130-nic
          multus:
            networkName: vlan130-primary
      - op: add
        path: /spec/template/spec/evictionStrategy
        value: LiveMigrate
            # evictionStrategy: LiveMigrate — during node drain, this VM is live-migrated
      # to another node rather than being stopped. Required for zero-downtime
      # node maintenance on production banking VMs.
      # Without this, a node drain STOPS the VM (brief downtime for payment processing).
      # Requires: RWX PVC (ocs-storagecluster-ceph-rbd-virtualization ✓),
      # secondary interface with explicit MAC ✓, no SR-IOV ✓.
      # Section 11 covers evictionStrategy in full.

10 — ApplicationSet — deploying VMs to dev and production

One ApplicationSet generates two Application objects — dev and production. The sync policy differs between environments by design: dev auto-syncs so VM configuration changes are tested immediately, production requires a manual sync so an engineer reviews the Argo CD diff before the running VM's definition is updated.

Argo CD syncing a VirtualMachine CR does not restart the VM

This is the most important behaviour to understand before enabling Argo CD on VMs. When Argo CD syncs a changed VirtualMachine CR, it applies the new CR to Kubernetes — but KubeVirt only restarts the VM if the change requires it. Most VirtualMachine CR changes (runStrategy, labels, annotations) take effect without a VM restart. Changes to the VM hardware spec (CPU, memory, disk) take effect on the next VMI start — not immediately. If you want the change to take effect while the VM is running, use hot-plug (Sections 18–19) or VirtualMachine live restart. Argo CD syncing the VM CR is a configuration update, not a power cycle.

applicationset-banking-vms.yaml — ApplicationSet for VM workloads
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: banking-vms
  namespace: gitops
spec:
  generators:
    - list:
        elements:
          - env: dev
            namespace: virt-workloads-dev
            syncPolicy: automated
            selfHeal: "true"
                        # dev: auto-sync + self-heal.
            # A developer editing dev-banking-app-vm directly is reverted by Argo CD
            # within 3 minutes. Git is the source of truth even for dev VMs.
          - env: production
            namespace: virt-workloads
            syncPolicy: none
            selfHeal: "false"
                        # production: manual sync only. No self-heal.
            # An engineer editing vm-banking-app-01 directly during an incident
            # (increasing memory for an OOM issue) is NOT reverted by Argo CD.
            # The manual change survives until properly committed to Git
            # and the Application is synced. This is the correct production behaviour.
  template:
    metadata:
      name: banking-vms-{{env}}
      namespace: gitops
    spec:
      project: virt-workloads
      source:
        repoURL: git@gitlab.novacrestbank.local:gitops/banking-config.git
        targetRevision: HEAD
        path: vms/banking-app-vm/overlays/{{env}}
                # Each environment points at its own Kustomize overlay directory.
        # dev     → vms/banking-app-vm/overlays/dev
        # production → vms/banking-app-vm/overlays/production
      destination:
        server: https://kubernetes.default.svc
        namespace: {{namespace}}
      syncPolicy:
        automated:
          prune: false
                    # prune: false for both environments.
          # Deleting a VirtualMachine CR via Git prune would STOP the VM.
          # VM deletion from Git should always be a deliberate, manual operation —
          # not an automatic prune triggered by removing a YAML file.
          # To decommission a VM: stop it first, then remove from Git, then manually sync.
          selfHeal: {{selfHeal}}
        syncOptions:
          - ServerSideApply=true
            # Server-side apply: correct handling of VirtualMachine CR fields managed
          # by KubeVirt controllers (status, VMI references). Prevents field ownership
          # conflicts between Argo CD and the KubeVirt operator.
          - CreateNamespace=false
            # Namespaces are pre-created by the platform team with proper labels,
          # NetworkPolicy, and resource quotas — not auto-created by Argo CD.

11 — evictionStrategy — live migration during node drain

When a bare-metal node is drained — for a firmware update, an OCP upgrade, or a hardware replacement — Kubernetes evicts the pods on the node. For regular pods, eviction means deletion: the pod is deleted and rescheduled on another node. For VMs, deletion means stopping the guest OS — a brief but real outage for payment-processing workloads.

evictionStrategy: LiveMigrate changes this behaviour. When the KubeVirt virt-handler on the node receives a drain signal, instead of stopping the VM, it triggers a live migration to another available bare-metal node. The VM's guest OS continues running throughout the migration. The node drain completes only after all VMs with evictionStrategy: LiveMigrate have successfully migrated off. The result: maintenance drains cause zero downtime for production banking VMs.

What evictionStrategy requires

Three conditions must all be true for LiveMigrate to work during a drain:

  • RWX PVC: The VM's boot disk must be on a ReadWriteMany PVC. During live migration, both the source and destination virt-launcher pods must access the disk simultaneously. ocs-storagecluster-ceph-rbd-virtualization provides RWX — confirmed in VIRT-03 and VIRT-04.
  • Explicit macAddress: The VM must have an explicit MAC address on its secondary VLAN 130 interface. Without it, the destination VMI gets a new random MAC, and the gratuitous ARP mechanism cannot correctly update the leaf switch's MAC table. Configured in the production Kustomize overlay — Section 09.
  • No SR-IOV: SR-IOV VFs are bound to a specific physical NIC on a specific node. SR-IOV VMs cannot be live-migrated. evictionStrategy: LiveMigrate on an SR-IOV VM causes the drain to fail — KubeVirt cannot migrate and cannot stop (because evictionStrategy says LiveMigrate). Use evictionStrategy: None for SR-IOV VMs and plan cold migration for their maintenance.

12 — VirtualMachineSnapshot — point-in-time disk capture

A VirtualMachineSnapshot captures the state of a VM and all its disks at a specific point in time. It uses the CSI VolumeSnapshot API under the hood — Ceph RBD creates a snapshot of the VM's PVC at the storage level. The snapshot is nearly instantaneous (Ceph RBD snapshots are copy-on-write) and does not require the VM to be powered off.

What a snapshot captures

A VirtualMachineSnapshot captures the VirtualMachine CR (the definition) and a CSI VolumeSnapshot of each disk PVC. The CSI snapshot is a point-in-time copy-on-write clone of the PVC at the block level. After the snapshot, the VM continues running and writing to its disk — the snapshot data is preserved separately by Ceph's copy-on-write mechanism. The snapshot does not include RAM state — it is a disk snapshot, not a full VM checkpoint.

Online snapshots and qemu-guest-agent quiescing

A snapshot taken while the VM is running may capture a disk in an inconsistent state if the guest OS has buffered writes that have not been flushed to disk. qemu-guest-agent solves this: KubeVirt sends a filesystem freeze command through the agent before triggering the CSI snapshot, and a thaw command after. The freeze flushes all pending writes and prevents new ones during the brief snapshot window. The result is a crash-consistent (if qemu-guest-agent is not installed) or application-consistent (if it is installed and the application supports freeze hooks) snapshot. All NovaCrestBank VMs have qemu-guest-agent installed — confirmed in VIRT-04 Section 17.

14 — OADP — what it is and how it backs up VMs

OADP (OpenShift API for Data Protection) is the Red Hat operator that provides application-consistent backup and restore for Kubernetes workloads — including VMs. Where VirtualMachineSnapshot (Sections 12–13) provides a fast, in-cluster point-in-time snapshot using Ceph RBD copy-on-write, OADP provides durable, scheduled backups that are stored externally — in ODF's S3-compatible object storage (MCG) — and can survive complete ODF cluster failure.

OADP vs VirtualMachineSnapshot — when to use each

CapabilityVirtualMachineSnapshotOADP
Storage locationInside ODF (Ceph RBD snapshot) — same clusterExternal S3 (MCG ObjectBucketClaim) — survives ODF failure
SpeedNear-instantaneous (copy-on-write)Full backup: minutes to hours depending on disk size
SchedulingManual onlyAutomatic via BackupSchedule CR (cron expression)
VM state requirementRunning or stoppedRunning (with quiesce hook) or stopped
Recovery fromAccidental file deletion, bad software update (in-cluster)ODF failure, cluster disaster, ransomware, off-site DR
RetentionManual — must be explicitly deletedConfigured TTL — automatically expired by OADP

How OADP backs up a running VM

OADP uses Velero under the hood. For VM disk backup, the process is: OADP instructs qemu-guest-agent to freeze the VM's filesystem (same mechanism as VirtualMachineSnapshot), triggers a CSI VolumeSnapshot of the VM's PVC via the Kubernetes VolumeSnapshot API, thaws the filesystem, and then uploads the VolumeSnapshot data to the configured S3 backup store (MCG ObjectBucketClaim backed by Ceph RGW). The upload happens asynchronously after the freeze/thaw — the VM's freeze window is only a few seconds. The data in S3 is the complete disk image at the point the snapshot was taken. backup-schedule-vms.yaml — nightly VM backup schedule
apiVersion: velero.io/v1
kind: Schedule
metadata:
  name: ncb-vm-nightly-backup
  namespace: openshift-adp
spec:
  schedule: "0 2 * * *"
    # 02:00 UTC daily. Runs during the low-traffic window for NovaCrestBank
  # (02:00 UTC = 12:00 AEST — overnight for Australian banking operations).
  # Cron: minute hour day-of-month month day-of-week
  template:
    includedNamespaces:
      - virt-workloads
        # Back up only the production VM namespace.
      # Dev VMs (virt-workloads-dev) are excluded — dev disk data is not
      # worth the backup storage cost. Dev VMs can be recreated from the
      # base Kustomize overlay and the RHEL 9 base image.
    includedResources:
      - virtualmachines
      - persistentvolumeclaims
      - persistentvolumes
      - secrets
        # secrets: includes cloud-init Secrets alongside the VM definition.
      # The VM cannot boot correctly without the cloud-init Secret (hostname, SSH keys).
      # Including it in the backup ensures a complete, bootable restore.
    labelSelector:
      matchLabels:
        app: banking-app
            # Back up only VMs with the banking-app label.
      # Excludes utility VMs or any non-banking VM that may be in the namespace.
      # The app: banking-app label is set in the base VirtualMachine CR (Section 07).
    snapshotVolumes: true
        # true: include PVC disk data in the backup using CSI VolumeSnapshots.
    # Without this: OADP backs up PVC metadata only — not the actual disk data.
    # A restore from a backup with snapshotVolumes: false would create empty PVCs.
    # The VM would boot to an empty disk. Always set true for VM backups.
    defaultVolumesToFsBackup: false
        # false: do not use filesystem-level backup (kopia) for VM PVCs.
    # VM PVCs are Block-mode volumes — kopia cannot perform filesystem backup
    # on raw block devices. CSI VolumeSnapshot (controlled by snapshotVolumes: true)
    # is the correct mechanism for Block-mode VM disk PVCs.
    hooks:
      resources:
        - name: vm-quiesce-hook
          includedNamespaces:
            - virt-workloads
          labelSelector:
            matchLabels:
              app: banking-app
          pre:
            - exec:
                container: compute
                command:
                  - /usr/bin/bash
                  - -c
                  - virsh qemu-agent-command 1 '{"execute":"guest-fsfreeze-freeze"}'
                timeout: 30s
                                # Pre-backup hook: freeze the VM's filesystem via qemu-guest-agent.
                # Flushes all pending writes and prevents new ones during the snapshot window.
                # The snapshot window (freeze to thaw) is typically 1–3 seconds.
                # timeout: 30s — if the freeze does not complete in 30 seconds (application
                # is holding a lock), the backup fails rather than taking an inconsistent snapshot.
          post:
            - exec:
                container: compute
                command:
                  - /usr/bin/bash
                  - -c
                  - virsh qemu-agent-command 1 '{"execute":"guest-fsfreeze-thaw"}'
                timeout: 30s
    ttl: 720h0m0s
        # 720 hours = 30 days. Backups older than 30 days are automatically deleted.
    # NovaCrestBank's backup retention policy: 30 days for production VM backups.
    # PCI-DSS Req 12.3.1 requires documented data retention — this CR is the documentation.
    # For longer retention: increase ttl, or configure OADP to copy to a
    # long-term S3 storage tier (Glacier equivalent) before the 30-day expiry.

18 — CPU and memory hot-plug — what it is and when to use it

CPU and memory hot-plug is the ability to add vCPU cores or memory to a running VM without rebooting the guest OS. KubeVirt supports hot-plug for both CPU and memory since KubeVirt 1.0 (included in OCP Virt 4.14+). Hot-plug allows capacity increases on production banking VMs during periods of elevated load without a maintenance window.

How hot-plug works in KubeVirt

When a hot-plug request is issued (by patching the VirtualMachine CR or using virtctl addvolume-equivalent commands), KubeVirt sends a request to the running QEMU process via virt-handler. QEMU adds the CPU or memory device to the virtual hardware. The guest OS receives a hardware change event and decides how to handle it — modern Linux kernels (RHEL 9 uses Linux 5.14+) automatically online new CPUs and memory blocks without requiring a reboot. Windows Server 2022 also supports hot-plug CPU and memory for the Standard and Datacenter editions.

Hot-plug does not change the VM's permanent specification — it is a runtime adjustment. After a VM reboot, the VM starts with the original specification from the VirtualMachine CR. To make a hot-plug change permanent, update the VirtualMachine CR (via Git commit → Argo CD sync) and perform a VM restart.

Hot-plug CPU
Hot-plug memory

When to use: Payment processing VM experiencing high CPU utilisation during end-of-month batch processing. Need to increase vCPU count from 8 to 12 without stopping the VM and without a maintenance window.

Requirement: CPU hot-plug requires maxSockets to be set in the VirtualMachine CR to define the maximum possible CPU count. KubeVirt pre-allocates physical CPU pinning slots up to this maximum. If maxSockets is not set, hot-plug CPU is not available on the VM.

Limitation: Can only add CPUs — cannot remove (hot-unplug CPU is not supported by Linux guests). To decrease vCPUs, a VM reboot is required.

When to use: Core banking API VM has OOMKilled pods due to insufficient JVM heap space during peak transaction load. Need to increase memory from 32 GiB to 48 GiB immediately without stopping the service.

Requirement: maxGuest must be set in the VirtualMachine CR to define the maximum possible memory. KubeVirt pre-allocates hugepage slots up to this maximum. If hugepages are used (as in production), hugepage allocation for hot-plug memory must be available on the node.

Limitation: Memory hot-unplug is also not supported by most guest OSes. Increasing memory is non-disruptive. Decreasing requires a reboot.

19 — Applying CPU and memory hot-plug to a running VM

First, the VirtualMachine CR must declare the maximum hot-plug capacity via maxSockets and maxGuest. These fields must be set before the VM is running — they cannot be hot-plugged themselves. Set them in the production Kustomize overlay so they are committed to Git and applied on the next VM restart or Argo CD sync.

Add maxSockets and maxGuest to production overlay patch
# Add to the production overlay patch (vms/banking-app-vm/overlays/production/vm-patch.yaml)
# These fields enable hot-plug headroom at VM start

- op: add
  path: /spec/template/spec/domain/cpu/maxSockets
  value: 4
  # maxSockets: 4 — allows scaling from 8 cores (current) up to 4×2 = 8 sockets × 2 threads...
  # KubeVirt CPU topology: maxSockets × cores × threads = maximum vCPU count.
  # With cores: 8 and maxSockets: 4 the maximum is 8 × 4 = 32 vCPUs hot-plugged.
  # Physical pinning slots for up to 32 cores are reserved from the isolated pool at VM start.
  # Only 8 are active initially — the rest are available for hot-plug without restart.

- op: add
  path: /spec/template/spec/domain/memory/maxGuest
  value: "64Gi"
  # maxGuest: 64Gi — allows memory hot-plug up to 64 GiB.
  # The VM starts with guest: 32Gi.
  # Hugepage slots for up to 64 GiB (64 × 1 GiB hugepages) are reserved from
  # the node's hugepage pool at VM start.
  # The node has 400 × 1 GiB hugepages total — 64 reserved for this VM's max.
  # Remaining hugepages are available for other VMs on the same node.
