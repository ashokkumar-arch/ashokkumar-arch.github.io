01 — The VM networking problem — why masquerade is not enough

The test VM created in VIRT-04 uses a masquerade interface for its primary network connection. Masquerade is NAT — the VM gets an IP inside the OVN pod overlay network, and traffic to and from it is translated through the virt-launcher pod's IP. The VM is not directly reachable from outside the OCP cluster. For a test VM used to verify that KubeVirt is working, this is sufficient. For banking application VMs, it is not acceptable.

What banking application VMs need

Consider a payment-processing VM that has been running in the existing hypervisor environment for years. It is reachable at IP 10.130.0.50 from the load balancer that distributes transactions to it, from the core banking system it calls for account data, from the monitoring system that polls it for health, and from the firewall rules that control its traffic. Its FQDN is vm-banking-app-01.novacrestbank.local and is hardcoded in configuration files across the data centre.

When this VM is migrated to OpenShift Virtualization, all of those integrations must continue working without reconfiguration. A masquerade interface on 10.0.2.2 does not satisfy this requirement. What is needed is a secondary network interface that connects the VM directly to VLAN 130 on the physical Cisco fabric — giving it a real, routable IP of 10.130.0.50 reachable from anywhere that has a route to 10.130.0.0/24.

The four-layer mental model — understand this before any YAML

Layer 1 — Physical NIC: The bare-metal node's LACP bond has sub-interfaces bond0.130 and bond0.140 tagged to VLANs 130 and 140 on the Cisco leaf switches. Configured in VIRT-02 via NMState.

Layer 2 — Node bridge (NNCP): A NodeNetworkConfigurationPolicy CR uses NMState to create Linux bridges (br-vlan130, br-vlan140) on every bare-metal node and connects the bond sub-interfaces to them. OVN-Kubernetes localnet taps into these bridges. Section 06 creates these.

Layer 3 — Named network (NAD): A NetworkAttachmentDefinition CR declares a named secondary network that Multus can attach to pod or VM interfaces. The NAD references the bridge on the node. Section 07 creates these.

Layer 4 — VM interface: The VirtualMachine CR references the NAD by name. Multus creates a secondary NIC inside the virt-launcher pod, connected through the bridge to the physical VLAN. Section 09 shows this in a complete VM YAML.

02 — Multus — what it is and how it works

Multus is a CNI (Container Network Interface) meta-plugin. To understand what a meta-plugin is, you first need to understand what CNI is and what its limitation is for VM workloads.

What CNI is — the standard

CNI is the standard interface between a container runtime (cri-o on RHCOS) and a network plugin. When a pod is created, cri-o calls the configured CNI plugin with the pod's details. The CNI plugin creates a network interface inside the pod's network namespace, assigns an IP address, and sets up routing. When the pod is deleted, CNI tears down the interface. In a standard OCP cluster, OVN-Kubernetes is the sole CNI plugin — it creates one network interface (eth0) per pod and connects it to the OVN overlay.

The limitation: standard CNI allows exactly one network interface per pod. One plugin, one interface, one IP. For VMs that need a primary interface for OCP cluster communication and one or more secondary interfaces for production VLAN connectivity, this is insufficient.

What Multus does

Multus acts as a broker between cri-o and multiple CNI plugins. When a pod annotated with additional network references starts, Multus intercepts the CNI call, calls OVN-Kubernetes for the primary interface (eth0), then calls the appropriate secondary CNI plugin for each additional interface. The result for a VM pod:

  • eth0: Created by OVN-Kubernetes. Pod overlay IP. Cluster-internal communication — OCP API, DNS, service discovery.
  • eth1: Created by the secondary CNI plugin called by Multus. Connected to VLAN 130 via the br-vlan130 bridge. VM's production IP on 10.130.0.0/24.

In OpenShift Virtualization, virt-launcher pods are the pods that run VMs. Multus gives virt-launcher pods multi-interface capability — and the VM guest OS sees both interfaces through the QEMU virtual NIC emulation inside the virt-launcher.

Why OVN localnet — not Linux bridge CNI — for NovaCrestBank

Two CNI plugins can serve VM secondary networks: Linux bridge CNI and OVN-Kubernetes localnet. Linux bridge CNI creates a kernel bridge per VLAN on each node via MachineConfig — it works but does not integrate with OVN. OVN localnet creates a logical switch in OVN that maps to the physical network — it integrates fully with OCP's network control plane, supports live migration MAC mobility natively (Section 16), and allows OVN ACL-based policy enforcement alongside NetworkPolicy. For NovaCrestBank's Model B (post-NSX) target state, OVN localnet is the correct choice because it provides the foundation for OVN-based policy enforcement without NSX.

03 — OVN-Kubernetes localnet — connecting VMs to physical VLANs

OVN-Kubernetes localnet is a topology mode where an OVN logical switch port is mapped directly to a physical network through a node's Linux bridge. Frames entering and leaving a localnet port bypass GENEVE tunnelling entirely — they are raw 802.1Q-tagged Ethernet frames on the physical VLAN.

How localnet differs from the standard OVN overlay

In the standard OVN overlay, traffic between pods on different nodes travels through GENEVE tunnels. The physical network carries only encapsulated tunnel frames — the pods' MAC addresses and IPs are invisible to the switch. In localnet, there is no tunnel. A VM with a localnet interface on VLAN 130 sends 802.1Q-tagged frames with VLAN 130 directly onto the bond sub-interface. The Cisco leaf switch sees the VM's MAC address as if the VM were a physical server plugged into the switch port.

04 — Two models — Model A (NSX present) and Model B (OVN-only)

NovaCrestBank's VIRT series spans a transition from a full NSX-T environment to a post-NSX OVN-Kubernetes-only architecture. VM networking must work correctly in both states. The critical design constraint: VM YAML does not change between Model A and Model B.

Model A — NSX present (now through VIRT-11)

When active: All guides from VIRT-05 through VIRT-10. NSX remains operational.

  • VM secondary interfaces on VLAN 130 / 140 via OVN localnet
  • NSX T0 gateway holds BGP sessions to leaf switches
  • NSX T0 advertises 10.130.0.0/24 and 10.140.0.0/24 to leaves
  • NSX gateway firewall enforces inter-zone policy (VLAN 130 ↔ 70, 130 ↔ 140)
  • OVN ACLs handle intra-VLAN traffic only

Advantage: Existing NSX DFW policy applies to migrated VMs immediately. No policy re-implementation at migration time.

Model B — OVN-only (after VIRT-11 NSX decommission)

When active: After VIRT-11 completes. NSX is fully decommissioned.

  • Same VM secondary interfaces — no VM reconfiguration
  • OVN EgressGateway holds BGP sessions to leaf switches
  • Kubernetes NetworkPolicy enforces all inter-zone policy
  • Leaf switches route directly to 10.130.0.0/24 and 10.140.0.0/24
  • No NSX licensing cost

Advantage: Single control plane for all policy — containers and VMs. No NSX operational overhead.

The key architectural decision — VM YAML is identical in both models

The VirtualMachine CR, NetworkAttachmentDefinition, and NodeNetworkConfigurationPolicy are identical in Model A and Model B. The VM's VLAN 130 secondary interface works the same way regardless of who enforces inter-zone policy. Transitioning from Model A to Model B is a control-plane-only operation: BGP sessions move from NSX Edge nodes to OVN EgressGateway nodes, and DFW rules are translated to NetworkPolicy objects. No VM is stopped, reconfigured, or migrated. This is the correct architecture for a phased platform transition in a banking environment.

05 — Model A architecture — traffic flow during the NSX transition period

Trace exactly how a VM on VLAN 130 reaches DC01 on VLAN 70 during the Model A period. This trace is what confirms that no additional firewall rules need to be configured in OCP — NSX already handles the inter-zone policy.

  1. VM sends packet: vm-banking-app-01 (10.130.0.50) sends LDAP request to DC01 (10.70.0.10). The packet exits eth1 (VLAN 130 secondary interface), tagged VLAN 130, through bond0.130 on the bare-metal node, out the Cisco leaf trunk port.
  2. Leaf routes the packet: Leaf-01 receives the frame on the bare-metal node's port-channel. Destination 10.70.0.10 is looked up in the routing table. The route to 10.70.0.0/24 is via the NSX T0 gateway (ASN 65010, learned via BGP on VLANs 102/103).
  3. NSX gateway firewall evaluates: The NSX T0 receives the packet at the uplink interface. The gateway firewall evaluates it against the configured rules. The existing rule permitting LDAP (TCP 389) and LDAPS (TCP 636) from the OCP network zone to the Infrastructure zone matches — the packet is permitted.
  4. NSX T1 delivers to DC01: NSX T1-Infra routes the packet to seg-infra-services and delivers it to DC01 at 10.70.0.10.
  5. Return path: DC01 replies to 10.130.0.50. NSX T0 has a BGP route to 10.130.0.0/24 (advertised by leaf-01 and leaf-02). NSX forwards to the correct leaf, which delivers to the bare-metal node, and OVN localnet delivers to the VM's eth1 interface.
NSX gateway firewall — not DFW microsegementation — for bare-metal VMs

NSX Distributed Firewall (DFW) microsegementation requires the NSX guest introspection agent running inside the hypervisor (ESXi) where the VM runs. OCP Virt VMs run on RHCOS bare-metal nodes — not ESXi. NSX cannot introspect into KubeVirt VMs. Therefore, Model A policy enforcement uses NSX gateway firewall rules at the T0/T1 boundary (applied to all traffic crossing between zones) rather than per-VM DFW rules. This provides zone-level segmentation (VLAN 130 ↔ VLAN 70) but not per-VM microsegementation. Per-VM policy enforcement in Model B uses Kubernetes NetworkPolicy, which does work with virt-launcher pods. PCI-DSS v4.0 Req 1.3.2

06 — NodeNetworkConfigurationPolicy — VLAN 130 bridge on bare-metal nodes

The NodeNetworkConfigurationPolicy (NNCP) is a Kubernetes CR that applies NMState configuration to a set of nodes declaratively. It is the operational successor to the one-time NMState apply in VIRT-02 — where VIRT-02 applied NMState once during provisioning to set static IPs, an NNCP is continuously reconciled. If a node's network configuration drifts, the NNCP controller re-applies the desired state automatically.

What the NNCP creates — the Linux bridge

The NNCP creates a Linux bridge named br-vlan130 on every bare-metal node and makes bond0.130 (the VLAN-tagged sub-interface) a port of that bridge. OVN-Kubernetes localnet connects its logical switch to this bridge. When OVN delivers a frame for a VM on VLAN 130, it goes into the bridge, out bond0.130, tagged VLAN 130, and onto the physical fabric. When the leaf switch sends a frame to the VM, it arrives tagged VLAN 130 on bond0.130, enters the bridge, and OVN delivers it to the correct VM tap interface.

nncp-vlan130.yaml — NodeNetworkConfigurationPolicy — every field annotated
apiVersion: nmstate.io/v1
kind: NodeNetworkConfigurationPolicy
metadata:
  name: vlan130-bridge-policy
    # Cluster-scoped (no namespace). Name identifies the VLAN and purpose.
spec:
  nodeSelector:
    workload-type: virtualization
        # Target only the 5 bare-metal nodes — not ESXi-hosted OCP workers.
    # ESXi workers do not have bond0.130 — applying there would fail and
    # the NNCP would report Degraded. The nodeSelector prevents this.
  desiredState:
    interfaces:
      - name: br-vlan130
        type: linux-bridge
                # linux-bridge: a standard Linux kernel bridge.
        # Acts as a Layer 2 switch between OVN's localnet port and bond0.130.
        # OVN injects VM frames into the bridge; the bridge forwards them
        # to bond0.130 which tags them VLAN 130 and sends them to the leaf switch.
        # The bridge name (br-vlan130) must match the NAD in Section 07.
        state: up
        bridge:
          options:
            stp:
              enabled: false
                            # STP MUST be disabled for VM secondary bridges.
              # Spanning Tree Protocol places new bridge ports in blocking state
              # for 30 seconds while it checks for loops. When live migration
              # completes and the virt-launcher pod on the destination node attaches
              # a new tap interface to this bridge, STP would block that tap for
              # 30 seconds — the VM appears to lose VLAN 130 connectivity for 30s.
              # This is unacceptable for payment-processing VMs with sub-second SLOs.
              # There is no loop risk here: the bridge has one uplink (bond0.130)
              # and multiple VM tap interfaces — no redundant paths.
          port:
            - name: bond0.130
              # bond0.130: the VLAN 130 sub-interface on the LACP bond.
            # Created in VIRT-02 NMState configuration. Adding it here as a
            # bridge port connects the bridge to the physical VLAN 130 segment.
        ipv4:
          enabled: false
                    # The bridge has no IP address. IPs belong to VMs connected to it.
          # A bridge with an IP would cause the RHCOS node to respond to ARP
          # requests on VLAN 130 — potentially intercepting VM traffic.
          # Always disable IP on VM networking bridges.
        ipv6:
          enabled: false

07 — NetworkAttachmentDefinition — VLAN 130 — every field annotated

The NetworkAttachmentDefinition (NAD) is the Multus object that declares a named secondary network. A VirtualMachine CR references the NAD by name to request a secondary interface. Every field in the NAD controls how that interface behaves — get one wrong and the interface either does not attach, allows MAC spoofing, or breaks live migration.

nad-vlan130.yaml — NetworkAttachmentDefinition — every field annotated
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: vlan130-primary
    # Namespace-scoped. VMs reference this NAD by name within the same namespace.
  # Create one NAD per namespace that needs the VLAN 130 network.
  namespace: virt-workloads
  annotations:
    k8s.v1.cni.cncf.io/resourceName: bridge.network.kubevirt.io/br-vlan130
        # resourceName: tells OCP Virt's scheduler which bridge this NAD uses.
    # The scheduler ensures VMs referencing this NAD are only placed on nodes
    # where br-vlan130 exists — all 5 bare-metal nodes after the NNCP applied.
    # Format: bridge.network.kubevirt.io/ + bridge name (must match NNCP bridge name).
spec:
  config: |
    {
      "cniVersion": "0.3.1",
            # CNI spec version 0.3.1 — current stable, compatible with all plugins used.
      "name": "vlan130-primary",
            # Must match metadata.name. Used in Multus logs and in the VMI status annotation.
      "type": "cnv-bridge",
            # type: "cnv-bridge" — OCP Virtualization bridge CNI plugin.
      # cnv-bridge wraps the Linux bridge CNI with two critical additions:
      # 1. Live migration MAC mobility: the VM's MAC address is preserved when
      #    virt-launcher moves to a new node. The bridge CNI plugin knows to
      #    send a gratuitous ARP on behalf of the VM after migration completes.
      # 2. OCP Virt admission webhook integration for interface validation.
      # Do NOT use type: "bridge" (raw Linux bridge CNI) for VM secondary networks.
      # Raw bridge CNI does NOT support MAC mobility — live migration changes the
      # VM's MAC address and causes 30-300 seconds of network disruption until
      # the upstream ARP cache expires and the new MAC is learned.
      "bridge": "br-vlan130",
            # Bridge name created by the NNCP in Section 06. Exact match required.
      # The CNI plugin connects the VM's tap interface to this Linux bridge.
      # The bridge then forwards frames to bond0.130 → VLAN 130 on the leaf switch.
      "macspoofchk": true,
            # MAC spoofing protection — a compromised VM may attempt to send frames
      # with a source MAC address that is not its own (to impersonate another VM
      # or host on VLAN 130). macspoofchk: true configures the Linux bridge to
      # drop any frame from the VM's tap interface whose source MAC does not match
      # the MAC assigned to that tap by KubeVirt.
      # PCI-DSS v4.0 Req 1.3.3 requires preventing unauthorised source address use.
      # Only disable macspoofchk for interfaces that legitimately use multiple MACs
      # (e.g., SR-IOV bonded VFs or nested virtualisation — neither applies here).
      "ipam": {}
            # ipam: {} — empty. The guest OS manages its own IP on this interface.
      # cloud-init assigns the static IP (10.130.0.50/24) to eth1 at boot (Section 09).
      # If ipam were populated (e.g., host-local IPAM with a CIDR), Multus would try
      # to assign an IP to the tap interface on the host side — conflicting with
      # cloud-init's IP assignment in the guest and causing duplicate IP errors.
      # Always use ipam: {} for VM secondary networks where the guest owns its IP.
    }

08 — NetworkAttachmentDefinition — VLAN 140

The VLAN 140 NAD is structurally identical to VLAN 130. Two values change: the name, and the bridge reference. Every design decision — type: cnv-bridge, macspoofchk: true, ipam: {} — applies identically for the same reasons documented in Section 07.

nad-vlan140.yaml — NetworkAttachmentDefinition for VLAN 140
apiVersion: k8s.cni.cncf.io/v1
kind: NetworkAttachmentDefinition
metadata:
  name: vlan140-dmz
  namespace: virt-workloads
  annotations:
    k8s.v1.cni.cncf.io/resourceName: bridge.network.kubevirt.io/br-vlan140
spec:
  config: |
    {
      "cniVersion": "0.3.1",
      "name": "vlan140-dmz",
      "type": "cnv-bridge",
      "bridge": "br-vlan140",
      "macspoofchk": true,
      "ipam": {}
    }
Why VLAN 130 and VLAN 140 are separate NADs — not a single NAD with VLAN tagging

Some CNI plugins support specifying a VLAN ID inside the NAD config — creating one NAD that handles multiple VLANs. The cnv-bridge plugin does not use this model. Each VLAN gets its own bridge (br-vlan130, br-vlan140) and its own NAD. This is intentional: separate bridges provide separate Layer 2 domains. A VM connected to vlan130-primary can only communicate with other VLAN 130 devices at Layer 2. There is no accidental Layer 2 adjacency between VLAN 130 and VLAN 140 VMs — inter-VLAN traffic must go through the router (leaf SVI and NSX gateway in Model A, or leaf SVI and OVN EgressGateway in Model B), where policy is enforced.

09 — Adding a secondary network interface to a VirtualMachine CR

With the NNCP (bridge on node) and NAD (named network) in place, adding a secondary VLAN 130 interface to a VM requires two additions to the VirtualMachine CR: an entry in spec.template.spec.networks declaring the network source, and an entry in spec.template.spec.domain.devices.interfaces declaring the interface type. The cloud-init userData configures the guest OS side of the interface.

vm-banking-app-01.yaml — relevant sections showing secondary interface
# ── inside spec.template.spec.domain.devices.interfaces ─────────────────
interfaces:
  - masquerade: {}
    name: default
        # Primary interface — OVN overlay. Unchanged from VIRT-04 test VM.
    # Provides cluster-internal access: OCP API, DNS, monitoring, cert-manager.

  - bridge: {}
    name: vlan130-nic
    macAddress: "02:00:00:0a:82:32"
        # bridge: {} — interface type for OVN localnet secondary networks.
    # Tells KubeVirt to create a tap interface and connect it to the bridge
    # specified in the NAD (br-vlan130 via the cnv-bridge CNI plugin).
    #
    # macAddress: "02:00:00:0a:82:32" — MANDATORY for live migration correctness.
    # If macAddress is omitted, KubeVirt assigns a random MAC at each VMI start.
    # The destination VMI (started on the new node during migration) gets a
    # different random MAC. After migration, the VM has a new MAC — the upstream
    # switch still has the old MAC in its ARP cache → traffic blackhole for
    # up to 300 seconds until ARP cache expires.
    # With an explicit MAC, both source and destination VMI use the same MAC.
    # The gratuitous ARP (sent by cnv-bridge after migration) updates the switch's
    # MAC table immediately — zero traffic loss.
    #
    # MAC convention: 02:00:00: prefix (locally administered, unicast)
    # + last 3 octets of VM IP in hex: 10.130.0.50 → 0a:82:32
    # This convention makes it trivial to derive the VM's IP from its MAC
    # and vice versa — valuable during network troubleshooting.

# ── inside spec.template.spec.networks ──────────────────────────────────
networks:
  - name: default
    pod: {}
        # pod: {} — primary interface uses the pod overlay network (OVN).

  - name: vlan130-nic
    multus:
      networkName: vlan130-primary
            # networkName: the NAD name in the same namespace.
      # Multus looks up this NAD, reads its CNI config (cnv-bridge / br-vlan130),
      # and calls the cnv-bridge plugin to create the secondary tap interface.
      # The VM guest OS sees this as eth1 — a standard NIC.

# ── cloud-init userData — configures eth1 in the guest OS ────────────────
volumes:
  - name: cloudinitdisk
    cloudInitNoCloud:
      userData: |
        #cloud-config
        hostname: vm-banking-app-01
        fqdn: vm-banking-app-01.novacrestbank.local
        network:
          version: 2
          ethernets:
            eth0:
              dhcp4: false
            eth1:
              dhcp4: false
              addresses:
                - 10.130.0.50/24
              gateway4: 10.130.0.3
              nameservers:
                addresses: [10.70.0.10, 10.70.0.11]
                search: [novacrestbank.local]
            # cloud-init configures eth1 on first boot:
      # - Static IP 10.130.0.50/24 — the VM's production IP on VLAN 130
      # - gateway4: 10.130.0.3 — HSRP VIP on leaf-01/02 for VLAN 130 (VIRT-01)
      # - nameservers: DC01 and DC02 — the domain DNS servers
      # - search: novacrestbank.local — short names resolve in the domain
      # eth0 is set to dhcp4: false — the OVN overlay assigns its IP internally,
      # not through DHCP. Cloud-init must not interfere with eth0.

12 — NetworkPolicy for VM workloads — Model B intra-cluster policy

In Model B, Kubernetes NetworkPolicy enforces policy on virt-launcher pod traffic within the OCP cluster. NetworkPolicy applies to pods — virt-launcher pods are pods — so the same mechanism works for VMs. However, there is an important scope clarification: NetworkPolicy controls traffic on the pod's primary interface (eth0, OVN overlay). 13 — EgressGateway — replacing NSX T0 BGP in Model B

In Model A, the NSX T0 gateway (ASN 65010) holds BGP sessions to leaf-01 (ASN 65001) and leaf-02 (ASN 65002) and advertises 10.130.0.0/24 and 10.140.0.0/24. After NSX decommission, those BGP sessions terminate — the leaves lose their routes to VM subnets. OVN-Kubernetes EgressGateway takes over this function.

EgressGateway designates specific OCP nodes as egress points for specific subnets and establishes BGP peering from those nodes to the leaf switches using the same ASN (65010) that NSX T0 used. From the leaf switch's perspective, the BGP peer IP changes from the NSX Edge node IP to an OCP node IP — but the routes advertised (10.130.0.0/24, 10.140.0.0/24) and the ASN (65010) remain the same. The BGP reconfiguration on the leaves is a peer address change, not a routing topology change.

EgressGateway is configured in VIRT-11 — not here

EgressGateway configuration happens during the NSX decommission process in VIRT-11. The BGP peer configuration on leaf-01 and leaf-02 must be updated (old peer: NSX Edge node IPs on VLAN 102/103; new peer: OCP EgressGateway node IPs on VLAN 110) in a coordinated cutover that takes the BGP session down for less than 30 seconds. VIRT-11 documents this step-by-step. This section exists so that the network planning team can verify VIRT-11's EgressGateway plan against the current Model A BGP configuration.

14 — SR-IOV — what it is and when to use it

SR-IOV (Single Root I/O Virtualization) is a PCIe hardware feature that allows one physical NIC to present multiple independent virtual NICs (called Virtual Functions, or VFs) to the operating system. Each VF can be assigned exclusively to a VM, giving the VM direct, kernel-bypass access to the NIC hardware without going through OVN-Kubernetes or the Linux network stack.

The trade-off — performance vs portability

SR-IOV VMs have significantly lower network latency and higher throughput than OVN localnet VMs at very high packet rates, because the VM's NIC driver writes directly to the NIC's hardware ring buffers. However, SR-IOV VMs cannot be live-migrated — the VF is bound to a specific physical NIC on a specific node. If the node running an SR-IOV VM needs maintenance, the VM must be cold-migrated (stopped, moved, restarted). For most banking application VMs, OVN localnet is the correct choice. SR-IOV is reserved for:

Use SR-IOVUse OVN localnet
High-frequency trading (HFT) — sub-100μs latency requirementPayment processing VMs — live migration required for maintenance
DPDK workloads — require userspace NIC accessCore banking VMs — NetworkPolicy enforcement needed
RDMA (database replication over InfiniBand or RoCE)All VMs where live migration is a requirement
>500k packets per second where OVN overhead is measurableDevelopment and test VMs — operational simplicity preferred

sriov-policy.yaml — SriovNetworkNodePolicy for Mellanox ConnectX-7 on VLAN 130
apiVersion: sriovnetwork.openshift.io/v1
kind: SriovNetworkNodePolicy
metadata:
  name: ncb-sriov-vlan130
  namespace: sriov-network-operator
spec:
  resourceName: ncb_sriov_vlan130
    # resourceName: exposed to the Kubernetes device plugin.
  # VMs request SR-IOV VFs by adding this resource name to their resource limits.
  nodeSelector:
    workload-type: virtualization
  numVfs: 8
    # 8 VFs per matching NIC per node. Mellanox ConnectX-7 supports up to 127.
  # 8 is conservative — sufficient for 2-4 HFT VMs per node with VF headroom.
  nicSelector:
    vendor: "15b3"
        # 15b3 = Mellanox Technologies (NVIDIA). PCI vendor ID.
    # Verify: oc debug node/bm-virt-01 -- chroot /host lspci -nn | grep -i mellanox
    deviceID: "101b"
        # 101b = ConnectX-7 device ID (MT2910).
    # Verify: lspci -nn | grep -i connectx — look for [15b3:XXXX] at end of line.
  deviceType: netdevice
    # netdevice: kernel driver mode — VF appears as a standard Linux netdev in the guest.
  # Use vfio-pci for DPDK workloads (passes VF to userspace without kernel driver).
  vlan: 130
    # Hardware port VLAN — each VF's port is tagged to VLAN 130 at the NIC level.
  # The VM guest sees an untagged interface; VLAN tagging happens in hardware.
  # This is different from OVN localnet where the LACP bond sub-interface carries the tag.
SR-IOV VMs cannot be live-migrated — plan maintenance accordingly

A VF is a hardware resource bound to a specific physical NIC on a specific node. KubeVirt cannot move a VF-backed interface to another node during live migration — the VF simply does not exist on the destination node. Attempting to live-migrate an SR-IOV VM results in a migration failure. When a bare-metal node running SR-IOV VMs requires maintenance (firmware update, hardware replacement), those VMs must be cold-migrated: gracefully shut down within the guest OS, then recreated on another node. Plan SR-IOV VM placement to minimise maintenance disruption — for example, spread HFT VMs across multiple nodes so that maintaining one node takes down only a subset of HFT capacity.

16 — MAC mobility — how live migration preserves VM network identity

When a VM is live-migrated from bm-virt-02 to bm-virt-04, the Cisco leaf switch still has its MAC address table entry pointing to Po212 (bm-virt-02's port-channel). Traffic destined for the VM's MAC is forwarded to the wrong node until the switch updates its MAC table. Without an active mechanism to update it, this causes a traffic blackhole lasting until the MAC table entry ages out — up to 300 seconds depending on the switch configuration.

The gratuitous ARP mechanism

When migration completes and the VM is fully running on bm-virt-04, the cnv-bridge CNI plugin (the reason for choosing type: cnv-bridge over type: bridge in the NAD) sends a gratuitous ARP on behalf of the VM. A gratuitous ARP is an ARP reply sent without a prior ARP request — its purpose is to update the ARP caches of other devices on the same VLAN and trigger MAC table updates in switches.

The gratuitous ARP contains: sender MAC = 02:00:00:0a:82:32 (the VM's MAC), sender IP = 10.130.0.50. When leaf-01 receives this frame on Po214 (bm-virt-04's port-channel), it updates its MAC address table: 02:00:00:0a:82:32 is now on Po214. All subsequent frames destined for 10.130.0.50 are forwarded to bm-virt-04. The update is instantaneous — zero traffic loss for the VM's VLAN 130 connectivity.

17 — Migration network — bandwidth analysis and architecture decision

KubeVirt performs live migration memory transfer over the pod primary network by default — VLAN 110 for the bare-metal nodes. During migration, guest memory pages are streamed from source to destination node at up to bandwidthPerMigration (64 MiB/s, set in the HyperConverged CR in VIRT-04). The question is whether this migration traffic requires a dedicated network interface.

Dedicated migration network vs default primary network for NovaCrestBank

Decision: default primary network (VLAN 110). No dedicated migration VLAN.

Bandwidth analysis: VLAN 110 is the LACP bond (25 Gbps = ~3 GiB/s usable). Migration bandwidth limit: 5 concurrent migrations × 64 MiB/s = 320 MiB/s ≈ 2.6 Gbps. OCP cluster traffic (API calls, etcd, pod-to-pod): typically 100–500 Mbps. Total peak: ~3.1 Gbps — within the 3 GiB/s capacity with moderate headroom.

Rejected: dedicated migration VLAN on VLAN 120. VLAN 120 is reserved for ODF Ceph replication. During a node drain (the most common cause of multiple simultaneous migrations), Ceph also begins OSD rebalancing on VLAN 120. Combining migration traffic with Ceph rebalancing on VLAN 120 would create contention at the worst moment. Keeping migration on VLAN 110 separates the traffic concerns cleanly.

Revisit when: VM density increases such that parallelMigrationsPerCluster must exceed 5, or if migration latency begins to impact the P99 latency SLOs of VMs sharing VLAN 110 during maintenance drains.

18 — DNS for VMs — the four-element hostname chain

A VM on VLAN 130 with IP 10.130.0.50 is not automatically registered in DNS. DNS records were pre-created in VIRT-02b Section 06. This section verifies the complete four-element hostname chain that must be consistent for a VM to have correct DNS identity — especially for Windows domain join and Kerberos authentication.

The four elements — all must agree

  1. DNS A record: vm-banking-app-01.novacrestbank.local → 10.130.0.50
  2. DNS PTR record: 10.130.0.50 → vm-banking-app-01.novacrestbank.local
  3. cloud-init hostname: hostname: vm-banking-app-01 and fqdn: vm-banking-app-01.novacrestbank.local
  4. Guest OS hostname: What hostname -f reports inside the guest after cloud-init runs

If any element disagrees — for example, cloud-init sets the hostname to vm-banking-app-01 but the DNS PTR record points to vm-app-01.novacrestbank.local — Active Directory domain join fails. The DC performs a forward and reverse DNS lookup of the joining computer's name before accepting the join request, and any mismatch is rejected.
