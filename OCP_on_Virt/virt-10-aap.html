01 — What AAP is — from first principles

Ansible Automation Platform (AAP) is Red Hat's enterprise automation framework. At its core, Ansible is an agent-less configuration management and orchestration tool: it connects to target systems over SSH (Linux) or WinRM (Windows), executes tasks described in YAML files called playbooks, and leaves the target in the desired state. Agent-less means nothing needs to be installed on the target — only Python and SSH access are required on Linux targets; WinRM must be enabled on Windows targets.

AAP adds the enterprise layer on top of open-source Ansible: Automation Controller (the web UI and REST API, formerly AWX/Tower), Automation Hub (content management for Ansible Collections), containerised Execution Environments (Python environments that run playbooks reproducibly), and Event-Driven Ansible (EDA — triggering playbooks automatically from external events such as Alertmanager webhooks).

Where AAP fits in the VIRT automation stack

Three tools manage NovaCrestBank's VM workloads. Each operates at a different layer.

ToolWhat it managesWhat it cannot do
Argo CD (VIRT-07)The VirtualMachine CR — the Kubernetes definition. Ensures the CR matches Git.Cannot touch the guest OS inside the VM.
ACM (VIRT-09)The cluster fleet — governance policies, cluster provisioning, fleet observability.Cannot configure what runs inside a VM guest.
AAP (VIRT-10)The guest OS — packages, services, users, firewall, AD join, compliance, patching. Also orchestrates OCP API calls when external conditions (Alertmanager alerts) trigger VM lifecycle operations.Cannot manage Kubernetes resources declaratively (that is Argo CD's role).
The boundary: Argo CD owns the VM definition, AAP owns the guest OS

Argo CD creates and maintains the VirtualMachine CR. AAP connects to the running VM via SSH and configures what runs inside it. Both tools act on the same VM simultaneously without conflict because they operate at different levels: Argo CD at the Kubernetes API level, AAP at the guest OS SSH level. Neither needs to know the other exists to do its job correctly.

02 — Playbook anatomy — every directive explained

A playbook is a YAML file describing automation work. The goal of this section is that after reading it, you can write a playbook from a blank file and make an independent decision about every line — not just copy from examples. Every structural keyword is explained with its valid values, its effect, and when to use which option.

complete-playbook-skeleton.yaml — every directive annotated
---
# --- marks the start of a YAML document. Optional but conventional for playbooks.
# A playbook is a LIST of plays. Each play is a YAML dict (mapping of keys to values).
# The leading hyphen (-) marks each play as a list item.

- name: Descriptive name for this play
  # name: (string) Human-readable label shown in output and AAP job logs.
  # Not a variable — purely descriptive. Best practice: describe WHAT the play does.
  # Example: "Install and configure qemu-guest-agent on banking VMs"

  hosts: virt_vms
  # hosts: (string or list) Which machines this play targets.
  # Sources: inventory file, dynamic inventory plugin, or extra_vars.
  # Values:
  #   all              → every host in inventory
  #   virt_vms         → group named virt_vms in inventory
  #   10.130.0.50      → specific IP address
  #   localhost        → run on the AAP execution node (no SSH to remote host)
  #   "web:&db"        → intersection: hosts in BOTH web AND db groups
  #   "all:!dev_vms"   → all hosts EXCEPT those in dev_vms group
  # In NovaCrestBank: kubevirt.io dynamic inventory plugin populates virt_vms
  # with the VLAN 130 IPs of all Running VMIs in the virt-workloads namespace.

  gather_facts: true
  # gather_facts: (bool, default true) Run the 'setup' module on each target BEFORE tasks.
  # The setup module collects OS facts — variables prefixed with ansible_:
  #   ansible_os_family        → "RedHat" (RHEL/CentOS/Rocky) or "Windows" or "Debian"
  #   ansible_distribution     → "RedHat", "CentOS", "Ubuntu", "Windows"
  #   ansible_distribution_version → "9.3"
  #   ansible_memtotal_mb      → 32768
  #   ansible_processor_cores  → 8
  #   ansible_hostname         → "vm-banking-app-01"
  #   ansible_default_ipv4.address → primary IP
  # Use these in when: conditions to make one playbook work across multiple OS types.
  # Set gather_facts: false when:
  #   - Target is not yet reachable (pre-boot provisioning play)
  #   - Play only uses Kubernetes API via delegate_to: localhost
  #   - Speed is critical and facts are not needed (saves 2-5 seconds per host)

  become: true
  # become: (bool, default false) Escalate privileges via sudo/su/pbrun.
  # Required for tasks needing root: package install, /etc/ edits, service restart.
  # The user escalated TO is controlled by become_user (default: root).
  # become: false is correct for tasks that run as the SSH user without escalation.
  # On Windows: do not use become — connect with admin WinRM credentials directly.

  become_user: root
  # become_user: (string, default "root") Which user to escalate to.
  # Can be overridden per-task: e.g., become_user: appuser for app-specific tasks.

  connection: ssh
  # connection: (string, default "ssh") How Ansible connects to the target.
  # ssh     → standard SSH (Linux/UNIX, default)
  # winrm   → Windows Remote Management (required for all Windows targets)
  # local   → run tasks on the controller itself (no remote connection)
  # kubectl → exec into a Kubernetes pod
  # smart   → SSH if OpenSSH supports ControlMaster, else paramiko

  vars:
    # vars: (dict) Play-level variables. Available to ALL tasks in this play.
    # Variable precedence (lowest to highest):
    #   role defaults → inventory group_vars → inventory host_vars
    #   → playbook vars → task vars → set_fact → extra_vars
    # extra_vars (AAP Survey, --extra-vars CLI) ALWAYS wins over everything.
    # Use vars for constants that might change between runs or environments.
    vm_name: "vm-banking-app-01"
    vm_namespace: "virt-workloads"
    vm_ip: "10.130.0.50"
    ad_domain: "novacrestbank.local"

  vars_files:
    # vars_files: (list of paths) Load variables from external YAML files.
    # Loaded in order — later files override earlier ones.
    # Separate variable definitions from playbook logic: cleaner, reusable.
    - vars/ncb_common.yml
    - vars/banking_vm.yml

  serial: 2
  # serial: (int or string or list) Run the play on N hosts at a time.
  # Without serial: all hosts in the play run ALL tasks in parallel.
  # serial: 2 → process 2 hosts, wait for completion, then process next 2.
  # serial: "20%" → process 20% of inventory at a time.
  # serial: [1, 5, 10] → ramp up: 1 host first, then 5, then 10.
  # Use for: rolling patches, rolling restarts, avoiding thundering herd on APIs.
  # Banking VM patching: serial: 1 for payment VMs, serial: 5 for dev VMs.

  max_fail_percentage: 20
  # max_fail_percentage: (int 0-100) Abort the play if this % of hosts fail.
  # Prevents a bad patch from cascading: if 3/10 VMs fail (30% > 20%), stop.
  # 0 → abort on the first failure.
  # 100 → never abort (run to completion regardless of failures).
  # Works with serial: applies the threshold per serial batch.

  tasks:
    # tasks: (list) The ordered list of operations to perform on each host.
    # Tasks run IN ORDER, one at a time per host (unless async is used).
    # If a task fails: that host is removed from the play (no more tasks run on it)
    # unless ignore_errors: true is set on that task.
    # Each task is a dict with: name + one module call + optional directives.

        # ── TASK: package installation ────────────────────────────────────
        - name: Install required packages
      ansible.builtin.package:
        # The module name IS the key in the task dict.
        # Format: collection.namespace.module_name
        # ansible.builtin → core modules shipped with Ansible
        # community.general → community-maintained general-purpose modules
        # kubernetes.core → Kubernetes API modules
        # netapp.ontap → NetApp ONTAP/AFF modules
        # ansible.windows → Windows-specific modules
        # f5networks.f5_modules → F5 BigIP modules
        name:
          # Module OPTIONS are indented under the module name key.
          # Each module has its own set of options — documented at docs.ansible.com.
          # name: which package(s) to manage. Accepts a string or list.
          - qemu-guest-agent
          - rsyslog
          - auditd
        state: present
          # state: desired state for the package.
          # present → install if not installed (idempotent: safe to re-run)
          # latest  → install AND upgrade to latest available version
          # absent  → remove the package if installed
          # Idempotency is fundamental to Ansible: every module checks current state
          # before acting. If state matches desired: report "ok" and do nothing.
          # If state differs: make the change and report "changed".
      notify: restart qemu-guest-agent
      # notify: (string or list) Trigger a HANDLER when this task reports "changed".
      # If the packages were already installed (ok): handler is NOT triggered.
      # Handlers run ONCE at the END of the play, even if notified multiple times.
      # Handler name must EXACTLY match the string in the handlers: section.
      # Use for: restart a service after its config changes, reload after install.

        # ── TASK: register task output ───────────────────────────────────
        - name: Query VMI status from Kubernetes API
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        name: "{{ vm_name }}"
          # {{ var_name }} is Jinja2 template syntax — Ansible's variable substitution.
          # At runtime, {{ vm_name }} becomes the value of the vm_name variable.
          # Jinja2 also supports filters (pipe operator):
          #   {{ vm_name | upper }}                     → "VM-BANKING-APP-01"
          #   {{ mem_gib | int * 1024 }}                → integer arithmetic
          #   {{ var | default('fallback') }}           → use fallback if undefined
          #   {{ var | default(omit) }}                 → omit the option if undefined
          #   {{ list | selectattr('key','eq','val') }} → filter list by attribute
          #   {{ ip.split('.')[-1] }}                   → last octet of an IP
        namespace: "{{ vm_namespace }}"
      register: vmi_info
      # register: (string) Save the module's return value into a variable.
      # Every module returns a dict. Common fields:
      #   changed: bool   — did the module change anything?
      #   failed: bool    — did the module fail?
      #   msg: string     — human-readable message
      #   rc: int         — return code (command/shell modules)
      #   stdout: string  — command output (command/shell modules)
      #   resources: list — Kubernetes objects (k8s_info module)
      #   json: dict      — parsed JSON response body (uri module)
      # Access registered values in subsequent tasks:
      #   vmi_info.resources[0].status.phase
      #   vmi_info.resources[0].status.interfaces[1].ipAddress
      delegate_to: localhost
      # delegate_to: (string) Run THIS task on a DIFFERENT host than the play's hosts.
      # localhost → run on the AAP execution node (the Ansible controller).
      # Why: kubernetes.core.k8s_info talks to the Kubernetes API.
      # The API is reachable from the AAP node — not from inside the VM guest OS.
      # Without delegate_to: Ansible tries to run the module inside the VM via SSH
      # where kubectl/k8s libraries are not installed.
      # delegate_to overrides the connection for this one task only.
      run_once: true
      # run_once: (bool) Run this task on ONLY the first host in the play.
      # Used when a task produces shared data (e.g., a single API call that
      # returns information valid for all hosts). The result is shared via hostvars.

        # ── TASK: conditional execution ──────────────────────────────────
        - name: Join Active Directory (RHEL only)
      ansible.builtin.command:
        cmd: "realm join --user=svc-ocp-virt-mgmt {{ ad_domain }}"
        stdin: "{{ ad_password }}"
          # command: run a single command on the target.
          # Does NOT use a shell (no pipes, no redirects, no shell variables).
          # Safer than shell for most cases — use shell only when pipes/redirects needed.
          # stdin: feed a string to the command's standard input (for password prompts).
      when: ansible_os_family == "RedHat"
      # when: (Jinja2 expression) Task runs ONLY when the expression evaluates to True.
      # ansible_os_family is an Ansible fact collected by gather_facts.
      # "RedHat" matches RHEL, CentOS, Rocky Linux, AlmaLinux, Fedora.
      # "Windows" matches Windows Server 2019/2022/2025.
      # "Debian" matches Ubuntu, Debian.
      # Multiple conditions:
      #   when: ansible_os_family == "RedHat" and vm_ip is defined
      #   when: ansible_os_family in ["RedHat", "Debian"]
      #   when: not (skip_ad_join | default(false))
      #   when: result.rc != 0
      no_log: true
      # no_log: (bool) Suppress this task's output and arguments from ALL logs.
      # Required whenever a task references a credential (password, token, key).
      # Without no_log: the password appears in AAP job logs — PCI-DSS violation.
      # Apply to any task that has a secret in cmd, stdin, body, or module options.
      changed_when: false
      # changed_when: (Jinja2 expression or bool) Override when a task reports "changed".
      # The command module always reports changed=true because it ran a command.
      # changed_when: false → always report "ok" (used for read-only commands).
      # changed_when: "'Updated' in result.stdout" → changed only if output says so.
      # changed_when: result.rc != 0 → changed only on non-zero exit.
      # Correct changed reporting is essential for idempotency and handler triggering.

        # ── TASK: loops ───────────────────────────────────────────────────
        - name: Open required ports in guest firewall
      ansible.posix.firewalld:
        port: "{{ item.port }}/{{ item.proto }}"
        state: enabled
        permanent: true
          # permanent: true → write the rule to the permanent config (survives reboot).
          # Without permanent: rule exists only in the running config (lost on restart).
          # Use both permanent: true AND immediate: true for the rule to take effect
          # immediately AND persist after reboot.
        immediate: true
        zone: public
      loop:
        # loop: (list) Run this task once for each item in the list.
        # {{ item }} refers to the current element inside the task.
        # {{ item.key }} accesses a key if the item is a dict.
        # loop replaces the deprecated with_items syntax.
        # Other loop sources: loop: "{{ range(1,6) | list }}"  → [1,2,3,4,5]
        #                     loop: "{{ my_list_variable }}"   → iterate a variable
        - {port: "8443", proto: "tcp"}
        - {port: "9100", proto: "tcp"}
      loop_control:
        label: "{{ item.port }}/{{ item.proto }}"
          # loop_control.label: what to display per iteration in the output.
          # Without this: Ansible prints the full item dict — unreadable for complex objects.
          # Other loop_control options:
          #   pause: 2      → wait 2 seconds between iterations
          #   index_var: idx → create a variable with the loop iteration index (0,1,2...)
      when: ansible_os_family == "RedHat"

        # ── TASK: until/retries for polling ──────────────────────────────
        - name: Wait for VM to reach Running phase
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        name: "{{ vm_name }}"
        namespace: "{{ vm_namespace }}"
      register: vmi_status
      until:
        vmi_status.resources | length > 0 and
        vmi_status.resources[0].status.phase == "Running"
      # until: (Jinja2 expression) Retry the task until the expression is True.
      # Evaluated after every attempt. Task fails if retries are exhausted.
      # Essential for polling asynchronous operations (VM boot, PVC bind, etc.).
      retries: 30
      # retries: (int) Maximum number of retry attempts. Default: 3.
      # 30 retries × 20 seconds = up to 10 minutes waiting for the VM to boot.
      delay: 20
      # delay: (int, seconds) How long to wait between retries. Default: 5.
      delegate_to: localhost

        # ── TASK: failed_when and ignore_errors ───────────────────────────
        - name: Check if reboot is required after patching
      ansible.builtin.command:
        cmd: needs-restarting -r
      register: needs_reboot
      failed_when: false
      # failed_when: (Jinja2 expression or bool) Override when a task reports "failed".
      # failed_when: false → never fail this task, even on non-zero exit code.
      # needed here because needs-restarting exits 1 when a reboot IS required —
      # that is normal (not an error). We capture rc and evaluate it in when:.
      # Other patterns:
      #   failed_when: result.rc not in [0, 2]  → fail only on unexpected codes
      #   failed_when: "'ERROR' in result.stderr"  → fail only on specific output
      changed_when: false

        # ── TASK: set_fact, pause, debug, assert ─────────────────────────
        - name: Store derived MAC address as a fact for later use
      ansible.builtin.set_fact:
        # set_fact: create a new variable at runtime.
        # Available to all subsequent tasks in the current play.
        # Essential for storing computed values derived from registered output.
        # set_fact variables persist across plays in the same playbook run (unlike vars).
        mac_address:
          "02:00:00:{{ vm_ip.split('.')[2] | int | format('%02x') }}:{{ vm_ip.split('.')[3] | int | format('%02x') }}"

        - name: Wait 30 seconds for KubeVirt to apply the hot-plug
      ansible.builtin.pause:
        # pause: wait for a fixed duration or until the operator presses Enter.
        # seconds: N  → wait N seconds
        # minutes: N  → wait N minutes
        # prompt: "Verify the VM before continuing"  → interactive pause (not for automation)
        seconds: 30

        - name: Log the provisioning result to job output
      ansible.builtin.debug:
        # debug: print a message or variable to job output. Makes no changes.
        # msg: "string {{ variable }}"  → print a message
        # var: variable_name            → pretty-print a variable's contents
        msg: "VM {{ vm_name }} provisioned at {{ vm_ip }} — MAC {{ mac_address }}"

        - name: Assert the VM is Running after provisioning
      ansible.builtin.assert:
        # assert: fail the task (and the play) if a condition is not True.
        # Used for post-action validation — the "verify" step in every workflow.
        # that: list of Jinja2 expressions — ALL must be True to pass.
        that:
          - vmi_status.resources[0].status.phase == "Running"
        fail_msg: "VM did not reach Running phase within the timeout"
        success_msg: "VM is Running — provisioning complete"

  handlers:
    # handlers: tasks triggered by notify, run ONCE at the end of the play.
    # Even if 10 tasks notify the same handler, it runs only once.
    # Handler names must exactly match the string in notify:.
    # Handlers are deduped: notifying "restart sshd" 5 times → one restart.
    # Force handlers to run immediately: meta: flush_handlers task.
    - name: restart qemu-guest-agent
      ansible.builtin.service:
        name: qemu-guest-agent
        state: restarted
          # state: started/stopped/restarted/reloaded
          # reloaded → SIGHUP (reload config without dropping connections — use for sshd)
          # restarted → full stop + start (drops connections — use for qemu-guest-agent)
          # enabled: true/false → controls whether service starts on boot (separate from state)

03 — The real-world module catalogue

A module is the unit of work in every task. The module name is the KEY in a task dict — its name determines what action is taken. Ansible executes the module as a Python script on the target (or locally for API-based modules). Every module returns a standardised JSON result containing at minimum: changed, failed, and msg. The full return dict is available via register.

pb-netapp-pvc.yaml — NetApp AFF volume and OCP PVC — every option annotated

- name: Provision NetApp AFF data volume and bind to OCP PVC

    volume_size_gb: 200

            name: "{{ vm_name }}-data-pvc"

              - ReadWriteMany
            resources:
              requests:
                storage: "{{ volume_size_gb }}Gi"

08 — Windows Server VM migration from VCF via AAP

09 — Guest OS patching — RHEL and Windows at scale

Guest OS patching is the most frequent AAP operation for VM workloads. Without automation: 50 VMs means 50 SSH sessions, 50 manual dnf update runs, 50 individual reboots. With AAP: one Job Template run, configurable batch size via serial, pre-patch snapshots, and post-patch validation — all logged in the AAP job output for audit evidence.

pb-patch-vms.yaml — fleet OS patching with serial batches and pre-patch snapshot
---
- name: Patch RHEL VM guest OS — pre-patch snapshot, update, reboot, validate
  hosts: rhel_virt_vms
  gather_facts: true
  become: true
  serial: 2
    # serial: 2 → patch 2 VMs simultaneously, wait for completion, then patch next 2.
  # Prevents all VMs rebooting simultaneously (disrupts banking services).
  # serial: 1 for payment processing VMs (maximum safety), serial: 5 for dev VMs.
  max_fail_percentage: 20
    # max_fail_percentage: 20 → abort patching if more than 20% of VMs fail.
  # A bad patch killing 3/10 VMs (30%) stops the run before it reaches all VMs.

  tasks:

    - name: Take pre-patch VirtualMachineSnapshot via Kubernetes API
      kubernetes.core.k8s:
        state: present
        definition:
          apiVersion: snapshot.kubevirt.io/v1alpha1
          kind: VirtualMachineSnapshot
          metadata:
            name:
              "pre-patch-{{ inventory_hostname | replace('.','--') }}-{{ ansible_date_time.date }}"
            namespace: virt-workloads
          spec:
            source:
              apiGroup: kubevirt.io
              kind: VirtualMachine
              name: "{{ hostvars[inventory_hostname].vm_name }}"
                            # hostvars[inventory_hostname]: access any variable for the current host
              # from the inventory. vm_name is set in host_vars/ for each VM.
      delegate_to: localhost

    - name: Apply all available security patches via dnf
      ansible.builtin.dnf:
        name: "*"
                # "*": update ALL installed packages.
        # For security-only updates: use name: [] with security: true.
        state: latest
        security: false
                # security: false → update all packages (full quarterly patch).
        # security: true → install only RHSA (Red Hat Security Advisory) packages.
        # Use security: true for patch cycles outside maintenance windows.
        exclude:
          - kernel*
            # Exclude kernel updates — they require reboot AND pre-approval.
          # Kernel updates run via a separate Job Template with a dedicated change request.
      register: dnf_update

    - name: Check if a reboot is required
      ansible.builtin.command:
        cmd: needs-restarting -r
      register: needs_reboot
      failed_when: false
            # needs-restarting exits 1 when a reboot is required — NOT an error.
      # failed_when: false lets us capture rc=1 without the task failing.
      # Evaluate: when: needs_reboot.rc == 1 on the next reboot task.
      changed_when: false

    - name: Reboot the VM guest if required
      ansible.builtin.reboot:
        reboot_timeout: 300
        test_command: whoami
        msg: "Rebooting for post-patch restart — AAP job {{ tower_job_id }}"
      when: needs_reboot.rc == 1

    - name: Gather service facts to verify post-patch state

12 — Automated VM scale-up — CPU and memory hot-plug via AAP

An operator submits a scale-up request through the AAP Survey (entering new CPU socket count and memory in GiB). AAP validates available hugepage capacity on the node where the VM is running, applies the VirtualMachine patch, and asserts the guest OS sees the new resources — all without touching kubectl.

pb-vm-scaleup.yaml — CPU/memory hot-plug with capacity validation
---
- name: Validate capacity and apply CPU/memory hot-plug to running VM
  hosts: localhost
  gather_facts: false
  vars:
    target_cpu_sockets: "{{ new_cpu_sockets | int }}"
    target_memory_gi: "{{ new_memory_gib | int }}"

  tasks:

    - name: Get VMI details — current state and node placement
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        name: "{{ vm_name }}"
        namespace: "{{ vm_namespace }}"
      register: vmi_info

    - name: Store current node and memory as facts
      ansible.builtin.set_fact:
        vm_node: "{{ vmi_info.resources[0].status.nodeName }}"
        current_memory_gi:
          "{{ vmi_info.resources[0].spec.template.spec.domain.memory.guest
             | regex_replace('Gi','') | int }}"
                  # regex_replace('Gi',''): strip the 'Gi' suffix from '32Gi' → '32'.
          # | int: convert the string '32' to the integer 32.
          # Then compare: if target_memory_gi > current_memory_gi + free_hugepages → fail.

    - name: Check free hugepages on the target node
      kubernetes.core.k8s_exec:
        namespace: stackrox
        pod:
          "collector-{{ vm_node | replace('.novacrestbank.local','') | replace('-','') }}"
        command:
          - chroot
          - /host
          - grep
          - HugePages_Free
          - /proc/meminfo
      register: hugepages_info

    - name: Extract free hugepage count as an integer
      ansible.builtin.set_fact:
        free_hugepages:
          "{{ hugepages_info.stdout
             | regex_search('HugePages_Free:\\s+(\\d+)', '\\1')
             | first | int }}"
        memory_delta_gi:
          "{{ target_memory_gi | int - current_memory_gi | int }}"

    - name: Fail immediately if insufficient hugepages — no change made
      ansible.builtin.fail:
        msg: |
          Insufficient hugepages on {{ vm_node }}.
          Requested: {{ memory_delta_gi }} GiB additional.
          Available: {{ free_hugepages }} hugepages (1 GiB each).
          Scale-up aborted — the VM is unchanged.
      when: (memory_delta_gi | int) > (free_hugepages | int)

    - name: Patch VirtualMachine with new CPU socket count and memory
      kubernetes.core.k8s:
        state: patched
                # state: patched → strategic merge patch — only specified fields are updated.
        # Unlike present (which may replace the whole resource), patched
        # changes ONLY what you declare in definition. Equivalent to oc patch --type=merge.
        api_version: kubevirt.io/v1
        kind: VirtualMachine
        name: "{{ vm_name }}"
        namespace: "{{ vm_namespace }}"
        definition:
          spec:
            template:
              spec:
                domain:
                  cpu:
                    sockets: "{{ target_cpu_sockets }}"
                  memory:
                    guest: "{{ target_memory_gi }}Gi"

    - name: Wait 30 seconds for KubeVirt to send hot-plug to QEMU
      ansible.builtin.pause:
        seconds: 30

    - name: Re-query VMI to verify hot-plug took effect
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        name: "{{ vm_name }}"
        namespace: "{{ vm_namespace }}"
      register: vmi_after

    - name: Assert scale-up was applied
      ansible.builtin.assert:
        that:
          - vmi_after.resources[0].status.currentCPUTopology.sockets | int
            == target_cpu_sockets | int
        fail_msg: "CPU hot-plug did not take effect — sockets mismatch"
        success_msg:
          "Scale-up complete: {{ target_cpu_sockets }} sockets, {{ target_memory_gi }} GiB RAM"

14 — Automated live migration between bare-metal nodes

AAP automates live migrations that would otherwise require manual virtctl migrate commands. Two scenarios benefit: scheduled maintenance drains (migrate all VMs off a node before patching it) and alert-driven migrations (a VM is on a node with a failing OSD or CPU thermal event — migrate to safety automatically).

pb-live-migrate.yaml — drain a node by migrating all VMs off it
---
- name: Drain bare-metal node by live-migrating all VMs to other nodes
  hosts: localhost
  gather_facts: false
  vars:
    target_node: "{{ node_to_drain }}"
        # node_to_drain: from the AAP Survey — operator specifies which node to drain.
    # e.g., "bm-virt-03.novacrestbank.local"

  tasks:

    - name: Find all VMIs running on the target node
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        namespace: virt-workloads
        field_selectors:
          - "status.nodeName={{ target_node }}"
                    # field_selectors: filter by resource field values (not label selectors).
          # status.nodeName is a VMI field — this returns only VMIs on the target node.
          # Used instead of label selectors when the filter is on status fields.
      register: vms_on_node

    - name: Log which VMs will be migrated
      ansible.builtin.debug:
        msg: "Migrating {{ vms_on_node.resources | length }} VMs off {{ target_node }}"

      kubernetes.core.k8s:
        state: present
        definition:
          apiVersion: kubevirt.io/v1
          kind: VirtualMachineInstanceMigration
          metadata:
            name:
              "aap-migrate-{{ item.metadata.name }}-{{ ansible_date_time.epoch }}"
            namespace: virt-workloads
          spec:
            vmiName: "{{ item.metadata.name }}"
      loop: "{{ vms_on_node.resources }}"
            # loop over the list of VMIs. Creates one VMIM per VM.
      # KubeVirt queues migrations respecting parallelMigrationsPerCluster
      # from HyperConverged CR — they do not all run simultaneously.
      loop_control:
        label: "{{ item.metadata.name }}"
                # label: show only the VM name in loop output (not the full VMI dict).

    - name: Wait until no VMIs remain on the target node
      kubernetes.core.k8s_info:
        api_version: kubevirt.io/v1
        kind: VirtualMachineInstance
        namespace: virt-workloads
        field_selectors:
          - "status.nodeName={{ target_node }}"
      register: remaining
      until: remaining.resources | length == 0
      retries: 60
      delay: 30
            # Poll every 30 seconds for up to 30 minutes.
      # Migration of 4 production VMs typically completes within 5-10 minutes.

    - name: Confirm node is clear and safe for maintenance
      ansible.builtin.debug:
        msg:
          "Node {{ target_node }} is clear. {{ vms_on_node.resources | length }} VMs migrated. Safe to proceed."

# It defines: sources (where events come from) and rules (IF condition THEN action).
# Rulebooks are deployed to EDA via the AAP UI and run as always-on Activations.

prometheusrule-vm-alerts.yaml — alert conditions that trigger the EDA pipeline
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: vm-performance-alerts
  namespace: virt-workloads
spec:
  groups:
    - name: vm.performance
      rules:
        - alert: VMHighCPUSaturation
          expr: |
            rate(kubevirt_vmi_vcpu_seconds[5m]) /
            on(name,namespace) kubevirt_vmi_number_of_vcpus > 0.95
                    # expr: PromQL expression that triggers the alert when true.
          # rate(kubevirt_vmi_vcpu_seconds[5m]): vCPU seconds consumed per second over 5 minutes.
          # Divided by the total vCPU count: gives per-vCPU utilisation ratio.
          # > 0.95: alert when average vCPU utilisation exceeds 95%.
          for: 5m
                    # for: the condition must be true for 5 minutes before firing.
          # Prevents transient CPU spikes (5-second burst) from triggering migration.
          labels:
            severity: critical
          annotations:
            summary: "VM {{ $labels.name }} CPU saturation > 95% for 5 minutes"
            description: "VM {{ $labels.name }} on node {{ $labels.node }} has sustained CPU saturation. EDA will trigger live migration."

        - alert: VMMemoryStarvation
          expr:
            kubevirt_vmi_memory_available_bytes / kubevirt_vmi_memory_resident_bytes < 0.05
                    # Alert when available memory is less than 5% of resident memory.
          # This indicates imminent OOM — the VM is about to start swapping or OOM-killing.
          for: 3m
          labels:
            severity: critical
          annotations:
            summary: "VM {{ $labels.name }} memory starvation — < 5% available"

17 — VM backup orchestration via AAP

OADP (VIRT-07) provides scheduled VM backup automatically. AAP complements OADP for two scenarios: on-demand backup triggered by a change management event (take a snapshot before a major application upgrade), and backup verification (trigger a test restore of the most recent backup and verify the restored VM boots successfully). The playbook below demonstrates both patterns.

pb-vm-backup.yaml — on-demand VirtualMachineSnapshot and OADP backup verification
---
- name: Take on-demand snapshot and trigger OADP backup before a major change
  hosts: localhost
  gather_facts: false
  vars:
    snapshot_name:
      "pre-change-{{ vm_name }}-{{ ansible_date_time.date }}-{{ ansible_date_time.hour }}h"
    backup_name:
      "aap-ondemand-{{ vm_name }}-{{ ansible_date_time.epoch }}"

  tasks:

    - name: Create VirtualMachineSnapshot (in-cluster, fast)
      kubernetes.core.k8s:
        state: present
        definition:
          apiVersion: snapshot.kubevirt.io/v1alpha1
          kind: VirtualMachineSnapshot
          metadata:
            name: "{{ snapshot_name }}"
            namespace: "{{ vm_namespace }}"
          spec:
            source:
              apiGroup: kubevirt.io
              kind: VirtualMachine
              name: "{{ vm_name }}"
            failureDeadline: 5m

    - name: Wait for snapshot to complete
      kubernetes.core.k8s_info:
        api_version: snapshot.kubevirt.io/v1alpha1
        kind: VirtualMachineSnapshot
        name: "{{ snapshot_name }}"
        namespace: "{{ vm_namespace }}"
      register: snap_status
      until:
        snap_status.resources[0].status.readyToUse == true
      retries: 15
      delay: 20

    - name: Trigger on-demand OADP backup (off-cluster, durable)
      kubernetes.core.k8s:
        state: present
        definition:
          apiVersion: velero.io/v1
          kind: Backup
          metadata:
            name: "{{ backup_name }}"
            namespace: openshift-adp
          spec:
            includedNamespaces:
              - "{{ vm_namespace }}"
            labelSelector:
              matchLabels:
                kubevirt.io/vm: "{{ vm_name }}"
            snapshotVolumes: true
            defaultVolumesToFsBackup: false
            ttl: 168h0m0s
                        # ttl: 168 hours = 7 days. On-demand backups kept for 7 days.
            # Nightly scheduled backups (from VIRT-07) have 30-day TTL.

    - name: Wait for OADP backup to complete
      kubernetes.core.k8s_info:
        api_version: velero.io/v1
        kind: Backup
        name: "{{ backup_name }}"
        namespace: openshift-adp
      register: backup_status
      until:
        backup_status.resources[0].status.phase in ["Completed", "Failed", "PartiallyFailed"]
      retries: 40
      delay: 60

    - name: Fail if backup did not complete successfully
      ansible.builtin.fail:
        msg:
          "OADP backup {{ backup_name }} ended with status: {{ backup_status.resources[0].status.phase }}"
      when:
        backup_status.resources[0].status.phase != "Completed"

    - name: Log backup success
      ansible.builtin.debug:
        msg: |
          ✓ Snapshot: {{ snapshot_name }} — readyToUse=true
          ✓ OADP backup: {{ backup_name }} — Completed
          Both artefacts are available for restore if the upcoming change causes issues.
