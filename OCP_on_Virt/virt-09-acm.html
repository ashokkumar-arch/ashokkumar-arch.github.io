01 — What ACM is — the four pillars from first principles

Red Hat Advanced Cluster Management (ACM) is the control plane for managing a fleet of Kubernetes clusters — not just one. Everything built in VIRT-01 through VIRT-08 operates on NovaCrestBank's single OCP cluster. ACM raises the abstraction level: instead of managing resources in a single cluster, ACM manages clusters themselves — provisioning them, governing what runs on them, distributing applications across them, and observing their health from a single hub.

The key mental model: ACM's relationship to an OCP cluster is analogous to an OCP cluster's relationship to individual pods. OCP abstracts away the node — you declare a Pod and OCP decides which node runs it. ACM abstracts away the cluster — you declare a policy or an application and ACM decides which cluster enforces it or hosts it.

1
Centralised Policy Governance & Compliance
SecOps · DevSecOps · PCI-DSS · NIST

ACM deploys security and configuration policies from the Hub to all managed clusters simultaneously. A policy that enforces the NovaCrestBank LDAP Identity Provider, or that requires evictionStrategy: LiveMigrate on all production VMs, is defined once on the Hub and propagated to every cluster in the fleet automatically. Compliance status — Pass/Fail per policy per cluster — is visible on the Hub's governance dashboard.

  • Enforcing Identity Providers (LDAP, OIDC) across all clusters from one definition
  • Security and network guardrails — SCCs, NetworkPolicy, admission webhooks
  • Detecting and remediating configuration drift — any cluster that diverges from policy is flagged and optionally auto-remediated
2
Multi-Cluster Application Lifecycle & GitOps
DevOps · GitOps · Placement · Argo CD integration

ACM's application model uses Placement rules to decide which clusters receive which applications — based on cluster labels, resource availability, or geographic location. When combined with Argo CD (deployed in VIRT-07), ACM becomes the placement engine and Argo CD becomes the delivery engine. The result: a GitOps pipeline that knows not just what to deploy but which clusters in the fleet should receive each workload.

  • Placement rules via cluster labels — target bare-metal VM clusters, exclude dev clusters, prefer clusters with available hugepage capacity
  • Unified application mapping — one Application definition, many cluster targets
  • ACM channels (Git, Helm, OCI) as the source of truth for multi-cluster deployment
3
Automated Cluster Lifecycle Management
Platform Engineering · Cluster Factories · Fleet Upgrades

ACM provisions new OCP clusters using Hive (for infrastructure-provider clusters) or the Assisted Installer (for bare-metal clusters). A "cluster factory" is an ACM ClusterDeployment template that creates a new, fully configured cluster — including all governance policies, monitoring, and GitOps configuration — in a reproducible, self-service way. Fleet-wide upgrades manage rolling OCP version updates across all clusters from the Hub.

  • Simplified provisioning via ClusterDeployment templates and SiteConfig (ZTP)
  • Cluster APIs — programmatic cluster creation for self-service developer environments
  • Fleet-wide fleet upgrades — coordinated rolling OCP upgrades across all managed clusters
4
Cross-Cluster Observability & Multi-Tenant Search
SRE · Platform visibility · Centralised metrics

ACM's observability stack aggregates metrics, alerts, and resource inventory from all managed clusters into a single Hub-level view. The global search box indexes every Kubernetes object across every managed cluster — finding a specific VirtualMachine CR across a 20-cluster fleet takes seconds rather than iterating through individual cluster contexts.

  • Global search box — find any VM, pod, node, or policy across all clusters instantly
  • Centralised metric aggregation via Thanos — single Grafana dashboard for fleet-wide VM metrics
  • Multi-cluster alert consolidation — one PagerDuty notification for a cluster-wide event, not N cluster-specific alerts

02 — Why ACM for VM management — what it adds beyond VIRT-07 GitOps

VIRT-07 put VirtualMachine CRs under Argo CD management. That solves the single-cluster GitOps problem. ACM solves a different, larger problem: what happens when NovaCrestBank runs VM workloads on more than one cluster?

The single-cluster ceiling

NovaCrestBank's current VIRT environment is one OCP cluster with five bare-metal nodes. As the platform matures, additional clusters are inevitable: a dedicated DR cluster in a second data centre, a developer cluster where VMs can be tested without competing for bare-metal hugepages, a production expansion cluster when the five bare-metal nodes reach capacity, or an edge cluster at a bank branch. Each new cluster needs the same governance policies (LDAP IdP, evictionStrategy enforcement, ACS deployment), the same monitoring, the same upgrade cadence, and the same GitOps configuration.

Without ACM, every new cluster is a manual repetition of VIRT-01 through VIRT-08 — with the risk that each repetition introduces subtle differences. With ACM, the Hub enforces consistency. Every new cluster registered with ACM receives all governance policies automatically. The platform engineer's job changes from "configure this cluster" to "register this cluster and let the Hub handle it."

03 — ACM architecture — Hub, Managed Clusters, and the data plane

ACM has a two-tier architecture. The Hub cluster runs the ACM control plane. Managed clusters run the ACM agent (klusterlet). Every managed cluster reports its resource inventory to the Hub and receives policies and application deployments from the Hub.

apiVersion: operator.open-cluster-management.io/v1
kind: MultiClusterHub
metadata:
  name: multiclusterhub
  namespace: open-cluster-management
    # One MultiClusterHub per cluster — the Hub is a singleton.
  # The name "multiclusterhub" is conventional and expected by ACM tooling.
spec:
  disableHubSelfManagement: false
    # false: the Hub cluster also manages itself as a managed cluster.
  # This means governance policies and search indexing apply to the Hub cluster too.
  # The Hub appears in the ACM console as "local-cluster" alongside other managed clusters.
  availabilityConfig: High
    # High: deploys Hub components with multiple replicas for HA.
  # Basic: single replica. Use High for production — the Hub is a critical control plane.
  enableClusterBackup: true
    # true: enable the Hub backup operator. Backs up ACM Hub configuration
  # (managed cluster registrations, policies, applications) to an S3 bucket.
  # If the Hub cluster fails, the Hub can be restored to a new cluster
  # and all managed clusters re-register automatically. Uses OADP under the hood.
  overrides:
    components:
      - name: cluster-backup
        enabled: true
      - name: multicluster-observability
        enabled: true
                # Enable the ACM observability stack — Thanos multi-cluster metric aggregation.
        # Required for Section 18 (centralised metric aggregation).
        # Requires an ObjectBucketClaim for Thanos metric storage (uses MCG, same as OADP).
      - name: search
        enabled: true
                # Enable the global search indexer.
        # Required for Section 17 (global VM search across all clusters).

06 — Cluster labels — the foundation of all ACM targeting

Cluster labels are the mechanism ACM uses to target policies and applications to specific subsets of managed clusters. Every ACM object that selects clusters (PlacementRule, ManagedClusterSet, Policy namespace binding) uses labels to identify its targets. Getting the label taxonomy correct before creating any policies is essential — retrofitting labels onto clusters that already have policies applied is error-prone.

Apply NovaCrestBank label taxonomy to local-cluster
apiVersion: cluster.open-cluster-management.io/v1
kind: ManagedCluster
metadata:
  name: local-cluster
  labels:
    environment: production
        # environment: production | development | staging | dr
    # Primary targeting dimension — policies that apply only to production clusters
    # use matchLabels: environment: production.
    region: ap-southeast-2
        # region: AP-Southeast-2 (Sydney) for the on-premises VCF cluster.
    # Used to target disaster recovery policies and cross-region placement rules.
    # When a DR cluster in a different region is added, its region label differs —
    # enabling placement rules that target the primary region vs the DR region.
    platform: baremetal-virt
        # platform: baremetal-virt — this cluster has bare-metal nodes with hugepages
    # and OCP Virt installed. VM workload policies and applications target this label.
    # A cloud cluster would have platform: rosa or platform: eks.
    # This label is how the VM placement rules in Section 13 target this cluster.
    compliance: pci-dss
        # compliance: pci-dss — this cluster is in scope for PCI-DSS.
    # PCI-DSS governance policies target clusters with this label.
    # A development cluster might have compliance: non-pci — receiving less strict policies.
    vendor: OpenShift
        # vendor: OpenShift (auto-populated by ACM). Used to scope policies to OCP
    # clusters only — a future EKS cluster would have vendor: EKS and would not
    # receive OCP-specific policies (SCC, route53 annotations, etc.).

07 — How ACM Governance works — Policy, PlacementRule, PolicySet

ACM Governance uses three objects that work together: the Policy defines what the desired state is, the PlacementRule (or the newer Placement object) defines which clusters receive the policy, and the PlacementBinding connects them. A PolicySet groups related policies into a named set that is targeted as a unit.

How a policy propagates to a managed cluster

When a Policy is created on the Hub and a PlacementBinding connects it to a PlacementRule that selects environment=production clusters, ACM creates a copy of the Policy's inner manifest as a ConfigurationPolicy on each selected managed cluster via the klusterlet. The klusterlet evaluates the ConfigurationPolicy against the cluster's actual state — if the cluster's OAuth configuration does not match the Policy's desired state, the klusterlet reports a violation. If remediationAction: enforce, the klusterlet applies the desired state directly, bringing the cluster into compliance.

Inform vs Enforce — the mandatory observation period for ACM policies

ACM policies have two remediation modes — inform (report violations, do not fix) and enforce (apply the desired state automatically). The correct operational pattern is identical to ACS's 14-day baseline rule: every new policy starts in inform mode for at least one week, so violations can be reviewed and understood before auto-remediation makes changes to production clusters. A policy switched directly to enforce without an observation period may auto-remediate something that was intentionally different — causing a production incident. All policies in this guide are created in inform mode with an explicit note of when to switch to enforce.

08 — Enforcing Identity Providers — LDAP IdP on all clusters

NovaCrestBank's OCP cluster uses AD LDAP as its identity provider (configured via the OAuth CR during the OCP-VCF series). If a second cluster is added — the DR cluster, a dev cluster — that cluster must also be configured with the same LDAP IdP. Without ACM governance, this is a manual step that is easily forgotten. With ACM governance, the IdP configuration is a Policy: any managed cluster that does not have the correct OAuth LDAP configuration is immediately flagged as non-compliant.

policy-idp-ldap.yaml — enforce LDAP IdP on all production OCP clusters
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-idp-ldap
  namespace: policies
    # Policies live in a dedicated namespace on the Hub — not in open-cluster-management.
  # The "policies" namespace is conventional. All Hub policies go here.
  annotations:
    policy.open-cluster-management.io/categories: PR.AC Identity Management and Access Control
    policy.open-cluster-management.io/controls: PR.AC-4 Access Permissions
    policy.open-cluster-management.io/standards: NIST SP 800-53
        # These annotations populate the ACM Governance dashboard's compliance categories.
    # They map each policy to a standard and control for compliance reporting.
spec:
  remediationAction: inform
    # inform: report violations only. Start here, switch to enforce after 7 days.
  # enforce would apply the OAuth CR to any non-compliant cluster automatically.
  disabled: false
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: config-idp-ldap
        spec:
          remediationAction: inform
          severity: high
          object-templates:
            - complianceType: musthave
              objectDefinition:
                                # The inner object: the cluster OAuth CR that must exist on every managed cluster.
                apiVersion: config.openshift.io/v1
                kind: OAuth
                metadata:
                  name: cluster
                spec:
                  identityProviders:
                    - name: NovaCrestBank-AD
                      type: LDAP
                      ldap:
                        url: ldaps://dc01.novacrestbank.local:636/...
                        insecure: false
                                # complianceType: musthave — the cluster's OAuth CR must contain
                        # an identityProvider named NovaCrestBank-AD of type LDAP pointing to DC01.
                        # If a managed cluster has no LDAP IdP, or has a different IdP URL,
                        # it is flagged as non-compliant on the ACM Governance dashboard.
                        # In enforce mode: the klusterlet would apply this OAuth CR to the cluster,
                        # configuring LDAP authentication automatically on cluster registration.
---
apiVersion: policy.open-cluster-management.io/v1
kind: PlacementBinding
metadata:
  name: binding-idp-ldap
  namespace: policies
placementRef:
  name: placement-prod-ocp
  kind: PlacementRule
subjects:
  - name: policy-idp-ldap
    kind: Policy
    apiGroup: policy.open-cluster-management.io
---
apiVersion: apps.open-cluster-management.io/v1
kind: PlacementRule
metadata:
  name: placement-prod-ocp
  namespace: policies
spec:
  clusterConditions:
    - status: "True"
      type: ManagedClusterConditionAvailable
  clusterSelector:
    matchLabels:
      environment: production
      vendor: OpenShift
            # Targets all clusters labelled environment=production AND vendor=OpenShift.
      # local-cluster matches both — the LDAP policy is evaluated against it.
      # A future dev cluster (environment=development) is excluded from this policy.
      # A future ROSA cluster (vendor=OpenShift, environment=production) is included.

09 — Security and network guardrails — SCCs and NetworkPolicy

Two categories of guardrail policies are essential for NovaCrestBank's fleet: SCC policies (ensuring the kubevirt-controller SCC exists and that privileged SCCs cannot be granted to arbitrary service accounts) and NetworkPolicy policies (ensuring the default-deny NetworkPolicy from VIRT-08 exists in the virt-workloads namespace on every cluster that runs VM workloads).

policy-vm-network-guardrails.yaml — enforce VM NetworkPolicy on all baremetal-virt clusters
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-vm-network-guardrails
  namespace: policies
  annotations:
    policy.open-cluster-management.io/standards: PCI-DSS v4.0
    policy.open-cluster-management.io/controls: 1.3.1 1.3.2
spec:
  remediationAction: enforce
    # enforce: if the NetworkPolicy is absent from virt-workloads on any baremetal-virt cluster,
  # the klusterlet creates it automatically. This is safe to enforce immediately —
  # the NetworkPolicy is additive and does not remove existing permissions,
  # it only enforces the default-deny plus the allowed exceptions.
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: config-virt-deny-all
        spec:
          remediationAction: enforce
          severity: critical
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: networking.k8s.io/v1
                kind: NetworkPolicy
                metadata:
                  name: default-deny-ingress-virt
                  namespace: virt-workloads
                spec:
                  podSelector: {}
                  policyTypes:
                    - Ingress
                    # This is the exact NetworkPolicy from VIRT-08 Section 10.
                  # If a new cluster is added with platform=baremetal-virt, this NetworkPolicy
                  # is automatically created in its virt-workloads namespace by the klusterlet.
                  # The platform engineer does not need to manually apply it.

10 — VM configuration governance — evictionStrategy and hugepages

ACM Governance can enforce VirtualMachine CR configuration requirements across all production VM clusters. Two configuration properties are governance-enforced at NovaCrestBank: evictionStrategy: LiveMigrate on all production VMs (to ensure zero-downtime node maintenance) and the presence of a hugepage request (to ensure banking VMs always use 1 GiB hugepages and never fall back to standard pages).

policy-vm-eviction-strategy.yaml — enforce evictionStrategy on banking VMs
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-vm-eviction-strategy
  namespace: policies
  annotations:
    policy.open-cluster-management.io/standards: NIST SP 800-53
    policy.open-cluster-management.io/controls: CP-7 Alternate Processing Site
spec:
  remediationAction: inform
    # inform: report VMs without evictionStrategy: LiveMigrate — do not auto-remediate.
  # Auto-remediation of evictionStrategy would modify running VM specs —
  # safer to alert the platform team and let them update via the GitOps workflow.
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: config-vm-eviction
        spec:
          remediationAction: inform
          severity: high
          namespaceSelector:
            include:
              - virt-workloads
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: kubevirt.io/v1
                kind: VirtualMachine
                metadata:
                  namespace: virt-workloads
                  labels:
                    environment: production
                spec:
                  template:
                    spec:
                      evictionStrategy: LiveMigrate
                                            # complianceType: musthave — every VirtualMachine in virt-workloads
                      # with label environment=production must have evictionStrategy: LiveMigrate.
                      # Any VM missing this field is flagged NonCompliant on the ACM dashboard.
                      # The platform team then raises a GitOps PR to add evictionStrategy
                      # to the production Kustomize overlay for the non-compliant VM.

11 — Detecting and remediating configuration drift

Configuration drift occurs when a cluster's actual state diverges from its desired state — a direct oc apply that bypasses GitOps, an automated process that modifies a resource, or an operator upgrade that changes a field. ACM Governance detects drift continuously: the klusterlet re-evaluates every ConfigurationPolicy on its cluster every 10 minutes (configurable). Any drift appears as a NonCompliant status on the ACM Governance dashboard within minutes.

jumphost — simulate drift and observe ACM detection
# Simulate drift: manually remove the default-deny NetworkPolicy from virt-workloads
# (as might happen if an engineer accidentally deleted it during troubleshooting)

12 — ACM as the bridge between GitOps and multi-cluster placement

In VIRT-07, Argo CD manages VMs on one cluster. ACM + Argo CD together manage VMs across a fleet of clusters. The architectural pattern: ACM decides where to deploy (which clusters), Argo CD decides what to deploy (the Kustomize overlay content from Git). ACM's ApplicationSet generator integration with Argo CD makes this seamless — ACM Placement generates a cluster list, Argo CD's ACM generator creates one Application per cluster automatically.

How ACM integrates with the VIRT-07 Argo CD instance

ACM registers itself as a GitOpsCluster — it tells the Argo CD instance running in the gitops namespace about every managed cluster. Argo CD gains the ability to deploy to all ACM-managed clusters, not just the local cluster. The ApplicationSet in VIRT-07 used a static list generator (two environments: dev and production). With ACM integration, the generator becomes a ClusterDecisionResource — it queries ACM for all clusters matching a label selector and generates one Argo CD Application per cluster automatically.

gitopscluster-ncb.yaml — register ACM clusters with the VIRT-07 Argo CD instance
apiVersion: apps.open-cluster-management.io/v1beta1
kind: GitOpsCluster
metadata:
  name: ncb-gitops-cluster
  namespace: gitops
    # Same namespace as the Argo CD instance from VIRT-07.
spec:
  argoServer:
    cluster: local-cluster
    argoNamespace: gitops
        # Points to the ncb-argocd instance in the gitops namespace.
    # ACM will register all managed clusters matching the Placement below
    # as Argo CD destination clusters. Argo CD ApplicationSets can then
    # use the ACM cluster generator to dynamically target these clusters.
  placementRef:
    kind: Placement
    apiGroup: cluster.open-cluster-management.io
    name: placement-baremetal-virt-clusters
        # The Placement object selects which managed clusters are registered with Argo CD.
    # Initially: local-cluster only (platform=baremetal-virt).
    # When a second bare-metal VM cluster is registered with ACM, it is automatically
    # added to Argo CD's cluster list and becomes available as an ApplicationSet target.

13 — Placement rules via labels — targeting clusters for VM workloads

ACM's Placement API selects clusters based on label expressions, cluster conditions, and resource requirements. For VM workloads, placement rules can express: "deploy this VM workload to clusters that have bare-metal nodes with hugepages available, in the production environment, in the AP-Southeast-2 region."

placement-vm-workloads.yaml — target bare-metal VM clusters for banking VM placement
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: placement-banking-vms
  namespace: policies
spec:
  numberOfClusters: 1
    # Select 1 cluster from all that match the predicates below.
  # Increase to 2 when a second baremetal-virt cluster is registered —
  # ACM will place the banking VM application on both clusters simultaneously.
  predicates:
    - requiredClusterSelector:
        labelSelector:
          matchLabels:
            environment: production
            platform: baremetal-virt
            compliance: pci-dss
                        # All three labels must match:
            # - environment=production: no dev clusters
            # - platform=baremetal-virt: must have bare-metal nodes with hugepages
            # - compliance=pci-dss: must be in scope for PCI-DSS (VIRT-08 security controls applied)
            # local-cluster matches all three. A future dev cluster (environment=development)
            # does not match — VM production workloads are not placed there.
  prioritizerPolicy:
    mode: Exact
    configurations:
      - scoreCoordinate:
          type: AddOn
          addOn:
            resourceName: hugepages
            scoreName: freeHugepages
        weight: 1
                # When multiple baremetal-virt clusters exist, prefer the cluster
        # with the most free hugepages — ensuring VM placement on the cluster
        # with the most available memory capacity for the hugepage-backed VMs.

14 — Unified application mapping — Subscription and Channel

ACM's application model uses a Channel (the source — a Git repository, Helm chart, or OCI registry) and a Subscription (the binding between a Channel and a set of clusters via Placement). This is distinct from Argo CD's Application model — ACM's Subscription works at a lower level and can be used without Argo CD, or in combination with it for GitOps delivery.

channel-and-subscription.yaml — ACM application mapping for VM workloads
# Channel: points to the banking-config Git repository
apiVersion: apps.open-cluster-management.io/v1
kind: Channel
metadata:
  name: banking-config-git
  namespace: banking-config-channel
spec:
  type: Git
  pathname: git@gitlab.novacrestbank.local:gitops/banking-config.git
    # The same banking-config repo used in VIRT-07 by Argo CD.
  # ACM Channel and Argo CD source can both reference the same Git repo —
  # they do not conflict. ACM's Subscription can coexist with Argo CD Applications.
  secretRef:
    name: banking-config-git-creds
---
# Subscription: binds the Channel to the Placement
apiVersion: apps.open-cluster-management.io/v1
kind: Subscription
metadata:
  name: banking-vm-subscription
  namespace: policies
  labels:
    app: banking-vms
spec:
  channel: banking-config-channel/banking-config-git
  placement:
    placementRef:
      kind: Placement
      name: placement-banking-vms
            # References the Placement from Section 13.
      # ACM delivers the vms/banking-app-vm/overlays/production path from the Channel
      # to all clusters selected by placement-banking-vms.
      # When a second baremetal-virt cluster is registered, the Placement selects it
      # and the Subscription automatically delivers the production VM overlay to it too.
  packageFilter:
    filterRef:
      name: vms/banking-app-vm/overlays/production

15 — Cluster factories — self-service infrastructure provisioning

A cluster factory is the pattern of using ACM's ClusterDeployment (for vSphere/cloud provider clusters) or SiteConfig/ZTP (Zero Touch Provisioning, for bare-metal clusters) to create new OCP clusters in a fully automated, reproducible way. The cluster factory produces a new cluster that is pre-configured — it emerges from the provisioning process already registered with ACM, already receiving all governance policies, already integrated with GitOps.

SiteConfig — bare-metal cluster provisioning via ZTP

For NovaCrestBank's bare-metal expansion cluster (a second set of DL385 Gen11 nodes for capacity expansion or DR), the Zero Touch Provisioning (ZTP) pipeline uses ACM + GitOps to provision an OCP cluster from bare hardware. A SiteConfig YAML committed to Git is the complete specification of the new cluster: node names, IP addresses, BIOS settings, OCP version, and the initial MachineConfigPool configuration. The ZTP pipeline watches the Git repository, generates the Assisted Installer manifests, and provisions the cluster without any manual SSH or BMC interaction.

siteconfig-ncb-dr.yaml — ZTP SiteConfig for a second bare-metal VM cluster
apiVersion: ran.openshift.io/v1
kind: SiteConfig
metadata:
  name: ncb-dr-site
  namespace: siteconfig
spec:
  baseDomain: novacrestbank.local
  pullSecretRef:
    name: pullsecret
  clusters:
    - clusterName: ncb-ocp-dr
      clusterType: Standard
      clusterLabels:
        environment: production
        platform: baremetal-virt
        compliance: pci-dss
        region: ap-southeast-4
                # The cluster labels are applied at provisioning time.
        # Immediately upon cluster registration with ACM, all policies that match
        # these labels are evaluated against ncb-ocp-dr:
        # - policy-idp-ldap (environment=production, vendor=OpenShift) → evaluates LDAP IdP
        # - policy-vm-network-guardrails (baremetal-virt) → creates NetworkPolicy in virt-workloads
        # - policy-vm-eviction-strategy → checks VMs for evictionStrategy: LiveMigrate

            name: bm-dr-01-bmc-secret

16 — Fleet-wide upgrades — rolling OCP version management

ACM manages OCP cluster upgrades through the ManagedClusterSet and ClusterCurator. Rather than logging into each cluster and running oc adm upgrade separately, ACM orchestrates upgrades across the fleet with controlled sequencing — dev clusters upgrade first, production clusters upgrade after dev is validated, DR clusters upgrade last.

managedclusterset and upgrade policy
# Group clusters into sets for coordinated upgrade waves
apiVersion: cluster.open-cluster-management.io/v1beta2
kind: ManagedClusterSet
metadata:
  name: production-virt-clusters
spec:
  clusterSelector:
    selectorType: LabelSelector
    labelSelector:
      matchLabels:
        environment: production
        platform: baremetal-virt
---
# ACM Policy to enforce the target OCP version across all production clusters
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: policy-ocp-version
  namespace: policies
spec:
  remediationAction: inform
    # inform: report clusters running older than the target version.
  # Do NOT enforce: cluster upgrades require change management approval — not auto-remediation.
  # The policy reports which clusters need upgrading; engineers trigger upgrades via ClusterCurator.
  policy-templates:
    - objectDefinition:
        apiVersion: policy.open-cluster-management.io/v1
        kind: ConfigurationPolicy
        metadata:
          name: config-ocp-version
        spec:
          remediationAction: inform
          severity: medium
          object-templates:
            - complianceType: musthave
              objectDefinition:
                apiVersion: config.openshift.io/v1
                kind: ClusterVersion
                metadata:
                  name: version
                status:
                  history:
                    - version: 4.16.x
                      state: Completed
                                    # Clusters running 4.16.x with state Completed are Compliant.
                      # Clusters on 4.15.x are NonCompliant — flagged in ACM Governance dashboard.
                      # The platform engineer reviews the flag, raises a change request,
                      # and uses ClusterCurator to trigger the upgrade with pre/post hooks.

18 — Centralised metric aggregation — Thanos multi-cluster

ACM's observability stack deploys a Thanos-based metric aggregation layer on the Hub. Each managed cluster runs a metrics collector (a minimal observability add-on) that forwards selected Prometheus metrics to the Hub's Thanos Receive component. The Hub's Thanos Query aggregates metrics from all clusters behind a single query endpoint — enabling Grafana dashboards that show VM metrics from all clusters in one view.

multiclusterobservability.yaml — configure which metrics to collect from managed clusters
apiVersion: observability.open-cluster-management.io/v1beta2
kind: MultiClusterObservability
metadata:
  name: observability
  namespace: open-cluster-management-observability
spec:
  storageConfig:
    metricObjectStorage:
      name: thanos-object-storage
      key: thanos.yaml
            # Secret containing MCG S3 credentials for Thanos metric storage.
      # Uses the same MCG (NooBaa) bucket infrastructure as OADP (VIRT-07) —
      # a separate bucket is created for Thanos metric storage.
  observabilityAddonSpec:
    enableMetrics: true
    interval: 300
        # 300 seconds = 5 minute scrape interval for Hub-level aggregated metrics.
    # The local cluster Prometheus still scrapes at 30s — this is the aggregation interval.
  advanced:
    customMetrics:
      - group: kubevirt
        resource: virtualmachines
        names:
          - kubevirt_vmi_phase_count
            # Number of VMIs by phase (Running, Pending, Failed) per cluster.
          # Enables a fleet-level "VM health" dashboard in Grafana.
          - kubevirt_vmi_memory_used_bytes
            # VM memory consumption per cluster — visible in fleet-wide capacity planning.
          - kubevirt_vmi_vcpu_seconds
            # VM CPU usage — fleet-wide CPU utilisation for capacity planning.
          - kubevirt_vmi_network_traffic_bytes_total
            # VM network I/O per cluster — useful for identifying migration storm traffic.

19 — ACM as the hybrid cloud bridge — on-premises to public cloud

ACM's architecture is provider-agnostic: it manages OCP clusters on bare metal, VMware vSphere, AWS, Azure, GCP, IBM Cloud, and any Kubernetes cluster that can run the klusterlet agent. This makes ACM the natural bridge for NovaCrestBank's hybrid cloud expansion — the same governance policies, application placement, and observability that manage the on-premises bare-metal VM cluster extend to public cloud clusters without architectural changes.

Three hybrid cloud use cases for NovaCrestBank

Cloud burst for dev/test VM capacity: When the five bare-metal nodes are at capacity and developers need additional VM resources for testing, a ROSA (Red Hat OpenShift Service on AWS) cluster can be provisioned and registered with ACM. The dev Kustomize overlay (from VIRT-07) is placed on the ROSA cluster via Placement. Developers get VM capacity without waiting for hardware procurement. When the dev workload completes, the ROSA cluster is deprovisioned. ACM policies ensure dev VMs on ROSA receive the same IdP configuration and security guardrails as on-premises dev VMs.

Disaster recovery cluster in a second region: The DR cluster (in ap-southeast-4, Melbourne) is managed by ACM alongside the production cluster (in ap-southeast-2, Sydney). The same governance policies apply. ACM's Placement rules can be configured to prefer the primary region and failover to the DR region automatically based on cluster availability conditions. OADP (from VIRT-07) backs up VM disk data to ODF MCG on the primary cluster; cross-region replication delivers backup data to the DR cluster's ODF for recovery.

Regulatory-isolated workloads: If a future regulatory requirement mandates that certain banking data cannot leave Australian jurisdiction, ACM Placement rules enforce this through regional label selectors — only clusters labelled region: ap-southeast-2 or region: ap-southeast-4 receive those workloads. Non-Australian clusters are structurally excluded from receiving regulated VM workloads regardless of capacity.

ACM does not replace VCF — it manages the OCP clusters that replace VCF workloads

A common question: does ACM manage VMware VCF directly? No. ACM manages OCP clusters. The VCF hypervisor environment (ESXi, vCenter, NSX) is managed by vCenter/SDDC Manager, not ACM. ACM's role in the VIRT series is to manage the OCP clusters that are receiving the VM workloads being migrated from VCF. As VMs migrate from VCF (VIRT-06) to OCP Virt, and as additional OCP Virt clusters are provisioned (cluster factory in Section 15), all of those OCP Virt clusters come under ACM management. Eventually — when all VCF workloads are migrated and NSX is decommissioned (VIRT-11) — VCF is removed from the picture entirely, and ACM is the management plane for the entire NovaCrestBank cluster fleet.
