01 — What KubeVirt is — VMs as Kubernetes objects

Before installing anything, understand what KubeVirt is at a conceptual level. This understanding is what makes every configuration decision that follows logical rather than mechanical.

The problem KubeVirt solves

Kubernetes is designed for containerised workloads. A container is a process — it starts, runs, and exits. Kubernetes excels at scheduling, scaling, and healing processes. But banks and enterprises have thousands of virtual machines running workloads that cannot be containerised: legacy Windows applications with kernel-level dependencies, commercial off-the-shelf software licensed per physical host, stateful applications with complex network identity requirements, and guest operating systems that must be maintained as full VMs for compliance auditing reasons.

The traditional answer was to run two separate platforms: a hypervisor for VMs, and Kubernetes for containers. Two operational stacks, two monitoring systems, two upgrade cycles, two sets of expertise. KubeVirt eliminates this duplication by making the Kubernetes API itself the management plane for virtual machines.

How KubeVirt works — the core mechanism

KubeVirt introduces three things to a Kubernetes cluster:

Custom Resource Definitions (CRDs): New object types — VirtualMachine, VirtualMachineInstance, DataVolume, and others — that extend the Kubernetes API vocabulary. A VirtualMachine CR is to a VM what a Deployment CR is to a containerised application: a declarative description of the desired state.

Controllers: Kubernetes controllers that watch these CRDs and reconcile the cluster state to match. When a VirtualMachine CR is created with running: true, the VM controller creates a VirtualMachineInstance (VMI), which the VMI controller schedules onto a node. When a VM is stopped, the controller deletes the VMI but preserves the VirtualMachine CR — just as deleting a Pod does not delete its Deployment.

The virt-launcher pod: The actual mechanism that runs a VM on a node. For each running VM, KubeVirt creates a virt-launcher Pod on the selected node. Inside this pod runs a QEMU/KVM process — the standard Linux virtualisation stack. The virt-launcher pod is what the Kubernetes scheduler sees. The VM's guest OS runs inside the QEMU process inside the pod. From the cluster's perspective, a running VM is a pod with specific resource requests and a specific node affinity.

Why this architecture matters for NovaCrestBank

Because VMs are Kubernetes objects, every existing OCP platform capability applies to them automatically. Argo CD can manage VM configuration as GitOps. cert-manager can issue certificates for VMs. Prometheus scrapes VM metrics. NetworkPolicy controls VM network traffic. ACS enforces security policy on VM workloads. AAP automates guest OS operations via the Kubernetes API. The bank's platform team does not need to learn a separate hypervisor management system — they use the same tools they already use for containers.

02 — KubeVirt components — every process explained

KubeVirt deploys five distinct components on the cluster. Each runs as one or more pods in the openshift-cnv namespace (the namespace used by OpenShift Virtualization, which is Red Hat's productised distribution of KubeVirt). Understanding what each component does tells you exactly what is broken when a VM fails to start.

virt-operator
KubeVirt Operator
The lifecycle manager for all other KubeVirt components. Reads the HyperConverged CR and deploys virt-api, virt-controller, virt-handler, and virt-launcher to match the desired configuration. Handles KubeVirt upgrades — when the HCO is updated, virt-operator orchestrates a rolling update of all components. If virt-operator is unhealthy, no other KubeVirt component can be updated — but running VMs continue because they are QEMU processes, not operator-managed.
Runs as: Deployment, 2 replicas. If both replicas are down, running VMs keep running but cannot be created, stopped, or migrated.
virt-api
KubeVirt API Server
Extends the Kubernetes API server with VM-specific endpoints. When you run virtctl console vm-name or virtctl migrate vm-name, the virtctl command talks to virt-api — not to the standard Kubernetes API. virt-api handles VM console proxying, VNC access, and live migration triggering. It is not in the critical path for VM I/O — a running VM does not need virt-api to continue running.
Runs as: Deployment, 2 replicas. Loss of virt-api breaks console access and migration triggers but does not stop running VMs.
virt-controller
VM and VMI Controller
Watches VirtualMachine and VirtualMachineInstance CRs and reconciles their state. When a VM is set to running: true, virt-controller creates a VMI and a virt-launcher Pod. When a VMI is in Succeeded phase (crashed and not set to restart), virt-controller evaluates the VM's runStrategy to decide whether to restart it. Also manages the VM lifecycle events: start, stop, pause, restart, and migration coordination.
Runs as: Deployment, 2 replicas (active/standby leader election). Loss of virt-controller stops new VMs from starting but does not affect running VMs.
virt-handler
Node Agent (DaemonSet)
Runs on every node that hosts VMs — one pod per node, deployed as a DaemonSet. virt-handler is the node-level agent: it talks to the local QEMU process inside each virt-launcher pod on the same node, applies network configuration changes to running VMs, triggers in-memory snapshots, and coordinates live migration at the source and destination node level. virt-handler on the source node negotiates with virt-handler on the destination node during a live migration.
Runs on: all bare-metal nodes (bm-virt-01 through 05). Loss of virt-handler on one node means VMs on that node cannot be migrated — but they continue running.
virt-launcher
VM Pod (one per running VM)
Not a deployment — a Pod created for each running VM. Contains the QEMU/KVM process that runs the guest OS. The virt-launcher pod has privileged access to the node's KVM device (/dev/kvm), the VM's disk PVC (mounted as a block device), and the VM's network interfaces (created by Multus and the CNI plugins). When a VM stops, its virt-launcher pod is deleted. When a VM is live-migrated, a new virt-launcher pod starts on the destination node before the source pod exits.
One per running VM. If a virt-launcher pod is killed (not the VM stopped gracefully), KubeVirt treats it as a VM crash and applies the runStrategy to decide whether to restart.

03 — KubeVirt CRDs — every object type explained

KubeVirt extends Kubernetes with several new object types. Before creating any VM, understand what each object is, what creates it, and what its lifecycle is. Confusing a VirtualMachine with a VirtualMachineInstance is the most common cause of unexpected VM behaviour.

CRD kindShort nameCreated byLifecycleWhat it represents
VirtualMachine vm Platform engineer (GitOps / kubectl apply) Persistent — survives VM stop/start. Deleted manually. The VM definition. Equivalent to a Deployment — it describes what the VM should be (CPU, memory, disks, network, runStrategy) and manages its desired running state. A stopped VM still has a VirtualMachine CR.
VirtualMachineInstance vmi virt-controller (when VM is started) Ephemeral — exists only while the VM is running. Deleted when VM stops. A running VM instance. Equivalent to a Pod — it tracks the specific node, IP, phase, and real-time state of one running VM. One VMI per running VM. oc get vmi shows all running VMs.
DataVolume dv Platform engineer, or created by VirtualMachine CR Persistent — survives VM stop/start. Holds the VM's disk data. A PVC with an import process. Combines PVC creation with data import (from URL, container image, or another PVC) in one object. Used for VM boot disks. The DataVolume imports the OS image into the PVC once, then the VM uses the PVC.
VirtualMachineInstanceMigration vmim Platform engineer (virtctl migrate) or virt-controller (automated) Ephemeral — deleted after migration completes or fails. A live migration request. Creating a VMIM object triggers the migration from source to destination node. The VMIM tracks migration progress (Pending → Scheduling → Preparing → Transferring → Final → Succeeded).
VirtualMachineSnapshot vmsnapshot Platform engineer Persistent until deleted. Captures VM state at a point in time. A point-in-time snapshot of a VM and all its disks. Uses the CSI VolumeSnapshot API under the hood — Ceph RBD creates a snapshot of the PVC. Can be used to restore a VM to its snapshot state via VirtualMachineRestore.
HyperConverged hco Platform engineer (one per cluster) Persistent — the top-level configuration object for all of OCP Virt. The single CR that configures the entire OpenShift Virtualization stack. Setting fields here propagates configuration to KubeVirt, CDI (Containerized Data Importer), SSP (Scheduling, Scale, and Performance), and all other sub-operators. Do not edit KubeVirt, CDI, or SSP CRs directly — edit HyperConverged.
VirtualMachine vs VirtualMachineInstance — the distinction you must know

When you run oc get vm and see a VM listed, that VM may or may not be running. The VM CR persists regardless. When you run oc get vmi and see a VMI listed, that VM is running right now on a specific node. If oc get vmi returns nothing for a VM that shows in oc get vm, the VM is stopped. This is intentional — the VM definition is preserved while the VM is off, exactly as a Deployment CR persists when all its Pods are scaled to zero.

04 — VM lifecycle — from YAML to running guest OS

Trace the complete path from oc apply -f vm.yaml to a running guest OS — every step, every component involved, every object created. This understanding is what allows you to diagnose exactly where a VM start failure occurred.

VM start sequence — every step and component
Step 1
oc apply -f vm.yaml → kube-apiserver creates VirtualMachine CR in etcd
Step 2
virt-controller watches VirtualMachine CR → sees running: true → creates VirtualMachineInstance CR
Step 3
virt-controller creates virt-launcher Pod with specific resource requests (CPU cores, memory, hugepages) and node affinity for bare-metal nodes
Step 4
kube-scheduler selects target node (bm-virt-0X) — respects VM node affinity, CPU pinning, hugepage availability, and existing VM placement
Step 5
virt-handler on bm-virt-0X receives the VMI assignment → prepares the node: mounts ODF PVC as block device, creates network interfaces via Multus
Step 6
virt-launcher pod starts QEMU/KVM process → guest OS boots from the PVC block device → VM enters Running phase
Step 7
VMI phase transitions: Pending → Scheduling → Scheduled → Running → visible in oc get vmi
Where VM start failures occur — and what to check

Stuck at Pending: The scheduler cannot find a node that satisfies the VM's resource requests. Check: are bare-metal nodes ready? Do they have enough free hugepages? Is the node tainted and does the VM have the correct toleration?

Stuck at Scheduling: virt-handler on the target node is unhealthy. Check: oc get pods -n openshift-cnv -l kubevirt.io=virt-handler — all virt-handler pods must be Running.

Stuck at Scheduled / CrashLoopBackOff on virt-launcher: QEMU failed to start. The most common causes: the PVC is not bound (DataVolume import failed), the hugepage allocation failed (node does not have enough free 1Gi hugepages), or the KVM device is unavailable (/dev/kvm not present — check BIOS VT-x setting from VIRT-02).

05 — What the HyperConverged Operator is

OpenShift Virtualization is not a single operator — it is a family of operators, each managing a different aspect of the virtualisation stack. The HyperConverged Operator (HCO) is the meta-operator: it installs and manages all the other operators through a single HyperConverged CR.

Sub-operatorWhat it managesHow HCO configures it
KubeVirt operator Core VM runtime (virt-controller, virt-handler, virt-launcher, QEMU) HCO writes a KubeVirt CR with the configuration from HyperConverged spec
CDI (Containerized Data Importer) DataVolume imports — downloading OS images from URLs or container registries into PVCs HCO writes a CDI CR. CDI is what makes DataVolume objects work.
SSP (Scheduling, Scale and Performance) VM templates, node labelling for VM placement, golden image management HCO writes an SSP CR. SSP manages the common VM templates in OCP console.
MTQ (ManagedTenant Quota) Resource quota management for VM workloads across namespaces HCO writes an MTQ CR when quota management is enabled.
NetworkAddonsOperator Additional network plugins: Multus, bridge CNI, SR-IOV, OVN-Kubernetes integration for VMs HCO writes a NetworkAddonsConfig CR. This is what deploys Multus to bare-metal nodes.
The single configuration principle — only edit HyperConverged, never the sub-operators

All sub-operator CRs (KubeVirt, CDI, SSP, NetworkAddonsConfig) are owned by HCO. If you edit them directly, HCO reconciles them back to the configuration specified in the HyperConverged CR on the next reconciliation loop — typically within 30 seconds. Any direct sub-operator edit is overwritten. The only supported way to change the virtualisation configuration is to change the HyperConverged CR.

07 — HyperConverged CR — every field explained

The HyperConverged CR is the master configuration object for the entire OpenShift Virtualization stack. Creating it triggers the deployment of all KubeVirt, CDI, SSP, NetworkAddonsOperator, and MTQ components. Every field is annotated — applying this CR without understanding each field is not acceptable for a production banking environment.

hyperconverged.yaml — NovaCrestBank HyperConverged CR — every field annotated
apiVersion: hco.kubevirt.io/v1beta1
kind: HyperConverged
metadata:
  name: kubevirt-hyperconverged
    # kubevirt-hyperconverged: the required name. Only one HyperConverged object
  # is allowed per cluster. Attempting to create a second causes an error.
  # This is the cluster-wide singleton that configures all of OCP Virt.
  namespace: openshift-cnv
spec:

    # ── Feature Gates — capabilities to enable ──────────────────────────
  featureGates:
    enableCommonBootImageImport: true
        # enableCommonBootImageImport: true — pre-imports Red Hat common VM boot images
    # (RHEL 9, Windows Server 2022, CentOS Stream) into the cluster as DataSources.
    # These become available as base images for new VMs via the OCP console template
    # gallery. Required for the VM creation templates in the console to work.
    # Requires internet access from the cluster to pull images from quay.io.
    # In an air-gapped environment: set to false and manually import images.

    deployKubeSecondaryDNS: false
        # deployKubeSecondaryDNS: false — KubeSecondaryDNS provides DNS names for
    # VM interfaces on secondary networks. Set to false here because VM networking
    # (Multus, OVN localnet, NetworkAttachmentDefinitions) is configured in VIRT-05.
    # Enable in VIRT-05 when secondary networks are configured.

    deployVmConsoleProxy: true
        # deployVmConsoleProxy: true — deploys the VM console proxy that allows
    # the OCP web console to provide in-browser VNC/serial console access to VMs.
    # Required for the "Open console" button in the OCP Virt UI to work.
    # Without this, console access requires the virtctl CLI only.

    nonRoot: true
        # nonRoot: true — runs virt-launcher pods as non-root user.
    # This is a security hardening feature. virt-launcher used to require root
    # to access /dev/kvm and manage network interfaces. With nonRoot: true,
    # KubeVirt uses Linux capabilities (CAP_NET_ADMIN, CAP_SYS_PTRACE) instead
    # of root access. This reduces the blast radius of a virt-launcher compromise.
    # Required by PCI-DSS Req 6.3.3 (run processes with minimum necessary privileges).

    # ── LiveMigration — live migration behaviour ────────────────────────
  liveMigrationConfig:
    parallelMigrationsPerCluster: 5
        # Maximum number of concurrent live migrations cluster-wide.
    # 5 is the default. During maintenance drains, this limits how many VMs
    # migrate simultaneously to avoid overwhelming the CephFS RWX storage
    # and the ODF replication network (VLAN 120).

    parallelOutboundMigrationsPerNode: 2
        # Maximum concurrent migrations FROM a single node.
    # 2 allows two VMs to migrate off a draining node simultaneously.
    # Higher values migrate faster during maintenance but stress the source
    # node's network bandwidth and ODF storage more.

    bandwidthPerMigration: 64Mi
        # Maximum network bandwidth per migration, in bytes/second.
    # 64Mi = 64 MiB/s per migration. With parallelMigrationsPerCluster: 5,
    # worst case = 5 × 64 MiB/s = 320 MiB/s on VLAN 120 (ODF replication).
    # VLAN 120 is 25 Gbps per node = ~3 GiB/s capacity.
    # 320 MiB/s leaves ample headroom for ODF replication traffic.

    completionTimeoutPerGiB: 800
        # Seconds allowed per GiB of VM memory during migration.
    # 800 seconds per GiB. A 32 GiB VM has 32 × 800 = 25,600 seconds (~7 hours)
    # to complete migration before KubeVirt declares it failed.
    # Increase for VMs with high memory write rates (they dirty pages faster
    # than they can be transferred, requiring more iterations).

    progressTimeout: 150
        # If migration makes no progress for 150 seconds, it is aborted.
    # Progress = memory pages being transferred to the destination.
    # If the source VM is dirtying pages faster than migration can transfer them
    # AND no progress is made for 150s, the migration fails gracefully —
    # the VM keeps running on the source node unchanged.

    # ── ResourceRequirements — virt-launcher resource tuning ────────────
  resourceRequirements:
    vmiCPUAllocationRatio: 10
        # CPU overcommit ratio for VMs that do NOT use CPU pinning.
    # 10 means 10 vCPUs can be allocated for every 1 physical CPU core.
    # For NovaCrestBank's banking application VMs with CPU pinning (configured
    # in the PerformanceProfile), this ratio does not apply — pinned VMs get
    # dedicated physical cores regardless of this setting.
    # This ratio applies to dev/test VMs without CPU pinning requirements.

    # ── StorageImport — default StorageClass for DataVolume imports ──────
  storageImport:
    insecureRegistries:
      - harbor.apps.ocp.novacrestbank.local
        # harbor.apps.ocp.novacrestbank.local: NovaCrestBank's internal Harbor registry.
      # CDI (DataVolume importer) treats this registry as trusted even though it
      # uses an internal CA certificate (NovaCrestBank-Issuing-CA).
      # Without this entry, DataVolume imports from Harbor fail with TLS errors
      # because CDI cannot verify the internal CA that signed Harbor's certificate.
      # The cert-manager certificate for Harbor is issued by IssuingCA — correct.
      # But CDI's internal trust store does not include IssuingCA.
      # Listing Harbor here tells CDI to skip TLS verification for this registry only.

    # ── CertConfig — certificate rotation for internal KubeVirt PKI ─────
  certConfig:
    ca:
      duration: 48h0m0s
      renewBefore: 24h0m0s
    server:
      duration: 24h0m0s
      renewBefore: 12h0m0s
            # KubeVirt uses its own internal PKI for component-to-component communication
      # (virt-api ↔ virt-handler, migration tunnels between virt-launcher pods).
      # These are internal certificates — not visible to the guest OS or to external
      # clients. The defaults (48h CA, 24h server) are appropriate for production.
      # Shorter durations increase rotation frequency (better security) but add
      # more certificate rotation events to the audit log.

10 — PerformanceProfile — latency-sensitive VM tuning

NovaCrestBank's bare-metal nodes host latency-sensitive banking application VMs — payment processing, real-time fraud detection, core banking API. These VMs have strict P99 latency requirements. Without kernel tuning, the Linux scheduler, CPU power management, and memory management subsystems introduce latency spikes that violate those SLOs.

The PerformanceProfile CR (from the Node Tuning Operator) applies a complete set of kernel-level optimisations to the bare-metal nodes: real-time kernel configuration, CPU isolation, hugepage pre-allocation, interrupt affinity, and C-state/P-state management. It is applied as a MachineConfig — the MCO applies it to the nodes and reboots them to take effect.

What PerformanceProfile configures — and why

SettingWhat it doesWhy it matters for banking VMs
CPU isolation Reserves specific CPU cores exclusively for VM vCPUs. The Linux scheduler, kernel threads, and system processes are restricted to the remaining "housekeeping" cores. Without isolation, the kernel scheduler can preempt a VM's vCPU thread for a kernel task — introducing latency spikes of 100μs–10ms. Payment processing systems with sub-millisecond SLOs cannot tolerate this.
Hugepages (1 GiB) Pre-allocates 1 GiB memory pages instead of 4 KiB pages. VM guest RAM is backed by hugepages. The standard Linux TLB (Translation Lookaside Buffer) covers 4 KiB pages. A 32 GiB VM with 4 KiB pages requires 8 million TLB entries — causing constant TLB misses and latency spikes. With 1 GiB hugepages, a 32 GiB VM needs only 32 TLB entries — essentially zero TLB pressure.
IRQ affinity Routes hardware interrupts (NIC, disk, timer) to housekeeping cores only — not to the isolated VM cores. An NIC interrupt on a core running a payment-service VM thread preempts that thread for interrupt handling — adding unpredictable latency. Pinning interrupts to housekeeping cores eliminates this.
CPU power management Disables C-states deeper than C1 and P-state frequency scaling on isolated cores. When a CPU core enters a deep sleep state (C6, C7) to save power, waking it takes 100μs–1ms. A VM vCPU on a core that was sleeping cannot execute guest instructions until the core wakes up. For latency-sensitive VMs, the core must stay in C1 (shallow sleep) or C0 (active) at all times.
NUMA topology Ensures VM vCPUs and their memory are allocated from the same NUMA domain. AMD EPYC 9004 (Genoa) has two NUMA domains per socket. Accessing memory in the remote NUMA domain adds ~100ns latency per access compared to local NUMA. For a VM performing millions of memory operations per second, this adds up to measurable P99 latency increases.

11 — Hugepages — what they are and why VMs require them

Hugepages were configured in Section 10. This section explains exactly what hugepages are and why VMs — specifically — require them. This explanation belongs before the first VM is created, because every VM YAML in this series explicitly requests hugepages, and you must understand why before writing the request.

What standard memory pages are — the baseline

Linux manages physical memory in pages — fixed-size chunks. The standard page size on x86-64 is 4 KiB (4,096 bytes). The CPU's Memory Management Unit (MMU) translates virtual addresses (what the process sees) to physical addresses (where data actually lives in RAM) using page tables. To avoid consulting the page table on every memory access — which would be extremely slow — the CPU caches recent translations in the TLB (Translation Lookaside Buffer).

The TLB has a fixed number of entries — typically 64 to 1,024 entries depending on the CPU model. With 4 KiB pages, a TLB with 1,024 entries can cache translations for 1,024 × 4 KiB = 4 MiB of memory. When a process accesses memory outside those 4 MiB, the TLB misses — the CPU must walk the page table, which takes 10–100 nanoseconds. This is called a TLB miss penalty.

Why VMs are different from containers

A container typically uses tens to hundreds of megabytes of RAM. Its memory access patterns are concentrated — the active working set fits comfortably within what the TLB can cache. TLB pressure is manageable.

A virtual machine has a completely different profile. A 32 GiB banking application VM contains an entire operating system, JVM, application libraries, and active data — all accessing memory across the full 32 GiB address space. With 4 KiB pages, the 32 GiB guest address space requires 8,388,608 page table entries. The TLB can cache maybe 1,024 of them. TLB miss rate is extremely high, and each miss adds 10–100 ns to every memory operation the guest performs.

How hugepages solve this

With 1 GiB hugepages, the same 32 GiB VM requires only 32 page table entries — all 32 fit comfortably in the TLB simultaneously. TLB miss rate drops to near zero. The guest's memory operations run at full speed without the TLB miss penalty.

The practical impact for NovaCrestBank: a payment-processing VM with high transaction throughput running on a node with 1 GiB hugepages processes transactions at 15–30% higher throughput than the same VM on a node with standard 4 KiB pages — because more CPU cycles are spent on actual computation and fewer on TLB miss handling.

Hugepage allocation is at boot time — it cannot be done on demand

Hugepages must be pre-allocated when the node boots — before the Linux memory allocator fragments physical RAM into thousands of small allocations. Trying to allocate 1 GiB hugepages after the system has been running for hours fails because contiguous 1 GiB physical memory regions are no longer available. This is why the PerformanceProfile (applied via MCO) allocates hugepages at boot time. The count: 400 in Section 10 reserves 400 × 1 GiB = 400 GiB of contiguous physical memory at boot. If a VM requests more hugepage-backed RAM than is available on the node, the VM stays in Pending state — it does not fall back to standard pages.

12 — CPU pinning — NUMA-aware VM placement

CPU pinning means a VM's vCPUs are mapped to specific physical CPU cores that are exclusively dedicated to that VM. The Linux scheduler does not schedule any other thread on those cores while the VM is running. Combined with the NUMA topology policy from the PerformanceProfile, CPU-pinned VMs on NovaCrestBank's DL385 Gen11 nodes achieve consistent sub-millisecond processing latency.

How CPU pinning works in KubeVirt

CPU pinning is requested in the VirtualMachine CR using the dedicatedCpuPlacement: true field under the VM's CPU configuration. When this is set, KubeVirt communicates with the Kubernetes CPU Manager (which must be in static policy — configured automatically by the PerformanceProfile) to allocate dedicated physical cores for the VM's vCPUs.

The CPU Manager then uses the NUMA topology policy (single-numa-node from the PerformanceProfile) to ensure the allocated cores all come from the same NUMA domain as the VM's hugepage memory allocation. The result: a VM's vCPUs and its memory are co-located on the same NUMA domain — no cross-NUMA latency.

When to use CPU pinning and when not to

VM typeUse CPU pinning?Reason
Payment processing serviceYesSub-millisecond P99 latency SLO requires dedicated cores with no scheduler preemption
Real-time fraud detectionYesSame — real-time decision pipeline cannot tolerate CPU preemption spikes
Core banking API VMYesHigh-frequency transaction processing benefits from dedicated cores
Reporting batch processorNoBatch processing has no real-time SLO — shared CPU access is acceptable
Development/test VMNoNo production SLO — sharing cores improves node VM density
Windows Server migration targetDependsUse CPU pinning if the migrated application has latency SLOs; shared otherwise
CPU pinning reduces VM density — understand the trade-off

Each CPU-pinned VM exclusively holds its vCPUs — those cores cannot be used by any other VM or workload while the pinned VM is running. A node with 60 isolated cores running three 16-vCPU pinned VMs (3 × 16 = 48 cores used) has only 12 isolated cores remaining for additional pinned VMs. With the CPU overcommit ratio (vmiCPUAllocationRatio: 10), 60 isolated cores support 600 shared vCPUs of non-pinned VMs. CPU pinning is correct for latency-sensitive banking VMs — but applying it universally to all VMs wastes the overcommit capability of the platform.

13 — Before the YAML — understanding VirtualMachine CR

The VirtualMachine CR is the most complex object created in the VIRT series. Before showing the YAML, every structural element must be understood — because a mistake in VM YAML does not produce an error immediately. The VM CR is accepted by the API, the VMI is created, and the virt-launcher pod starts — and then fails in QEMU with an error that points back to a field in the original YAML.

The three sections of a VirtualMachine CR

spec.running / spec.runStrategy: Controls the VM's power state. runStrategy: Always means the VM should always be running — if it crashes, KubeVirt restarts it. runStrategy: Manual means the VM starts and stops only on explicit request. Use Always for production banking VMs (treat them like Kubernetes Deployments with high availability). Use Manual for development VMs that should not auto-restart.

spec.template.spec: The VM instance specification — CPU, memory, hugepages, disks, networks, and the guest OS configuration. This is the richest section and where most configuration happens. It maps to the VMI spec when the VM starts.

spec.dataVolumeTemplates: Inline DataVolume definitions that are created when the VM is first created. Used to import the OS disk image automatically. If the DataVolume import fails, the VM will not start — the VM waits for the DataVolume to be in Succeeded phase before creating the VMI.

NodeNetworkConfigurationPolicy — mentioned in previous guides, used for VMs here

In VIRT-05 (VM Networking), the NodeNetworkConfigurationPolicy (NNCP) CR is used to configure bond and VLAN sub-interfaces on bare-metal nodes for VM secondary networks. An NNCP is a declarative description of the desired NMState configuration for a set of nodes — exactly as the NMState YAML applied in VIRT-02 Section 10, but managed as a Kubernetes CR rather than a one-time apply. Every NNCP field has the same semantics as the NMState fields annotated in VIRT-02 Section 09. When VIRT-05 introduces NNCPs, each field references the NMState annotation from VIRT-02. Nothing is assumed — every directive is explained when it first appears.

14 — DataVolume — importing a guest OS image

A DataVolume is a PVC with an import process attached. When a DataVolume is created, CDI (Containerized Data Importer) starts an importer pod that downloads the OS image from the specified source, writes it into the PVC, and then terminates. The VM uses the PVC as its boot disk after the DataVolume reaches Succeeded phase. Understanding DataVolume is essential before creating any VM.

Where NovaCrestBank sources VM images

Red Hat provides QCOW2 format images for RHEL and Windows guest VMs through the Red Hat Customer Portal. These images are downloaded once, uploaded to Harbor (NovaCrestBank's internal container registry), and then referenced by DataVolumes. No VM import ever pulls directly from the internet — all image sources are internal.

For the first test VM, a RHEL 9 QCOW2 image is used. The image is stored in Harbor as a container image wrapping the QCOW2 file (using the virtctl image-upload command, covered below).
