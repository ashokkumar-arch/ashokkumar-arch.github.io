01 — Why NSX can be decommissioned — what OVN-Kubernetes replaces

NSX was the software-defined networking layer for NovaCrestBank's VCF environment. It provided three capabilities: an overlay network (Geneve tunnels between ESXi hosts), a distributed firewall (DFW — stateful L4 policy enforcement inside each ESXi kernel), and a T0/T1 gateway hierarchy for north-south traffic routing and BGP peering with the physical Cisco leaf switches. As of VIRT-10, every VM workload has been migrated from VCF to OCP Virt. The VMs that depended on the NSX overlay no longer exist on ESXi. That fact — that no traffic-forwarding VM remains on the NSX dataplane — is the prerequisite for decommissioning NSX.

NSX provided (pre-migration)
  • Geneve overlay between ESXi hosts for VM-to-VM traffic on logical segments
  • NSX DFW: stateful east-west firewall inside every ESXi kernel for segment-level micro-segmentation
  • T0 gateway: BGP speaker peering with Cisco leaf-01 and leaf-02 for VLAN 110/130/140 route advertisement
  • T1 gateways: NAT, load balancing, and routing between tenant segments
  • NSX Edge nodes: appliances running T0/T1 in dedicated ESXi VMs
  • NSX Manager: 3-node cluster providing the control plane and REST API
OVN-Kubernetes provides (post-migration)
  • Geneve overlay between OCP nodes (bare-metal) for pod and VM primary interface traffic
  • Kubernetes NetworkPolicy + OVN ACLs: east-west enforcement between pods and VM primary interfaces
  • OVN EgressGateway: BGP speaker (via MetalLB BGP mode) peering with leaf-01 and leaf-02 for OCP egress route advertisement
  • VLAN 130/140 traffic: OVN localnet — direct physical fabric, no overlay needed
  • No Edge node equivalent — routing runs in OVN on the bare-metal nodes themselves
  • No NSX Manager — OVN control plane is embedded in OCP (ovnkube-master pods in openshift-ovn-kubernetes)
The one thing that cannot be decommissioned until BGP is migrated

Everything in NSX can be removed except the T0 gateway's BGP peering with the Cisco leaf switches — until OVN EgressGateway takes over those BGP sessions. The T0 is currently advertising the OCP cluster subnets (VLAN 110: 10.110.0.0/24, VLAN 130: 10.130.0.0/24, VLAN 140: 10.140.0.0/24) to the leaf switches via BGP. If the T0 BGP sessions drop before OVN takes them over, those subnets become unreachable from the rest of the network — including the banking application VLAN 130 IPs. Section 09 covers the exact cutover procedure: add OVN BGP peer, verify route advertisement, then withdraw the NSX T0 peer. Never remove NSX before this sequence is complete.

02 — Decommission sequence — order matters, rollback at each phase

The NSX decommission proceeds in five phases. Each phase has a validation gate and a rollback procedure. No phase begins until the previous phase's gate is green. The guiding principle: remove policy before removing infrastructure. DFW rules are replaced by NetworkPolicies first; BGP sessions are migrated second; only then is the NSX dataplane (Edge nodes) removed; then the NSX control plane (Manager); then the physical fabric cleanup (TEP VLANs).

1
DFW → NetworkPolicy translation and validation
Weeks 1–2 · Policy — no traffic impact
Audit all NSX DFW rules. Translate each rule to the OVN equivalent (Kubernetes NetworkPolicy, OVN ACL, or guest OS firewall). Apply translated policies. Run parallel validation — verify the same traffic that DFW permitted/denied is permitted/denied by the new policy. Deactivate DFW rules one section at a time, verifying no traffic disruption before deactivating the next.
  • Rollback: re-enable the DFW section — instant, no infrastructure change required
2
BGP migration — NSX T0 → OVN EgressGateway
Week 3 · One-time cutover during maintenance window · Sub-minute impact
Configure OVN EgressGateway and MetalLB BGP on the bare-metal nodes to peer with leaf-01 and leaf-02. Verify OVN is advertising the correct prefixes. During a maintenance window: withdraw the NSX T0 BGP peering. The leaf switches fail over to the OVN-originated routes. Verify connectivity. If connectivity fails: re-enable NSX T0 BGP immediately (rollback is re-enabling one BGP neighbor on the T0).
  • Rollback: re-enable BGP neighbor on NSX T0 — connectivity restored in <60 seconds
3
NSX Edge node decommission
Week 4 · Post-BGP migration · No traffic impact if Phase 2 succeeded
Remove T0 and T1 gateway configurations from NSX Manager. Power off and delete the Edge node VMs from vCenter. Remove the Edge node transport nodes from NSX Manager.
  • Rollback: not straightforward — Edge VMs must be re-deployed from OVA. This phase is one-way. Ensure Phase 2 is fully validated before proceeding.
4
NSX Manager cluster decommission
Week 4 · After Edge decommission
Remove all host transport nodes (ESXi hosts) from NSX. Uninstall the NSX vibs from ESXi hosts. Power off and delete the 3 NSX Manager VMs. Release the NSX licenses from VMware's licensing portal.
  • Rollback: not possible after Manager VMs are deleted. This phase is irreversible.
5
Leaf switch TEP VLAN removal and VCF exit
Week 5 · Final cleanup
Remove the NSX TEP VLAN configuration from the Cisco Nexus leaf switches (VLANs and port-channel subinterfaces used for Geneve encapsulation between ESXi and Edge nodes). Decommission vCenter, SDDC Manager, and remaining ESXi hosts. Close VMware contracts.
  • Rollback: leaf configuration can be restored from backup. Non-critical at this stage.

04 — NSX DFW model — what it does and how OVN replaces it

NSX Distributed Firewall (DFW) is a stateful L4 firewall that runs inside every ESXi kernel. Every packet from every VM passes through the DFW kernel module before it reaches the virtual switch. Rules are grouped into Sections (with a priority order) and target NSX Security Groups (dynamic groups of VMs matched by tag, name pattern, or segment). DFW provides micro-segmentation — fine-grained east-west traffic control between VMs on the same segment — without any traffic hairpin to a physical firewall.

In OCP, the equivalent is the combination of three controls. Kubernetes NetworkPolicy enforces east-west policy at the OVN overlay level — controlling traffic between pods and VM primary interfaces. OVN ACLs (applied at the localnet port level) control traffic on VLAN 130/140 secondary interfaces. Guest OS firewall (firewalld on RHEL, Windows Firewall via GPO) controls inbound traffic on the VM's secondary interface from the physical fabric. No single OCP control maps 1:1 to DFW — the three-control model together provides equivalent coverage.

NSX DFW capabilityOVN / OCP equivalentScope
East-west between VMs on the same segmentKubernetes NetworkPolicy between pods/VMIs in the same namespaceOVN overlay (primary interface traffic)
East-west between VMs on different segmentsNetworkPolicy with namespaceSelector across namespacesOVN overlay
North-south from external to VM (VLAN 130/140)Guest OS firewalld / Windows Firewall (VIRT-08 Sections 13–14)Physical fabric (localnet — OVN ACL not yet used)
L7 application-layer rules (e.g., allow HTTP but block HTTPS)Not available in standard NetworkPolicy — requires a Service Mesh (Istio/OpenShift Service Mesh)N/A for NovaCrestBank Phase 1

05 — DFW rule-by-rule translation to Kubernetes NetworkPolicy

NovaCrestBank's NSX DFW has four sections relevant to the migrated VM workloads. Each section is translated below to the OVN equivalent that was applied in VIRT-08. The translation follows a consistent pattern: NSX Security Group → Kubernetes label selector, NSX service (port/protocol) → NetworkPolicy ports, NSX source/destination → NetworkPolicy from/to.

DFW Section 1: VM-to-VM banking traffic

NSX DFW rule → Kubernetes NetworkPolicy translation
# ── NSX DFW Rule (original) ─────────────────────────────────────────────
# Section: Banking-App-East-West
# Name: Allow API Gateway to Banking App
# Source: NSX Security Group "SG-API-Gateway" (VMs tagged app:api-gateway)
# Destination: NSX Security Group "SG-Banking-App" (VMs tagged app:banking-app)
# Service: TCP port 8080
# Action: Allow
# Applied-To: SG-Banking-App

# ── OVN Kubernetes NetworkPolicy (replacement) ───────────────────────────
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-gateway-to-banking-app
  namespace: virt-workloads
    # Namespace-scoped: applies to pods in virt-workloads.
  # NSX DFW was cluster-scoped across all segments — NetworkPolicy is namespace-scoped.
  # For cross-namespace traffic: use namespaceSelector in the from/to block.
spec:
  podSelector:
    matchLabels:
      app: banking-app
        # podSelector: the "Applied-To" equivalent.
    # This NetworkPolicy applies to pods labelled app=banking-app.
    # In OCP Virt: the virt-launcher pod for vm-banking-app-01 has
    # this label set on the VirtualMachine CR (labels.app: banking-app).
    # The label propagates to the virt-launcher pod automatically.
  ingress:
    - from:
        - podSelector:
          matchLabels:
            app: api-gateway
                # from.podSelector: the "Source" equivalent.
        # Matches pods labelled app=api-gateway in the same namespace.
        # NSX DFW matched by Security Group tag — here we match by Kubernetes label.
        # The label is on the VirtualMachine CR for vm-api-gateway-01.
      ports:
        - protocol: TCP
          port: 8080
                    # ports: the "Service" equivalent. TCP port 8080.
          # NetworkPolicy ports are INGRESS ports on the destination pod —
          # matching the DFW destination service (port 8080 on SG-Banking-App).
  policyTypes:
    - Ingress
---
# ── DFW Rule 2: Allow banking app to ODF storage ─────────────────────────
# NSX: SG-Banking-App → SG-ODF-Storage, TCP 3260 (iSCSI) + TCP 6789/3300 (Ceph mon)
# OVN: virt-launcher pods (banking-app) → ODF pods (openshift-storage namespace)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-banking-app-to-odf
  namespace: openshift-storage
    # This NetworkPolicy is in the openshift-storage namespace — not virt-workloads.
  # It permits ingress TO ODF pods FROM the virt-workloads namespace.
  # Cross-namespace: use namespaceSelector to identify the source namespace.
spec:
  podSelector:
    matchLabels:
      app: rook-ceph-osd
  ingress:
    - from:
        - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: virt-workloads
                    # namespaceSelector: select pods from a different namespace.
          # kubernetes.io/metadata.name is automatically set on all namespaces.
          # This permits all pods from virt-workloads to reach ODF OSD pods.
          # Note: ODF-to-VM storage traffic uses the pod network (OVN overlay)
          # for the RBD protocol — no VLAN 130 involvement for storage access.
      ports:
        - protocol: TCP
          port: 6800
                    # Ceph OSD messenger port (6800-7300 range — OVN allows the base port here;
          # for production use a port range via endPort in NetworkPolicy v1.25+)
        - protocol: TCP
          port: 6789
                    # Ceph Monitor port — used for cluster coordination

06 — What DFW does that NetworkPolicy cannot — and the compensating controls

Kubernetes NetworkPolicy controls traffic at the OVN overlay level. Three DFW capabilities have no direct NetworkPolicy equivalent, and each requires a specific compensating control.

DFW capabilityNetworkPolicy limitationCompensating control
VLAN 130/140 east-west
DFW enforced rules between VMs communicating via VLAN 130 IPs directly (same segment, not through T0)		 NetworkPolicy controls OVN overlay traffic only. VLAN 130 traffic goes directly through the physical fabric — NetworkPolicy has no visibility into it. Guest OS firewalld (VIRT-08 Section 13) restricts inbound connections on eth1 (VLAN 130) to specific ports. Cisco leaf ACLs at VLAN boundary provide inter-VLAN enforcement. For Phase 2+ (VIRT-11 complete): OVN localnet ACLs via AdminNetworkPolicy can be applied at the port level.
Layer 7 inspection
NSX DFW with context profiles could permit HTTP but deny HTTPS on the same port, or match specific URL patterns		 NetworkPolicy is L3/L4 only — port and protocol matching only. No application-layer awareness. NovaCrestBank Phase 1 does not require L7 DFW rules. The migrated banking VMs use port-based access control. If L7 is required in future: OpenShift Service Mesh (Istio) provides L7 policy between service mesh-enrolled workloads.
Identity-based rules (NSX Security Tag)
DFW rules that dynamically include VMs when they receive a specific NSX tag (applied via vCenter or NSX API)		 NetworkPolicy label selectors are static — a pod must have the label at creation time. Dynamic tag application (equivalent to tagging a running VM) requires modifying the pod label — which is not possible without restarting the pod. ACM Governance policies (VIRT-09 Section 10) enforce required labels on all VirtualMachine CRs. The label is set at VM creation and maintained by ACM policy. Label drift triggers ACM enforcement. VirtualMachine CR labels propagate to virt-launcher pods automatically — no pod restart required.

08 — OVN EgressGateway — replacing NSX T0 as the BGP speaker

OVN-Kubernetes provides BGP advertisement capability through MetalLB operating in BGP mode. MetalLB is deployed as the OCP LoadBalancer provider and configured with BGP peers pointing to leaf-01 and leaf-02. It then advertises the required subnets — the same 6 prefixes currently advertised by the NSX T0 — from the bare-metal OCP nodes directly to the leaf switches. When the leaf switches see the same prefixes arriving via both NSX T0 (AS 65099) and OVN/MetalLB (AS 65100), they install both paths. The NSX T0 path is then withdrawn, leaving only the OVN path.

metallb-bgp-config.yaml — MetalLB BGP configuration for OVN EgressGateway
# Install MetalLB operator first via OLM, then configure BGP mode
apiVersion: metallb.io/v1beta1
kind: MetalLB
metadata:
  name: metallb
  namespace: metallb-system
spec:
  nodeSelector:
    workload-type: virtualization
    # Run MetalLB speakers only on the 5 bare-metal nodes.
  # These nodes have direct L2 adjacency with the Cisco leaf switches
  # (VLAN 110 port-channels Po211–Po215 from VIRT-01).
  # MetalLB BGP sessions originate from the bare-metal node IPs (10.110.0.11–.15).
---
# BGP peer: leaf-01
apiVersion: metallb.io/v1beta2
kind: BGPPeer
metadata:
  name: leaf-01
  namespace: metallb-system
spec:
  peerAddress: 10.10.0.1
    # peerAddress: leaf-01 management IP (from VIRT-01 SVI configuration).
  peerASN: 65001
    # peerASN: leaf-01's BGP ASN — 65001 (from VIRT-01 BGP configuration).
  myASN: 65100
    # myASN: OCP cluster's BGP ASN — 65100.
  # Different from NSX T0's ASN (65099) so the leaf switches can distinguish
  # routes originating from OVN vs routes originating from NSX T0.
  # When both are active: ECMP across both paths for redundancy.
  holdTime: 90s
  keepaliveTime: 30s
  nodeSelectors:
    - matchLabels:
        workload-type: virtualization
---
# BGP peer: leaf-02 (identical structure)
apiVersion: metallb.io/v1beta2
kind: BGPPeer
metadata:
  name: leaf-02
  namespace: metallb-system
spec:
  peerAddress: 10.10.0.2
  peerASN: 65002
  myASN: 65100
  holdTime: 90s
  keepaliveTime: 30s
  nodeSelectors:
    - matchLabels:
        workload-type: virtualization
---
# IPAddressPool: the subnets MetalLB advertises via BGP

11 — NSX Manager cluster decommission

The NSX Manager is a 3-node appliance cluster (management VMs on ESXi) that provides the NSX control plane: the REST API, the policy engine, and the central inventory of transport nodes, segments, and gateways. With the T0/T1 gateways deleted and the Edge nodes removed, the Manager has no remaining forwarding-plane entities to manage. The decommission sequence: remove ESXi host transport nodes (uninstall NSX vibs from each ESXi), then power off and delete the 3 Manager VMs.

Confirm ESXi hosts are no longer transport nodes before proceeding

NSX installs kernel modules (vibs) into each ESXi host that was a transport node — this enables the DFW and the Geneve overlay within the ESXi kernel. These vibs must be removed cleanly via NSX Manager before the Manager VMs are deleted. If the Manager VMs are deleted first, the vibs cannot be removed through the normal API path — manual ESXi vib removal via esxcli is required. Always remove ESXi transport nodes from NSX Manager before deleting the Manager VMs.
