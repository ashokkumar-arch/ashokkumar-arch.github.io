01 — Why a dedicated Windows Server guide

In a banking environment with formal change management, every Active Directory change requires its own change request, its own approval, and its own implementation record. The Windows Server and AD team operates independently from the platform team — they cannot be expected to open six different guides across the VIRT series to discover what DNS records need to be added, what DHCP scopes need to exist, and what OU structure is expected.

This guide is the single document that answers the question: "What does the OpenShift Virtualization project need from Active Directory, DNS, DHCP, and PKI — and what change request do I need to raise?" Every Windows Server change the entire VIRT series makes is documented here, in the order it needs to happen, with the exact procedure for each change.

For the platform engineer: this guide also serves as confirmation that the existing VCF Windows Server infrastructure is reused without modification — you do not need to reconfigure AD integration, rebuild a PKI hierarchy, or create new GPOs. What you do need to know is precisely documented in Section 03 (the complete change register).

Audience — this guide serves two teams

Windows/AD administrator: Sections 03 through 20 are yours. Every change that requires AD access is here with step-by-step procedures. Section 21 is the sign-off checklist you return signed before the platform team proceeds to VIRT-03.

Platform/OCP engineer: Section 02 confirms that the existing AD integration, PKI trust, and OAuth configuration require zero changes. Sections 05 and 09 are verification steps you run from the jump host to confirm the AD team's work is complete before VIRT-03.

The OpenShift on VCF + NSX series (Series II) established a complete Windows Server infrastructure that the VIRT series inherits in full. Before describing what changes, it is important to document precisely what does not change — because in a banking environment, the absence of change is as important to document as the presence of change.

The architecture decision that makes this possible

The reason so little changes in Windows Server for the VIRT series is the architecture decision documented in VIRT-00 ADR-VIRT-01: the bare-metal nodes join the existing OCP cluster rather than forming a new one. If a new cluster were created, it would require a new additionalTrustBundle, new AD OAuth integration, a new cert-manager ClusterIssuer, and potentially a new DNS zone for a new cluster domain. By joining the existing cluster, all of that infrastructure is already in place. One cluster. One Windows Server integration. The VIRT series adds five nodes to something that already exists — not a second parallel system.

04 — DNS — what the VIRT series needs and why

DNS is the foundational dependency for everything else. Before a VM can be reached by name, before a migrated VM can join AD, before the OCP cluster can resolve bare-metal node FQDNs — DNS must be correct. This section establishes the complete DNS picture for the VIRT series before any record is created.

The DNS server — no change needed

DNS is served by DC01 and DC02 using Active Directory-integrated zones for novacrestbank.local. The Assisted Installer embedded DC01 (10.70.0.10) as the DNS server for all bare-metal nodes in the DHCP scope (VIRT-00) and in the NMState static configuration (VIRT-02). The bare-metal nodes already use DC01/DC02 for DNS. No DNS server configuration changes are needed — only new records in existing zones.

The naming convention — consistent with the VCF series

All new FQDNs follow the pattern established in the VCF series: <hostname>.novacrestbank.local. VM workload records follow the same convention: vm-<function>-<number>.novacrestbank.local. This ensures that any engineer who knows the naming convention from the VCF guides can immediately understand the VIRT naming without a separate lookup.

Why DNS records for VMs must exist before the VM is created

When a VM is created in OpenShift Virtualization (VIRT-05) or migrated from VMware (VIRT-06), the VM guest OS starts and expects to be reachable by its hostname. If the DNS A record does not exist at that moment, several things fail immediately: Windows VMs cannot join the domain (AD domain join requires forward DNS resolution to succeed before the join is attempted), monitoring agents cannot register the VM by name, and the NSX/NetworkPolicy rules that reference FQDNs cannot resolve the VM's IP. Creating DNS records before the VM exists — a planned, deliberate act — is the correct sequence.

05 — DNS: bare-metal node records — verification

The bare-metal node DNS records were created in VIRT-00 Section 07d as a provisioning prerequisite. This section verifies they are correct and complete. If they are not present, VIRT-02 could not have succeeded — but verify anyway, because DNS replication between DC01 and DC02 occasionally lags and a record visible on DC01 may not yet be visible on DC02.

jumphost — verify bare-metal node DNS records on both DCs
# Verify from both DC01 and DC02 — replication must be complete on both [admin@jumphost ~]$ for dc in 10.70.0.10 10.70.0.11; do echo "=== DNS server: ${dc} ===" for n in 01 02 03 04 05; do ip="10.110.0.1${n#0}" fwd=$(dig +short bm-virt-${n}.novacrestbank.local @${dc} 2>/dev/null) rev=$(dig +short -x ${ip} @${dc} 2>/dev/null) status="✓" [[ "${fwd}" != "${ip}" ]] && status="✗ FORWARD MISSING" [[ -z "${rev}" ]] && status="${status} ✗ PTR MISSING" echo " bm-virt-${n}: A=${fwd} PTR=${rev} ${status}" done done === DNS server: 10.70.0.10 === bm-virt-01: A=10.110.0.11 PTR=bm-virt-01.novacrestbank.local. ✓ bm-virt-02: A=10.110.0.12 PTR=bm-virt-02.novacrestbank.local. ✓ bm-virt-03: A=10.110.0.13 PTR=bm-virt-03.novacrestbank.local. ✓ bm-virt-04: A=10.110.0.14 PTR=bm-virt-04.novacrestbank.local. ✓ bm-virt-05: A=10.110.0.15 PTR=bm-virt-05.novacrestbank.local. ✓ === DNS server: 10.70.0.11 === bm-virt-01: A=10.110.0.11 PTR=bm-virt-01.novacrestbank.local. ✓ bm-virt-02: A=10.110.0.12 PTR=bm-virt-02.novacrestbank.local. ✓ bm-virt-03: A=10.110.0.13 PTR=bm-virt-03.novacrestbank.local. ✓ bm-virt-04: A=10.110.0.14 PTR=bm-virt-04.novacrestbank.local. ✓ bm-virt-05: A=10.110.0.15 PTR=bm-virt-15.novacrestbank.local. ✓ ✓ All bare-metal node A and PTR records are present and identical on both DC01 and DC02. ✓ AD DNS replication is complete for these records.
If records are present on DC01 but missing on DC02
Cause: AD DNS replication has not completed. AD-integrated DNS zones replicate via AD replication — not DNS zone transfer. If DC01 and DC02 replication is healthy, new records propagate within 15 seconds. If replication is delayed or broken, records may not appear on DC02 for minutes or not at all.
Check replication health: On DC01, run repadmin /showrepl in PowerShell. Any replication failure appears as a non-zero error count. If replication is failing, fix it before proceeding — DNS that works on DC01 but not DC02 means half the environment loses DNS when DC01 is unavailable.
Force immediate replication: repadmin /syncall /AdeP on DC01 forces replication to all partners immediately. Wait 30 seconds and re-run the verification command above.

06 — DNS: VLAN 130 VM workload records

VLAN 130 (vm-workload-primary, 10.130.0.0/24) is where banking application VMs migrated from the hypervisor environment will live. DNS records for these VMs must exist before the VMs are created or migrated — because the VM's first action after boot is often a DNS lookup (to find a domain controller for AD join, or to find another service it depends on).

Which VMs get records in VLAN 130

Two categories of VMs use VLAN 130 DNS records:

  • New VMs created in OpenShift Virtualization (VIRT-05): These are VMs built from templates directly on the OCP Virt platform. DNS records are created here before the VM template is deployed.
  • VMs migrated from the hypervisor environment (VIRT-06 MTV): These VMs may already have DNS records in the existing environment pointing to their old IPs (on NSX segments). After migration, their IPs change to the VLAN 130 range. Section 08 covers the record update procedure for migrated VMs — this section covers new VMs only.

Use static IPs for VMs that run critical banking services

The DNS records above use static IPs. This is correct for VMs running banking services — their IP must be predictable and stable. A VM running a payment processing service cannot have its IP change because a DHCP lease expired. Static IP + DNS A record is the correct pattern. DHCP (Section 11) is for VMs that do not need a predictable IP — development VMs, test environments, short-lived batch processors.

07 — DNS: VLAN 140 VM workload DMZ records

VLAN 140 (vm-workload-dmz, 10.140.0.0/24) hosts VMs that require stricter security separation from the primary VM workload — for example, VMs hosting partner-facing APIs, pre-production environments, or systems designated as DMZ-adjacent by the security team. The DNS procedure is identical to Section 06.

FQDNTypeIPPurpose
vm-api-gateway-01.novacrestbank.localA + PTR10.140.0.50Partner API gateway VM — DMZ-adjacent, external partner connectivity
vm-preprod-01.novacrestbank.localA + PTR10.140.0.100Pre-production environment VM — isolated from production VLAN 130
The difference between VLAN 130 and VLAN 140 in DNS terms

From DNS Manager's perspective, records for both VLANs go into the same novacrestbank.local forward zone. The difference is in the IP range — VLAN 130 VMs have IPs in 10.130.0.0/24 and their PTR records go in 130.10.10.in-addr.arpa. VLAN 140 VMs have IPs in 10.140.0.0/24 and their PTR records go in 140.10.10.in-addr.arpa. DNS itself does not enforce security separation — that is NSX gateway firewall (Model A) or Kubernetes NetworkPolicy (Model B) in VIRT-05.

08 — DNS: post-migration record updates — MTV reference procedure

This section documents the DNS update procedure for VMs migrated from the hypervisor environment in VIRT-06. It is documented here so the AD team understands the full scope of change before VIRT-06 begins. The actual execution happens during the VIRT-06 migration cutover — but the procedure is approved here.

What changes when a VM is migrated

A VM running in the current hypervisor environment has an IP address on one of the NSX overlay segments (for example, 10.70.0.x on seg-infra-services, or on a banking application segment). After migration to OpenShift Virtualization, the VM's new network interface is on VLAN 130 (10.130.0.0/24) or VLAN 140 (10.140.0.0/24).

The VM's hostname does not change. The DNS FQDN does not change. Only the IP address changes. The DNS update is: delete the old A record and PTR record, create a new A record with the new IP, create a new PTR record in the VLAN 130/140 reverse zone.

The migration DNS update procedure

This procedure is executed during the cutover window in VIRT-06. Document it here so the AD team can prepare and have the DNS Manager open and ready during the migration window.

Step 1 — Record the old IP before migration begins
Before starting MTV migration, record the VM's current IP: nslookup <vm-fqdn> 10.70.0.10
Record both the A record IP and the reverse zone it lives in. You will need this to delete the old PTR record.
Step 2 — Delete the old A record
In DNS Manager: Forward Lookup Zones → novacrestbank.local
Find the VM's hostname. Right-click → Delete. Confirm.
Do not delete the record until after the VM has stopped responding at the old IP — deleting it too early causes a DNS outage for any system that depends on the VM by name.
Step 3 — Delete the old PTR record
In DNS Manager: navigate to the reverse zone matching the VM's old IP.
Find the PTR record for the old IP. Right-click → Delete. Confirm.
Step 4 — Create the new A record with the VLAN 130 IP
Forward Lookup Zones → novacrestbank.local → Right-click → New Host (A or AAAA)
Name: same hostname as before (only the IP changes)
IP: new IP from VLAN 130 (10.130.0.x) or VLAN 140 (10.140.0.x)
☑ Check "Create associated pointer (PTR) record"
Click Add Host
Step 5 — Verify from jump host
nslookup <vm-fqdn> 10.70.0.10 must return the new IP.
nslookup <new-ip> 10.70.0.10 must return the VM's FQDN.
Allow 15–30 seconds for AD DNS replication before verifying against DC02.
DNS TTL and the migration window

AD-integrated DNS zones in NovaCrestBank use the default TTL of 1 hour. Any system that cached the VM's old IP will continue using it for up to 1 hour after the DNS record is updated. For banking systems where sub-minute cutover is required, the migration window must account for DNS TTL. The pre-migration checklist in VIRT-06 includes a TTL reduction step — temporarily setting the VM's DNS record TTL to 60 seconds one hour before migration begins, ensuring clients refresh quickly after cutover.

Why the VLAN 110 discovery scope is temporary in purpose but permanent in configuration

The VLAN 110 DHCP scope (10.110.0.20–.250) was created for the Assisted Installer discovery phase. 11 — DHCP: new scope for VLAN 130 (vm-workload-primary)

VMs on VLAN 130 that do not use a static IP — for example, development VMs, test instances, or short-lived batch processors — need DHCP. The scope is created on DC01 and automatically replicated to DC02 through the DHCP failover partnership that was established in the Classic series.

Why not give every VM a static IP?

Static IPs are correct for VMs that run production services with stable hostnames and firewall rules referencing their IP. But not every VM needs a static IP — a developer spinning up a test VM should not have to coordinate with the AD team to reserve an IP address. DHCP for the 10.130.0.50–.200 range handles dynamic VMs, while 10.130.0.10–.49 is reserved for static assignments (like the VMs in Section 06). The split is clean and administratively manageable.

The DHCP relay on the leaf switch is already in place

In VIRT-01 Section 08, the VLAN 130 SVI on leaf-01 and leaf-02 was configured with ip helper-address 10.70.0.10. 13 — DHCP: reservations for migrated VMs

Some VMs migrated from the hypervisor environment are configured to use DHCP rather than static IPs — but they need the same IP every time they boot. A DHCP reservation satisfies both requirements: the VM uses DHCP (no changes to the VM's network configuration), but DC01 always assigns it the same IP based on its MAC address.

When to use DHCP reservations vs static IPs

Use a DHCP reservation when: the VM's OS is configured for DHCP (changing it to static requires OS-level change which is a separate change request), or the VM is a development/test VM where a static IP assignment is administratively disproportionate to the risk. Use a true static IP (configured in the VM via cloud-init or NMState) when: the VM runs a production service, the IP appears in firewall rules or monitoring configurations, or the VM's IP must be stable across even a DC01/DC02 DHCP failure.

Creating a DHCP reservation

A DHCP reservation requires the VM's MAC address. For a VM migrated from VMware, the MAC address is available in vCenter before migration. For a VM running on OpenShift Virtualization, the MAC is in the VirtualMachine CR (set by KubeVirt or explicitly specified).

16 — New OU for OpenShift Virtualization VMs

When Windows Server VMs are migrated from the hypervisor environment to OpenShift Virtualization (VIRT-06) and join the AD domain, they need an OU to join into. Creating a dedicated OU for OpenShift Virtualization-hosted VMs provides two operational benefits: it visually separates OCP-hosted VMs from hypervisor-hosted VMs in AD management tooling, and it allows future Group Policy targeting if VM-specific policies are needed (for example, a GPO that applies only to VMs running on OCP Virt).

Why a dedicated OU rather than using the existing Servers OU

The existing OU=Servers works for all server computer accounts. Migrated VMs could join there without issue. However, as the VM migration progresses and more banking application VMs move to OpenShift Virtualization, the ability to apply a GPO specifically to OCP-hosted VMs — for example, a Group Policy that configures Windows VMs to report their OpenShift node as the primary management endpoint to monitoring tools — requires a dedicated OU. Creating the OU now, before the first VM is migrated, is cheaper than reorganising AD after migration.
