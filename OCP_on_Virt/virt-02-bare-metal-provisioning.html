01 — What this guide does — and what it does not

This guide provisions five HP ProLiant DL385 Gen11 bare-metal servers as RHCOS worker nodes and joins them to the existing OCP cluster that was built in the OpenShift on VCF + NSX series. By the end of this guide, running oc get nodes will show five new bm-virt-xx nodes in Ready state alongside the existing master and worker nodes.

This guide does not install OpenShift Virtualization (the HyperConverged Operator). It does not configure ODF. It does not create any VMs. Those are VIRT-03 and VIRT-04. The only thing this guide does is add bare-metal nodes to the cluster. Everything else comes after.

Why this order? Because OpenShift Virtualization requires nodes before it can be installed. ODF requires nodes with raw disks before it can be configured. The node provisioning step is the foundation — it must be solid and verified before anything is built on top of it.

Why the Assisted Installer Add Hosts flow — not manual coreos-installer

There are two ways to add bare-metal nodes to an existing OCP cluster. The manual method downloads the RHCOS live ISO, boots each server from it, runs coreos-installer with the ignition config URL, approves CSRs, and repeats for each node. It works — but it is a sequence of manual steps with multiple places for error to hide.

The Assisted Installer Add Hosts flow uses the same agent-based technology that Red Hat uses for its own internal deployments. You generate one discovery ISO, boot all five servers from it simultaneously, and the Assisted Installer handles discovery, pre-installation validation, and the RHCOS installation on each node in a managed sequence. It also produces detailed logs for each node — a complete audit trail of what happened on each server during provisioning. For a banking environment with change management requirements, this audit trail is valuable. For a consulting architect portfolio, demonstrating familiarity with the Assisted Installer shows current knowledge of Red Hat's recommended approach.

02 — Pre-flight checks — run all before powering any server

The Classic series' approach to pre-flight checks applies here without exception: every check is a gate. If any gate fails, stop and fix it before continuing. The 20 minutes spent fixing a pre-flight failure saves hours of debugging a failed installation.

The non-negotiable rule: do not power on any bare-metal server until all five pre-flight gates pass. A node that boots before the DNS records exist, before the DHCP relay is working, or before the iLO credentials are set will fail in a way that produces confusing symptoms — because the failure happens in the first 30 seconds of boot, before any useful logs are written.

02a — Gate 1: Cluster health

Adding nodes to a degraded cluster compounds the problem. Before touching any server, verify the cluster is fully healthy. Every cluster operator must be Available, not Progressing, and not Degraded.

02b — Gate 2: DNS — forward and reverse for all five nodes

The Assisted Installer agent performs a reverse DNS lookup of the node's IP during discovery. If the PTR record does not exist or does not resolve to the correct FQDN, the node fails the validation phase with a hostname mismatch error. Both forward (A) and reverse (PTR) records must exist and resolve correctly before any node is powered on.

These records were created in VIRT-00 Section 07d. This gate confirms they are actually resolving from the jump host — not just that they were entered in DNS Manager.

jumphost — DNS gate — forward and reverse
# Gate 2: Test forward (A) and reverse (PTR) DNS for all 5 nodes
# DNS server is DC01 at 10.70.0.10

[admin@jumphost ~]$ for n in 01 02 03 04 05; do
  fwd=$(dig +short bm-virt-${n}.novacrestbank.local @10.70.0.10)
  rev=$(dig +short -x ${fwd} @10.70.0.10 2>/dev/null)
  echo "bm-virt-${n}: A=${fwd} PTR=${rev}"
done

DNS gate failure — causes and fixes
A record returns empty: The forward A record was not created in DNS Manager on DC01. Open DNS Manager → Forward Lookup Zones → novacrestbank.local → right-click → New Host (A or AAAA) → enter the name (bm-virt-01) and IP (10.110.0.11) → check "Create associated pointer (PTR) record" → Add Host.
PTR record returns empty but A record works: The reverse zone does not exist. Open DNS Manager → Reverse Lookup Zones → right-click → New Zone → Primary zone → AD-integrated → enter network ID 10.110.0. → finish. Then re-create the A record with "Create associated pointer" checked.
DNS times out (no response): The jump host's /etc/resolv.conf does not point to DC01. Verify with cat /etc/resolv.conf — it must show nameserver 10.70.0.10.

03 — What the Assisted Installer is — from first principles

Before running any command, understand what the Assisted Installer is and what role each of its components plays. This understanding is what allows you to diagnose problems when they occur — because you know where each piece of the process lives and what it is responsible for.

The problem the Assisted Installer solves

Adding a bare-metal node to an OCP cluster involves a sequence of steps that must happen in the right order: boot the server with the correct OS image, apply the correct ignition configuration, install RHCOS to disk, reboot, have the kubelet register with the API server, and then have a cluster administrator approve the Certificate Signing Request (CSR) that the new node submits. If any step fails, the node does not join the cluster — and the error is often cryptic because it happens before the OS is fully operational and before any logging infrastructure is running.

The Assisted Installer wraps this entire sequence in a managed workflow. It runs a lightweight agent on each node during the provisioning process that communicates with a service (the Assisted Installer service, running either in the cloud or on-cluster) to coordinate the provisioning, validate the hardware, and report progress. The engineer interacts with a single interface — the Assisted Installer UI or API — rather than managing each node individually.

The four components you will work with

The Assisted Installer service: For clusters managed by Red Hat (ROSA, OCP on cloud), the Assisted Installer service runs at api.openshift.com and is accessed via the Red Hat Hybrid Cloud Console. For on-premises clusters like NovaCrestBank's, the Assisted Installer service is accessed via the Console's "Add hosts" workflow for an existing cluster. This service manages the provisioning lifecycle — it tracks each node's state (discovering, validating, installing, installed, failed) and coordinates the ISO generation.

The discovery ISO: A bootable ISO image generated specifically for your cluster. It contains the Assisted Installer agent pre-configured with your cluster's API endpoint and pull secret. When a node boots from this ISO, the agent automatically discovers the node's hardware (CPU count, RAM, disk inventory, NIC list), runs pre-installation validations, and registers the node with the Assisted Installer service — all without any manual interaction on the node console.

The Assisted Installer agent: A process running inside the discovery ISO's RAM disk on each booted node. It communicates outbound to the Assisted Installer service. It reports hardware inventory, runs validation checks (does the node have enough RAM? is the disk large enough? are the NICs configured correctly?), and then executes the RHCOS installation when the engineer triggers it from the UI or API. The agent runs entirely in RAM — it does not touch the node's disk until installation is explicitly triggered.

CSR approval: After RHCOS is installed and the node reboots into its new OS, the kubelet process starts and submits a Certificate Signing Request to the cluster's API server. This CSR is the node's way of saying "I exist and I want to join the cluster." A cluster administrator must approve this CSR — it is a security gate that prevents unauthorised nodes from joining. This guide covers CSR approval in Section 13.

What "Add Hosts" means for an existing cluster

The Assisted Installer "Add Hosts" flow is specifically designed for the use case of adding bare-metal nodes to an existing cluster — which is exactly what VIRT-02 does. It differs from a full cluster installation in one important way: the existing cluster's API endpoint and credentials are used to generate the discovery ISO. The new nodes join the existing cluster — they do not form a new one. The existing master nodes, existing worker nodes, and all existing operators continue to run without interruption during the provisioning of the new nodes.

04 — The Add Hosts flow — understanding the complete sequence before running it

Here is the complete sequence of what happens from the moment you trigger the Add Hosts workflow to the moment the new nodes appear as Ready in the cluster. Read this entire section before running any command — knowing the full sequence in advance means you can recognise where you are in the process when something unexpected happens.

  1. Generate discovery ISO (Section 05): The openshift-install agent command generates a bootable ISO specific to your cluster. This ISO contains the Assisted Installer agent, your cluster's API endpoint, and your pull secret. It takes approximately 2 minutes.
  2. Mount ISO via iLO virtual media (Section 06): The ISO is served from the jump host's HTTP server (set up in VIRT-00 Section 07b) and mounted to each server's virtual DVD drive via the iLO Redfish API. No physical media is involved.
  3. Boot each server from virtual DVD (Section 07): Each server is powered on and booted from the virtual DVD. The server boots into RAM — it does not touch its local NVMe disks. The Assisted Installer agent starts and sends a DHCP discover on VLAN 110 to get a temporary IP from DC01's discovery DHCP scope.
  4. Agent registers with the cluster: Once the node has an IP, the agent calls the cluster's infrastructure API endpoint (https://api.ocp.novacrestbank.local:6443) to register itself and report its hardware inventory. This is why gate 1 (cluster health) and gate 2 (DNS — specifically the API endpoint) must pass before this step.
  5. Discovery and validation (Section 08): The Assisted Installer validates each node's hardware against the minimum requirements: CPU count, RAM, disk size, NIC count, network connectivity. Validation results appear in the Assisted Installer UI within 3–5 minutes of the node booting.
  6. Configure static IPs via NMState (Section 10): Before triggering installation, each node's permanent network configuration is set via NMState. This assigns the static IP (10.110.0.11–.15), configures the LACP bond (bond0), and creates VLAN sub-interfaces for VLANs 110, 120, 130, and 140.
  7. Trigger installation (Section 11): Installation is triggered on all five nodes simultaneously. The agent writes the RHCOS image to each node's OS disk (the 2 × 480 GB SSD RAID-1 volume), applies the ignition configuration, and reboots the node.
  8. Node reboots into RHCOS: After installation, each node reboots from its local disk. The Assisted Installer agent is no longer running — RHCOS is now fully in control. The kubelet starts and submits a CSR to the API server.
  9. CSR approval (Section 13): The cluster administrator approves the pending CSRs. Each node typically requires two CSR approvals — one for the node's client certificate and one for its serving certificate.
  10. Node joins and becomes Ready: After CSR approval, the node completes its bootstrap sequence, the DaemonSets (kubelet, CSI drivers, monitoring agents) deploy to it, and it transitions to Ready state. The entire process from boot to Ready takes approximately 20–25 minutes per batch of nodes.

05 — Generate the discovery ISO

The discovery ISO is generated using the openshift-install agent create image command. This command requires two input files: install-config.yaml (which defines cluster parameters) and agent-config.yaml (which defines the hosts being added). Both files are consumed and deleted by the command — always keep backups.

What install-config.yaml provides for Add Hosts

For an Add Hosts operation on an existing cluster, the install-config.yaml does not define a new cluster — it tells the agent which existing cluster to join. The critical fields are the cluster name (metadata.name: ocp) and base domain (baseDomain: novacrestbank.local), which together form the cluster's API endpoint: api.ocp.novacrestbank.local. The agent uses this endpoint to register discovered nodes with the cluster.

What agent-config.yaml provides

agent-config.yaml is specific to the Assisted Installer Add Hosts flow. It defines the hosts being added — their MAC addresses (used to match discovery reports to specific servers) and the rendezvous IP (the IP of one node that acts as the coordination point during discovery). The rendezvous node does not have a special role after installation — any node can be the rendezvous. Use bm-virt-01 (10.110.0.11) as the rendezvous.

09 — NMState — what it is and every directive explained

Before configuring static IPs and NIC bonding in Section 10, you must understand NMState completely. The Classic series standard applies: no configuration appears without a full explanation of every concept and every directive.

What NMState is — the fundamental concept

NMState is a declarative network configuration API for Linux. Instead of running a sequence of ip link, ip addr, and ip route commands — which are imperative ("do this now") and do not survive reboots unless separately persisted — you write a YAML file describing the desired state of the network. NMState reads the YAML and configures the network to match. If the system is rebooted, NMState re-applies the desired state automatically at boot.

For RHCOS specifically, NMState is the only supported mechanism for persistent network configuration. RHCOS's root filesystem is read-only — you cannot edit /etc/sysconfig/network-scripts files directly and have the changes persist across reboots. NMState configuration is applied via the Machine Config Operator, which stores it in etcd and re-applies it if the node is rebuilt. Every network configuration change on an RHCOS node is a NMState change.

The NMState structure — five layers in the file

A NMState configuration file for the DL385 Gen11 nodes has five distinct layers. Each layer configures a specific aspect of the networking stack. Understanding the layer model tells you exactly where to look when a specific network interface has a problem.

Layer 1 — Physical interfaces: The raw NIC ports (eth0, eth1, eth2, eth3). At this layer, NMState configures the physical NIC's speed, duplex, and whether it participates in a bond. No IP addresses are assigned at this layer.

Layer 2 — Bond: A logical interface (bond0) that aggregates multiple physical NICs into a single logical link. NMState configures the bond mode (802.3ad for LACP), LACP timing (fast or slow — fast sends PDUs every second instead of every 30 seconds, enabling faster failover detection), and which physical interfaces are bond members. No IP addresses here either — the bond is the Layer 2 aggregation. IPs go on Layer 3.

Layer 3 — VLAN sub-interfaces: Logical interfaces on top of the bond (bond0.110, bond0.120, bond0.130, bond0.140). Each sub-interface is associated with a VLAN ID. Frames sent on bond0.110 are tagged with VLAN ID 110. Frames received from the switch with VLAN ID 110 are delivered to bond0.110. This is 802.1Q VLAN tagging at the OS level.

Layer 4 — IP addressing: Static IP addresses are assigned to VLAN sub-interfaces. bond0.110 gets the node IP (10.110.0.11 for bm-virt-01). bond0.120 gets the ODF replication IP (10.120.0.11). bond0.130 and bond0.140 have no static IP — VM traffic uses them at Layer 2, and VMs get their own IPs.

Layer 5 — Routing: The default route (0.0.0.0/0 via 10.110.0.3) is assigned to bond0.110 — the node's primary interface. All traffic that is not destined for the local subnets exits via this gateway (the HSRP VIP on the leaf switches). A specific host route or route policy can be added for ODF replication traffic to ensure it always uses bond0.120 — not the default gateway.

Why static IPs must be set before installation — not after

After installation, each node reboots into RHCOS and the kubelet starts. The kubelet registers with the cluster API using the node's IP address. If the node comes up with a DHCP IP from the discovery pool (10.110.0.20–250), the kubelet registers with that temporary IP. After DHCP lease expiry, the node's IP changes — and the cluster still thinks the node is at the old IP. This causes CSR approval to fail and the node appears as NotReady. Static IP configuration before installation prevents this entirely.

13 — CSRs — what they are and how to approve them

After each node reboots into RHCOS and the kubelet starts, the node submits a Certificate Signing Request (CSR) to the cluster's API server. A CSR is the Kubernetes mechanism by which a new node proves its identity and requests a certificate that allows it to communicate securely with the cluster. Without an approved CSR, the node cannot join the cluster — it stays in NotReady state indefinitely.

Why CSR approval is a manual step

CSR approval is manual because it is a security gate. Any machine that can reach the cluster API can submit a CSR claiming to be any node name. If CSR approval were automatic, a compromised machine in the network could submit a CSR claiming to be master-0, get approved, and receive a certificate that grants it the permissions of a master node. Manual approval by a cluster administrator is the verification step that says "yes, I know about this node, I racked it, and I am approving it intentionally."

Two CSRs per node — why two

Each node actually submits two CSRs in sequence. The first CSR is the client CSR — it is submitted by the node bootstrap agent and requests a client certificate allowing the node to authenticate to the API server. The second CSR is the server CSR — submitted by the kubelet after the client CSR is approved, it requests a serving certificate allowing the API server to authenticate communications coming from the node. Both must be approved. If only the first is approved, the node appears in oc get nodes but stays NotReady because the serving certificate is missing.

15 — MachineSet — labelling and tainting bare-metal nodes

At this point, the five bare-metal nodes are Ready and accepting all workloads. This is not the desired production state. The nodes must be labelled and tainted so that VM workloads are isolated to bare-metal nodes and container workloads remain on ESXi-hosted workers. This section applies those labels and taints.

What labels do — and what taints do

Labels are metadata tags attached to nodes. Workloads can use node selectors or node affinity rules to prefer or require nodes with specific labels. Labels alone do not prevent workloads from landing on a node — they only attract workloads that actively select for those labels. A pod with no node selector can still land on a labelled node.

Taints are repellents. A taint on a node repels all pods that do not have a matching toleration. The taint kubevirt.io/virt=true:NoSchedule means "do not schedule any pod on this node unless it has a toleration for kubevirt.io/virt=true." The HyperConverged Operator (installed in VIRT-04) automatically adds this toleration to all virt-launcher pods (the pods that run VMs). Regular application pods — Deployments, StatefulSets, Jobs — do not have this toleration, so they never land on tainted bare-metal nodes. This achieves clean workload isolation without a separate cluster.

jumphost — label and taint all five bare-metal nodes
[admin@jumphost ~]$ for node in bm-virt-01 bm-virt-02 bm-virt-03 bm-virt-04 bm-virt-05; do
  # Label: identifies this node as a bare-metal worker in the virtualization pool
  # kubevirt.io/schedulable=true tells the virt-handler DaemonSet to run here
  oc label node ${node} \
    node-role.kubernetes.io/bm-worker="" \
    kubevirt.io/schedulable="true" \
    workload-type=virtualization \
    --overwrite

  # Taint: prevents regular container workloads from landing here
  # NoSchedule — pods without this toleration are not scheduled here
  # Does not evict pods already running — only affects new scheduling decisions
  oc adm taint node ${node} \
    kubevirt.io/virt=true:NoSchedule \
    --overwrite

  echo "${node}: labelled and tainted"
done
