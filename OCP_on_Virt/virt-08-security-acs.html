01 — What ACS sees on bare-metal VM nodes — and what it cannot

ACS was deployed in OCP-OPS-03 for NovaCrestBank's existing OCP workloads. Its Collector DaemonSet runs on every node, using eBPF probes to observe system calls from every container. In the VIRT environment, the bare-metal nodes run virt-launcher pods — each pod contains a running QEMU process that hosts a VM guest OS. ACS sees the virt-launcher pod as it sees any other pod. The key distinction is what is visible inside the virt-launcher versus what is happening inside the VM guest OS.

The three visibility layers for VM workloads

Layer 1 — The virt-launcher pod (fully visible to ACS): The virt-launcher pod is a standard Kubernetes pod running on the bare-metal node. ACS's Collector eBPF probe observes every system call from the virt-launcher process: the QEMU process that starts, the vhost-net threads that handle VM I/O, the libvirt management socket. ACS can detect anomalous behaviour in the virt-launcher itself — unexpected process executions, unexpected network connections made by the virt-launcher pod, unexpected file writes by QEMU. This is the same level of visibility ACS has for any container.

Layer 2 — Guest OS processes through QEMU (partially visible): The VM's guest OS runs as a QEMU subprocess. Guest OS I/O passes through the QEMU virtual device drivers. Network traffic that goes through the masquerade primary interface (VLAN 110, OVN overlay) is visible to ACS as network connections from the virt-launcher pod — ACS sees the source and destination IP but not the content. Process executions inside the guest OS are not visible to ACS — they are inside the QEMU virtualisation boundary, which the host's eBPF probes cannot cross.

Layer 3 — Guest OS internal processes (not visible to ACS): What the banking application running inside the VM does — file opens, internal TCP connections between services within the VM, process trees — is not visible to ACS via eBPF. The eBPF probe runs on the host kernel; it cannot observe system calls made inside the guest kernel. This is the fundamental visibility boundary for VM workloads: ACS sees the container (virt-launcher), not what is happening inside the VM.

The visibility gap — and what compensates for it

For containerised workloads, ACS provides complete visibility inside every running pod. For VM workloads, ACS's visibility stops at the virt-launcher boundary. What compensates for this gap? Three controls work together:

qemu-guest-agent: Installed in all NovaCrestBank VMs (confirmed in VIRT-04). The agent provides a bidirectional channel between the host and the guest OS. KubeVirt uses it for filesystem freeze/thaw (OADP backup). With additional configuration, it can also report guest process lists and guest file system events — supplementing ACS's host-level visibility.

Guest OS hardening (Sections 13–14): Rather than relying on external detection, the guest OS is hardened to reduce the attack surface — CIS Benchmark controls, minimal package sets, audit logging from within the guest. Guest OS audit logs are forwarded to Splunk from within the VM, providing the process-level visibility inside the VM that ACS cannot.

Network traffic analysis: ACS sees all network traffic leaving the VM via the masquerade interface (primary, OVN overlay) and can detect unexpected external connections even if the guest OS process that initiated them is not visible. PCI-DSS v4.0 Req 11.5.1

03 — Extending the Collector DaemonSet to bare-metal nodes

The ACS Collector DaemonSet deploys to all nodes by default — it does not require special configuration to reach bare-metal nodes. However, the bare-metal nodes carry the kubevirt.io/virt=true:NoSchedule taint. Without a toleration, the Collector DaemonSet cannot schedule on tainted nodes.

The kubevirt.io/virt taint — does the Collector tolerate it?

The ACS Collector DaemonSet in OCP-OPS-03 was configured for ESXi-hosted nodes that do not carry the kubevirt.io/virt taint. If the Collector was deployed before the bare-metal nodes were tainted (in VIRT-02), the existing DaemonSet may already be running on bare-metal nodes — the taint only blocks new scheduling; it does not evict pods that were already running when the taint was added. However, if the Collector pod on a bare-metal node is restarted for any reason (node reboot, pod crash, ACS upgrade), it will fail to reschedule without the toleration. Verify the toleration is present in the SecuredCluster CR and patch it if absent.

Patch SecuredCluster to add kubevirt toleration to Collector DaemonSet
# Check if the Collector DaemonSet already tolerates the kubevirt taint
# If not: patch the SecuredCluster CR to add the toleration
apiVersion: platform.stackrox.io/v1alpha1
kind: SecuredCluster
metadata:
  name: stackrox-secured-cluster-services
  namespace: stackrox
spec:
  clusterName: ncb-ocp-cluster
  centralEndpoint: central.stackrox.svc:443
  perNode:
    collector:
      collection: EBPF
            # EBPF: use eBPF probes for system call observation.
      # EBPF is the modern, kernel-module-free collection method.
      # Required for bare-metal RHCOS nodes — kernel modules are not supported on RHCOS.
      # CORE_BPF is available in ACS 4.3+ and has lower overhead than EBPF on newer kernels.
      # For OCP 4.16 with RHCOS kernel 5.14+, CORE_BPF is the preferred option — but
      # EBPF is fully supported and proven.
    tolerations:
      - key: kubevirt.io/virt
        operator: Exists
        effect: NoSchedule
                # The bare-metal nodes carry kubevirt.io/virt=true:NoSchedule.
        # This toleration allows the Collector DaemonSet pod to schedule on tainted nodes.
        # Without this toleration, the Collector pod is evicted from bare-metal nodes
        # on any restart — creating a silent eBPF visibility gap on VM-hosting nodes.
        # Effect: ACS would have no visibility into virt-launcher pods after any Collector restart.
      - key: cluster.ocs.openshift.io/openshift-storage
        operator: Exists
        effect: NoSchedule
                # The bare-metal nodes also carry the ODF storage label/taint.
        # Adding this toleration ensures Collector survives any future storage taint additions.

04 — What ACS sees inside virt-launcher pods

With the Collector running on bare-metal nodes, ACS builds a behavioural baseline for virt-launcher pods. Understanding what is normal for a virt-launcher pod — and what deviates from normal — is what makes the baseline meaningful and the policy alert thresholds correct.

Normal virt-launcher process tree

A healthy virt-launcher pod running a RHEL 9 banking application VM has a predictable process tree. ACS's Collector observes every process execution via eBPF and builds this as the baseline for the deployment:

ACS Central — normal process baseline for a virt-launcher pod
# In ACS Central UI: Workloads → Deployments → virt-workloads → vm-banking-app-01
# Process Discovery tab shows the observed process baseline:

PID 1    /usr/bin/virt-launcher-monitor    ← virt-launcher entrypoint
  PID 2  /usr/bin/virt-launcher            ← KubeVirt VM manager
    PID 3  /usr/libexec/qemu-kvm [args]   ← QEMU/KVM — the VM hypervisor process
              -machine q35                 ← x86-64 machine type
              -cpu host                    ← host-passthrough (EPYC 9354P)
              -m 32G                       ← 32 GiB guest memory
              -object memory-backend-file  ← hugepage-backed guest RAM
              -drive file=/dev/vda         ← VM boot disk (ODF RBD PVC)
    PID 4  /usr/sbin/virtqemud             ← libvirt daemon for QEMU management
    PID 5  /usr/bin/qemu-ga               ← qemu-guest-agent — filesystem quiesce

✓ This is the complete normal process tree. Every process is expected.
  ACS baseline: virt-launcher-monitor, virt-launcher, qemu-kvm, virtqemud, qemu-ga.
  Any process outside this set during the baseline period creates a deviation alert.
What would indicate virt-launcher compromise

Shell process spawned by QEMU: If the QEMU process spawns /bin/bash or /bin/sh, this indicates a VM escape exploit — an attacker has broken out of the guest OS and is executing code on the host. This is an extremely high-severity event. ACS detects it at the eBPF level immediately.

Unexpected network connection from virt-launcher: virt-launcher makes only known network connections — to the OVN overlay (for the masquerade interface) and to ODF storage (RBD PVC). An unexpected connection to an external IP from the virt-launcher process indicates host-level compromise following a VM escape.

File write outside known paths: QEMU writes only to the PVC block device and to its socket files in /var/run/kubevirt-private. Unexpected writes to /tmp, /usr/bin, or the node's filesystem indicate post-escape activity.

05 — VM-specific ACS policies — what to enforce and when

ACS ships with over 70 built-in security policies. Some apply directly to VM workloads. Others need to be scoped carefully — policies written for containerised applications may generate false positives against virt-launcher pods because virt-launcher behaves differently from a standard application container.

Privileged Container Directly relevant — virt-launcher pods require elevated capabilities (CAP_NET_ADMIN, KVM device access). 06 — virt-launcher runtime behaviour — building the baseline correctly

The most important step before enforcing any ACS policy on the virt-workloads namespace is the 14-day baseline observation period — and it must cover virt-launcher pods specifically. virt-launcher behaves differently from standard application containers. Without a correct baseline, legitimate virt-launcher activity generates false-positive alerts and enforcement causes VM operations to fail.

What happens during a live migration — and why the baseline must include it

During live migration, the virt-launcher pod on the destination node is created fresh. KubeVirt sets up the migration tunnel — the source virt-launcher pod opens a network connection to the destination virt-launcher pod on the migration port (port 49152 by default). This is an unexpected network connection from the source virt-launcher's perspective — it is connecting to a previously unseen IP (the destination virt-launcher pod IP) on a previously unseen port.

Without a baseline that includes live migration traffic, the ACS Network Baseline Violation policy fires every time a VM is live-migrated. In a Enforce mode, this would kill the source virt-launcher pod mid-migration — crashing the VM. Before switching Network Baseline Violation to Enforce mode, verify that at least one live migration has occurred during the observation period and that the migration network connection appears in the baseline.

08 — Admission controller for VirtualMachine CRs

ACS's admission controller intercepts pod creation requests and evaluates them against security policies before the pod runs. However, the ACS admission controller intercepts Pod admission — not VirtualMachine CR admission. When a VirtualMachine CR is applied to Kubernetes, the ACS admission controller does not see it — it is a custom resource. ACS sees the admission of the virt-launcher Pod that KubeVirt creates when the VM starts.

What the admission controller evaluates for virt-launcher pods

When a VM is started and KubeVirt creates the virt-launcher pod, the ACS admission controller evaluates the pod spec against all configured policies. At this point, ACS can block the virt-launcher pod from starting if it violates a policy — which effectively prevents the VM from starting. 09 — ACS network graph — observing VM traffic patterns

ACS's network graph shows all observed network connections enriched with Kubernetes metadata. For VM workloads, the network graph shows connections made by virt-launcher pods — which includes the VM's traffic that flows through the masquerade primary interface (OVN overlay, VLAN 110). The network graph does not show traffic on the VM's secondary VLAN 130/140 interface — that traffic bypasses the OVN overlay entirely (it is localnet traffic that goes directly to the physical fabric). Section 11 explains this scope limitation and what compensates for it.

10 — NetworkPolicy for virt-workloads namespace

NetworkPolicy objects control traffic on the OVN pod overlay network — the primary interface of virt-launcher pods. This covers cluster-internal VM-to-VM traffic through the overlay (not through the physical VLANs), traffic from monitoring components to VMs, and migration tunnel traffic. Apply the ACS-generated NetworkPolicy after confirming all legitimate connections are in the baseline.

networkpolicies-virt-workloads.yaml — generated from ACS baseline and committed to Git
# Policy 1: Default-deny all ingress to virt-workloads pods (virt-launcher pods)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress-virt
  namespace: virt-workloads
spec:
  podSelector: {}
    # Applies to all pods in the namespace — including all virt-launcher pods.
  # No ingress rules = deny all pod-network ingress by default.
  # Note: this does NOT block VLAN 130/140 traffic (see Section 11).
  policyTypes:
    - Ingress
---
# Policy 2: Allow Prometheus scrape from openshift-monitoring
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring-scrape
  namespace: virt-workloads
spec:
  podSelector:
    matchLabels:
      kubevirt.io: virt-launcher
  ingress:
    - from:
        - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: openshift-monitoring
      ports:
        - protocol: TCP
          port: 9100
---
# Policy 3: Allow live migration tunnel between virt-launcher pods in this namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-migration-tunnel
  namespace: virt-workloads
spec:
  podSelector:
    matchLabels:
      kubevirt.io: virt-launcher
  ingress:
    - from:
        - podSelector:
          matchLabels:
            kubevirt.io: virt-launcher
      ports:
        - protocol: TCP
          port: 49152
                    # Port 49152: the KubeVirt live migration tunnel port.
          # Source (migrating FROM) virt-launcher pod connects to destination pod on this port.
          # This NetworkPolicy allows virt-launcher pods to communicate only with
          # other virt-launcher pods on port 49152 — not with any other pods.

11 — VLAN 130/140 traffic — what NetworkPolicy cannot control

Kubernetes NetworkPolicy controls traffic on the OVN pod overlay network. It does not control traffic on the VM's secondary VLAN 130/140 interfaces — those interfaces connect directly to the physical fabric via OVN localnet, bypassing the OVN overlay entirely. A NetworkPolicy object in the virt-workloads namespace has no effect on traffic flowing in and out of the VM's VLAN 130 interface at 10.130.0.50.

This is not a gap — it is by design. VLAN 130/140 traffic is controlled by:

  • Model A (NSX present): NSX gateway firewall rules at the T0/T1 boundary control inter-zone traffic (VLAN 130 ↔ VLAN 70 for AD, VLAN 130 ↔ VLAN 140). NSX DFW controls intra-zone traffic where it has guest introspection visibility.
  • Model B (post-NSX, VIRT-11): OVN ACLs applied at the localnet port level — configured via OVN-Kubernetes annotations, not standard Kubernetes NetworkPolicy. VIRT-11 documents this in full.
  • Cisco leaf ACLs: VLAN-level access control lists on the Cisco Nexus leaf switches control which VLANs can reach which subnets — a coarse but effective first line of defence for VLAN 130/140 traffic.
  • Guest OS firewall: The VM's own firewalld/iptables rules control inbound connections on VLAN 130/140. These are configured during guest OS hardening (Sections 13–14).
ACS network graph — VLAN 130 traffic is not visible

The ACS network graph shows connections made by pod primary interfaces through the OVN overlay. Traffic from VM eth1 (VLAN 130, 10.130.0.50) to a remote system (10.70.0.10 DC01) does not appear in the ACS network graph because it never enters the OVN overlay — it flows directly from the Linux bridge (br-vlan130) through bond0.130 to the Cisco leaf, and from there to NSX T0 / DC01. ACS does not see this traffic. For VLAN 130/140 traffic monitoring, the data sources are NSX flow logs, Cisco NetFlow from the leaf switches, and Windows/Linux audit logs from the VMs themselves — all forwarded to Splunk.

13 — RHEL 9 guest OS hardening — CIS Benchmark Level 1

ACS provides security visibility for the virt-launcher pod (the host-level container). The VM guest OS requires its own hardening — applied during the initial boot via cloud-init, maintained via RHEL system profiles, and audited via guest OS audit logs forwarded to Splunk. The target is CIS Red Hat Enterprise Linux 9 Benchmark Level 1.

Key CIS Benchmark Level 1 controls applied via cloud-init

The cloud-init userData in the VirtualMachine CR (committed to Git as a Kubernetes Secret in VIRT-07) applies hardening on first boot. The key controls:

cloud-init hardening snippet — applied on first VM boot via cloudInitNoCloud secretRef
#cloud-config — RHEL 9 guest OS hardening (CIS Benchmark Level 1)
runcmd:
    # ── Minimal package set ─────────────────────────────────────────────
    - dnf remove -y telnet rsh-server ypbind tftp-server xinetd
    # Remove services not needed by the banking application.
  # Minimal attack surface — no legacy remote access protocols.

    # ── SSH hardening ───────────────────────────────────────────────────
    - sed -i 's/^#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config
    - sed -i 's/^#PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
    - sed -i 's/^#MaxAuthTries.*/MaxAuthTries 4/' /etc/ssh/sshd_config
    - sed -i 's/^#ClientAliveInterval.*/ClientAliveInterval 300/' /etc/ssh/sshd_config
    - sed -i 's/^#ClientAliveCountMax.*/ClientAliveCountMax 0/' /etc/ssh/sshd_config
    # CIS 5.2.10: PermitRootLogin no. CIS 5.2.11: PasswordAuthentication no.
  # SSH key-only access — the key is injected via cloud-init users.ssh_authorized_keys.
  # ClientAliveInterval + CountMax=0: SSH sessions idle for 5 minutes are terminated.

    # ── Audit logging ───────────────────────────────────────────────────
    - systemctl enable --now auditd
    - cat >> /etc/audit/rules.d/banking-vm.rules << 'EOF'
      -w /etc/passwd -p wa -k passwd-change
      -w /etc/sudoers -p wa -k sudoers-change
      -a exit,always -F arch=b64 -S execve -k exec-banking
      -a exit,always -F arch=b64 -S open,openat -F exit=-EACCES -k access-denied
      EOF
    # auditd captures: all writes to /etc/passwd and /etc/sudoers (privilege escalation attempts),
  # all process executions (execve — the guest-internal equivalent of ACS eBPF process monitoring),
  # all permission-denied file access attempts. Logs forwarded to Splunk via rsyslog.

    # ── Kernel hardening ────────────────────────────────────────────────
    - cat >> /etc/sysctl.d/99-cis-banking-vm.conf << 'EOF'
      net.ipv4.conf.all.accept_redirects = 0
      net.ipv4.conf.default.accept_redirects = 0
      net.ipv4.conf.all.log_martians = 1
      net.ipv4.conf.default.log_martians = 1
      net.ipv4.tcp_syncookies = 1
      kernel.randomize_va_space = 2
      EOF
    - sysctl -p /etc/sysctl.d/99-cis-banking-vm.conf
    # ASLR enabled (randomize_va_space: 2), ICMP redirect rejection, TCP SYN cookie protection.
  # Martian packet logging — packets with impossible source addresses logged to /var/log/messages.

    # ── Firewalld — allow only known ports on VLAN 130 interface ────────
    - systemctl enable --now firewalld
    - firewall-cmd --permanent --zone=public --add-interface=eth1
    - firewall-cmd --permanent --zone=public --remove-service=dhcpv6-client
    - firewall-cmd --permanent --zone=public --add-port=8443/tcp
    # Port 8443: banking application HTTPS API — the only inbound port on VLAN 130.
  # All other inbound ports on eth1 (VLAN 130) are blocked by firewalld.
  # This is the guest-internal firewall that controls VLAN 130 traffic
  # (which NetworkPolicy cannot control — Section 11).
    - firewall-cmd --reload

    # ── Forward audit logs to Splunk via rsyslog ────────────────────────
    - cat > /etc/rsyslog.d/99-splunk-hecfwd.conf << 'EOF'
      *.* @@splunk-hec.novacrestbank.local:514
      EOF
    - systemctl restart rsyslog
    # Forward all syslog and audit events to Splunk via syslog-over-TCP.
  # This is the guest-internal log forwarding that provides the inside-VM visibility
  # that ACS eBPF cannot provide (ACS sees the virt-launcher container, not guest processes).
