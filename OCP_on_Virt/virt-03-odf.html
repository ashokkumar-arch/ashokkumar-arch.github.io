01 — Why ODF on bare-metal — not vSAN, not another approach

Before installing anything, understand why ODF is the correct storage choice for the OpenShift Virtualization environment. This is not an automatic decision — it requires comparing the available options against the specific requirements of VM workloads on bare-metal nodes.

The storage requirement that vSAN cannot meet

vSAN — the storage system used by the existing VCF environment — provides excellent block storage for ESXi-hosted VMs. It is RAID-protected, performant, and well-integrated with vCenter. So why not use it for the bare-metal nodes too?

The answer is physical architecture. vSAN runs inside ESXi — it uses the VMDKs and vSphere storage policies to present PVCs to OCP pods and VMs. The five HP DL385 Gen11 bare-metal nodes do not run ESXi. They run RHCOS directly on metal. There is no vSphere CSI driver, no VMDK, no vCenter to manage storage. The vSphere storage stack simply does not exist on these nodes.

ODF is the Ceph-based storage system that runs natively inside OCP — as Kubernetes pods on the bare-metal RHCOS nodes. It uses the raw NVMe disks directly. It presents storage to VM workloads through Kubernetes StorageClasses and PersistentVolumeClaims — the same API that all OCP workloads already use. No vCenter, no ESXi, no VMDK.

What ODF provides specifically for OpenShift Virtualization

Two storage capabilities are specifically required for VM workloads on OCP Virt — and both are provided by ODF but not by vSAN on bare-metal:

ReadWriteMany (RWX) for live migration: When a running VM is live-migrated from one bare-metal node to another, both the source and destination node must simultaneously access the VM's boot disk for a brief period during the migration. This requires a RWX-capable volume. ODF's CephFS provides RWX. Without it, live migration fails at the storage handoff step and the VM must be cold-migrated (stopped, moved, restarted) — not acceptable for banking workloads with availability SLOs.

The ocs-storagecluster-ceph-rbd-virtualization StorageClass: ODF ships a StorageClass specifically tuned for VM boot disks. It configures optimal block sizes, pre-allocation behaviour, and image features (layering, deep-flatten, exclusive-lock, object-map) that match the I/O pattern of a running VM guest OS. Using the generic ceph-rbd StorageClass for VM boot disks works but does not apply these optimisations.

02 — What ODF is — Ceph running as an OCP-native operator

OpenShift Data Foundation is Red Hat's enterprise storage platform for OCP. Under the hood it is Ceph — the open-source distributed storage system — packaged, operated, and supported by Red Hat as an OCP Operator. Before installing anything, understand what Ceph is, because every configuration decision in this guide follows from Ceph's architecture.

What Ceph is

Ceph is a software-defined storage platform that pools raw disks across multiple nodes and presents them as a unified storage cluster. Data is distributed and replicated across disks and nodes — no single disk failure and no single node failure causes data loss. Ceph has no single point of failure: its metadata, monitor quorum, and data placement are all distributed. NovaCrestBank's ODF cluster pools the raw NVMe disks across all five bare-metal nodes and presents storage through three OCP-facing APIs simultaneously: block (for VM boot disks), file (for shared workloads), and object (S3-compatible, for backups).

How ODF wraps Ceph

The ODF Operator translates Ceph administration into Kubernetes reconciliation. A StorageCluster CR declares the desired Ceph cluster configuration — which nodes, which disks, how many replicas. The ODF Operator reads this CR and deploys all Ceph daemons as OCP pods. When a daemon pod crashes, the Operator reschedules it. When a new disk is added to a node, the Operator provisions a new OSD pod for it. Ceph operations that previously required the ceph CLI are now observable and manageable through oc get and oc describe.

Why ODF over standalone Ceph

You could deploy Ceph directly on the bare-metal nodes without ODF, using the Rook operator directly. But ODF adds: Red Hat support, OLM-managed lifecycle, the NooBaa MCG layer for object storage, OADP integration for application-consistent backups, and the Ceph Dashboard integrated into the OCP console. For a production banking environment, the support lifecycle and integration with the rest of the OCP platform make ODF the correct choice over standalone Rook-Ceph.

03 — Ceph components — every daemon explained before we deploy them

ODF deploys six distinct types of Ceph daemon, each running as one or more OCP pods on the bare-metal nodes. Understanding what each daemon does is essential — because when ODF is unhealthy, the error is always in a specific daemon, and knowing its role tells you exactly what has failed and what it means for storage availability.

MON
Monitor
Maintains the cluster map — the authoritative record of which OSDs exist, which are up, and where every piece of data lives. Runs in quorum: a majority vote is required for any cluster state change. Losing 2 of 3 MONs loses quorum — the cluster stops accepting writes (goes read-only) to prevent data corruption. MONs must be distributed across different physical nodes — two MONs on the same node means a single node failure can break quorum.
NovaCrestBank: 3 MON pods — one on each of bm-virt-01, 02, 03. Quorum requires 2 of 3.
OSD
Object Storage Daemon
One OSD per disk. Stores the actual data. When data is written to one OSD (the primary), it replicates to two replica OSDs on two different nodes (with replication factor 3). Handles recovery when a peer OSD fails — detects the failure, contacts MON, and begins re-replicating the affected data to healthy OSDs. OSDs use BlueStore as their internal storage format: direct access to raw block device, no filesystem overhead, write-ahead log for durability.
NovaCrestBank: 20 OSD pods — 4 per node × 5 nodes. Each OSD manages one 3.84 TB NVMe disk.
MGR
Manager
Runs cluster management modules: the Ceph Dashboard (web UI accessible from OCP console), Prometheus metrics export (the source of all ODF metrics that appear in OBS-01 dashboards), the balancer module (periodically rebalances data across OSDs to equalise disk usage), and the orchestrator (coordinates OSD provisioning with the ODF Operator). One active + one standby for HA. If the active MGR pod crashes, the standby takes over — cluster operations continue.
NovaCrestBank: 2 MGR pods — active on one node, standby on another.
MDS
Metadata Server
Required only for CephFS (the shared filesystem). Manages filesystem metadata — directory trees, file attributes, inode-to-object mapping. Applications do not interact with MDS directly; the CephFS kernel client or FUSE client talks to MDS transparently. One active + one standby per filesystem. If the active MDS fails, the standby takes over within seconds. Not needed if only block (CephRBD) storage is used — but ODF deploys MDS by default when CephFS is enabled.
NovaCrestBank: 2 MDS pods for the default CephFS filesystem. Required for RWX PVCs used by VM live migration.
RGW
RADOS Gateway
An HTTP gateway in front of the Ceph RADOS layer that exposes an S3-compatible and Swift-compatible object storage API. RGW is Ceph's native object storage — data stored through RGW is protected by Ceph's replication just like block and file data. Used when you need S3-compatible object storage with Ceph-managed redundancy. Alternative to MCG (NooBaa) for simpler S3 use cases without multi-cloud federation needs.
NovaCrestBank: RGW deployed for OADP backup buckets. MCG (NooBaa) used for ObjectBucketClaims.
MCG
Multi-Cloud Gateway
NooBaa-based S3 gateway that abstracts multiple backing stores (Ceph RGW, AWS S3, Azure Blob) behind a unified S3 API. Provides namespace buckets, data federation across cloud and on-premises, and lifecycle policies. Used when you need ObjectBucketClaims (dynamically provisioned S3 buckets via the Kubernetes API) or multi-cloud storage spanning on-premises and cloud providers. MCG is what OADP uses for its backup store in the NovaCrestBank configuration.
NovaCrestBank: MCG deployed — backs OADP backup ObjectBucketClaims. NooBaa database on ceph-rbd PVC.
The quorum rule — why 3 nodes is the minimum for any converged ODF deployment

Ceph MON quorum requires a majority — more than half the MON daemons must be reachable for the cluster to accept writes. With 3 MONs (one per node), losing 1 node leaves 2 MONs running — quorum is maintained. With 2 MONs, losing 1 leaves 1 MON — quorum is broken and the cluster stops accepting writes. This is why ODF requires a minimum of 3 nodes. NovaCrestBank uses 5 nodes — losing any 2 nodes simultaneously still leaves 3 MONs in quorum. NIST SP 800-53 CP-7

04 — Three storage types — block, file, and object

ODF provides all three fundamental storage types simultaneously from the same Ceph cluster. The type you choose for each workload depends on its access pattern — not its size or importance. The wrong type causes either a functional failure (pod stuck in Pending because PVC cannot be mounted) or a performance problem.

TypeCeph technologyAccess modeWhen to useNovaCrestBank use cases
Block (RBD) Ceph RADOS Block Device ReadWriteOnce One pod or VM reads/writes a device-like volume. Databases, application state. Highest performance — direct block I/O via kernel RBD driver. VM boot disks (via ceph-rbd-virtualization StorageClass), PostgreSQL data, Prometheus TSDB
File (CephFS) Ceph Filesystem via MDS ReadWriteMany Multiple pods or VMs must simultaneously mount the same volume. Shared workspaces, live migration staging. Lower performance than RBD — metadata overhead of a filesystem. VM live migration staging disk (source and destination node both mount the volume during migration). Required by OpenShift Virtualization live migration.
Object (S3) Ceph RGW or MCG (NooBaa) ObjectBucketClaim
(not a PVC)		 Unstructured data accessed via HTTP S3 API. Not mounted as a filesystem. Logs, backups, images, large binary data. OADP backup store, OpenShift image registry backing store (migration from MinIO)
The most common ODF mistake for VM workloads

Using ReadWriteOnce (RBD block) for a VM's boot disk when that VM will be live-migrated. During live migration, OCP Virt needs both the source and destination node to access the disk simultaneously — which requires ReadWriteMany. The correct StorageClass is ocs-storagecluster-ceph-rbd-virtualization, which is configured for ReadWriteMany. VMs created with ReadWriteOnce PVCs can only be cold-migrated (stopped first). Section 18 covers this StorageClass in full.

05 — Converged topology — why all 5 nodes run ODF

NovaCrestBank's ODF deployment uses a converged topology: all five DL385 Gen11 nodes run both VM workloads (via OpenShift Virtualization) and ODF storage daemons (OSD, MON, MGR, MDS). This is different from the VCF series ODF deployment (STORAGE-01) which uses dedicated storage nodes.

Converged vs dedicated storage nodes for the VIRT environment

Rejected: dedicated storage nodes. Adding 3 dedicated storage nodes to the bare-metal environment would require 3 additional HP DL385 Gen11 servers, additional switch ports, iLO cabling, and OS provisioning. The bare-metal environment already has 5 nodes — adding 3 more for storage-only work doubles the hardware scope and cost. For a platform consolidation project whose business case includes TCO reduction, this is difficult to justify.

Decision: converged topology, 5 nodes. All 5 DL385 nodes contribute 4 NVMe disks each to ODF (20 OSDs total) while also running VM workloads. ODF resource limits (CPU and memory requests/limits on OSD and MON pods) ensure storage daemons do not starve VM workloads. With 64 CPU cores and 512 GiB RAM per node, each node has substantial headroom for both OSD daemons (~4 GiB RAM per OSD × 4 = 16 GiB) and VM workloads (remaining ~490 GiB). The 5-node converged topology tolerates 2 simultaneous node failures — stronger resilience than a 3-node dedicated configuration.

06 — Disk verification — raw NVMe disks confirmed on all nodes

Before installing the Local Storage Operator, verify that the raw NVMe disks are visible as unformatted block devices on every bare-metal node. The LSO discovers and claims raw block devices — it cannot use a disk that already has a filesystem, a partition table, or an existing Ceph OSD on it. This verification runs from the jump host using oc debug to open a shell on each node.

If an NVMe disk shows a FSTYPE — root causes and fixes
Disk shows FSTYPE=xfs or ext4: A previous OSD or filesystem was created on this disk. The LSO will not claim a disk with an existing filesystem. Wipe the disk: oc debug node/<node> -- chroot /host wipefs -a /dev/nvme0n1. Verify FSTYPE is empty before proceeding.
Disk shows FSTYPE=ceph_bluestore: A previous Ceph OSD exists on this disk from an earlier ODF installation attempt. Run ceph-volume lvm zap /dev/nvme0n1 from inside a debug pod with ceph tools available, or wipefs -a /dev/nvme0n1 followed by dd if=/dev/zero of=/dev/nvme0n1 bs=4M count=10 to clear the BlueStore signature.
NVMe disk count is fewer than 4: The HP SmartArray controller may have claimed a disk for RAID. Verify iLO → System Information → Storage → NVMe that all 4 NVMe drives are visible as individual devices, not as a RAID set. NVMe drives for ODF must be in passthrough/JBOD mode, not RAID.

07 — Node labelling and tainting for ODF

ODF uses a specific label to identify which nodes should run storage daemons (OSDs, MONs, MGRs). Without this label, the LSO and ODF Operator do not know which nodes to target. The taint ensures that non-ODF pods do not land on storage nodes and compete with OSD I/O — but in a converged topology, we do not use the full storage taint. Here is why.

Why converged nodes are NOT fully tainted for ODF

In a dedicated storage node deployment (like the VCF series STORAGE-01), the taint node.ocs.openshift.io/storage=true:NoSchedule is applied to storage nodes to prevent application pods from landing on them. This is correct when the nodes run only ODF — they should not run anything else.

In NovaCrestBank's converged topology, the same nodes run both ODF daemons and VM workloads (via OCP Virt virt-launcher pods). Applying the ODF storage taint would prevent virt-launcher pods from scheduling here — unless those pods also tolerate the storage taint, which requires a change to the HyperConverged Operator configuration that adds complexity.

The converged nodes already carry the kubevirt.io/virt=true:NoSchedule taint from VIRT-02. ODF pods must tolerate this taint to schedule on these nodes. This is configured in the StorageCluster CR's placement section (Section 14). The ODF label is still required — it is what tells LSO and ODF which nodes have the OSD disks.

jumphost — label nodes for ODF
# Add the ODF node selector label to all 5 bare-metal nodes

08 — What the Local Storage Operator is — and why ODF needs it

ODF cannot use raw disk devices directly off the node. It needs PersistentVolumes (PVs) — the Kubernetes abstraction for storage that has been provisioned and is available to be claimed. Raw block devices on RHCOS nodes are not automatically PVs. The Local Storage Operator (LSO) bridges this gap: it discovers raw block devices on labelled nodes and creates PVs from them. Without LSO, ODF has no storage to claim.

The three LSO objects

LocalVolumeDiscovery: A DaemonSet-backed object that runs a discovery agent on each labelled storage node. The agent scans the node's block devices and reports what it finds — device name, size, vendor, model, filesystem presence, and whether the device is currently in use. Results appear as LocalVolumeDiscoveryResult objects. This is read-only — discovery does not change anything on the disk.

LocalVolumeSet: Tells LSO which discovered devices to claim and what StorageClass to create PVs for. Uses selectors to filter devices — you specify that you want NonRotational disks larger than 100 GiB with no filesystem. LSO claims matching devices and creates one PV per device. Once a device is claimed, it will not be claimed again — LSO tracks ownership.

LocalVolume (not used here): An older LSO object that claims specific named devices (e.g., /dev/nvme0n1). Requires manually specifying each device path on each node. LocalVolumeSet is the correct modern approach — it uses selectors and scales automatically as new nodes and disks are added.

Why volumeMode: Block and not Filesystem

LSO creates PVs in volumeMode: Block for ODF — not Filesystem. Ceph OSD daemons use BlueStore as their internal storage format. BlueStore requires raw block device access — it manages its own internal structures directly on the block device. If the PV were in Filesystem mode, the OS would place a filesystem on the device first, and Ceph's BlueStore would be written on top of that filesystem — adding an unnecessary translation layer and reducing performance. volumeMode: Block gives OSD pods direct access to the raw NVMe block device, which is what BlueStore requires.

09 — Install the Local Storage Operator

lso-subscription.yaml — Local Storage Operator Subscription
# Create the openshift-local-storage namespace first
apiVersion: v1
kind: Namespace
metadata:
  name: openshift-local-storage
  # The dedicated namespace for all LSO objects and pods.
  # LSO discovery agents run as DaemonSet pods here.
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: local-operator-group
  namespace: openshift-local-storage
spec:
  targetNamespaces:
    # targetNamespaces must match the namespace — LSO is a single-namespace operator.
    # Omitting this causes OLM to create an all-namespaces OperatorGroup which conflicts
    # with LSO's expected scope and produces an InstallPlan error.
    - openshift-local-storage
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: local-storage-operator
  namespace: openshift-local-storage
spec:
  channel: stable
    # channel: stable — LSO uses a single stable channel, unlike ODF which uses
  # version-pinned channels (stable-4.16). LSO's stable channel always tracks
  # the version compatible with the installed OCP version.
  name: local-storage-operator
  source: redhat-operators
    # redhat-operators: LSO is a Red Hat certified operator — not community-operators.
  sourceNamespace: openshift-marketplace
  installPlanApproval: Manual
    # Manual: require explicit approval before installing. This prevents automatic
  # upgrades from changing the LSO version without a change management review.
  # In a banking environment, every operator upgrade is a change.

10 — LocalVolumeDiscovery — run discovery on all 5 nodes

The LocalVolumeDiscovery object deploys a DaemonSet that runs a discovery agent on each node matching the node selector. The agent scans block devices and reports results as LocalVolumeDiscoveryResult objects — one per node. Discovery is read-only. It does not claim or modify any disk. It is the reconnaissance step that tells LSO what is available before anything is claimed.

lso-discovery.yaml — LocalVolumeDiscovery
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeDiscovery
metadata:
  name: auto-discover-devices
  namespace: openshift-local-storage
spec:
  nodeSelector:
    nodeSelectorTerms:
      - matchExpressions:
          - key: cluster.ocs.openshift.io/openshift-storage
              operator: Exists
              # Exists operator: any node that has this label key (regardless of value)
          # is included. We labelled all 5 bare-metal nodes with this key in Section 07.
          # This discovery runs only on bare-metal nodes — not on the ESXi-hosted
          # OCP workers from the VCF environment, which do not have this label.

11 — LocalVolumeSet — claim NVMe disks as PersistentVolumes

The LocalVolumeSet is where LSO actually claims the discovered disks and creates PersistentVolumes from them. Every field has a specific purpose — the wrong value causes either the wrong disks to be claimed or the correct disks to be missed.

lso-volumeset.yaml — LocalVolumeSet — every field explained
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeSet
metadata:
  name: local-block
    # local-block: the conventional name for the ODF LSO LocalVolumeSet.
  # ODF documentation and the StorageCluster CR reference this name implicitly
  # through the StorageClass it creates (localblock).
  namespace: openshift-local-storage
spec:
  nodeSelector:
    nodeSelectorTerms:
      - matchExpressions:
          - key: cluster.ocs.openshift.io/openshift-storage
              operator: Exists

  storageClassName: localblock
    # localblock: the StorageClass LSO creates for the PVs it provisions.
  # The ODF StorageCluster CR references this StorageClass to claim PVs for OSDs.
  # Do not change this name — the StorageCluster CR expects "localblock" by convention.
  volumeMode: Block
    # Block mode (not Filesystem): OSD devices must be raw block devices.
  # Ceph BlueStore manages its own internal format directly on the raw block device.
  # Filesystem mode would add a kernel filesystem between BlueStore and the disk —
  # unnecessary overhead, reduced performance, and not supported for ODF OSDs.
  deviceInclusionSpec:
    deviceTypes:
      - disk
        # disk only: include physical disk devices. Exclude:
      # "part" (partitions — would incorrectly claim individual partitions),
      # "loop" (loop devices used for container images),
      # "rom" (optical drives).
      # The OS disks (sda/sdb) are type "disk" but will be excluded by the
      # filesystem check — sda has FSTYPE=vfat (RHCOS boot partition).
    minSize: 1Ti
      # 1 TiB minimum size: the NVMe OSD disks are 3.84 TB — well above this threshold.
    # The OS SSDs (480 GiB) are below 1 TiB and would be excluded here even if they
    # had no filesystem. Double protection: size filter AND filesystem check both
    # prevent OS disks from being claimed as OSD devices.
    deviceMechanicalProperties:
      - NonRotational
        # NonRotational: NVMe and SATA SSD only. Excludes spinning HDDs (Rotational).
      # ODF performance on HDDs is significantly worse than on SSDs — write amplification,
      # OSD journal I/O, and recovery rebalancing all perform poorly on spinning media.
      # Explicitly requiring NonRotational prevents accidental claiming of an HDD
      # if one is ever installed on these nodes.
    fsType: ""

    # A device with an existing filesystem (vfat, xfs, ext4) would be claimed
    # only if fsType is not specified. Setting it to "" explicitly excludes
    # all formatted devices — another layer of protection against claiming the OS disk.

15 — Topology awareness and failure domains

Ceph's replication model distributes data across failure domains to ensure that no single hardware failure causes data loss. In NovaCrestBank's 5-node converged deployment, each node is its own failure domain. Understanding how Ceph uses topology is essential — it determines what failures you can survive and what hardware failure sequence would cause a cluster outage.

What a failure domain is

A failure domain is a unit of hardware failure that is treated as correlated — if one component in the domain fails, all components in the same domain may fail together. In a bare-metal server deployment with no rack-level redundancy configuration, each physical server is a separate failure domain. If bm-virt-01 loses power, all 4 of its OSD disks go offline simultaneously — but the disks on bm-virt-02 through bm-virt-05 are unaffected.

Ceph's CRUSH map — how topology is expressed

Ceph uses a data structure called the CRUSH map (Controlled Replication Under Scalable Hashing) to model the physical topology of the storage cluster. The CRUSH map organises OSDs into a tree of containers: OSDs → hosts → racks → data centres → regions. ODF builds this tree automatically from the Kubernetes node labels during StorageCluster deployment — each bare-metal node becomes a "host" bucket in the CRUSH tree.

When Ceph writes 3 replicas of a data chunk, the CRUSH algorithm places each replica in a different "host" bucket — meaning on a different physical node. This guarantees that losing one entire bare-metal node (and all its 4 OSD disks) still leaves 2 replicas intact on other nodes.

The 5-node advantage over a 3-node minimum

A 3-node ODF cluster meets the minimum requirement (3 MONs, 3 failure domains for replica: 3) but has no tolerance for simultaneous multi-node failures. NovaCrestBank's 5-node cluster can lose any 2 nodes simultaneously and remain fully operational. For a banking environment where storage availability directly impacts payment processing, this additional resilience justifies the larger hardware footprint. NIST SP 800-53 CP-7

18 — ceph-rbd-virtualization — the VM StorageClass

The ocs-storagecluster-ceph-rbd-virtualization StorageClass is the most important StorageClass for the VIRT series. It is used for every VM boot disk in VIRT-04 and every DataVolume in VIRT-06 (MTV migration). Understanding why it exists as a separate class from ceph-rbd is essential.

What makes it different from ceph-rbd

The ceph-rbd-virtualization StorageClass has three differences from the generic ceph-rbd class:

Access mode: ReadWriteMany. The generic ceph-rbd StorageClass creates PVCs with ReadWriteOnce access mode — only one node can mount the PVC at a time. The ceph-rbd-virtualization class uses ReadWriteMany — multiple nodes can mount the PVC simultaneously. This is required for live migration: during migration, the source node and destination node both need access to the VM's boot disk for a brief window. Without RWX access, live migration fails.

RBD image features: VM-optimised set. Ceph RBD volumes have configurable features (layering, deep-flatten, exclusive-lock, object-map, fast-diff). The VM StorageClass enables the specific combination of features that OpenShift Virtualization requires for snapshots, cloning (DataVolume clones), and correct locking behaviour when two nodes access the same volume during live migration.

Disk pre-allocation: thick provisioning. The VM StorageClass pre-allocates the full disk capacity on creation. This is different from thin provisioning (the default for ceph-rbd) where space is allocated as data is written. For VM boot disks, pre-allocation ensures the VM never encounters a "no space" error mid-operation because the allocated space was already full on the Ceph cluster.
