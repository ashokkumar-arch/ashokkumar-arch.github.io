01 — What MTV is — the Migration Toolkit for Virtualization explained

MTV (Migration Toolkit for Virtualization) is a Red Hat operator that automates the migration of virtual machines from hypervisor platforms — VMware vSphere, Red Hat Virtualization, or OpenStack — to OpenShift Virtualization. It handles everything that would otherwise be a manual sequence: connecting to the source hypervisor, converting the disk format, transferring disk data, configuring the VM network and storage in the destination, and starting the migrated VM.

What MTV does that manual migration cannot scale

Migrating a VM manually from vSphere involves: exporting the VMDK from vCenter, converting it to QCOW2 with qemu-img, uploading to ODF using virtctl image-upload, creating a VirtualMachine CR by hand, mapping the VMware portgroup to the correct OVN localnet NAD, and verifying the result. For one VM: 2–4 hours. For fifty VMs: weeks — with inconsistency at every manual step. MTV automates all of this through a declarative API. A MigrationPlan specifies which VMs to migrate, how networks map from VMware portgroups to OCP NADs, and how storage maps from vSAN to ODF StorageClasses. MTV executes the plan, tracks progress per VM, retries on transient failures, and produces a migration report.

What stays in VCF after VIRT-06 — the two-platform transition period

VIRT-06 migrates banking application workload VMs from the VCF environment to OpenShift Virtualization bare-metal nodes. The VCF environment continues to run during this period — it still hosts the management VMs (SDDC Manager, vCenter, NSX Manager) and any workload VMs not yet migrated. VCF is not decommissioned in VIRT-06. Both platforms operate simultaneously until NSX decommission in VIRT-11. VIRT-06 migrates workload VMs only — not management infrastructure.

02 — How MTV moves a VM — the disk-level mechanism

Understanding what happens at the disk and network level during migration allows you to set realistic expectations, diagnose failures, and explain the process to application teams. MTV performs a specific sequence of operations — each has a measurable duration and a failure mode.

Cold migration — the complete six-phase sequence

Phase 1
Source VM power-off
MTV signals vCenter to gracefully shut down the VM via VMware Tools or forcefully powers it off. The VM's disks become quiesced — no in-flight writes. Data consistency is guaranteed from this moment. The VM is unavailable from here until it starts on OCP Virt.
Phase 2
Disk export via VDDK
MTV connects to VMware VDDK (Virtual Disk Development Kit) on the ESXi host and reads the VMDK block by block. VDDK provides a direct data path to the underlying storage — not through the guest OS or the production fabric. Transfer happens over the VMware management network (VLAN 10 in NovaCrestBank). Without VDDK, MTV falls back to HTTP streaming — approximately 10× slower.
Phase 3
Format conversion — VMDK to raw
MTV converts the VMDK format on the fly during transfer. The conversion happens in the MTV importer pod's memory — data is never written to disk in VMDK format on the OCP side. Output is raw block data written directly into a new ODF PVC.
Phase 4
PVC creation in ODF
MTV creates a PersistentVolumeClaim backed by ocs-storagecluster-ceph-rbd-virtualization for each VM disk. PVC size matches the VM disk. Thick provisioning means full capacity is reserved in Ceph immediately — no thin provisioning growth events later.
Phase 5
VirtualMachine CR creation
MTV creates a VirtualMachine CR in the destination namespace. CPU count, memory size, and firmware type (BIOS or UEFI) are read from the source VM's vSphere configuration. Network interfaces are mapped from VMware portgroups to OCP NADs using the NetworkMap. The migrated PVC is attached as the boot disk.
Phase 6
VM starts on OCP Virt
MTV starts the VirtualMachine. The guest OS boots from the migrated disk. For RHEL VMs, virtio drivers are pre-installed — clean boot. For Windows VMs without virtio drivers pre-installed before migration, the first boot triggers hardware detection and the VirtIO driver ISO may need to be attached to complete the boot cleanly.

Transfer rate and duration estimation

MTV transfers disk data from ESXi (via VDDK) to the OCP cluster. Effective rate depends on the ESXi host's management NIC bandwidth (1G on NovaCrestBank's VCF VLAN 10) and ODF write throughput. Expect 100–150 MiB/s per VM.

VDDK — the VMware proprietary library MTV requires

MTV uses VMware's VDDK for efficient disk access. VDDK must be downloaded separately from VMware Customer Connect, packaged as a container image, and pushed to Harbor before any migration. Without VDDK, MTV falls back to HTTP streaming (~10× slower — not suitable for production migrations). Section 05 covers VDDK image preparation as the mandatory first step of MTV installation.

Cold — wave 1: vm-win-test-01, vm-rhel-test-01, vm-preprod-01. These VMs have maintenance windows and no production traffic SLO.

Warm — wave 2: vm-banking-app-01 (payment processing), vm-banking-app-02 (core banking API), vm-api-gateway-01 (partner gateway). Strict uptime SLOs — warm migration limits downtime to 3–5 minutes during cutover.

04 — MTV components — every object type explained

MTV extends Kubernetes with six CRDs. A mistake in a Provider or StorageMap propagates to every MigrationPlan that references it — understand each object before creating any plan.

Provider
Source and Destination
Declares a connection to a virtualisation platform. A vsphere Provider connects to vCenter. An openshift Provider is the destination local OCP cluster. MTV discovers the source inventory (VMs, networks, datastores) through the Provider connection and makes it available for mapping in plans.
NetworkMap
VMware portgroup to OCP NAD
Declares how VMware distributed portgroups map to OCP NetworkAttachmentDefinitions. A VM connected to seg-banking-app in VMware is connected to vlan130-primary in OCP after migration. Without a NetworkMap, migrated VMs have no secondary network interfaces and cannot be reached by their production IPs.
StorageMap
VMware datastore to OCP StorageClass
Declares how VMware datastores map to OCP StorageClasses. All VMs on vSAN map to ocs-storagecluster-ceph-rbd-virtualization. Without a StorageMap, MTV cannot create PVCs for migrated VM disks and the migration fails at Phase 4.
Plan
The migration specification
Declares which VMs to migrate, referencing the Provider, NetworkMap, and StorageMap. Specifies cold or warm migration. A single Plan can migrate multiple VMs in parallel. The Plan is validated by MTV before any Migration is created — validation catches missing mappings and unreachable VMs before disk transfer begins.
Migration
One execution of a Plan
A Migration is created to execute a Plan. Tracks real-time progress per VM — current phase, bytes transferred, error details. Multiple Migrations can reference the same Plan for retries (if a transfer fails) or for phased execution (migrate some VMs now, others in the next wave).
Hook
Pre/post migration automation
An Ansible playbook or shell script that MTV runs before or after a VM migration. Used for pre-migration quiescing (stop the application gracefully before disk freeze), post-migration smoke tests (verify the application responds on its new IP), and monitoring system updates (register the new IP with the observability stack).

06 — Source provider — vCenter (NovaCrestBank VCF environment)

A Provider CR declares a connection to a virtualisation platform. MTV uses this connection to discover VM inventory, read VMDK data via VDDK, and control VM power state during migration. Create the credentials Secret first — the Provider references it, not the credentials directly.

provider-vcenter.yaml — Source Provider — every field annotated
apiVersion: forklift.konveyor.io/v1beta1
kind: Provider
metadata:
  name: vcenter-ocp
  namespace: openshift-mtv
spec:
  type: vsphere
    # vsphere: connects to a VMware vCenter instance.
  # Other valid types: ovirt (Red Hat Virtualization), openstack, ova, openshift.
  url: https://vcenter-ocp.novacrestbank.local/sdk
    # vcenter-ocp: the OCP workload vCenter (hosts the ESXi cluster with VMs to migrate).
  # /sdk is the standard vSphere SDK API path. Must use HTTPS.
  # Not vcenter-mgmt — the management vCenter hosts only VCF management VMs.
  secret:
    name: vcenter-ocp-creds
    namespace: openshift-mtv
        # Secret containing: user=svc-mtv-vcenter@novacrestbank.local, password=..., insecureSkipVerify=false.
    # Account svc-mtv-vcenter requires: VM read, power-off (cold), snapshot create/delete (warm CBT),
    # VDDK data access. Retrieved from CyberArk via ESO — same pattern as all service accounts.
  settings:
    vddkInitImage: harbor.apps.ocp.novacrestbank.local/mtv/vddk:8.0.3
        # Specifies VDDK image per-Provider. Allows different VDDK versions for different
    # vCenter versions if needed. In NovaCrestBank, one vCenter, one VDDK version.

07 — Destination provider — OpenShift Virtualization

The destination Provider declares the local OCP cluster as a migration target. Unlike the source Provider (which requires credentials and a URL), the destination Provider for the local cluster uses the in-cluster service account — MTV already has access to the cluster it runs on.

provider-ocp.yaml — Destination Provider (local OCP cluster)
apiVersion: forklift.konveyor.io/v1beta1
kind: Provider
metadata:
  name: host
    # "host": the conventional name for the local cluster destination Provider.
  # MTV looks for this name as the default destination when MigrationPlans omit
  # the destinationProvider field. Always use "host" for the local cluster.
  namespace: openshift-mtv
spec:
  type: openshift
  url: ""
    # Empty URL = use the in-cluster Kubernetes API server.
  # MTV's forklift-controller ServiceAccount has RBAC to create VirtualMachine,
  # PVC, and DataVolume objects in any namespace — the destination namespace
  # is specified in the MigrationPlan's targetNamespace field.

08 — NetworkMap — VMware portgroups to OVN localnet NADs

The NetworkMap is the translation table between VMware distributed portgroup IDs and OCP NetworkAttachmentDefinition names. Use the internal vSphere portgroup IDs — not the display names, which can be renamed without notice. Retrieve IDs from the MTV inventory API or the MTV UI under Providers → vcenter-ocp → Networks.

networkmap-ncb.yaml — NetworkMap — every field annotated
apiVersion: forklift.konveyor.io/v1beta1
kind: NetworkMap
metadata:
  name: ncb-network-map
  namespace: openshift-mtv
spec:
  map:
        # Each entry maps one VMware portgroup to one OCP network.
    - source:
        id: network-1001
                # Internal vSphere ID of DPortGroup-seg-banking-app.
        # Stable across vCenter renames — always use ID, not display name.
      destination:
        type: multus
                # multus: connect via a secondary NAD (not the pod overlay).
        # Use "pod" only for VMs that need cluster-internal access but no routable VLAN IP.
        name: vlan130-primary
        namespace: virt-workloads
                # The NAD created in VIRT-05 Section 07. Migrated VM's banking app interface
        # maps to VLAN 130 and gets its production IP on 10.130.0.0/24.
    - source:
        id: network-1002
                # DPortGroup-seg-dmz-app → VLAN 140 DMZ NAD
      destination:
        type: multus
        name: vlan140-dmz
        namespace: virt-workloads
    - source:
        id: network-1003
                # seg-infra-services → OVN pod overlay (management/cluster access only)
      destination:
        type: pod
  provider:
    source:
      name: vcenter-ocp
      namespace: openshift-mtv
    destination:
      name: host
      namespace: openshift-mtv

09 — StorageMap — vSAN to ODF StorageClass

storagemap-ncb.yaml — StorageMap — every field annotated
apiVersion: forklift.konveyor.io/v1beta1
kind: StorageMap
metadata:
  name: ncb-storage-map
  namespace: openshift-mtv
spec:
  map:
    - source:
        id: datastore-101
                # Internal vSphere ID of the vSAN datastore.
        # Retrieve from MTV UI: Providers → vcenter-ocp → Storage.
      destination:
        storageClass: ocs-storagecluster-ceph-rbd-virtualization
                # The VM-tuned StorageClass from VIRT-03 Section 18:
        # - Thick provisioned — full capacity reserved at PVC creation
        # - VM-optimised RBD image features (layering, deep-flatten, exclusive-lock, object-map)
        # - ReadWriteMany access mode — live migration supported for migrated VMs.
        # Do NOT use ocs-storagecluster-ceph-rbd (ReadWriteOnce — breaks live migration
        # for migrated VMs just as it would for natively created VMs).
        accessMode: ReadWriteMany
                # Explicitly set RWX — overrides the StorageClass default if needed.
        # Migrated VMs must support live migration: RWX is the requirement.
  provider:
    source:
      name: vcenter-ocp
      namespace: openshift-mtv
    destination:
      name: host
      namespace: openshift-mtv

11 — MigrationPlan — cold migration — every field annotated

migrationplan-cold.yaml — cold migration for test and pre-production VMs
apiVersion: forklift.konveyor.io/v1beta1
kind: Plan
metadata:
  name: ncb-cold-migration-wave1
    # Name identifies the migration wave. NovaCrestBank migrates in waves —
  # wave1: test/preprod VMs (cold). wave2: production VMs (warm).
  namespace: openshift-mtv
spec:
  description: "Wave 1 — cold migration of test and pre-production VMs"
  provider:
    source:
      name: vcenter-ocp
      namespace: openshift-mtv
    destination:
      name: host
      namespace: openshift-mtv
  map:
    network:
      name: ncb-network-map
      namespace: openshift-mtv
    storage:
      name: ncb-storage-map
      namespace: openshift-mtv
  targetNamespace: virt-workloads
    # targetNamespace: the OCP namespace where migrated VMs are created.
  # VirtualMachine CRs, PVCs, and DataVolumes are created here after migration.
  # The NADs referenced by the NetworkMap (vlan130-primary, vlan140-dmz) must
  # exist in this namespace — verified in pre-flight Check 4.
  warm: false
    # warm: false — cold migration. Source VMs must be powered off before transfer.
  # warm: true enables CBT-based incremental sync (see Section 14 for warm migration).
  vms:
    - id: vm-101
      name: vm-win-test-01
            # id: internal vSphere VM ID (moref). Retrieve from MTV inventory API or UI.
      # name: display name for migration plan reference. Does not have to match vCenter name
      # but should for clarity. The migrated VirtualMachine CR uses the vCenter VM name.
    - id: vm-102
      name: vm-rhel-test-01
    - id: vm-103
      name: vm-preprod-01
  preserveClusterCpuModel: true
    # preserveClusterCpuModel: true — use the destination cluster's CPU model
  # rather than the exact source host CPU model.
  # Why: the DL385 Gen11 (EPYC 9354P) is a different CPU than the ESXi hosts (Xeon).
  # If the source VM was configured with host-model CPU in vSphere and we tried to
  # emulate the exact Xeon model on EPYC hardware, some CPU features would not be
  # available. Setting this to true tells KubeVirt to use the cluster's native CPU
  # model — EPYC — which all VMs on the bare-metal nodes share.
  # Result: migrated VMs may gain access to EPYC-specific CPU extensions (AVX-512,
  # SHA-NI) that the source Xeon did not have. Applications must be tested if they
  # depend on specific CPU features being absent.

13 — Warm migration — what CBT is and how cutover works

Warm migration is the method for production VMs that cannot tolerate extended downtime. Understanding the mechanism — specifically Changed Block Tracking (CBT) and the pre-copy/cutover phases — is essential for planning the maintenance window correctly.

What Changed Block Tracking (CBT) is

CBT is a VMware feature that tracks which disk blocks have changed since a given point in time. When CBT is enabled on a VM, vSphere maintains a change log for each VMDK. MTV uses the CBT API to retrieve only the changed blocks during incremental synchronisation — not the entire disk. This is what makes warm migration efficient: instead of transferring a 200 GiB disk again during the final cutover, MTV only transfers the blocks that changed since the last sync — typically a few hundred megabytes for a production banking application VM.

The three phases of warm migration

Phase 1 — Initial copy: MTV creates a snapshot of the VM's disk in vSphere (using CBT). The full disk data is transferred to ODF — identical to a cold migration transfer, but the VM stays running. This is the longest phase — 20–90 minutes depending on disk size. The VM continues serving production traffic throughout.

Phase 2 — Incremental sync (pre-copy): After the initial copy, MTV uses CBT to identify blocks that changed while Phase 1 was running and syncs those changes to ODF. Then it syncs again. And again. Each sync iteration is shorter than the previous because the change delta is smaller — the VM's ODF copy converges toward the live disk state. MTV continues pre-copy sync iterations until the engineer triggers cutover.

Phase 3 — Cutover: The engineer triggers cutover during a maintenance window (even a brief one — 3–10 minutes). MTV performs one final CBT sync, powers off the source VM in vSphere, performs a final incremental sync to capture any last changes, and starts the VM in OCP Virt. The ODF copy of the disk is now current. Total cutover downtime = final sync time + VM boot time.

CBT must be enabled on source VMs before warm migration — verify first

CBT is not enabled by default on all VMware VMs. It must be explicitly enabled in vSphere. Without CBT, MTV cannot perform incremental sync and falls back to transferring the entire disk at every sync iteration — eliminating the warm migration advantage. Verify CBT status before creating a warm MigrationPlan: in vCenter, right-click the VM → Edit Settings → VM Options → Advanced → Edit Configuration → check for ctkEnabled = TRUE. If not set, enable CBT and take a snapshot to initialise the CBT log.

16 — DNS record update — TTL strategy and cutover procedure

After migration, the VMs have new IP addresses on VLAN 130 (or VLAN 140). The old DNS records point to their old IPs on the NSX overlay segments. Any system that resolves the VM's hostname gets the old IP until the DNS record is updated. The DNS update is the last step before a migrated VM is officially "live" on OCP Virt.

TTL reduction before migration — reduce cache impact

DNS records have a TTL (Time To Live) — the number of seconds a resolver may cache the record before checking DNS again. The standard TTL in NovaCrestBank's AD-integrated DNS zone is 3600 seconds (1 hour). A migrated VM with a new IP will not be reachable by its old cached DNS record for up to 1 hour after the DNS update.

The correct preparation: at least 1 hour before migration begins, reduce the DNS TTL for the migrating VM's A record to 60 seconds. After 1 hour, all caches will have expired their old TTL entries and will start using the 60-second TTL. After cutover, the DNS record is updated with the new IP. All caches expire within 60 seconds — traffic switches to the new IP within 1 minute of the DNS update.

DC01 — PowerShell — TTL reduction and post-migration DNS update
# Step 1 (1+ hour before cutover): reduce TTL to 60 seconds for production VMs
PS C:\> $zone = "novacrestbank.local"
foreach ($vm in @("vm-banking-app-01","vm-banking-app-02","vm-api-gateway-01")) {
    Set-DnsServerResourceRecord `
        -ZoneName $zone `
        -OldInputObject (Get-DnsServerResourceRecord -ZoneName $zone -Name $vm -RRType A) `
        -NewInputObject (Get-DnsServerResourceRecord -ZoneName $zone -Name $vm -RRType A |
            ForEach-Object { $_.TimeToLive = [TimeSpan]::FromSeconds(60); $_ })
    Write-Host "TTL reduced to 60s for ${vm}"
}
TTL reduced to 60s for vm-banking-app-01
TTL reduced to 60s for vm-banking-app-02
TTL reduced to 60s for vm-api-gateway-01

# Wait 1+ hour for TTL changes to propagate through all DNS caches
# Then proceed with migration cutover (Section 15)

# Step 2 (immediately after cutover): update A record to new VLAN 130 IP
# Old IP was on NSX overlay segment. New IP is on VLAN 130.
PS C:\> $migrations = @{
    "vm-banking-app-01" = "10.130.0.50"
    "vm-banking-app-02" = "10.130.0.51"
    "vm-api-gateway-01" = "10.140.0.50"
}

foreach ($vm in $migrations.Keys) {
    $newIP = $migrations[$vm]
    # Remove old A record
    Remove-DnsServerResourceRecord -ZoneName $zone -Name $vm -RRType A -Force
    # Create new A record with new IP (PTR auto-created if "Create PTR" is checked)
    Add-DnsServerResourceRecordA `
        -ZoneName $zone `
        -Name $vm `
        -IPv4Address $newIP `
        -TimeToLive ([TimeSpan]::FromSeconds(3600)) `
        -CreatePtr
    Write-Host "Updated: ${vm} → ${newIP}"
}

18 — Rollback — when and how to revert a migration

Not every migration succeeds cleanly. Know the rollback procedure before starting any production migration — not after a failure has already occurred.

When rollback is appropriate

Rollback is appropriate when post-migration validation reveals that the migrated VM is not functioning correctly and the issue cannot be resolved quickly (within the agreed maintenance window for warm migration, or within acceptable downtime for cold migration). Rollback is specifically the ability to power the source VM back on in vSphere and restore service there while the OCP Virt issue is investigated.

Rollback is only possible if the source VM has not been deleted

The source VM in vSphere is left in place (powered off) after migration. Rollback is straightforward: power on the source VM in vSphere, update DNS back to the old IP, and declare the migration reverted. Delete the migrated VM from OCP Virt until the issue is resolved. Once the source VM is deleted from vSphere, rollback requires restoring from backup — a significantly more complex operation. Do not delete source VMs until the migration is fully validated and stable for at least 7 days.
