The two upgrade types — z-stream patch and minor version
Not all OCP upgrades are the same. Understanding the two types — and the different risk profiles, preparation requirements, and execution timelines each carries — is the foundation of a mature upgrade strategy.

Z-stream (patch) upgrade — 4.16.x → 4.16.y

Updates within the same minor version. Changes the third number only: 4.16.3 → 4.16.7. Contains: bug fixes, security patches, CVE remediation. Does NOT change: API compatibility, operator CRD schemas, RHCOS major kernel version. Risk profile: Low. These upgrades are routine maintenance — apply within 30 days of release under NovaCrestBank's patching policy. Preparation: minimal (verify cluster is healthy, take etcd backup). Execution time: 45–90 minutes for a 9-node cluster.

Minor version upgrade — 4.16 → 4.17

Updates the minor version number. Changes the second number: 4.16.x → 4.17.y. May change: API deprecations and removals, operator CRD schema changes, RHCOS kernel version, default security policy changes. Risk profile: Moderate to High. These upgrades require formal planning, operator compatibility assessment, application testing in non-production environments, change management approval, and a maintenance window. Execution time: 2–4 hours for a 9-node cluster. At NovaCrestBank, EUS channels mean this happens at most once per 18 months.

The upgrade version format decoded
OCP version numbers like 4.16.7 — what does each number mean and what changes at each position?
OCP uses semantic versioning with a specific meaning for each component: 4.16.7 = major.minor.patch.

Major (4): The platform generation. OCP 4.x is the Kubernetes-native, operator-driven generation. A major version change (3 → 4) is a complete architectural change — not an upgrade, a migration. NovaCrestBank will not face a major version change in any foreseeable planning horizon. Major version 4 is the long-term platform.

Minor (16): The feature release. Red Hat releases a new minor version approximately every 4 months. Each minor version adds new features, deprecates old APIs, and introduces new operator versions. Minor versions have a defined support window — typically 18 months for EUS versions, 9 months for standard versions. When a minor version reaches End of Life, the cluster must be upgraded to a supported minor version.

Patch (7): The z-stream release. Red Hat releases z-stream updates every 2–4 weeks. Each z-stream contains accumulated bug fixes and security patches for that minor version. The highest z-stream is always the recommended version — never run a lower z-stream than necessary. z-stream updates never remove APIs or change behaviour — they only fix bugs and patch CVEs.

NovaCrestBank's practical rule: always run the latest z-stream within the current minor version. Never lag more than 3 z-streams behind (that implies an unpatched CVE has been open for 6–12 weeks). For minor version upgrades, use EUS channels and plan thoroughly — no more than twice per year, ideally once per year.
UPG-01 · Section 02
The Cluster Version Operator — the engine that drives every upgrade
The Cluster Version Operator (CVO) is the OCP component responsible for managing the cluster's version. Every upgrade — from a z-stream patch to a major minor version — is orchestrated by the CVO. Understanding what the CVO does and how to observe its state is essential for managing upgrades.
What the CVO does and how it relates to the cluster operators
When an upgrade is initiated, who actually performs the upgrade? Is there a single process that upgrades everything, or multiple components?
The CVO is the conductor — it orchestrates but does not directly implement the upgrade. When an upgrade is triggered, the CVO pulls the new release payload from the OCP release registry (mirrored to Harbor in NovaCrestBank's air-gapped environment). The release payload is a container image containing the manifests and images for every component in the new OCP version. The CVO applies these manifests to update each cluster operator in a defined order. Each cluster operator is then responsible for upgrading its own component.

For example: the CVO tells the Machine Config Operator (MCO) that its new version has arrived. The MCO picks up the new machine config (RHCOS version, kernel parameters) and begins a rolling drain-and-reboot of each node. The CVO waits for the MCO to report that all nodes are updated before proceeding to the next operator. This ordered, operator-driven approach means: the upgrade is declarative (desired state in the release payload), self-healing (if a step fails, the CVO retries), and observable (each operator reports its progress as a ClusterOperator condition).

UPG-01 · Section 03
The update graph — safe paths between versions
OCP upgrades cannot go from any version to any other version directly. Red Hat maintains an update graph that defines the safe paths — which source versions are tested and supported to upgrade to which target versions. Understanding the graph prevents broken upgrades.
Why you cannot skip versions — the update graph enforces tested paths
If NovaCrestBank is on 4.16.3 and 4.16.9 is current, can the cluster upgrade directly to 4.16.9? Or must it go through 4.16.4 → 4.16.5 → ... → 4.16.9?
For z-stream upgrades within the same minor version: yes, you can jump directly. 4.16.3 → 4.16.9 is a tested and supported path. Red Hat tests all z-stream skips within a minor version. The update graph explicitly lists which z-streams can upgrade to which target z-streams, and all previous z-streams can typically upgrade directly to the latest.

For minor version upgrades: you cannot always jump directly. The update graph defines specific tested paths. For example, upgrading from 4.14 to 4.16 requires going through 4.15 — the graph does not include a direct 4.14 → 4.16 edge. Attempting to specify a target version that is not in the update graph results in the CVO rejecting the upgrade with an error: "the requested update to X.Y.Z is not available from the current state."

Why the restriction exists: each minor version upgrade involves migrating data (etcd schemas, API objects), updating RHCOS, and migrating operator CRDs. Skipping a minor version means skipping a migration step, which can corrupt cluster state. Red Hat tests each specific version-to-version edge and only adds it to the graph when it is confident the path is safe. An edge that appears in the graph has been tested with real clusters running the source version upgrading to the target version.

How to check the graph for NovaCrestBank:
Online: https://access.redhat.com/labs/ocpupgradegraph/update_path
CLI tool: oc adm upgrade --to=4.17.0 — the CVO responds with whether the path is available or not.
The availableUpdates field in the ClusterVersion object (shown in Section 02) lists exactly the versions the CVO knows are safe to upgrade to from the current state.

Update channels — stable, fast, candidate, and EUS
The update channel controls which versions the CVO offers as available updates. The channel is not a risk level — it is a selection filter for which versions of the update graph are visible.

Why subscribing to the wrong channel causes problems
If NovaCrestBank accidentally switches from eus-4.16 to stable-4.16, the CVO immediately offers all stable z-stream updates — which is fine. But the availableUpdates list changes: stable-4.16 may offer 4.17.x as an available update (the standard path), bypassing the EUS path requirements. If a z-stream upgrade is accidentally initiated through the stable channel to a non-EUS 4.17 version, the cluster is no longer on an EUS-qualified path and loses the 18-month support window. Always verify the channel is correct before initiating any upgrade.
UPG-01 · Section 05
Why EUS matters for banking — 18-month support windows and change management
EUS (Extended Update Support) is not just a technical feature — it is a strategic alignment between Red Hat's support calendar and the realities of managing software upgrades in a regulated banking environment.
The specific advantages of EUS for NovaCrestBank's upgrade governance
Standard OCP minor versions have 9-month support windows. EUS versions have 18-month windows. Why does this matter for banking operations specifically?
A 9-month support window means the cluster must complete a minor version upgrade within 9 months of the current version's release date — or the cluster reaches End of Life and Red Hat stops providing security fixes. For NovaCrestBank, completing a minor version upgrade in under 9 months is operationally challenging for four specific reasons.

1. Change Advisory Board cycles. NovaCrestBank's CAB (Change Advisory Board) reviews and approves significant infrastructure changes quarterly. A minor version upgrade is a significant change — it requires CAB review, risk assessment, rollback planning documentation, and approval. The CAB process alone takes 6–8 weeks from initial submission to approval. With a 9-month window: submit to CAB immediately after the new version releases, get 6–8 weeks of approval cycle, leave 1–2 months for the actual upgrade preparation and execution. No margin for issues.

2. Regulatory examination timing. FFIEC and APRA (Australian Prudential Regulation Authority) examinations are scheduled 6–12 months in advance. Performing a cluster upgrade during an examination period or within 30 days before/after an examination creates additional risk and scrutiny. An 18-month window means NovaCrestBank can schedule upgrades to avoid examination periods.

3. Operator ecosystem maturity. Red Hat Marketplace operators (ACS, Pipelines, GitOps, Logging) have their own support matrices — they support specific OCP minor versions. A newly released OCP 4.17 may not yet have all NovaCrestBank's operators certified for that version when 4.17 releases. With an 18-month EUS window, NovaCrestBank can wait 3–6 months for all operators to certify against the new OCP minor version before planning the upgrade.

4. Application regression testing. NovaCrestBank runs 50 banking microservices. Each minor OCP upgrade may introduce API deprecations or behavioural changes that require application testing — Kubernetes API group changes, OCP Route changes, security context defaults. 18 months provides time for thorough regression testing without rushing.

UPG-01 · Section 06
The Machine Config Operator — how RHCOS nodes are updated
When the CVO upgrades the cluster, the Machine Config Operator (MCO) is responsible for updating each node's operating system. Understanding how the MCO works explains the rolling reboot pattern and why node upgrades take the time they do.
1
CVO delivers new MachineConfig
The CVO applies the new release payload which includes an updated MachineConfig for the new RHCOS version. The MachineConfig specifies: the new RHCOS OSTree commit (the immutable root filesystem version), kernel parameters, systemd units, and CRI-O configuration. The MCO detects this new MachineConfig and begins the rolling update.
2
MachineConfigPool triggers
The MCO uses MachineConfigPools (MCPs) to manage node groups. NovaCrestBank has three MCPs: master (the 3 control plane nodes), infra (the 3 infra nodes), and worker (the 3 worker nodes). The MCO processes MCPs in a specific order: master first, then the others. Within each pool, nodes are drained and updated one at a time.
3
Node drain, reboot, and rejoin
For each node the MCO processes: (1) the kubelet cordon marks the node unschedulable, (2) the kubelet drains all running pods (with respect to PodDisruptionBudgets — it will not evict a pod if doing so would violate the PDB), (3) the MCO applies the new RHCOS OSTree commit via rpm-ostree (this is an atomic transaction — the new filesystem is staged, not applied in-place), (4) the node reboots into the new RHCOS, (5) after reboot, the kubelet re-registers and the node becomes schedulable. Pod scheduling resumes. Time per node: 10–15 minutes (drain + OSTree stage + reboot + kubelet startup).
4
MachineConfigPool status
The MCO reports progress via the MachineConfigPool's status. machineCount: total nodes in the pool. updatedMachineCount: nodes that have been updated. readyMachineCount: updated nodes that are Ready and schedulable. The pool is complete when all three counts are equal. The CVO waits for the master MCP to complete before allowing worker and infra pools to begin.
UPG-01 · Section 07
Cluster operators — the components that upgrade in order
OCP is composed of approximately 30 cluster operators — each responsible for one platform component. During an upgrade, the CVO applies each operator's new manifests in a defined dependency order. The upgrade is only complete when all cluster operators report Available=True.
Cluster operators vs OLM operators — a critical distinction
Cluster operators (managed by the CVO) are the built-in OCP platform components: etcd, kube-apiserver, network, dns, authentication, monitoring, ingress, image-registry, and ~25 others. They upgrade automatically with the cluster — you do not manage their upgrades directly. OLM operators (managed by OLM subscriptions) are the operators NovaCrestBank installed: ACS, Pipelines, GitOps, Logging, Tempo, OpenTelemetry. These do NOT upgrade automatically with the cluster — they have their own Subscription and InstallPlan approval workflow (OPS-01 Section 16). After a cluster upgrade, NovaCrestBank must separately upgrade each OLM operator to a version certified for the new OCP minor version.

UPG-01 · Section 08
Rolling order — masters first, then infra and workers
The MCO upgrades nodes in a strict order within each MachineConfigPool, and processes pools in a dependency-driven order. Understanding the order explains why the upgrade is safe for a 3-master cluster and how banking applications remain available throughout.
Phase 1
Control plane operators upgrade (no node reboots yet)
The CVO first upgrades the cluster operators: etcd, kube-apiserver, kube-controller-manager, kube-scheduler, and openshift-apiserver. Each operator upgrades its own pods using rolling updates — new pods come up, old pods terminate. For etcd (3 members): upgraded one member at a time, always maintaining quorum (2 of 3 members healthy). The API server uses rolling pods with no downtime. This phase: 15–30 minutes. Banking applications continue running normally — they are unaffected by control plane pod rolling updates.
Phase 2
Master node RHCOS updates (3 nodes, sequential)
The MCO processes the master MachineConfigPool one node at a time. For each master: drain (etcd gracefully steps down from leadership, all pods evicted), apply new RHCOS, reboot, rejoin. With 3 masters, quorum is maintained throughout (2 of 3 available at all times while one reboots). Each master takes 10–15 minutes: drain (2–3 min) + RHCOS apply (3–5 min) + reboot (2–3 min) + kubelet startup (1–2 min). Total phase 2: ~45 minutes for 3 masters. Banking applications on worker nodes are completely unaffected.
Phase 3
Infra and worker node RHCOS updates (6 nodes, sequential or parallel pools)
After all masters are updated, the MCO processes infra and worker MCPs. NovaCrestBank's 3 infra nodes and 3 worker nodes each process sequentially. During infra node draining: monitoring stack pods (Prometheus, Alertmanager) are temporarily relocated to the remaining infra nodes — they have anti-affinity rules ensuring they do not all land on one node. During worker node draining: banking application pods are evicted and rescheduled to the two remaining workers. PodDisruptionBudgets (configured in the GitOps config repo overlays) ensure the MCO never evicts a pod if it would leave fewer than the minimum required replicas available. Each node: ~12 minutes. Total phase 3: ~72 minutes for 6 nodes (if both pools process simultaneously).
Phase 4
Remaining platform operator upgrades (after nodes are updated)
After all nodes are running the new RHCOS, the CVO completes the upgrade of remaining cluster operators that depend on the new RHCOS version (monitoring, ingress, image-registry, etc.). The ClusterVersion PROGRESSING=True until all operators report the new version. Total phase 4: 15–20 minutes. When PROGRESSING=False and VERSION shows the new version: upgrade is complete.
Total upgrade time estimate for NovaCrestBank's 9-node cluster
Phase 1 (control plane operators): 15–30 min + Phase 2 (3 masters): 45 min + Phase 3 (6 infra + worker nodes, parallel): 72 min + Phase 4 (remaining operators): 20 min = 2.5–3 hours total for a z-stream upgrade. A minor version upgrade (4.16 → 4.17) is longer due to additional migration steps: typically 3.5–4.5 hours. Plan a 6-hour maintenance window to include pre-upgrade verification and post-upgrade health checks with comfortable margin.
UPG-01 · Section 09
Zero-downtime mechanics — how banking apps survive a node drain
An OCP upgrade drains and reboots every worker node. NovaCrestBank's banking applications must remain available throughout. Three Kubernetes mechanisms ensure this: PodDisruptionBudgets, replica count, and rolling deployment strategy.
What happens to the payment-service when worker-0 is drained during an upgrade
The payment-service runs 3 replicas on 3 worker nodes (one per node with anti-affinity). When the MCO drains worker-0, what happens to the payment-service replica that was running there?
The MCO calls the Kubernetes API to cordon and drain worker-0. The drain command evicts pods respecting PodDisruptionBudgets. The payment-service has a PDB (configured in the production Kustomize overlay in the GitOps config repo, OPS-05 Section 13) that says: minAvailable=2 — at least 2 payment-service replicas must be available at all times.

The drain sequence for the payment-service:
1. The MCO requests eviction of the payment-service pod on worker-0.
2. Kubernetes checks the PDB: evicting this pod would leave 2 replicas running on worker-1 and worker-2. 2 ≥ minAvailable(2) — the eviction is permitted.
3. The pod on worker-0 is gracefully terminated (SIGTERM → 30-second graceful shutdown window → SIGKILL if not done).
4. The OCP scheduler creates a replacement pod. With worker-0 cordoned, the replacement lands on worker-1 or worker-2 (whichever has capacity). Now there are still 2 running replicas: one on worker-1, one on worker-2 (plus the replacement scheduling).
5. The Service load balancer (and NSX LB) continues routing payment API traffic to the 2 remaining pods on worker-1 and worker-2. Traffic is never interrupted.
6. worker-0 reboots and returns to Ready. The replacement pod continues running on worker-1 or worker-2 (the scheduler does not automatically rebalance — the pod stays where it is until the next rolling update or restart).

What would break zero downtime: if the payment-service had minAvailable=3 in the PDB (or maxUnavailable=0), the MCO would be unable to evict the pod without violating the PDB — the drain would hang indefinitely. Conversely, if the payment-service had only 1 replica (no replicas to spare), draining the node would cause a brief outage. The banking applications must have at least 2 replicas AND a PDB that allows at least 1 to be unavailable during maintenance.

UPG-01 · Section 10
Planning the upgrade — timeline and change management
A cluster upgrade is a Significant Change at NovaCrestBank — it requires formal planning, CAB approval, a maintenance window, rollback planning, and stakeholder notification. This section provides the complete planning timeline.
T-8w
8 weeks before: initial planning and operator compatibility
Check Red Hat's upgrade graph for the target version path. Verify all NovaCrestBank OLM operators have versions certified for the target OCP minor version (ACS, Pipelines, GitOps, Logging, Loki, Tempo, OTel). Check the Red Hat Customer Portal for known issues with the target version on vSphere/VCF platforms. Identify the maintenance window date (off-peak: Saturday 10 PM – Sunday 4 AM). Draft the change request for CAB submission.
T-6w
6 weeks before: CAB submission and stakeholder notification
Submit the change request to the Change Advisory Board. Include: upgrade scope (version from/to), maintenance window, impact assessment (which services may experience brief reduced capacity during node drains), rollback plan (etcd backup procedure if reverting is required), test plan (verification steps post-upgrade), and approval requirements. Notify the banking operations team, the SOC team, and the CISO of the planned maintenance window date.
T-2w
2 weeks before: oc-mirror and payload staging
Run oc-mirror to pull the target OCP release payload and all updated operator images into Harbor. Verify all images are present in Harbor. Test the mirror integrity. Confirm the CVO can see the target version in availableUpdates from the Harbor mirror. If any images are missing, 2 weeks is enough time to identify and resolve the issue before the maintenance window.
T-1w
1 week before: complete pre-upgrade health checks
Run all pre-upgrade health checks (Sections 13–15): all cluster operators Available, all nodes Ready, etcd health, PDB configuration, storage health, monitoring stack healthy. Document the current cluster state for comparison after the upgrade. Take a full etcd backup (Section 18). If any health check fails, decide whether to remediate now or postpone the maintenance window.
Maintenance window
The upgrade execution (UPG-02 guide)
Execute the upgrade per UPG-02. Take a fresh etcd backup immediately before initiating the upgrade. Monitor the upgrade using the checks in UPG-02. Post-upgrade: run verification checks (all operators at new version, all nodes Ready, all banking applications responding). Declare upgrade complete and close the change request.
T+1w
1 week after: post-upgrade review
Review monitoring dashboards for any regressions introduced by the upgrade. Review Splunk for any unusual events during and after the upgrade. Update operator subscriptions to new certified versions (OLM operators that need manual version pinning). Document any issues encountered and lessons learned. Update the upgrade runbook.

Operator compatibility — checking OLM operators before a minor version upgrade
OLM operators (ACS, Pipelines, GitOps, Logging, Loki, Tempo, OTel) have specific OCP version support matrices. Before a minor version upgrade, every installed operator must have a version certified for the target OCP minor version.
An incompatible OLM operator can cause cluster-level problems after a minor version upgrade
An OLM operator that is not certified for a new OCP minor version may: use deprecated Kubernetes APIs that were removed in the new version (causing the operator's webhooks to fail all requests), have incompatible CRD schema versions (causing existing Custom Resources to become unrecognisable), or conflict with changed security context defaults (causing operator pods to be rejected by the new OCP admission webhook). UPG-01 · Section 16
What to do when a cluster operator is Degraded before the upgrade
A Degraded cluster operator blocks the upgrade. This section covers the diagnostic approach and resolution patterns for the most common causes of operator degradation before an upgrade.
The diagnostic approach — always read the operator's message before guessing
A cluster operator shows DEGRADED=True. What is the correct first step?
The first step is always to read the operator's degraded message — not to restart pods or apply changes. The message tells you specifically what is wrong. Operators are designed to provide human-readable degradation reasons in their status conditions. Reading the message first saves significant time over trial-and-error remediation.

The most common causes of cluster operator degradation and their resolutions:

etcd operator Degraded: Almost always a member health issue. Check etcdctl endpoint health --cluster. If a member is unhealthy, check the etcd pod logs on that member's node and the node's disk health. etcd degradation before an upgrade is a stop-everything situation — resolve before proceeding.

monitoring operator Degraded: Usually a storage issue (Prometheus PVC full or VolumeSnapshot failing). Check oc get pvc -n openshift-monitoring. If the Prometheus PVC is at 100%, delete old TSDB blocks or resize the PVC.

machine-config operator Degraded: A node is stuck in a configuration update. Check oc get machineconfigpools and oc get nodes. If a node has DEGRADED=True in its MCP, check the node's MCD (machine-config-daemon) pod logs on that node.

ingress operator Degraded: Often a certificate issue (the ingress default certificate is expired or cannot be renewed). Check oc get clusteroperator ingress -o yaml for the specific message. Check cert-manager CertificateRequest status if using cert-manager for the ingress certificate.

UPG-01 · Section 18
etcd backup — the mandatory pre-upgrade safety net
An etcd backup taken immediately before the upgrade is the cluster's safety net. If the upgrade causes unrecoverable cluster state corruption, the backup allows restoring the cluster to the pre-upgrade state. Taking the backup is not optional — it is mandatory change management requirement at NovaCrestBank.
There is no supported rollback for an OCP upgrade — etcd backup is the only recovery option
OCP does not support downgrading a cluster version. Once an upgrade begins, it must complete. If a completed upgrade causes a critical problem (an application is incompatible with the new version, an operator fails on the new version, a known regression is discovered post-upgrade), the recovery path is to restore the cluster from the pre-upgrade etcd backup. This is a destructive operation — it restores the cluster to the pre-upgrade state but loses all changes made after the backup was taken. This is why the backup must be taken as close to the upgrade start as possible — ideally within 30 minutes of triggering the upgrade.
