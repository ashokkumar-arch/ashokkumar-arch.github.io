UPG-02 · Section 02
Pause MachineConfigPools — controlling when nodes reboot
Pausing the worker and infra MachineConfigPools before triggering the upgrade gives the operations team full control over when nodes reboot. Without pausing, the MCO begins rebooting worker nodes immediately after masters complete — no opportunity to monitor the control plane upgrade before applying it to workload nodes.
Why pause the worker and infra MCPs — not the master MCP
What does pausing a MachineConfigPool actually do, and why pause workers and infra but not masters?
Pausing a MachineConfigPool tells the MCO: "do not apply any new MachineConfig to nodes in this pool until I explicitly un-pause it." When the upgrade delivers a new MachineConfig (with the new RHCOS version), paused pools receive the new config into their desired state but do not apply it — the nodes in paused pools continue running the current RHCOS version until un-paused.

Why not pause masters: the master pool MUST update first. The etcd and API server upgrades depend on RHCOS-level changes being applied to master nodes. Pausing the master pool would prevent the control plane from upgrading, which would block all subsequent operator upgrades. The CVO has a strict dependency: control plane operator upgrades complete, then master RHCOS, then everything else. Pausing masters would stall the entire upgrade indefinitely.

Why pause workers and infra: after the master RHCOS update completes, the MCO would immediately start draining worker and infra nodes without any pause for human review. In NovaCrestBank's banking environment: the operations team wants to confirm the control plane is fully healthy at the new version before allowing worker nodes (which run the banking applications) to reboot. A 30-minute review window after masters complete gives confidence that the upgrade is proceeding correctly before exposing banking application pods to node drains.
jumphost.novacrestbank.local — pause worker and infra MCPs
# Pause the worker MachineConfigPool
[svc-ocp@jumphost ~]$ oc patch machineconfigpool worker \
  --type merge \

The upgrade is triggered with a single oc adm upgrade command. This tells the CVO to begin applying the new release payload. The command returns immediately — the upgrade runs asynchronously and is monitored separately.

UPG-02 · Section 04
Triggering via the OCP web console — GUI walkthrough
The OCP web console provides a visual upgrade interface. Both the CLI and GUI result in the same CVO command — use whichever is comfortable. The GUI provides a visual progress indicator and shows the upgrade path clearly.
1
Navigate to Cluster Settings
In the OCP web console (console-openshift-console.apps.ocp.novacrestbank.local), log in as a cluster-admin account. Navigate to: Administration → Cluster Settings. The current version (4.16.7) is displayed prominently. If updates are available, an "Update" button appears next to the version. If the button is greyed out, the CVO has not yet discovered available updates from the Harbor mirror.
2
Select the target version
Click "Update". A modal dialog appears showing available update versions from the current channel (eus-4.16). Select version 4.16.9 from the dropdown. The dialog shows: the version, the channel it comes from, a link to the release notes, and whether this is a recommended update. Confirm: channel is eus-4.16, version is 4.16.9, and "Recommended update" is displayed.
3
Confirm and initiate
Click "Update" in the modal. The page immediately shows a progress bar under "Cluster Operators" and "Worker Machines" (and "Control Plane" for a minor version upgrade). The progress bar advances as each CVO task completes. The web console provides the same information as the oc get clusterversion watch — both are acceptable monitoring approaches.
4
Monitor progress in the console
The Cluster Settings page shows: "Updating cluster" with the source and target versions, a progress percentage, and the current step name. Click "View update graph" to see the visual graph of the update path. The "Conditions" tab shows the CVO's current conditions (Progressing, Available, RetrievedUpdates). For each step that completes, the percentage advances. This page auto-refreshes every 30 seconds.
GUI vs CLI — when to use each during the upgrade
Use the GUI for: initiating the upgrade (visual confirmation of the selected version before committing), monitoring overall progress during long upgrade phases (the percentage bar), and the post-upgrade summary. Use the CLI for: diagnosing stalled operators (requires detailed condition inspection), watching MachineConfigPool node-by-node progress, checking individual node status during the RHCOS reboot phase, and taking any remediation actions. The CLI provides more depth; the GUI provides better at-a-glance status during normal progress.

UPG-02 · Section 08
What the MCO does during a node upgrade — inside the node
From the outside, the MCO drains a node and reboots it. From the inside, it uses RHCOS's rpm-ostree atomic update system to stage the new OS version before rebooting. Understanding this explains why RHCOS upgrades are reliable and rollback-capable.
Why RHCOS upgrades are atomic and rollback-capable
Traditional Linux upgrades (apt, yum) apply packages in-place — if the upgrade fails halfway, the system is in an inconsistent state. Why is RHCOS different?
RHCOS uses rpm-ostree, which manages the operating system as an immutable filesystem tree (an "OSTree commit"). The current running OS is one commit. An upgrade stages a new commit alongside the current one — without touching the running filesystem — and then updates the bootloader to boot into the new commit on the next restart. If the new commit fails to boot (kernel panic, boot failure), the bootloader falls back to the previous commit automatically. The system comes back up on the old OS and the MCO reports the failure.

The staging process during an RHCOS upgrade:
1. rpm-ostree deploy <new-ostree-commit-hash> — downloads the new OS tree layers from the registry (Harbor mirror in NovaCrestBank's case), overlays the layers onto the new commit alongside the current one. The running filesystem is never touched.
2. The node reboots into the new commit. The bootloader is configured with the new commit as the default and the old commit as the fallback.
3. If the new RHCOS boots successfully and the kubelet starts and registers with the API server, the MCO marks the node updated and moves to the next node.
4. If the new RHCOS fails to boot, the bootloader automatically falls back to the previous commit after 3 failed boot attempts. The node comes back on the old OS, the MCO detects this, and marks the MachineConfigPool Degraded — alerting the operations team that this specific node failed to update.

UPG-02 · Section 12
Why NSX and vSAN are unaffected by the OCP upgrade
NovaCrestBank's networking (NSX-05 DFW, NSX-T) and storage (vSAN) infrastructure exist at the vSphere layer, below the OCP layer. An OCP upgrade does not touch either system.
The relationship between OCP, NSX, and vSAN — layered independence
When a master node reboots during the RHCOS upgrade, does anything happen to the NSX-T segments, DFW rules, or vSAN datastores?
Nothing. OCP runs on top of the vSphere/VCF platform — it does not manage NSX or vSAN, and upgrades to OCP do not trigger any action in either system.

NSX-T perspective: The NSX-T data plane is implemented in ESXi kernel modules — not in the OCP nodes. The NSX-T manager (10.10.0.60–62) and NSX-T Edge nodes (10.10.0.31–32) are separate VMs managed by VCF and NSX-T, independent of OCP. The OCP nodes are VMs that connect to NSX-T overlay segments (seg-ocp-nodes on NCB-T1-OCP) — but the segment definition, DFW policies, and T0/T1 gateways are all in NSX-T's control plane. When an OCP node reboots, the NSX-T VDS kernel module on the ESXi host ensures the VM's network connectivity is restored immediately when the VM powers back on. The DFW rules, firewall policies, and routing tables in NSX-T are completely unchanged by the OCP upgrade.

vSAN perspective: OCP nodes' PVCs are vSphere volumes (VMDKs) managed by the vSphere CSI driver. The CSI driver runs as a DaemonSet in OCP — its pods are evicted during node drains like any other pod, and rescheduled to the remaining nodes during the update. During the drain: the CSI driver pods on the draining node gracefully terminate, PVCs that were attached to pods on that node are detached by the CSI driver (volumes are not deleted — only detached), and when pods reschedule to other nodes, the CSI driver reattaches the PVCs on the new nodes. vSAN itself (the distributed storage fabric running in the ESXi kernel) is not affected — the VMDK files remain intact on vSAN throughout the OCP node reboot.

StatefulSets during node drains: the monitoring stack's Prometheus StatefulSets and LokiStack's StatefulSets have PersistentVolumeClaims. When their pods are evicted from a draining node, the PVCs are detached from that node and reattached to the node where the replacement pod lands. This reattach takes 30–60 seconds — during which the pod is in Pending state. This is normal and expected during node drains with StatefulSet pods.
NSX-T Manager cluster10.10.0.60–62 (Management Domain VMs) — completely unaffected. No OCP upgrade action touches it.
NSX-T Edge nodes10.10.0.31–32 — unaffected. BGP sessions to spine/leaf remain up. T0/T1 gateways continue routing throughout OCP upgrade.
NSX-05 DFW rulesDFW policies are applied at the ESXi kernel level — not in the OCP nodes. Rules persist across OCP node reboots without any change management action.
vSAN datastoresvSAN cluster (3-node RAID-6 FTT:2 for management domain) unaffected. VMDK files remain on vSAN. PVCs detach and reattach as pods reschedule — vSAN data is never at risk.
NSX LB virtual serversNCB-VS-OCP-API (10.50.0.200:6443) and NCB-VS-OCP-HTTPS (10.50.0.201:443) — continue serving traffic. Pool members (master IPs) remain reachable because at most 1 master reboots at a time.

UPG-02 · Section 17
Post-upgrade: update OLM operators to certified versions
The cluster upgrade is complete but OLM operators (ACS, Pipelines, GitOps, Logging, Loki, Tempo, OTel) do not upgrade automatically. For a z-stream upgrade (4.16.7 → 4.16.9), OLM operators typically do not require version updates — they are certified across all z-streams of the same minor version. For a minor version upgrade (4.16 → 4.17), each OLM operator must be upgraded to a version certified for 4.17.
z-stream vs minor version — when OLM operators need updating
We just completed a z-stream upgrade (4.16.7 → 4.16.9). Do the OLM operators (ACS, Pipelines, GitOps, etc.) need to be updated?
For a z-stream upgrade: almost certainly not. OLM operators certify against OCP minor versions (4.16.x), not specific z-streams. An operator certified for OCP 4.16 works on all 4.16.x releases — 4.16.0 through 4.16.99. After a 4.16.7 → 4.16.9 z-stream upgrade, check if any operator has a new version available that contains bug fixes or security patches, but there is no requirement to update.

For a minor version upgrade (4.16 → 4.17): yes, OLM operators may need updating. The OLM Subscription channel for each operator must be updated to the channel that supports 4.17. For example: Red Hat ACS's stable channel may need to be verified to include a version certified for 4.17. The operator update process is: verify the new certified version is available in Harbor (from the pre-upgrade oc-mirror run), update the Subscription channel or version pin if needed, approve the new InstallPlan through the NovaCrestBank change management process.
