GITOPS-03 · Section 01
Pipeline overview — 8 stages, one complete Jenkinsfile
The NovaCrestBank banking microservice pipeline runs every time a commit merges to the main branch of a source repository. It takes a Java source commit and produces a signed, scanned, deployed container image in staging — and opens a reviewed merge request for production. Every stage has a specific security or quality purpose, not just a technical purpose.

The complete Jenkinsfile — full source before the stage walkthrough
The complete Jenkinsfile for the payment-service pipeline is shown here in its entirety before each stage is analysed in detail. This file lives in gitlab.corp.novacrestbank.com/banking/payment-service at the repository root. Sections 03–10 walk through each stage with full annotation.
banking/payment-service/Jenkinsfile — complete pipeline
// NovaCrestBank — payment-service CI/CD pipeline
// Triggered by: GitLab webhook on push to main branch
// Agent: ncb-banking-agent pod template (GITOPS-02 Section 15)

pipeline {
  agent {
    kubernetes {
      label 'ncb-banking-agent'
      defaultContainer 'jnlp'
    }
  }

  environment {
    SERVICE_NAME    = 'payment-service'
    HARBOR_REGISTRY = 'harbor.apps.ocp.corp.novacrestbank.com'
    HARBOR_PROJECT  = 'banking-apps'
    IMAGE_BASE      = "${HARBOR_REGISTRY}/${HARBOR_PROJECT}/${SERVICE_NAME}"
    IMAGE_TAG       = "${env.GIT_COMMIT[0..7]}"
    IMAGE_FULL      = "${IMAGE_BASE}:${IMAGE_TAG}"
    CONFIG_REPO     = 'git@gitlab.corp.novacrestbank.com:gitops/banking-config.git'
    ARGO_APP_STAGING  = 'payment-service-staging'
    ARGO_APP_PROD     = 'payment-service-production'
    SONAR_PROJECT   = 'ncb-payment-service'
    ACS_CENTRAL_URL = 'https://central.stackrox.svc:443'
  }

  options {
    timeout(time: 45, unit: 'MINUTES')
    buildDiscarder(logRotator(numToKeepStr: '30'))
    disableConcurrentBuilds()
    gitLabConnection('NovaCrestBank GitLab')
  }

  stages {

    // ── STAGE 1: CHECKOUT ──────────────────────────────────────────────────────
    stage('Checkout') {
      steps {
        container('jnlp') {
          checkout scm
          script {
            env.GIT_COMMIT_FULL = sh(
              script: 'git rev-parse HEAD', returnStdout: true).trim()
            env.GIT_AUTHOR     = sh(
              script: 'git log -1 --format="%an <%ae>"', returnStdout: true).trim()
            env.GIT_MESSAGE    = sh(
              script: 'git log -1 --format="%s"', returnStdout: true).trim()
          }
          updateGitlabCommitStatus name: 'jenkins-pipeline', state: 'running'
        }
      }
    }

    // ── STAGE 2: BUILD AND TEST ────────────────────────────────────────────────
    stage('Build and Test') {
      steps {
        container('maven') {
          sh '''
            mvn clean verify \
              -Dmaven.test.failure.ignore=false \
              -B -q
          '''
          sh """
            mvn sonar:sonar \
              -Dsonar.projectKey=${SONAR_PROJECT} \
              -Dsonar.host.url=https://sonarqube.apps.ocp.corp.novacrestbank.com \
              -Dsonar.qualitygate.wait=true \
              -B -q
          """
          junit 'target/surefire-reports/**/*.xml'
        }
      }
    }

    // ── STAGE 3: BUILD IMAGE WITH BUILDAH ─────────────────────────────────────
    stage('Build Image') {
      steps {
        container('buildah') {
          sh """
            buildah bud \
              --no-cache \
              --layers \
              --label "git.commit=${GIT_COMMIT_FULL}" \
              --label "git.author=${GIT_AUTHOR}" \
              --label "build.timestamp=\$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
              --label "pipeline.build=${BUILD_NUMBER}" \
              -t ${IMAGE_FULL} \
              -f Dockerfile .
          """
        }
      }
    }

    // ── STAGE 4: PUSH TO HARBOR AND SIGN WITH COSIGN ──────────────────────────
    stage('Push and Sign') {
      steps {
        container('buildah') {
          withCredentials([[
            $class: 'UsernamePasswordMultiBinding',
            credentialsId: 'jenkins-harbor-push-credentials',
            usernameVariable: 'HARBOR_USER',
            passwordVariable: 'HARBOR_PASS'
          ]]) {
            sh """
              buildah push \
                --creds "${HARBOR_USER}:${HARBOR_PASS}" \
                --tls-verify=true \
                ${IMAGE_FULL} \
                docker://${IMAGE_FULL}
            """
            script {
              env.IMAGE_DIGEST = sh(
                script: """skopeo inspect \
                  --creds "${HARBOR_USER}:${HARBOR_PASS}" \
                  docker://${IMAGE_FULL} \
                  | python3 -c 'import json,sys; print(json.load(sys.stdin)["Digest"])'""",
                returnStdout: true).trim()
            }
            sh """
              COSIGN_PASSWORD=\$(cat /etc/cosign/cosignPassword) \
              cosign sign \
                --key /etc/cosign/cosign.key \
                --tls-verify=true \
                --annotations="pipeline.build=${BUILD_NUMBER}" \
                --annotations="git.commit=${GIT_COMMIT_FULL}" \
                --annotations="signed.by=ncb-jenkins-pipeline" \
                ${IMAGE_BASE}@${IMAGE_DIGEST}
            """
          }
        }
      }
    }

    // ── STAGE 5: ACS VULNERABILITY SCAN GATE ──────────────────────────────────
    stage('ACS Scan Gate') {
      steps {
        container('jnlp') {
          withCredentials([[
            $class: 'StringBinding',
            credentialsId: 'acs-api-token',
            variable: 'ACS_TOKEN'
          ]]) {
            sh """
              roxctl image check \
                --endpoint ${ACS_CENTRAL_URL} \
                --token \${ACS_TOKEN} \
                --image ${IMAGE_BASE}@${IMAGE_DIGEST} \
                --severity CRITICAL \
                --output table
            """
          }
        }
      }
    }

    // ── STAGE 6: GITOPS STAGING DEPLOY ────────────────────────────────────────
    stage('Deploy to Staging') {
      steps {
        container('jnlp') {
          withCredentials([[
            $class: 'SSHUserPrivateKeyBinding',
            credentialsId: 'jenkins-config-repo-deploy-key',
            keyFileVariable: 'SSH_KEY'
          ]]) {
            sh """
              export GIT_SSH_COMMAND="ssh -i \${SSH_KEY} -o StrictHostKeyChecking=no"
              git clone ${CONFIG_REPO} config-repo
              cd config-repo

              cd apps/${SERVICE_NAME}/overlays/staging
              kustomize edit set image \\
                ${IMAGE_BASE}=${IMAGE_BASE}@${IMAGE_DIGEST}

              git config user.email "ncb-pipeline@corp.novacrestbank.com"
              git config user.name  "NovaCrestBank Pipeline"
              git add kustomization.yaml
              git commit -m "staging: promote ${SERVICE_NAME} to ${IMAGE_TAG}

              Image: ${IMAGE_BASE}@${IMAGE_DIGEST}
              Build: ${BUILD_URL}
              Commit: ${GIT_COMMIT_FULL}
              Author: ${GIT_AUTHOR}"

              git push origin main
            """
            sh """
              timeout 300 bash -c '
              while true; do
                STATUS=\$(argocd app get ${ARGO_APP_STAGING} \\
                  --output json 2>/dev/null \\
                  | python3 -c "import json,sys; d=json.load(sys.stdin); \\
                    print(d[\"status\"][\"sync\"][\"status\"], \\
                          d[\"status\"][\"health\"][\"status\"])")
                echo "Argo CD staging: \${STATUS}"
                echo "\${STATUS}" | grep -q "Synced Healthy" && break
                sleep 15
              done'
            """
          }
        }
      }
    }

    // ── STAGE 7: INTEGRATION TESTS ────────────────────────────────────────────
    stage('Integration Tests') {
      steps {
        container('maven') {
          sh """
            mvn verify \
              -Pintegration-tests \
              -Dtarget.base.url=https://payment-service-staging.apps.ocp.corp.novacrestbank.com \
              -Dmaven.test.failure.ignore=false \
              -B -q
          """
          junit 'target/failsafe-reports/**/*.xml'
        }
      }
    }

    // ── STAGE 8: PRODUCTION MERGE REQUEST ─────────────────────────────────────
    stage('Production Promotion MR') {
      steps {
        container('jnlp') {
          withCredentials([[
            $class: 'StringBinding',
            credentialsId: 'gitlab-api-token',
            variable: 'GL_TOKEN'
          ], [
            $class: 'SSHUserPrivateKeyBinding',
            credentialsId: 'jenkins-config-repo-deploy-key',
            keyFileVariable: 'SSH_KEY'
          ]]) {
            sh """
              export GIT_SSH_COMMAND="ssh -i \${SSH_KEY} -o StrictHostKeyChecking=no"
              cd config-repo

              BRANCH="pipeline/promote-${SERVICE_NAME}-${IMAGE_TAG}"
              git checkout -b \${BRANCH}

              cd apps/${SERVICE_NAME}/overlays/production
              kustomize edit set image \\
                ${IMAGE_BASE}=${IMAGE_BASE}@${IMAGE_DIGEST}

              git add kustomization.yaml
              git commit -m "production: promote ${SERVICE_NAME} to ${IMAGE_TAG}

              Image: ${IMAGE_BASE}@${IMAGE_DIGEST}
              Source commit: ${GIT_COMMIT_FULL}
              Author: ${GIT_AUTHOR}
              Message: ${GIT_MESSAGE}
              Jenkins build: ${BUILD_URL}
              Integration tests: PASSED
              ACS scan: PASSED (no CRITICAL CVEs)"

              git push origin \${BRANCH}

              curl -sk -X POST \\
                "https://gitlab.corp.novacrestbank.com/api/v4/projects/\$(
                  curl -sk "https://gitlab.corp.novacrestbank.com/api/v4/projects?search=banking-config" \\
                    -H "PRIVATE-TOKEN: \${GL_TOKEN}" \\
                  | python3 -c "import json,sys; print([p for p in json.load(sys.stdin) \\
                      if 'gitops/banking-config' in p['path_with_namespace']][0]['id'])"
                )/merge_requests" \\
                -H "PRIVATE-TOKEN: \${GL_TOKEN}" \\
                -H "Content-Type: application/json" \\
                -d "{
                  \\"source_branch\\": \\"\${BRANCH}\\",
                  \\"target_branch\\": \\"main\\",
                  \\"title\\": \\"production: promote ${SERVICE_NAME} to ${IMAGE_TAG}\\",
                  \\"description\\": \\"$(cat << 'MRDESC'
## Production Promotion — ${SERVICE_NAME} ${IMAGE_TAG}

### Change Summary
Promoting \`${SERVICE_NAME}\` from the previously running version to \`${IMAGE_TAG}\`.

**Image digest (immutable reference):**
\`${IMAGE_BASE}@${IMAGE_DIGEST}\`

### Evidence
| Gate | Result |
|------|--------|
| Unit tests | ✅ PASSED (see Jenkins build) |
| SonarQube quality gate | ✅ PASSED |
| Buildah image build | ✅ Non-root, no daemon |
| cosign signature | ✅ Signed with NovaCrestBank key |
| ACS vulnerability scan | ✅ No CRITICAL CVEs |
| Staging deployment | ✅ Synced and Healthy |
| Integration tests | ✅ PASSED |

### Links
- Jenkins build: ${BUILD_URL}
- Source commit: ${GIT_COMMIT_FULL}
- Author: ${GIT_AUTHOR}
- Commit message: ${GIT_MESSAGE}

### Reviewers
@platform-engineers — please review and approve this production promotion.
MRDESC
                  )\\",
                  \\"assignee_id\\": null,
                  \\"labels\\": \\"production-promotion,pipeline-automated\\",
                  \\"remove_source_branch\\": true
                }"
            """
          }
        }
      }
    }
  }

  post {
    success {
      updateGitlabCommitStatus name: 'jenkins-pipeline', state: 'success'
      addGitLabMRComment comment: """✅ Pipeline PASSED — ${SERVICE_NAME} ${IMAGE_TAG}
        Image: \`${IMAGE_FULL}\`
        Digest: \`${IMAGE_DIGEST ?: 'see build log'}\`
        ACS scan: No CRITICAL CVEs
        Staging: Synced and Healthy
        Integration tests: PASSED
        Production MR: opened — awaiting platform engineer approval"""
    }
    failure {
      updateGitlabCommitStatus name: 'jenkins-pipeline', state: 'failed'
      addGitLabMRComment comment: "❌ Pipeline FAILED at stage: ${currentBuild.displayName}. See ${BUILD_URL}console"
    }
    always {
      cleanWs()
    }
  }
}
GITOPS-03 · Section 03
Stage 1 — Checkout and environment setup
The checkout stage does more than clone the repository. It establishes the complete traceability context for every subsequent artifact produced by this pipeline run — the commit SHA that becomes the image tag, the author that appears in audit logs, and the message that becomes the body of the production MR.
Stage 1 — Checkout — annotated
stage('Checkout') {
  steps {
    container('jnlp') {
      checkout scm
            // checkout scm: Jenkins checks out the commit that triggered this build.
              // The GitLab webhook payload contained the commit SHA — Jenkins uses it
              // to ensure it builds exactly the commit that was merged, not HEAD
              // (which may have advanced if another commit merged simultaneously).

      script {
        env.GIT_COMMIT_FULL = sh(
          script: 'git rev-parse HEAD', returnStdout: true).trim()
                // GIT_COMMIT_FULL: the 40-character commit SHA.
                  // Used in: image labels, cosign annotations, production MR body.
                  // Every artifact is traceable back to this exact commit.

        env.GIT_AUTHOR = sh(
          script: 'git log -1 --format="%an <%ae>"', returnStdout: true).trim()
                // GIT_AUTHOR: "Alice Smith "
                  // Appears in: production MR body, image labels, build log.
                  // Satisfies the PCI-DSS Req 10 "who initiated the change" requirement —
                  // even the production MR (opened by the pipeline service account) names
                  // the human whose commit triggered the build.

        env.GIT_MESSAGE = sh(
          script: 'git log -1 --format="%s"', returnStdout: true).trim()
                // GIT_MESSAGE: the first line of the commit message.
                  // "Add NPP payment rail adapter" becomes part of the production MR title,
                  // giving reviewers context on what feature they are approving.
      }

      updateGitlabCommitStatus name: 'jenkins-pipeline', state: 'running'
            // Immediately mark the commit as "pipeline running" in the GitLab UI.
              // The merge request that triggered this build now shows a yellow dot
              // indicating CI is in progress. Developers can see pipeline status
              // in GitLab without opening Jenkins.
    }
  }
}
GITOPS-03 · Section 04
Stage 2 — Maven build, unit tests, and SonarQube quality gate
The build stage runs in the maven container. It produces the application JAR, validates every unit test passes, and submits the code to SonarQube for static analysis. The SonarQube quality gate uses -Dsonar.qualitygate.wait=true — the pipeline blocks until SonarQube's server-side analysis completes and returns a pass or fail verdict.
Why SonarQube is a blocking gate — not a non-blocking advisory scan
SonarQube could run as a non-blocking background scan — the image builds regardless and developers are notified of quality issues separately. Why does NovaCrestBank use -Dsonar.qualitygate.wait=true which blocks the pipeline?
A non-blocking SonarQube scan creates the advisory trap: developers see the scan results after the image is already in Harbor and deployed to staging, at which point fixing the issue requires another full pipeline run. If the quality issue is a CRITICAL security hotspot (SQL injection pattern, hardcoded credential, insecure random number generator), the code with the security flaw has already been built into an image that may have been pushed to Harbor before the vulnerability was discovered.

With qualitygate.wait=true: the SonarQube analysis completes before the image is built. A CRITICAL security hotspot detected by SonarQube stops the pipeline at Stage 2. No image is built. No container with that vulnerability enters Harbor. The developer fixes the issue and the pipeline re-runs from the beginning. This is the correct order: detect before build, not after deploy.

The quality gate is configured in SonarQube with NovaCrestBank-specific thresholds: zero BLOCKER or CRITICAL security hotspots, code coverage above 70% for new code, and no new security vulnerabilities from the OWASP Top 10 categories. These thresholds are maintained by the platform team in SonarQube and applied consistently across all banking microservices.
Stage 2 — Build and Test — annotated
stage('Build and Test') {
  steps {
    container('maven') {
      sh '''
        mvn clean verify \
          -Dmaven.test.failure.ignore=false \
          -B -q
      '''
            // clean verify: compile → unit test → package → verify.
              // -Dmaven.test.failure.ignore=false: any failing unit test fails the build.
              //   Default Maven behaviour is to continue on test failure with a warning.
              //   This flag makes test failures hard failures — no image is built if tests fail.
              // -B: batch mode — no interactive prompts (required in CI).
              // -q: quiet — suppresses INFO logs, shows only warnings and errors.
              //   Use -B without -q during debugging to see full Maven output.

      sh """
        mvn sonar:sonar \
          -Dsonar.projectKey=${SONAR_PROJECT} \
          -Dsonar.host.url=https://sonarqube.apps.ocp.corp.novacrestbank.com \
          -Dsonar.qualitygate.wait=true \
          -B -q
      """
            // sonar:sonar: submit code to SonarQube for analysis.
              // -Dsonar.qualitygate.wait=true: BLOCKING — Maven waits for SonarQube
              //   to complete analysis (typically 30-60 seconds) and return the gate verdict.
              //   If the quality gate fails: Maven exits with code 1 → stage fails → pipeline stops.
              //   If the quality gate passes: Maven exits with code 0 → pipeline continues.

      junit 'target/surefire-reports/**/*.xml'
            // junit: Jenkins parses the Maven Surefire XML reports and records
              // test results in the build history. Enables trending graphs and
              // per-test failure notifications in the Jenkins UI.
    }
  }
}
GITOPS-03 · Section 05
Stage 3 — Buildah image build — why not Docker, why not root
The image build stage uses Buildah running in the buildah container. The choice of Buildah over Docker is not stylistic — it is a specific security requirement enforced by NovaCrestBank's ACS policy and OCP's SCC (Security Context Constraints). The section explains each Buildah flag and why Docker-in-Docker would be rejected.
Why Buildah and not Docker — the three reasons
Docker is the industry-standard container build tool. Most CI examples use Docker. Why does NovaCrestBank's pipeline use Buildah?
Three specific reasons, each grounded in NovaCrestBank's OCP security posture.

Reason 1 — Docker-in-Docker requires a privileged container; OCP and NovaCrestBank's ACS policy prohibit privileged containers in the jenkins namespace. Docker requires a Docker daemon. Running a Docker daemon inside a Kubernetes pod requires either mounting the host's /var/run/docker.sock (giving the container full root access to the host's Docker daemon — worse than privileged) or running Docker-in-Docker (requiring a privileged: true security context). OCP's restricted-v2 SCC — the default for all pods without explicit privilege escalation — prohibits both. The jenkins namespace does not have a SCC exception granting privileged access, and it should not. Buildah builds OCI images using Linux user namespace isolation with no daemon and no privileged mode. It satisfies the same restricted-v2 SCC that all other pods run under.

Reason 2 — Buildah produces standard OCI images, not Docker-specific images. The images Buildah produces are OCI-compliant and indistinguishable from Docker-built images in Harbor, in OCP's container runtime (CRI-O), and in any OCI-compliant registry. There is no compatibility trade-off — the image is just as functional, just built by a different tool.

Reason 3 — NovaCrestBank's ACS policy for the jenkins namespace includes a check: "Image built with Docker daemon: FAIL." ACS inspects image labels during deployment. An image built via Docker-in-Docker carries Docker-specific metadata that ACS's policy engine can detect. Buildah images do not carry this metadata and pass the policy check.
Stage 3 — Build Image — annotated
stage('Build Image') {
  steps {
    container('buildah') {
      sh """
        buildah bud \
          --no-cache \
          --layers \
          --label "git.commit=${GIT_COMMIT_FULL}" \
          --label "git.author=${GIT_AUTHOR}" \
          --label "build.timestamp=\$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
          --label "pipeline.build=${BUILD_NUMBER}" \
          -t ${IMAGE_FULL} \
          -f Dockerfile .
      """
            // buildah bud: "Build Under Dockerfile" — compatible with standard Dockerfile syntax.
              //   Same Dockerfile used in development works with Buildah in CI.

              // --no-cache: always rebuild from scratch.
              //   Agent pods are ephemeral (podRetention=never) — there is no layer cache
              //   from a previous build. --no-cache makes this explicit and prevents
              //   any stale cache from a previous workspace from affecting the build.

              // --layers: enable layer caching within this build only (multi-stage Dockerfiles).
              //   Speeds up builds that use multi-stage patterns (compile stage + runtime stage).
              //   This is not a cross-build cache — it only applies to layers within one build.

              // --label: embed traceability metadata into the image.
              //   git.commit: the exact source code this image was built from.
              //   git.author: who wrote the code — visible in image inspectors.
              //   build.timestamp: when this image was built.
              //   pipeline.build: the Jenkins build number for log correlation.
              //   These labels appear in "docker inspect" / "skopeo inspect" output
              //   and in Harbor's image detail view — full traceability without a separate SBOM.

              // -t ${IMAGE_FULL}: tag the image as harbor.../banking-apps/payment-service:a1b2c3d4
              //   The tag is the 8-character commit SHA (GIT_COMMIT[0..7]).
              //   Not "latest" — never "latest" in production.
              //   "latest" is a mutable tag. A mutable tag means the image referenced in
              //   the config repo could change under Argo CD without a config change.
              //   Commit SHA tags are immutable — the tag always refers to this exact image.
    }
  }
}
GITOPS-03 · Section 06
Stage 4 — Push to Harbor and sign with cosign — why cosign, why the digest
Stage 4 has two sequential operations: push the image to Harbor (producing an immutable digest), then sign that specific digest with cosign. The sequence matters — the signature is over the digest, not the tag. This section explains the cryptographic model and why the config repo references the digest, not the tag.
Why sign the digest and not the tag — the mutable tag attack
The image is tagged payment-service:a1b2c3d4. The cosign sign command signs payment-service@sha256:abc123... (the digest). Why sign the digest and not the tag?
A tag is a mutable pointer. A digest is an immutable cryptographic identity. Signing a tag signs a pointer that could be moved to point at a different image without invalidating the signature.

If cosign signed the tag payment-service:a1b2c3d4: an attacker who compromises Harbor and replaces the content of that tag with a malicious image would not invalidate the cosign signature — the signature was on the tag name, not the content. The ACS admission controller would accept the malicious image because the tag has a valid signature.

If cosign signs the digest payment-service@sha256:abc123...: the signature is cryptographically bound to the specific byte content of the image (SHA-256 of the image manifest). Any change to the image produces a different digest. A malicious image has a different digest — it does not have a valid NovaCrestBank cosign signature — and the ACS admission controller rejects it.

This is also why the GitOps config repo uses digest references, not tags: kustomize edit set image harbor.../payment-service=harbor.../payment-service@sha256:abc123 pins the Deployment to the exact signed image. Argo CD deploys exactly that image — a tag rotation in Harbor cannot silently change what Argo CD deploys.

stage('Push and Sign') {
  steps {
    container('buildah') {
      withCredentials([[...]]) {

        sh """buildah push \
          --creds "${HARBOR_USER}:${HARBOR_PASS}" \
          --tls-verify=true \

          docker://${IMAGE_FULL}"""
                // buildah push: send the locally-built image to Harbor.
                  // --tls-verify=true: verify Harbor's TLS certificate.
                  //   Harbor's cert is signed by NovaCrestBank-IssuingCA.
                  //   The buildah container's /etc/ssl/certs includes the NCB CA bundle
                  //   (deployed via MachineConfig to all worker nodes — UPG-03 Section 09).
                  // docker://: push using the Docker registry protocol (OCI-compatible).
                  // Harbor assigns the image an immutable SHA-256 digest on receipt.

        script {
          env.IMAGE_DIGEST = sh(
            script: """skopeo inspect \
              --creds "${HARBOR_USER}:${HARBOR_PASS}" \
              docker://${IMAGE_FULL} \
              | python3 -c 'import json,sys; print(json.load(sys.stdin)["Digest"])'""",
            returnStdout: true).trim()
        }
                // skopeo inspect: query Harbor for the digest of the image we just pushed.
                  // Returns: {"Digest": "sha256:abc123def456..."}
                  // IMAGE_DIGEST is now available to all subsequent stages as an env var.
                  // This is the immutable identifier used in: cosign sign, config repo commit,
                  // ACS scan command, and production MR body. Every subsequent reference
                  // to this image uses @sha256:... — never the mutable tag again.

        sh """
          COSIGN_PASSWORD=\$(cat /etc/cosign/cosignPassword) \
          cosign sign \
            --key /etc/cosign/cosign.key \
            --tls-verify=true \
            --annotations="pipeline.build=${BUILD_NUMBER}" \
            --annotations="git.commit=${GIT_COMMIT_FULL}" \
            --annotations="signed.by=ncb-jenkins-pipeline" \
            ${IMAGE_BASE}@${IMAGE_DIGEST}
        """
                // cosign sign: cryptographically sign the image digest.
                  // --key /etc/cosign/cosign.key: the NovaCrestBank signing private key,
                  //   mounted as a Secret volume at /etc/cosign in the buildah container.
                  // --annotations: embed verifiable metadata in the signature.
                  //   These annotations are visible when running "cosign verify" —
                  //   an auditor can verify that this specific pipeline build signed this image.
                  // ${IMAGE_BASE}@${IMAGE_DIGEST}: signing the DIGEST, not the tag.
                  //   The signature is stored in Harbor as an OCI artifact attached to
                  //   the image manifest — no separate signature registry needed.
                  // After this command: the image in Harbor has a cosign signature.
                  //   The ACS admission controller can verify the signature at deployment time.
                  //   An unsigned copy of this image (or a tampered copy) would fail admission.
      }
    }
  }
}

After the image is in Harbor and signed, ACS scans it for known CVEs using roxctl image check. This is a blocking gate — a CRITICAL CVE fails the pipeline. The scan uses the image digest (not the tag) to ensure ACS scans the exact image that was pushed.
Stage 5 — ACS Scan Gate — annotated

GITOPS-03 · Section 08
Stage 6 — GitOps staging deploy — commit image digest to the config repo
Stage 6 is the handoff from CI to CD. Jenkins commits the new image digest reference to the staging overlay in the GitOps config repo, and then waits for Argo CD to sync staging to the new state. The wait confirms the deployment configuration is valid before integration tests run.
Why push directly to main for staging but open an MR for production
Stage 6 commits directly to main in the config repo (for the staging overlay). Stage 8 opens a merge request (for the production overlay). Why different workflows for the two environments?
The asymmetry is intentional and reflects the different risk profiles of staging and production changes.

Staging: Staging exists to validate that the application works in a production-like environment before reaching production. If staging has a problem, the consequence is: a developer sees a failing staging deployment and integration tests, and the pipeline stops before opening the production MR. No customer is affected. A direct push to the staging overlay is appropriate — the staging environment's purpose is to absorb and surface failures quickly. Requiring a manual review for staging deployments would add 30+ minutes to every pipeline run with no additional safety benefit (a staging failure is already safe-to-fail by definition).

Production: Production serves real banking transactions. A misconfigured deployment, an application regression, or a security issue that reached production affects real customers and creates regulatory exposure. The merge request is the mandatory human review gate. A platform engineer reads the diff (one line: the image digest reference), verifies the ACS scan result and integration test evidence are attached, and approves. This 5-minute human step is the investment NovaCrestBank makes to prevent production incidents caused by CI/CD automation errors.

GITOPS-03 · Section 10
Stage 8 — Production MR — why a pull request rather than a direct push
Stage 8 opens a merge request against the production overlay in the config repo. This is the only step in the pipeline that requires human action before production deployment occurs. The decision to use an MR rather than a direct push is the single most important architectural choice in the entire pipeline.
Why a merge request rather than a direct push to the production overlay
The pipeline has already run unit tests, SonarQube quality gate, ACS vulnerability scan, staging deployment, and integration tests. At this point, the image is thoroughly validated. Why introduce a human approval step rather than promoting directly to production?
The merge request is not a technical gate — it is a regulatory and organisational gate. The six preceding stages validate the technical quality of the image. The MR validates that a human with production deployment authority has reviewed and authorised this specific deployment at this specific time.

Reason 1 — PCI-DSS Req 6.4 explicitly requires human authorisation for production changes. A fully automated production deployment — even one that passed six quality gates — does not satisfy Req 6.4.2 ("All changes to system components in the production environment are authorised through a formal change control process"). Automation can validate technical quality. It cannot provide human judgment about: is now the right time to deploy this change? Are there conflicting changes in flight? Is there a customer communication that needs to go out simultaneously? Is there a regulatory reason to delay this specific change?

Reason 2 — The MR is the change management record. NovaCrestBank's CAB (Change Advisory Board) process for routine deployments is satisfied by the production promotion MR — it contains the change description, the evidence of testing, the approver's identity, and the timestamp. This is the "lightweight change record" approved by the CAB for fully-automated pipelines: a MR with defined approval criteria, not a full CAB meeting for every routine deployment.

Reason 3 — The MR prevents the pipeline from becoming a production deployment backdoor. If the pipeline had direct push access to the production overlay, a compromise of the pipeline service account would give an attacker the ability to deploy arbitrary images to production (albeit still constrained by the cosign signing requirement). With the MR gate, even if the pipeline service account is compromised, the attacker can only open MRs — they cannot approve them. The approval requires a separate human user with Maintainer role in the config repo, whose credentials are independent of the pipeline service account.

GITOPS-03 · Section 12
Agent declaration — how containers are selected per stage
The pipeline uses a single pod template (ncb-banking-agent) with multiple containers. Each stage declares which container its steps run in. Understanding the container selection model explains why buildah steps must be in the buildah container and not the maven container.
Pod
One agent pod per pipeline run
When the pipeline starts, one pod is created with four containers: jnlp (coordination), maven (Java builds), buildah (image builds and cosign). All four containers share: the same pod filesystem (workspace is shared via emptyDir), the same pod network (all containers can reach each other on localhost), and the same Secret mounts (cosign key at /etc/cosign is visible to the buildah container because it is mounted at the pod level).
Selection
container('name') selects where steps run
container('maven') { sh 'mvn ...' } runs the shell command inside the maven container's process namespace — where the JDK and Maven binary exist. container('buildah') { sh 'buildah bud ...' } runs inside the buildah container — where the Buildah, Skopeo, and cosign binaries exist. Running a Buildah command outside the buildah container would fail: the binary does not exist in other containers.
Shared
Shared workspace across containers
The Jenkins workspace (/home/jenkins/agent/workspace/payment-service) is a shared emptyDir volume mounted in all containers. The Maven JAR produced in Stage 2 (by the maven container) is visible to the buildah container in Stage 3 without any copy step — both containers see the same workspace directory. This is why the Buildah stage can run buildah bud -f Dockerfile . where the Dockerfile references COPY target/payment-service.jar /app/ — the JAR is already in the shared workspace.
GITOPS-03 · Section 13
Credentials binding — secrets in the pipeline without hardcoding
Every credential in the pipeline is bound from a Jenkins credential (itself stored as a Kubernetes Secret in the jenkins namespace) using withCredentials. The credential value is injected as an environment variable and is automatically masked in the build log — the raw value never appears in any log, file, or GitLab MR comment.

Credential mapping — Kubernetes Secret to Jenkins credential to pipeline variable

Harbor push credentialsK8s Secret jenkins-harbor-push-credentials → Jenkins ID jenkins-harbor-push-credentials → HARBOR_USER / HARBOR_PASS env vars in Stage 4
cosign signing keyK8s Secret jenkins-cosign-signing-key → mounted at /etc/cosign/cosign.key and /etc/cosign/cosignPassword in the buildah container via pod volume mount (GITOPS-02 Section 15)
GitLab API tokenK8s Secret jenkins-gitlab-api-token → Jenkins ID gitlab-api-token → GL_TOKEN env var in Stage 8
Config repo SSH keyK8s Secret banking-config-repo → Jenkins ID jenkins-config-repo-deploy-key → SSH_KEY file path in Stage 6 and Stage 8
ACS API tokenK8s Secret jenkins-acs-api-token → Jenkins ID acs-api-token → ACS_TOKEN env var in Stage 5
Why credentials are never in the Jenkinsfile, in Git, or in environment variable blocks
If a credential appeared in the Jenkinsfile (e.g., environment { HARBOR_PASS = 'password123' }), it would be committed to the source repo and visible to anyone with read access to the banking/payment-service repo — all banking developers. Even if removed later, the credential would remain in Git history. withCredentials binds values that exist only in the Jenkins credential store (which lives in /var/jenkins_home, not in Git) into environment variables that are masked in all log output. The Jenkinsfile can be public (or read by all developers) without leaking any credentials.
GITOPS-03 · Section 14
Environment variables — traceability from git commit to running image
The environment block at the top of the Jenkinsfile defines the variables that thread traceability through every stage. The image tag is the commit SHA. The image is referenced by digest. Every artifact that the pipeline produces carries the commit SHA that produced it.
IMAGE_TAG = env.GIT_COMMIT[0..7]The 8-character commit SHA becomes the image tag. Never "latest". A mutable tag means the GitOps config repo could reference a tag that changes its meaning without a config change — defeating the immutability guarantee of GitOps.
IMAGE_DIGEST (set in Stage 4)The SHA-256 digest assigned by Harbor when the image is pushed. All subsequent stages reference the image by @sha256:... digest, not by tag. The production MR commits the digest to the config repo — Argo CD deploys the exact signed byte content, not a tag that could be updated.
CONFIG_REPOThe SSH URL of the GitOps config repo. Centralised here so if the GitLab hostname changes (domain migration, disaster recovery), only one line changes in the Jenkinsfile.
ARGO_APP_STAGINGThe Argo CD Application name for staging. The argocd app get command in Stage 6 uses this name. Defined here rather than hardcoded in the stage script for maintainability.
ACS_CENTRAL_URLACS Central's internal service URL. Internal (in-cluster) URL — not the external Route. roxctl running in the agent pod reaches ACS Central via the cluster's internal service network, not via the public Route. This is faster, does not consume external bandwidth, and avoids dependence on the OCP Router during the scan.

GITOPS-03 · Section 16
The complete trust chain — from git push to signed running pod
Every link in the trust chain from a developer's commit to a running pod in production is cryptographically anchored or policy-enforced. An attacker trying to insert a malicious workload into production must break multiple independent controls simultaneously.
Link 1
Commit author verification
GitLab push rules enforce @novacrestbank.com email on all commits (GITOPS-00 Section 16). A commit from an anonymous or external email address is rejected at push time. Every commit is attributable to a specific AD-authenticated user.
Link 2
Source code review
Protected branch (push_access_level=0) on main branch in the source repo. The commit reached main only because it passed peer review and was merged via MR. No developer can bypass the review process with a direct push.
Link 3
Build provenance
The image was built by an authorised Jenkins pipeline triggered by the GitLab webhook. The build embedded the commit SHA, author, and build number as immutable image labels. These labels are inspectable in Harbor and verifiable against the GitLab commit history.
Link 4
cosign signature
The image was signed with the NovaCrestBank signing key immediately after being pushed to Harbor. The signature is over the image digest — it is cryptographically impossible to modify the image without invalidating the signature. The signing key is accessible only to the Jenkins ServiceAccount in the jenkins namespace.
Link 5
ACS vulnerability gate
ACS scanned the image at this specific digest before it could proceed. No CRITICAL CVEs were present at scan time. The scan result is recorded in the ACS audit log and in the Jenkins build log (linked from the production MR).
Link 6
Production MR human approval
A named platform engineer reviewed the evidence and approved the production promotion MR. The approval is timestamped and immutable in GitLab's audit log. The approval constitutes the formal change management authorisation required by PCI-DSS Req 6.4.2.
Link 7
ACS admission controller enforcement
When Argo CD deploys the Deployment to the banking-production namespace, OCP's admission webhook calls ACS. ACS verifies the image has a valid NovaCrestBank cosign signature. An unsigned image, a signature from an unknown key, or a signature over a different digest — all rejected at admission, before any container starts.

GITOPS-03 · Section 18
Diagnosing pipeline failures — reading the build log
When the pipeline fails, the build log is the first diagnostic tool. Each stage produces specific error patterns. This section maps the most common failure modes to their root causes.
Stage 2
Maven build fails — unit test failure vs compilation failure
Compilation failure: BUILD FAILURE ... COMPILATION ERROR — a syntax error in Java code. Fix: review the failing file, the compiler error shows the exact line. Unit test failure: Tests run: X, Failures: Y, Errors: Z — check the Surefire report linked in the build. SonarQube quality gate failure: ANALYSIS FAILED due to Quality Gate failure — log into SonarQube at sonarqube.apps.ocp.corp.novacrestbank.com and view the project dashboard to see which specific condition failed (security hotspot, coverage below threshold).
Stage 4
Harbor push failure — auth vs network vs certificate
Auth failure: unauthorized: authentication required — the jenkins-harbor-push-credentials credential may be expired or incorrect. Verify the Harbor service account is active. Network failure: connection refused or no such host — the buildah container cannot reach Harbor. Check: NSX DFW rules allowing jenkins namespace → Harbor. Certificate failure: x509: certificate signed by unknown authority — the NCB CA bundle is not in the buildah container's trust store. Verify the MachineConfig (UPG-03 Section 09) that deploys the CA bundle to worker nodes is applied.
Stage 5
ACS scan failure — CVE found vs connectivity
CVE gate failure: FAIL: Policy "No Critical CVEs" — 1 violation found — the build log shows the CVE ID, affected package, and fix version. The developer must update the dependency and re-run the pipeline. ACS connectivity failure: connection refused to central.stackrox.svc:443 — ACS Central pod may be down or the jenkins namespace NetworkPolicy blocks the connection. Check oc get pods -n stackrox for Central pod health.
Stage 6
Staging deploy timeout — Argo CD sync issue
Timeout after 300s: Argo CD did not sync and mark staging Healthy within 5 minutes. Check: argocd app get payment-service-staging — if Degraded, the pod is CrashLoopBackOff. Check the staging pod logs: oc logs -n banking-staging -l app=payment-service. Common cause: the new image has a startup configuration error that unit tests did not catch (missing environment variable, wrong database connection string). GitLab push failure: Permission denied (publickey) — the jenkins-config-repo-deploy-key SSH key may have expired in GitLab.
Stage 7
Integration test failure — application vs infrastructure
HTTP 4xx from the API: an application-level regression — the new code returns an unexpected response. Check the integration test failure report (linked from the build) and the staging application logs. Connection refused / timeout: an infrastructure issue — the NSX NetworkPolicy may not allow the jenkins namespace to reach banking-staging on port 443, or the staging Route is not reachable. Check: oc get route -n banking-staging and NSX DFW rule for jenkins → banking-staging.

What GITOPS-04 adds on top of GITOPS-03

GITOPS-03 defines what gets promoted to production (the image digest). GITOPS-04 defines how Argo CD deploys it across multiple environments using an ApplicationSet — the Kustomize base-plus-overlay pattern, sync waves for deployment ordering, health checks, and the Argo CD web UI walkthrough. The config repo structure introduced in Section 07 of this guide (base/ and overlays/{dev,staging,production}/) is the exact structure that the GITOPS-04 ApplicationSet configuration reads.
