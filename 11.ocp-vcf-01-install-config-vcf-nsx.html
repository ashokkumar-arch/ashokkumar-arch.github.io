OCP-VCF-01 · Section 01
Why the VCF install-config.yaml is different from standalone vSphere
The install-config.yaml for OpenShift on VMware Cloud Foundation is not a minor variation of the standalone vSphere install-config — it uses a completely different platform.vsphere schema. This section explains why, and what architectural reason drives each change.

The fundamental question
We already have the standalone vSphere install-config from ocp-installation.html. Can we just change the vCenter hostname and reuse it?
No. The standalone vSphere install-config uses a flat schema that reflects a single vCenter managing everything. VCF uses a completely different hierarchical schema that reflects VCF's architecture — a Management Domain vCenter separate from a Workload Domain vCenter, NSX overlay segments instead of VLAN portgroups, NSX Load Balancer VIPs instead of keepalived, and a signed certificate chain instead of a self-signed certificate.

Using the standalone schema against a VCF environment will produce errors like failed to find network, x509: certificate signed by unknown authority, and the installer will be unable to place VMs correctly because it will not understand the VCF Workload Domain topology.

The new schema — introduced in OCP 4.12 — uses platform.vsphere.vcenters (a list), platform.vsphere.failureDomains (topology regions and zones), and platform.vsphere.topology.networks (segment reference). This schema was designed specifically for multi-vCenter environments like VCF.

The two install-config schemas side by side

Standalone vSphere schema (flat)

Single platform.vsphere block with scalar fields: vcenter:, datacenter:, network:, apiVIP:, ingressVIP:. One vCenter, one datacenter, one portgroup name, two VIP addresses. Simple but cannot express topology for VCF Workload Domains.

VCF / multi-vCenter schema (hierarchical)

platform.vsphere.vcenters (list — can express multiple vCenters), platform.vsphere.failureDomains (topology with region/zone/server/topology), topology.networks (NSX segment list), apiVIPs/ingressVIPs (lists). Explicitly expresses VCF topology.

OCP-VCF-01 · Section 02
VCF architecture that shapes the install-config — two vCenters, NSX, and the OCP Workload Domain
Every field that differs in the VCF install-config exists because of a specific architectural reality in NovaCrestBank's VCF deployment. Understanding the architecture makes the YAML obvious rather than arbitrary.

NovaCrestBank VCF — the two-domain architecture

Dom 1
Management Domain
Managed by vcenter-mgmt.novacrestbank.local. Contains: SDDC Manager, vcenter-mgmt itself, vcenter-ocp (the OCP Workload Domain vCenter), NSX Manager cluster (nsx-mgr-01/02/03), NSX Edge nodes (edge-01, edge-02), and infra VMs (DC01, DC02, RootCA, IssuingCA). Runs on esxi-mgmt-01 through esxi-mgmt-04. The OCP installer never touches this domain directly.
Dom 2
OCP Workload Domain
Managed by vcenter-ocp.novacrestbank.local. Contains: esxi-ocp-01 through esxi-ocp-05 (the cluster that runs OCP VMs), vsanDatastore (the datastore for all OCP node VMDKs), NCB-OCP-Datacenter, NCB-OCP-Cluster. The OCP installer connects only to this vCenter. All VM creation happens here.
NSX
Shared NSX fabric
One NSX Manager cluster manages BOTH domains. The segment seg-ocp-nodes (10.50.0.0/24) is an NSX overlay segment visible in vcenter-ocp's network inventory as type OPAQUE_NETWORK. The install-config references this segment by name — not a VLAN ID. NSX LB Virtual Servers serve the API VIP (10.50.0.200:6443) and Ingress VIP (10.50.0.201:443/80) — replacing keepalived entirely.

Why this architecture matters for the install-config

OCP-VCF-01 · Section 03
The six things that change — standalone vSphere vs VCF + NSX, field by field
Reading this section alongside ocp-installation.html shows exactly what is different and why — six changes, each with a clear VCF architectural reason
How to read this section
Each change is shown as a side-by-side comparison: the standalone vSphere value on the left (from ocp-installation.html) versus the VCF+NSX value on the right. Every change has a "Why" that connects it to the VCF architecture explained in Section 02.

Change 1 — vCenter server: vcenter → vcenter-ocp

Standalone vSphere

vcenter: vcenter.novacrestbank.local

Single vCenter manages everything — ESXi hosts, VMs, storage. The installer connects here for all operations.

VCF + NSX

server: vcenter-ocp.novacrestbank.local

The OCP Workload Domain vCenter. OCP VMs live in this domain. The installer must connect here — not vcenter-mgmt — because esxi-ocp-01 through esxi-ocp-05 are only managed by vcenter-ocp.

Change 2 — network: portgroup → NSX segment

Standalone vSphere

network: PG-VLAN50-OCP-Nodes

A vDS distributed portgroup backed by VLAN 50. The installer attaches VM NICs to this portgroup. VLAN ID is the underlying transport.

VCF + NSX

networks: [seg-ocp-nodes]

An NSX overlay segment. No VLAN ID in the traditional sense — it uses GENEVE encapsulation over VLAN 100 (Host TEP). Visible in vcenter-ocp as type OPAQUE_NETWORK. The name must match exactly as it appears in vcenter-ocp's network inventory.

Change 3 — VIPs: scalar strings → lists, keepalived → NSX LB

Standalone vSphere

apiVIP: 10.50.0.200
ingressVIP: 10.50.0.201

Scalar strings. VIPs served by keepalived (VRRP) running inside the OCP nodes themselves. The VIP floats between nodes on L2. Requires the VIP to be on the same /24 subnet as the nodes.

VCF + NSX

apiVIPs: [10.50.0.200]
ingressVIPs: [10.50.0.201]

Lists (plural). VIPs served by NSX Load Balancer Virtual Servers (NCB-VS-OCP-API, NCB-VS-OCP-HTTPS) created in NSX-04. No keepalived. No L2 adjacency requirement. The VIPs already exist before any OCP VM is created.

Change 4 — Certificate trust: not needed → certThumbprint + additionalTrustBundle

Standalone vSphere

The standalone vSphere install used insecureSkipVerify: true or relied on the vCenter bundled self-signed certificate. No additionalTrustBundle needed for vCenter trust.

VCF + NSX

certThumbprint: AABB...7899

WIN-VCF-03 replaced vcenter-ocp's self-signed cert with an IssuingCA-signed cert. The installer validates the cert against this SHA-256 thumbprint. additionalTrustBundle contains the full CA chain (IssuingCA + RootCA) so RHCOS nodes trust the internal PKI.

Change 5 — Datacenter and cluster: NovaCrestBank-DC → NCB-OCP-Datacenter

Standalone vSphere

datacenter: NovaCrestBank-DC

One datacenter object in one vCenter. Simple flat reference.

VCF + NSX

datacenter: NCB-OCP-Datacenter
computeCluster: /NCB-OCP-Datacenter/host/NCB-OCP-Cluster

The OCP Workload Domain in vcenter-ocp has its own datacenter and cluster objects. Full inventory paths are required. Names must exactly match what exists in vcenter-ocp — not vcenter-mgmt.

Change 6 — Schema structure: flat → failureDomains hierarchy

Standalone vSphere

Flat platform.vsphere block. All fields at the same level: vcenter:, datacenter:, network:, diskType:. No topology concept.

VCF + NSX

platform.vsphere.vcenters[] list (connection), platform.vsphere.failureDomains[] list (topology with region/zone/server/topology sub-object). The topology sub-object contains datacenter, computeCluster, datastore, resourcePool, networks. Hierarchical — designed for multi-vCenter, multi-datacenter VCF deployments.

OCP-VCF-01 · Section 06
Collect the vcenter-ocp certificate thumbprint
The certThumbprint field pins the installer to the exact certificate presented by vcenter-ocp. It is the SHA-256 fingerprint of the IssuingCA-signed certificate installed in WIN-VCF-03 — 64 uppercase hex characters, no colons.
Why certThumbprint exists — and why it is different from additionalTrustBundle
We already provide the CA chain in additionalTrustBundle. Why is a separate certThumbprint field also required?
They serve different purposes. additionalTrustBundle tells RHCOS nodes to trust certificates signed by NovaCrestBank-IssuingCA — it is a general CA trust anchor. certThumbprint tells the installer "I specifically expect vcenter-ocp to present a certificate with this exact SHA-256 fingerprint" — it is a per-server identity pin.

If an attacker performed a man-in-the-middle attack between the jump host and vcenter-ocp, they might present a certificate signed by a different CA that the installer doesn't trust, or they might present a certificate signed by a CA that somehow made it into the trust store. The thumbprint prevents both — even if trust is somehow compromised, the fingerprint comparison provides a second check. It also catches accidental misconfiguration: if someone accidentally points the installer at the wrong vCenter, the thumbprint won't match and the installer will refuse to proceed.
jumphost.novacrestbank.local — collect and format the vcenter-ocp thumbprint
# Get the SHA-256 fingerprint of vcenter-ocp's current certificate
[svc-ocp@jumphost ~]$ echo | openssl s_client \
  -connect vcenter-ocp.novacrestbank.local:443 2>/dev/null \
  | openssl x509 -fingerprint -sha256 -noout
SHA256 Fingerprint=AA:BB:CC:DD:EE:FF:00:11:22:33:44:55:66:77:88:99:AA:BB:CC:DD:EE:FF:00:11:22:33:44:55:66:77:88:99

# The install-config requires the thumbprint WITHOUT colons
# Strip the colons and save to an environment variable
[svc-ocp@jumphost ~]$ VCENTER_THUMBPRINT=$(echo | openssl s_client \

# Verify: must be exactly 64 characters, all uppercase hex
[svc-ocp@jumphost ~]$ echo -n $VCENTER_THUMBPRINT | wc -c

OCP-VCF-01 · Section 07
Build the CA bundle for additionalTrustBundle
The additionalTrustBundle field contains the NovaCrestBank CA chain in PEM format. Every line — including the BEGIN/END markers — must be indented by exactly two spaces. IssuingCA first, then RootCA.
The three most common additionalTrustBundle errors
1. Tabs instead of spaces. YAML parsers reject tab indentation. Every line must have exactly 2 spaces of leading indent — use a text editor with visible whitespace.
2. Missing RootCA. The chain must include BOTH IssuingCA AND RootCA. The installer needs the full chain to validate vcenter-ocp's cert all the way to a self-signed root. IssuingCA alone is not enough.
3. Blank line after the |. The PEM content must start on the very next line after additionalTrustBundle: |. A blank line between the pipe and the first -----BEGIN produces an empty trust bundle.
jumphost.novacrestbank.local — build and verify the CA chain
# Copy CA certs from WIN-VCF-03 to the jump host
# (scp or WinSCP from C:\NovaCrest-Certs\ to ~/ocp-kit/certs/)

# Verify both files are valid PEM certificates
[svc-ocp@jumphost certs]$ openssl x509 -in issuingca.cer -noout -subject -issuer
subject=CN=NovaCrestBank-IssuingCA, O=NovaCrestBank, C=AU
issuer=CN=NovaCrestBank-RootCA, O=NovaCrestBank, C=AU
         ↑ IssuingCA issued by RootCA — correct ✓

[svc-ocp@jumphost certs]$ openssl x509 -in rootca.cer -noout -subject -issuer
subject=CN=NovaCrestBank-RootCA, O=NovaCrestBank, C=AU
issuer=CN=NovaCrestBank-RootCA, O=NovaCrestBank, C=AU
         ↑ RootCA is self-signed (subject == issuer) — correct ✓

# Build the combined chain: IssuingCA first, then RootCA
[svc-ocp@jumphost certs]$ cat issuingca.cer rootca.cer > chain.pem

# Verify the chain validates correctly
[svc-ocp@jumphost certs]$ openssl verify -CAfile rootca.cer issuingca.cer
issuingca.cer: OK

# Verify vcenter-ocp's cert validates against this chain
[svc-ocp@jumphost certs]$ echo | openssl s_client \
  -connect vcenter-ocp.novacrestbank.local:443 2>/dev/null \
  | openssl x509 | openssl verify -CAfile chain.pem
stdin: OK
      ↑ vcenter-ocp's certificate validates against our CA chain ✓

# Preview the correctly-indented format for the YAML
# Every line needs exactly 2 spaces of leading indent
[svc-ocp@jumphost certs]$ sed 's/^/  /' chain.pem | head -5

  MIIFzDCCA7SgAwIBAgIQN2uhh...
  ...

OCP-VCF-01 · Section 11
failureDomains — topology and VCF Workload Domain objects
The failureDomains section is where the VCF Workload Domain topology is expressed. It maps region/zone labels to actual vCenter objects. This is the section that does not exist at all in the standalone vSphere install-config.
What failureDomains is — and why VCF needs it when standalone vSphere did not
What is a failureDomain and why does VCF require this section when the standalone vSphere install-config did not have it?
A failure domain is OpenShift's abstraction for "a group of infrastructure that can fail together" — analogous to an AWS availability zone. In a VCF environment, you may have multiple Workload Domains, each with its own vCenter, each treated as a separate availability zone for OCP.

The standalone vSphere install-config did not need this concept because it assumed one vCenter, one datacenter, one cluster — all described with flat scalar fields. VCF changes this: the OCP Workload Domain is one specific region+zone within a larger VCF fabric, and the installer needs to know the full topology of that domain.

For NovaCrestBank's single OCP Workload Domain, we use one failure domain with arbitrary region/zone labels — the labels themselves don't matter as long as they're consistent. What matters are the topology fields inside, which express the exact vCenter object paths.

The server field inside failureDomains must match exactly one of the servers listed in vcenters above. This is how the installer knows which connection details to use for this topology.

OCP-VCF-01 · Section 12
topology.networks — the NSX segment reference
The single most VCF-specific field in the entire install-config. This is what changes when you move from a VLAN portgroup to an NSX overlay network — and understanding why it works requires understanding how NSX segments appear in vCenter.
How NSX segments appear in vCenter — and why the name is all you need
In standalone vSphere we referenced a VLAN portgroup by name: PG-VLAN50-OCP-Nodes. In VCF we reference an NSX segment: seg-ocp-nodes. There's no VLAN ID anywhere. How does the installer know what network to attach the VMs to?
NSX-T registers all overlay segments with vCenter through the NSX-vCenter integration. From vCenter's perspective, seg-ocp-nodes appears in the network inventory alongside vDS portgroups — but with a type of OPAQUE_NETWORK instead of DISTRIBUTED_PORTGROUP. The installer calls the vCenter API, finds the network named seg-ocp-nodes of type OPAQUE_NETWORK, and attaches VM NICs to it. NSX handles all the underlying GENEVE encapsulation transparently.

No VLAN ID is needed because NSX overlay segments don't use VLANs for their data plane — they use GENEVE encapsulation over the Host TEP VLAN (VLAN 100 in this environment). The VM just sees a standard Ethernet NIC. The NSX host agent on esxi-ocp-01 through esxi-ocp-05 intercepts the traffic and encapsulates it in GENEVE tunnels.

Critical: the name is case-sensitive. seg-ocp-nodes is not the same as Seg-OCP-Nodes. The installer performs an exact string match against vCenter's network inventory. Verify the exact name from the vCenter REST API before writing it into the YAML.

OCP-VCF-01 · Section 13
apiVIPs and ingressVIPs — NSX Load Balancer replaces keepalived
In standalone vSphere, OCP uses keepalived (VRRP) running inside the master/infra nodes to serve API and Ingress VIPs. In VCF+NSX, these VIPs are served by NSX LB Virtual Servers that already existed before the first OCP node boots. This is one of the most architecturally significant differences.
keepalived vs NSX LB — what actually changes
In ocp-installation.html the VIPs were served by keepalived inside the OCP nodes. Now NSX LB serves them. What is the practical difference?
With keepalived, the VIP is a "floating IP" that lives inside the OCP cluster — it is served by a VRRP instance running on an OCP master or infra node. It requires L2 adjacency: the VIP must be on the same /24 subnet as the OCP nodes because VRRP works by broadcasting to the same L2 segment. The installer waits for a node to boot and advertise the VIP before the API becomes reachable.

With NSX LB, the VIP is a Virtual Server on the NSX Edge nodes. It exists before any OCP node is created. The NSX LB listens on 10.50.0.200:6443 and 10.50.0.201:443/80 from the moment NCB-VS-OCP-API and NCB-VS-OCP-HTTPS were created in NSX-04. No L2 adjacency requirement — the VIP can be on any routable NSX subnet.

For the install-config this means: apiVIPs and ingressVIPs are IPs that the installer can validate via DNS lookup right now, before any VM is created. The installer checks that api.ocp.novacrestbank.local resolves to the apiVIP value. If they don't match, the installer fails immediately.

Note: the field names changed from singular to plural between schema versions: apiVIP (old) → apiVIPs (list). Always use the list form for OCP 4.12+.

OCP-VCF-01 · Section 14
networking — pod and service CIDRs
The networking section is identical in structure to standalone vSphere. The CIDRs must not overlap with any physical or NSX network in the environment. Verified against all VCF-01 subnets.
OVN-Kubernetes and NSX — two overlay layers that coexist
OCP uses OVN-Kubernetes for pod-level networking (pod-to-pod, pod-to-service). NSX handles node-level networking (GENEVE tunnels between ESXi hosts, routing between segments, LB VIPs). They operate at completely different layers and do not conflict. OVN-K runs inside the node VMs. NSX runs between the node VMs at the hypervisor level. A banking application's network traffic passes through both layers in sequence: application pod → OVN-K overlay → node VM NIC → NSX GENEVE tunnel → destination node NIC → OVN-K overlay → destination pod.

OCP-VCF-01 · Section 16
additionalTrustBundle — the CA chain for vcenter-ocp and RHCOS nodes
The additionalTrustBundle contains the NovaCrestBank CA chain injected into every RHCOS node during provisioning. Without it, nodes cannot trust vcenter-ocp's IssuingCA-signed certificate and the vSphere CSI driver will fail to connect after install.
What additionalTrustBundle does — two separate jobs
We already have certThumbprint for vcenter-ocp. What does additionalTrustBundle add that certThumbprint doesn't cover?
certThumbprint is used by the installer binary on the jump host to verify vcenter-ocp during the install itself. additionalTrustBundle is baked into the RHCOS ignition config and injected into every node VM during provisioning — it gives the running nodes a permanent trust anchor for the internal PKI.

After install, the vSphere CSI driver running inside OCP nodes must connect to vcenter-ocp to provision persistent volumes. The CSI driver runs on RHCOS. Without additionalTrustBundle, the CSI driver would reject vcenter-ocp's IssuingCA-signed certificate with x509: certificate signed by unknown authority — exactly the same error you would see on any Linux system that doesn't trust your internal CA.

Additionally, RHCOS nodes need to trust the internal CA to: use LDAPS to DC01 (after AD OAuth is configured in OCP-VCF-02), trust any internal service that presents IssuingCA-signed certificates, and validate mutual TLS between cluster components. The additionalTrustBundle is a one-time injection at install time — changing it later requires an MCO (Machine Config Operator) update that triggers a rolling node reboot.
install-config.yaml — additionalTrustBundle (format)
additionalTrustBundle: |
    # The | (literal block scalar) preserves newlines exactly.
    # Content starts on the NEXT line — no blank line between | and -----BEGIN.
    # Every line of PEM content must be indented by exactly 2 spaces.
    # Tabs instead of spaces = YAML parser error.
    # IssuingCA first, then RootCA (same order as chain.pem from Section 07).
    -----BEGIN CERTIFICATE-----
    MIIFzDCCA7SgAwIBAgIQN2uhh...   ← IssuingCA (NovaCrestBank-IssuingCA)
    ... (all IssuingCA PEM lines, each prefixed with 2 spaces) ...
    -----END CERTIFICATE-----
    -----BEGIN CERTIFICATE-----
    MIIFxTCCA62gAwIBAgIUN2u...    ← RootCA (NovaCrestBank-RootCA)
    ... (all RootCA PEM lines, each prefixed with 2 spaces) ...
    -----END CERTIFICATE-----
    # Use this command to generate the correctly-indented block:
    #   sed 's/^/  /' ~/ocp-kit/certs/chain.pem
    # Paste the output of that command directly after the | character above.
