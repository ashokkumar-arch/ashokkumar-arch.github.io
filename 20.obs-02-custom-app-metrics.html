The four golden signals — applied to NovaCrestBank banking APIs
Google's Site Reliability Engineering book identifies four signals that, when measured together, give a complete picture of a service's health. For banking APIs, these four signals map directly to the customer experience and regulatory expectations. Everything in this guide is built around them.

Why four signals — not twenty metrics — is the correct starting point
A banking microservice can expose hundreds of metrics — JVM heap, thread count, database connection pool, cache hit rate, HTTP response codes, GC pause duration, and many more. Why start with four signals rather than monitoring everything?
More metrics is not better monitoring. It is more noise with the same signal buried deeper. The danger of measuring everything is alert fatigue — when Grafana dashboards show 40 panels and Alertmanager fires on 15 different thresholds, engineers stop reading the dashboards and start dismissing alerts. The four golden signals are the minimum set that answers the question an on-call engineer actually asks at 2 AM: "Is the service working? Is it working well? Is it going to keep working?"

For a banking payments API specifically, the four signals map to concrete customer and regulatory consequences. Latency above 500ms on a payment API means customers experience a "hung" transaction — they do not know whether the payment submitted or not, which causes duplicate submission attempts. Traffic below baseline means customers are not reaching the service — something upstream has failed. Error rate above 0.1% on a payment endpoint means real failed transactions — customer complaints and potential regulatory reporting. Saturation approaching 100% thread pool usage means the service is about to start rejecting requests with 503 responses.

Start with the four signals. Make them work perfectly — the right metrics, the right PromQL, the right alert thresholds, the right Grafana panels. Add additional metrics only when the four signals are insufficient to diagnose a real incident that occurred.
Signal 1 — Latency: how long does a request take?
What it is: The time between a client sending a request and receiving the response. Measured as a distribution (not an average) because the tail — P99 or P99.9 — is what customers experience when things go wrong.

Banking relevance: UK and Australian payment schemes (BPAY, NPP, Faster Payments) have defined SLAs for payment processing latency. A payment API that processes 99% of requests in under 200ms but takes 30 seconds for 1% has an unacceptable tail even though its average looks healthy. Only a percentile measure reveals the tail.

NovaCrestBank SLO: P99 latency < 500ms for the payment submission endpoint. P50 < 80ms. Measured over a 5-minute rolling window. Alert when P99 exceeds 500ms for 5 consecutive minutes.
Signal 2 — Traffic: how much demand is the service receiving?
What it is: The rate of requests hitting the service — typically measured as requests per second (RPS) or transactions per second (TPS). For a database, it might be queries per second or bytes transferred per second.

Banking relevance: Banking traffic has strong diurnal patterns — peak during business hours (9 AM–5 PM), end-of-month batch processing spikes, low overnight. A traffic drop outside business hours is normal. A traffic drop during business hours at 2 PM on a weekday is an incident — it means customers cannot reach the service. Traffic monitoring detects upstream failures (load balancer misconfiguration, DNS failure) that no service-internal metric can catch.

NovaCrestBank SLO: Alert when traffic drops below 20% of the rolling 7-day average for the same hour of the week during business hours. Alert when traffic exceeds 200% of the rolling average (possible DDoS or loop).
Signal 3 — Errors: what fraction of requests are failing?
What it is: The rate of explicitly failed requests — HTTP 5xx responses, business logic errors (insufficient funds returned as 422), or requests that succeed technically but return incorrect results. Includes both infrastructure errors (the service is crashing) and application errors (a payment is rejected for a business reason).

Banking relevance: Payment API errors fall into two categories that must be measured separately. Infrastructure errors (5xx) represent service failures — the payment may or may not have been processed; the client does not know. Business errors (4xx — insufficient funds, invalid account) are expected and should be monitored for anomaly detection (a sudden spike in "insufficient funds" responses at 3 AM is suspicious). Only 5xx errors affect the SLO; 4xx errors affect fraud detection.

NovaCrestBank SLO: 5xx error rate < 0.1% over any 5-minute window. Alert at 0.05% (warning) and 0.1% (critical). Separately track 4xx business errors for fraud anomaly detection.
Signal 4 — Saturation: how full is the service?
What it is: How much of the service's capacity is being used — what fraction of the service's ability to handle additional load remains. For a thread-pool-based Java service like the payment-service, saturation is thread pool utilisation. For a CPU-bound service, it is CPU utilisation. For a memory-bound process, it is heap usage as a fraction of the max heap.

Banking relevance: Saturation is the predictive signal — high saturation precedes latency increases and errors. A payment-service running at 90% thread pool utilisation will have higher latency (threads queue) and eventually return 503 (thread pool exhausted) before any error rate metric rises. Measuring saturation at 80% and alerting before the service degrades gives the operations team time to scale before customers notice.

NovaCrestBank SLO: Thread pool utilisation < 80% sustained. JVM heap utilisation < 75% sustained. Alert at 80% (warning). Above 90%, latency increases are already occurring — alert critical and scale immediately.
OBS-02 · Section 02
What to measure and what not to measure
Every metric the payment-service exposes is scraped by Prometheus every 30 seconds, stored in TSDB, and queried by Grafana. Cardinality — the number of unique label combinations — determines whether a metric is sustainable or will cause storage and performance problems.
The cardinality problem — why high-cardinality labels destroy monitoring
Should the payment-service include a label for the customer account ID or the transaction ID in its metrics? That would allow per-customer latency analysis.
Never use high-cardinality values — customer IDs, transaction IDs, user email addresses, session tokens — as Prometheus metric labels. This is one of the most common and most damaging mistakes in Prometheus instrumentation.

Here is why. A single metric with a customer ID label — payment_processing_seconds{customer_id="NC-12345678"} — creates one Prometheus time series per unique customer ID. NovaCrestBank has 500,000 customers. That single metric creates 500,000 time series — each scraped every 30 seconds, each stored in TSDB, each tracked by Prometheus's memory index. At 500,000 time series for one metric, Prometheus's memory usage balloons by several gigabytes. The TSDB becomes enormous. Queries against this metric take minutes instead of milliseconds. Prometheus starts dropping samples. The monitoring system collapses under the weight of one badly instrumented metric.

The rule: labels should have low, bounded cardinality — a small, known set of possible values that will not grow proportionally with the number of users or transactions. Good label values: endpoint (10–20 possible values), status_code (5–10 values), environment (3 values), payment_type (5 values: BPAY, NPP, OSKO, Direct Debit, Direct Credit). Bad label values: customer ID, transaction ID, user email, session token, correlation ID.

For per-transaction analysis, use distributed tracing (OBS-04 — Tempo) or structured logs (OBS-03 — LokiStack) — both are designed for high-cardinality data. Prometheus is not.

Micrometer — the instrumentation library for Spring Boot
Micrometer is to metrics what SLF4J is to logging — a vendor-neutral instrumentation facade. The payment-service uses Micrometer with the Prometheus registry backend. Understanding Micrometer's abstraction model makes the code in Section 06 obvious.
Why Micrometer rather than the Prometheus Java client directly
The Prometheus project provides its own Java client library (io.prometheus:simpleclient). Why use Micrometer instead?
The Prometheus Java client is Prometheus-specific — if NovaCrestBank later needs to send metrics to a different backend (Datadog, New Relic, StatsD, or a future system), all instrumentation code must be rewritten. Micrometer is a metrics facade — the application code uses Micrometer's API, and a registry backend translates to the specific wire format. Changing from Prometheus to another system is a configuration change, not a code change.

More practically: Micrometer integrates with Spring Boot Actuator out of the box. Adding one dependency and two configuration lines gives the payment-service automatic instrumentation for: all HTTP request latency and counts (via @Timed and Spring MVC instrumentation), JVM metrics (heap, GC pauses, thread count, class loading), Spring data source metrics (connection pool size, active connections, pending requests), Tomcat/Undertow thread pool metrics. The application team gets the platform metrics for free and only writes custom Micrometer code for business-domain metrics (payment outcome, processing time by payment type).

Micrometer's three core instruments — mapped to Prometheus types

Counter
MeterRegistry.counter()
Increments only. Maps to a Prometheus Counter. Use for: total payment submissions, total payment successes, total payment failures, total API requests. The raw counter value is meaningless — always use rate(counter[5m]) in PromQL to get the per-second rate. Micrometer automatically appends _total to the metric name to follow Prometheus naming convention: payment_submissions becomes payment_submissions_total in the /metrics output.
Timer
MeterRegistry.timer()
Measures durations. Maps to a Prometheus Histogram (with configurable buckets) plus an accompanying Counter for total call count. Use for: HTTP request duration, payment processing duration, database query duration. Micrometer automatically records both the count and the duration. In PromQL, use histogram_quantile(0.99, rate(timer_name_seconds_bucket[5m])) to get P99 latency. The Timer is the most important instrument for banking API monitoring — it gives you both throughput (the count) and latency (the histogram) from a single instrument.
Gauge
MeterRegistry.gauge()
Observes a current value. Maps to a Prometheus Gauge. Use for: queue depth (how many payments are awaiting processing), connection pool active connections, cache size, thread pool active threads. The Gauge reads the value each time Prometheus scrapes the /metrics endpoint — you provide a supplier function that returns the current value. Unlike Counter and Timer, Gauge values are queried directly in PromQL without a rate function.

Adding Micrometer to the payment-service — pom.xml
Two Maven dependencies add Prometheus metrics exposition to the payment-service. The Spring Boot Actuator BOM manages version compatibility automatically — no version numbers need to be specified for Micrometer artifacts.
pom.xml — Micrometer and Spring Boot Actuator dependencies
<!-- Spring Boot Actuator — provides /actuator/prometheus endpoint -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-actuator</artifactId>
    <!-- No version — inherited from spring-boot-starter-parent BOM -->
</dependency>

<!-- Micrometer Prometheus registry — translates Micrometer metrics to Prometheus format -->
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-registry-prometheus</artifactId>
    <!-- No version — managed by spring-boot-dependencies BOM -->
    <!-- This is all that is needed to enable Prometheus metrics.
       With these two dependencies, the /actuator/prometheus endpoint is
       automatically available when the application starts.
       Spring Boot auto-configuration detects the Prometheus registry on the
       classpath and registers it — zero additional Java code required for
       the automatic platform metrics (JVM, HTTP, datasource, thread pool). -->
</dependency>

<!-- Optional: Micrometer tracing bridge for Tempo/Zipkin (used in OBS-04) -->
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
What automatic metrics you get for free — before writing a single line of instrumentation code
With these two dependencies, the payment-service automatically exposes: all JVM metrics (heap by pool, GC pause duration and frequency, thread count by state, class loading), all Tomcat/Undertow thread pool metrics (active threads, max threads, pending queue), all Spring MVC HTTP metrics (request count and latency by URI, method, and status), all HikariCP connection pool metrics (active connections, idle connections, pending acquires, acquisition time), and all Spring Cache metrics (if caching is used). This set of automatic metrics covers three of the four golden signals — traffic (HTTP request count), latency (HTTP request duration), and saturation (thread pool, connection pool, heap) — before any custom code is written. Only the business-domain metrics (payment outcome by type) require custom instrumentation.
OBS-02 · Section 05
Spring Boot Actuator configuration — application.yaml
Spring Boot Actuator is disabled for most endpoints by default for security reasons. The application.yaml enables the Prometheus endpoint, configures metric tags that appear as labels on every metric, and sets histogram bucket boundaries for latency measurement.
application.yaml — Actuator and Micrometer configuration — every field explained
management:
  endpoints:
    web:
      exposure:
        include: prometheus,health,info
          # Expose only these three endpoints. Never expose "env", "beans",
              # "mappings", or "*" — these reveal internal application structure.
              # prometheus: the Prometheus metrics scrape endpoint (/actuator/prometheus)
              # health:     liveness and readiness probes — required by OCP DeploymentConfig
              # info:       application version information — useful for dashboards
  endpoint:
    prometheus:
      enabled: true
    health:
      show-details: never
        # never: health endpoint returns {"status":"UP"} only — not the detailed
            # component health breakdown (DB connection status, disk space, etc.)
            # That detail is only for internal monitoring, not external exposure.
            # For Prometheus liveness/readiness, "UP" is sufficient.
  metrics:
    tags:
      application: payment-service
      team: banking-platform
      environment: ${spring.profiles.active:production}
        # Common tags: added as labels to EVERY metric this application exposes.
            # application: "payment-service" — allows filtering all metrics from this service
            # team: "banking-platform" — for multi-team Grafana dashboards
            # environment: injected from Spring profile — "production" by default.
            # These tags mean every metric looks like:
            # http_server_requests_seconds_count{application="payment-service",
            #   environment="production", team="banking-platform", ...}
    distribution:
      percentiles-histogram:
        http.server.requests: true
        payment.processing.duration: true
          # percentiles-histogram: true tells Micrometer to emit histogram buckets
              # (the _bucket time series) for these metrics instead of just count+sum.
              # WITHOUT this: Micrometer emits count, sum, and max — enough to compute
              # average latency but NOT P99 (you need buckets for histogram_quantile()).
              # WITH this: Micrometer emits _bucket, _count, _sum — enabling P99/P95/P50.
              # Must be true for every Timer metric where you want percentile queries.
      sla:
        http.server.requests: 50ms,100ms,200ms,500ms,1s,2s,5s
        payment.processing.duration: 100ms,200ms,500ms,1s,2s,5s,10s
          # sla: the histogram bucket boundaries (upper bounds).
              # Prometheus records how many requests fell BELOW each boundary.
              # For payment processing: boundaries at 100ms, 200ms, 500ms, 1s, 2s, 5s, 10s
              # match NovaCrestBank's SLO tiers. Fine-grained buckets near the SLO threshold
              # (500ms) allow precise histogram_quantile() computation.
              # Bucket boundaries are configured HERE in application.yaml — they cannot
              # be changed without a code deployment and service restart.
    enable:
      jvm: true
      process: true
      tomcat: true
      hikaricp: true
        # Explicitly enable each metric group. All are enabled by default when
            # the dependency is on the classpath, but being explicit documents the intent.

Writing custom metrics — Counter, Timer, and Gauge in Java
The automatic Micrometer metrics cover platform signals. The business-domain signals — payment outcome rates, processing time by payment type, queue depth — require custom instrumentation in the payment-service Java code. This section shows the complete pattern for each instrument type.
PaymentMetrics.java — custom instrumentation for the four golden signals
@Component
public class PaymentMetrics {

    private final MeterRegistry registry;

    // ── SIGNAL 1: LATENCY — Timer per endpoint and payment type ─────────────────
    private final Timer paymentSubmitTimer;
    private final Timer paymentAuthoriseTimer;

    // ── SIGNAL 2: TRAFFIC — Counter by payment type and outcome ─────────────────
    private final Counter.Builder paymentCounterBuilder;

    // ── SIGNAL 4: SATURATION — Gauge for processing queue depth ─────────────────
    private final Queue<PaymentRequest> processingQueue;

    public PaymentMetrics(MeterRegistry registry,
                          Queue<PaymentRequest> processingQueue) {
        this.registry = registry;
        this.processingQueue = processingQueue;

        // Timer: payment submission latency with histogram buckets
        // Metric name: payment_submit_duration_seconds
        // Additional tags: payment_type is added per-call (see recordPayment() below)
        paymentSubmitTimer = Timer.builder("payment.submit.duration")
            .description("End-to-end payment submission latency including authorisation")
            .publishPercentileHistogram()   // emit _bucket series for histogram_quantile()
            .minimumExpectedValue(Duration.ofMillis(10))   // bucket lower bound
            .maximumExpectedValue(Duration.ofSeconds(10))  // bucket upper bound
            .register(registry);

        paymentAuthoriseTimer = Timer.builder("payment.authorise.duration")
            .description("Time to receive authorisation from the payment rail")
            .publishPercentileHistogram()
            .register(registry);

        // Counter builder: payment outcomes — low cardinality labels only
        // Metric name: payment_outcomes_total
        // Labels: payment_type (BPAY/NPP/OSKO/etc), outcome (success/failed/declined)
        paymentCounterBuilder = Counter.builder("payment.outcomes")
            .description("Count of payment outcomes by type and result");

        // Gauge: processing queue depth — reads queue size on each Prometheus scrape
        // Metric name: payment_queue_depth
        // This is the saturation signal — high queue depth precedes latency increases
        Gauge.builder("payment.queue.depth", processingQueue, Queue::size)
            .description("Number of payments currently awaiting processing")
            .tag("queue", "main-processing")
            .register(registry);
    }

    /**
     * Record a completed payment with its outcome.
     * Called after every payment submission attempt — success or failure.
     * 
     * @param paymentType  BPAY, NPP, OSKO, DirectDebit, DirectCredit
     * @param outcome      success, insufficient_funds, invalid_account, timeout, system_error
     * @param duration     measured by the caller with System.nanoTime() or Instant
     */
    public void recordPayment(String paymentType, String outcome, Duration duration) {
        // Record the Timer — captures duration for latency percentiles
        paymentSubmitTimer.record(duration);

        // Increment the Counter — captures throughput and outcome distribution
        // Labels (tags) are added per-call — low cardinality values from the approved list
        paymentCounterBuilder
            .tag("payment_type", paymentType)  // ~5 values: BPAY, NPP, OSKO, ...
            .tag("outcome", outcome)            // ~8 values: success, declined, ...
            .register(registry)
            .increment();
    }

    /**
     * @Timed annotation on service methods — alternative to manual Timer.record().
     * Spring AOP intercepts the method call and records the duration automatically.
     * Use @Timed for methods where Duration is not already available.
     */
    @Timed(value = "payment.validation.duration",
           description = "Time to validate payment request fields",
           percentiles = {0.5, 0.95, 0.99},
           histogram = true)
    public ValidationResult validatePayment(PaymentRequest request) {
        // Spring AOP records start/end time automatically.
        // The @Timed annotation handles everything — no Timer.record() call needed here.
        return validator.validate(request);
    }
}
Why the Timer is registered once but tagged per-call
The paymentSubmitTimer is built once in the constructor without tags. Tags are added in recordPayment() via .tag("payment_type", paymentType). This is Micrometer's tag-per-call pattern — it creates one time series per unique tag combination (BPAY/success, BPAY/failed, NPP/success, etc.) rather than one monolithic timer. Building the Timer once prevents re-registering it on every call. This is safe because Micrometer's registry is idempotent — registering the same name with the same tags returns the existing meter.

User workload monitoring namespace setup
With enableUserWorkload: true set in OBS-01, the User Workload Prometheus watches all namespaces for ServiceMonitor objects. The banking-production namespace needs one configuration object — a user-workload-monitoring-config ConfigMap — to tune the user workload Prometheus for this namespace's scrape volume.
Two ConfigMaps — one for the platform stack, one for the user workload stack
OBS-01 configured cluster-monitoring-config in openshift-monitoring — this controls the platform Prometheus. The user workload Prometheus is configured separately via user-workload-monitoring-config in openshift-user-workload-monitoring. These are independent — changes to one do not affect the other. For NovaCrestBank's banking application metrics, the user workload ConfigMap sets scrape limits that prevent a badly instrumented application from overwhelming the User Workload Prometheus with millions of time series.
user-workload-monitoring-config — user workload Prometheus tuning
apiVersion: v1
kind: ConfigMap
metadata:
  name: user-workload-monitoring-config
  namespace: openshift-user-workload-monitoring
    # This exact name and namespace — the CMO reads only from here.
data:
  config.yaml: |
        prometheus:
          retention: 24h
            # User workload Prometheus retains only 24 hours of data locally.
              # This is intentional — the platform Prometheus (15 days) captures
              # infrastructure metrics long-term. Application metrics are queried
              # via Thanos Querier which spans both Prometheus instances.
              # 24h retention keeps the user workload PVC small.
          enforcedSampleLimit: 50000
            # Maximum samples (individual metric values) per scrape across
              # ALL ServiceMonitors in ALL namespaces.
              # If a single scrape returns more than 50,000 samples, Prometheus
              # drops the excess and records a scrape error.
              # This prevents a badly instrumented service (for example, one that
              # inadvertently creates high-cardinality metrics) from overwhelming
              # the User Workload Prometheus.
              # With 50 banking services × ~200 metrics each = ~10,000 samples
              # per full scrape cycle. 50,000 provides substantial headroom.
          enforcedTargetLimit: 200
            # Maximum number of scrape targets (individual pod endpoints) across
              # all ServiceMonitors. 200 covers 50 services × 4 replicas each.
              # Prevents runaway autodiscovery from targeting unexpected pods.
          nodeSelector:
            node-role.kubernetes.io/infra: ""
          tolerations:
            - key: node-role.kubernetes.io/infra
              effect: NoSchedule
            # User Workload Prometheus also runs on infra nodes — same isolation
              # as the platform Prometheus. Application metrics scraping should not
              # compete with application workloads for resources on worker nodes.

SLI, SLO, and error budget — the banking reliability model
Alert thresholds are not arbitrary numbers — they derive from SLOs (Service Level Objectives) which derive from SLIs (Service Level Indicators). Understanding the relationship makes every alert threshold a business decision, not a technical guess.
SLI, SLO, and error budget — what each term means in operational terms
What is the difference between an SLI, an SLO, and an error budget? And how do they relate to Prometheus alert thresholds?
An SLI (Service Level Indicator) is a measurable metric that quantifies user experience. An SLO (Service Level Objective) is a target for that metric. An error budget is the amount of time the SLO can be missed before consequences occur.

For the payment-service payment submission endpoint:

SLI: The fraction of payment submission requests that complete with P99 latency under 500ms, measured over a rolling 30-day window.
SLO: This fraction must be at least 99.5% — meaning fewer than 0.5% of requests can exceed 500ms P99 over any rolling 30-day period.
Error budget: 0.5% of requests can exceed 500ms. Over 30 days with an average of 5 TPS, that is approximately 0.005 × 5 × 86400 × 30 = 64,800 "slow" requests per month before the SLO is breached.

The alert threshold relationship: The Prometheus alert fires when the P99 exceeds 500ms for 5 consecutive minutes. This is a burn rate alert — it fires when the service is consuming the error budget faster than it can sustain for the full 30-day window. 5 minutes at double the SLO threshold burns through 10 minutes of error budget. The alert gives the team time to investigate and resolve before the 30-day SLO is actually breached.

Why this matters for banking regulation: The SLO is the technical basis for the service's availability commitments to customers. For payment APIs under the Australian Banking Code of Practice, Faster Payment obligations, or NPP Participant Agreement, the bank must demonstrate it can meet defined processing time SLAs. Prometheus SLO alerts with error budgets provide the continuous monitoring evidence that auditors and regulators expect.

Saturation alert — thread pool and JVM heap approaching limits
Saturation alerts are the predictive signal — they fire before customers see latency increases or errors. High thread pool or heap utilisation is the early warning that the service is approaching its capacity limit.
PrometheusRule — saturation alerts (thread pool, heap, queue depth)
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: payment-service-saturation-alerts
  namespace: banking-production
  labels: {role: alert-rules}
spec:
  groups:
    - name: ncb.payment-service.saturation
      rules:

        - alert: NCBPaymentServiceThreadPoolSaturated

                sum(tomcat_threads_current_threads{namespace="banking-production",application="payment-service"})
              / sum(tomcat_threads_config_max_threads{namespace="banking-production",application="payment-service"})
              > 0.80
            # Thread pool utilisation above 80%.
                # tomcat_threads_current_threads: threads currently handling requests.
                # tomcat_threads_config_max_threads: configured maximum (e.g. 200).
                # At 80%: queuing is beginning, latency is about to increase.
                # Action: check if a scale-out is needed (increase replicas via HPA),
                # or if a slow downstream dependency is holding threads (check DB latency).
          for: 5m
          labels: {severity: warning, team: banking-platform}
          annotations:
            summary: payment-service thread pool is {{ $value | humanizePercentage }} saturated
            description: "payment-service Tomcat thread pool is {{ $value | humanizePercentage }} in use. Latency increases are likely. Scale out or investigate thread-blocking operations (database, downstream services)."

        - alert: NCBPaymentServiceHeapHigh
          expr: |
                sum(jvm_memory_used_bytes{area="heap",namespace="banking-production",application="payment-service"})

              > 0.75

                # Above 75%, the GC is running more frequently to reclaim space.

          for: 10m

          annotations:
            summary: payment-service JVM heap is {{ $value | humanizePercentage }} — GC pressure increasing
            description: "JVM heap usage is {{ $value | humanizePercentage }}. G1GC will begin more frequent collections above 75% heap utilisation. Check for memory leaks: oc exec -n banking-production deployment/payment-service -- jcmd 1 VM.native_memory"

        - alert: NCBPaymentQueueDepthHigh
          expr: |
            payment_queue_depth{namespace="banking-production"} > 100
            # Processing queue depth above 100 — payments backing up.
                # Queue depth growing means the service is consuming payments
                # slower than they are arriving — unsustainable.
                # At 100+ queued payments: latency for queued payments is already high.
          for: 2m
          labels: {severity: critical, team: banking-platform}
          annotations:
            summary: Payment processing queue depth is {{ $value }} — payments backing up
            description: "{{ $value }} payments are queued for processing. Processing throughput has fallen behind submission rate. Investigate: payment processor thread starvation, downstream rail timeouts, database deadlocks."

Dashboard structure — the NovaCrestBank banking microservice dashboard layout

Dashboard design principle — what an on-call engineer needs at 2 AM
How should the Grafana dashboard be organised? Should each panel show a different metric?
An on-call engineer woken at 2 AM by a PagerDuty alert has one question: "What is broken and how bad is it?" The dashboard must answer that question in under 30 seconds without requiring the engineer to scroll, think, or run queries. This determines the entire layout.

Row 1 — Status overview (stat panels): four large numbers answering "is everything broadly OK right now?" — current P99 latency, current error rate, current request rate, current queue depth. Row 2 — Traffic: request rate over time. A traffic drop at an unexpected time is immediately visible as a cliff edge on the graph. A traffic spike causing the saturation issue is visible as a spike.

Row 3 — Latency: P50, P95, and P99 time series overlaid. The gap between P50 and P99 reveals tail latency — a service where P50=50ms but P99=800ms has a specific class of problem (intermittent slow operations, GC pauses, connection pool waits) that differs from a service where P50 and P99 are both 800ms (general overload).

Row 5 — Saturation: thread pool %, heap %, queue depth over time. This row answers "is this an overload issue or a correctness issue?" — if thread pool is at 95% when latency spiked, the cause is capacity exhaustion. If thread pool is at 20% when latency spiked, the cause is a slow external dependency.
