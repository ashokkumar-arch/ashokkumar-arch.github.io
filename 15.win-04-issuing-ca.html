Windows Server · Win-04 · Section 02
Enterprise CA vs Standalone CA — the most important distinction in AD CS
Understanding why the Issuing CA must be an Enterprise CA — and what that unlocks
Enterprise CA — what it requires and what it enables
The Root CA was a Standalone CA. Why must the Issuing CA be an Enterprise CA?
Because Enterprise CA is the only type that supports certificate templates, AD-based authorisation, and auto-enrolment — the three features that make a CA operationally practical.

Enterprise CA requirements: The CA server must be domain-joined. The account running the CA installation must be a Domain Admin (or Enterprise Admin for the first CA in a forest). This is why the Issuing CA joins novacrestbank.local before AD CS is installed.

What Enterprise CA enables:
1. Certificate templates — pre-defined certificate profiles stored in Active Directory. A template defines the key usage, subject alternative names, validity period, and EKUs (Enhanced Key Usages) for a class of certificates. Administrators create templates once; all issued certificates for that purpose use the template automatically. Without templates, every certificate request must be manually configured each time.

2. AD-based authorisation — the CA checks Active Directory to determine who is allowed to request which types of certificates. A Domain Controller requesting a Domain Controller certificate is verified against AD group membership before the CA issues it. This prevents unauthorised systems from obtaining certificates for sensitive roles.

3. Auto-enrolment — domain-joined machines automatically request certificates without administrator involvement. Group Policy tells every machine "you are authorised to request a Computer certificate from the Issuing CA" — the machine's certificate client does it silently in the background. DCs automatically get Domain Controller certificates for LDAPS. This is what enables encrypted LDAP (LDAPS) without manual certificate requests on each DC.
Subordinate CA — why not just create another Root CA
Why make the Issuing CA a Subordinate CA signed by the Root CA — why not create it as its own standalone Root CA?
Because a second independent Root CA would require separate trust distribution — every system would need to trust two separate Root CAs. With a subordinate CA, there is only one root of trust to manage.

If the Issuing CA were its own Root CA: administrators would need to push two Root CA certificates to every machine's Trusted Root store. Every certificate viewer would show two separate certificate chains. If one Root CA is compromised, the other is unaffected — but this also means you cannot revoke the compromised CA using the other one (they have no relationship). Managing two independent PKI hierarchies doubles the operational complexity.

With a subordinate CA: one root of trust (NovaCrestBank-Root-CA) signs the Issuing CA. The Issuing CA can be revoked by the Root CA if compromised. Machines only need to trust one Root CA certificate. The entire PKI has a single authoritative point of trust. This is the correct architecture.

Why 4096-bit key on the Issuing CA — same as Root CA
The Root CA uses 4096-bit RSA. Should the Issuing CA also use 4096, or is 2048 sufficient for a CA that only lasts 10 years?
4096-bit on the Issuing CA for consistency and a unified security posture — but 2048-bit would also be cryptographically adequate for a 10-year Issuing CA.

The security argument: a 10-year Issuing CA with a 2048-bit key is secure today. 2048-bit RSA will remain computationally infeasible to break for the foreseeable future within a 10-year window. The choice of 4096 is about consistency: NovaCrestBank's PKI uses 4096 throughout, making the key strength policy simple — "all CAs use 4096-bit RSA." This eliminates decision fatigue on future CA renewals and makes the PKI specification easy to document for auditors.

There is a minor performance consideration: RSA-4096 signing is slower than RSA-2048. For the Issuing CA which may sign thousands of certificates per year, this is measurable but negligible on modern hardware — the difference is milliseconds per signing operation.

Why Certification Authority Web Enrollment is added on the Issuing CA but not the Root CA
Web Enrollment was explicitly excluded from the Root CA. Why enable it on the Issuing CA?
Because the Issuing CA must serve certificate requests from non-domain-joined devices — particularly ESXi hosts and the Thales HSM — that cannot use the automatic enrolment protocol.

Domain-joined Windows machines use the Microsoft Certificate Enrolment Protocol (CEP/CES or the older DCOM-based autoenrolment) to request certificates automatically. This protocol requires domain membership and Kerberos authentication.

Non-domain devices (ESXi hosts, Linux VMs, network appliances) cannot use this protocol. Web Enrollment provides a browser-based fallback at https://issuingca.novacrestbank.local/certsrv. An administrator logs into the web interface with domain credentials, submits a CSR (certificate signing request) file from the non-domain device, and downloads the signed certificate. This is how ESXi host certificates and OCP certificates will be requested during the vSphere-03 guide.

Web Enrollment on an online Issuing CA is safe — the Issuing CA is always running anyway, and Web Enrollment requires authentication before any certificate is issued.

Why "Send a certificate request to existing enterprise CA" cannot be used here
The wizard offers to send the request automatically to a parent CA. Why use the file-based method instead?
Because the parent CA (NovaCrestBank-Root-CA) is a Standalone CA — not an Enterprise CA — and it is offline.

The automatic send option uses the Microsoft Certificate Enrollment Protocol (MSCEP), which only works with Enterprise CAs in the same Active Directory forest. The Root CA is a Standalone CA specifically because it must not be domain-joined. Additionally, the Root CA is powered off — even if the protocol worked, there is no one to receive the request. The file-based CSR method is the only correct approach for an offline Standalone Root CA.

Step-by-step: sign the Issuing CA CSR on the Root CA

  • 1
    Copy the CSR file from the Issuing CA to the Root CA

    On the Root CA, open File Explorer. In the address bar, type \\issuingca.novacrestbank.local\c$ and press Enter. Authenticate with NOVACRESTBANK\Administrator credentials. Copy IssuingCA.req to C:\IssuingCA.req on the Root CA.

    Why the Root CA can access the Issuing CA's C$ share
    The Root CA is a standalone server (not domain-joined) but it can still connect to the domain network share using explicit domain credentials. When Windows prompts for credentials, enter NOVACRESTBANK\Administrator — the Root CA will authenticate against DC01's NTLM service and gain access to the share.
  • 2
    Submit the CSR to the Root CA

    On the Root CA: Server Manager → Tools → Certification Authority. Right-click NovaCrestBank-Root-CA → All Tasks → Submit new request. Browse to C:\IssuingCA.req. Click Open. The CSR is submitted and goes to Pending Requests.

  • 3
    Issue the certificate — set validity to 10 years first

    Click Pending Requests in the left tree. The Issuing CA request appears. Before issuing, set the validity period. By default the Root CA would issue it for 1 year (its default for issued certificates). You must override this to 10 years:

    Command Prompt — ROOTCA
    # Override the default CA issued cert validity period to 10 years
# This must be done BEFORE issuing the Issuing CA certificate
C:\> certutil -setreg CA\ValidityPeriodUnits 10
Old Value: 1  →  New Value: 10
C:\> certutil -setreg CA\ValidityPeriod "Years"
Old Value: Years  →  New Value: Years
C:\> net stop certsvc && net start certsvc
CertSvc stopped. CertSvc started.

↑ Root CA will now issue the Issuing CA cert for 10 years instead of 1 year

    Back in the Certification Authority console: right-click the pending request → All Tasks → Issue. The certificate moves to Issued Certificates.

  • 4
    Export the signed certificate as a .p7b file

    Click Issued Certificates. Double-click the Issuing CA certificate entry. Click Details tab → Copy to File. Select PKCS #7 (.P7B). Tick "Include all certificates in the certification path". Save as C:\IssuingCA-signed.p7b.

  • 5
    Copy the signed certificate back to the Issuing CA

    On the Root CA, open File Explorer → navigate to \\issuingca.novacrestbank.local\c$ → copy C:\IssuingCA-signed.p7b to C:\IssuingCA-signed.p7b on the Issuing CA.

  • 6
    Also publish a new CRL (while the Root CA is powered on)

    While the Root CA is already powered on, publish an updated CRL: right-click Revoked Certificates → All Tasks → Publish → New CRL. This ensures the CRL is fresh and covers the newly-issued Issuing CA certificate's serial number in the "not revoked" state.

  • 7
    Shut down the Root CA

    Start → Shut down. Wait for Windows to power off. Confirm in vCenter that the VM shows "Powered Off." The Root CA is offline again. Total Root CA uptime: approximately 15–20 minutes.

Windows Server · Win-04 · Section 15
Configure CDP and AIA on the Issuing CA
Setting the CRL and AIA URLs that will appear in every leaf certificate issued — pointing to the same pki.novacrestbank.local web server as the Root CA
Same URL base as the Root CA — same IIS server on DC01
The Issuing CA's CRL files and its certificate are published to the same http://pki.novacrestbank.local/pki/ location as the Root CA. Windows Server · Win-04 · Section 16
Set CRL period and publish the initial CRL
The Issuing CA CRL has a much shorter validity period than the Root CA — weekly publication with daily Delta CRLs
Why the Issuing CA CRL is published weekly — not annually like the Root CA
The Root CA CRL lasts 52 weeks. Why does the Issuing CA CRL last only 7 days?
Because the Issuing CA issues far more certificates than the Root CA, and its revocation list must stay current to be useful.

The Root CA issues maybe two certificates in its entire lifetime (the Issuing CA certificate, and a replacement if needed). Its CRL is a list of "revoked Issuing CA certificates" — almost certainly empty forever. A 52-week CRL is fine because there is almost never anything to put on it, and keeping the Root CA powered on only once a year is a strong security practice.

The Issuing CA issues hundreds of certificates over its lifetime — vCenter, ESXi hosts, OCP certs, DC certs, computer certs for every domain member. If any of these are compromised, they must be revoked immediately and the revocation list updated. A 7-day CRL means: the longest a revoked certificate remains trusted by clients who have cached the previous CRL is 7 days. This is a reasonable tradeoff between freshness and operational overhead.

Delta CRLs (daily) provide an additional mechanism: instead of downloading the full CRL every 7 days, clients download a small "delta CRL" daily that contains only the changes since the last full CRL. This reduces bandwidth and latency for revocation checks.

Windows Server · Win-04 · Section 17
Certificate templates — what they are and why they matter
The feature that makes the Enterprise CA operationally practical — pre-defined certificate profiles for each use case
What a certificate template is
What exactly is a certificate template — and what does it define?
A certificate template is a pre-configured certificate blueprint stored in Active Directory that defines every aspect of how a certificate should be issued for a specific purpose.

A template defines:
Key Usage — what the certificate's key can be used for (Digital Signature, Key Encipherment, Certificate Signing, etc.)
Extended Key Usage (EKU) — the specific application purpose (Server Authentication, Client Authentication, Code Signing, Email Protection, etc.)
Subject — whether the subject is supplied by the requester or built from AD attributes
Validity period — how long the issued certificate is valid
Key length minimum — the minimum key size the CA will accept for this template
Enrolment permissions — which AD groups are allowed to request this type of certificate (e.g., "Domain Controllers" group can request Domain Controller certificates; "Authenticated Users" can request Computer certificates)
Auto-enrolment — whether machines auto-request this certificate type via Group Policy

Without templates, every certificate request must specify all these parameters manually. With templates, an administrator creates the template once and all subsequent certificates of that type use those parameters automatically.

How templates work in practice — the flow

1. Admin creates "Web Server" template in Certificate Templates console
2. Admin enables the template on the Issuing CA ("templates to issue" list)
3. vCenter admin requests a TLS certificate → selects "Web Server" template
4. Issuing CA validates: is the requester in the allowed AD group? → Yes ✓
5. Issuing CA issues cert using template parameters (2yr validity, TLS EKU, 2048+ key)
6. vCenter installs the cert → browsers trust it via the Root CA chain

Windows Server · Win-04 · Section 22
Auto-enrolment — how it works end-to-end
The mechanism that makes certificate management invisible — domain machines request and renew certificates automatically without any administrator action
The complete auto-enrolment flow — from GPO to certificate in the machine store
Auto-enrolment sounds complex. What actually happens step by step when a domain machine gets its certificate automatically?
1. Group Policy defines the certificate enrolment settings. A GPO linked to the domain sets "Certificate Services Client — Auto-Enrollment Settings" to "Enabled" with options to renew expired certificates and update certificates that use certificate templates. This GPO applies to all computers in the domain.

2. The machine receives the GPO. When a domain computer applies Group Policy (at startup, login, or the 90-minute background refresh interval), it receives the auto-enrolment settings.

3. The machine's certificate client contacts the Issuing CA. The certificate client (certmgr in the background) queries Active Directory for the list of certificate templates. It finds NovaCrestBank-Computer in the list with Autoenroll permission for Domain Computers. The machine's computer account is in Domain Computers — it is authorised.

4. The machine generates a private key and CSR. Locally, without any network communication, the machine generates a new RSA key pair. The public key is packaged into a CSR with the machine's AD DNS name as the subject.

5. The CSR is submitted to the Issuing CA. The certificate client sends the CSR to NovaCrestBank-Issuing-CA using the DCOM-based Certificate Enrollment Protocol, authenticated with Kerberos (using the machine's computer account credentials).

6. The CA validates and issues. The Issuing CA verifies the machine is in Domain Computers, verifies the template allows Autoenroll for Domain Computers, and issues the certificate using the NovaCrestBank-Computer template settings.

7. The certificate is installed in the machine store. The certificate is automatically placed in the Certificates (Local Computer) → Personal store. The machine now has a valid, CA-signed certificate. This entire process happens silently, with no administrator or user involvement.
