Application & CI/CD Pipeline

How the Flask dashboard was built across three development iterations, containerised with Docker, tested locally against live Timestream data, and then deployed automatically to ECS Fargate through a fully automated CodePipeline on every code commit.

01 — Overview 02 — Infrastructure 03 — IoT Pipeline 04 — App & CI/CD 05 — Enterprise Extension
The Flask dashboard — what it does

The Dashboard Application

The dashboard is a Python Flask web application. Its job is simple: on every HTTP request to /, query Timestream for the last 10 sensor readings, format them as a readable table, and render an HTML page. Flask is a lightweight Python web framework — it handles the HTTP server, routing, and templating, so the application code focuses entirely on the Timestream query and data presentation.

The application uses boto3 (the AWS Python SDK) to connect to Timestream. Inside ECS Fargate, boto3 uses the ECS task IAM role for authentication — it does not need credentials passed in as environment variables. The task role has timestream:Query and timestream:Select scoped to the specific database and table ARNs, so the application can only read from that one table — nothing else.

Three development iterations — why they happened

Version History

The dashboard went through three distinct iterations during development, each solving a specific problem discovered during testing. This is normal software development — building, testing against real data, identifying issues, and iterating. The three versions were built and tested locally in Docker before any was pushed to CodeCommit for pipeline deployment.

Version 1
Initial working version
First working Flask app that connected to Timestream and returned sensor data. Confirmed the query worked and the application could authenticate using the ECS task role. Data displayed but with formatting issues.
Version 2
Human-readable metrics
Addressed the raw float display issue — temperature, humidity, and air quality values were rounded to 2 decimal places using round(float(col['ScalarValue']), 2). The timestamp was still displaying in raw Timestream format.
Version 3 (final)
Human-readable timestamp
Fixed the timestamp format. Timestream returns timestamps with nanosecond precision (2024-01-15 10:30:45.000000000000). Version 3 parses and reformats using Python's datetime.strptime to display as 2024-01-15 10:30:45.

The final app.py — key sections explained

The Timestream query:

query_string = """
SELECT * FROM "EnvironmentalMonitoring"."SensorData"
ORDER BY time DESC LIMIT 10
"""

ORDER BY time DESC LIMIT 10 retrieves the 10 most recent readings. This always hits the memory store (since data less than 24 hours old lives there) — sub-millisecond latency for the dashboard load.

The timestamp parsing that fixed the v2 display issue:

elif column_names[i] == 'Time':
    data[column_names[i]] = datetime.strptime(
        col['ScalarValue'],
        '%Y-%m-%d %H:%M:%S.%f000000'
    ).strftime('%Y-%m-%d %H:%M:%S')

Timestream returns timestamps as strings with microsecond precision followed by six zeros (representing nanoseconds that are always zero). The strptime format string accounts for this by matching %f (microseconds) followed by the literal string 000000, then reformatting to the clean %Y-%m-%d %H:%M:%S output.

Docker — local testing before pipeline deployment

Containerisation and Local Testing

Each version of the Flask app was containerised and tested locally before being committed to CodeCommit. This validates that the container works correctly in isolation before the CI/CD pipeline is involved. The Dockerfile is straightforward:

FROM python:3.9-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 80
CMD ["python", "app.py"]

Python 3.9-slim is used rather than the full Python image — it produces a significantly smaller image (removes development tools, documentation, and unused libraries), which means faster ECR pulls and smaller attack surface.

Local testing with AWS credentials

Testing locally required passing AWS credentials to the container so boto3 could authenticate to Timestream:

docker run \
  -e AWS_ACCESS_KEY_ID=<ACCESS_KEY> \
  -e AWS_SECRET_ACCESS_KEY=<SECRET_KEY> \
  -e AWS_DEFAULT_REGION=us-east-1 \
  -p 80:80 scems-dashboard-v3
This is a local development pattern only — not used in production: Passing IAM credentials as environment variables to a container is acceptable for local testing on a developer's machine. It is never done in production because the credentials are visible in the container's environment and can be extracted. In ECS Fargate, boto3 automatically uses the ECS task IAM role — no credentials are ever passed in. The docker run command above was only used for local iteration before the pipeline was wired up.
The CI/CD pipeline — all four stages

CodePipeline — From Commit to Deployed Container

Once version 3 was working correctly in local Docker testing, it was pushed to the CodeCommit repository. CodePipeline detected the commit and automatically ran the four-stage pipeline to build and deploy it to ECS Fargate without any manual intervention.

1
Source — CodeCommit
Trigger: Git push to main
CodePipeline polls the CodeCommit repository for changes to the main branch. When a commit is detected, it downloads the source code into the pipeline's S3 artifact bucket and passes it to the Build stage. The S3 artifact bucket is created by the CloudFormation template with versioning enabled — every pipeline execution's source snapshot is retained.
2
Build — CodeBuild
buildspec.yml
CodeBuild executes the buildspec.yml from the repository root. The build specification has three phases. Pre-build: authenticate to ECR using the AWS CLI's get-login-password command piped to docker login. Build: run docker build to build the image and tag it with both the CodeBuild build ID (for traceability) and latest. Post-build: push both tags to ECR and write imagedefinitions.json. 
phases:
  pre_build:
    commands:
      - aws ecr get-login-password --region $AWS_DEFAULT_REGION |
          docker login --username AWS --password-stdin $ECR_REPO_URI
  build:
    commands:
      - docker build -t $ECR_REPO_URI:$CODEBUILD_BUILD_ID .
      - docker tag $ECR_REPO_URI:$CODEBUILD_BUILD_ID $ECR_REPO_URI:latest
  post_build:
    commands:
      - docker push $ECR_REPO_URI:$CODEBUILD_BUILD_ID
      - docker push $ECR_REPO_URI:latest
      - printf '[{"name":"scems-dashboard","imageUri":"%s"}]'
          $ECR_REPO_URI:$CODEBUILD_BUILD_ID > imagedefinitions.json
3
Deploy — ECS Fargate
imagedefinitions.json
The Deploy stage reads imagedefinitions.json from the CodeBuild output artifact. This file maps the container name (scems-dashboard) to the new ECR image URI including the build-specific tag. CodePipeline passes this to ECS, which registers a new task definition revision pointing to the new image and performs a rolling deployment — launching new tasks with the new image before draining the old ones. Zero-downtime deployment.
4
Verification
ALB health check
After ECS launches the new tasks, the ALB target group health check (GET / returning HTTP 200) determines when the new tasks are healthy. Only healthy tasks receive traffic. The old tasks are drained and terminated after the new ones pass health checks. The ALB DNS name continues serving requests throughout the entire deployment.
Why the build ID tag matters: Tagging every image with both latest and the CodeBuild build ID means you always know which pipeline run produced which image. If the new version causes issues, you can roll back to the previous task definition revision (which still references the previous build ID tag) without rebuilding. latest alone provides no traceability.
