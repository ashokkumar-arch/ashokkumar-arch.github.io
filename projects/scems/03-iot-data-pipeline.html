IoT Data Pipeline

The complete data path from a simulated environmental sensor publishing over MQTT, through IoT Core's Topic Rule engine, through Lambda's transformation and write logic, and into Timestream's time-series storage — with KMS encryption at every stage where data is at rest.

The pipeline — step by step

From Sensor to Database — Every Step Explained

1

Device Authentication — X.509 Mutual TLS

Before the sensor can publish a single message, it must authenticate to IoT Core. This is not a username and password — it is mutual TLS (mTLS) using X.509 certificates. Both sides authenticate each other: the device proves its identity to IoT Core using its client certificate, and IoT Core proves its identity to the device using AWS's root CA certificate.

The client certificate was created during the EC2 UserData bootstrap and is stored at /opt/iot/certs/EnvironmentalSensor.pem.crt. The private key is at /opt/iot/certs/EnvironmentalSensor.pem.key. The AWS root CA certificate is downloaded from Amazon's certificate authority at bootstrap time.

The AWS IoT Python SDK handles the TLS handshake automatically — you provide the file paths and the endpoint, and the SDK takes care of presenting the certificate, verifying the server's certificate, and establishing the encrypted channel on port 8883.

myAWSIoTMQTTClient = AWSIoTMQTTClient("EnvironmentalSensor-001")
myAWSIoTMQTTClient.configureEndpoint(IOT_ENDPOINT, 8883)
myAWSIoTMQTTClient.configureCredentials(
    ROOT_CA_PATH,
    PRIVATE_KEY_PATH,
    CERTIFICATE_PATH
)
Why mTLS and not API keys? API keys are strings — they can be copied, shared, or leaked in logs. An X.509 private key is a cryptographic asset that never leaves the device. Even if someone reads the key file off the EC2 instance, they cannot use it to authenticate from another device without also having access to the registered certificate's identity in IoT Core. For IoT devices at scale, certificate-based authentication is the industry standard.
2

IoT Policy — What the Device Is Allowed to Do

Having a valid certificate is not enough — the certificate must also be attached to an IoT Policy that defines what the device is permitted to do. The policy attached to the EnvironmentalSensor certificate grants four actions:

{
  "Effect": "Allow",
  "Action": [
    "iot:Connect",
    "iot:Publish",
    "iot:Subscribe",
    "iot:Receive"
  ],
  "Resource": "*"
}

iot:Connect allows the device to establish a connection to IoT Core. iot:Publish allows it to send messages to topics. iot:Subscribe and iot:Receive allow it to receive messages — useful for command-and-control scenarios where the cloud sends instructions back to the device.

Capstone simplification — wildcard Resource: The Resource is set to *, meaning the device could theoretically publish to any topic. In a production deployment with many devices, each device's policy would scope the Resource to its own client ID and specific topic: arn:aws:iot:region:account:topic/environmental/sensor/${iot:ClientId}. This prevents one compromised device from publishing to another device's topic.
3

MQTT Publish — the Sensor Message

Once authenticated and connected, the sensor publishes a JSON payload to the environmental/sensor topic every 5 seconds. MQTT is a lightweight publish-subscribe protocol designed for IoT — it runs over TCP, uses a broker (IoT Core plays the role of the broker), and is far more efficient than HTTP for high-frequency sensor data.

payload = json.dumps({
    "deviceId": "EnvironmentalSensor-001",
    "temperature": round(random.uniform(15.0, 35.0), 2),
    "humidity": round(random.uniform(30.0, 80.0), 2),
    "airQuality": round(random.uniform(0.0, 500.0), 2),
    "timestamp": int(time.time())
})
myAWSIoTMQTTClient.publish("environmental/sensor", payload, 1)

The QoS (Quality of Service) level is 1, which means "at least once delivery" — IoT Core will confirm receipt of the message and retry if the confirmation is not received. QoS 0 is fire-and-forget; QoS 1 guarantees the message is delivered but may result in duplicates if the network is unreliable. For sensor data where losing a reading is acceptable but sending the same reading twice is harmless, QoS 1 is the right choice.

4

IoT Topic Rule — Routing Messages to Lambda

The IoT Topic Rule is the routing engine inside IoT Core. It listens on the environmental/sensor topic and triggers an action whenever a message arrives. The rule uses SQL to filter messages:

SELECT * FROM 'environmental/sensor'

SELECT * passes the entire message payload through to the action. A production rule might filter on specific conditions — for example, SELECT * FROM 'environmental/sensor' WHERE temperature > 30 to trigger an alarm only when temperature exceeds a threshold — but for this project, all readings are forwarded.

The action is a Lambda invocation. The rule holds a LambdaAction that references the Lambda function ARN. A separate AWS::Lambda::Permission resource grants lambda:InvokeFunction to iot.amazonaws.com scoped to this specific rule ARN — without this permission, IoT Core cannot invoke the function even though the rule tells it to.

Why this permission model matters: The Lambda permission resource is what separates IoT Core's ability to invoke this function from anyone else's. Even if an attacker could craft a message that looked like it came from IoT Core, they could not invoke the Lambda directly — only IoT Core's service principal acting on behalf of this specific rule can invoke it.
5

Lambda — Transform and Write to Timestream

The Lambda function receives the IoT message payload as its event, transforms it into Timestream's required multi-measure record format, and writes it to the database. The function is defined inline in CloudFormation (Python 3.9 runtime).

First, it discovers the Timestream write endpoint. Timestream uses a cell-based architecture with regional endpoint discovery — rather than hardcoding an endpoint, the correct approach is to call describe_endpoints() first:

write_client = boto3.client('timestream-write')
endpoint = write_client.describe_endpoints()['Endpoints'][0]['Address']
ts_write = boto3.client(
    'timestream-write',
    endpoint_url=f'https://{endpoint}'
)

Then it constructs the multi-measure record. Timestream's multi-measure format allows multiple metrics to be stored in a single row — more efficient than the alternative single-measure format which would create one row per metric:

record = {
    'Dimensions': [
        {'Name': 'DeviceId', 'Value': payload.get('deviceId', 'unknown')}
    ],
    'MeasureName': 'EnvironmentalReading',
    'MeasureValueType': 'MULTI',
    'MeasureValues': [
        {'Name': 'temperature', 'Value': str(payload['temperature']),
         'Type': 'DOUBLE'},
        {'Name': 'humidity', 'Value': str(payload['humidity']),
         'Type': 'DOUBLE'},
        {'Name': 'airQuality', 'Value': str(payload['airQuality']),
         'Type': 'DOUBLE'}
    ],
    'Time': str(int(time.time() * 1000)),
    'TimeUnit': 'MILLISECONDS'
}

The record is then written using write_records() with the database name and table name. If the write fails, Lambda logs the error to CloudWatch Logs — the execution role has the necessary CloudWatch Logs permissions.

6

Timestream — Time-Series Storage with KMS Encryption

Amazon Timestream is purpose-built for time-series data. Unlike a relational database, Timestream automatically manages data tiering — recent data stays in the memory store (fast, expensive), and older data is automatically moved to the magnetic store (slower, cheap). For environmental monitoring data, the access pattern is almost always "show me the last N readings" rather than deep historical queries, which makes Timestream's tiering model ideal.

The database is defined in CloudFormation with a customer-managed KMS key:

EnvironmentalMonitoringDB:
  Type: AWS::Timestream::Database
  Properties:
    DatabaseName: EnvironmentalMonitoring
    KmsKeyId: !GetAtt KMSKey.Arn

The table defines the retention periods — how long data stays in the memory store before being moved to the magnetic store, and how long it stays in the magnetic store before being deleted:

SensorDataTable:
  Type: AWS::Timestream::Table
  Properties:
    DatabaseName: !Ref EnvironmentalMonitoringDB
    TableName: SensorData
    RetentionProperties:
      MemoryStoreRetentionPeriodInHours: "24"
      MagneticStoreRetentionPeriodInDays: "365"

24 hours in memory store means the dashboard (which queries the last 10 readings) always hits hot storage with sub-millisecond latency. Data older than 24 hours moves to magnetic store where it is available for longer-term analysis at lower cost.

KMS encryption in context: Every record written by Lambda is encrypted using the customer-managed KMS key before being stored. When the ECS Fargate task queries Timestream for the dashboard, Timestream decrypts the data transparently — but only because the ECS task role has kms:Decrypt on that specific key ARN. If a different service tried to query the same Timestream table without the right KMS permission, the query would fail even if that service had timestream:Query.
