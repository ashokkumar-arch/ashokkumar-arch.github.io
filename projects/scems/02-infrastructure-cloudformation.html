Why CloudFormation — the IaC decision

Infrastructure as Code from Day One

The decision to put everything in a single CloudFormation template rather than clicking through the console was deliberate. In a production environment, infrastructure that lives only in the console is infrastructure that cannot be version-controlled, peer-reviewed, reproduced, or audited. CloudFormation enforces a discipline: every resource, every configuration, every permission must be explicitly declared.

A single template also means the entire environment can be torn down with aws cloudformation delete-stack and rebuilt identically from scratch — which is exactly what was done repeatedly during development as the design evolved. For a sandbox environment that is billed by the hour, this also matters for cost control.

The template is published at: ashokkumar-arch/scems-capstone/infrastructure/scems-ecs.yaml — every resource described in this guide is visible and inspectable there.
VPC and networking — what was built

VPC Design

The template creates a dedicated VPC for the project rather than using the default VPC. This is the correct baseline even for a capstone — it provides network isolation, explicit CIDR control, and a clean security group boundary.

What the template creates

VPC10.0.0.0/16 CIDR
Public Subnet A10.0.1.0/24 · us-east-1a
Public Subnet B10.0.2.0/24 · us-east-1b
Internet GatewayAttached to VPC
Route Table0.0.0.0/0 → IGW
Subnet associationsBoth subnets → Route Table
Capstone simplification — all resources in public subnets: All subnets use MapPublicIpOnLaunch: true and there are no private subnets. The EC2 sensor, ECS Fargate tasks, and Lambda all have public IPs or internet exposure. In a production enterprise deployment, only the ALB would sit in public subnets — all compute, Lambda, and the ECS service would be in private subnets with a NAT Gateway for outbound access. See the Enterprise Extension guide for the production VPC architecture.

Security groups

Three security groups are defined — one per tier:

ALB Security Group: Inbound HTTP (port 80) from 0.0.0.0/0 — the ALB must be reachable from the internet for the dashboard. Outbound all traffic to the VPC.

ECS Security Group: Inbound on port 80 from the ALB security group only — not from the internet directly. The ALB is the only entry point to the ECS tasks. This is the correct pattern even in the capstone.

EC2 (IoT Sensor) Security Group: Inbound SSH (port 22) from 0.0.0.0/0 for console access during development. Outbound all traffic for IoT Core MQTT over TLS (port 8883). In production, SSH would be replaced by SSM Session Manager and port 22 would be closed entirely — see the Enterprise Extension guide.

EC2 UserData — full automation of IoT device setup

How the IoT Sensor Configures Itself

The most complex part of the CloudFormation template is the EC2 UserData script — a 100+ line shell script that runs automatically when the instance launches and turns a blank Amazon Linux 2 instance into a fully configured, registered, publishing IoT device. Nothing is done manually.

The UserData script executes in this sequence:

Phase 1 — System preparation

yum update -y
yum install -y python3 python3-pip aws-cli jq
pip3 install AWSIoTPythonSDK boto3

Installs the AWS IoT Python SDK (which handles MQTT, X.509 mutual TLS, and the connection lifecycle) and the AWS Python SDK (boto3) for any AWS API calls the sensor script needs to make.

Phase 2 — X.509 certificate creation and IoT registration

This is the critical security step. Rather than using a pre-created certificate, the UserData script creates the X.509 certificate at launch time using the AWS CLI, which calls the IoT Core certificate authority:

aws iot create-keys-and-certificate \
  --set-as-active \
  --certificate-pem-outfile /opt/iot/certs/EnvironmentalSensor.pem.crt \
  --private-key-outfile /opt/iot/certs/EnvironmentalSensor.pem.key \
  --region us-east-1

The certificate is immediately marked as active and written to the instance's local filesystem. The certificate ID is extracted from the response using jq and stored for the next step.

Phase 3 — IoT Thing attachment and policy attachment

aws iot attach-thing-principal \
  --thing-name EnvironmentalSensor \
  --principal $CERT_ARN

aws iot attach-policy \
  --policy-name EnvironmentalSensorPolicy \
  --target $CERT_ARN

This associates the freshly created certificate with both the IoT Thing (which was already created by CloudFormation) and the IoT Policy (which defines what the device is allowed to do on IoT Core). Without both attachments, the device cannot authenticate.

Phase 4 — IoT endpoint discovery

IOT_ENDPOINT=$(aws iot describe-endpoint \
  --endpoint-type iot:Data-ATS \
  --query 'endpointAddress' \
  --output text)

Each AWS account has a unique IoT Core data endpoint. The script retrieves it dynamically at runtime rather than hardcoding it — this is the correct approach because IoT endpoints are account-specific and would be wrong if the template were deployed to a different account.

Phase 5 — Python sensor script generation

The UserData script writes a complete Python sensor simulator to /opt/iot/sensor.py using a heredoc. The sensor script uses the AWS IoT Python SDK to establish an mTLS connection to IoT Core using the certificate created in Phase 2, then publishes simulated readings every 5 seconds:

payload = json.dumps({
    "deviceId": "EnvironmentalSensor-001",
    "temperature": round(random.uniform(15.0, 35.0), 2),
    "humidity": round(random.uniform(30.0, 80.0), 2),
    "airQuality": round(random.uniform(0.0, 500.0), 2),
    "timestamp": int(time.time())
})

The topic is environmental/sensor — exactly what the IoT Topic Rule listens on.

Phase 6 — systemd service installation

[Unit]
Description=IoT Environmental Sensor
After=network.target

[Service]
ExecStart=/usr/bin/python3 /opt/iot/sensor.py
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target

The sensor is registered as a systemd service with Restart=on-failure. If the MQTT connection drops or the script crashes, systemd automatically restarts it after 5 seconds. This makes the sensor self-healing without any manual intervention.

KMS key policy — why it is designed this way

Customer-Managed KMS Key

The Timestream database is encrypted at rest using a customer-managed KMS key rather than AWS-managed encryption. This gives the account owner explicit control over key rotation, usage auditing, and key deletion — which matters for compliance and for the ability to demonstrate that encryption is something you configured, not something that was there by default.

The key policy structure

The KMS key policy has two statements. The first grants kms:* to the root account principal — this is the standard bootstrapping pattern that enables IAM policies to delegate specific KMS permissions to IAM roles. Without this statement, no IAM role could use the key regardless of what permissions the role's policy grants.

The second aspect of the design is what is absent: there is no kms:* granted to any service role. Lambda gets kms:Decrypt via its execution role policy, scoped to this key's ARN. The ECS task role gets kms:Decrypt scoped to this key's ARN. Neither role can rotate the key, schedule deletion, or grant others access to it.

Why this matters: If the Lambda execution role were compromised, the attacker could decrypt data from Timestream but could not rotate or delete the key, could not grant themselves or others access to the key, and could not use the key for any other purpose. The blast radius of a credential compromise is bounded by the scope of the role's permissions.
Resource dependency graph

Deployment Order and DependsOn

CloudFormation resolves dependencies between resources automatically when they reference each other via !Ref or !GetAtt. However, some dependencies are not expressed through references and must be declared explicitly using DependsOn. The template uses this in several places:

The IoT Topic Rule depends on the Lambda function being created first, because the rule's action references the Lambda ARN and must grant invoke permission to IoT Core. The ECS Service depends on the ALB Listener being fully created, because ECS needs the listener to exist before it can register targets. The CodePipeline depends on the S3 artifact bucket and both the CodeBuild project and ECS service existing.

CloudFormation handles the parallel creation of independent resources automatically — the VPC, KMS key, IAM roles, ECR repository, and IoT Thing are all created in parallel since none depends on the others. Only the resources that genuinely depend on others wait.
