Project context

What This Project Is — and What It Deliberately Is Not

This project was built as an academic capstone in a sandbox AWS environment, which means some production constraints were intentionally simplified — the ALB public IP is accessed directly in the browser rather than through Route 53 and ACM-issued TLS, SSH is open to the internet rather than replaced by SSM Session Manager, and all resources sit in public subnets rather than a VPC with public/private subnet separation.

Every one of those simplifications is deliberate and acknowledged. The purpose of this series is twofold: first, to document exactly what was built and how it works; and second, in the final guide (Enterprise Extension), to show precisely how a production enterprise deployment would close each of those gaps — including security hardening, resilience, observability, and Cloud FinOps cost optimisation.

Why this framing matters: A Consulting Architect is not expected to have built every system to production grade in a personal lab. What they are expected to demonstrate is that they understand the delta between a proof-of-concept and a production deployment — and can articulate exactly what it takes to close it. That is what the Enterprise Extension guide is for.
Problem statement

The Problem Being Solved

A municipal government responsible for smart city initiatives needs real-time monitoring of environmental parameters — temperature, humidity, and air quality — across multiple sensor sites. Their existing infrastructure is on-premises, limited by high maintenance costs, no real-time processing capability, no automated backup, and no integrated security monitoring.

The objective: design and deploy a scalable, secure, and cost-effective cloud system that ingests sensor data in real time, stores it as time-series data, surfaces it through a web dashboard, and automates the entire application delivery pipeline — with security and cost monitoring built in from the start.

AWS services — what each one does

Service Map

IoT Layer
AWS IoT Core
Manages device connectivity and message routing. The EC2 sensor connects using X.509 certificate mutual TLS (mTLS) authentication. Messages published to the environmental/sensor MQTT topic are routed by a Topic Rule to the Lambda processor.
Compute — Sensor
Amazon EC2 (t2.micro)
Simulates the physical IoT sensor device. The EC2 UserData script automatically installs AWS IoT SDK, creates X.509 certificates, attaches the IoT policy and Thing, and starts a systemd service that publishes temperature, humidity, and air quality readings every 5 seconds.
Compute — Processing
AWS Lambda
Serverless function triggered by the IoT Topic Rule. Receives the sensor payload, transforms it into Timestream's multi-measure record format, and writes it to the time-series database. Runs only when data arrives — zero idle cost.
Storage
Amazon Timestream
Fully managed time-series database. Receives sensor records from Lambda. KMS-encrypted at rest using a customer-managed key. Queried by the Flask dashboard using SQL. The EnvironmentalMonitoring database contains the SensorData table.
Encryption
AWS KMS
Customer-managed KMS key encrypts the Timestream database at rest. Both Lambda and the ECS task role have kms:Decrypt scoped explicitly to this key ARN — not to *.
Application
Amazon ECS Fargate
Runs the Flask web dashboard as a Docker container without managing servers. Task definition specifies 0.25 vCPU and 0.5 GB memory. The ECS task role grants timestream:Query and timestream:Select scoped to the specific database and table ARNs.
Networking
Application Load Balancer
Distributes incoming HTTP traffic to the ECS Fargate tasks. Health check on / path. The ALB DNS name is the public endpoint for the dashboard in this capstone. In production, this would sit behind Route 53 and ACM TLS — see the Enterprise Extension guide.
Image Registry
Amazon ECR
Private container registry storing the Flask dashboard Docker image. CodeBuild pushes new image versions here after each build. ECS pulls the image from ECR when launching new tasks.
CI/CD
AWS CodePipeline
Orchestrates the three-stage pipeline: Source (CodeCommit) → Build (CodeBuild) → Deploy (ECS). Automatically triggered on every commit to the CodeCommit repository.
CI/CD
AWS CodeBuild
Executes the buildspec.yml — authenticates to ECR, builds the Docker image, tags it with the CodeBuild build ID, pushes to ECR, and writes imagedefinitions.json for the ECS deploy stage.
Source Control
AWS CodeCommit
Git repository hosting the Flask application source. The pipeline watches the main branch for commits. HTTPS Git credentials used for authentication.
Security Monitoring
AWS Security Hub
Enabled with AWS Foundational Security Best Practices standard. Continuously evaluates resource configurations against security controls and surfaces findings in a centralised dashboard.
Data Protection
AWS Backup
Automated backup plan covering EC2 instances and EBS volumes. Daily backups with 30-day retention. Backup vault with vault lock enabled for tamper-proof backup storage.
Cost Management
AWS Budgets
Monthly cost budget with alert at 80% threshold. Notification sent to the project administrator email. Estimated total monthly cost for all services: $70. See the Enterprise Extension guide for Cloud FinOps optimisation strategies.
IAM role design

Least-Privilege IAM Architecture

Six distinct IAM roles are designed — one per service boundary. Each role grants only the permissions that specific service needs to perform its function, scoped to specific resource ARNs where possible rather than wildcards.

RolePrincipalKey PermissionsScope
Lambda Execution Role lambda.amazonaws.com timestream:WriteRecords, timestream:DescribeEndpoints, kms:Decrypt, CloudWatch Logs Scoped to the specific Timestream table ARN and KMS key ARN — not *
ECS Task Execution Role ecs-tasks.amazonaws.com ECR pull permissions, CloudWatch Logs create/write Allows ECS to pull the container image from ECR and write task logs
ECS Task Role ecs-tasks.amazonaws.com timestream:Query, timestream:Select, kms:Decrypt, timestream:DescribeEndpoints Scoped to specific Timestream database and table ARNs and the KMS key ARN
CodeBuild Service Role codebuild.amazonaws.com ECR authentication and push, CloudWatch Logs scoped to the specific log group prefix, S3 for artifacts CloudWatch Logs scoped to build-specific log group — not account-wide
CodePipeline Service Role codepipeline.amazonaws.com CodeCommit read, CodeBuild start, ECS register task + update service, S3 artifacts, ECR describe Uses AWS managed CodePipeline policy pattern — production would scope to specific pipeline resources
EC2 Instance Profile ec2.amazonaws.com IoT credential provider (to exchange EC2 identity for IoT credentials), CloudWatch Logs for sensor logs No direct Timestream or Lambda access — sensor communicates only through IoT Core
Design principle: The Lambda role gets timestream:WriteRecords only on the specific table ARN — not timestream:* on *. The ECS task role gets timestream:Query and timestream:Select only on the specific database and table ARNs. The KMS kms:Decrypt on both roles is scoped to the specific key ARN. This is least-privilege applied correctly, not just claimed.
Implementation environment

How the Project Was Built

The implementation was developed and tested on a CentOS 9 Stream virtual machine running on VMware Workstation Pro (free for personal use). AWS CLI was configured with IAM user access keys for the sandbox account. The Flask dashboard application went through three development iterations locally (building and testing Docker images versions 1, 2, and 3) before the final version was pushed to CodeCommit and deployed automatically to ECS Fargate through the CI/CD pipeline.

The entire AWS infrastructure — 35+ resources — was deployed with a single command:

aws cloudformation create-stack \
  --stack-name scems \
  --template-body file://scems-ecs.yaml \
  --parameters ParameterKey=KeyName,ParameterValue=IoT_Device_VM \
  --capabilities CAPABILITY_NAMED_IAM
