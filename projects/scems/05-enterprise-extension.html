Enterprise Extension —
Production Hardening & FinOps

How a production enterprise deployment of SCEMS would differ from the capstone. Every simplification made in the sandbox is named, explained, and closed — covering network architecture, security hardening, resilience, observability, CI/CD maturity, and a structured Cloud FinOps cost optimisation model.

Why this guide exists: A Consulting Architect is not judged on whether their lab matches production perfectly — they are judged on whether they understand the gap and can close it. This is how consulting architects think when reviewing a client's existing deployment.
01 · Security Hardening

Network & Access Security

VPC Architecture — Public vs Private Subnet Separation

The capstone places all resources in public subnets with MapPublicIpOnLaunch: true. This means every EC2 instance, ECS Fargate task, and Lambda function has a public IP address reachable from the internet. In a production enterprise deployment, this is never the case for compute resources.

The correct architecture separates resources by their need for internet exposure:

Internet
Users / IoT Devices / Admin
Public subnets
ALB (HTTPS only)
NAT Gateway
Private subnets
ECS Fargate tasks
Lambda
EC2 (IoT sensor)
CodeBuild

The ALB is the only resource with a public IP. All compute resources sit in private subnets with no public IPs. Outbound internet access for private resources (for IoT Core MQTT, ECR pulls, Timestream API calls) goes through the NAT Gateway in the public subnet. VPC Endpoints for IoT Core, Timestream, ECR, and SSM eliminate the need for even NAT Gateway traffic for those services — keeping API traffic within the AWS backbone entirely.

Capstone gap
SSH port 22 open to 0.0.0.0/0
Replace with AWS SSM Session Manager. No open inbound ports on EC2. Session access is controlled by IAM policy, logged to CloudTrail and S3, and works without a bastion host or VPN.
Capstone gap
IoT policy Resource: "*"
Scope to specific topic ARNs per device: arn:aws:iot:region:account:topic/environmental/${iot:ClientId}. Prevents a compromised device certificate from publishing to other devices' topics.
Capstone gap
ALB serving plain HTTP
Add ACM-issued TLS certificate to the ALB listener (HTTPS on port 443). Add HTTP→HTTPS redirect listener on port 80. Add Route 53 CNAME or alias record pointing to the ALB DNS name.
Capstone gap
CodePipeline role uses wildcard policy
Replace the AWS managed CodePipeline boilerplate policy with a custom policy scoped to the specific pipeline's CodeCommit repo, CodeBuild project, ECS service, and S3 artifact bucket ARNs.
Capstone gap
No WAF on ALB
Associate an AWS WAF WebACL with the ALB. Enable OWASP Top 10 managed rule group, rate-based rule limiting requests per IP (prevents scraping), and geo-restriction if the dashboard serves a defined region only.
Capstone gap
No account-level threat detection
Enable GuardDuty for ML-based threat detection (unusual API calls, cryptocurrency mining, data exfiltration patterns) and CloudTrail for complete API audit logging to S3 with Object Lock for tamper-proof retention.
02 · Resilience

High Availability & Disaster Recovery

ECS Fargate — Multi-AZ with Auto Scaling

The capstone deploys a single ECS task. In production, the ECS service would be configured with a minimum of 2 desired tasks spread across multiple Availability Zones. Application Auto Scaling would be configured to scale based on ALB request count or ECS service CPU utilisation — adding tasks during load spikes and removing them during quiet periods.

Timestream itself is inherently multi-AZ — data is replicated across multiple AZs within the region. This does not require any configuration; it is built into the service. Documenting this explicitly matters for client conversations about availability — it is a strength of the managed service choice.

Lambda Dead-Letter Queue

If the Lambda function fails to write to Timestream (network error, permissions issue, Timestream throttle), the IoT Topic Rule retries the invocation. If all retries fail, the message is lost unless a Dead-Letter Queue (DLQ) is configured. In production, an SQS queue is configured as the Lambda DLQ — failed invocations land in the queue for manual inspection and reprocessing, rather than being silently dropped.

AWS Backup — Extended to Cross-Region

The capstone configures AWS Backup with a 30-day retention policy. In production for a smart city government client, the backup plan would include a cross-region backup copy rule — replicating backups to a secondary region for geographic disaster recovery. The Recovery Time Objective (RTO) and Recovery Point Objective (RPO) for the system would be documented in a Business Impact Assessment and the backup retention period set accordingly.

03 · Observability

Monitoring, Alerting & Tracing

CloudWatch — What to Monitor

The capstone notes CloudWatch as optional and does not implement it. In production, the following metrics and alarms would be configured as a baseline:

Lambda error rate alarm
Alarm when Errors / Invocations > 5% over a 5-minute window. SNS notification to the operations team. This detects Timestream write failures before they cause a visible data gap in the dashboard.
ECS task restart alarm
CloudWatch Container Insights tracks ECS task restart counts. Alarm when a task restarts more than twice in 10 minutes — indicates a recurring application crash that needs investigation.
ALB 5xx rate alarm
Alarm when HTTPCode_Target_5XX_Count > 10 in 5 minutes. 5xx errors from ECS tasks indicate application failures, not client errors — they require operator attention.
IoT message delivery alarm
IoT Core publishes RuleMessageThrottled and TopicMatch metrics to CloudWatch. Alarm when RuleMessageThrottled > 0 — indicates the IoT Topic Rule is being throttled and sensor data is being lost.
Timestream write latency
Monitor WriteRecords.SuccessfulRequestLatency. Unexpectedly high write latency from Lambda indicates a Timestream performance issue that will eventually manifest as Lambda timeouts.
Security Hub findings alarm
EventBridge rule triggering SNS when Security Hub generates a HIGH or CRITICAL severity finding. Security issues surface to the operator within minutes of detection rather than waiting for a periodic dashboard review.

X-Ray Distributed Tracing

With X-Ray enabled on Lambda, every invocation generates a trace showing the complete execution timeline — time spent on the Timestream describe_endpoints() call, time on the write_records() call, and any errors at each step. For a system where Lambda is the bridge between IoT Core and Timestream, this makes it possible to distinguish "Timestream is slow" from "Lambda code has an inefficiency" without adding any logging code to the function.

04 · CI/CD Maturity

Pipeline Hardening

Image Security Scanning in CodeBuild

The capstone's buildspec.yml builds and pushes the Docker image with no security scanning. In production, the build phase would include a container image vulnerability scan using ECR's built-in scanning (powered by Clair) or an explicit Trivy scan step. The pipeline would fail if HIGH or CRITICAL vulnerabilities are found in the base image or installed packages — preventing known-vulnerable images from reaching production.

Manual Approval Before Production

The capstone pipeline deploys automatically from CodeCommit commit to ECS Fargate with no human review. For a government smart city system, an additional pipeline stage would be added between Build and Deploy: a Manual Approval action requiring a named approver to review the build output and confirm deployment. This creates a formal change control gate — auditable, traceable, and reversible.

ECR Lifecycle Policy

Without an ECR lifecycle policy, every image pushed by CodeBuild accumulates in the repository indefinitely. At daily deployments, this is 365 image versions per year. An ECR lifecycle policy retaining only the last 10 tagged images eliminates this cost entirely:

{
  "rules": [{
    "rulePriority": 1,
    "description": "Retain last 10 tagged images",
    "selection": {
      "tagStatus": "tagged",
      "tagPrefixList": ["build-"],
      "countType": "imageCountMoreThan",
      "countNumber": 10
    },
    "action": { "type": "expire" }
  }]
}
05 · Cloud FinOps

Cost Optimisation — How Enterprises Save Money

Cloud FinOps is the discipline of bringing financial accountability to cloud spending — making cost visible, allocating it correctly, and optimising it without sacrificing reliability. The capstone estimates $70/month for the sandbox. The table below shows how a production enterprise deployment of SCEMS would apply FinOps principles to reduce the ongoing operational cost.

Service / AreaFinOps OptimisationMechanismSavings
ECS Fargate Compute Savings Plans Commit to a baseline amount of Fargate compute (vCPU-hours and GB-hours) for 1 or 3 years. Savings Plans apply automatically to any Fargate usage up to the committed amount. No instance type lock-in — unlike Reserved Instances. Up to 52%
Timestream Memory store retention tuning Memory store is priced at ~$0.036/GB-hour. Magnetic store is ~$0.03/GB-month — roughly 25× cheaper per GB stored. Setting memory store retention to 1 hour (sufficient for a dashboard showing last 10 readings) and magnetic store to 365 days means only 1 hour of data lives in the expensive tier at any time. Significant
S3 (logs, artifacts, backups) S3 Intelligent-Tiering Enable S3 Intelligent-Tiering on the artifact bucket, ALB access log bucket, CloudTrail bucket, and AWS Backup output bucket. Objects not accessed for 30 days automatically move to the Infrequent Access tier (40% lower storage cost) with no retrieval penalty for compliance-driven access. 40% on cold data
Lambda Memory optimisation via Lambda Power Tuning Lambda pricing = duration × memory. The default 128MB may not be optimal. Lambda Power Tuning (an open-source AWS Step Functions state machine) runs the function at multiple memory configurations and reports cost vs duration. For Timestream write functions, 256MB often delivers the same or better duration at lower cost than 512MB. 10–30% on Lambda cost
ECR Lifecycle policy As described in the CI/CD section — retaining only the last 10 tagged images via lifecycle policy eliminates the cost of accumulated build artifacts. ECR storage is $0.10/GB-month; 365 images at ~150MB each = ~5.5GB = $0.55/month that grows indefinitely without a lifecycle policy. Eliminates waste
IoT Core Message batching at the device IoT Core pricing is per message. At 1 message/5 seconds, the sensor publishes 17,280 messages/day. If the use case permits, batching 5 readings per message (publish every 25 seconds) reduces messages by 80% at the cost of a 25-second data freshness lag — acceptable for environmental monitoring, significant for billing. 80% on IoT messages
All services Resource tagging strategy Apply Project: SCEMS, Environment: prod, CostCenter: SmartCity-IT tags to every resource. AWS Cost Explorer groups costs by tag — enabling showback (visibility) and chargeback (allocation) to specific city departments. AWS Config rule required-tags enforces tagging compliance and flags untagged resources automatically. Visibility & governance
All services Cost Anomaly Detection AWS Cost Anomaly Detection uses ML to identify unexpected spending spikes — for example, if a bug causes Lambda to invoke 10,000 times per minute instead of once per 5 seconds. Anomaly alerts via SNS catch runaway costs before they accumulate. This is the safety net beneath AWS Budgets alerts. Risk mitigation

The FinOps Operating Model

Cost optimisation is not a one-time activity — it is an operating discipline. For a production smart city deployment, the FinOps operating model would include a weekly Cost Explorer review by the infrastructure team (10 minutes, checking for anomalies and unexpected service growth), a monthly right-sizing review using AWS Compute Optimizer recommendations for any EC2 instances, a quarterly Savings Plans review to evaluate commitment levels against actual usage, and an annual architecture review to evaluate whether newer services (Graviton-based Fargate, Timestream Influx DB compatibility mode) offer cost improvements.

FinOps principle: The goal is not the lowest possible cost — it is the best value at acceptable reliability. Cutting the memory store retention to 1 minute saves money but means the dashboard query hits magnetic store latency on every page load. The right answer balances cost against the user experience the application is supposed to deliver. That trade-off is a business decision, not a technical one — and surfacing it clearly to stakeholders is the consulting architect's job.
